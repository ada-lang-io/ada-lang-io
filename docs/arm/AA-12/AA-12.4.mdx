<MarginText>1</MarginText>
<p>[ A generic formal object can be used to pass a value or variable to a generic unit.] <br /></p>


#### Language Design Principles

<AnnotatedOnly>
<MarginText>1.a</MarginText>
<Admonition type="aarm" aarm="note">
A generic formal object of mode <strong>in</strong> is like a constant initialized to the value of the <code><a href="/docs/arm/AA-12/AA-12.3#S0318">explicit_generic_actual_parameter</a></code>.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>1.b</MarginText>
<Admonition type="aarm" aarm="note">
A generic formal object of mode <strong>in out</strong> is like a renaming of the <code><a href="/docs/arm/AA-12/AA-12.3#S0318">explicit_generic_actual_parameter</a></code>. <br />
</Admonition>
</AnnotatedOnly>


#### Syntax

<MarginText>2/3</MarginText>
<MarginInfo items={["AI95-00423-01", "AI05-0005-1", "AI05-0183-1"]} />

<CodeBlock>
<code>formal{"_"}object{"_"}declaration</code><a id="S0319"></a><code> ::= </code><br />    <code><a href="/docs/arm/AA-3/AA-3.3#S0033">defining_identifier_list</a></code> : <code><a href="/docs/arm/AA-6/AA-6.1#S0208">mode</a></code> [<code><a href="/docs/arm/AA-3/AA-3.10#S0083">null_exclusion</a></code>] <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code> [:= <code><a href="/docs/arm/AA-3/AA-3.7#S0063">default_expression</a></code>]<br />        [<code><a href="/docs/arm/AA-13/AA-13.1#S0346">aspect_specification</a></code>];<br />  |  <code><a href="/docs/arm/AA-3/AA-3.3#S0033">defining_identifier_list</a></code> : <code><a href="/docs/arm/AA-6/AA-6.1#S0208">mode</a></code> <code><a href="/docs/arm/AA-3/AA-3.10#S0084">access_definition</a></code> [:= <code><a href="/docs/arm/AA-3/AA-3.7#S0063">default_expression</a></code>]<br />        [<code><a href="/docs/arm/AA-13/AA-13.1#S0346">aspect_specification</a></code>];<br />

#### Name Resolution Rules

<MarginText>3</MarginText>
<p>The expected type for the <code><a href="/docs/arm/AA-3/AA-3.7#S0063">default_expression</a></code>, if any, of a formal object is the type of the formal object.<br /></p>

<MarginText>4</MarginText>
<p>For a generic formal object of mode <strong>in</strong>, the expected type for the actual is the type of the formal.<br /></p>

<MarginText>5/2</MarginText>
<MarginInfo items={["AI95-00423-01"]} />
<p>For a generic formal object of mode <strong>in out</strong>, the type of the actual shall resolve to the type determined by the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code>, or for a <code><a href="/docs/arm/AA-12/AA-12.4#S0319">formal_object_declaration</a></code> with an <code><a href="/docs/arm/AA-3/AA-3.10#S0084">access_definition</a></code>, to a specific anonymous access type. If the anonymous access type is an access-to-object type, the type of the actual shall have the same designated type as that of the <code><a href="/docs/arm/AA-3/AA-3.10#S0084">access_definition</a></code>. If the anonymous access type is an access-to-subprogram type, the type of the actual shall have a designated profile which is type conformant with that of the <code><a href="/docs/arm/AA-3/AA-3.10#S0084">access_definition</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>5.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>See the corresponding rule for <code><a href="/docs/arm/AA-8/AA-8.5#S0239">object_renaming_declaration</a></code>s for a discussion of the reason for this rule. <br />
</Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>6</MarginText>
<p>If a generic formal object has a <code><a href="/docs/arm/AA-3/AA-3.7#S0063">default_expression</a></code>, then the mode shall be <strong>in</strong> [(either explicitly or by default)]; otherwise, its mode shall be either <strong>in</strong> or <strong>in out</strong>. <br /></p>

<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Mode <strong>out</strong> is not allowed for generic formal objects. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>7</MarginText>
<p>For a generic formal object of mode <strong>in</strong>, the actual shall be an <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code>. For a generic formal object of mode <strong>in out</strong>, the actual shall be a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> that denotes a variable for which renaming is allowed (see <a href="/docs/arm/AA-8/AA-8.5#Subclause_8.5.1">8.5.1</a>). <br /></p>

<AnnotatedOnly>
<MarginText>7.a</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>The part of this that requires an <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> or <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is a Name Resolution Rule, but that's too pedantic to worry about. (The part about denoting a variable, and renaming being allowed, is most certainly <em>not</em> a Name Resolution Rule.) <br />
</Admonition>
</AnnotatedOnly>

<MarginText>8/2</MarginText>
<MarginInfo items={["AI95-00287-01", "AI95-00423-01"]} />
<p>In the case where the type of the formal is defined by an <code><a href="/docs/arm/AA-3/AA-3.10#S0084">access_definition</a></code>, the type of the actual and the type of the formal:<br /></p>

<MarginText>8.1/2</MarginText>
<MarginInfo items={["AI95-00423-01"]} />
<ul>
<li>shall both be access-to-object types with statically matching designated subtypes and with both or neither being access-to-constant types; or <br /></li>
<MarginText>8.2/2</MarginText>
<MarginInfo items={["AI95-00423-01"]} />
<li>shall both be access-to-subprogram types with subtype conformant designated profiles. <br /></li>
</ul>
<MarginText>8.3/5</MarginText>
<MarginInfo items={["AI95-00423-01", "AI12-0287-1"]} />
<p>For a <code><a href="/docs/arm/AA-12/AA-12.4#S0319">formal_object_declaration</a></code> of mode <strong>in out</strong> with a <code><a href="/docs/arm/AA-3/AA-3.10#S0083">null_exclusion</a></code> or an <code><a href="/docs/arm/AA-3/AA-3.10#S0084">access_definition</a></code> that has a <code><a href="/docs/arm/AA-3/AA-3.10#S0083">null_exclusion</a></code>, the subtype of the actual matching the <code><a href="/docs/arm/AA-12/AA-12.4#S0319">formal_object_declaration</a></code> shall exclude null. In addition, if the actual matching the <code><a href="/docs/arm/AA-12/AA-12.4#S0319">formal_object_declaration</a></code> statically denotes the generic formal object of mode <strong>in out</strong> of another generic unit <em>G</em>, and the instantiation containing the actual occurs within the body of <em>G</em> or within the body of a generic unit declared within the declarative region of <em>G</em>, then the declaration of the formal object of <em>G</em> shall have a <code><a href="/docs/arm/AA-3/AA-3.10#S0083">null_exclusion</a></code>. In addition to the places where Legality Rules normally apply (see <a href="/docs/arm/AA-12/AA-12.3">12.3</a>), this rule applies also in the private part of an instance of a generic unit.<br /></p>

<AnnotatedOnly>
<MarginText>8.a/5</MarginText>
<MarginInfo items={["AI95-00287-01", "AI95-00423-01", "AI12-0287-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>This rule prevents &ldquo;lying&rdquo;. <strong>Null</strong> must never be the value of an object with an explicit <code><a href="/docs/arm/AA-3/AA-3.10#S0083">null_exclusion</a></code>. The &ldquo;in addition&rdquo; part is an assume-the-worst rule which prevents trouble in generic bodies (including bodies of child units) when the subtype of the formal object of mode <strong>in out</strong> excludes null implicitly. Since a generic formal object of mode <strong>in</strong> is like a constant initialized to the value of the actual, the runtime check performed by the initialization is enough to prevent lying; thus we don't need a Legality Rule for such objects. <br />
</Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>9/2</MarginText>
<MarginInfo items={["AI95-00255-01", "AI95-00423-01"]} />
<p>A <code><a href="/docs/arm/AA-12/AA-12.4#S0319">formal_object_declaration</a></code> declares a generic formal object. The default mode is <strong>in</strong>. For a formal object of mode <strong>in</strong>, the nominal subtype is the one denoted by the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code> or <code><a href="/docs/arm/AA-3/AA-3.10#S0084">access_definition</a></code> in the declaration of the formal. For a formal object of mode <strong>in out</strong>, its type is determined by the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code> or <code><a href="/docs/arm/AA-3/AA-3.10#S0084">access_definition</a></code> in the declaration; its nominal subtype is nonstatic, even if the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code> denotes a static subtype; for a composite type, its nominal subtype is unconstrained if the first subtype of the type is unconstrained[, even if the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code> denotes a constrained subtype]. <br /></p>

<AnnotatedOnly>
<MarginText>9.a/2</MarginText>
<MarginInfo items={["AI95-00255-01"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>We require that the subtype is unconstrained because a formal <strong>in out</strong> acts like a renaming, and thus the given subtype is ignored for purposes of matching; any value of the type can be passed. Thus we can assume only that the object is constrained if the first subtype is constrained (and thus there can be no unconstrained subtypes for the type). If we didn't do this, it would be possible to rename or take 'Access of components that could disappear due to an assignment to the whole object. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.b/2</MarginText>
<MarginInfo items={["AI95-00423-01"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>The two &ldquo;even if&rdquo; clauses are OK even though they don't mention <code><a href="/docs/arm/AA-3/AA-3.10#S0084">access_definition</a></code>s; an access subtype can neither be a static subtype nor be a composite type. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>10/2</MarginText>
<MarginInfo items={["AI95-00269-01"]} />
<p>In an instance, a <code><a href="/docs/arm/AA-12/AA-12.4#S0319">formal_object_declaration</a></code> of mode <strong>in</strong> is a <em>full constant declaration</em> and declares a new stand-alone constant object whose initialization expression is the actual, whereas a <code><a href="/docs/arm/AA-12/AA-12.4#S0319">formal_object_declaration</a></code> of mode <strong>in out</strong> declares a view whose properties are identical to those of the actual. <br /></p>

<AnnotatedOnly>
<MarginText>10.a/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>These rules imply that generic formal objects of mode <strong>in</strong> are passed by copy (or are built-in-place for a limited type), whereas generic formal objects of mode <strong>in out</strong> are passed by reference.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.b</MarginText>
<Admonition type="aarm" aarm="note">
Initialization and finalization happen for the constant declared by a <code><a href="/docs/arm/AA-12/AA-12.4#S0319">formal_object_declaration</a></code> of mode <strong>in</strong> as for any constant; see <a href="/docs/arm/AA-3/AA-3.3#Subclause_3.3.1">3.3.1</a>, &ldquo;<a href="/docs/arm/AA-3/AA-3.3#Subclause_3.3.1">Object Declarations</a>&rdquo; and <a href="/docs/arm/AA-7/AA-7.6">7.6</a>, &ldquo;<a href="/docs/arm/AA-7/AA-7.6">Assignment and Finalization</a>&rdquo;.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.c</MarginText>
<Admonition type="aarm" aarm="note">
In an instance, the subtype of a generic formal object of mode <strong>in</strong> is as for the equivalent constant. In an instance, the subtype of a generic formal object of mode <strong>in out</strong> is the subtype of the corresponding generic actual. <br />
</Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>11</MarginText>
<p>For the evaluation of a <code><a href="/docs/arm/AA-12/AA-12.3#S0317">generic_association</a></code> for a formal object of mode <strong>in</strong>, a constant object is created, the value of the actual parameter is converted to the nominal subtype of the formal object, and assigned to the object[, including any value adjustment &mdash; see <a href="/docs/arm/AA-7/AA-7.6">7.6</a>]. <br /></p>

<AnnotatedOnly>
<MarginText>11.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This includes evaluating the actual and doing a subtype conversion, which might raise an exception. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.b</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The rule for evaluating a <code><a href="/docs/arm/AA-12/AA-12.3#S0317">generic_association</a></code> for a formal object of mode <strong>in out</strong> is covered by the general Dynamic Semantics rule in <a href="/docs/arm/AA-12/AA-12.3">12.3</a>. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   The constraints that apply to a generic formal object of mode <strong>in out</strong> are those of the corresponding generic actual parameter (not those implied by the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code> that appears in the <code><a href="/docs/arm/AA-12/AA-12.4#S0319">formal_object_declaration</a></code>). Therefore, to avoid confusion, it is recommended that the name of a first subtype be used for the declaration of such a formal object. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Constraint checks are done at instantiation time for formal objects of mode <strong>in</strong>, but not for formal objects of mode <strong>in out</strong>. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>

#### Extensions to Ada 83

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>12.b</MarginText>
<Admonition type="aarm" aarm="note">
In Ada 83, it is forbidden to pass a (nongeneric) formal parameter of mode <strong>out</strong>, or a subcomponent thereof, to a generic formal object of mode <strong>in out</strong>. This restriction is removed in Ada 95. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>

#### Wording Changes from Ada 83

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>12.c</MarginText>
<Admonition type="aarm" aarm="note">
We make &ldquo;<code><a href="/docs/arm/AA-6/AA-6.1#S0208">mode</a></code>&rdquo; explicit in the syntax. RM83 refers to the mode without saying what it is. This is also more uniform with the way (nongeneric) formal parameters are defined.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.d</MarginText>
<Admonition type="aarm" aarm="note">
We considered allowing mode <strong>out</strong> in Ada 95, for uniformity with (nongeneric) formal parameters. The semantics would be identical for modes <strong>in out</strong> and <strong>out</strong>. (Note that generic formal objects of mode <strong>in out</strong> are passed by reference. Note that for (nongeneric) formal parameters that are allowed to be passed by reference, the semantics of <strong>in out</strong> and <strong>out</strong> is the same. The difference might serve as documentation. The same would be true for generic formal objects, if <strong>out</strong> were allowed, so it would be consistent.) We decided not to make this change, because it does not produce any important benefit, and any change has some cost. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>

</AnnotatedOnly>
<AnnotatedOnly>

The actual for such an object must be built-in-place via a <code><a href="/docs/arm/AA-6/AA-6.4#S0218">function_call</a></code> or <code><a href="/docs/arm/AA-4/AA-4.3#S0106">aggregate</a></code>, see <a href="/docs/arm/AA-7/AA-7.5">7.5</a>.<br />
</Admonition>

<AnnotatedOnly>
<MarginText>12.f/2</MarginText>
<MarginInfo items={["AI95-00423-01"]} />

A generic formal object can have a <code><a href="/docs/arm/AA-3/AA-3.10#S0083">null_exclusion</a></code> or an anonymous access type. <br />
</Admonition>
</AnnotatedOnly>
