<MarginText>1</MarginText>
<p>[A <code><a href="../AA-12/AA-12.1#S0310">generic_declaration</a></code> declares a generic unit, which is either a generic subprogram or a generic package. A <code><a href="../AA-12/AA-12.1#S0310">generic_declaration</a></code> includes a <code><a href="../AA-12/AA-12.1#S0313">generic_formal_part</a></code> declaring any generic formal parameters. A generic formal parameter can be an object; alternatively (unlike a parameter of a subprogram), it can be a type, a subprogram, or a package.] <br /></p>

<MarginText>2</MarginText>

<CodeBlock>

</CodeBlock>

</CodeBlock>
<MarginText>7</MarginText>
<p class="Indented2">The only form of <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code> allowed within a <code><a href="../AA-12/AA-12.1#S0313">generic_formal_part</a></code> is a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code> [(that is, the <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code> shall not include an explicit <code><a href="../AA-3/AA-3.2#S0029">constraint</a></code>)]. The defining name of a generic subprogram shall be an <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> [(not an <code><a href="../AA-6/AA-6.1#S0202">operator_symbol</a></code>)]. <br /></p>

<AnnotatedOnly>
<MarginText>7.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The reason for forbidding <code><a href="../AA-3/AA-3.2#S0029">constraint</a></code>s in <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code>s is that it simplifies the elaboration of <code><a href="../AA-12/AA-12.1#S0310">generic_declaration</a></code>s (since there is nothing to evaluate), and that it simplifies the matching rules, and makes them more checkable at compile time. <br /></Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>8/2</MarginText>
<MarginInfo items={["AI95-00434-01"]} />
<p>A <code><a href="../AA-12/AA-12.1#S0310">generic_declaration</a></code> declares a generic unit &mdash; a generic package, generic procedure, or generic function, as appropriate.<br /></p>

<MarginText>9</MarginText>
<p>An entity is a <em>generic formal</em> entity if it is declared by a <code><a href="../AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code>. &ldquo;Generic formal&rdquo;, or simply &ldquo;formal&rdquo;, is used as a prefix in referring to objects, subtypes (and types), functions, procedures and packages, that are generic formal entities, as well as to their respective declarations. [Examples: &ldquo;generic formal procedure&rdquo; or a &ldquo;formal integer type declaration&rdquo;.]<br /></p>

<MarginText>9.1/5</MarginText>
<MarginInfo items={["AI12-0371-1"]} />
<p>The list of <code><a href="../AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code>s of a <code><a href="../AA-12/AA-12.1#S0313">generic_formal_part</a></code> form a <em>declaration list</em> of the generic unit.<br /></p>

<AnnotatedOnly>
<MarginText>9.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Aspect specifications (see <a href="../AA-13/AA-13.1#Subclause_13.1.1">13.1.1</a>) given in a generic formal part can only use declarations given in the formal part, and not those in the visible part of the generic unit. <br /></Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>10</MarginText>
<p>The elaboration of a <code><a href="../AA-12/AA-12.1#S0310">generic_declaration</a></code> has no effect. <br /></p>

<AnnotatedOnly>
<MarginText>11</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 1   Outside a generic unit a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes the <code><a href="../AA-12/AA-12.1#S0310">generic_declaration</a></code> denotes the generic unit. In contrast, within the declarative region of the generic unit, a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes the <code><a href="../AA-12/AA-12.1#S0310">generic_declaration</a></code> denotes the current instance. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.a</MarginText>
<Admonition type="aarm" aarm="proof">
<strong></strong>This is stated officially as part of the &ldquo;current instance&rdquo; rule in <a href="../AA-8/AA-8.6">8.6</a>, &ldquo;<a href="../AA-8/AA-8.6">The Context of Overload Resolution</a>&rdquo;. See also <a href="../AA-12/AA-12.3">12.3</a>, &ldquo;<a href="../AA-12/AA-12.3">Generic Instantiation</a>&rdquo;. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 2   Within a generic <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code>, the name of this program unit acts as the name of a subprogram. Hence this name can be overloaded, and it can appear in a recursive call of the current instance. For the same reason, this name cannot appear after the reserved word <strong>new</strong> in a (recursive) <code><a href="../AA-12/AA-12.3#S0315">generic_instantiation</a></code>.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>13/5</MarginText>
<MarginInfo items={["AI12-0447-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 3   A <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code> or <code><a href="../AA-12/AA-12.6#S0339">default_name</a></code> appearing in a <code><a href="../AA-12/AA-12.1#S0313">generic_formal_part</a></code> is not evaluated during elaboration of the <code><a href="../AA-12/AA-12.1#S0313">generic_formal_part</a></code>; instead, it is evaluated when used. However, the  usual visibility rules apply to any <code><a href="../AA-4/AA-4.1#S0091">name</a></code> used in a default, with name resolution performed based on the location of the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> within the <code><a href="../AA-12/AA-12.1#S0313">generic_formal_part</a></code>.  <br /></Admonition>
</AnnotatedOnly>


#### Examples

<MarginText>14</MarginText>
<p><em>Examples of generic formal parts:</em> <br /></p>

<MarginText>15</MarginText>
<CodeBlock language="ada">
generic     --  parameterless {"\n"}
<MarginText>16</MarginText>
generic{"\n"}   Size : Natural;  --  formal object {"\n"}
<MarginText>17</MarginText>
generic{"\n"}   Length : Integer := 200;          -- formal object with a default expression{"\n"}
<MarginText>18</MarginText>
   Area   : Integer := Length*Length; -- formal object with a default expression{"\n"}
<MarginText>19</MarginText>
generic{"\n"}   type Item  is private;                       -- formal type{"\n"}   type Index is ({"<"}{">"});                          -- formal type{"\n"}   type Row   is array(Index range {"<"}{">"}) of Item; -- formal type{"\n"}   with function "{"<"}"(X, Y : Item) return Boolean;    -- formal subprogram {"\n"}

</CodeBlock>
<MarginText>20</MarginText>
<p><em>Examples of generic declarations declaring generic subprograms Exchange and Squaring:</em> <br /></p>

<MarginText>21</MarginText>
<CodeBlock language="ada">
generic{"\n"}   type Elem is private;{"\n"}procedure Exchange(U, V : in out Elem);{"\n"}
<MarginText>22/5</MarginText>
<MarginInfo items={["AI12-0178-1"]} />
generic{"\n"}   type Item ({"<"}{">"}) is private;{"\n"}   with function "*"(U, V : Item) return Item is {"<"}{">"};{"\n"}function Squaring(X : Item) return Item;{"\n"}

</CodeBlock>
<MarginText>23</MarginText>
<p><em>Example of a generic declaration declaring a generic package:</em> <br /></p>

<MarginText>24</MarginText>
<CodeBlock language="ada">
generic{"\n"}   type Item   is private;{"\n"}   type Vector is array (Positive range {"<"}{">"}) of Item;{"\n"}   with function Sum(X, Y : Item) return Item;{"\n"}package On_Vectors is{"\n"}   function Sum  (A, B : Vector) return Vector;{"\n"}   function Sigma(A    : Vector) return Item;{"\n"}   Length_Error : exception;{"\n"}end On_Vectors;{"\n"}

</CodeBlock>

#### Extensions to Ada 83

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>24.a</MarginText>
<Admonition type="aarm" aarm="note">
The syntax rule for <code><a href="../AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code> is modified to allow the reserved words <strong>tagged</strong> and <strong>abstract</strong>, to allow formal derived types, and to allow formal packages.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.b</MarginText>
<Admonition type="aarm" aarm="note">
<code><a href="../AA-8/AA-8.4#S0235">Use_clause</a></code>s are allowed in <code><a href="../AA-12/AA-12.1#S0313">generic_formal_part</a></code>s. This is necessary in order to allow a <code><a href="../AA-8/AA-8.4#S0235">use_clause</a></code> within a formal part to provide direct visibility of declarations within a generic formal package. <br /></Admonition>
</AnnotatedOnly>

#### Wording Changes from Ada 83

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>24.c/3</MarginText>
<MarginInfo items={["AI05-0299-1"]} />
<Admonition type="aarm" aarm="note">
The syntax for <code><a href="../AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code> and <code><a href="../AA-12/AA-12.5#S0323">formal_type_definition</a></code> is split up into more named categories. The rules for these categories are moved to the appropriate subclauses. The names of the categories are changed to be more intuitive and uniform. For example, we changed <code>generic_parameter_declaration</code> to <code><a href="../AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code>, because the thing it declares is a generic formal, not a generic. In the others, we abbreviate &ldquo;generic_formal&rdquo; to just &ldquo;formal&rdquo;. We can't do that for <code><a href="../AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code>, because of confusion with normal formal parameters of subprograms. <br /></Admonition>
</AnnotatedOnly>
