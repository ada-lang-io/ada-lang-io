<MarginText>1</MarginText>
<p>[ Formal subprograms can be used to pass callable entities to a generic unit.] <br /></p>


#### Language Design Principles

<AnnotatedOnly>
<MarginText>1.a</MarginText>
<Admonition type="aarm" aarm="note">
Generic formal subprograms are like renames of the <code><a href="/docs/arm/AA-12/AA-12.3#S0318">explicit_generic_actual_parameter</a></code>. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>4.1/2</MarginText>
<MarginInfo items={["AI95-00260-02", "AI95-00348-01"]} />
<p class="Indented2">A <code><a href="/docs/arm/AA-12/AA-12.6#S0338">subprogram_default</a></code> of <strong>null</strong> shall not be specified for a formal function or for a <code><a href="/docs/arm/AA-12/AA-12.6#S0337">formal_abstract_subprogram_declaration</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>4.a/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>There are no null functions because the return value has to be constructed somehow. We don't allow null for abstract formal procedures, as the operation is dispatching. It doesn't seem appropriate (or useful) to say that the implementation of something is null in the formal type and all possible descendants of that type. This also would define a dispatching operation that doesn't correspond to a slot in the tag of the controlling type, which would be a new concept. Finally, additional rules would be needed to define the meaning of a dispatching null procedure (for instance, the convention of such a subprogram should be intrinsic, but that's not what the language says). It doesn't seem worth the effort. <br />
</Admonition>
</AnnotatedOnly>


#### Name Resolution Rules

<MarginText>5</MarginText>
<p>The expected profile for the <code><a href="/docs/arm/AA-12/AA-12.6#S0339">default_name</a></code>, if any, is that of the formal subprogram. <br /></p>

<AnnotatedOnly>
<MarginText>5.a/3</MarginText>
<MarginInfo items={["AI05-0299-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>This rule, unlike others in this subclause, is observed at compile time of the <code><a href="/docs/arm/AA-12/AA-12.1#S0310">generic_declaration</a></code>.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b</MarginText>
<Admonition type="aarm" aarm="note">
The evaluation of the <code><a href="/docs/arm/AA-12/AA-12.6#S0339">default_name</a></code> takes place during the elaboration of each instantiation that uses the default, as defined in <a href="/docs/arm/AA-12/AA-12.3">12.3</a>, &ldquo;<a href="/docs/arm/AA-12/AA-12.3">Generic Instantiation</a>&rdquo;. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>6</MarginText>
<p>For a generic formal subprogram, the expected profile for the actual is that of the formal subprogram. <br /></p>


#### Legality Rules

<MarginText>7/3</MarginText>
<MarginInfo items={["AI05-0239-1"]} />
<p>The profiles of the formal and any named default shall be mode conformant. <br /></p>

<AnnotatedOnly>
<MarginText>7.a/3</MarginText>
<MarginInfo items={["AI05-0299-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>This rule, unlike others in this subclause, is checked at compile time of the <code><a href="/docs/arm/AA-12/AA-12.1#S0310">generic_declaration</a></code>. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>8/3</MarginText>
<MarginInfo items={["AI05-0239-1"]} />
<p>The profiles of the formal and actual shall be mode conformant. <br /></p>

<MarginText>8.1/2</MarginText>
<MarginInfo items={["AI95-00423-01"]} />
<p>For a parameter or result subtype of a <code><a href="/docs/arm/AA-12/AA-12.6#S0335">formal_subprogram_declaration</a></code> that has an explicit <code><a href="/docs/arm/AA-3/AA-3.10#S0083">null_exclusion</a></code>:<br /></p>

<MarginText>8.2/5</MarginText>
<MarginInfo items={["AI12-0183-1", "AI12-0287-1"]} />
<ul>
<li>if the actual matching the <code><a href="/docs/arm/AA-12/AA-12.6#S0335">formal_subprogram_declaration</a></code> statically denotes a generic formal subprogram of another generic unit <em>G</em>, and the instantiation containing the actual occurs within the body of a generic unit <em>G</em> or within the body of a generic unit declared within the declarative region of the generic unit <em>G</em>, then the corresponding parameter or result type of the formal subprogram of <em>G</em> shall have a <code><a href="/docs/arm/AA-3/AA-3.10#S0083">null_exclusion</a></code>;<br /></li>
<MarginText>8.3/2</MarginText>
<li>otherwise, the subtype of the corresponding parameter or result type of the actual matching the <code><a href="/docs/arm/AA-12/AA-12.6#S0335">formal_subprogram_declaration</a></code> shall exclude null. In addition to the places where Legality Rules normally apply (see <a href="/docs/arm/AA-12/AA-12.3">12.3</a>), this rule applies also in the private part of an instance of a generic unit. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>8.a/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This rule prevents &ldquo;lying&rdquo;. <strong>Null</strong> must never be the value of a parameter or result with an explicit <code><a href="/docs/arm/AA-3/AA-3.10#S0083">null_exclusion</a></code>. The first bullet is an assume-the-worst rule which prevents trouble in generic bodies (including bodies of child generics) when the formal subtype excludes null implicitly. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>8.4/5</MarginText>
<MarginInfo items={["AI12-0204-1"]} />
<p>If the named default, if any, is a prefixed view, the prefix of that view shall denote an object for which renaming is allowed (see <a href="/docs/arm/AA-8/AA-8.5#Subclause_8.5.1">8.5.1</a>). Similarly, if the actual subprogram in an instantiation is a prefixed view, the prefix of that view shall denote an object for which renaming is allowed.<br /></p>

<AnnotatedOnly>
<MarginText>8.b/5</MarginText>
<MarginInfo items={["AI12-0204-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>The prefix in such a case is essentially renamed at the point of the instantiation and passed to any calls of the formal subprogram in the generic. If the prefix isn't legal to rename, that doesn't make sense (and allowing it might end up passing a nonexistent object to some calls). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>8.5/5</MarginText>
<MarginInfo items={["AI95-00260-02", "AI05-0296-1"]} />
<p>If a formal parameter of a <code><a href="/docs/arm/AA-12/AA-12.6#S0337">formal{"_"}abstract{"_"}subprogram{"_"}declaration</a></code> is of a specific tagged type <em>T</em> or of an anonymous access type designating a specific tagged type <em>T</em>, <em>T</em> is called a <em>controlling type</em> of the <code><a href="/docs/arm/AA-12/AA-12.6#S0337">formal{"_"}abstract{"_"}subprogram{"_"}declaration</a></code>. Similarly, if the result of a <code><a href="/docs/arm/AA-12/AA-12.6#S0337">formal{"_"}abstract{"_"}subprogram{"_"}declaration</a></code> for a function is of a specific tagged type <em>T</em> or of an anonymous access type designating a specific tagged type <em>T</em>, <em>T</em> is called a controlling type of the <code><a href="/docs/arm/AA-12/AA-12.6#S0337">formal{"_"}abstract{"_"}subprogram{"_"}declaration</a></code>. A <code><a href="/docs/arm/AA-12/AA-12.6#S0337">formal{"_"}abstract{"_"}subprogram{"_"}declaration</a></code> shall have exactly one controlling type, and that type shall not be incomplete. <br /></p>

<AnnotatedOnly>
<MarginText>8.c/2</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The specific tagged type could be any of a formal tagged private type, a formal derived type, a formal interface type, or a normal tagged type. While the last case doesn't seem to be very useful, there isn't any good reason for disallowing it. This rule ensures that the operation is a dispatching operation of some type, and that we unambiguously know what that type is.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.d/2</MarginText>
<Admonition type="aarm" aarm="note">
We informally call a subprogram declared by a <code><a href="/docs/arm/AA-12/AA-12.6#S0337">formal{"_"}abstract{"_"}subprogram{"_"}declaration</a></code> an <em>abstract formal subprogram</em>, but we do not use this term in normative wording. (We do use it often in these notes.) <br />
</Admonition>
</AnnotatedOnly>

<MarginText>8.6/5</MarginText>
<MarginInfo items={["AI95-00260-02", "AI12-0165-1"]} />
<p>The actual subprogram for a <code><a href="/docs/arm/AA-12/AA-12.6#S0337">formal{"_"}abstract{"_"}subprogram{"_"}declaration</a></code> shall be:<br /></p>

<MarginText>8.7/5</MarginText>
<ul>
<li>a dispatching operation of the controlling type; or<br /></li>
<MarginText>8.8/5</MarginText>
<li>if the controlling type is a formal type, and the actual type corresponding to that formal type is a specific type <em>T</em>, a dispatching operation of type <em>T</em>; or<br /></li>
<MarginText>8.9/5</MarginText>
<li>if the controlling type is a formal type, and the actual type is a class-wide type <em>T</em>'Class, an implicitly declared subprogram corresponding to a primitive operation of type <em>T</em> (see Static Semantics below). <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>8.e/2</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>We mean the controlling type of the <code><a href="/docs/arm/AA-12/AA-12.6#S0337">formal{"_"}abstract{"_"}subprogram{"_"}declaration</a></code>, of course. Saying that gets unwieldy and redundant (so says at least one reviewer, anyway). <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.e.1/5</MarginText>
<MarginInfo items={["AI12-0165-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>This means that the actual is a primitive operation of the controlling type, an abstract formal subprogram, or the implicitly available primitive operation for a class-wide type.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.e.2/5</MarginText>
<MarginInfo items={["AI12-0005-1", "AI12-0165-1"]} />
<Admonition type="aarm" aarm="note">
An explicit class-wide operation cannot be used for the actual of a formal abstract subprogram. Such an operation is never primitive (only specific types have primitive subprograms).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.f/5</MarginText>
<MarginInfo items={["AI12-0165-1"]} />
<Admonition type="aarm" aarm="note">
This could happen in a case like: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.g/2</MarginText>
<CodeBlock language="ada">
generic{"\n"}   type T({"<"}{">"}) is tagged private;{"\n"}   with procedure Foo (Obj : in T) is abstract;{"\n"}package P ...{"\n"}
<MarginText>8.h/2</MarginText>
package New{"_"}P is new P (Something'Class, Some{"_"}Proc);{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>8.i/5</MarginText>
<MarginInfo items={["AI12-0165-1"]} />
<Admonition type="aarm" aarm="note">
If Some{"_"}Proc is an explicit class-wide operation, the instantiation here is illegal, because Some{"_"}Proc is not a primitive operation of Something'Class (there are no such operations).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.i.1/5</MarginText>
<MarginInfo items={["AI12-0165-1"]} />
<Admonition type="aarm" aarm="note">
However, if Some{"_"}Proc is a primitive operation of type Something, then the instantiation is legal; the actual is the implicitly generated subprogram described in Static Semantics below. This is not a problem, since the rules given in <a href="/docs/arm/AA-12/AA-12.5#Subclause_12.5.1">12.5.1</a> explain how this routine dispatches even though its parameter is class-wide.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.i.2/5</MarginText>
<MarginInfo items={["AI12-0165-1"]} />
<Admonition type="aarm" aarm="note">
We allow this special case because it is possible for a class-wide operation to be primitive inside of an instance, and the contract model does not allow us to make such cases illegal. As such, it seems inconsistent to not allow the same in explicit instantiations.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.j/5</MarginText>
<MarginInfo items={["AI12-0005-1", "AI12-0165-1"]} />
<Admonition type="aarm" aarm="note">
Specifically, since it is possible for a formal tagged type to be instantiated with a class-wide type, it is possible for the (real) controlling type to be class-wide as in the following unusual case:<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.k/5</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
<CodeBlock language="ada">
generic{"\n"}   type NT({"<"}{">"}) is new T with private;{"\n"}   -- Presume that T has the following primitive operation:{"\n"}   -- procedure Bar (Obj : in T);{"\n"}package Gr ...{"\n"}
<MarginText>8.l/2</MarginText>
package body Gr is{"\n"}   package New{"_"}P2 is new P (NT, Foo ={">"} Bar);{"\n"}end Gr;{"\n"}
<MarginText>8.m/2</MarginText>
package New{"_"}Gr is new Gr (Something'Class);{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>8.n/5</MarginText>
<MarginInfo items={["AI12-0165-1"]} />
<Admonition type="aarm" aarm="note">
The instantiation of New{"_"}P2 is legal, since Bar is a dispatching operation of the actual type of the controlling type of the abstract formal subprogram Foo. Again, the rules given in <a href="/docs/arm/AA-12/AA-12.5#Subclause_12.5.1">12.5.1</a> explain how this routine dispatches.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.o/2</MarginText>
<Admonition type="aarm" aarm="note">
Note that this legality rule never needs to be rechecked in an instance (that contains a nested instantiation). The rule only talks about the actual type of the instantiation; it does not require looking further; if the actual type is in fact a formal type, we do not intend looking at the actual for that formal. <br />
</Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>9</MarginText>
<p>A <code><a href="/docs/arm/AA-12/AA-12.6#S0335">formal_subprogram_declaration</a></code> declares a generic formal subprogram. The types of the formal parameters and result, if any, of the formal subprogram are those determined by the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code>s given in the <code><a href="/docs/arm/AA-12/AA-12.6#S0335">formal_subprogram_declaration</a></code>; however, independent of the particular subtypes that are denoted by the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code>s, the nominal subtypes of the formal parameters and result, if any, are defined to be nonstatic, and unconstrained if of an array type [(no applicable index constraint is provided in a call on a formal subprogram)]. In an instance, a <code><a href="/docs/arm/AA-12/AA-12.6#S0335">formal_subprogram_declaration</a></code> declares a view of the actual. The profile of this view takes its subtypes and calling convention from the original profile of the actual entity, while taking the formal parameter <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code>s and <code><a href="/docs/arm/AA-3/AA-3.7#S0063">default{"_"}expression</a></code>s from the profile given in the <code><a href="/docs/arm/AA-12/AA-12.6#S0335">formal{"_"}subprogram{"_"}declaration</a></code>. The view is a function or procedure, never an entry. <br /></p>

<AnnotatedOnly>
<MarginText>9.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This rule is intended to be the same as the one for renamings-as-declarations, where the <code><a href="/docs/arm/AA-12/AA-12.6#S0335">formal_subprogram_declaration</a></code> is analogous to a renaming-as-declaration, and the actual is analogous to the renamed view. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>9.1/3</MarginText>
<MarginInfo items={["AI05-0071-1", "AI05-0131-1"]} />
<p>If a <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code> in the profile of the <code><a href="/docs/arm/AA-12/AA-12.6#S0335">formal_subprogram_declaration</a></code> denotes a formal private or formal derived type and the actual type for this formal type is a class-wide type <em>T</em>'Class, then for the purposes of resolving the corresponding actual subprogram at the point of the instantiation, certain implicit declarations may be available as possible resolutions as follows:<br /></p>

<MarginText>9.2/3</MarginText>
<p class="Indented4">For each primitive subprogram of <em>T</em> that is directly visible at the point of the instantiation, and that has at least one controlling formal parameter, a corresponding implicitly declared subprogram with the same defining name, and having the same profile as the primitive subprogram except that <em>T</em> is systematically replaced by <em>T</em>'Class in the types of its profile, is potentially use-visible. The body of such a subprogram is as defined in <a href="/docs/arm/AA-12/AA-12.5#Subclause_12.5.1">12.5.1</a> for primitive subprograms of a formal type when the actual type is class-wide. <br /></p>

<AnnotatedOnly>
<MarginText>9.b/3</MarginText>
<MarginInfo items={["AI05-0071-1", "AI05-0131-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>This gives the same capabilities to formal subprograms as those that primitive operations of the formal type have when the actual type is class-wide. We do not want to discourage the use of explicit declarations for (formal) subprograms! <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.c/3</MarginText>
<MarginInfo items={["AI05-0071-1", "AI05-0131-1"]} />
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>Although the above wording seems to require constructing implicit versions of all of the primitive subprograms of type <em>T</em>, it should be clear that a compiler only needs to consider those that could possibly resolve to the corresponding actual subprogram. For instance, if the formal subprogram is a procedure with two parameters, and the actual subprogram name is Bar (either given explicitly or by default), the compiler need not consider primitives that are functions, that have the wrong number of parameters, that have defining names other than Bar, and so on; thus it does not need to construct implicit declarations for those primitives. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.d/3</MarginText>
<MarginInfo items={["AI05-0071-1", "AI05-0131-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>Functions that only have a controlling result and do not have a controlling parameter of <em>T</em> are not covered by this rule, as any call would be required to raise Program{"_"}Error by <a href="/docs/arm/AA-12/AA-12.5#Subclause_12.5.1">12.5.1</a>. It is better to detect the error earlier than at run time. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>10</MarginText>
<p>If a generic unit has a <code><a href="/docs/arm/AA-12/AA-12.6#S0338">subprogram_default</a></code> specified by a box, and the corresponding actual parameter is omitted, then it is equivalent to an explicit actual parameter that is a usage name identical to the defining name of the formal.<br /></p>

<MarginText>10.1/2</MarginText>
<MarginInfo items={["AI95-00348-01"]} />
<p>If a generic unit has a <code><a href="/docs/arm/AA-12/AA-12.6#S0338">subprogram_default</a></code> specified by the reserved word <strong>null</strong>, and the corresponding actual parameter is omitted, then it is equivalent to an explicit actual parameter that is a null procedure having the profile given in the <code><a href="/docs/arm/AA-12/AA-12.6#S0335">formal{"_"}subprogram{"_"}declaration</a></code>.<br /></p>

<MarginText>10.2/2</MarginText>
<MarginInfo items={["AI95-00260-02"]} />
<p>The subprogram declared by a <code><a href="/docs/arm/AA-12/AA-12.6#S0337">formal{"_"}abstract{"_"}subprogram{"_"}declaration</a></code> with a controlling type <em>T</em> is a dispatching operation of type <em>T</em>. <br /></p>

<AnnotatedOnly>
<MarginText>10.a/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This is necessary to trigger all of the dispatching operation rules. It otherwise would not be considered a dispatching operation, as formal subprograms are never primitive operations. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11/5</MarginText>
<MarginInfo items={["AI12-0442-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 1   The matching rules for formal subprograms state requirements that are similar to those applying to <code><a href="/docs/arm/AA-8/AA-8.5#S0242">subprogram_renaming_declaration</a></code>s (see <a href="/docs/arm/AA-8/AA-8.5#Subclause_8.5.4">8.5.4</a>). In particular, the name of a parameter of the formal subprogram can be different from that of the corresponding parameter of the actual subprogram; similarly, for these parameters, <code><a href="/docs/arm/AA-3/AA-3.7#S0063">default_expression</a></code>s can be different.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 2   The constraints that apply to a parameter of a formal subprogram are those of the corresponding formal parameter of the matching actual subprogram (not those implied by the corresponding <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code> in the <code>{"_"}specification</code> of the formal subprogram). A similar remark applies to the result of a function. Therefore, to avoid confusion, it is recommended that the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> of a first subtype be used in any declaration of a formal subprogram. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>13</MarginText>
<Admonition type="aarm" aarm="note">

</Admonition>

<MarginText>14</MarginText>
<Admonition type="aarm" aarm="note">

#### Incompatibilities With Ada 2005

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>24.d/3</MarginText>
<MarginInfo items={["AI05-0296-1"]} />
<Admonition type="aarm" aarm="note">
It is now illegal to declare a formal abstract subprogram whose controlling type is incomplete. It was never intended to allow that, and such a type would have to come from outside of the generic unit in Ada 2005, so it is unlikely to be useful. Moreover, a dispatching call on the subprogram is likely to fail in many implementations. So it is very unlikely that any code will need to be changed because of this new rule. <br />
</Admonition>
</AnnotatedOnly>

#### Incompatibilities With Ada 2012

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>24.g/5</MarginText>
<MarginInfo items={["AI12-0204-1"]} />
<Admonition type="aarm" aarm="correction">
<strong></strong> Added a rule to ensure that for an actual subprogram that is a prefixed view, the prefix continues to exist during the life of the instantiation. If the prefix is a subcomponent that depends on discriminants, Ada 2005 and 2012 would have allowed the prefix while Ada 2022 would not. Without this change, explicit forms (renaming the object and then using that in instantiations) would be safer than directly using the prefixed view; that's inconsistent with other kinds of actual subprograms. <br />
</Admonition>
