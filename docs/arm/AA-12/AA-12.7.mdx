<MarginText>1</MarginText>
<p>[ Formal packages can be used to pass packages to a generic unit. The <code><a href="../AA-12/AA-12.7#S0340">formal_package_declaration</a></code> declares that the formal package is an instance of a given generic package. Upon instantiation, the actual package has to be an instance of that generic package.] <br /></p>

#### Legality Rules

<MarginText>4</MarginText>
<p>The <em>generic{"_"}package{"_"}</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> shall denote a generic package (the <em>template</em> for the formal package); the formal package is an instance of the template.<br /></p>

<MarginText>4.1/3</MarginText>
<MarginInfo items={["AI05-0025-1"]} />
<p>The <em>generic{"_"}formal{"_"}parameter{"_"}</em><code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code> of a <code><a href="../AA-12/AA-12.7#S0342">formal_package_association</a></code> shall denote a <code><a href="../AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code> of the template. If two or more formal subprograms of the template have the same defining name, then named associations are not allowed for the corresponding actuals.<br /></p>

<MarginText>4.2/3</MarginText>
<MarginInfo items={["AI95-00398-01"]} />
<p>A <code><a href="../AA-12/AA-12.7#S0341">formal_package_actual_part</a></code> shall contain at most one <code><a href="../AA-12/AA-12.7#S0342">formal_package_association</a></code> for each formal parameter. If the <code><a href="../AA-12/AA-12.7#S0341">formal_package_actual_part</a></code> does not include &ldquo;<strong>others</strong> ={">"} {"<"}{">"}&rdquo;, each formal parameter without an association shall have a <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code> or <code><a href="../AA-12/AA-12.6#S0338">subprogram_default</a></code>.<br /></p>

<MarginText>4.3/3</MarginText>
<MarginInfo items={["AI05-0200-1"]} />
<p>The rules for matching between <code><a href="../AA-12/AA-12.7#S0342">formal_package_association</a></code>s and the generic formals of the template are as follows: <br /></p>

<MarginText>4.4/3</MarginText>
<ul>
<li>If all of the <code><a href="../AA-12/AA-12.7#S0342">formal_package_association</a></code>s are given by generic associations, the <code><a href="../AA-12/AA-12.3#S0318">explicit_generic_actual_parameter</a></code>s of the <code><a href="../AA-12/AA-12.7#S0342">formal_package_association</a></code>s shall be legal for an instantiation of the template.<br /></li>
<MarginText>4.5/5</MarginText>
<MarginInfo items={["AI05-0200-1"]} />
<li>If a <code><a href="../AA-12/AA-12.7#S0342">formal_package_association</a></code> for a formal type <em>T</em> of the template is given by {"<"}{">"}, then the <code><a href="../AA-12/AA-12.7#S0342">formal_package_association</a></code> for any other <code><a href="../AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code> of the template that mentions <em>T</em> directly or indirectly shall also be given by {"<"}{">"}.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>4.a/3</MarginText>
<MarginInfo items={["AI05-0200-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>The above rule is simple to state, though it does not reflect the fact that the formal package functions like an instantiation of a special kind, where each box association for a <code><a href="../AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code> <em>F</em> is replaced with a new entity <em>F</em>' that has the same characteristics as <em>F</em>: if <em>F</em> is a formal discrete type then <em>F</em>' is a discrete type, if <em>F</em> is a formal subprogram then <em>F</em>' is a subprogram with a similar signature, etc. In practice this is achieved by making the association into a copy of the declaration of the generic formal.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>5/2</MarginText>
<MarginInfo items={["AI95-00317-01"]} />
<p>The actual shall be an instance of the template. If the <code><a href="../AA-12/AA-12.7#S0341">formal_package_actual_part</a></code> is ({"<"}{">"}) or (<strong>others</strong> ={">"} {"<"}{">"}), [then the actual may be any instance of the template]; otherwise, certain of the actual parameters of the actual instance shall match the corresponding actual parameters of the formal package, determined as follows:<br /></p>

<MarginText>5.1/2</MarginText>
<MarginInfo items={["AI95-00317-01"]} />
<ul>
<li>If the <code><a href="../AA-12/AA-12.7#S0341">formal{"_"}package{"_"}actual{"_"}part</a></code> includes <code><a href="../AA-12/AA-12.3#S0317">generic_association</a></code>s as well as associations with {"<"}{">"}, then only the actual parameters specified explicitly with <code><a href="../AA-12/AA-12.3#S0317">generic_association</a></code>s are required to match;<br /></li>
<MarginText>5.2/2</MarginText>
<MarginInfo items={["AI95-00317-01"]} />
<li>Otherwise, all actual parameters shall match[, whether any actual parameter is given explicitly or by default]. <br /></li>
</ul>
<MarginText>5.3/2</MarginText>
<MarginInfo items={["AI95-00317-01"]} />
<p>The rules for matching of actual parameters between the actual instance and the formal package are as follows:<br /></p>

<MarginText>6/2</MarginText>
<MarginInfo items={["AI95-00317-01"]} />
<ul>
<li>For a formal object of mode <strong>in</strong>, the actuals match if they are static expressions with the same value, or if they statically denote the same constant, or if they are both the literal <strong>null</strong>. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We can't simply require full conformance between the two actual parameter expressions, because the two expressions are being evaluated at different times. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>7</MarginText>
<ul>
<li>For a formal subtype, the actuals match if they denote statically matching subtypes. <br /></li>
<MarginText>8</MarginText>
<li>For other kinds of formals, the actuals match if they statically denote the same entity. <br /></li>
</ul>
<MarginText>8.1/1</MarginText>
<MarginInfo items={["AI95-00213-01"]} />
<p>{"{"}<em>8652/0039</em>{"}"} For the purposes of matching, any actual parameter that is the name of a formal object of mode <strong>in</strong> is replaced by the formal object's actual expression (recursively). <br /></p>


#### Static Semantics

<MarginText>9</MarginText>
<p>A <code><a href="../AA-12/AA-12.7#S0340">formal_package_declaration</a></code> declares a generic formal package.<br /></p>

<MarginText>10/2</MarginText>
<MarginInfo items={["AI95-00317-01"]} />
<p>The visible part of a formal package includes the first list of <code><a href="../AA-3/AA-3.11#S0088">basic_declarative_item</a></code>s of the <code><a href="../AA-7/AA-7.1#S0230">package{"_"}specification</a></code>. In addition, for each actual parameter that is not required to match, a copy of the declaration of the corresponding formal parameter of the template is included in the visible part of the formal package. If the copied declaration is for a formal type, copies of the implicit declarations of the primitive subprograms of the formal type are also included in the visible part of the formal package.<br /></p>

<AnnotatedOnly>
<MarginText>10.a/2</MarginText>
<MarginInfo items={["AI95-00317-01"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>If the <code><a href="../AA-12/AA-12.7#S0341">formal_package_actual_part</a></code> is ({"<"}{">"}), then the declarations that occur immediately within the <code><a href="../AA-12/AA-12.1#S0313">generic_formal_part</a></code> of the template for the formal package are visible outside the formal package, and can be denoted by expanded names outside the formal package. If only some of the actual parameters are given by {"<"}{">"}, then the declaration corresponding to those parameters (but not the others) are made visible. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.b/3</MarginText>
<MarginInfo items={["AI05-0005-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>We always want either the actuals or the formals of an instance to be nameable from outside, but never both. If both were nameable, one would get some funny anomalies since they denote the same entity, but, in the case of types at least, they might have different and inconsistent sets of primitive operators due to predefined operator &ldquo;reemergence&rdquo;, Formal derived types exacerbate the difference. We want the implicit declarations of the <code><a href="../AA-12/AA-12.1#S0313">generic_formal_part</a></code> as well as the explicit declarations, so we get operations on the formal types. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.c</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>A generic formal package is a package, and is an instance. Hence, it is possible to pass a generic formal package as an actual to another generic formal package. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>11/2</MarginText>
<MarginInfo items={["AI95-00317-01"]} />
<p>For the purposes of matching, if the actual instance <em>A</em> is itself a formal package, then the actual parameters of <em>A</em> are those specified explicitly or implicitly in the <code><a href="../AA-12/AA-12.7#S0341">formal_package_actual_part</a></code> for <em>A</em>, plus, for those not specified, the copies of the formal parameters of the template included in the visible part of <em>A</em>.<br /></p>

#### Wording Changes from Ada 95

</AnnotatedOnly>
<AnnotatedOnly>

<MarginInfo items={["AI95-00213-01"]} />
<Admonition type="aarm" aarm="note">
{"{"}<em>8652/0039</em>{"}"} <strong>Corrigendum:</strong> Corrected the description of formal package matching to say that formal parameters are always replaced by their actual parameters (recursively). This matches the actual practice of compilers, as the ACATS has always required this behavior.<br />
</Admonition>

<AnnotatedOnly>

</Admonition>
</AnnotatedOnly>

#### Incompatibilities With Ada 2005

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>25.e/3</MarginText>
<MarginInfo items={["AI05-0025-1", "AI05-0200-1"]} />
<Admonition type="aarm" aarm="correction">
<strong></strong> Added missing rules for parameters of generic formal package that parallel those in <a href="../AA-12/AA-12.3">12.3</a>, as well as some specific to {"<"}{">"} parameters. These are technically incompatibilities because generic formal package parameters that Ada 95 and Ada 2005 would have considered legal now have to be rejected. But this should not be an issue in practice as such formal parameters could not have matched any actual generics. And it is quite likely that implementations already enforce some of these rules. <br />
</Admonition>
</AnnotatedOnly>
