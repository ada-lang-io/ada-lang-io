#### Dynamic Semantics

<MarginText>2/5_D.2.1</MarginText>
<MarginInfo items={["AI95-00321-01", "AI12-0454-1"]} />
<p>A task can become a <i>running task</i> only if it is ready (see Clause <a href="/docs/arm/AA-9/">9</a>) and the execution resources required by that task are available. Processors are allocated to tasks based on each task's active priority.<br /></p>

<MarginText>3_D.2.1</MarginText>
<p>It is implementation defined whether, on a multiprocessor, a task that is waiting for access to a protected object keeps its processor busy. <br /></p>

<AnnotatedOnly>
<MarginText>3.a_D.2.1</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<b></b>Whether, on a multiprocessor, a task that is waiting for access to a protected object keeps its processor busy.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>4/5_D.2.1</MarginText>
<MarginInfo items={["AI95-00321-01", "AI12-0119-1"]} />
<p><i>Task dispatching</i> is the process by which a logical thread of control associated with a ready task is selected for execution on a processor. This selection is done during the execution of such a logical thread of control, at certain points called <i>task dispatching points</i>. Such a logical thread of control reaches a task dispatching point whenever it becomes blocked, and when its associated task terminates. [Other task dispatching points are defined throughout this Annex for specific policies.] Below we talk in terms of tasks, but in the context of a parallel construct, a single task can be represented by multiple logical threads of control, each of which can appear separately on a ready queue. <br /></p>

<AnnotatedOnly>
<MarginText>4.a_D.2.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>On multiprocessor systems, more than one task can be chosen, at the same time, for execution on more than one processor, as explained below. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>5/2_D.2.1</MarginText>
<MarginInfo items={["AI95-00321-01"]} />
<p><i>Task dispatching policies</i> are specified in terms of conceptual <i>ready queues</i> and task states. A ready queue is an ordered list of ready tasks. The first position in a queue is called the <i>head of the queue</i>, and the last position is called the <i>tail of the queue</i>. A task is <i>ready</i> if it is in a ready queue, or if it is running. Each processor has one ready queue for each priority value. At any instant, each ready queue of a processor contains exactly the set of tasks of that priority that are ready for execution on that processor, but are not running on any processor; that is, those tasks that are ready, are not running on any processor, and can be executed using that processor and other available resources. A task can be on the ready queues of more than one processor. <br /></p>

<AnnotatedOnly>
<MarginText>5.a_D.2.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>The core language defines a ready task as one that is not blocked. Here we refine this definition and talk about ready queues. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>6/2_D.2.1</MarginText>
<MarginInfo items={["AI95-00321-01"]} />
<p>Each processor also has one <i>running task</i>, which is the task currently being executed by that processor. Whenever a task running on a processor reaches a task dispatching point it goes back to one or more ready queues; a task (possibly the same task) is then selected to run on that processor. The task selected is the one at the head of the highest priority nonempty ready queue; this task is then removed from all ready queues to which it belongs. <br /></p>

<AnnotatedOnly>
<MarginText>6.a_D.2.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>There is always at least one task to run, if we count the idle task. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>7/5_D.2.1</MarginText>
<MarginInfo items={["AI95-00321-01", "AI05-0166-1", "AI12-0241-1"]} />
<p>A call of Yield and a <code><a href="/docs/arm/AA-9/AA-9.6#S0266">delay_statement</a></code> are task dispatching points for all language-defined policies.<br /></p>

<AnnotatedOnly>
<MarginText>3.b/2_D.2.2</MarginText>
<MarginInfo items={["AI95-00355-01"]} />
<Admonition type="aarm" aarm="reason">
<b></b>Each ready queue is managed by exactly one policy. Anything else would be chaos. The ready queue is determined by the active priority. However, how the active priority is calculated is determined by the policy; in order to break out of this circle, we have to say that the active priority is calculated by the method determined by the policy of the base priority. <br />
</Admonition>

## D.2.3  Preemptive Dispatching

<MarginText>1/3_D.2.3</MarginText>
<MarginInfo items={["AI95-00321-01", "AI05-0299-1"]} />
<p>[This subclause defines a preemptive task dispatching policy.] <br /></p>


#### Static Semantics

<MarginText>2/2_D.2.3</MarginText>
<MarginInfo items={["AI95-00355-01"]} />
<p>The <i>policy{"_"}</i><code><a href="/docs/arm/AA-2/AA-2.3#S0002">identifier</a></code> FIFO{"_"}Within{"_"}Priorities is a task dispatching policy.<br /></p>


#### Dynamic Semantics

<MarginText>3/2_D.2.3</MarginText>
<MarginInfo items={["AI95-00321-01"]} />
<p>When FIFO{"_"}Within{"_"}Priorities is in effect, modifications to the ready queues occur only as follows:<br /></p>

<MarginText>4/2_D.2.3</MarginText>
<MarginInfo items={["AI95-00321-01"]} />
<ul>
<li>When a blocked task becomes ready, it is added at the tail of the ready queue for its active priority.<br /></li>
<MarginText>5/2_D.2.3</MarginText>
<li>When the active priority of a ready task that is not running changes, or the setting of its base priority takes effect, the task is removed from the ready queue for its old active priority and is added at the tail of the ready queue for its new active priority, except in the case where the active priority is lowered due to the loss of inherited priority, in which case the task is added at the head of the ready queue for its new active priority.<br /></li>
<MarginText>6/2_D.2.3</MarginText>
<li>When the setting of the base priority of a running task takes effect, the task is added to the tail of the ready queue for its active priority.<br /></li>
<MarginText>7/2_D.2.3</MarginText>
<li>When a task executes a <code><a href="/docs/arm/AA-9/AA-9.6#S0266">delay_statement</a></code> that does not result in blocking, it is added to the tail of the ready queue for its active priority. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>7.a/2_D.2.3</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>If the delay does result in blocking, the task moves to the &ldquo;delay queue&rdquo;, not to the ready queue. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>8/2_D.2.3</MarginText>
<MarginInfo items={["AI95-00321-01"]} />
<p>Each of the events specified above is a task dispatching point (see <a href="/docs/arm/AA-D/AA-D.2#Subclause_D.2.1">D.2.1</a>).<br /></p>

<MarginText>9/2_D.2.3</MarginText>
<MarginInfo items={["AI95-00321-01"]} />
<p>A task dispatching point occurs for the currently running task of a processor whenever there is a nonempty ready queue for that processor with a higher priority than the priority of the running task. The currently running task is said to be <i>preempted</i> and it is added at the head of the ready queue for its active priority.<br /></p>


#### Implementation Requirements

<MarginText>10/2_D.2.3</MarginText>
<MarginInfo items={["AI95-00333-01"]} />
<p>An implementation shall allow, for a single partition, both the task dispatching policy to be specified as FIFO{"_"}Within{"_"}Priorities and also the locking policy (see <a href="/docs/arm/AA-D/AA-D.3">D.3</a>) to be specified as Ceiling{"_"}Locking. <br /></p>

<AnnotatedOnly>
<MarginText>10.a/2_D.2.3</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>This is the preferred combination of the FIFO{"_"}Within{"_"}Priorities policy with a locking policy, and we want that combination to be portable. <br />
</Admonition>
</AnnotatedOnly>


#### Documentation Requirements

<MarginText>11/2_D.2.3</MarginText>
<MarginInfo items={["AI95-00321-01"]} />
<p><i>Priority inversion</i> is the duration for which a task remains at the head of the highest priority nonempty ready queue while the processor executes a lower priority task. The implementation shall document:<br /></p>

<MarginText>12/2_D.2.3</MarginText>
<ul>
<li>The maximum priority inversion a user task can experience due to activity of the implementation (on behalf of lower priority tasks), and <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>12.a/2_D.2.3</MarginText>
<Admonition type="aarm" aarm="note">
<b>Documentation Requirement: </b>The maximum priority inversion a user task can experience from the implementation.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>13/2_D.2.3</MarginText>
<ul>
<li>whether execution of a task can be preempted by the implementation processing of delay expirations for lower priority tasks, and if so, for how long. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>13.a/2_D.2.3</MarginText>
<Admonition type="aarm" aarm="note">
<b>Documentation Requirement: </b>The amount of time that a task can be preempted for processing on behalf of lower-priority tasks.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14/2_D.2.3</MarginText>
<MarginInfo items={["AI95-00321-01"]} />
<Admonition type="aarm" aarm="note">
NOTE 1   If the active priority of a running task is lowered due to loss of inherited priority (as it is on completion of a protected operation) and there is a ready task of the same active priority that is not running, the running task continues to run (provided that there is no higher priority task).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>15/2_D.2.3</MarginText>
<MarginInfo items={["AI95-00321-01"]} />
<Admonition type="aarm" aarm="note">
NOTE 2   Setting the base priority of a ready task causes the task to move to the tail of the queue for its active priority, regardless of whether the active priority of the task actually changes.<br />
</Admonition>
</AnnotatedOnly>

## D.2.4  Non-Preemptive Dispatching

<MarginText>1/3_D.2.4</MarginText>
<MarginInfo items={["AI95-00298-01", "AI05-0299-1"]} />
<p>[This subclause defines a non-preemptive task dispatching policy.] <br /></p>


#### Static Semantics

<MarginText>2/2_D.2.4</MarginText>
<MarginInfo items={["AI95-00298-01", "AI95-00355-01"]} />
<p>The <i>policy{"_"}</i><code><a href="/docs/arm/AA-2/AA-2.3#S0002">identifier</a></code> Non{"_"}Preemptive{"_"}FIFO{"_"}Within{"_"}Priorities is a task dispatching policy.<br /></p>

<MarginText>2.1/3_D.2.4</MarginText>
<MarginInfo items={["AI05-0166-1"]} />
<p>The following language-defined library package exists: <br /></p>

<MarginText>2.2/5_D.2.4</MarginText>
<MarginInfo items={["AI12-0241-1", "AI12-0302-1"]} />
<CodeBlock language="ada">
package Ada.Dispatching.Non{"_"}Preemptive{"\n"}  with Preelaborate, Nonblocking, Global ={">"} in out synchronized is{"\n"}  procedure Yield{"_"}To{"_"}Higher;{"\n"}  procedure Yield{"_"}To{"_"}Same{"_"}Or{"_"}Higher renames Yield;{"\n"}end Ada.Dispatching.Non{"_"}Preemptive;{"\n"}

</CodeBlock>
<MarginText>2.3/3_D.2.4</MarginText>
<MarginInfo items={["AI05-0166-1", "AI05-0264-1"]} />
<p>A call of Yield{"_"}To{"_"}Higher is a task dispatching point for this policy. If the task at the head of the highest priority ready queue has a higher active priority than the calling task, then the calling task is preempted.<br /></p>

<AnnotatedOnly>
<MarginText>2.a/3_D.2.4</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>For language-defined policies other than Non{"_"}Preemptive{"_"}FIFO{"_"}Within{"_"}Priorities, a higher priority task should never be on a ready queue while a lower priority task is executed. Thus, for such policies, Yield{"_"}To{"_"}Higher does nothing.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>2.b/3_D.2.4</MarginText>
<Admonition type="aarm" aarm="note">
Yield{"_"}To{"_"}Higher is <i>not</i> a potentially blocking operation; it can be used during a protected operation. That is allowed, as under the predefined Ceiling{"_"}Locking policy any task with a higher priority than the protected operation cannot call the operation (that would violate the locking policy). An implementation-defined locking policy may need to define the semantics of Yield{"_"}To{"_"}Higher differently. <br />
</Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>3/2_D.2.4</MarginText>
<MarginInfo items={["AI95-00355-01"]} />
<p>Non{"_"}Preemptive{"_"}FIFO{"_"}Within{"_"}Priorities shall not be specified as the <i>policy{"_"}</i><code><a href="/docs/arm/AA-2/AA-2.3#S0002">identifier</a></code> of <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code> Priority{"_"}Specific{"_"}Dispatching (see <a href="/docs/arm/AA-D/AA-D.2#Subclause_D.2.2">D.2.2</a>).<br /></p>

<AnnotatedOnly>
<MarginText>3.a/2_D.2.4</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>The non-preemptive nature of this policy could cause the policies of higher priority tasks to malfunction, missing deadlines and having unlimited priority inversion. That would render the use of such policies impotent and misleading. As such, this policy only makes sense for a complete system. <br />
</Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>4/2_D.2.4</MarginText>
<MarginInfo items={["AI95-00298-01"]} />
<p>When Non{"_"}Preemptive{"_"}FIFO{"_"}Within{"_"}Priorities is in effect, modifications to the ready queues occur only as follows:<br /></p>

<MarginText>5/2_D.2.4</MarginText>
<MarginInfo items={["AI95-00298-01"]} />
<ul>
<li>When a blocked task becomes ready, it is added at the tail of the ready queue for its active priority.<br /></li>
<MarginText>6/2_D.2.4</MarginText>
<li>When the active priority of a ready task that is not running changes, or the setting of its base priority takes effect, the task is removed from the ready queue for its old active priority and is added at the tail of the ready queue for its new active priority.<br /></li>
<MarginText>7/2_D.2.4</MarginText>
<li>When the setting of the base priority of a running task takes effect, the task is added to the tail of the ready queue for its active priority.<br /></li>
<MarginText>8/2_D.2.4</MarginText>
<li>When a task executes a <code><a href="/docs/arm/AA-9/AA-9.6#S0266">delay_statement</a></code> that does not result in blocking, it is added to the tail of the ready queue for its active priority. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>8.a/2_D.2.4</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>If the delay does result in blocking, the task moves to the &ldquo;delay queue&rdquo;, not to the ready queue. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>9/3_D.2.4</MarginText>
<MarginInfo items={["AI05-0166-1"]} />
<p>For this policy, blocking or termination of a task, a <code><a href="/docs/arm/AA-9/AA-9.6#S0266">delay_statement</a></code>, a call to Yield{"_"}To{"_"}Higher, and a call to Yield{"_"}To{"_"}Same{"_"}Or{"_"}Higher or Yield are the only task dispatching points (see <a href="/docs/arm/AA-D/AA-D.2#Subclause_D.2.1">D.2.1</a>). <br /></p>

<AnnotatedOnly>
<MarginText>9.a/3_D.2.4</MarginText>
<MarginInfo items={["AI05-0166-1"]} />
<Admonition type="aarm" aarm="ramification">
<b></b>A <code><a href="/docs/arm/AA-9/AA-9.6#S0266">delay_statement</a></code> is always a task dispatching point even if it is not blocking. Similarly, a call to Yield{"_"}To{"_"}Higher is never blocking, but it is a task dispatching point In each of these cases, they can cause the current task to stop running (it is still ready). Otherwise, the running task continues to run until it is blocked.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.b/5_D.2.4</MarginText>
<MarginInfo items={["AI12-0299-1"]} />
<Admonition type="aarm" aarm="note">
This rule supersedes the Implementation Permission of <a href="/docs/arm/AA-D/AA-D.2#Subclause_D.2.1">D.2.1</a>; an implementation that adds additional task dispatching points to this policy is incorrect. <br />
</Admonition>
</AnnotatedOnly>


#### Implementation Requirements

<MarginText>10/2_D.2.4</MarginText>
<MarginInfo items={["AI95-00333-01"]} />
<p>An implementation shall allow, for a single partition, both the task dispatching policy to be specified as Non{"_"}Preemptive{"_"}FIFO{"_"}Within{"_"}Priorities and also the locking policy (see <a href="/docs/arm/AA-D/AA-D.3">D.3</a>) to be specified as Ceiling{"_"}Locking. <br /></p>

<AnnotatedOnly>
<MarginText>10.a/2_D.2.4</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>This is the preferred combination of the Non{"_"}Preemptive{"_"}FIFO{"_"}Within{"_"}Priorities policy with a locking policy, and we want that combination to be portable. <br />
</Admonition>
</AnnotatedOnly>


#### Implementation Permissions

<MarginText>11/3_D.2.4</MarginText>
<MarginInfo items={["AI95-00298-01", "AI05-0229-1", "AI05-0269-1"]} />
<p>Since implementations are allowed to round all ceiling priorities in subrange System.Priority to System.Priority'Last (see <a href="/docs/arm/AA-D/AA-D.3">D.3</a>), an implementation may allow a task of a partition using the Non{"_"}Premptive{"_"}FIFO{"_"}Within{"_"}Priorities policy to execute within a protected object without raising its active priority provided the associated protected unit does not contain any subprograms with aspects Interrupt{"_"}Handler or Attach{"_"}Handler specified, nor does the unit have aspect Interrupt{"_"}Priority specified. When the locking policy (see <a href="/docs/arm/AA-D/AA-D.3">D.3</a>) is Ceiling{"_"}Locking, an implementation taking advantage of this permission shall ensure that a call to Yield{"_"}to{"_"}Higher that occurs within a protected action uses the ceiling priority of the protected object (rather than the active priority of the task) when determining whether to preempt the task. <br /></p>

<AnnotatedOnly>
<MarginText>11.a.1/3_D.2.4</MarginText>
<MarginInfo items={["AI05-0269-1"]} />
<Admonition type="aarm" aarm="reason">
<b></b>We explicitly require that the ceiling priority be used in calls to Yield{"_"}to{"_"}Higher in order to prevent a risk of priority inversion and consequent loss of mutual exclusion when Yield{"_"}to{"_"}Higher is used in a protected object. This requirement might lessen the value of the permission (as the current Ceiling{"_"}Priority will have to be maintained in the TCB), but loss of mutual exclusion cannot be tolerated. The primary benefit of the permission (eliminating the need for preemption at the end of a protected action) is still available. As noted above, an implementation-defined locking policy will need to specify the semantics of Yield{"_"}to{"_"}Higher, including this case. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>10/2_D.2.5</MarginText>
<MarginInfo items={["AI95-00355-01"]} />
<p>For Round{"_"}Robin{"_"}Within{"_"}Priorities, the dispatching rules for FIFO{"_"}Within{"_"}Priorities apply with the following additional rules:<br /></p>

<MarginText>11/2_D.2.5</MarginText>
<ul>
<li>When a task is added or moved to the tail of the ready queue for its base priority, it has an execution time budget equal to the quantum for that priority level. This will also occur when a blocked task becomes executable again.<br /></li>
<MarginText>12/2_D.2.5</MarginText>
<li>When a task is preempted (by a higher priority task) and is added to the head of the ready queue for its priority level, it retains its remaining budget.<br /></li>
<MarginText>13/2_D.2.5</MarginText>
<li>While a task is executing, its budget is decreased by the amount of execution time it uses. The accuracy of this accounting is the same as that for execution time clocks (see <a href="/docs/arm/AA-D/AA-D.14">D.14</a>). <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>13.a/2_D.2.5</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Note that this happens even when the task is executing at a higher, inherited priority, and even if that higher priority is dispatched by a different policy than round robin. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>14/2_D.2.5</MarginText>
<ul>
<li>When a task has exhausted its budget and is without an inherited priority (and is not executing within a protected operation), it is moved to the tail of the ready queue for its priority level. This is a task dispatching point.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>14.a/2_D.2.5</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>In this case, it will be given a budget as described in the first bullet.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.b/2_D.2.5</MarginText>
<Admonition type="aarm" aarm="note">
The rules for FIFO{"_"}Within{"_"}Priority (to which these bullets are added) say that a task that has its base priority set to a Round Robin priority is moved to the tail of the ready queue for its new priority level, and then will be given a budget as described in the first bullet. That happens whether or not the task's original base priority was a Round Robin priority. <br />
</Admonition>
</AnnotatedOnly>
