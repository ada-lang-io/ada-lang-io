<MarginText>1</MarginText>
<p>[A <code><a href="../AA-6/AA-6.1#S0207">parameter_specification</a></code> declares a formal parameter of mode <strong>in</strong>, <strong>in out</strong>, or <strong>out</strong>.] <br /></p>


#### Static Semantics

<MarginText>2</MarginText>
<p>A parameter is passed either <em>by copy</em> or <em>by reference</em>. [When a parameter is passed by copy, the formal parameter denotes a separate object from the actual parameter, and any information transfer between the two occurs only before and after executing the subprogram. When a parameter is passed by reference, the formal parameter denotes (a view of) the object denoted by the actual parameter; reads and updates of the formal parameter directly reference the actual parameter object.]<br /></p>

<MarginText>3/3</MarginText>
<MarginInfo items={["AI05-0142-4", "AI05-0262-1"]} />
<p>A type is a <em>by-copy type</em> if it is an elementary type, or if it is a descendant of a private type whose full type is a by-copy type. A parameter of a by-copy type is passed by copy, unless the formal parameter is explicitly aliased.<br /></p>

<MarginText>4</MarginText>
<p>A type is a <em>by-reference type</em> if it is a descendant of one of the following: <br /></p>

<MarginText>5</MarginText>
<ul>
<li>a tagged type;<br /></li>
<MarginText>6</MarginText>
<li>a task or protected type;<br /></li>
<MarginText>7/3</MarginText>
<MarginInfo items={["AI05-0096-1"]} />
<li>an explicitly limited record type; <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>7.a/3</MarginText>
<MarginInfo items={["AI05-0096-1"]} />
<Admonition type="aarm" aarm="note">
<em>This paragraph was deleted.</em><br /></Admonition>
</AnnotatedOnly>

<MarginText>8</MarginText>
<ul>
<li>a composite type with a subcomponent of a by-reference type;<br /></li>
<MarginText>9</MarginText>
<li>a private type whose full type is a by-reference type. <br /></li>
</ul>
<MarginText>10/5</MarginText>
<MarginInfo items={["AI05-0142-4", "AI05-0188-1", "AI12-0027-1", "AI12-0236-1", "AI12-0317-1"]} />
<p>A parameter of a by-reference type is passed by reference, as is an explicitly aliased parameter of any type. Each value of a by-reference type has an associated object.  For a value conversion, the associated object is the anonymous result object if such an object is created (see <a href="../AA-4/AA-4.6">4.6</a>); otherwise it is the associated object of the operand. In other cases, the object associated with the evaluated operative constituent of the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> or <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> (see <a href="../AA-4/AA-4.4">4.4</a>) determines its associated object. <br /></p>

<AnnotatedOnly>
<MarginText>10.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>By-reference parameter passing makes sense only if there is an object to reference; hence, we define such an object for each case.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.b</MarginText>
<Admonition type="aarm" aarm="note">
Since tagged types are by-reference types, this implies that every value of a tagged type has an associated object. This simplifies things, because we can define the tag to be a property of the object, and not of the value of the object, which makes it clearer that object tags never change.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.b.1/5</MarginText>
<MarginInfo items={["AI12-0317-1"]} />
<Admonition type="aarm" aarm="note">
A construct like parenthesized expression or <code><a href="../AA-4/AA-4.7#S0163">qualified_expression</a></code> is ignored for the purposes of determining the associated object; for a <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>, it is relevant only in that it determines which <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> defines the associated object.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.c</MarginText>
<Admonition type="aarm" aarm="note">
We considered simplifying things even more by making every value (and therefore every expression) have an associated object. After all, there is little semantic difference between a constant object and a value. However, this would cause problems for untagged types. In particular, we would have to do a constraint check on every read of a type conversion (or a renaming thereof) in certain cases.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.d/2</MarginText>
<MarginInfo items={["AI95-00318-02"]} />
<Admonition type="aarm" aarm="note">
We do not want this definition to depend on the view of the type; privateness is essentially ignored for this definition. Otherwise, things would be confusing (does the rule apply at the call site, at the site of the declaration of the subprogram, at the site of the return statement?), and requiring different calls to use different mechanisms would be an implementation burden.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.e</MarginText>
<Admonition type="aarm" aarm="note">
<a href="../AA-C/AA-C.6">C.6</a>, &ldquo;<a href="../AA-C/AA-C.6">Shared Variable Control</a>&rdquo; says that a composite type with an atomic or volatile subcomponent is a by-reference type, among other things.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.f</MarginText>
<Admonition type="aarm" aarm="note">
Every value of a limited by-reference type is the value of one and only one limited object. The <em>associated object</em> of a value of a limited by-reference type is the object whose value it represents. Two values of a limited by-reference type are the <em>same</em> if and only if they represent the value of the same object.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.g</MarginText>
<Admonition type="aarm" aarm="note">
We say &ldquo;by-reference&rdquo; above because these statements are not always true for limited private types whose underlying type is nonlimited (unfortunately). <br /></Admonition>
</AnnotatedOnly>

<MarginText>11/3</MarginText>
<MarginInfo items={["AI05-0240-1"]} />
<p>For other parameters, it is unspecified whether the parameter is passed by copy or by reference. <br /></p>

<AnnotatedOnly>
<MarginText>11.a/3</MarginText>
<MarginInfo items={["AI05-0005-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>There is no need to incorporate the discussion of AI83-00178, which requires pass-by-copy for certain kinds of actual parameters, while allowing pass-by-reference for others. This is because we explicitly indicate that a function creates an anonymous constant object for its result (see <a href="../AA-6/AA-6.5">6.5</a>). We also provide a special dispensation for instances of Unchecked_Conversion to return by reference (see <a href="../AA-13/AA-13.9">13.9</a>). <br /></Admonition>
</AnnotatedOnly>


#### Bounded (Run-Time) Errors

<MarginText>12/3</MarginText>
<MarginInfo items={["AI05-0240-1"]} />
<p>If one <code><a href="../AA-4/AA-4.1#S0091">name</a></code> denotes a part of a formal parameter, and a second <code><a href="../AA-4/AA-4.1#S0091">name</a></code> denotes a part of a distinct formal parameter or an object that is not part of a formal parameter, then the two <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s are considered <em>distinct access paths</em>. If an object is of a type for which the parameter passing mechanism is not specified and is not an explicitly aliased parameter, then it is a bounded error to assign to the object via one access path, and then read the value of the object via a distinct access path, unless the first access path denotes a part of a formal parameter that no longer exists at the point of the second access [(due to leaving the corresponding callable construct).] The possible consequences are that Program_Error is raised, or the newly assigned value is read, or some old value of the object is read. <br /></p>

<AnnotatedOnly>
<MarginText>12.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>For example, if we call &ldquo;P(X ={">"} Global_Variable, Y ={">"} Global_Variable)&rdquo;, then within P, the names &ldquo;X&rdquo;, &ldquo;Y&rdquo;, and &ldquo;Global_Variable&rdquo; are all distinct access paths. If Global_Variable's type is neither pass-by-copy nor pass-by-reference, then it is a bounded error to assign to Global_Variable and then read X or Y, since the language does not specify whether the old or the new value would be read. On the other hand, if Global_Variable's type is pass-by-copy, then the old value would always be read, and there is no error. Similarly, if Global_Variable's type is defined by the language to be pass-by-reference, then the new value would always be read, and again there is no error. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.b</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We are saying <em>assign</em> here, not <em>update</em>, because updating any subcomponent is considered to update the enclosing object.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.c</MarginText>
<Admonition type="aarm" aarm="note">
The &ldquo;still exists&rdquo; part is so that a read after the subprogram returns is OK.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.d</MarginText>
<Admonition type="aarm" aarm="note">
If the parameter is of a by-copy type, then there is no issue here &mdash; the formal is not a view of the actual. If the parameter is of a by-reference type, then the programmer may depend on updates through one access path being visible through some other access path, just as if the parameter were of an access type. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.e</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>The implementation can keep a copy in a register of a parameter whose parameter-passing mechanism is not specified. If a different access path is used to update the object (creating a bounded error situation), then the implementation can still use the value of the register, even though the in-memory version of the object has been changed. However, to keep the error properly bounded, if the implementation chooses to read the in-memory version, it has to be consistent -- it cannot then assume that something it has proven about the register is true of the memory location. For example, suppose the formal parameter is L, the value of L(6) is now in a register, and L(6) is used in an <code><a href="../AA-4/AA-4.1#S0096">indexed_component</a></code> as in &ldquo;A(L(6)) := 99;&rdquo;, where A has bounds 1..3. If the implementation can prove that the value for L(6) in the register is in the range 1..3, then it need not perform the constraint check if it uses the register value. However, if the memory value of L(6) has been changed to 4, and the implementation uses that memory value, then it had better not alter memory outside of A.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.f</MarginText>
<Admonition type="aarm" aarm="note">
Note that the rule allows the implementation to pass a parameter by reference and then keep just part of it in a register, or, equivalently, to pass part of the parameter by reference and another part by copy. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.g</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We do not want to go so far as to say that the mere presence of aliasing is wrong. We wish to be able to write the following sorts of things in standard Ada: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.h</MarginText>
<CodeBlock language="ada">
procedure Move ( Source  : in  String;{"\n"}                 Target  : out String;{"\n"}                 Drop    : in  Truncation := Error;{"\n"}                 Justify : in  Alignment  := Left;{"\n"}                 Pad     : in  Character  := Space);{"\n"}-- Copies elements from Source to Target (safely if they overlap){"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>12.i</MarginText>
<Admonition type="aarm" aarm="note">
This is from the standard string handling package. It would be embarrassing if this couldn't be written in Ada!<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.j</MarginText>
<Admonition type="aarm" aarm="note">
The &ldquo;then&rdquo; before &ldquo;read&rdquo; in the rule implies that the implementation can move a read to an earlier place in the code, but not to a later place after a potentially aliased assignment. Thus, if the subprogram reads one of its parameters into a local variable, and then updates another potentially aliased one, the local copy is safe &mdash; it is known to have the old value. For example, the above-mentioned Move subprogram can be implemented by copying Source into a local variable before assigning into Target.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.k</MarginText>
<Admonition type="aarm" aarm="note">
For an <code><a href="../AA-5/AA-5.2#S0173">assignment_statement</a></code> assigning one array parameter to another, the implementation has to check which direction to copy at run time, in general, in case the actual parameters are overlapping slices. For example: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.l</MarginText>
<CodeBlock language="ada">
procedure Copy(X : in out String; Y: String) is{"\n"}begin{"\n"}    X := Y;{"\n"}end Copy;{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>12.m</MarginText>
<Admonition type="aarm" aarm="note">
It would be wrong for the compiler to assume that X and Y do not overlap (unless, of course, it can prove otherwise). <br /></Admonition>
</AnnotatedOnly>
