#### Name Resolution Rules

<p>A <em>formal parameter</em> is an object [directly visible within a <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code>] that represents the actual parameter passed to the subprogram in a call; it is declared by a <code><a href="../AA-6/AA-6.1#S0207">parameter_specification</a></code>. For a formal parameter, the expected type for its <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code>, if any, is that of the formal parameter. </p>

#### Legality Rules

<p>{"{"}<em>{"AI05-0143-1"}</em>{"}"} The <em>parameter mode</em> of a formal parameter conveys the direction of information transfer with the actual parameter: <strong>in</strong>, <strong>in out</strong>, or <strong>out</strong>. Mode <strong>in</strong> is the default, and is the mode of a parameter defined by an <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>. </p>
<Admonition type="aarm" aarm="note" title="Note: ">
<p><em>This paragraph was deleted.</em>{"{"}<em>{"AI05-0143-1"}</em>{"}"} </p></Admonition>

<p>A <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code> is only allowed in a <code><a href="../AA-6/AA-6.1#S0207">parameter_specification</a></code> for a formal parameter of mode <strong>in</strong>.</p>
<p>{"{"}<em>{"AI95-00348-01"}</em>{"}"} {"{"}<em>{"AI05-0177-1"}</em>{"}"} {"{"}<em>{"AI05-0229-1"}</em>{"}"} A <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code> or a <code><a href="../AA-12/AA-12.1#S0311">generic_subprogram_declaration</a></code> requires a completion [unless the Import aspect (see B.1) is True for the declaration; the completion shall be a body or a <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code> (see 8.5)]. [A completion is not allowed for an <code><a href="../AA-3/AA-3.9#S0076">abstract_subprogram_declaration</a></code> (see 3.9.3), a <code><a href="../AA-6/AA-6.7#S0227">null_procedure_declaration</a></code> (see 6.7), or an <code><a href="../AA-6/AA-6.8#S0228">expression_function_declaration</a></code> (see 6.8).] </p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>{"{"}<em>{"AI95-00348-01"}</em>{"}"} {"{"}<em>{"AI05-0177-1"}</em>{"}"} Abstract subprograms , null procedures, and expression functions are not declared by <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code>s, and so do not require completion (although the latter two can <em>be</em> completions). Protected subprograms are declared by <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code>s, and so require completion. Note that an abstract subprogram is a subprogram, a null procedure is a subprogram, an expression function is a subprogram, and a protected subprogram is a subprogram, but a generic subprogram is not a subprogram. </p></Admonition>

<p><strong>Proof: </strong>{"{"}<em>{"AI05-0229-1"}</em>{"}"} When the Import aspect is True for any entity, no completion is allowed (see B.1). </p></Admonition>

<p>A <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes a formal parameter is not allowed within the <code><a href="../AA-6/AA-6.1#S0206">formal_part</a></code> in which it is declared, nor within the <code><a href="../AA-6/AA-6.1#S0206">formal_part</a></code> of a corresponding body or <code><a href="../AA-9/AA-9.5#S0258">accept_statement</a></code>. </p>

<p><strong></strong>By contrast, <code><a href="../AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code>s are visible to subsequent declarations in the same <code><a href="../AA-12/AA-12.1#S0313">generic_formal_part</a></code>. </p></Admonition>


#### Static Semantics

<p>The <em>profile</em> of (a view of) a callable entity is either a <code><a href="../AA-6/AA-6.1#S0204">parameter_profile</a></code> or <code><a href="../AA-6/AA-6.1#S0205">parameter_and_result_profile</a></code>[; it embodies information about the interface to that entity - for example, the profile includes information about parameters passed to the callable entity. All callable entities have a profile - enumeration literals, other subprograms, and entries. An access-to-subprogram type has a designated profile.] Associated with a profile is a calling convention. A <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code> declares a procedure or a function, as indicated by the initial reserved word, with name and profile as given by its specification.</p>
<p>{"{"}<em>{"AI95-00231-01"}</em>{"}"} {"{"}<em>{"AI95-00318-02"}</em>{"}"} The nominal subtype of a formal parameter is the subtype determined by the optional <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code> and the <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>, or defined by the <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>, in the <code><a href="../AA-6/AA-6.1#S0207">parameter_specification</a></code>. The nominal subtype of a function result is the subtype determined by the optional <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code> and the <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>, or defined by the <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>, in the <code><a href="../AA-6/AA-6.1#S0205">parameter_and_result_profile</a></code>. </p>
<p>{"{"}<em>{"AI05-0142-4"}</em>{"}"} An <em>explicitly aliased parameter</em> is a formal parameter whose <code><a href="../AA-6/AA-6.1#S0207">parameter_specification</a></code> includes the reserved word <strong>aliased</strong>.</p>
<p>{"{"}<em>{"AI95-00231-01"}</em>{"}"} {"{"}<em>{"AI95-00254-01"}</em>{"}"} {"{"}<em>{"AI95-00318-02"}</em>{"}"} An <em>access parameter</em> is a formal <strong>in</strong> parameter specified by an <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>. An <em>access result type</em> is a function result type specified by an <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>. An access parameter or result type is of an anonymous access type (see 3.10). [Access parameters of an access-to-object type allow dispatching calls to be controlled by access values. Access parameters of an access-to-subprogram type permit calls to subprograms passed as parameters irrespective of their accessibility level.]</p>
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<p><strong></strong>{"{"}<em>{"AI95-00318-02"}</em>{"}"} Access result types have normal accessibility and thus don't have any special properties worth noting here. </p></Admonition>

<p>The <em>subtypes of a profile</em> are: </p>
<p>For any non-access parameters, the nominal subtype of the parameter.</p>
<p>{"{"}<em>{"AI95-00254-01"}</em>{"}"} For any access parameters of an access-to-object type, the designated subtype of the parameter type.</p>
<p>{"{"}<em>{"AI95-00254-01"}</em>{"}"} {"{"}<em>{"AI05-0164-1"}</em>{"}"} For any access parameters of an access-to-subprogram type, the subtypes of the designated profile of the parameter type.</p>
<p>{"{"}<em>{"AI95-00231-01"}</em>{"}"} {"{"}<em>{"AI95-00318-02"}</em>{"}"} For any non-access result, the nominal subtype of the function result.</p>
<p>{"{"}<em>{"AI95-00318-02"}</em>{"}"} For any access result type of an access-to-object type, the designated subtype of the result type.</p>
<p>{"{"}<em>{"AI95-00318-02"}</em>{"}"} {"{"}<em>{"AI05-0164-1"}</em>{"}"} For any access result type of an access-to-subprogram type, the subtypes of the designated profile of the result type.</p>
<p>[ The <em>types of a profile</em> are the types of those subtypes.]</p>
<p>{"{"}<em>{"AI95-00348-01"}</em>{"}"} {"{"}<em>{"AI05-0177-1"}</em>{"}"} [A subprogram declared by an <code><a href="../AA-3/AA-3.9#S0076">abstract_subprogram_declaration</a></code> is abstract; a subprogram declared by a <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code> is not. See 3.9.3, "Abstract Types and Subprograms". Similarly, a procedure declared by a <code><a href="../AA-6/AA-6.7#S0227">null_procedure_declaration</a></code> is a null procedure; a procedure declared by a <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code> is not. See 6.7, "Null Procedures". Finally, a function declared by an <code><a href="../AA-6/AA-6.8#S0228">expression_function_declaration</a></code> is an expression function; a function declared by a <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code> is not. See 6.8, "Expression Functions".]</p>
<p>{"{"}<em>{"AI95-00218-03"}</em>{"}"} [An <code><a href="../AA-8/AA-8.3#S0234">overriding_indicator</a></code> is used to indicate whether overriding is intended. See 8.3.1, "Overriding Indicators".] </p>

#### Dynamic Semantics

<p>{"{"}<em>{"AI95-00348-01"}</em>{"}"} The elaboration of a <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code> has no effect. </p>
<Admonition type="aarm" aarm="note" title="Note: ">
<p>NOTE 1   A <code><a href="../AA-6/AA-6.1#S0207">parameter_specification</a></code> with several identifiers is equivalent to a sequence of single <code><a href="../AA-6/AA-6.1#S0207">parameter_specification</a></code>s, as explained in 3.3.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>NOTE 2   Abstract subprograms do not have bodies, and cannot be used in a nondispatching call (see 3.9.3, "Abstract Types and Subprograms").</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>NOTE 3   The evaluation of <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code>s is caused by certain calls, as described in 6.4.1. They are not evaluated during the elaboration of the subprogram declaration.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>NOTE 4   Subprograms can be called recursively and can be called concurrently from multiple tasks. </p></Admonition>

## 6.1.1  Preconditions and Postconditions

<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0247-1"}</em>{"}"} {"{"}<em>{"AI12-0045-1"}</em>{"}"} {"{"}<em>{"AI12-0220-1"}</em>{"}"} {"{"}<em>{"AI12-0272-1"}</em>{"}"} {"{"}<em>{"AI12-0396-1"}</em>{"}"} For a noninstance subprogram [(including a generic formal subprogram)], a generic subprogram, an entry, or an access-to-subprogram type, the following language-defined assertion aspects may be specified with an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> (see 13.1.1):</p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>{"{"}<em>{"AI12-0045-1"}</em>{"}"} "Noninstance subprogram" excludes a subprogram that is an instance of a generic subprogram. In that case, the aspects should be specified on the generic subprogram. If preconditions or postconditions need to be added to an instance of a generic subprogram, it can be accomplished by creating a separate subprogram specification and then completing that specification with a renames-as-body of the instance. </p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p><strong>Proof: </strong>{"{"}<em>{"AI12-0272-1"}</em>{"}"} A generic formal subprogram is a subprogram, and there are no rules to prevent using these attributes on it. </p></Admonition>

<p>{"{"}<em>{"AI12-0220-1"}</em>{"}"} PreThis aspect specifies a specific precondition for a callable entity or an access-to-subprogram type; it shall be specified by an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, called a <em>specific precondition expression</em>. If not specified for an entity, the specific precondition expression for the entity is the enumeration literal True.</p>
<Admonition type="aarm" aarm="note" title="Note: ">
<p><strong>To be honest: </strong>In this and the following rules, we are talking about the enumeration literal True declared in package Standard (see A.1), and not some other value or identifier True. That matters as some rules depend on full conformance of these expressions, which depends on the specific declarations involved. </p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p><strong>Aspect Description for </strong><strong>Pre: </strong>Precondition; a condition that is expected to hold true before a call.</p></Admonition>

<p>{"{"}<em>{"AI05-0254-1"}</em>{"}"} {"{"}<em>{"AI05-0262-1"}</em>{"}"} {"{"}<em>{"AI12-0418-1"}</em>{"}"} Pre'ClassThis aspect specifies a class-wide precondition for a dispatching operation of a tagged type and its descendants; it shall be specified by an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, called a <em>class-wide precondition expression</em>. If not specified for an entity, then if no other class-wide precondition applies to the entity, the class-wide precondition expression for the entity is the enumeration literal True.</p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>{"{"}<em>{"AI05-0254-1"}</em>{"}"} If other class-wide preconditions apply to the entity and no class-wide precondition is specified, no class-wide precondition is defined for the entity; of course, the class-wide preconditions (of ancestors) that apply are still going to be checked. We need subprograms that don't have ancestors and don't specify a class-wide precondition to have a class-wide precondition of True, so that adding such a precondition to a descendant has no effect (necessary as a dispatching call through the root routine would not check any precondition).</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0220-1"}</em>{"}"} Pre'Class cannot be specified on an access-to-subprogram type because of a Legality Rule found in 13.1.1 that limits 'Class aspects to tagged types and primitive subprograms of tagged types. The same is true for Post'Class (below). </p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p><strong>Aspect Description for </strong><strong>Pre'Class: </strong>Precondition that applies to corresponding subprograms of descendant types.</p></Admonition>

<p>{"{"}<em>{"AI12-0220-1"}</em>{"}"} PostThis aspect specifies a specific postcondition for a callable entity or an access-to-subprogram type; it shall be specified by an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, called a <em>specific postcondition expression</em>. If not specified for an entity, the specific postcondition expression for the entity is the enumeration literal True.</p>
<Admonition type="aarm" aarm="note" title="Note: ">
<p><strong>Aspect Description for </strong><strong>Post: </strong>Postcondition; a condition that will hold true after a call.</p></Admonition>

<p>{"{"}<em>{"AI05-0262-1"}</em>{"}"} {"{"}<em>{"AI12-0418-1"}</em>{"}"} Post'ClassThis aspect specifies a class-wide postcondition for a dispatching operation of a tagged type and its descendants; it shall be specified by an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, called a <em>class-wide postcondition expression</em>. If not specified for an entity, the class-wide postcondition expression for the entity is the enumeration literal True. </p>
<Admonition type="aarm" aarm="note" title="Note: ">
<p><strong>Aspect Description for </strong><strong>Post'Class: </strong>Postcondition that applies to corresponding subprograms of descendant types.</p></Admonition>

<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<p><strong></strong>{"{"}<em>{"AI12-0005-1"}</em>{"}"} In the AARM notes below, we use the terms "inherited" and "inheritance" informally with respect to class-wide pre/post-conditions, to mean that the aspect applies to corresponding subprograms in descendant types. </p></Admonition>

#### Name Resolution Rules

<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} The expected type for a precondition or postcondition expression is any boolean type.</p>
<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0262-1"}</em>{"}"} {"{"}<em>{"AI12-0113-1"}</em>{"}"} {"{"}<em>{"AI12-0159-1"}</em>{"}"} {"{"}<em>{"AI12-0170-1"}</em>{"}"} {"{"}<em>{"AI12-0418-1"}</em>{"}"} Within the expression for a Pre'Class or Post'Class aspect for a primitive subprogram <em>S</em> of a tagged type <em>T</em>, a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes a formal parameter (or <em>S</em>'Result) of type <em>T</em> is interpreted as though it had a (notional) nonabstract  type <em>NT</em> that is a formal derived type whose ancestor type is <em>T</em>, with directly visible primitive operations. Similarly, a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes a formal access parameter (or <em>S</em>'Result for an access result) of type access-to-<em>T</em> is interpreted as having type access-to-<em>NT</em>. [The result of this interpretation is that the only operations that can be applied to such <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s are those defined for such a formal derived type.]</p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>{"{"}<em>{"AI12-0159-1"}</em>{"}"} This ensures that the expression is well-defined for any primitive subprogram of a type descended from <em>T</em>. </p></Admonition>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>{"{"}<em>{"AI12-0170-1"}</em>{"}"} The operations of <em>NT</em> are also nonabstract, so the rule against a call of an abstract subprogram does not trigger for a class-wide precondition or postcondition. </p></Admonition>

<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0264-1"}</em>{"}"} {"{"}<em>{"AI12-0418-1"}</em>{"}"} For an <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code> with <code><a href="../AA-4/AA-4.1#S0101">attribute_designator</a></code> Old, if the attribute reference has an expected type (or class of types) or shall resolve to a given type, the same applies to the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code>; otherwise, the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> shall be resolved independently of context.</p>

#### Legality Rules

<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0230-1"}</em>{"}"} The Pre or Post aspect shall not be specified for an abstract subprogram or a null procedure. [Only the Pre'Class and Post'Class aspects may be specified for such a subprogram.]</p>
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<p><strong></strong>{"{"}<em>{"AI05-0183-1"}</em>{"}"} Pre'Class and Post'Class can only be specified on primitive routines of tagged types, by a blanket rule found in 13.1.1. </p></Admonition>

<p>{"{"}<em>{"AI05-0247-1"}</em>{"}"} {"{"}<em>{"AI05-0254-1"}</em>{"}"} If a type <em>T</em> has an implicitly declared subprogram <em>P</em> inherited from a parent type <em>T1</em> and a homograph (see 8.3) of <em>P</em> from a progenitor type <em>T2</em>, and</p>
<p>the corresponding primitive subprogram <em>P1</em> of type <em>T1</em> is neither null nor abstract; and</p>
<p>the class-wide precondition expression True does not apply to <em>P1</em> (implicitly or explicitly); and</p>
<p>there is a class-wide precondition expression that applies to the corresponding primitive subprogram <em>P2</em> of <em>T2</em> that does not fully conform to any class-wide precondition expression that applies to <em>P1</em>, </p>
<p>{"{"}<em>{"AI05-0247-1"}</em>{"}"} {"{"}<em>{"AI05-0254-1"}</em>{"}"} then:</p>
<p>If the type <em>T</em> is abstract, the implicitly declared subprogram <em>P</em> is <em>abstract</em>.</p>
<p>Otherwise, the subprogram <em>P</em> <em>requires overriding</em> and shall be overridden with a nonabstract subprogram.</p>
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<p><strong></strong>We use the term "requires overriding" here so that this rule is taken into account when calculating visibility in 8.3; otherwise we would have a mess when this routine is overridden. </p></Admonition>

<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>Such an inherited subprogram would necessarily violate the Liskov Substitutability Principle (LSP) if called via a dispatching call from an ancestor other than the one that provides the called body. In such a case, the class-wide precondition of the actual body is stronger than the class-wide precondition of the ancestor. If we did not enforce that precondition for the body, the body could be called when the precondition it knows about is False - such "counterfeiting" of preconditions has to be avoided. But enforcing the precondition violates LSP. We do not want the language to be implicitly creating bodies that violate LSP; the programmer can still write an explicit body that calls the appropriate parent subprogram. In that case, the violation of LSP is explicitly in the code and obvious to code reviewers (both human and automated).</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>We have to say that the subprogram is abstract for an abstract type in this case, so that the next concrete type has to override it for the reasons above. Otherwise, inserting an extra level of abstract types would eliminate the requirement to override (as there is only one declared operation for the concrete type), and that would be bad for the reasons given above. </p></Admonition>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>This requires the set of class-wide preconditions that apply to the interface routine to be strictly stronger than those that apply to the concrete routine. Since full conformance requires each name to denote the same declaration, it is unlikely that independently declared preconditions would conform. This rule does allow "diamond inheritance" of preconditions, and of course no preconditions at all match.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>We considered adopting a rule that would allow examples where the expressions would conform after all inheritance has been applied, but this is complex and is not likely to be common in practice. Since the penalty here is just that an explicit overriding is required, the complexity is too much. </p></Admonition>

<p>{"{"}<em>{"AI05-0247-1"}</em>{"}"} If a renaming of a subprogram or entry <em>S1</em> overrides an inherited subprogram <em>S2</em>, then the overriding is illegal unless each class-wide precondition expression that applies to <em>S1</em> fully conforms to some class-wide precondition expression that applies to <em>S2</em> and each class-wide precondition expression that applies to <em>S2</em> fully conforms to some class-wide precondition expression that applies to <em>S1</em>.</p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>Such an overriding subprogram would violate LSP, as the precondition of <em>S1</em> would usually be different (and thus stronger) than the one known to a dispatching call through an ancestor routine of <em>S2</em>. This is always OK if the preconditions match, so we always allow that. </p></Admonition>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>This only applies to primitives of tagged types; other routines cannot have class-wide preconditions. </p></Admonition>

<p>{"{"}<em>{"AI12-0131-1"}</em>{"}"} Pre'Class shall not be specified for an overriding primitive subprogram of a tagged type <em>T</em> unless the Pre'Class aspect is specified for the corresponding primitive subprogram of some ancestor of <em>T</em>.</p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>Any such Pre'Class will have no effect, as it will be <strong>or</strong>ed with True. As such, it is highly misleading for readers, especially for those who are determining the assumptions that can be made in the body of the primitive subprogram. Note that in this case there is nothing explicit that might indicate that the class-wide precondition is ineffective. This rule does not prevent explicitly writing an ineffective class-wide precondition (for instance, if the parent subprogram has explicitly specified a precondition of True). </p></Admonition>

<p>{"{"}<em>{"AI12-0131-1"}</em>{"}"} In addition to the places where Legality Rules normally apply (see 12.3), these rules also apply in the private part of an instance of a generic unit. </p>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>A Pre'Class defaults to True only if no class-wide preconditions are inherited for the subprogram. The same is true for Post'Class. </p></Admonition>

<Admonition type="aarm" aarm="reason" title="Reason: ">

<p>{"{"}<em>{"AI12-0113-1"}</em>{"}"} References to formal parameters of <em>S</em> (or to <em>S</em> itself) are replaced with references to the corresponding formal parameters of the corresponding inherited or overriding subprogram <em>S</em> (or to the corresponding subprogram <em>S</em> itself).</p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>We have to define the corresponding expression this way as overriding routines are only required to be subtype conformant; in particular, the parameter names can be different. So we have to talk about corresponding parameters without mentioning any names. </p></Admonition>

<p>{"{"}<em>{"AI12-0113-1"}</em>{"}"} {"{"}<em>{"AI12-0412-1"}</em>{"}"} If the primitive subprogram <em>S</em> is not abstract, but the given descendant of <em>T</em> is abstract, then a nondispatching call on <em>S</em> is illegal if any Pre'Class or Post'Class aspect that applies to <em>S</em> is other than a static boolean expression. <Admonition type="aarm" aarm="note" title="Note: ">
<p><em>This paragraph was deleted.</em></p></Admonition>

<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>{"{"}<em>{"AI12-0412-1"}</em>{"}"} The above rules mean that a concrete primitive of an abstract type is effectively treated as abstract, if any nontrivial Pre'Class or Post'Class aspects apply to it. This makes sense because we are using a notional formal derived type model for such aspects, and an abstract type is not permitted as an actual type for such a formal type. If we didn't do this, the evaluation of the precondition or postcondition of a concrete subprogram of an abstract type could possibly call abstract functions.. </p></Admonition>

<p><strong></strong>{"{"}<em>{"AI12-0412-1"}</em>{"}"} As this Reference Manual was frozen, a significant incompatibility has come to light with the above rule. It is likely that the above rule will be adjusted; check with ARG work at <a href="http://www.ada_auth.org/arg.html">www.ada_auth.org/arg.html</a> to find the adjusted rules. </p></Admonition>

<p>{"{"}<em>{"AI12-0280-2"}</em>{"}"} For a <code><a href="../AA-4/AA-4.5#S0151">case_expression</a></code> that is not repeatedly evaluated, a subexpression of any <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is conditionally evaluated, and its determining expressions include the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the <code><a href="../AA-4/AA-4.5#S0151">case_expression</a></code>;</p>
<p>{"{"}<em>{"AI12-0280-2"}</em>{"}"} For a short-circuit control form that is not repeatedly evaluated, a subexpression of the right-hand operand is conditionally evaluated, and its determining expressions include the left-hand operand of the short-circuit control form;</p>
<p>{"{"}<em>{"AI12-0280-2"}</em>{"}"} For a membership test that is not repeatedly evaluated, a subexpression of a <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code> other than the first is conditionally evaluated, and its determining expressions include the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> and the preceding <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code>s of the membership test. </p>
<p>{"{"}<em>{"AI12-0280-2"}</em>{"}"} A conditionally evaluated subexpression is <em>determined to be unevaluated</em> at run time if its set of determining expressions are all known on entry, and when evaluated on entry their values are such that the given subexpression is not evaluated. </p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>To be precise, a conditionally evaluated expression is determined to be unevaluated (including all of its subexpressions) under the following circumstances: </p></Admonition>

<p>Within an <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code>, a <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> with an associated <code><a href="../AA-4/AA-4.5#S0150">condition</a></code> that evaluates to False, or a <code><a href="../AA-4/AA-4.5#S0150">condition</a></code> or <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> where a condition of a preceding part of the <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code> evaluates to True;</p>
<p>Within a <code><a href="../AA-4/AA-4.5#S0151">case_expression</a></code>, a <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> with an <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code> that is not covered by the value of the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>;</p>
<p>The right-hand operand of a short-circuit control form where the left-hand operand evaluates to False for <strong>and then</strong> or True for <strong>or else</strong>;</p>
<p>A <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code> of a membership test where the individual membership test defined by any prior <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code> evaluates to True. </p>
<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} For a <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> X that denotes an object of a nonlimited type, the following attribute is defined: </p>
<p>X'Old{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0262-1"}</em>{"}"} {"{"}<em>{"AI05-0273-1"}</em>{"}"} {"{"}<em>{"AI12-0032-1"}</em>{"}"} {"{"}<em>{"AI12-0280-2"}</em>{"}"} Each X'Old in a postcondition expression that is enabled, other than those that occur in subexpressions that are determined to be unevaluated, denotes a constant that is implicitly declared at the beginning of the subprogram body, entry body, or accept statement.</p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>{"{"}<em>{"AI12-0280-2"}</em>{"}"} If X'Old occurs in a subexpression that is determined to be unevaluated, then there is no associated constant, and no evaluation of the prefix takes place. In general, this will require evaluating one or more known-on-entry subexpressions before creating and initializing any X'Old constants. Note that any 'Old in a known-on-entry subexpression evaluated this way represents the current value of the prefix (the 'Old itself can be ignored).</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0193-1"}</em>{"}"} In the case of an accept statement, the constant is declared inside of the rendezvous. It is considered part of the initialization of the postcondition check, which is part of the rendezvous by definition (see 9.5.2). </p></Admonition>

<p>{"{"}<em>{"AI12-0032-1"}</em>{"}"} {"{"}<em>{"AI12-0159-1"}</em>{"}"} The implicitly declared entity denoted by each occurrence of X'Old is declared as follows:</p>
<p>If X is of an anonymous access type defined by an <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code> <em>A</em> then </p>
<CodeBlock>
X'Old : constant A := X;{"\n"}
</CodeBlock>

<p>If X is of a specific tagged type <em>T</em> then </p>
<CodeBlock>
anonymous : constant T'Class := T'Class(X);{"\n"}X'Old : T renames T(anonymous);{"\n"}
</CodeBlock>

<p>where the name X'Old denotes the object renaming. </p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>This means that the underlying tag associated with X'Old is that of X and not that of the nominal type of X. </p></Admonition>

<p>Otherwise </p>
<CodeBlock>
X'Old : constant S := X;{"\n"}
</CodeBlock>

<p>where <em>S</em> is the nominal subtype of X. This includes the case where the type of <em>S</em> is an anonymous array type or a universal type. </p>
<p>{"{"}<em>{"AI12-0032-1"}</em>{"}"} {"{"}<em>{"AI12-0185-1"}</em>{"}"} {"{"}<em>{"AI12-0388-1"}</em>{"}"} The type and nominal subtype of X'Old are as implied by the above definitions.</p>
<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0262-1"}</em>{"}"} {"{"}<em>{"AI05-0273-1"}</em>{"}"} {"{"}<em>{"AI12-0217-1"}</em>{"}"} {"{"}<em>{"AI12-0280-2"}</em>{"}"} Reference to this attribute is only allowed within a postcondition expression. The <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of an Old <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code> shall not contain a Result <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code>, nor an Old <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code>, nor a use of an entity declared within the postcondition expression but not within <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> itself (for example, the loop parameter of an enclosing <code><a href="../AA-4/AA-4.5#S0153">quantified_expression</a></code>). The <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of an Old <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code> shall statically name (see 4.9) an entity, unless the <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code> is unconditionally evaluated, or is conditionally evaluated where all of the determining expressions are known on entry.</p>
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<p><strong></strong>The <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> X can be any nonlimited object that obeys the syntax for prefix other than the few exceptions given above (discussed below). Useful cases are: the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> of a formal parameter of mode [<strong>in</strong>] <strong>out</strong>, the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> of a global variable updated by the subprogram, a function call passing those as parameters, a subcomponent of those things, etc.</p></Admonition>

<p>A qualified expression can be used to make an arbitrary expression into a valid prefix, so T'(X + Y)'Old is legal, even though (X + Y)'Old is not. Of course, in this case "+"(X, Y)'Old is also legal, but the qualified expression is arguably more readable.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>Note that F(X)'Old and F(X'Old) are not necessarily equal. The former calls F(X) and saves that value for later use during the postcondition. The latter saves the value of X, and during the postcondition, passes that saved value to F. In most cases, the former is what one wants (but it is not always legal, see below).</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0032-1"}</em>{"}"} If X has controlled parts, adjustment and finalization are implied by the implicit constant declaration. Similarly, the implicit constant declaration defines the accessibility level of X'Old.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>If postconditions are disabled, we want the compiler to avoid any overhead associated with saving 'Old values.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>'Old makes no sense for limited types, because its implementation involves copying. It might make semantic sense to allow build-in-place, but it's not worth the trouble. </p></Admonition>

<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>{"{"}<em>{"AI05-0273-1"}</em>{"}"} {"{"}<em>{"AI12-0280-2"}</em>{"}"} {"{"}<em>{"AI12-0005-1"}</em>{"}"} Since the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of an Old <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code> is evaluated when the subprogram is called (if it is evaluated at all), we cannot allow it to include values that do not exist at that time (like 'Result and loop parameters of <code><a href="../AA-4/AA-4.5#S0153">quantified_expression</a></code>s). We also do not allow the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> itself to include 'Old references, as those would be redundant (because the evaluation of the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code>, if it occurs, already happens on entry to the subprogram ), and allowing them would require some sort of order to the implicit constant declarations (because in A(I'Old)'Old, we surely would want the value of I'Old evaluated before the A(I'Old) is evaluated).</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI05-0273-1"}</em>{"}"} {"{"}<em>{"AI12-0280-2"}</em>{"}"} An additional rule applies when it cannot be determined on entry to the subprogram whether the Old <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code> will or will not be evaluated when the overall postcondition expression is evaluated. In such cases, we require that the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of the Old <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code> to statically name some object. This is necessary because the Old <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code>es have to be evaluated when the subprogram is called if there is any possibility that they might be needed; the compiler cannot in general know whether they will be needed in the postcondition expression. To see the problem, consider:</p></Admonition>

<CodeBlock>
Table : array (1..10) of Integer := ...{"\n"}procedure Bar (I : in out Natural){"\n"}   with Post ={">"} I {">"} 0 and then Table(I)'Old = 1; -- Illegal{"\n"}
</CodeBlock>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0005-1"}</em>{"}"} In this example, the compiler cannot know on entry what will be the value of I when the subprogram returns (since the subprogram execution can change it), and thus it does not know whether Table(I)'Old will be needed then. Thus it has to always create an implicit constant and evaluate Table(I) when Bar is called (because not having the value when it is needed is not acceptable). But if I = 0 when the subprogram is called, that evaluation will raise Constraint_Error, and that will happen even if I is unchanged by the subprogram and the value of Table(I)'Old is not ultimately needed. It's easy to see how a similar problem could occur for a dereference of an access type. This would be mystifying (since the point of the short circuit is to eliminate this possibility, but it cannot do so). Therefore, we require the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of any Old <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code> in such a context to statically name an object, which eliminates anything that could change during execution.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>It is easy to work around most errors that occur because of this rule. Just move the 'Old to the outer object, before any indexing, dereferences, or components. (That does not work for function calls, however, nor does it work for array indexing if the index can change during the execution of the subprogram.) </p></Admonition>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>{"{"}<em>{"AI12-0032-1"}</em>{"}"} An accept statement for a task entry with enabled postconditions such as </p></Admonition>

<CodeBlock>
accept E do{"\n"}   statements{"\n"}exception{"\n"}   handlers{"\n"}end;{"\n"}
</CodeBlock>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>behaves (at runtime) as follows: </p></Admonition>

<CodeBlock>
accept E do{"\n"}   declare{"\n"}      declarations, if any, of 'Old constants{"\n"}   begin{"\n"}      begin{"\n"}         statements{"\n"}      exception{"\n"}         handlers{"\n"}      end;{"\n"}      postcondition checks{"\n"}   end;{"\n"}end;{"\n"}
</CodeBlock>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0032-1"}</em>{"}"} Preconditions are checked by the caller before the rendezvous begins. Postcondition expressions might, of course, reference 'Old constants.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0032-1"}</em>{"}"} In the case of a protected operation with enabled postconditions, 'Old constant declarations (if any) are elaborated after the start of the protected action. Postcondition checks (which might reference these constants) are performed before the end of the protected action as described below.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0280-2"}</em>{"}"} The <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of an Old <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code> has to statically name an entity if it appears within a repeatedly evaluated expression. </p></Admonition>

<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0220-1"}</em>{"}"} For a <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> F that denotes a function declaration or an access-to-function type, the following attribute is defined: </p>
<p>F'Result{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0262-1"}</em>{"}"} {"{"}<em>{"AI12-0185-1"}</em>{"}"} {"{"}<em>{"AI12-0220-1"}</em>{"}"} {"{"}<em>{"AI12-0388-1"}</em>{"}"} Within a postcondition expression for F, denotes the return object of the function call for which the postcondition expression is evaluated. <p>{"{"}<em>{"AI12-0143-1"}</em>{"}"} For a <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> E that denotes an entry declaration of an entry family (see 9.5.2), the following attribute is defined: </p>
<p>E'Index{"{"}<em>{"AI12-0143-1"}</em>{"}"} Within a precondition or postcondition expression for entry family E, denotes the value of the entry index for the call of E. The nominal subtype of this attribute is the entry index subtype.</p>

#### Dynamic Semantics

<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0247-1"}</em>{"}"} {"{"}<em>{"AI05-0290-1"}</em>{"}"} Upon a call of the subprogram or entry, after evaluating any actual parameters, precondition checks are performed as follows:</p>
<p>The specific precondition check begins with the evaluation of the specific precondition expression that applies to the subprogram or entry, if it is enabled; if the expression evaluates to False, Assertions.Assertion_Error is raised; if the expression is not enabled, the check succeeds.</p>
<p>The class-wide precondition check begins with the evaluation of any enabled class-wide precondition expressions that apply to the subprogram or entry. If and only if all the class-wide precondition expressions evaluate to False, Assertions.Assertion_Error is raised.</p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>The class-wide precondition expressions of the entity itself as well as those of any parent or progenitor operations are evaluated, as these expressions apply to the corresponding operations of all descendants.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>Class-wide precondition checks are performed for all appropriate calls, but only enabled precondition expressions are evaluated. Thus, the check would be trivial if no precondition expressions are enabled. </p></Admonition>

<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0247-1"}</em>{"}"} {"{"}<em>{"AI05-0254-1"}</em>{"}"} {"{"}<em>{"AI05-0269-1"}</em>{"}"} {"{"}<em>{"AI12-0166-1"}</em>{"}"} The precondition checks are performed in an arbitrary order, and if any of the class-wide precondition expressions evaluate to True, it is not specified whether the other class-wide precondition expressions are evaluated. The precondition checks and any check for elaboration of the subprogram body are performed in an arbitrary order. In a call on a protected operation, the checks are performed before starting the protected action. For an entry call, the checks are performed prior to checking whether the entry is open.</p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>We need to explicitly allow short-circuiting of the evaluation of the class-wide precondition check if any expression fails, as it consists of multiple expressions; we don't need a similar permission for the specific precondition check as it consists only of a single expression. Nothing is evaluated for the call after a check fails, as the failed check propagates an exception. </p></Admonition>

<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0247-1"}</em>{"}"} {"{"}<em>{"AI05-0254-1"}</em>{"}"} {"{"}<em>{"AI05-0262-1"}</em>{"}"} {"{"}<em>{"AI05-0290-1"}</em>{"}"} Upon successful return from a call of the subprogram or entry, prior to copying back any by-copy <strong>in out</strong> or <strong>out</strong> parameters, the postcondition check is performed. This consists of the evaluation of any enabled specific and class-wide postcondition expressions that apply to the subprogram or entry. If any of the postcondition expressions evaluate to False, then Assertions.Assertion_Error is raised. The postcondition expressions are evaluated in an arbitrary order, and if any postcondition expression evaluates to False, it is not specified whether any other postcondition expressions are evaluated. The postcondition check, and any constraint or predicate checks associated with <strong>in out</strong> or <strong>out</strong> parameters are performed in an arbitrary order.</p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>The class-wide postcondition expressions of the entity itself as well as those of any parent or progenitor operations are evaluated, as these apply to all descendants; in contrast, only the specific postcondition of the entity applies. Postconditions can always be evaluated inside the invoked body. </p></Admonition>

<p>{"{"}<em>{"AI12-0032-1"}</em>{"}"} For a call to a task entry, the postcondition check is performed before the end of the rendezvous; for a call to a protected operation, the postcondition check is performed before the end of the protected action of the call. The postcondition check for any call is performed before the finalization of any implicitly-declared constants associated (as described above) with Old <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code>s but after the finalization of any other entities whose accessibility level is that of the execution of the callable construct. </p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>{"{"}<em>{"AI12-0032-1"}</em>{"}"} If a postcondition references the implicitly-declared constant associated with an Old attribute, the postcondition must be evaluated before the constant is finalized. One way to think of this is to imagine declaring a controlled object between any implicit "'Old" constant declarations and any explicit declarations, then performing postcondition checks during the finalization of this object. </p></Admonition>

<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0262-1"}</em>{"}"} If a precondition or postcondition check fails, the exception is raised at the point of the call[; the exception cannot be handled inside the called subprogram or entry]. Similarly, any exception raised by the evaluation of a precondition or postcondition expression is raised at the point of call.</p>
<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0247-1"}</em>{"}"} {"{"}<em>{"AI05-0254-1"}</em>{"}"} {"{"}<em>{"AI05-0262-1"}</em>{"}"} {"{"}<em>{"AI12-0113-1"}</em>{"}"} {"{"}<em>{"AI12-0159-1"}</em>{"}"} For any call to a subprogram or entry <em>S</em> (including dispatching calls), the checks that are performed to verify specific precondition expressions and specific and class-wide postcondition expressions are determined by those for the subprogram or entry actually invoked. Note that the class-wide postcondition expressions verified by the postcondition check that is part of a call on a primitive subprogram of type <em>T</em> includes all class-wide postcondition expressions originating in any progenitor of <em>T</em>[, even if the primitive subprogram called is inherited from a type <em>T1</em> and some of the postcondition expressions do not apply to the corresponding primitive subprogram of <em>T1</em>]. Any operations within a class-wide postcondition expression that were resolved as primitive operations of the (notional) formal derived type <em>NT</em>, are in the evaluation of the postcondition bound to the corresponding operations of the type identified by the controlling tag of the call on <em>S</em>.[ This applies to both dispatching and non-dispatching calls on <em>S</em>.]</p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>This applies to access-to-subprogram calls, dispatching calls, and to statically bound calls. We need this rule to cover statically bound calls as well, as specific pre- and postconditions are not inherited, but the subprogram might be.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>For concrete subprograms, we require the original specific postcondition to be evaluated as well as the inherited class-wide postconditions in order that the semantics of an explicitly defined wrapper that does nothing but call the original subprogram is the same as that of an inherited subprogram.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>Note that this rule does not apply to class-wide preconditions; they have their own rules mentioned below. </p></Admonition>

<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0247-1"}</em>{"}"} {"{"}<em>{"AI05-0254-1"}</em>{"}"} {"{"}<em>{"AI12-0113-1"}</em>{"}"} {"{"}<em>{"AI12-0159-1"}</em>{"}"} The class-wide precondition check for a call to a subprogram or entry <em>S</em> consists solely of checking the class-wide precondition expressions that apply to the denoted callable entity (not necessarily to the one that is invoked). Any operations within such an expression that were resolved as primitive operations of the (notional) formal derived type <em>NT</em> are in the evaluation of the precondition bound to the corresponding operations of the type identified by the controlling tag of the call on <em>S</em>.[ This applies to both dispatching and non-dispatching calls on <em>S</em>.]</p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>For a dispatching call, we are talking about the Pre'Class(es) that apply to the subprogram that the dispatching call is resolving to, not the Pre'Class(es) for the subprogram that is ultimately dispatched to. The class-wide precondition of the resolved call is necessarily the same or stronger than that of the invoked call. For a statically bound call, these are the same; for an access-to-subprogram, (which has no class-wide preconditions of its own), we check the class-wide preconditions of the invoked routine.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0233-1"}</em>{"}"} Since this check is based on the "callable entity", it does not depend on the view of the entity. This matters any time the ancestor type (if any) of the partial view differs from the parent type of the full view. In such a case, the view of the callable entity associated with the full view might inherit a Pre'Class while the view of the same callable entity associated with the partial view does not. </p></Admonition>

<Admonition type="aarm" aarm="implementation-note" title="Implementation Note: ">
<p><strong></strong>These rules imply that logically, class-wide preconditions of routines must be checked at the point of call (other than for access-to-subprogram calls, which must be checked in the body, probably using a wrapper). Specific preconditions that might be called with a dispatching call or via an access-to-subprogram value must be checked inside of the subprogram body. In contrast, the postcondition checks always need to be checked inside the body of the routine. Of course, an implementation can evaluate all of these at the point of call for statically bound calls if the implementation uses wrappers for dispatching bodies and for 'Access values.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>There is no requirement for an implementation to generate special code for routines that are imported from outside of the Ada program. That's because there is a requirement on the programmer that the use of interfacing aspects do not violate Ada semantics (see B.1). That includes making pre- and postcondition checks. For instance, if the implementation expects routines to make their own postcondition checks in the body before returning, C code can be assumed to do this (even though that is highly unlikely). That's even though the formal definition of those checks is that they are evaluated at the call site. Note that pre- and postconditions can be very useful for verification tools (even if they aren't checked), because they tell the tool about the expectations on the foreign code that it most likely cannot analyze. </p></Admonition>

<p>{"{"}<em>{"AI12-0195-1"}</em>{"}"} For the purposes of the above rules, a call on an inherited subprogram is considered to involve a call on a subprogram <em>S</em>' whose body consists only of a call (with appropriate conversions) on the non-inherited subprogram <em>S</em> from which the inherited subprogram was derived. It is not specified whether class-wide precondition or postcondition expressions that are equivalent (with respect to which non-inherited function bodies are executed) for <em>S</em> and <em>S</em>' are evaluated once or twice. If evaluated only once, the value returned is used for both associated checks.</p>
<Admonition type="aarm" aarm="implementation-note" title="Implementation Note: ">
<p><strong></strong>If the class-wide pre- and postcondition expressions are equivalent for <em>S</em> and <em>S</em>' because none of the primitive subprograms called in those expressions were overridden, no wrapper is needed. Otherwise, a wrapper is presumably needed to provide the correct logic. </p></Admonition>

<p>{"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0247-1"}</em>{"}"} {"{"}<em>{"AI05-0254-1"}</em>{"}"} {"{"}<em>{"AI12-0220-1"}</em>{"}"} For a call via an access-to-subprogram value, precondition and postcondition checks performed are as determined by the subprogram or entry denoted by the prefix of the Access attribute reference that produced the value. In addition, a precondition check of any precondition expression associated with the access-to-subprogram type is performed. Similarly, a postcondition check of any postcondition expression associated with the access-to-subprogram type is performed.</p>
<Admonition type="aarm" aarm="implementation-note" title="Implementation Note: ">
<p><strong></strong>{"{"}<em>{"AI12-0220-1"}</em>{"}"} A call via an access-to-subprogram value can be considered equivalent (with respect to dynamic semantics) to a call to a notional "wrapper" subprogram which has the Pre and Post aspects and the profile of the access-to-subprogram type and whose body contains (and returns, in the case of a function) only a call to the designated subprogram. However, other evaluation orders for the checks are allowed beyond those allowed by strictly following this model. This equivalence can be used to determine the appropriate point at which the constant associated with an Old attribute reference in the Post aspect for an access-to-subprogram type is elaborated and finalized. </p></Admonition>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>{"{"}<em>{"AI12-0220-1"}</em>{"}"} In the case of type conversion between two access-to-subprogram types, the Pre and Post aspects of the source type of the conversion play no role in any subsequent call via the conversion result; only the Pre and Post aspects of the target type of the conversion are relevant in that case. The same applies in the case of a "conversion" (using the term loosely) which is accomplished by combining a dereference and an Access attribute reference, as in Some_Pointer.<strong>all</strong>'Access. </p></Admonition>

<p>{"{"}<em>{"AI12-0272-1"}</em>{"}"} [For a call on a generic formal subprogram, precondition and postcondition checks performed are as determined by the subprogram or entry denoted by the actual subprogram, along with any specific precondition and specific postcondition of the formal subprogram itself.]</p>
<Admonition type="aarm" aarm="note" title="Note: ">
<p><strong>Proof: </strong>{"{"}<em>{"AI12-0272-1"}</em>{"}"} This follows from the general Dynamic Semantics rules given above, but we mention it explicitly so that there can be no doubt that it is intended. </p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p><strong>To be honest: </strong>{"{"}<em>{"AI12-0371-1"}</em>{"}"} The specific precondition and postcondition that apply to a generic formal subprogram also apply to any renaming of that subprogram, even if that renaming is visible in the instance and called from outside of the generic instance. </p></Admonition>


#### Implementation Permissions

<p>{"{"}<em>{"AI12-0280-2"}</em>{"}"} {"{"}<em>{"AI12-0373-1"}</em>{"}"} An implementation may evaluate a known-on-entry subexpression of a postcondition expression of an entity at the place where X'Old constants are created for the entity, with the normal evaluation of the postcondition expression, or both.</p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>{"{"}<em>{"AI12-0280-2"}</em>{"}"} We allow the evaluation of known-on-entry subexpressions when they might be needed to determine whether to create a particular 'Old constant. We allow them to be evaluated later as well, or for the results to be saved somehow. This permission shouldn't matter, as the results ought to be same wherever they are evaluated and there should not be any side effects. The main effect of the permission is to determine when any exceptions caused by such subexpressions may be raised. We never require waiting to determine the value of such subexpressions, even if they aren't used to determine the creation of a constant for 'Old. </p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>NOTE 1   {"{"}<em>{"AI05-0145-2"}</em>{"}"} {"{"}<em>{"AI05-0262-1"}</em>{"}"} {"{"}<em>{"AI12-0442-1"}</em>{"}"} A precondition is checked just before the call. If another task can change any value that the precondition expression depends on, the precondition can evaluate to False within the subprogram or entry body.</p></Admonition>

<p>NOTE 2   {"{"}<em>{"AI12-0312-1"}</em>{"}"} For an example of the use of these aspects and attributes, see the Streams Subsystem definitions in 13.13.1. </p></Admonition>

#### Inconsistencies With Ada 2012

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0032-1"}</em>{"}"} <strong>Corrigendum:</strong> The Old attribute is defined more carefully. This changes the nominal subtype and place of declaration of the attribute compared to the published Ada 2012 Reference Manual. In extreme cases, this could change the runtime behavior of the attribute (for instance, the tag might be different). The changes are most likely going to prevent bugs by being more intuitive, but it is possible that a program that previously worked might fail.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0113-1"}</em>{"}"} {"{"}<em>{"AI12-0159-1"}</em>{"}"} <strong>Corrigendum:</strong> Eliminated unintentional redispatching from class-wide preconditions and postconditions. This means that a different body might be evaluated for a statically bound call to a routine that has a class-wide precondition or postcondition. The change means that the behavior of Pre and Pre'Class will be the same for a particular subprogram, and that the known behavior of the operations can be assumed within the body of that subprogram for Pre'Class. We expect that this change will primarily fix bugs, as it will make Pre'Class and Post'Class work more like expected. In the case where redispatching is desired, an explicit conversion to a class-wide type can be used.</p></Admonition>

<Admonition type="aarm" aarm="correction" title="Correction:">
<p>{"{"}<em>{"AI12-0166-1"}</em>{"}"} <strong></strong> Specified that precondition checks always take place before starting a protected action. Original Ada 2012 left this unspecified, so if an implementation made the checks after starting the protected action, and a program depended upon that, the program might fail in a different compiler. But such a program was depending on unspecified behavior anyway, and thus was never portable; as such, such programs should be rare.</p></Admonition>

<Admonition type="aarm" aarm="correction" title="Correction:">
<p>{"{"}<em>{"AI12-0195-1"}</em>{"}"} <strong></strong> Specified that an inherited subprogram check both the original and new versions of a class-wide precondition. If a call on an inherited subprogram fails the original class-wide precondition when it passes the new class-wide precondition, then the call will fail the precondition check whereas it would have passed in original Ada 2012. (A similar possibility exists for class-wide postconditions.) This can only happen if the overriding subprograms somehow fail to follow the guidelines of LSP, so this should be rare (the entire point of class-wide preconditions and postconditions is to use them in cases where LSP is followed). </p></Admonition>


#### Incompatibilities With Ada 2012

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0045-1"}</em>{"}"} {"{"}<em>{"AI12-0005-1"}</em>{"}"} <strong>Corrigendum:</strong> Precondition and postcondition aspects cannot be specified on instances of generic subprograms (they should be specified on the generic subprogram instead). This was (unintentionally) allowed by the Ada 2012 standard. These are not allowed on instances as there is no corresponding way to add preconditions and postconditions to subprograms declared within the instance of a generic package. Therefore, allowing specification on a subprogram instance could present a maintenance problem in the future if the entity needs to be converted to a generic package (a common conversion).</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0131-1"}</em>{"}"} <strong>Corrigendum:</strong> Pre'Class is no longer allowed to be specified for an overriding primitive subprogram unless there are also inherited class-wide precondittions. This incompatibility prevents cases where the explicit Pre'Class is counterfeited by an implicit class-wide precondition of True. This rule should catch more bugs than it creates; the programmer should have written Pre rather than Pre'Class in this case (or written Pre'Class on the original subprogram, not an overriding). Note that this incompatibility eliminates what otherwise would be an inconsistency with original Ada 2012, where precondition checks that would have previously been made for a statically bound call would no longer be made. That dynamic change was necessary to eliminate cases where the evaluated class-wide precondition on a dispatching call would have been weaker than the class-wide precondition of a statically bound call. (The original Ada 2012 violated the LSP semantics that class-wide preconditions were intended to model.)</p></Admonition>

<Admonition type="aarm" aarm="correction" title="Correction:">
<p>{"{"}<em>{"AI12-0198-1"}</em>{"}"} <strong></strong> A component <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> in an array aggregate can now be potentially unevaluated, requiring the prefix to be statically determined. Existing code that uses the Old attribute with a dynamic prefix in such contexts will now be illegal. However, in many cases, the existing code will not do what the programmer is expecting (as Old is evaluated textually, once per occurrence, while array aggregate components are evaluated once per component). In addition, Old is a new Ada 2012 feature, so most Ada legacy code will not contain it. The problem is usually easily fixed by moving Old to an outer object (such as the entire aggregate). </p></Admonition>

## 6.1.2  The Global and Global'Class Aspects

<p>{"{"}<em>{"AI12-0425-1"}</em>{"}"} {"{"}<em>{"AI12-0439-1"}</em>{"}"} The Global and Global'Class aspects of a program unit are used to identify the objects global to the unit that can be read or written during its execution. </p>

#### Static Semantics

<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} For a subprogram, an entry, an access-to-subprogram type, a task unit, a protected unit, or a library package or generic library package, the following language-defined aspect may be specified with an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> (see 13.1.1):</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} {"{"}<em>{"AI12-0425-1"}</em>{"}"} GlobalThe Global aspect shall be specified with a <code><a href="../AA-6/AA-6.1#S0209">global_aspect_definition</a></code>.</p>
<Admonition type="aarm" aarm="note" title="Note: ">
<p><strong>Aspect Description for </strong><strong>Global: </strong>Global object usage contract.</p></Admonition>

<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} {"{"}<em>{"AI12-0422-1"}</em>{"}"} The Global aspect identifies the set of variables (which, for the purposes of this clause, includes all constants except those which are known to have no variable views (see 3.3)) that are global to a callable entity or task body, and that are read or updated as part of the execution of the callable entity or task body. If specified for a protected unit, it refers to all of the protected operations of the protected unit. Constants of any type may also be mentioned in a Global aspect.</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} If not specified or otherwise defined below, the aspect defaults to the Global aspect for the enclosing library unit if the entity is declared at library level, and to Unspecified otherwise. If not specified for a library unit, the aspect defaults to Global ={">"} <strong>null</strong> for a library unit that is declared Pure, and to Global ={">"} Unspecified otherwise. </p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} For a dispatching subprogram, the following language-defined aspect may be specified with an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> (see 13.1.1):</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} {"{"}<em>{"AI12-0425-1"}</em>{"}"} Global'ClassThe Global'Class aspect shall be specified with a <code><a href="../AA-6/AA-6.1#S0209">global_aspect_definition</a></code>. This aspect identifies an upper bound on the set of variables global to a dispatching operation that can be read or updated as a result of a dispatching call on the operation. If not specified, the aspect defaults to the Global aspect for the dispatching subprogram.</p>
<Admonition type="aarm" aarm="note" title="Note: ">
<p><strong>Aspect Description for </strong><strong>Global'Class: </strong>Global object usage contract inherited on derivation.</p></Admonition>

<p>{"{"}<em>{"AI12-0380-1"}</em>{"}"} Together, we refer to the Global and Global'Class aspects as <em>global</em> aspects.</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} A <code><a href="../AA-6/AA-6.1#S0209">global_aspect_definition</a></code> defines the Global or Global'Class aspect of some entity. The Global aspect identifies the sets of global variables that can be read, written, or modified as a side effect of executing the operation(s) associated with the entity. The Global'Class aspect associated with a dispatching operation of type <em>T</em> represents a restriction on the Global aspect on a corresponding operation of any descendant of type <em>T</em>.</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} {"{"}<em>{"AI12-0416-1"}</em>{"}"} {"{"}<em>{"AI12-0442-1"}</em>{"}"} The Global aspect for a callable entity defines the global variables that can be referenced as part of a call on the entity, including any assertion expressions that apply to the call (even if not enabled), such as preconditions, postconditions, predicates, and type invariants.</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} {"{"}<em>{"AI12-0439-1"}</em>{"}"} The Global aspect for an access-to-subprogram object (or subtype) identifies the global variables that can be referenced when calling via the object (or any object of that subtype) including assertion expressions that apply.</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} {"{"}<em>{"AI12-0405-1"}</em>{"}"} For a predefined operator of an elementary type, the function representing an enumeration literal, or any other static function (see 4.9), the Global aspect is <strong>null</strong>. For a predefined operator of a composite type, the Global aspect of the operator defaults to that of the enclosing library unit (unless a Global aspect is specified for the type - see H.7).</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} {"{"}<em>{"AI12-0380-1"}</em>{"}"} The following is defined in terms of operations that are performed by or on behalf of an entity. The rules on operations apply to the entity(s) associated with those operations.</p>
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<p><strong></strong>{"{"}<em>{"AI12-0380-1"}</em>{"}"} The operations performed by a callable entity are those associated with the body of the entity. For other kinds of entities (such as subtypes, see H.7), we explicitly list the associated operations. </p></Admonition>

<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} The global variables associated with any <code><a href="../AA-6/AA-6.1#S0211">global_mode</a></code> can be read as a side effect of an operation. The <strong>in out</strong> and <strong>out</strong> <code><a href="../AA-6/AA-6.1#S0211">global_mode</a></code>s together identify the set of global variables that can be updated as a side effect of an operation. Creating an access-to-variable value that designates an object is considered an update of the designated object, and creating an access-to-constant value that designates an object is considered a read of the designated object.</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} The overall set of objects associated with each <code><a href="../AA-6/AA-6.1#S0211">global_mode</a></code> includes all objects identified for the mode in the <code><a href="../AA-6/AA-6.1#S0209">global_aspect_definition</a></code>.</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} A <code><a href="../AA-6/AA-6.1#S0213">global_set</a></code> identifies a <em>global variable set</em> as follows:</p>
<p><strong>all</strong> identifies the set of all global variables;</p>
<p><strong>synchronized</strong> identifies the set of all synchronized variables (see 9.10), as well as variables of a composite type all of whose non-discriminant subcomponents are synchronized;</p>
<p><code><a href="../AA-6/AA-6.1#S0215">global_name</a></code>{"{"}, <code><a href="../AA-6/AA-6.1#S0215">global_name</a></code>{"}"} identifies the union of the sets of variables identified by the <code><a href="../AA-6/AA-6.1#S0215">global_name</a></code>s in the list, for the following forms of <code><a href="../AA-6/AA-6.1#S0215">global_name</a></code>:</p>
<p><em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> identifies the specified global variable (or constant);</p>
<p><em>package_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> identifies the set of all variables declared in the private part or body of the package, or anywhere within a private descendant of the package. </p>

#### Legality Rules

<p>{"{"}<em>{"AI12-0079-1"}</em>{"}"} Within a <code><a href="../AA-6/AA-6.1#S0209">global_aspect_definition</a></code>, a given <code><a href="../AA-6/AA-6.1#S0211">global_mode</a></code> shall be specified at most once. Similarly, within a <code><a href="../AA-6/AA-6.1#S0209">global_aspect_definition</a></code>, a given entity shall be named at most once by a <code><a href="../AA-6/AA-6.1#S0215">global_name</a></code>.</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} If an entity (other than a library package or generic library package) has a Global aspect other than Unspecified or <strong>in out all</strong>, then the associated operation(s) shall read only those variables global to the entity that are within the global variable set associated with the <strong>in</strong>, <strong>in out</strong>, or <strong>out</strong> <code><a href="../AA-6/AA-6.1#S0211">global_mode</a></code>s, and the operation(s) shall update only those variables global to the entity that are within the global variable set associated with either the <strong>in out</strong> or <strong>out</strong> <code><a href="../AA-6/AA-6.1#S0211">global_mode</a></code>s. In the absence of the No_Hidden_Indirect_Globals restriction (see H.4), this ignores objects reached via a dereference of an access value.  The above rule includes any possible Global effects of calls occurring during the execution of the operation, except for the following excluded calls:</p>
<p>calls to formal subprograms;</p>
<p>calls associated with operations on formal subtypes;</p>
<p>calls through formal objects of an access-to-subprogram type;</p>
<p>calls through access-to-subprogram parameters;</p>
<p>calls on operations with Global aspect Unspecified.</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} The possible Global effects of these excluded calls (other than those that are Unspecified) are taken into account by the caller of the original operation, by presuming they occur at least once during its execution. For calls that are not excluded, the possible Global effects of the call are those permitted by the Global aspect of the associated entity, or by its Global'Class aspect if a dispatching call.</p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>For a predefined equality operator of a composite type, the possible Global effects includes those of the equality operations invoked as part of the evaluation of the operator (which might not be predefined and thus might have a different Global specification than the component types). </p></Admonition>

<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} If a Global aspect other than Unspecified or <strong>in out all</strong> applies to an access-to-subprogram type, then the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of an Access <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code> producing a value of such a type shall denote a subprogram whose Global aspect is not Unspecified and is <em>covered</em> by that of the result type, where a global aspect <em>G1</em> is <em>covered</em> by a global aspect <em>G2</em> if the set of variables that <em>G1</em> identifies as readable or updatable is a subset of the corresponding set for <em>G2</em>. Similarly on a conversion to such a type, the operand shall be of a named access-to-subprogram type whose Global aspect is covered by that of the target type.</p>
<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} If an implementation-defined <code><a href="../AA-6/AA-6.1#S0211">global_mode</a></code> applies to a given set of variables, an implementation-defined rule determines what sort of references to them are permitted.</p>
<p>{"{"}<em>{"AI12-0079-1"}</em>{"}"} For a subprogram that is a dispatching operation of a tagged type <em>T</em>, each mode of its Global aspect shall identify a subset of the variables identified by the corresponding mode, or by the <strong>in out</strong> mode, of the Global'Class aspect of a corresponding dispatching subprogram of any ancestor of <em>T</em>, unless the aspect of that ancestor is Unspecified.</p>

#### Implementation Permissions

<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} {"{"}<em>{"AI12-0444-1"}</em>{"}"} An implementation can allow some references to a constant object which are not accounted for by the Global or Global'Class aspect when it is considered a variable in the above rules, if the implementation can determine that the object is in fact immutable.</p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>In particular, this allows the implementation to violate privacy in order to determine whether a constant needs to be covered by a Global aspect. </p></Admonition>

<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} Implementations may perform additional checks on calls to operations with an Unspecified Global aspect to ensure that they do not violate any limitations associated with the point of call.</p>
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<p><strong></strong>In this sense, Global ={">"} Unspecified is not permission to violate the caller's Global restrictions. It is rather that the implementor of the subprogram is presuming other means are being used to ensure safety. Note the No_Unspecified_Globals Restriction (H.4), which prevents the use of Unspecified with the Global aspect in a given partition. </p></Admonition>

<p>{"{"}<em>{"AI12-0079-3"}</em>{"}"} Implementations may extend the syntax or semantics of the Global aspect in an implementation-defined manner[; for example, supporting additional <code><a href="../AA-6/AA-6.1#S0211">global_mode</a></code>s].</p>
<Admonition type="aarm" aarm="implementation-defined" title="Implementation defined: ">
<p><strong></strong>Any extensions of the Global aspect.</p></Admonition>

<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>This is intended to allow preexisting usages from SPARK 2014 to remain acceptable in conforming implementations, as well as to provide flexibility for future enhancements. Note the word "extend" in this permission; we expect that any aspect usage that conforms with the (other) rules of this clause will be accepted by any Ada implementation, regardless of any implementation-defined extensions. </p></Admonition>
