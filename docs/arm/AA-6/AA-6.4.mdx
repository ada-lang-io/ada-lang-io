<MarginText>1</MarginText>
<p>A <em>subprogram call</em> is either a <code><a href="../AA-6/AA-6.4#S0217">procedure_call_statement</a></code> or a <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code>; [it invokes the execution of the <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code>. The call specifies the association of the actual parameters, if any, with formal parameters of the subprogram.] <br /></p>


#### Syntax

<MarginText>2</MarginText>

<CodeBlock>
<code>procedure_call_statement</code><a id="S0217"></a><code> ::= </code><br />    <em>procedure_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>;<br />  | <em>procedure_</em><code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> <code><a href="../AA-6/AA-6.4#S0219">actual_parameter_part</a></code>;<br />

</CodeBlock>
<MarginText>3</MarginText>

<CodeBlock>
<code>function_call</code><a id="S0218"></a><code> ::= </code><br />    <em>function_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code><br />  | <em>function_</em><code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> <code><a href="../AA-6/AA-6.4#S0219">actual_parameter_part</a></code><br />

</CodeBlock>
<AnnotatedOnly>
<MarginText>3.a/3</MarginText>
<MarginInfo items={["AI05-0005-1"]} />
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>For the purpose of non-syntax rules, infix operator calls are considered <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code>s. See <a href="../AA-6/AA-6.6">6.6</a>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>4</MarginText>

<CodeBlock>
<code>actual_parameter_part</code><a id="S0219"></a><code> ::= </code><br />    (<code><a href="../AA-6/AA-6.4#S0220">parameter_association</a></code> {"{"}, <code><a href="../AA-6/AA-6.4#S0220">parameter_association</a></code>{"}"})<br />

</CodeBlock>
<MarginText>5</MarginText>

<CodeBlock>
<code>parameter_association</code><a id="S0220"></a><code> ::= </code><br />   [<em>formal_parameter_</em><code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code> ={">"}] <code><a href="../AA-6/AA-6.4#S0221">explicit_actual_parameter</a></code><br />

</CodeBlock>
<MarginText>6</MarginText>

<CodeBlock>
<code>explicit_actual_parameter</code><a id="S0221"></a><code> ::= </code><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> | <em>variable_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code><br />

</CodeBlock>
<MarginText>7/5</MarginText>
<MarginInfo items={["AI12-0416-1"]} />
<p class="Indented2">A <code><a href="../AA-6/AA-6.4#S0220">parameter_association</a></code> is <em>named</em> or <em>positional</em> according to whether or not the <em>formal_parameter_</em><code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code> is specified. For the <code><a href="../AA-6/AA-6.4#S0220">parameter_association</a></code>s of a single <code><a href="../AA-6/AA-6.4#S0219">actual_parameter_part</a></code> or <code><a href="../AA-5/AA-5.5#S0188">iterator_actual_parameter_part</a></code>, any  positional associations shall precede any named associations. Named associations are not allowed if the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> in a subprogram call is an <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>7.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This means that the formal parameter names used in describing predefined attributes are to aid presentation of their semantics, but are not intended for use in actual calls. <br /></Admonition>
</AnnotatedOnly>


#### Name Resolution Rules

<MarginText>8/2</MarginText>
<MarginInfo items={["AI95-00310-01"]} />
<p>The <code><a href="../AA-4/AA-4.1#S0091">name</a></code> or <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> given in a <code><a href="../AA-6/AA-6.4#S0217">procedure_call_statement</a></code> shall resolve to denote a callable entity that is a procedure, or an entry renamed as (viewed as) a procedure. The <code><a href="../AA-4/AA-4.1#S0091">name</a></code> or <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> given in a <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code> shall resolve to denote a callable entity that is a function. The <code><a href="../AA-4/AA-4.1#S0091">name</a></code> or <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> shall not resolve to denote an abstract subprogram unless it is also a dispatching subprogram. [When there is an <code><a href="../AA-6/AA-6.4#S0219">actual_parameter_part</a></code>, the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> can be an <code><a href="../AA-4/AA-4.1#S0095">implicit_dereference</a></code> of an access-to-subprogram value.] <br /></p>

<AnnotatedOnly>
<MarginText>8.a.1/2</MarginText>
<MarginInfo items={["AI95-00310-01"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>This rule is talking about dispatching operations (which is a static concept) and not about dispatching calls (which is a dynamic concept). <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The function can be an operator, enumeration literal, attribute that is a function, etc. <br /></Admonition>
</AnnotatedOnly>

<MarginText>9</MarginText>
<p>A subprogram call shall contain at most one association for each formal parameter. Each formal parameter without an association shall have a <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code> (in the profile of the view denoted by the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> or <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code>). [This rule is an overloading rule (see <a href="../AA-8/AA-8.6">8.6</a>).] <br /></p>

<AnnotatedOnly>
<MarginText>9.a/3</MarginText>
<MarginInfo items={["AI05-0240-1"]} />
<Admonition type="aarm" aarm="proof">
<strong></strong>All Name Resolution Rules are overloading rules, see <a href="../AA-8/AA-8.6">8.6</a>. <br /></Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>9.1/5</MarginText>
<MarginInfo items={["AI95-00407-01", "AI12-0416-1"]} />
<p>If the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> or <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of a subprogram call denotes a prefixed view (see <a href="../AA-4/AA-4.1#Subclause_4.1.3">4.1.3</a>), the subprogram call is equivalent to a call on the underlying subprogram, with the first actual parameter being provided by the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of the prefixed view (or the Access attribute of this <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> if the first formal parameter is an access parameter), and the remaining actual parameters given by the <code><a href="../AA-6/AA-6.4#S0219">actual_parameter_part</a></code>, if any. <br /></p>


#### Dynamic Semantics

<MarginText>10/2</MarginText>
<MarginInfo items={["AI95-00345-01"]} />
<p>For the execution of a subprogram call, the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> or <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of the call is evaluated, and each <code><a href="../AA-6/AA-6.4#S0220">parameter_association</a></code> is evaluated (see <a href="../AA-6/AA-6.4#Subclause_6.4.1">6.4.1</a>). If a <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code> is used, an implicit <code><a href="../AA-6/AA-6.4#S0220">parameter_association</a></code> is assumed for this rule. These evaluations are done in an arbitrary order. The <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> is then executed, or a call on an entry or protected subprogram is performed (see <a href="../AA-3/AA-3.9#Subclause_3.9.2">3.9.2</a>). Finally, if the subprogram completes normally, then after it is left, any necessary assigning back of formal to actual parameters occurs (see <a href="../AA-6/AA-6.4#Subclause_6.4.1">6.4.1</a>).<br /></p>

<AnnotatedOnly>
<MarginText>10.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The implicit association for a default is only for this run-time rule. At compile time, the visibility rules are applied to the default at the place where it occurs, not at the place of a call. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.b</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>If the subprogram is inherited, see <a href="../AA-3/AA-3.4">3.4</a>, &ldquo;<a href="../AA-3/AA-3.4">Derived Types and Classes</a>&rdquo;.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.c</MarginText>
<Admonition type="aarm" aarm="note">
If the subprogram is protected, see <a href="../AA-9/AA-9.5#Subclause_9.5.1">9.5.1</a>, &ldquo;<a href="../AA-9/AA-9.5#Subclause_9.5.1">Protected Subprograms and Protected Actions</a>&rdquo;.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.d</MarginText>
<Admonition type="aarm" aarm="note">
If the subprogram is really a renaming of an entry, see <a href="../AA-9/AA-9.5#Subclause_9.5.3">9.5.3</a>, &ldquo;<a href="../AA-9/AA-9.5#Subclause_9.5.3">Entry Calls</a>&rdquo;.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.d.1/2</MarginText>
<MarginInfo items={["AI95-00345-01"]} />
<Admonition type="aarm" aarm="note">
If the subprogram is implemented by an entry or protected subprogram, it will be treated as a dispatching call to the corresponding entry (see <a href="../AA-9/AA-9.5#Subclause_9.5.3">9.5.3</a>, &ldquo;<a href="../AA-9/AA-9.5#Subclause_9.5.3">Entry Calls</a>&rdquo;) or protected subprogram (see <a href="../AA-9/AA-9.5#Subclause_9.5.1">9.5.1</a>, &ldquo;<a href="../AA-9/AA-9.5#Subclause_9.5.1">Protected Subprograms and Protected Actions</a>&rdquo;).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.e/2</MarginText>
<MarginInfo items={["AI95-00348-01"]} />
<Admonition type="aarm" aarm="note">
Normally, the <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> that is executed by the above rule is the one for the subprogram being called. For an enumeration literal, implicitly declared (but noninherited) subprogram, null procedure, or an attribute that is a subprogram, an implicit body is assumed. For a dispatching call, <a href="../AA-3/AA-3.9#Subclause_3.9.2">3.9.2</a>, &ldquo;<a href="../AA-3/AA-3.9#Subclause_3.9.2">Dispatching Operations of Tagged Types</a>&rdquo; defines which <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> is executed. <br /></Admonition>
</AnnotatedOnly>

<MarginText>10.1/5</MarginText>
<MarginInfo items={["AI95-00407-01", "AI12-0416-1"]} />
<p> <br /></p>

<MarginText>11/2</MarginText>
<MarginInfo items={["AI95-00318-02"]} />
<p>The exception Program_Error is raised at the point of a <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code> if the function completes normally without executing a return statement. <br /></p>

<AnnotatedOnly>
<MarginText>11.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>We are committing to raising the exception at the point of call, for uniformity &mdash; see AI83-00152. This happens after the function is left, of course.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.b</MarginText>
<Admonition type="aarm" aarm="note">
Note that there is no name for suppressing this check, since the check imposes no time overhead and minimal space overhead (since it can usually be statically eliminated as dead code). <br /></Admonition>
</AnnotatedOnly>

<MarginText>12/2</MarginText>
<MarginInfo items={["AI95-00231-01"]} />
<p>A <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code> denotes a constant, as defined in <a href="../AA-6/AA-6.5">6.5</a>; the nominal subtype of the constant is given by the nominal subtype of the function result. <br /></p>

## 6.4.1  Parameter Associations

<MarginText>1</MarginText>
<p>[ A parameter association defines the association between an actual parameter and a formal parameter.] <br /></p>


#### Language Design Principles

<AnnotatedOnly>
<MarginText>1.a</MarginText>
<Admonition type="aarm" aarm="note">
The parameter passing rules for <strong>out</strong> parameters are designed to ensure that the parts of a type that have implicit initial values (see <a href="../AA-3/AA-3.3#Subclause_3.3.1">3.3.1</a>) don't become &ldquo;de-initialized&rdquo; by being passed as an <strong>out</strong> parameter.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>1.b/3</MarginText>
<MarginInfo items={["AI05-0142-4"]} />
<Admonition type="aarm" aarm="note">
For explicitly aliased parameters of functions, we will ensure at the call site that a part of the parameter can be returned as part of the function result without creating a dangling pointer. We do this with accessibility checks at the call site that all actual objects of explicitly aliased parameters live at least as long as the function result; then we can allow them to be returned as access discriminants or anonymous access results, as those have the master of the function result. <br /></Admonition>
</AnnotatedOnly>


#### Name Resolution Rules

<MarginText>2/3</MarginText>
<MarginInfo items={["AI05-0118-1"]} />
<p>The <em>formal_parameter_</em><code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code> of a named <code><a href="../AA-6/AA-6.4#S0220">parameter_association</a></code> shall resolve to denote a <code><a href="../AA-6/AA-6.1#S0207">parameter_specification</a></code> of the view being called; this is the formal parameter of the association. The formal parameter for a positional <code><a href="../AA-6/AA-6.4#S0220">parameter_association</a></code> is the parameter with the corresponding position in the formal part of the view being called.<br /></p>

<AnnotatedOnly>
<MarginText>2.a/3</MarginText>
<MarginInfo items={["AI05-0118-1"]} />
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>For positional parameters, the &ldquo;corresponding position&rdquo; is calculated after any transformation of prefixed views. <br /></Admonition>
</AnnotatedOnly>

<MarginText>3</MarginText>
<p>The <em>actual parameter</em> is either the <code><a href="../AA-6/AA-6.4#S0221">explicit_actual_parameter</a></code> given in a <code><a href="../AA-6/AA-6.4#S0220">parameter_association</a></code> for a given formal parameter, or the corresponding <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code> if no <code><a href="../AA-6/AA-6.4#S0220">parameter_association</a></code> is given for the formal parameter. The expected type for an actual parameter is the type of the corresponding formal parameter. <br /></p>

<AnnotatedOnly>
<MarginText>3.a</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>The corresponding <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code> is the one of the corresponding formal parameter in the profile of the view denoted by the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> or <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of the call. <br /></Admonition>
</AnnotatedOnly>

<MarginText>4</MarginText>
<p>If the mode is <strong>in</strong>, the actual is interpreted as an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>; otherwise, the actual is interpreted only as a <code><a href="../AA-4/AA-4.1#S0091">name</a></code>, if possible. <br /></p>

<AnnotatedOnly>
<MarginText>4.a/4</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>This formally resolves the ambiguity present in the syntax rule for <code><a href="../AA-6/AA-6.4#S0221">explicit_actual_parameter</a></code>. This matters as an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> that is a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> is evaluated and represents a value while a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> by itself can be an object; if the mode is not <strong>in</strong>, we want the parameter to interpreted as an object. Note that we don't actually require that the actual be a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> if the mode is not <strong>in</strong>; we do that below.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.b/4</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
<Admonition type="aarm" aarm="note">
This wording uses "interpreted as" rather than "shall be" so that this rule is not used to resolve overloading; it is solely about evaluation as described above. We definitely do not want to allow oddities like the presence of parentheses requiring the selection of an <strong>in</strong> formal parameter as opposed to an otherwise matching <strong>in out</strong> parameter. <br /></Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>5</MarginText>
<p>If the mode is <strong>in out</strong> or <strong>out</strong>, the actual shall be a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes a variable. <br /></p>

<AnnotatedOnly>
<MarginText>5.a/5</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>We no longer need &ldquo;or a <code><a href="../AA-4/AA-4.6#S0162">type_conversion</a></code> whose argument is the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> of a variable&rdquo;, because a <code><a href="../AA-4/AA-4.6#S0162">type_conversion</a></code> is now a <code><a href="../AA-4/AA-4.1#S0091">name</a></code>, and a view conversion  of a variable is a variable while any other conversion (which should not be legal here) is a constant. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The requirement that the actual be a (variable) <code><a href="../AA-4/AA-4.1#S0091">name</a></code> is not an overload resolution rule, since we don't want the difference between <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> and <code><a href="../AA-4/AA-4.1#S0091">name</a></code> to be used to resolve overloading. For example: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.c</MarginText>
<CodeBlock language="ada">
procedure Print(X : in Integer; Y : in Boolean := True);{"\n"}procedure Print(Z : in out Integer);{"\n"}. . .{"\n"}Print(3); -- Ambiguous!{"\n"}  {"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>5.d</MarginText>
<Admonition type="aarm" aarm="note">
The above call to Print is ambiguous even though the call is not compatible with the second Print which requires an actual that is a (variable) <code><a href="../AA-4/AA-4.1#S0091">name</a></code> (&ldquo;3&rdquo; is an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, not a <code><a href="../AA-4/AA-4.1#S0091">name</a></code>). This requirement is a legality rule, so overload resolution fails before it is considered, meaning that the call is ambiguous. <br /></Admonition>
</AnnotatedOnly>

<MarginText>5.1/5</MarginText>
<MarginInfo items={["AI12-0074-1", "AI12-0159-1", "AI12-0377-1"]} />
<p>If the mode is <strong>out</strong>, the actual parameter is a view conversion, and the type of the formal parameter is  a scalar type , then <br /></p>

<MarginText>5.2/5</MarginText>
<ul>
<li>neither  the target type nor  the operand type has the Default_Value aspect specified; or <br /></li>
<MarginText>5.3/5</MarginText>
<li>both the target type and  the operand type  shall have the Default_Value aspect specified, and there shall exist a type (other than a root numeric type) that is an ancestor of both the target type and the operand type. <br /></li>
</ul>
<MarginText>5.4/4</MarginText>
<MarginInfo items={["AI12-0074-1", "AI12-0159-1"]} />
<p>In addition to the places where Legality Rules normally apply (see <a href="../AA-12/AA-12.3">12.3</a>), these rules also apply in the private part of an instance of a generic unit.<br /></p>

<AnnotatedOnly>
<MarginText>5.e/4</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>These rules are needed in order to ensure that a well-defined parameter value is passed. <br /></Admonition>
</AnnotatedOnly>

<MarginText>6/3</MarginText>
<MarginInfo items={["AI05-0102-1", "AI05-0142-4"]} />
<p>If the formal parameter is an explicitly aliased parameter, the type of the actual parameter shall be tagged or the actual parameter shall be an aliased view of an object. Further, if the formal parameter subtype <em>F</em> is untagged:<br /></p>

<MarginText>6.1/3</MarginText>
<ul>
<li>the subtype <em>F</em> shall statically match the nominal subtype of the actual object; or<br /></li>
<MarginText>6.2/3</MarginText>
<li>the subtype <em>F</em> shall be unconstrained, discriminated in its full view, and unconstrained in any partial view. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>6.a/3</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Tagged objects (and tagged <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>s for <strong>in</strong> parameters) do not need to be aliased. This matches the behavior of unaliased formal parameters of tagged types, which allow 'Access to be taken of the formal parameter regardless of the form of the actual parameter. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.b/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We need the subtype check on untagged actual parameters so that the requirements of 'Access are not lost. 'Access makes its checks against the nominal subtype of its prefix, and parameter passing can change that subtype. But we don't want this parameter passing to change the objects that would be allowed as the prefix of 'Access. This is particularly important for arrays, where we don't want to require any additional implementation burden. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.b.1/5</MarginText>
<MarginInfo items={["AI12-0005-1", "AI12-0095-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>We assume the worst in a generic body regarding whether  a formal subtype has a constrained partial view; specifically, in a generic body a discriminated subtype is considered to have a constrained partial view if it is a descendant of an untagged generic formal private or derived type (see <a href="../AA-12/AA-12.5#Subclause_12.5.1">12.5.1</a> for the formal definition of this rule). <br /></Admonition>
</AnnotatedOnly>

<MarginText>6.3/4</MarginText>
<MarginInfo items={["AI12-0095-1"]} />
<p>In addition to the places where Legality Rules normally apply (see <a href="../AA-12/AA-12.3">12.3</a>), these rules also apply in the private part of an instance of a generic unit.<br /></p>

<MarginText>6.4/3</MarginText>
<MarginInfo items={["AI05-0142-4", "AI05-0234-1"]} />
<p>In a function call, the accessibility level of the actual object for each explicitly aliased parameter shall not be statically deeper than the accessibility level of the master of the call (see <a href="../AA-3/AA-3.10#Subclause_3.10.2">3.10.2</a>).<br /></p>

<AnnotatedOnly>
<MarginText>6.c/3</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Since explicitly aliased parameters are either tagged or required to be objects, there is always an object (possibly anonymous) to talk about. This is discussing the static accessibility level of the actual object; it does not depend on any runtime information (for instance when the actual object is a formal parameter of another subprogram, it does not depend on the actual parameter of that other subprogram). <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.d/4</MarginText>
<MarginInfo items={["AI12-0095-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>This accessibility check (and its dynamic cousin as well) can only fail if the master of the function call (which is defined in the Heart of Darkness, or <a href="../AA-3/AA-3.10#Subclause_3.10.2">3.10.2</a> if you prefer) is different than the master directly enclosing the call. The most likely place where this will occur is in the initializer of an <code><a href="../AA-4/AA-4.8#S0164">allocator</a></code>; in almost all other cases this check will always pass. <br /></Admonition>
</AnnotatedOnly>

<MarginText>6.5/3</MarginText>
<MarginInfo items={["AI05-0144-2"]} />
<p>Two <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s are <em>known to denote the same object</em> if:<br /></p>

<MarginText>6.6/3</MarginText>
<ul>
<li>both <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s statically denote the same stand-alone object or parameter; or<br /></li>
<MarginText>6.7/3</MarginText>
<li>both <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s are <code><a href="../AA-4/AA-4.1#S0098">selected_component</a></code>s, their <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code>es are known to denote the same object, and their <code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code>s denote the same component; or<br /></li>
<MarginText>6.8/3</MarginText>
<li>both <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s are dereferences (implicit or explicit) and the dereferenced <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s are known to denote the same object; or<br /></li>
<MarginText>6.9/3</MarginText>
<li>both <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s are <code><a href="../AA-4/AA-4.1#S0096">indexed_component</a></code>s, their <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code>es are known to denote the same object, and each of the pairs of corresponding index values are either both static expressions with the same static value or both <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s that are known to denote the same object; or<br /></li>
<MarginText>6.10/3</MarginText>
<li>both <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s are <code><a href="../AA-4/AA-4.1#S0097">slice</a></code>s, their <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code>es are known to denote the same object, and the two <code><a href="../AA-4/AA-4.1#S0097">slice</a></code>s have statically matching index constraints; or<br /></li>
<MarginText>6.11/3</MarginText>
<li>one of the two <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s statically denotes a renaming declaration whose renamed <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> is known to denote the same object as the other, the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of any dereference within the renamed <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> is not a variable, and any <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> within the renamed <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> contains no references to variables nor calls on nonstatic functions. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>6.e/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This exposes known renamings of slices, indexing, and so on to this definition. In particular, if we have <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.f/3</MarginText>
<CodeBlock language="ada">
C : Character renames S(1);{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>6.g/3</MarginText>
<Admonition type="aarm" aarm="note">
then C and S(1) are known to denote the same object.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.h/3</MarginText>
<Admonition type="aarm" aarm="note">
We need the requirement that no variables occur in the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code>es of dereferences and in (index) <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s of the renamed object in order to avoid problems from later changes to those parts of renamed names. Consider:<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.i/3</MarginText>
<CodeBlock language="ada">
   type Ref is access Some_Type;{"\n"}   Ptr : Ref := new Some_Type'(...);{"\n"}   X : Some_Type renames Ptr.all;{"\n"}begin{"\n"}   Ptr := new Some_Type'(...);{"\n"}   P (Func_With_Out_Params (Ptr.all), X);{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>6.j/3</MarginText>
<Admonition type="aarm" aarm="note">
X and Ptr.<strong>all</strong> should not be known to denote the same object, since they denote different allocated objects (and this is not an unreasonable thing to do). <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.k/3</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>The exclusion of variables from renamed object_names is not enough to prevent altering the value of the name or expression by another access path. For instance, both <strong>in</strong> parameters passed by reference and access-to-constant values can designate variables. For the intended use of "known to be the same object", this is OK; the modification via another access path is very tricky and it is OK to reject code that would be buggy except for the tricky code. Assuming Element is an elementary type, consider the following example: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.l/3</MarginText>
<CodeBlock language="ada">
Global : Tagged_Type;{"\n"}
<MarginText>6.m/3</MarginText>
procedure Foo (Param : in Tagged_Type := Global) is{"\n"}   X : Element renames Some_Global_Array (Param.C);{"\n"}begin{"\n"}   Global.C := Global.C + 1;{"\n"}   Swap (X, Some_Global_Array (Param.C));{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>6.n/3</MarginText>
<Admonition type="aarm" aarm="note">
The rules will flag the call of procedure Swap as illegal, since X and Some_Global_Array (Parameter.C) are known to denote the same object (even though they will actually represent different objects if Param = Global). But this is only incorrect if the parameter actually is Global and not some other value; the error could exist for some calls. So this flagging seems harmless.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.o/3</MarginText>
<Admonition type="aarm" aarm="note">
Similar examples can be constructed using stand-alone composite constants with controlled or immutably limited components, and (as previously noted) with dereferences of access-to-constant values. Even when these examples flag a call incorrectly, that call depends on very tricky code (modifying the value of a constant); the code is likely to confuse future maintainers as well and thus we do not mind rejecting it. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.p/3</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Whether or not <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s or <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code>es are known to denote the same object is determined statically. If the name contains some dynamic portion other than a dereference, <code><a href="../AA-4/AA-4.1#S0096">indexed_component</a></code>, or <code><a href="../AA-4/AA-4.1#S0097">slice</a></code>, it is not "known to denote the same object".<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.q/3</MarginText>
<Admonition type="aarm" aarm="note">
These rules make no attempt to handle slices of objects that are known to be the same when the slices have dynamic bounds (other than the trivial case of bounds being defined by the same subtype), even when the bounds could be proven to be the same, as it is just too complex to get right and these rules are intended to be conservative. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.r/3</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>"Known to denote the same object" is intended to be an equivalence relationship, that is, it is reflexive, symmetric, and transitive. We believe this follows from the rules. For instance, given the following declarations: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.s/3</MarginText>
<CodeBlock language="ada">
S   : String(1..10);{"\n"}ONE : constant Natural := 1;{"\n"}R   : Character renames S(1);{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>6.t/3</MarginText>
<Admonition type="aarm" aarm="note">
the names R and S(1) are known to denote the same object by the sixth bullet, and S(1) and S(ONE) are known to denote the same object by the fourth bullet, so using the sixth bullet on R and S(ONE), we simply have to test S(1) vs. S(ONE), which we already know denote the same object. <br /></Admonition>
</AnnotatedOnly>

<MarginText>6.12/3</MarginText>
<MarginInfo items={["AI05-0144-2"]} />
<p>Two <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s are <em>known to refer to the same object</em> if <br /></p>

<MarginText>6.13/3</MarginText>
<ul>
<li>The two <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s are known to denote the same object; or<br /></li>
<MarginText>6.14/3</MarginText>
<li>One of the <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s is a <code><a href="../AA-4/AA-4.1#S0098">selected_component</a></code>, <code><a href="../AA-4/AA-4.1#S0096">indexed_component</a></code>, or <code><a href="../AA-4/AA-4.1#S0097">slice</a></code> and its <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> is known to refer to the same object as the other <code><a href="../AA-4/AA-4.1#S0091">name</a></code>; or<br /></li>
<MarginText>6.15/3</MarginText>
<li>One of the two <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s statically denotes a renaming declaration whose renamed <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> is known to refer to the same object as the other <code><a href="../AA-4/AA-4.1#S0091">name</a></code>. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>6.u/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This ensures that names Prefix.Comp and Prefix are known to refer to the same object for the purposes of the rules below. This intentionally does not include dereferences; we only want to worry about accesses to the same object, and a dereference changes the object in question. (There is nothing shared between an access value and the object it designates.) <br /></Admonition>
</AnnotatedOnly>

<MarginText>6.16/3</MarginText>
<MarginInfo items={["AI05-0144-2"]} />
<p>If a call <em>C</em> has two or more parameters of mode <strong>in out</strong> or <strong>out</strong> that are of an elementary type, then the call is legal only if:<br /></p>

<MarginText>6.17/5</MarginText>
<MarginInfo items={["AI12-0216-1", "AI12-0324-1"]} />
<ul>
<li>For each <code><a href="../AA-4/AA-4.1#S0091">name</a></code> <em>N</em> denoting an object of an elementary type that is passed as a parameter of mode <strong>in out</strong> or <strong>out</strong> to the call <em>C</em>, there is no other <code><a href="../AA-4/AA-4.1#S0091">name</a></code> among the other parameters of mode <strong>in out</strong> or <strong>out</strong> to <em>C</em> that is known to denote the same object.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>6.v/3</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>This means <em>visibly</em> an elementary type; it does not include partial views of elementary types (partial views are always composite). That's necessary to avoid having Legality Rules depend on the contents of the private part. <br /></Admonition>
</AnnotatedOnly>

<MarginText>6.18/3</MarginText>
<MarginInfo items={["AI05-0144-2"]} />
<p>If a construct <em>C</em> has two or more direct constituents that are <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s or <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s whose evaluation may occur in an arbitrary order, at least one of which contains a function call with an <strong>in out</strong> or <strong>out</strong> parameter, then the construct is legal only if:<br /></p>

<AnnotatedOnly>
<MarginText>6.w/3</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>All of the places where the language allows an arbitrary order can be found by looking in the index under "arbitrary order, allowed". Note that this listing includes places that don't involve <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s or <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s (such as checks or finalization). <br /></Admonition>
</AnnotatedOnly>

<MarginText>6.19/3</MarginText>
<ul>
<li>For each name <em>N</em> that is passed as a parameter of mode <strong>in out</strong> or <strong>out</strong> to some inner function call <em>C2</em> (not including the construct <em>C</em> itself), there is no other <code><a href="../AA-4/AA-4.1#S0091">name</a></code> anywhere within a direct constituent of the construct <em>C</em> other than the one containing <em>C2</em>, that is known to refer to the same object. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>6.x/3</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This requirement cannot fail for a procedure or entry call alone; there must be at least one function with an <strong>in out</strong> or <strong>out</strong> parameter called as part of a parameter expression of the call in order for it to fail. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.y/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>These rules prevent obvious cases of dependence on the order of evaluation of <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s or <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s. Such dependence is usually a bug, and in any case, is not portable to another implementation (or even another optimization setting).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.z/3</MarginText>
<Admonition type="aarm" aarm="note">
In the case that the top-level construct C is a call, these rules do not require checks for most <strong>in out</strong> parameters, as the rules about evaluation of calls prevent problems. Similarly, we do not need checks for short circuit operations or other operations with a defined order of evaluation. The rules about arbitrary order (see <a href="../AA-1/AA-1.1#Subclause_1.1.4">1.1.4</a>) allow evaluating parameters and writing parameters back in an arbitrary order, but not interleaving of evaluating parameters of one call with writing parameters back from another &mdash; that would not correspond to any allowed sequential order. <br /></Admonition>
</AnnotatedOnly>

<MarginText>6.21/3</MarginText>

<MarginText>9</MarginText>

<MarginText>10</MarginText>

<MarginText>13.a</MarginText>
<Admonition type="aarm" aarm="reason">

<li> <br /></li>
<MarginText>13.4/5</MarginText>

This is important so that a programmer can guarantee that all objects of a scalar type have a valid value with a carefully chosen Default_Value. <br /></Admonition>

<AnnotatedOnly>
<MarginText>13.d/3</MarginText>

In contrast, a type that does not have a Default_Value only need support the size of the subtype (since no values are passed in). <br /></Admonition>

<AnnotatedOnly>
<MarginText>13.e/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong> <br /></Admonition>

<ul>
<li>For a composite type with discriminants or that has implicit initial values for any subcomponents (see <a href="../AA-3/AA-3.3#Subclause_3.3.1">3.3.1</a>), the behavior is as for an <strong>in out</strong> parameter passed by copy[, except that no predicate check is performed]. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>14.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This ensures that no part of an object of such a type can become &ldquo;de-initialized&rdquo; by being part of an <strong>out</strong> parameter. <br /></Admonition>

<AnnotatedOnly>
<MarginText>14.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This includes an array type whose component type is an access type, and a record type with a component that has a <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code>, among other things. <br /></Admonition>

<AnnotatedOnly>
<MarginText>14.c/5</MarginText>
<MarginInfo items={["AI12-0333-1"]} />
<Admonition type="aarm" aarm="proof">
<strong></strong>No predicate check follows from the definition of subtype conversion in <a href="../AA-4/AA-4.6">4.6</a>. <br /></Admonition>

<MarginText>15/5</MarginText>
<MarginInfo items={["AI12-0439-1"]} />
<ul>
<li>For any other type, the formal parameter is uninitialized. If composite, a view conversion of the actual parameter to the nominal subtype of the formal is evaluated [(which can  raise Constraint_Error)], and the actual subtype of the formal is that of the view conversion. If elementary, the actual subtype of the formal is given by its nominal subtype. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>15.a/3</MarginText>
<MarginInfo items={["AI05-0228-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>This case covers scalar types that do not have Default_Value specified, and composite types whose subcomponent's subtypes do not have any implicit initial values. The view conversion for composite types ensures that if the lengths don't match between an actual and a formal array parameter, the Constraint_Error is raised before the call, rather than after. <br /></Admonition>
</AnnotatedOnly>

<MarginText>15.1/5</MarginText>
<MarginInfo items={["AI12-0377-1"]} />
<ul>
<li>Furthermore, if the type is a scalar type, and the actual parameter is a view conversion, then Program_Error is raised if either the target or the operand type has the Default_Value aspect specified, unless they both have the Default_Value aspect specified, and there is a type (other than a root numeric type) that is an ancestor of both the target type and the operand type.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>15.b/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This can only occur in the body of an instance of a generic unit. Legality Rules will catch all other cases. Implementations that macro-expand generics can always detect this case when the enclosing instance body is expanded. <br /></Admonition>
</AnnotatedOnly>

<MarginText>15.2/5</MarginText>
<MarginInfo items={["AI05-0142-4", "AI05-0234-1"]} />
<ul>
<li>In a function call, for each explicitly aliased parameter, a check is made that the accessibility level of the master of the actual object is not deeper than that of the master of the call (see <a href="../AA-3/AA-3.10#Subclause_3.10.2">3.10.2</a>). <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>15.c/3</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>If the actual object to a call <em>C</em> is a formal parameter of some function call <em>F</em>, no dynamic check against the master of the actual parameter of <em>F</em> is necessary. Any case which could fail the dynamic check is already statically illegal (either at the call site of <em>F</em>, or at the call site <em>C</em>). This is important, as it would require nasty distributed overhead to accurately know the dynamic accessibility of a formal parameter (all tagged and explicitly aliased parameters would have to carry accessibility levels). <br /></Admonition>
</AnnotatedOnly>

<MarginText>16</MarginText>
<p>A formal parameter of mode <strong>in out</strong> or <strong>out</strong> with discriminants is constrained if either its nominal subtype or the actual parameter is constrained.<br /></p>

<MarginText>17</MarginText>
<p>After normal completion and leaving of a subprogram, for each <strong>in out</strong> or <strong>out</strong> parameter that is passed by copy, the value of the formal parameter is converted to the subtype of the variable given as the actual parameter and assigned to it. These conversions and assignments occur in an arbitrary order. <br /></p>

<AnnotatedOnly>
<MarginText>17.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The conversions mentioned above during parameter passing might raise Constraint_Error &mdash; (see <a href="../AA-4/AA-4.6">4.6</a>). <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>17.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>If any conversion or assignment as part of parameter passing propagates an exception, the exception is raised at the place of the subprogram call; that is, it cannot be handled inside the <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code>. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>17.c</MarginText>
<Admonition type="aarm" aarm="proof">
<strong></strong>Since these checks happen before or after executing the <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code>, the execution of the <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> does not dynamically enclose them, so it can't handle the exceptions. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>17.d</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The variable we're talking about is the one denoted by the <em>variable_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> given as the <code><a href="../AA-6/AA-6.4#S0221">explicit_actual_parameter</a></code>. If this <em>variable_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> is a <code><a href="../AA-4/AA-4.6#S0162">type_conversion</a></code>, then the rules in <a href="../AA-4/AA-4.6">4.6</a> for assigning to a view conversion apply. That is, if X is of subtype S1, and the actual is S2(X), the above-mentioned conversion will convert to S2, and the one mentioned in <a href="../AA-4/AA-4.6">4.6</a> will convert to S1. <br /></Admonition>
</AnnotatedOnly>
