<MarginText>1/2</MarginText>
<MarginInfo items={["AI95-00348-01"]} />
<p>A <code><a href="/docs/arm/AA-6/AA-6.7#S0227">null_procedure_declaration</a></code> provides a shorthand to declare a procedure with an empty body. <br /></p>


#### Syntax

<MarginText>2/3</MarginText>
<MarginInfo items={["AI95-00348-01", "AI05-0183-1"]} />

<CodeBlock>
<code>null{"_"}procedure{"_"}declaration</code><a id="S0227"></a><code> ::= </code><br />   [<code><a href="/docs/arm/AA-8/AA-8.3#S0234">overriding_indicator</a></code>]<br />   <code><a href="/docs/arm/AA-6/AA-6.1#S0197">procedure_specification</a></code> <b>is</b> <b>null</b><br />       [<code><a href="/docs/arm/AA-13/AA-13.1#S0346">aspect_specification</a></code>];<br />

</CodeBlock>

#### Legality Rules

<MarginText>2.1/3</MarginText>
<MarginInfo items={["AI05-0177-1"]} />
<p>If a <code><a href="/docs/arm/AA-6/AA-6.7#S0227">null_procedure_declaration</a></code> is a completion, it shall be the completion of a <code><a href="/docs/arm/AA-6/AA-6.1#S0195">subprogram_declaration</a></code> or <code><a href="/docs/arm/AA-12/AA-12.1#S0311">generic_subprogram_declaration</a></code>. The profile of a <code><a href="/docs/arm/AA-6/AA-6.7#S0227">null_procedure_declaration</a></code> that completes a declaration shall conform fully to that of the declaration. <br /></p>


#### Static Semantics

<MarginText>3/5</MarginText>
<MarginInfo items={["AI95-00348-01", "AI05-0177-1", "AI05-0264-1", "AI12-0408-1"]} />
<p>A <code><a href="/docs/arm/AA-6/AA-6.7#S0227">null_procedure_declaration</a></code> that is not a completion declares a <i>null procedure</i>. A completion is not allowed for a <code><a href="/docs/arm/AA-6/AA-6.7#S0227">null_procedure_declaration</a></code>; however, a <code><a href="/docs/arm/AA-6/AA-6.7#S0227">null_procedure_declaration</a></code> can complete a previous declaration. <br /></p>

<AnnotatedOnly>
<MarginText>3.a/2</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>There are no null functions because the return value has to be constructed somehow; a function that always raises Program{"_"}Error doesn't seem very useful or worth the complication. <br />
</Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>4/5</MarginText>
<MarginInfo items={["AI95-00348-01", "AI12-0408-1"]} />
<p>The execution of a null procedure is invoked by a subprogram call. For the execution of a subprogram call on a null procedure, or on a procedure completed with a <code><a href="/docs/arm/AA-6/AA-6.7#S0227">null_procedure_declaration</a></code>, the execution of the <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code> has no effect. <br /></p>

<AnnotatedOnly>
<MarginText>4.a/2</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Thus, a null procedure is equivalent to the body <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.b/2</MarginText>
<CodeBlock language="ada">
begin{"\n"}   null;{"\n"}end;{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>4.c/2</MarginText>
<Admonition type="aarm" aarm="note">
with the exception that a null procedure can be used in place of a procedure specification. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>5/3</MarginText>
<MarginInfo items={["AI95-00348-01", "AI05-0177-1"]} />
<p>The elaboration of a <code><a href="/docs/arm/AA-6/AA-6.7#S0227">null_procedure_declaration</a></code> has no other effect than to establish that the null procedure can be called without failing the Elaboration{"_"}Check. <br /></p>


#### Examples

<MarginText>6/5</MarginText>
<MarginInfo items={["AI12-0429-1"]} />
<p><i>Example of the declaration of a null procedure:</i> <br /></p>

<MarginText>7/5</MarginText>
<MarginInfo items={["AI95-00433-01", "AI12-0440-1"]} />
<CodeBlock language="ada">
procedure Simplify(Expr : in out Expression) is null; -- see <a href="/docs/arm/AA-3/AA-3.9">3.9</a>{"\n"}-- By default, Simplify does nothing, but it can be overridden in extensions of Expression{"\n"}

</CodeBlock>
