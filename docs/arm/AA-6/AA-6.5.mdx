<MarginText>1/2</MarginText>
<MarginInfo items={["AI95-00318-02"]} />
<p>A <code><a href="../AA-6/AA-6.5#S0222">simple_return_statement</a></code> or <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code> (collectively called a <em>return statement</em>) is used to complete the execution of the innermost enclosing <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code>, <code><a href="../AA-9/AA-9.5#S0260">entry_body</a></code>, or <code><a href="../AA-9/AA-9.5#S0258">accept_statement</a></code>. <br /></p>

#### Name Resolution Rules

<MarginText>3/5</MarginText>
<MarginInfo items={["AI95-00318-02", "AI12-0173-1"]} />
<p>The <em>result subtype</em> of a function is the subtype denoted by the <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>, or defined by the <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>, after the reserved word <strong>return</strong> in the profile of the function. The expected type for the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, if any, of a <code><a href="../AA-6/AA-6.5#S0222">simple_return_statement</a></code> is the result type of the corresponding function.  The expected type for the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of an <code><a href="../AA-6/AA-6.5#S0224">extended_return_object_declaration</a></code>  is that of the <code><a href="../AA-6/AA-6.5#S0226">return_subtype_indication</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>3.a</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>The same applies to generic functions. <br /></Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>4/2</MarginText>
<MarginInfo items={["AI95-00318-02"]} />
<p>A return statement shall be within a callable construct, and it <em>applies to</em> the innermost callable construct or <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code> that contains it. A return statement shall not be within a body that is within the construct to which the return statement applies.<br /></p>

<AnnotatedOnly>
<MarginText>4.a/4</MarginText>
<MarginInfo items={["AI12-0089-1"]} />
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>The above also applies to generic subprograms, even though they are not callable constructs. (An instance of a generic subprogram is a callable construct, but not a generic subprogram itself.) <br /></Admonition>
</AnnotatedOnly>

<MarginText>5/5</MarginText>
<MarginInfo items={["AI95-00318-02", "AI05-0015-1", "AI12-0173-1"]} />
<p>A function body shall contain at least one return statement that applies to the function body, unless the function contains <code><a href="../AA-13/AA-13.8#S0357">code_statement</a></code>s. A <code><a href="../AA-6/AA-6.5#S0222">simple_return_statement</a></code> shall include an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> if and only if it applies to a function body. An <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code> shall apply to a function body. An <code><a href="../AA-6/AA-6.5#S0224">extended_return_object_declaration</a></code>  with the reserved word <strong>constant</strong> shall include an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>.<br /></p>

<AnnotatedOnly>
<MarginText>5.a/4</MarginText>
<MarginInfo items={["AI95-00318-02", "AI12-0022-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>The requirement that a function body has to have at least one return statement is a &ldquo;helpful&rdquo; restriction. There has been some interest in lifting this restriction, or allowing a raise statement to substitute for the return statement. However, there was enough interest in leaving it as is that we decided not to change it. Note that for Ada 2012, Corrigendum 1, a return statement whose expression is a <code><a href="../AA-11/AA-11.3#S0309">raise_expression</a></code> can be given in <em>any</em> function body (the <code><a href="../AA-11/AA-11.3#S0309">raise_expression</a></code> will match any type), so there is much less need to eliminate this rule. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b/2</MarginText>
<MarginInfo items={["AI95-00318-02"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>A return statement can apply to an <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code>, so a <code><a href="../AA-6/AA-6.5#S0222">simple_return_statement</a></code> without an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> can be given in one. However, neither <code><a href="../AA-6/AA-6.5#S0222">simple_return_statement</a></code> with an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> nor an <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code> can be given inside an <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code>, as they must apply (directly) to a function body.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b.1/4</MarginText>
<MarginInfo items={["AI12-0089-1"]} />
<Admonition type="aarm" aarm="note">
Since a "function body" includes a generic function body, this rule and all of the following Legality Rules apply to generic function bodies as well as non-generic function bodies. This is true even though a generic function is not a function. <br /></Admonition>
</AnnotatedOnly>

<MarginText>5.1/5</MarginText>
<MarginInfo items={["AI12-0173-1", "AI12-0418-1"]} />
<p>The <em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of an <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code></em> is the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> (if any) of the <code><a href="../AA-6/AA-6.5#S0224">extended_return_object_declaration</a></code> of the <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code>.<br /></p>

<AnnotatedOnly>
<MarginText>5.b.2/5</MarginText>
<MarginInfo items={["AI12-0173-1", "AI12-0418-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>The <em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a return statement</em> is either the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a <code><a href="../AA-6/AA-6.5#S0222">simple_return_statement</a></code> or the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of an <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code> as defined above. <br /></Admonition>
</AnnotatedOnly>

<MarginText>5.2/2</MarginText>
<MarginInfo items={["AI95-00318-02"]} />
<p>For an <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code> that applies to a function body:<br /></p>

<MarginText>5.3/5</MarginText>
<MarginInfo items={["AI95-00318-02", "AI05-0032-1", "AI05-0103-1"]} />
<ul>
<li>If the result subtype of the function is defined by a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>, the <code><a href="../AA-6/AA-6.5#S0226">return_subtype_indication</a></code> shall be a <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code>. The type of the <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code> shall be covered by the result type of the function. The subtype defined by the <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code> shall be statically compatible with the result subtype of the function; if the result type of the function is elementary, the two subtypes shall statically match. If the result subtype of the function is indefinite, then the subtype defined by the <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code> shall be a definite subtype, or there shall be an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>.<br /></li>
<MarginText>5.4/5</MarginText>
<MarginInfo items={["AI95-00318-02"]} />
<li>If the result subtype of the function is defined by an <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>, the <code><a href="../AA-6/AA-6.5#S0226">return_subtype_indication</a></code> shall be an <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>. The subtype defined by the <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code> shall statically match the result subtype of the function. [The accessibility level of this anonymous access subtype is that of the result subtype.] <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>5.b.3/4</MarginText>
<MarginInfo items={["AI12-0070-1"]} />
<Admonition type="aarm" aarm="proof">
<strong></strong>The accessibility of such anonymous access types is defined in the Heart of Darkness (aka <a href="../AA-3/AA-3.10#Subclause_3.10.2">3.10.2</a>). <br /></Admonition>
</AnnotatedOnly>

<MarginText>5.5/5</MarginText>
<MarginInfo items={["AI05-0032-1"]} />
<ul>
<li>If the result subtype of the function is class-wide, the accessibility level of the type of the subtype defined by the <code><a href="../AA-6/AA-6.5#S0226">return_subtype_indication</a></code> shall not be statically deeper than that of the master that elaborated the function body. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>5.b.4/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>In this case, the <code><a href="../AA-6/AA-6.5#S0226">return_subtype_indication</a></code> could be a specific type initialized by default; in that case there is no <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> to check. <br /></Admonition>
</AnnotatedOnly>

<MarginText>5.6/5</MarginText>
<MarginInfo items={["AI95-00318-02", "AI05-0032-1"]} />
<p>For any return statement that applies to a function body:<br /></p>

<MarginText>5.7/5</MarginText>
<MarginInfo items={["AI95-00318-02", "AI05-0188-1"]} />
<ul>
<li>[If the result subtype of the function is limited, then the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the return statement (if any) shall meet the restrictions described in <a href="../AA-7/AA-7.5">7.5</a>.] <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>5.c/3</MarginText>
<MarginInfo items={["AI05-0188-1"]} />
<Admonition type="aarm" aarm="note">
<em>This paragraph was deleted.</em><br /></Admonition>
</AnnotatedOnly>

<MarginText>5.8/5</MarginText>
<MarginInfo items={["AI95-00416-01", "AI05-0032-1", "AI05-0051-1"]} />
<ul>
<li>If the result subtype of the function is class-wide, the accessibility level of the type of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> (if any) of the return statement shall not be statically deeper than that of the master that elaborated the function body.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>5.d/4</MarginText>
<MarginInfo items={["AI05-0032-1", "AI05-0051-1", "AI12-0005-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>If the result type is class-wide, then there must be an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the return statement unless this is an <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code> whose <code><a href="../AA-6/AA-6.5#S0226">return_subtype_indication</a></code> is a specific type. We have a separate rule to cover that case. Note that if an <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code> has an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, then both this rule and the next one must be satisfied. <br /></Admonition>
</AnnotatedOnly>

<MarginText>5.9/5</MarginText>
<MarginInfo items={["AI05-0051-1"]} />
<ul>
<li>If the subtype determined by the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the <code><a href="../AA-6/AA-6.5#S0222">simple_return_statement</a></code> or by the <code><a href="../AA-6/AA-6.5#S0226">return_subtype_indication</a></code> has one or more access discriminants, the accessibility level of the anonymous access type of each access discriminant shall not be statically deeper than that of the master that elaborated the function body.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>5.d.1/3</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>We use the type used by the return statement rather than from the function return type since we want to check whenever the return object has access discriminants, even if the function return type doesn't have any (mostly for a class-wide type). <br /></Admonition>
</AnnotatedOnly>

<MarginText>5.10/5</MarginText>
<MarginInfo items={["AI05-0277-1", "AI12-0426-1"]} />
<p>If the reserved word  <strong>aliased</strong> is present in an <code><a href="../AA-6/AA-6.5#S0224">extended_return_object_declaration</a></code>, the type of the extended return object shall be immutably limited. <br /></p>


#### Static Semantics

<MarginText>5.11/5</MarginText>
<MarginInfo items={["AI95-00318-02", "AI05-0015-1", "AI05-0144-2"]} />
<p>Within an <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code>, the <em>return object</em> is declared with the given <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code>, with the nominal subtype defined by the <code><a href="../AA-6/AA-6.5#S0226">return_subtype_indication</a></code>. An <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code> with the reserved word <strong>constant</strong> is a full constant declaration that declares the return object to be a constant object. <br /></p>


#### Dynamic Semantics

<MarginText>5.12/5</MarginText>
<MarginInfo items={["AI95-00318-02", "AI95-00416-01", "AI05-0032-1", "AI12-0439-1"]} />
<p>For the execution of an <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code>, the <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code> or <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code> is elaborated. This creates the nominal subtype of the return object. If there is an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, it is evaluated and converted to the nominal subtype (which can  raise Constraint_Error &mdash; see <a href="../AA-4/AA-4.6">4.6</a>); the return object is created and the converted value is assigned to the return object. Otherwise, the return object is created and initialized by default as for a stand-alone object of its nominal subtype (see <a href="../AA-3/AA-3.3#Subclause_3.3.1">3.3.1</a>). If the nominal subtype is indefinite, the return object is constrained by its initial value. A check is made that the value of the return object belongs to the function result subtype. Constraint_Error is raised if this check fails. <br /></p>

<AnnotatedOnly>
<MarginText>5.e/2</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>If the result type is controlled or has a controlled part, appropriate calls on Initialize or Adjust are performed prior to executing the <code><a href="../AA-11/AA-11.2#S0304">handled_sequence_of_statements</a></code>, except when the initial expression is an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> (which requires build-in-place with no call on Adjust).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.f/3</MarginText>
<MarginInfo items={["AI05-0005-1"]} />
<Admonition type="aarm" aarm="note">
If the return statement is left without resulting in a return (for example, due to an exception propagated from the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> or the <code><a href="../AA-11/AA-11.2#S0304">handled_sequence_of_statements</a></code>, or a goto out of the <code><a href="../AA-11/AA-11.2#S0304">handled_sequence_of_statements</a></code>), if the return object has been created, it is finalized prior to leaving the return statement. If it has not been created when the return statement is left, it is not created or finalized.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.g/3</MarginText>
<MarginInfo items={["AI05-0032-1"]} />
<Admonition type="aarm" aarm="note">
Other rules ensure that the check required by this rule cannot fail unless the function has a class-wide result subtype where the associated specific subtype is constrained. In other cases, either the subtypes have to match or the function's subtype is unconstrained and needs no checking. <br /></Admonition>
</AnnotatedOnly>

<MarginText>6/2</MarginText>
<MarginInfo items={["AI95-00318-02"]} />
<p>For the execution of a <code><a href="../AA-6/AA-6.5#S0222">simple_return_statement</a></code>, the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> (if any) is first evaluated, converted to the result subtype, and then is assigned to the anonymous <em>return object</em>. <br /></p>

<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The conversion might raise Constraint_Error &mdash; (see <a href="../AA-4/AA-4.6">4.6</a>). <br /></Admonition>
</AnnotatedOnly>

<MarginText>7/2</MarginText>
<MarginInfo items={["AI95-00318-02", "AI95-00416-01"]} />
<p>[If the return object has any parts that are tasks, the activation of those tasks does not occur until after the function returns (see <a href="../AA-9/AA-9.2">9.2</a>).] <br /></p>

<AnnotatedOnly>
<MarginText>7.a/2</MarginText>
<Admonition type="aarm" aarm="proof">
<strong></strong>This is specified by the rules in <a href="../AA-9/AA-9.2">9.2</a>. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.b/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>Only the caller can know when task activations should take place, as it depends on the context of the call. If the function is being used to initialize the component of some larger object, then that entire object must be initialized before any task activations. Even after the outer object is fully initialized, task activations are still postponed until the <strong>begin</strong> at the end of the declarative part if the function is being used to initialize part of a declared object. <br /></Admonition>
</AnnotatedOnly>

<MarginText>8/5</MarginText>
<MarginInfo items={["AI95-00318-02", "AI95-00344-01", "AI05-0024-1", "AI05-0032-1", "AI12-0097-1", "AI12-0418-1"]} />
<p>If the result type of a function is a specific tagged type, the tag of the return object is that of the result type. If the result type is class-wide, the tag of the return object is that of the value of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the return statement, unless the return object is defined by an <code><a href="../AA-6/AA-6.5#S0224">extended_return_object_declaration</a></code> with a <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code> that is specific, in which case it is that of the type of the <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code>. A check is made that the master of the type identified by the tag of the result includes the elaboration of the master that elaborated the function body. If this check fails, Program_Error is raised. <br /></p>

<AnnotatedOnly>
<MarginText>8.a/2</MarginText>
<MarginInfo items={["AI95-00318-02"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>The first sentence is true even if the tag of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is different, which could happen if the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> were a view conversion or a dereference of an access value. Note that for a limited type, because of the restriction to <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>s and function calls (and no conversions), the tag will already match. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.b/2</MarginText>
<MarginInfo items={["AI95-00318-02"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>The first rule ensures that a function whose result type is a specific tagged type always returns an object whose tag is that of the result type. This is important for dispatching on controlling result, and allows the caller to allocate the appropriate amount of space to hold the value being returned (assuming there are no discriminants).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.c/3</MarginText>
<Admonition type="aarm" aarm="note">
The master check prevents the returned object from outliving its type. Note that this check cannot fail for a specific tagged type, as the tag represents the function's type, which necessarily must be declared outside of the function.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.d/3</MarginText>
<Admonition type="aarm" aarm="note">
We can't use the normal accessibility level &ldquo;deeper than&rdquo; check here because we may have &ldquo;incomparable&rdquo; levels if the masters belong to two different tasks. #### Implementation Permissions

<MarginText>24/3</MarginText>
<MarginInfo items={["AI95-00416-01", "AI05-0050-1"]} />
<p>For a function call used to initialize a composite object with a constrained nominal subtype or used to initialize a return object that is built in place into such an object:<br /></p>

<MarginText>24.1/3</MarginText>
<MarginInfo items={["AI05-0050-1"]} />
<ul>
<li>If the result subtype of the function is constrained, and conversion of an object of this subtype to the subtype of the object being initialized would raise Constraint_Error, then Constraint_Error may be raised before calling the function.<br /></li>
<MarginText>24.2/3</MarginText>
<MarginInfo items={["AI05-0050-1"]} />
<li>If the result subtype of the function is unconstrained, and a return statement is executed such that the return object is known to be constrained, and conversion of the return object to the subtype of the object being initialized would raise Constraint_Error, then Constraint_Error may be raised at the point of the call (after abandoning the execution of the function body). <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>24.a/3</MarginText>
<MarginInfo items={["AI95-00416-01", "AI05-0050-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>Without such a permission, it would be very difficult to implement &ldquo;built-in-place&rdquo; semantics. The intention is that the exception is raised at the same point that it would have been raised without the permission; it should not change handlers if the implementation switches between return-by-copy and built-in-place. This means that the exception is not handleable within the function, because in the return-by-copy case, the constraint check to verify that the result satisfies the constraints of the object being initialized happens after the function returns. This implies further that upon detecting such a situation, the implementation may need to simulate a goto to a point outside any local exception handlers prior to raising the exception. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.b/3</MarginText>
<MarginInfo items={["AI95-00416-01", "AI05-0050-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>These permissions do not apply in the case of an extended return object with mutable discriminants. That's necessary because in that case a return object can be created with the &ldquo;wrong&rdquo; discriminants and then changed to the &ldquo;right&rdquo; discriminants later (but before returning). We don't want this case raising an exception when the canonical semantics will not do so.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.c/3</MarginText>
<MarginInfo items={["AI05-0050-1"]} />
<Admonition type="aarm" aarm="note">
It's still possible to write a program that will raise an exception using this permission that would not in the canonical semantics. That could happen if a return statement with the &ldquo;wrong&rdquo; discriminants or bounds is abandoned (via an exception, or for an extended_return_statement, via an exit or goto statement), and then a return statement with the &ldquo;right&rdquo; discriminants or bounds is executed. The only solution for this problem is to not have the permission at all, but this is too unusual of a case to worry about the effects of the permission, especially given the implementation difficulties for built-in-place objects that this permission is intended to ease.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.d/3</MarginText>
<MarginInfo items={["AI05-0050-1"]} />
<Admonition type="aarm" aarm="note">
Note that the mutable-discriminant case only happens when built-in-place initialization is optional. This means that any difficulties associated with implementing built-in-place initialization without these permissions can be sidestepped by not building in place.<br /></Admonition>
</AnnotatedOnly>

<a id="Subclause_6.5.1"></a>

## 6.5.1  Nonreturning Subprograms

<MarginText>1/5</MarginText>
<MarginInfo items={["AI95-00329-01", "AI95-00414-01", "AI05-0229-1", "AI12-0269-1", "AI12-0418-1"]} />
<p>Specifying aspect No_Return to have the value True indicates that a subprogram  cannot return normally[; it may, for example, propagate an exception or loop forever]. <br /></p>

<AnnotatedOnly>
<MarginText>1.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Aspect No_Deposit will have to wait for Ada 2028 . :-) <br /></Admonition>
</AnnotatedOnly>

<p><em>Paragraphs 2 and 3 were moved to <a href="../AA-J/">Annex J</a>, &ldquo;<a href="../AA-J/">Obsolescent Features</a>&rdquo;.</em> <br /></p>


#### Static Semantics

<MarginText>3.1/5</MarginText>
<MarginInfo items={["AI05-0229-1", "AI12-0269-1"]} />
<p>For a subprogram  or generic subprogram , the following language-defined representation aspect may be specified: <br /></p>

<MarginText>3.2/3</MarginText>
<dt><br/>No_Return</dt>
<dl>
<dd>The type of aspect No_Return is Boolean. When aspect No_Return is True for an entity, the entity is said to be <em>nonreturning</em>.<br /></dd>
<MarginText>3.3/5</MarginText>
<MarginInfo items={["AI12-0423-1"]} />
<dd>If directly specified, the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> shall be a static expression. When not directly specified, if the subprogram is a primitive subprogram inherited by a derived type, then the aspect is True if any corresponding subprogram of the parent or progenitor types is nonreturning. Otherwise , the aspect is False.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>3.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Aspect Description for </strong><strong>No_Return: </strong>A subprogram  will not return normally.<br /></Admonition>
</AnnotatedOnly>

<MarginText>3.4/5</MarginText>
<MarginInfo items={["AI05-0229-1", "AI12-0269-1"]} />
<p>If a generic subprogram  is nonreturning, then so are its instances. If a subprogram  declared within a generic unit is nonreturning, then so are the corresponding copies of that subprogram  in instances.<br /></p>


#### Legality Rules

<MarginText>4/3</MarginText>
<MarginInfo items={["AI95-00329-01", "AI95-00414-01", "AI05-0229-1"]} />
<p>Aspect No_Return shall not be specified for a null procedure nor an instance of a generic unit.<br /></p>

<AnnotatedOnly>
<MarginText>4.a/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>A null procedure cannot have the appropriate nonreturning semantics, as it does not raise an exception or loop forever. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.b/5</MarginText>
<MarginInfo items={["AI05-0229-1", "AI12-0269-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>The procedure can be abstract. If a nonreturning subprogram  is renamed (anywhere) calls through the new name still have the nonreturning semantics. <br /></Admonition>
</AnnotatedOnly>

<MarginText>5/2</MarginText>
<MarginInfo items={["AI95-00329-01", "AI95-00414-01"]} />
<p>A return statement shall not apply to a nonreturning procedure or generic procedure.<br /></p>

<MarginText>5.1/5</MarginText>
<MarginInfo items={["AI12-0269-1"]} />
<p>Any return statement that applies to a nonreturning function or generic function shall be a <code><a href="../AA-6/AA-6.5#S0222">simple_return_statement</a></code> with an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> that is a <code><a href="../AA-11/AA-11.3#S0309">raise_expression</a></code>, a call on a nonreturning function, or a parenthesized <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of one of these.<br /></p>

<AnnotatedOnly>
<MarginText>5.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>We still require at least one return statement in a function; we just require that all such return statements don't actually return a value. <br /></Admonition>
</AnnotatedOnly>

<MarginText>6/5</MarginText>
<MarginInfo items={["AI95-00414-01", "AI12-0269-1"]} />
<p>A subprogram  shall be nonreturning if it overrides a dispatching nonreturning subprogram . In addition to the places where Legality Rules normally apply (see <a href="../AA-12/AA-12.3">12.3</a>), this rule applies also in the private part of an instance of a generic unit.<br /></p>

<AnnotatedOnly>
<MarginText>6.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This ensures that dispatching calls to nonreturning subprograms  will, in fact, not return. <br /></Admonition>
</AnnotatedOnly>

<MarginText>7/5</MarginText>
<MarginInfo items={["AI95-00414-01", "AI12-0269-1"]} />
<p>If a renaming-as-body completes a nonreturning subprogram  declaration, then the renamed subprogram  shall be nonreturning.<br /></p>

<AnnotatedOnly>
<MarginText>7.a/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This ensures that no extra code is needed to implement the renames (that is, no wrapper is needed) as the body has the same property. <br /></Admonition>
</AnnotatedOnly>

<p><em>Paragraph 8 was deleted.</em> <br /></p>


#### Dynamic Semantics

<MarginText>9/2</MarginText>
<MarginInfo items={["AI95-00329-01", "AI95-00414-01"]} />
<p>If the body of a nonreturning procedure completes normally, Program_Error is raised at the point of the call. <br /></p>

<AnnotatedOnly>
<MarginText>9.a/5</MarginText>
<MarginInfo items={["AI12-0269-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>Note that there is no name for suppressing this check, since the check represents a bug, imposes no time overhead, and minimal space overhead (since it can usually be statically eliminated as dead code). We don't need a similar rule for nonreturning functions as this is standard semantics for all functions that normally complete without a transfer of control (such as a return statement). <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.b/2</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>If a nonreturning procedure tries to return, we raise Program_Error. This is stated as happening at the call site, because we do not wish to allow the procedure to handle the exception (and then, perhaps, try to return again!). However, the expected run-time model is that the compiler will generate <strong>raise</strong> Program_Error at the end of the procedure body (but not handleable by the procedure itself), as opposed to doing it at the call site. (This is just like the typical run-time model for functions that fall off the end without returning a value). The reason is indirect calls: in P.<strong>all</strong>(...);, the compiler cannot know whether P designates a nonreturning procedure or a normal one. Putting the <strong>raise</strong> Program_Error in the procedure's generated code solves this problem neatly.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.c/2</MarginText>
<Admonition type="aarm" aarm="note">
Similarly, if one passes a nonreturning procedure to a generic formal parameter, the compiler cannot know this at call sites (in shared code implementations); the raise-in-body solution deals with this neatly. <br /></Admonition>
</AnnotatedOnly>
