<MarginText>1</MarginText>
<p>[Objects are created at run time and contain a value of a given type. An object can be created and initialized as part of elaborating a declaration, evaluating an <code><a href="../AA-4/AA-4.8#S0164">allocator</a></code>, <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, or <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code>, or passing a parameter by copy. Prior to reclaiming the storage for an object, it is finalized if necessary (see <a href="../AA-7/AA-7.6#Subclause_7.6.1">7.6.1</a>).] <br /></p>


#### Static Semantics

<MarginText>14/3</MarginText>
<MarginInfo items={["AI05-0054-2"]} />
<p>An object is either a <em>constant</em> object or a <em>variable</em> object. Similarly, a view of an object is either a <em>constant</em> or a <em>variable</em>. All views of a constant elementary object are constant. All views of a constant composite object are constant, except for parts that are of controlled or immutably limited types; variable views of those parts and their subcomponents may exist. In this sense, objects of controlled and immutably limited types are <em>inherently mutable</em>. A constant view of an object cannot be used to modify its value. The terms constant and variable by themselves refer to constant and variable views of objects.<br /></p>

<MarginText>14.a/5</MarginText>
<MarginInfo items={["AI12-0422-1"]} />
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>If some part of an object has a variable view, then the object as a whole has a variable view, and not all views of the object are constant. That's true even if only a subcomponent has a variable view. <br /></Admonition>

<MarginText>14.1/5</MarginText>
<MarginInfo items={["AI12-0422-1"]} />
<p>A constant object is <em>known to have no variable views</em> if it does not have a part that is immutably limited, or of a controlled type, private type, or private extension.<br /></p>

<MarginText>14.b/5</MarginText>
<MarginInfo items={["AI12-0422-1"]} />
<Admonition type="aarm" aarm="reason" title="Reason: ">
<strong></strong>This definition can be used in Legality Rules as it respects privacy. It is an assume-the-worst rule, as all private types and private extensions might have a controlled component. <br /></Admonition>

<MarginText>15</MarginText>
<p>The value of an object is <em>read</em> when the value of any part of the object is evaluated, or when the value of an enclosing object is evaluated. The value of a variable is <em>updated</em> when an assignment is performed to any part of the variable, or when an assignment is performed to an enclosing object. <br /></p>

<MarginText>15.a</MarginText>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>Reading and updating are intended to include read/write references of any kind, even if they are not associated with the evaluation of a particular construct. Consider, for example, the expression &ldquo;X.<strong>all</strong>(F)&rdquo;, where X is an access-to-array object, and F is a function. The implementation is allowed to first evaluate &ldquo;X.<strong>all</strong>&rdquo; and then F. Finally, a read is performed to get the value of the F'th component of the array. Note that the array is not necessarily read as part of the evaluation of &ldquo;X.<strong>all</strong>&rdquo;. This is important, because if F were to free X using Unchecked_Deallocation, we want the execution of the final read to be erroneous. <br /></Admonition>

<MarginInfo items={["AI05-0264-1", "AI12-0191-1", "AI12-0294-1"]} />
<p>At the place where a view of an object is defined, a <em>nominal subtype</em> is associated with the view. The <em>nominal type</em> of a view is the type of the nominal subtype of the view. The object's <em>actual subtype</em> (that is, its subtype) can be more restrictive than the nominal subtype of the view; it always is more restrictive if the nominal subtype is an <em>indefinite subtype</em>. A subtype is an indefinite subtype if it is an unconstrained array subtype, or if it has unknown discriminants or unconstrained discriminants without defaults (see <a href="../AA-3/AA-3.7">3.7</a>); otherwise, the subtype is a <em>definite</em> subtype [(all elementary subtypes are definite subtypes)]. [A class-wide subtype is defined to have unknown discriminants, and is therefore an indefinite subtype. An indefinite subtype does not by itself provide enough information to create an object; an additional <code><a href="../AA-3/AA-3.2#S0029">constraint</a></code> or explicit initialization <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is necessary (see <a href="../AA-3/AA-3.3#Subclause_3.3.1">3.3.1</a>). A component cannot have an indefinite nominal subtype.]<br /></p>

<MarginText>24.a.1/5</MarginText>
<Admonition type="aarm" aarm="glossary-entry" title="Glossary entry: ">
<strong></strong>The nominal subtype of a view of an object is the subtype specified when the view is defined.<br /></Admonition>

<MarginText>25.1/3</MarginText>
<MarginInfo items={["AI05-0008-1"]} />
<p>A view of a composite object is <em>known to be constrained</em> if:<br /></p>

<MarginText>25.2/5</MarginText>
<MarginInfo items={["AI12-0401-1"]} />
<ul>
<li>its nominal subtype is constrained and  not an untagged partial view, and it is neither a value conversion nor a <code><a href="../AA-4/AA-4.7#S0163">qualified_expression</a></code>; or<br /></li>
<MarginText>25.3/3</MarginText>
<li>its nominal subtype is indefinite; or<br /></li>
<MarginText>25.4/3</MarginText>
<MarginInfo items={["AI05-0008-1", "AI05-0093-1"]} />
<li>its type is immutably limited (see <a href="../AA-7/AA-7.5">7.5</a>); or<br /></li>
<MarginText>25.5/3</MarginText>
<li>it is part of a stand-alone constant (including a generic formal object of mode <strong>in</strong>); or<br /></li>
<MarginText>25.6/3</MarginText>
<li>it is part of a formal parameter of mode <strong>in</strong>; or<br /></li>
<MarginText>25.7/3</MarginText>
<li>it is part of the object denoted by a <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code> or <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>; or<br /></li>
<MarginText>25.8/5</MarginText>
<MarginInfo items={["AI12-0226-1", "AI12-0228-1"]} />
<li>it is a value conversion or <code><a href="../AA-4/AA-4.7#S0163">qualified_expression</a></code> where the operand denotes a view of a composite object that is known to be constrained; or<br /></li>
<MarginText>25.9/5</MarginText>
<MarginInfo items={["AI12-0228-1"]} />
<li>it is part of a constant return object of an <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code>; or<br /></li>
<MarginText>25.10/5</MarginText>
<MarginInfo items={["AI05-0008-1", "AI05-0041-1", "AI12-0228-1"]} />
<li>it is a dereference of a pool-specific access type, and there is no ancestor of its type that has a constrained partial view.<br /></li>
</ul>
<MarginText>25.a/3</MarginText>
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<strong></strong>We do not include dereferences of general access types because they might denote stand-alone aliased unconstrained variables. That's true even for access-to-constant types (the denoted object does not have to be a constant).<br /></Admonition>

<MarginText>25.b/5</MarginText>
<MarginInfo items={["AI05-0005-1", "AI05-0008-1", "AI12-0228-1"]} />
<Admonition type="aarm" aarm="note" title="Note: ">
We don't mention view conversions as there are no mutable tagged types (discriminant defaults are allowed only if the type is immutably limited), so all tagged view conversions are either of an indefinite type (if it has discriminants without defaults), an immutably limited type (if the discriminants do have defaults), or constrained (if there are no discriminants). This matches the first three bullets here, so all tagged view conversions are known to be constrained without needing to mention them explicitly. Untagged view conversions only can occur in parameter passing (as actuals to <strong>in out</strong> or <strong>out</strong> parameters), and "known to be constrained" is not used there. <br /></Admonition>

<MarginText>25.c/5</MarginText>
<MarginInfo items={["AI12-0228-1"]} />
<Admonition type="aarm" aarm="note" title="Note: ">
We don't need to mention the current instance of a (sub)type, either. If a current instance of a type or subtype appears in an aspect specification, it represents a value, so whether or not it is known to be constrained is irrelevant (the term is only defined for composite objects). Otherwise, the current instance of a type can only be used in an immutably limited type, so all such instances are known to be constrained by the third bullet. <br /></Admonition>

<MarginText>25.11/5</MarginText>
<MarginInfo items={["AI05-0008-1", "AI05-0041-1", "AI12-0228-1"]} />
<p>For the purposes of determining within a generic body whether an object is known to be constrained: <br /></p>

<MarginText>25.12/5</MarginText>
<ul>
<li>if a subtype is a descendant of an untagged generic formal private or derived type, and the subtype is not an unconstrained array subtype, it is not considered indefinite and is considered to have a constrained partial view;<br /></li>
<MarginText>25.13/5</MarginText>
<li>if a subtype is a descendant of a formal access type, it is not considered pool-specific.<br /></li>
</ul>

#### Extensions to Ada 83

<MarginText>28.a</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
There are additional kinds of objects (choice parameters and entry indices of entry bodies).<br /></Admonition>

<MarginText>28.b</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
The result of a function and of evaluating an aggregate are considered (constant) objects. This is necessary to explain the action of finalization on such things. Because a <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code> is also syntactically a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> (see <a href="../AA-4/AA-4.1">4.1</a>), the result of a <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code> can be renamed, thereby allowing repeated use of the result without calling the function again. <br /></Admonition>


#### Wording Changes from Ada 83

<MarginText>28.c/3</MarginText>
<MarginInfo items={["AI05-0299-1"]} />
<Admonition type="aarm" aarm="note" title="Note: ">
This subclause now follows the subclauses on types and subtypes, to cut down on the number of forward references.<br /></Admonition>

<MarginText>28.d</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
The term nominal subtype is new. It is used to distinguish what is known at compile time about an object's constraint, versus what its "true" run-time constraint is.<br /></Admonition>

<MarginText>28.e</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
The terms definite and indefinite (which apply to subtypes) are new. They are used to aid in the description of generic formal type matching, and to specify when an explicit initial value is required in an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code>.<br /></Admonition>

<MarginText>28.f</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
We have moved the syntax for <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> and <code><a href="../AA-3/AA-3.3#S0034">number_declaration</a></code> down into their respective subclauses, to keep the syntax close to the description of the associated semantics.<br /></Admonition>

<MarginText>28.g</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
We talk about variables and constants here, since the discussion is not specific to <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code>s, and it seems better to have the list of the kinds of constants juxtaposed with the kinds of objects.<br /></Admonition>

<MarginText>28.h</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
We no longer talk about indirect updating due to parameter passing. Parameter passing is handled in 6.2 and 6.4.1 in a way that there is no need to mention it here in the definition of read and update. Reading and updating now includes the case of evaluating or assigning to an enclosing object. <br /></Admonition>

<a id="Subclause_3.3.1"></a>

## 3.3.1  Object Declarations

<MarginText>1/3</MarginText>
<MarginInfo items={["AI05-0262-1"]} />
<p>An <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> declares a <em>stand-alone</em> object with a given nominal subtype and, optionally, an explicit initial value given by an initialization expression. For an array, access, task, or protected object, the <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> may include the definition of the (anonymous) type of the object. <br /></p>


#### Syntax

<MarginText>2/3</MarginText>
<MarginInfo items={["AI95-00385-01", "AI95-00406-01", "AI05-0183-1"]} />

<CodeBlock>
<code>object_declaration</code><a id="S0032"></a><code> ::= </code><br />    <code><a href="../AA-3/AA-3.3#S0033">defining_identifier_list</a></code> : [<strong>aliased</strong>] [<strong>constant</strong>] <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code> [:= <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>]<br />        [<code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>];<br />  | <code><a href="../AA-3/AA-3.3#S0033">defining_identifier_list</a></code> : [<strong>aliased</strong>] [<strong>constant</strong>] <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code> [:= <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>]<br />        [<code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>];<br />  | <code><a href="../AA-3/AA-3.3#S0033">defining_identifier_list</a></code> : [<strong>aliased</strong>] [<strong>constant</strong>] <code><a href="../AA-3/AA-3.6#S0051">array_type_definition</a></code> [:= <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>]<br />        [<code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>];<br />  | <code><a href="../AA-9/AA-9.1#S0245">single_task_declaration</a></code><br />  | <code><a href="../AA-9/AA-9.4#S0250">single_protected_declaration</a></code><br />

</CodeBlock>
<MarginText>3</MarginText>

<CodeBlock>
<code>defining_identifier_list</code><a id="S0033"></a><code> ::= </code><br />  <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code> {"{"}, <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code>{"}"}<br />

</CodeBlock>

#### Name Resolution Rules

<MarginText>4</MarginText>
<p>For an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> with an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> following the compound delimiter :=, the type expected for the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is that of the object. This <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is called the <em>initialization expression</em>. <br /></p>


#### Legality Rules

<MarginText>5/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />
<p>An <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> without the reserved word <strong>constant</strong> declares a variable object. If it has a <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code> or an <code><a href="../AA-3/AA-3.6#S0051">array_type_definition</a></code> that defines an indefinite subtype, then there shall be an initialization expression. <br /></p>


#### Static Semantics

<MarginText>6/3</MarginText>
<MarginInfo items={["AI05-0264-1", "AI05-0299-1"]} />
<p>An <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> with the reserved word <strong>constant</strong> declares a constant object. If it has an initialization expression, then it is called a <em>full constant declaration</em>. Otherwise, it is called a <em>deferred constant declaration</em>. The rules for deferred constant declarations are given in subclause <a href="../AA-7/AA-7.4">7.4</a>. The rules for full constant declarations are given in this subclause.<br /></p>

<MarginText>7</MarginText>
<p>Any declaration that includes a <code><a href="../AA-3/AA-3.3#S0033">defining_identifier_list</a></code> with more than one <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code> is equivalent to a series of declarations each containing one <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code> from the list, with the rest of the text of the declaration copied for each declaration in the series, in the same order as the list. The remainder of this Reference Manual relies on this equivalence; explanations are given for declarations with a single <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code>.<br /></p>

<MarginText>8/2</MarginText>
<MarginInfo items={["AI95-00385-01"]} />
<p>The <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code>, <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>, or full type definition of an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> defines the nominal subtype of the object. The <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> declares an object of the type of the nominal subtype. <br /></p>

<MarginText>8.a/2</MarginText>
<MarginInfo items={["AI95-00385-01"]} />
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<strong></strong>The phrase &ldquo;full type definition&rdquo; here includes the case of an anonymous array, access, task, or protected type. <br /></Admonition>

<MarginText>8.1/5</MarginText>
<MarginInfo items={["AI95-00373-01", "AI12-0192-1"]} />
<p>A component of an object is said to <em>require late initialization</em> if:  <br /></p>

<MarginText>8.2/5</MarginText>
<MarginInfo items={["AI12-0192-1", "AI12-0404-1"]} />
<ul>
<li>it has an access discriminant value constrained by a per-object expression; or<br /></li>
<MarginText>8.3/5</MarginText>
<MarginInfo items={["AI12-0192-1"]} />
<li>it has an initialization expression that includes a name denoting an access discriminant; or<br /></li>
<MarginText>8.4/5</MarginText>
<MarginInfo items={["AI12-0192-1"]} />
<li>it has an initialization expression that includes a reference to the current instance of the type either by name or implicitly as the target object of a call. <br /></li>
</ul>
<MarginText>8.b/2</MarginText>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<strong></strong>Such components can depend on the values of other components of the object. We want to initialize them as late and as reproducibly as possible. <br /></Admonition>


#### Dynamic Semantics

<MarginText>9/2</MarginText>
<MarginInfo items={["AI95-00363-01"]} />
<p>If a composite object declared by an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> has an unconstrained nominal subtype, then if this subtype is indefinite or the object is constant the actual subtype of this object is constrained. The constraint is determined by the bounds or discriminants (if any) of its initial value; the object is said to be <em>constrained by its initial value</em>. When not constrained by its initial value, the actual and nominal subtypes of the object are the same. If its actual subtype is constrained, the object is called a <em>constrained object</em>.<br /></p>

<MarginText>10</MarginText>
<p>For an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> without an initialization expression, any initial values for the object or its subcomponents are determined by the <em>implicit initial values</em> defined for its nominal subtype, as follows: <br /></p>

<MarginText>11</MarginText>
<ul>
<li>The implicit initial value for an access subtype is the null value of the access type.<br /></li>
<MarginText>11.1/5</MarginText>
<MarginInfo items={["AI05-0228-1", "AI12-0439-1"]} />
<li>The implicit initial value for a scalar subtype that has the Default_Value aspect specified is the value of that aspect converted to the nominal subtype (which can  raise Constraint_Error &mdash; see <a href="../AA-4/AA-4.6">4.6</a>, &ldquo;<a href="../AA-4/AA-4.6">Type Conversions</a>&rdquo;);<br /></li>
</ul>
<MarginText>11.a.1/3</MarginText>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>This is a Dynamic Semantics rule, so the visibility of the <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> is not relevant &mdash; if the full type for a private type has the Default_Value aspect specified, partial views of the type also have this implicit initial value. <br /></Admonition>

<MarginText>12</MarginText>
<ul>
<li>The implicit initial (and only) value for each discriminant of a constrained discriminated subtype is defined by the subtype.<br /></li>
<MarginText>13/5</MarginText>
<MarginInfo items={["AI05-0228-1", "AI12-0439-1"]} />
<li>For a (definite) composite subtype, the implicit initial value of each component with a <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code> is obtained by evaluation of this expression and conversion to the component's nominal subtype (which can  raise Constraint_Error), unless the component is a discriminant of a constrained subtype (the previous case), or is in an excluded <code><a href="../AA-3/AA-3.8#S0072">variant</a></code> (see <a href="../AA-3/AA-3.8#Subclause_3.8.1">3.8.1</a>). For each component that does not have a <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code>, if the composite subtype has the Default_Component_Value aspect specified, the implicit initial value is the value of that aspect converted to the component's nominal subtype; otherwise, any implicit initial values are those determined by the component's nominal subtype.<br /></li>
<MarginText>14</MarginText>
<li>For a protected or task subtype, there is an implicit component (an entry queue) corresponding to each entry, with its implicit initial value being an empty queue. <br /></li>
</ul>
<MarginText>14.a</MarginText>
<Admonition type="aarm" aarm="implementation-note" title="Implementation Note: ">
<strong></strong>The implementation may add implicit components for its own use, which might have implicit initial values. For a task subtype, such components might represent the state of the associated thread of control. For a type with dynamic-sized components, such implicit components might be used to hold the offset to some explicit component. <br /></Admonition>

<MarginText>15</MarginText>
<p>The elaboration of an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> proceeds in the following sequence of steps: <br /></p>

<MarginText>16/2</MarginText>
<dt><br/>a)</dt>
<MarginInfo items={["AI95-00385-01"]} />
<dl>
<dd>The <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code>, <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>, <code><a href="../AA-3/AA-3.6#S0051">array_type_definition</a></code>, <code><a href="../AA-9/AA-9.1#S0245">single_task_declaration</a></code>, or <code><a href="../AA-9/AA-9.4#S0250">single_protected_declaration</a></code> is first elaborated. This creates the nominal subtype (and the anonymous type in the last four cases).<br /></dd>
<MarginText>17/5</MarginText>
<dt><br/>b)</dt>
<MarginInfo items={["AI12-0439-1"]} />
<dd>If the <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> includes an initialization expression, the (explicit) initial value is obtained by evaluating the expression and converting it to the nominal subtype (which can  raise Constraint_Error &mdash; see <a href="../AA-4/AA-4.6">4.6</a>). <br /></dd>
<MarginText>18/2</MarginText>
<dt><br/>c)</dt>
<MarginInfo items={["AI95-00171-01", "AI95-00373-01"]} />
<dd>{"{"}<em>8652/0002</em>{"}"} The object is created, and, if there is not an initialization expression, the object is <em>initialized by default</em>. When an object is initialized by default, any per-object constraints (see <a href="../AA-3/AA-3.8">3.8</a>) are elaborated and any implicit initial values for the object or for its subcomponents are obtained as determined by the nominal subtype. Any initial values (whether explicit or implicit) are assigned to the object or to the corresponding subcomponents. As described in <a href="../AA-5/AA-5.2">5.2</a> and <a href="../AA-7/AA-7.6">7.6</a>, Initialize and Adjust procedures can be called. <br /></dd>
</dl>
<MarginText>18.a</MarginText>
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<strong></strong>For a per-object constraint that contains some per-object expressions and some non-per-object expressions, the values used for the constraint consist of the values of the non-per-object expressions evaluated at the point of the <code><a href="../AA-3/AA-3.2#S0023">type_declaration</a></code>, and the values of the per-object expressions evaluated at the point of the creation of the object.<br /></Admonition>

<MarginText>18.b</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
The elaboration of per-object constraints was presumably performed as part of the dependent compatibility check in Ada 83. If the object is of a limited type with an access discriminant, the <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code> is elaborated at this time (see <a href="../AA-3/AA-3.7">3.7</a>). <br /></Admonition>

<MarginText>18.c</MarginText>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<strong></strong>The reason we say that evaluating an explicit initialization expression happens before creating the object is that in some cases it is impossible to know the size of the object being created until its initial value is known, as in &ldquo;X: String := Func_Call(...);&rdquo;. The implementation can create the object early in the common case where the size can be known early, since this optimization is semantically neutral. <br /></Admonition>

<MarginText>19/2</MarginText>
<MarginInfo items={["AI95-00373-01"]} />
<dl>
<dd><em>This paragraph was deleted.</em><br /></dd>
</dl>
<MarginText>19.a</MarginText>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>Since the initial values have already been converted to the appropriate nominal subtype, the only Constraint_Errors that might occur as part of these assignments are for values outside their base range that are used to initialize unconstrained numeric subcomponents. See <a href="../AA-3/AA-3.5">3.5</a>. <br /></Admonition>

<MarginText>20/2</MarginText>
<MarginInfo items={["AI95-00373-01"]} />
<p>For the third step above, evaluations and assignments are performed in an arbitrary order subject to the following restrictions: <br /></p>

<MarginText>20.1/2</MarginText>
<MarginInfo items={["AI95-00373-01"]} />
<ul>
<li>Assignment to any part of the object is preceded by the evaluation of the value that is to be assigned. <br /></li>
</ul>
<MarginText>20.a.1/2</MarginText>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<strong></strong>Duh. But we ought to say it. Note that, like any rule in the Reference Manual, it doesn't prevent an &ldquo;as-if&rdquo; optimization; as long as the semantics as observed from the program are correct, the compiler can generate any code it wants. <br /></Admonition>

<MarginText>20.2/2</MarginText>
<MarginInfo items={["AI95-00373-01"]} />
<ul>
<li>The evaluation of a <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code> that includes the name of a discriminant is preceded by the assignment to that discriminant. <br /></li>
</ul>
<MarginText>20.a.2/2</MarginText>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<strong></strong>Duh again. But we have to say this, too. It's odd that Ada 95 only required the default expressions to be evaluated before the discriminant is used; it says nothing about discriminant values that come from <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code>s. <br /></Admonition>

<MarginText>20.3/2</MarginText>
<MarginInfo items={["AI95-00373-01"]} />
<ul>
<li>The evaluation of the <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code> for any component that depends on a discriminant is preceded by the assignment to that discriminant. <br /></li>
</ul>
<MarginText>20.a</MarginText>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<strong></strong>For example: <br /></Admonition>

<MarginText>20.b</MarginText>
<CodeBlock language="ada">
type R(D : Integer := F) is{"\n"}    record{"\n"}        S : String(1..D) := (others ={">"} G);{"\n"}    end record;{"\n"}
<MarginText>20.c</MarginText>
X : R;{"\n"}

</CodeBlock>
<MarginText>20.d</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
For the elaboration of the declaration of X, it is important that F be evaluated before the aggregate. <br /></Admonition>

<MarginText>20.4/3</MarginText>
<MarginInfo items={["AI95-00373-01", "AI05-0092-1"]} />
<ul>
<li>The assignments to any components, including implicit components, not requiring late initialization precede the initial value evaluations for any components requiring late initialization; if two components both require late initialization, then assignments to parts of the component occurring earlier in the order of the component declarations precede the initial value evaluations of the component occurring later. <br /></li>
</ul>
<MarginText>20.e/2</MarginText>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<strong></strong>Components that require late initialization can refer to the entire object during their initialization. We want them to be initialized as late as possible to reduce the chance that their initialization depends on uninitialized components. For instance: <br /></Admonition>

<MarginText>20.f/2</MarginText>
<CodeBlock language="ada">
type T (D : Natural) is{"\n"}  limited record{"\n"}    C1 : T1 (T'Access);{"\n"}    C2 : Natural := F (D);{"\n"}    C3 : String (1 .. D) := (others ={">"} ' ');{"\n"}  end record;{"\n"}

</CodeBlock>
<MarginText>20.g/2</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
Component C1 requires late initialization. The initialization could depend on the values of any component of T, including D, C2, or C3. Therefore, we want to it to be initialized last. Note that C2 and C3 do not require late initialization; they only have to be initialized after D.<br /></Admonition>

<MarginText>20.h/2</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
It is possible for there to be more than one component that requires late initialization. In this case, the language can't prevent problems, because all of the components can't be the last one initialized. In this case, we specify the order of initialization for components requiring late initialization; by doing so, programmers can arrange their code to avoid accessing uninitialized components, and such arrangements are portable. Note that if the program accesses an uninitialized component, <a href="../AA-13/AA-13.9#Subclause_13.9.1">13.9.1</a> defines the execution to be erroneous. <br /></Admonition>

<MarginText>21/5</MarginText>
<MarginInfo items={["AI05-0228-1", "AI12-0439-1"]} />
<p>[There is no implicit initial value defined for a scalar subtype unless the Default_Value aspect has been specified for the type.] In the absence of an explicit initialization or the specification of the Default_Value aspect, a newly created scalar object can  have a value that does not belong to its subtype (see <a href="../AA-13/AA-13.9#Subclause_13.9.1">13.9.1</a> and <a href="../AA-H/AA-H.1">H.1</a>). <br /></p>

<MarginText>21.a</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
<strong>To be honest: </strong>It could even be represented by a bit pattern that doesn't actually represent any value of the type at all, such as an invalid internal code for an enumeration type, or a NaN for a floating point type. It is a generally a bounded error to reference scalar objects with such &ldquo;invalid representations&rdquo;, as explained in <a href="../AA-13/AA-13.9#Subclause_13.9.1">13.9.1</a>, &ldquo;<a href="../AA-13/AA-13.9#Subclause_13.9.1">Data Validity</a>&rdquo;. <br /></Admonition>

<MarginText>21.b</MarginText>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>There is no requirement that two objects of the same scalar subtype have the same implicit initial &ldquo;value&rdquo; (or representation). It might even be the case that two elaborations of the same <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> produce two different initial values. However, any particular uninitialized object is default-initialized to a single value (or invalid representation). Thus, multiple reads of such an uninitialized object will produce the same value each time (if the implementation chooses not to detect the error). <br /></Admonition>

<MarginText>22</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
NOTE 1   Implicit initial values are not defined for an indefinite subtype, because if an object's nominal subtype is indefinite, an explicit initial value is required.<br /></Admonition>

<MarginText>23/5</MarginText>
<MarginInfo items={["AI05-0092-1", "AI05-0255-1", "AI12-0061-1", "AI12-0308-1"]} />
<Admonition type="aarm" aarm="note" title="Note: ">
NOTE 2   As indicated above, a stand-alone object is an object declared by an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code>. Similar definitions apply to &ldquo;stand-alone constant&rdquo; and &ldquo;stand-alone variable&rdquo;. A subcomponent of an object is not a stand-alone object, nor is an object that is created by an <code><a href="../AA-4/AA-4.8#S0164">allocator</a></code>. An object declared by a <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code>, <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code>, <code><a href="../AA-4/AA-4.3#S0119">iterated_component_association</a></code>, <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code>, <code><a href="../AA-6/AA-6.1#S0207">parameter_specification</a></code>, <code><a href="../AA-9/AA-9.5#S0263">entry_index_specification</a></code>, <code><a href="../AA-11/AA-11.2#S0306">choice_parameter_specification</a></code>, <code><a href="../AA-6/AA-6.5#S0225">extended_return_statement</a></code>, or a <code><a href="../AA-12/AA-12.4#S0319">formal_object_declaration</a></code> of mode <strong>in out</strong> is not considered a stand-alone object.<br /></Admonition>

<MarginText>24</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
NOTE 3   The type of a stand-alone object cannot be abstract (see <a href="../AA-3/AA-3.9#Subclause_3.9.3">3.9.3</a>). <br /></Admonition>

#### Extensions to Ada 83

<MarginText>33.a</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
The syntax rule for <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> is modified to allow the <strong>aliased</strong> reserved word.<br /></Admonition>

<MarginText>33.b</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
A variable declared by an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> can be constrained by its initial value; that is, a variable of a nominally unconstrained array subtype, or discriminated type without defaults, can be declared so long as it has an explicit initial value. In Ada 83, this was permitted for constants, and for variables created by allocators, but not for variables declared by <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code>s. This is particularly important for tagged class-wide types, since there is no way to constrain them explicitly, and so an initial value is the only way to provide a constraint. It is also important for generic formal private types with unknown discriminants.<br /></Admonition>

<MarginText>33.c</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
We now allow an <code><a href="../AA-3/AA-3.6#S0052">unconstrained_array_definition</a></code> in an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code>. This allows an object of an anonymous array type to have its bounds determined by its initial value. This is for uniformity: If one can write &ldquo;X: <strong>constant</strong> <strong>array</strong>(Integer <strong>range</strong> 1..10) <strong>of</strong> Integer := ...;&rdquo; then it makes sense to also allow &ldquo;X: <strong>constant</strong> <strong>array</strong>(Integer <strong>range</strong> {"<"}{">"}) <strong>of</strong> Integer := ...;&rdquo;. (Note that if anonymous array types are ever sensible, a common situation is for a table implemented as an array. Tables are often constant, and for constants, there's usually no point in forcing the user to count the number of elements in the value.) <br /></Admonition>

#### Inconsistencies With Ada 95

<MarginText>33.f/2</MarginText>
<MarginInfo items={["AI95-00363-01"]} />
<Admonition type="aarm" aarm="note" title="Note: ">
Unconstrained aliased objects of types with discriminants with defaults are no longer constrained by their initial values. This means that a program that raised Constraint_Error from an attempt to change the discriminants will no longer do so. The change only affects programs that depended on the raising of Constraint_Error in this case, so the inconsistency is unlikely to occur outside of the ACATS. This change may however cause compilers to implement these objects differently, possibly taking additional memory or time. This is unlikely to be worse than the differences caused by any major compiler upgrade. <br /></Admonition>

#### Inconsistencies With Ada 2012

<MarginText>33.n/5</MarginText>
<MarginInfo items={["AI12-0192-1"]} />
<Admonition type="aarm" aarm="correction" title="Correction:">
<strong></strong> Components of a protected type require late initialization if their initialization includes (implicitly) the current instance of the type. However, there do exist very unlikely cases where legal, portable Ada 2012 code would become erroneous. (See the discussion section of  for an example.) These are so unlikely that it is expected that they only exist in the minds of Ada lawyers. <br /></Admonition>

## 3.3.2  Number Declarations

<MarginText>1</MarginText>
<p>A <code><a href="../AA-3/AA-3.3#S0034">number_declaration</a></code> declares a named number. <br /></p>

<MarginText>1.a/3</MarginText>
<MarginInfo items={["AI05-0299-1"]} />
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<strong></strong>If a value or other property of a construct is required to be <em>static</em> that means it is required to be determined prior to execution. A <em>static</em> expression is an expression whose value is computed at compile time and is usable in contexts where the actual value might affect the legality of the construct. This is fully defined in subclause <a href="../AA-4/AA-4.9">4.9</a>. <br /></Admonition>


#### Syntax

<MarginText>2</MarginText>

<CodeBlock>
<code>number_declaration</code><a id="S0034"></a><code> ::= </code><br />     <code><a href="../AA-3/AA-3.3#S0033">defining_identifier_list</a></code> : <strong>constant</strong> := <em>static_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>;<br />

</CodeBlock>

#### Name Resolution Rules

<MarginText>3</MarginText>
<p>The <em>static_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> given for a <code><a href="../AA-3/AA-3.3#S0034">number_declaration</a></code> is expected to be of any numeric type.<br /></p>

<MarginText>3.1/5</MarginText>
<MarginInfo items={["AI12-0394-1"]} />
<p>A <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes a <code><a href="../AA-3/AA-3.3#S0034">number_declaration</a></code> is interpreted as a value of a universal type, unless the expected type for the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> is a non-numeric type with an Integer_Literal or Real_Literal aspect, in which case it is interpreted to be of its expected type.<br /></p>

<MarginText>3.a/5</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
<strong>To be honest: </strong>This is only a Name Resolution Rule; all named numbers are values of a universal type (see the Static Semantics below). We need this rule so that named numbers can match types with user-defined literals; we need the other rules so the value of the named number is well-defined in all cases. <br /></Admonition>


#### Legality Rules

<MarginText>4/3</MarginText>
<MarginInfo items={["AI05-0299-1"]} />
<p>The <em>static_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> given for a number declaration shall be a static expression, as defined by subclause <a href="../AA-4/AA-4.9">4.9</a>. <br /></p>


#### Static Semantics

<MarginText>5</MarginText>
<p>The named number denotes a value of type <em>universal_integer</em> if the type of the <em>static_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is an integer type. The named number denotes a value of type <em>universal_real</em> if the type of the <em>static_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is a real type.<br /></p>

<MarginText>6</MarginText>
<p>The value denoted by the named number is the value of the <em>static_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, converted to the corresponding universal type. <br /></p>


#### Dynamic Semantics

<MarginText>7</MarginText>
<p>The elaboration of a <code><a href="../AA-3/AA-3.3#S0034">number_declaration</a></code> has no effect. <br /></p>

<MarginText>7.a</MarginText>
<Admonition type="aarm" aarm="proof" title="Proof: ">
<strong></strong>Since the <em>static_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> was evaluated at compile time. <br /></Admonition>


#### Examples

<MarginText>8</MarginText>
<p><em>Examples of number declarations:</em> <br /></p>

<MarginText>9</MarginText>
<CodeBlock language="ada">
Two_Pi        : constant := 2.0*Ada.Numerics.Pi;   -- a real number (see <a href="../AA-A/AA-A.5">A.5</a>){"\n"}

<MarginInfo items={["AI95-00433-01"]} />

</CodeBlock>

#### Extensions to Ada 83

<MarginText>10.a</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
We now allow a static expression of any numeric type to initialize a named number. For integer types, it was possible in Ada 83 to use 'Pos to define a named number, but there was no way to use a static expression of some nonuniversal real type to define a named number. This change is upward compatible because of the preference rule for the operators of the root numeric types. <br /></Admonition>


#### Wording Changes from Ada 83

<MarginText>10.b</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
We have moved the syntax rule into this subclause.<br /></Admonition>

<MarginText>10.c</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
AI83-00263 describes the elaboration of a number declaration in words similar to that of an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code>. However, since there is no expression to be evaluated and no object to be created, it seems simpler to say that the elaboration has no effect. <br /></Admonition>
