<MarginText>1</MarginText>
<p><i>Scalar</i> types comprise enumeration types, integer types, and real types. Enumeration types and integer types are called <i>discrete</i> types; each value of a discrete type has a <i>position number</i> which is an integer value. Integer types and real types are called <i>numeric</i> types. [All scalar types are ordered, that is, all relational operators are predefined for their values.]<br /></p>


#### Syntax

<MarginText>2</MarginText>

<CodeBlock>
<code>range{"_"}constraint</code><a id="S0036"></a><code> ::= </code> <b>range</b> <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code><br />

</CodeBlock>
<MarginText>3</MarginText>

<CodeBlock>
<code>range</code><a id="S0037"></a><code> ::= </code> <code><a href="/docs/arm/AA-4/AA-4.1#S0102">range_attribute_reference</a></code><br />   | <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code> .. <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code><br />

</CodeBlock>
<AnnotatedOnly>
<MarginText>3.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>These need to be <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code>s rather than more general <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code>s because ranges appear in membership tests and other contexts where <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> .. <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> would be ambiguous. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>4</MarginText>
<p>A <i>range</i> has a <i>lower bound</i> and an <i>upper bound</i> and specifies a subset of the values of some scalar type (the <i>type of the range</i>). A range with lower bound L and upper bound R is described by &ldquo;L .. R&rdquo;. If R is less than L, then the range is a <i>null range</i>, and specifies an empty set of values. Otherwise, the range specifies the values of the type from the lower bound to the upper bound, inclusive. A value <i>belongs</i> to a range if it is of the type of the range, and is in the subset of values specified by the range. A value <i>satisfies</i> a range constraint if it belongs to the associated range. One range is <i>included</i> in another if all values that belong to the first range also belong to the second. <br /></p>


#### Name Resolution Rules

<MarginText>5</MarginText>
<p>For a <code><a href="/docs/arm/AA-3/AA-3.2#S0027">subtype_indication</a></code> containing a <code><a href="/docs/arm/AA-3/AA-3.5#S0036">range_constraint</a></code>, either directly or as part of some other <code><a href="/docs/arm/AA-3/AA-3.2#S0030">scalar_constraint</a></code>, the type of the <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code> shall resolve to that of the type determined by the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code> of the <code><a href="/docs/arm/AA-3/AA-3.2#S0027">subtype_indication</a></code>. For a <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code> of a given type, the <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code>s of the <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code> (likewise, the <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code>s of the equivalent <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code> for a <code><a href="/docs/arm/AA-4/AA-4.1#S0102">range_attribute_reference</a></code>) are expected to be of the type of the <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>5.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>In Ada 95, <code><a href="/docs/arm/AA-3/AA-3.2#S0029">constraint</a></code>s only appear within <code><a href="/docs/arm/AA-3/AA-3.2#S0027">subtype_indication</a></code>s; things that look like constraints that appear in type declarations are called something else like <code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code>s.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b/5</MarginText>
<MarginInfo items={["AI05-0299-1", "AI12-0449-1"]} />
<Admonition type="aarm" aarm="note">
We say "the expected type is ..." or "the type is expected to be ..." depending on which reads better. They are fundamentally equivalent, and both feed into the type resolution rules of <a href="/docs/arm/AA-8/AA-8.6">8.6</a>.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.c</MarginText>
<Admonition type="aarm" aarm="note">
In some cases, it doesn't work to use expected types. For example, in the above rule, we say that the &ldquo;type of the <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code> shall resolve to ...&rdquo; rather than &ldquo;the expected type for the <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code> is ...&rdquo; We then use &ldquo;expected type&rdquo; for the bounds. If we used &ldquo;expected&rdquo; at both points, there would be an ambiguity, since one could apply the rules of <a href="/docs/arm/AA-8/AA-8.6">8.6</a> either on determining the type of the range, or on determining the types of the individual bounds. It is clearly important to allow one bound to be of a universal type, and the other of a specific type, so we need to use &ldquo;expected type&rdquo; for the bounds. Hence, we used &ldquo;shall resolve to&rdquo; for the type of the range as a whole. There are other situations where &ldquo;expected type&rdquo; is not quite right, and we use &ldquo;shall resolve to&rdquo; instead. <br />
</Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>6</MarginText>
<p>The <i>base range</i> of a scalar type is the range of finite values of the type that can be represented in every unconstrained object of the type; it is also the range supported at a minimum for intermediate values during the evaluation of expressions involving predefined operators of the type. <br /></p>

<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>Note that in some machine architectures intermediates in an expression (particularly if static), and register-resident variables might accommodate a wider range. The base range does not include the values of this wider range that are not assignable without overflow to memory-resident objects.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>The base range of an enumeration type is the range of values of the enumeration type. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.c</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>If the representation supports infinities, the base range is nevertheless restricted to include only the representable finite values, so that 'Base'First and 'Base'Last are always guaranteed to be finite.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.d</MarginText>
<Admonition type="aarm" aarm="note">
<b>To be honest: </b>By a "value that can be assigned without overflow" we don't mean to restrict ourselves to values that can be represented exactly. Values between machine representable values can be assigned, but on subsequent reading, a slightly different value might be retrieved, as (partially) determined by the number of digits of precision of the type. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>7</MarginText>
<p>[A constrained scalar subtype is one to which a range constraint applies.] The <i>range</i> of a constrained scalar subtype is the range associated with the range constraint of the subtype. The <i>range</i> of an unconstrained scalar subtype is the base range of its type. <br /></p>


#### Dynamic Semantics

<MarginText>8</MarginText>
<p>A range is <i>compatible</i> with a scalar subtype if and only if it is either a null range or each bound of the range belongs to the range of the subtype. A <code><a href="/docs/arm/AA-3/AA-3.5#S0036">range_constraint</a></code> is <i>compatible</i> with a scalar subtype if and only if its range is compatible with the subtype. <br /></p>

<AnnotatedOnly>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Only <code><a href="/docs/arm/AA-3/AA-3.5#S0036">range_constraint</a></code>s (explicit or implicit) impose conditions on the values of a scalar subtype. The other <code><a href="/docs/arm/AA-3/AA-3.2#S0030">scalar_constraint</a></code>s, <code><a href="/docs/arm/AA-3/AA-3.5#S0050">digits_constraint</a></code>s and <code><a href="/docs/arm/AA-J/AA-J.3#S0367">delta_constraint</a></code>s impose conditions on the subtype denoted by the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code> in a <code><a href="/docs/arm/AA-3/AA-3.2#S0027">subtype_indication</a></code>, but don't impose a condition on the values of the subtype being defined. Therefore, a scalar subtype is not called <i>constrained</i> if all that applies to it is a <code><a href="/docs/arm/AA-3/AA-3.5#S0050">digits_constraint</a></code>. Decimal subtypes are subtle, because a <code><a href="/docs/arm/AA-3/AA-3.5#S0050">digits_constraint</a></code> without a <code><a href="/docs/arm/AA-3/AA-3.5#S0036">range_constraint</a></code> nevertheless includes an implicit <code><a href="/docs/arm/AA-3/AA-3.5#S0036">range_constraint</a></code>. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>9</MarginText>
<p>The elaboration of a <code><a href="/docs/arm/AA-3/AA-3.5#S0036">range_constraint</a></code> consists of the evaluation of the <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code>. The evaluation of a <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code> determines a lower bound and an upper bound. If <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code>s are given to specify bounds, the evaluation of the <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code> evaluates these <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code>s in an arbitrary order, and converts them to the type of the <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code>. If a <code><a href="/docs/arm/AA-4/AA-4.1#S0102">range_attribute_reference</a></code> is given, the evaluation of the <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code> consists of the evaluation of the <code><a href="/docs/arm/AA-4/AA-4.1#S0102">range_attribute_reference</a></code>.<br /></p>

#### Incompatibilities With Ada 83

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>63.a/1</MarginText>
<Admonition type="aarm" aarm="note">
S'Base is no longer defined for nonscalar types. One conceivable existing use of S'Base for nonscalar types is S'Base'Size where S is a generic formal private type. However, that is not generally useful because the actual subtype corresponding to S might be a constrained array or discriminated type, which would mean that S'Base'Size might very well overflow (for example, S'Base'Size where S is a constrained subtype of String will generally be 8 {"*"} (Integer'Last + 1)). For derived discriminated types that are packed, S'Base'Size might not even be well defined if the first subtype is constrained, thereby allowing some amount of normally required &ldquo;dope&rdquo; to have been squeezed out in the packing. Hence our conclusion is that S'Base'Size is not generally useful in a generic, and does not justify keeping the attribute Base for nonscalar types just so it can be used as a <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code>.<br />

#### Static Semantics

<MarginText>6/3_3.5.1</MarginText>
<MarginInfo items={["AI05-0006-1"]} />
<p>Each <code><a href="/docs/arm/AA-3/AA-3.5#S0039">enumeration_literal_specification</a></code> is the explicit declaration of the corresponding <i>enumeration literal</i>: it declares a parameterless function, whose defining name is the <code><a href="/docs/arm/AA-3/AA-3.1#S0022">defining{"_"}identifier</a></code> or <code><a href="/docs/arm/AA-3/AA-3.5#S0040">defining{"_"}character{"_"}literal</a></code>, and whose result subtype is the base subtype of the enumeration type. <br /></p>

<AnnotatedOnly>
<MarginText>6.a_3.5.1</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>This rule defines the profile of the enumeration literal, which is used in the various types of conformance. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.b_3.5.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>The parameterless function associated with an enumeration literal is fully defined by the <code><a href="/docs/arm/AA-3/AA-3.5#S0038">enumeration_type_definition</a></code>; a body is not permitted for it, and it never fails the Elaboration{"_"}Check when called. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.c/3_3.5.1</MarginText>
<MarginInfo items={["AI05-0006-1"]} />
<Admonition type="aarm" aarm="discussion">
<b></b>The result subtype is primarily a concern when an enumeration literal is used as the <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> of a case statement, due to the full coverage requirement based on the nominal subtype. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>7_3.5.1</MarginText>
<p>Each enumeration literal corresponds to a distinct value of the enumeration type, and to a distinct position number. The position number of the value of the first listed enumeration literal is zero; the position number of the value of each subsequent enumeration literal is one more than that of its predecessor in the list.<br /></p>

<MarginText>8_3.5.1</MarginText>
<p>[The predefined order relations between values of the enumeration type follow the order of corresponding position numbers.]<br /></p>

<MarginText>9_3.5.1</MarginText>
<p>[ If the same <code><a href="/docs/arm/AA-3/AA-3.1#S0022">defining_identifier</a></code> or <code><a href="/docs/arm/AA-3/AA-3.5#S0040">defining_character_literal</a></code> is specified in more than one <code><a href="/docs/arm/AA-3/AA-3.5#S0038">enumeration{"_"}type{"_"}definition</a></code>, the corresponding enumeration literals are said to be <i>overloaded</i>. At any place where an overloaded enumeration literal occurs in the text of a program, the type of the enumeration literal has to be determinable from the context (see <a href="/docs/arm/AA-8/AA-8.6">8.6</a>).] <br /></p>


#### Dynamic Semantics

<MarginText>10_3.5.1</MarginText>
<p>The elaboration of an <code><a href="/docs/arm/AA-3/AA-3.5#S0038">enumeration_type_definition</a></code> creates the enumeration type and its first subtype, which is constrained to the base range of the type. <br /></p>

<AnnotatedOnly>
<MarginText>10.a_3.5.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>The first subtype of a discrete type is always constrained, except in the case of a derived type whose parent subtype is Whatever'Base. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>11_3.5.1</MarginText>
<p>When called, the parameterless function associated with an enumeration literal returns the corresponding value of the enumeration type. <br /></p>

<AnnotatedOnly>
<MarginText>12_3.5.1</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   If an enumeration literal occurs in a context that does not otherwise suffice to determine the type of the literal, then qualification by the name of the enumeration type is one way to resolve the ambiguity (see <a href="/docs/arm/AA-4/AA-4.7">4.7</a>). <br />
</Admonition>
</AnnotatedOnly>

## 3.5.2  Character Types


#### Static Semantics

<MarginText>1_3.5.2</MarginText>
<p>An enumeration type is said to be a <i>character type</i> if at least one of its enumeration literals is a <code><a href="/docs/arm/AA-2/AA-2.5#S0015">character_literal</a></code>.<br /></p>

<MarginText>2/5_3.5.2</MarginText>
<MarginInfo items={["AI95-00285-01", "AI05-0181-1", "AI05-0262-1", "AI05-0266-1", "AI12-0263-1", "AI12-0450-1"]} />
<p>The predefined type Character is a character type whose values correspond to the 256 code points of Row 00 (also known as Latin-1) of the ISO/IEC 10646:2020 Basic Multilingual Plane (BMP). Each of the graphic characters of Row 00 of the BMP has a corresponding <code><a href="/docs/arm/AA-2/AA-2.5#S0015">character_literal</a></code> in Character. Each of the nongraphic characters of Row 00 has a corresponding language-defined name, which is not usable as an enumeration literal, but which is usable with the attributes Image, Wide{"_"}Image, Wide{"_"}Wide{"_"}Image, Value, Wide{"_"}Value, and Wide{"_"}Wide{"_"}Value; these names are given in the definition of type Character in <a href="/docs/arm/AA-A/AA-A.1">A.1</a>, &ldquo;<a href="/docs/arm/AA-A/AA-A.1">The Package Standard</a>&rdquo;, but are set in <i>italics</i>. <br /></p>

<AnnotatedOnly>
<MarginText>2.a/5_3.5.2</MarginText>
<MarginInfo items={["AI05-0262-1", "AI12-0263-1", "AI12-0450-1"]} />
<Admonition type="aarm" aarm="discussion">
<b></b><i>Code point</i> is defined in ISO/IEC 10646:2020. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>3/5_3.5.2</MarginText>
<MarginInfo items={["AI95-00285-01", "AI05-0262-1", "AI12-0263-1", "AI12-0450-1"]} />
<p>The predefined type Wide{"_"}Character is a character type whose values correspond to the 65536 code points of the ISO/IEC 10646:2020 Basic Multilingual Plane (BMP). Each of the graphic characters of the BMP has a corresponding <code><a href="/docs/arm/AA-2/AA-2.5#S0015">character_literal</a></code> in Wide{"_"}Character. The first 256 values of Wide{"_"}Character have the same <code><a href="/docs/arm/AA-2/AA-2.5#S0015">character_literal</a></code> or language-defined name as defined for Character. Each of the <code>graphic{"_"}character</code>s has a corresponding <code><a href="/docs/arm/AA-2/AA-2.5#S0015">character_literal</a></code>.<br /></p>

<MarginText>4/5_3.5.2</MarginText>
<MarginInfo items={["AI95-00285-01", "AI05-0262-1", "AI12-0263-1", "AI12-0450-1"]} />
<p>The predefined type Wide{"_"}Wide{"_"}Character is a character type whose values correspond to the 2147483648 code points of the ISO/IEC 10646:2020 character set. Each of the <code>graphic{"_"}character</code>s has a corresponding <code><a href="/docs/arm/AA-2/AA-2.5#S0015">character_literal</a></code> in Wide{"_"}Wide{"_"}Character. The first 65536 values of Wide{"_"}Wide{"_"}Character have the same <code><a href="/docs/arm/AA-2/AA-2.5#S0015">character_literal</a></code> or language-defined name as defined for Wide{"_"}Character.<br /></p>

<MarginText>5/3_3.5.2</MarginText>
<MarginInfo items={["AI95-00285-01", "AI05-0262-1"]} />
<p>The characters whose code point is larger than 16{"#"}FF{"#"} and which are not <code>graphic{"_"}character</code>s have language-defined names which are formed by appending to the string "Hex{"_"}" the representation of their code point in hexadecimal as eight extended digits. As with other language-defined names, these names are usable only with the attributes (Wide{"_"})Wide{"_"}Image and (Wide{"_"})Wide{"_"}Value; they are not usable as enumeration literals.<br /></p>

<AnnotatedOnly>
<MarginText>5.a/2_3.5.2</MarginText>
<MarginInfo items={["AI95-00285-01"]} />
<Admonition type="aarm" aarm="reason">
<b></b>The language-defined names are not usable as enumeration literals to avoid "polluting" the name space. Since Wide{"_"}Character and Wide{"_"}Wide{"_"}Character are defined in Standard, if the language-defined names were usable as enumeration literals, they would hide other nonoverloadable declarations with the same names in <b>use</b>-d packages.<br />
</Admonition>
</AnnotatedOnly>

## 3.5.3  Boolean Types


#### Static Semantics

<MarginText>1_3.5.3</MarginText>
<p>There is a predefined enumeration type named Boolean, [declared in the visible part of package Standard]. It has the two enumeration literals False and True ordered with the relation False {"<"} True. Any descendant of the predefined type Boolean is called a <i>boolean</i> type. <br /></p>

<AnnotatedOnly>
<MarginText>1.a_3.5.3</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>An implementation is not required to support enumeration representation clauses on boolean types that impose an unacceptable implementation burden. See <a href="/docs/arm/AA-13/AA-13.4">13.4</a>, &ldquo;<a href="/docs/arm/AA-13/AA-13.4">Enumeration Representation Clauses</a>&rdquo;. However, it is generally straightforward to support representations where False is zero and True is 2{"*"}{"*"}n &ndash; 1 for some n. <br />
</Admonition>
</AnnotatedOnly>

## 3.5.4  Integer Types

<MarginText>1_3.5.4</MarginText>
<p>An <code><a href="/docs/arm/AA-3/AA-3.5#S0041">integer_type_definition</a></code> defines an integer type; it defines either a <i>signed</i> integer type, or a <i>modular</i> integer type. The base range of a signed integer type includes at least the values of the specified range. A modular type is an integer type with all arithmetic modulo a specified positive <i>modulus</i>; such a type corresponds to an unsigned type with wrap-around semantics. <br /></p>


#### Syntax

<MarginText>2_3.5.4</MarginText>

<CodeBlock>
<code>integer{"_"}type{"_"}definition</code><a id="S0041"></a><code> ::= </code><br />    <code><a href="/docs/arm/AA-3/AA-3.5#S0042">signed_integer_type_definition</a></code> | <code><a href="/docs/arm/AA-3/AA-3.5#S0043">modular_type_definition</a></code><br />

</CodeBlock>
<MarginText>3_3.5.4</MarginText>

<CodeBlock>
<code>signed{"_"}integer{"_"}type{"_"}definition</code><a id="S0042"></a><code> ::= </code><br />    <b>range</b> <i>static{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code> .. <i>static{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code><br />

</CodeBlock>
<AnnotatedOnly>
<MarginText>3.a_3.5.4</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>We don't call this a <code><a href="/docs/arm/AA-3/AA-3.5#S0036">range_constraint</a></code>, because it is rather different &mdash; not only is it required to be static, but the associated overload resolution rules are different than for normal range constraints. A similar comment applies to <code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code>. This used to be <code>integer{"_"}range{"_"}specification</code> but when we added support for modular types, it seemed overkill to have three levels of syntax rules, and just calling these <code>signed{"_"}integer{"_"}range{"_"}specification</code> and <code>modular{"_"}range{"_"}specification</code> loses the fact that they are defining different classes of types, which is important for the generic type matching rules. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>4_3.5.4</MarginText>

<CodeBlock>
<code>modular{"_"}type{"_"}definition</code><a id="S0043"></a><code> ::= </code><b>mod</b> <i>static{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code><br />

</CodeBlock>

#### Name Resolution Rules

<MarginText>5/5_3.5.4</MarginText>
<MarginInfo items={["AI12-0444-1"]} />
<p>Each <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code> in a <code><a href="/docs/arm/AA-3/AA-3.5#S0042">signed_integer_type_definition</a></code> is expected to be of any integer type; they can be of different integer types. The <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> in a <code><a href="/docs/arm/AA-3/AA-3.5#S0043">modular_type_definition</a></code> is likewise expected to be of any integer type. <br /></p>


#### Legality Rules

<MarginText>6_3.5.4</MarginText>
<p>The <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code>s of a <code><a href="/docs/arm/AA-3/AA-3.5#S0042">signed_integer_type_definition</a></code> shall be static, and their values shall be in the range System.Min{"_"}Int .. System.Max{"_"}Int.<br /></p>

<MarginText>7_3.5.4</MarginText>
<p>The <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> of a <code><a href="/docs/arm/AA-3/AA-3.5#S0043">modular_type_definition</a></code> shall be static, and its value (the <i>modulus</i>) shall be positive, and shall be no greater than System.Max{"_"}Binary{"_"}Modulus if a power of 2, or no greater than System.Max{"_"}Nonbinary{"_"}Modulus if not. <br /></p>

<AnnotatedOnly>
<MarginText>7.a_3.5.4</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>For a 2's-complement machine, supporting nonbinary moduli greater than System.Max{"_"}Int can be quite difficult, whereas essentially any binary moduli are straightforward to support, up to 2{"*"}System.Max{"_"}Int+2, so this justifies having two separate limits. <br />
</Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>8_3.5.4</MarginText>
<p>The set of values for a signed integer type is the (infinite) set of mathematical integers[, though only values of the base range of the type are fully supported for run-time operations]. The set of values for a modular integer type are the values from 0 to one less than the modulus, inclusive.<br /></p>

<MarginText>9_3.5.4</MarginText>
<p>A <code><a href="/docs/arm/AA-3/AA-3.5#S0042">signed_integer_type_definition</a></code> defines an integer type whose base range includes at least the values of the <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code>s and is symmetric about zero, excepting possibly an extra negative value. A <code><a href="/docs/arm/AA-3/AA-3.5#S0042">signed_integer_type_definition</a></code> also defines a constrained first subtype of the type, with a range whose bounds are given by the values of the <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code>s, converted to the type being defined. <br /></p>

<AnnotatedOnly>
<MarginText>9.a/2_3.5.4</MarginText>
<MarginInfo items={["AI95-00114-01"]} />
<Admonition type="aarm" aarm="implementation-note">
<b></b>The base range of a signed integer type might be much larger than is necessary to satisfy the above requirements. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.a.1/1_3.5.4</MarginText>
<Admonition type="aarm" aarm="note">
<b>To be honest: </b>The conversion mentioned above is not an <i>implicit subtype conversion</i> (which is something that happens at overload resolution, see <a href="/docs/arm/AA-4/AA-4.6">4.6</a>), although it happens implicitly. Therefore, the freezing rules are not invoked on the type (which is important so that representation items can be given for the type). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>10_3.5.4</MarginText>
<p>A <code><a href="/docs/arm/AA-3/AA-3.5#S0043">modular_type_definition</a></code> defines a modular type whose base range is from zero to one less than the given modulus. A <code><a href="/docs/arm/AA-3/AA-3.5#S0043">modular_type_definition</a></code> also defines a constrained first subtype of the type with a range that is the same as the base range of the type.<br /></p>

<MarginText>11_3.5.4</MarginText>
<p>There is a predefined signed integer subtype named Integer[, declared in the visible part of package Standard]. It is constrained to the base range of its type. <br /></p>

<AnnotatedOnly>
<MarginText>11.a_3.5.4</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>Integer is a constrained subtype, rather than an unconstrained subtype. This means that on assignment to an object of subtype Integer, a range check is required. On the other hand, an object of subtype Integer'Base is unconstrained, and no range check (only overflow check) is required on assignment. For example, if the object is held in an extended-length register, its value might be outside of Integer'First .. Integer'Last. All parameter and result subtypes of the predefined integer operators are of such unconstrained subtypes, allowing extended-length registers to be used as operands or for the result. In an earlier version of Ada 95, Integer was unconstrained. However, the fact that certain Constraint{"_"}Errors might be omitted or appear elsewhere was felt to be an undesirable upward inconsistency in this case. Note that for Float, the opposite conclusion was reached, partly because of the high cost of performing range checks when not actually necessary. Objects of subtype Float are unconstrained, and no range checks, only overflow checks, are performed for them. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>12_3.5.4</MarginText>
<p>Integer has two predefined subtypes, [declared in the visible part of package Standard:] <br /></p>

<MarginText>13_3.5.4</MarginText>
<CodeBlock language="ada">
subtype Natural  is Integer range 0 .. Integer'Last;{"\n"}subtype Positive is Integer range 1 .. Integer'Last;{"\n"}

</CodeBlock>
<MarginText>14_3.5.4</MarginText>
<p>A type defined by an <code><a href="/docs/arm/AA-3/AA-3.5#S0041">integer_type_definition</a></code> is implicitly derived from <i>root{"_"}integer</i>, an anonymous predefined (specific) integer type, whose base range is System.Min{"_"}Int .. System.Max{"_"}Int. However, the base range of the new type is not inherited from <i>root{"_"}integer</i>, but is instead determined by the range or modulus specified by the <code><a href="/docs/arm/AA-3/AA-3.5#S0041">integer_type_definition</a></code>. [Integer literals are all of the type <i>universal{"_"}integer</i>, the universal type (see <a href="/docs/arm/AA-3/AA-3.4#Subclause_3.4.1">3.4.1</a>) for the class rooted at <i>root{"_"}integer</i>, allowing their use with the operations of any integer type.] <br /></p>

<AnnotatedOnly>
<MarginText>14.a_3.5.4</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>This implicit derivation is not considered exactly equivalent to explicit derivation via a <code><a href="/docs/arm/AA-3/AA-3.4#S0035">derived_type_definition</a></code>. In particular, integer types defined via a <code><a href="/docs/arm/AA-3/AA-3.4#S0035">derived_type_definition</a></code> inherit their base range from their parent type. A type defined by an <code><a href="/docs/arm/AA-3/AA-3.5#S0041">integer_type_definition</a></code> does not necessarily inherit its base range from <i>root{"_"}integer</i>. It is not specified whether the implicit derivation from <i>root{"_"}integer</i> is direct or indirect, not that it really matters. All we want is for all integer types to be descendants of <i>root{"_"}integer</i>.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.a.1/1_3.5.4</MarginText>
<MarginInfo items={["AI95-00152-01"]} />
<Admonition type="aarm" aarm="note">
{"{"}<i>8652/0099</i>{"}"} Note that this derivation does not imply any inheritance of subprograms. Subprograms are inherited only for types derived by a <code><a href="/docs/arm/AA-3/AA-3.4#S0035">derived{"_"}type{"_"}definition</a></code> (see <a href="/docs/arm/AA-3/AA-3.4">3.4</a>), or a <code><a href="/docs/arm/AA-7/AA-7.3#S0233">private{"_"}extension{"_"}declaration</a></code> (see <a href="/docs/arm/AA-7/AA-7.3">7.3</a>, <a href="/docs/arm/AA-7/AA-7.3#Subclause_7.3.1">7.3.1</a>, and <a href="/docs/arm/AA-12/AA-12.5#Subclause_12.5.1">12.5.1</a>). <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.b_3.5.4</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>It is the intent that even nonstandard integer types (see below) will be descendants of <i>root{"_"}integer</i>, even though they might have a base range that exceeds that of <i>root{"_"}integer</i>. This causes no problem for static calculations, which are performed without range restrictions (see <a href="/docs/arm/AA-4/AA-4.9">4.9</a>). However for run-time calculations, it is possible that Constraint{"_"}Error might be raised when using an operator of <i>root{"_"}integer</i> on the result of 'Val applied to a value of a nonstandard integer type. <br />
</Admonition>
</AnnotatedOnly>

#### Dynamic Semantics

<MarginText>18_3.5.4</MarginText>
<p>The elaboration of an <code><a href="/docs/arm/AA-3/AA-3.5#S0041">integer_type_definition</a></code> creates the integer type and its first subtype.<br /></p>

<MarginText>21_3.5.4</MarginText>

<MarginText>22_3.5.4</MarginText>

<MarginText>23_3.5.4</MarginText>

#### Implementation Permissions

<MarginText>24/5_3.5.4</MarginText>
<MarginInfo items={["AI12-0444-1"]} />
<p>For the execution of a predefined operation of a signed integer type, it is optional to raise Constraint{"_"}Error if the result is outside the base range of the type, so long as the correct result is produced. <br /></p>

<AnnotatedOnly>
<MarginText>24.a_3.5.4</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>Constraint{"_"}Error is never raised for operations on modular types, except for divide-by-zero (and <b>rem</b>/<b>mod</b>-by-zero). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>25_3.5.4</MarginText>
<p>An implementation may provide additional predefined signed integer types[, declared in the visible part of Standard], whose first subtypes have names of the form Short{"_"}Integer, Long{"_"}Integer, Short{"_"}Short{"_"}Integer, Long{"_"}Long{"_"}Integer, etc. Different predefined integer types are allowed to have the same base range. However, the range of Integer should be no wider than that of Long{"_"}Integer. Similarly, the range of Short{"_"}Integer (if provided) should be no wider than Integer. Corresponding recommendations apply to any other predefined integer types. An implementation may support base ranges for which there is no corresponding named integer type. The range of each first subtype should be the base range of its type. <br /></p>

<AnnotatedOnly>
<MarginText>25.a/5_3.5.4</MarginText>
<MarginInfo items={["AI12-0444-1"]} />
<Admonition type="aarm" aarm="implementation-defined">
<b></b>The predefined integer types declared in Standard.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>26/5_3.5.4</MarginText>
<MarginInfo items={["AI12-0444-1"]} />
<p>An implementation may provide <i>nonstandard integer types</i>, descendants of <i>root{"_"}integer</i> that are declared outside of the specification of package Standard, which may have different characteristics than a type defined by an <code><a href="/docs/arm/AA-3/AA-3.5#S0041">integer_type_definition</a></code>. For example, a nonstandard integer type can have an asymmetric base range or it can be disallowed as an array or loop index (a very long integer). Any type descended from a nonstandard integer type is also nonstandard. An implementation may place arbitrary restrictions on the use of such types; it is implementation defined whether operators that are predefined for &ldquo;any integer type&rdquo; are defined for a particular nonstandard integer type. [In any case, such types are not permitted as <code><a href="/docs/arm/AA-12/AA-12.3#S0318">explicit_generic_actual_parameter</a></code>s for formal scalar types &mdash; see <a href="/docs/arm/AA-12/AA-12.5#Subclause_12.5.2">12.5.2</a>.] <br /></p>

<AnnotatedOnly>
<MarginText>26.a_3.5.4</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<b></b>Any nonstandard integer types and the operators defined for them.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>27_3.5.4</MarginText>
<p>For a one's complement machine, the high bound of the base range of a modular type whose modulus is one less than a power of 2 may be equal to the modulus, rather than one less than the modulus. It is implementation defined for which powers of 2, if any, this permission is exercised.<br /></p>

<MarginText>27.1/1_3.5.4</MarginText>
<MarginInfo items={["AI95-00095-01"]} />
<p>{"{"}<i>8652/0003</i>{"}"} For a one's complement machine, implementations may support nonbinary modulus values greater than System.Max{"_"}Nonbinary{"_"}Modulus. It is implementation defined which specific values greater than System.Max{"_"}Nonbinary{"_"}Modulus, if any, are supported. <br /></p>

<AnnotatedOnly>
<MarginText>27.a.1/1_3.5.4</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>On a one's complement machine, the natural full word type would have a modulus of 2{"*"}{"*"}Word{"_"}Size&ndash;1. However, we would want to allow the all-ones bit pattern (which represents negative zero as a number) in logical operations. These permissions are intended to allow that and the natural modulus value without burdening implementations with supporting expensive modulus values. <br />
</Admonition>
</AnnotatedOnly>


#### Implementation Advice

<MarginText>28_3.5.4</MarginText>
<p>An implementation should support Long{"_"}Integer in addition to Integer if the target machine supports 32-bit (or longer) arithmetic. No other named integer subtypes are recommended for package Standard. Instead, appropriate named integer subtypes should be provided in the library package Interfaces (see <a href="/docs/arm/AA-B/AA-B.2">B.2</a>). <br /></p>

<AnnotatedOnly>
<MarginText>28.a.1/2_3.5.4</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<b></b>Long{"_"}Integer should be declared in Standard if the target supports 32-bit arithmetic. No other named integer subtypes should be declared in Standard.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>28.a_3.5.4</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>To promote portability, implementations should explicitly declare the integer (sub)types Integer and Long{"_"}Integer in Standard, and leave other predefined integer types anonymous. For implementations that already support Byte{"_"}Integer, etc., upward compatibility argues for keeping such declarations in Standard during the transition period, but perhaps generating a warning on use. A separate package Interfaces in the predefined environment is available for pre-declaring types such as Integer{"_"}8, Integer{"_"}16, etc. See <a href="/docs/arm/AA-B/AA-B.2">B.2</a>. In any case, if the user declares a subtype (first or not) whose range fits in, for example, a byte, the implementation can store variables of the subtype in a single byte, even if the base range of the type is wider. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>29_3.5.4</MarginText>
<p>An implementation for a two's complement machine should support modular types with a binary modulus up to System.Max{"_"}Int{"*"}2+2. An implementation should support a nonbinary modulus up to Integer'Last. <br /></p>

<AnnotatedOnly>
<MarginText>29.a.1/2_3.5.4</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<b></b>For a two's complement target, modular types with a binary modulus up to System.Max{"_"}Int{"*"}2+2 should be supported. A nonbinary modulus up to Integer'Last should be supported.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>29.a_3.5.4</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>Modular types provide bit-wise "<b>and</b>", "<b>or</b>", "<b>xor</b>", and "<b>not</b>" operations. It is important for systems programming that these be available for all integer types of the target hardware. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>29.b_3.5.4</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Note that on a one's complement machine, the largest supported modular type would normally have a nonbinary modulus. On a two's complement machine, the largest supported modular type would normally have a binary modulus. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>29.c_3.5.4</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>Supporting a nonbinary modulus greater than Integer'Last can impose an undesirable implementation burden on some machines. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>30_3.5.4</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 1   Integer literals are of the anonymous predefined integer type <i>universal{"_"}integer</i>. Other integer types have no literals. However, the overload resolution rules (see <a href="/docs/arm/AA-8/AA-8.6">8.6</a>, &ldquo;<a href="/docs/arm/AA-8/AA-8.6">The Context of Overload Resolution</a>&rdquo;) allow expressions of the type <i>universal{"_"}integer</i> whenever an integer type is expected.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>31_3.5.4</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 2   The same arithmetic operators are predefined for all signed integer types defined by a <code><a href="/docs/arm/AA-3/AA-3.5#S0042">signed_integer_type_definition</a></code> (see <a href="/docs/arm/AA-4/AA-4.5">4.5</a>, &ldquo;<a href="/docs/arm/AA-4/AA-4.5">Operators and Expression Evaluation</a>&rdquo;). In addition, for the unsigned types declared in the language-defined package Interfaces (see <a href="/docs/arm/AA-B/AA-B.2">B.2</a>), functions are defined that provide bit-wise shifting and rotating.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>32_3.5.4</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 3   Modular types match a <code><a href="/docs/arm/AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code> of the form "<b>type</b> T <b>is mod</b> {"<"}{">"};"; signed integer types match "<b>type</b> T <b>is range</b> {"<"}{">"};" (see <a href="/docs/arm/AA-12/AA-12.5#Subclause_12.5.2">12.5.2</a>). <br />

<AnnotatedOnly>
<MarginText>36.e_3.5.4</MarginText>

</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>36.f_3.5.4</MarginText>
<Admonition type="aarm" aarm="note">
We no longer explain integer type definition in terms of an equivalence to a normal type derivation, except to say that all integer types are by definition implicitly derived from <i>root{"_"}integer</i>. This is for various reasons.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>36.g_3.5.4</MarginText>
<Admonition type="aarm" aarm="note">
First of all, the equivalence with a type derivation and a subtype declaration was not perfect, and was the source of various AIs (for example, is the conversion of the bounds static? Is a numeric type a derived type with respect to other rules of the language?)<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>36.h_3.5.4</MarginText>
<Admonition type="aarm" aarm="note">
Secondly, we don't want to require that every integer size supported shall have a corresponding named type in Standard. Adding named types to Standard creates nonportabilities.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>36.i_3.5.4</MarginText>
<Admonition type="aarm" aarm="note">
Thirdly, we don't want the set of types that match a formal derived type "type T is new Integer;" to depend on the particular underlying integer representation chosen to implement a given user-defined integer type. Hence, we would have needed anonymous integer types as parent types for the implicit derivation anyway. We have simply chosen to identify only one anonymous integer type &mdash; <i>root{"_"}integer</i>, and stated that every integer type is derived from it.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>36.j_3.5.4</MarginText>
<Admonition type="aarm" aarm="note">
Finally, the &ldquo;fiction&rdquo; that there were distinct preexisting predefined types for every supported representation breaks down for fixed point with arbitrary smalls, and was never exploited for enumeration types, array types, etc. Hence, there seems little benefit to pushing an explicit equivalence between integer type definition and normal type derivation. <br />
</Admonition>
</AnnotatedOnly>

## 3.5.6  Real Types

<MarginText>1_3.5.6</MarginText>
<p>Real types provide approximations to the real numbers, with relative bounds on errors for floating point types, and with absolute bounds for fixed point types. <br /></p>


#### Syntax

<MarginText>2_3.5.6</MarginText>

<CodeBlock>
<code>real{"_"}type{"_"}definition</code><a id="S0044"></a><code> ::= </code><br />    <code><a href="/docs/arm/AA-3/AA-3.5#S0045">floating_point_definition</a></code> | <code><a href="/docs/arm/AA-3/AA-3.5#S0047">fixed_point_definition</a></code><br />

</CodeBlock>

#### Static Semantics

<MarginText>3_3.5.6</MarginText>
<p>A type defined by a <code><a href="/docs/arm/AA-3/AA-3.5#S0044">real_type_definition</a></code> is implicitly derived from <i>root{"_"}real</i>, an anonymous predefined (specific) real type. [Hence, all real types, whether floating point or fixed point, are in the derivation class rooted at <i>root{"_"}real</i>.] <br /></p>

<AnnotatedOnly>
<MarginText>3.a_3.5.6</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>It is not specified whether the derivation from <i>root{"_"}real</i> is direct or indirect, not that it really matters. All we want is for all real types to be descendants of <i>root{"_"}real</i>.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.a.1/1_3.5.6</MarginText>
<MarginInfo items={["AI95-00152-01"]} />
<Admonition type="aarm" aarm="note">
{"{"}<i>8652/0099</i>{"}"} Note that this derivation does not imply any inheritance of subprograms. Subprograms are inherited only for types derived by a <code><a href="/docs/arm/AA-3/AA-3.4#S0035">derived{"_"}type{"_"}definition</a></code> (see <a href="/docs/arm/AA-3/AA-3.4">3.4</a>), or a <code><a href="/docs/arm/AA-7/AA-7.3#S0233">private{"_"}extension{"_"}declaration</a></code> (see <a href="/docs/arm/AA-7/AA-7.3">7.3</a>, <a href="/docs/arm/AA-7/AA-7.3#Subclause_7.3.1">7.3.1</a>, and <a href="/docs/arm/AA-12/AA-12.5#Subclause_12.5.1">12.5.1</a>).<br />
</Admonition>
</AnnotatedOnly>

<MarginText>4_3.5.6</MarginText>
<p>[ Real literals are all of the type <i>universal{"_"}real</i>, the universal type (see <a href="/docs/arm/AA-3/AA-3.4#Subclause_3.4.1">3.4.1</a>) for the class rooted at <i>root{"_"}real</i>, allowing their use with the operations of any real type. Certain multiplying operators have a result type of <i>universal{"_"}fixed</i> (see <a href="/docs/arm/AA-4/AA-4.5#Subclause_4.5.5">4.5.5</a>), the universal type for the class of fixed point types, allowing the result of the multiplication or division to be used where any specific fixed point type is expected.] <br /></p>


#### Dynamic Semantics

<MarginText>5_3.5.6</MarginText>
<p>The elaboration of a <code><a href="/docs/arm/AA-3/AA-3.5#S0044">real_type_definition</a></code> consists of the elaboration of the <code><a href="/docs/arm/AA-3/AA-3.5#S0045">floating_point_definition</a></code> or the <code><a href="/docs/arm/AA-3/AA-3.5#S0047">fixed_point_definition</a></code>. <br /></p>


#### Implementation Requirements

<MarginText>6_3.5.6</MarginText>
<p>An implementation shall perform the run-time evaluation of a use of a predefined operator of <i>root{"_"}real</i> with an accuracy at least as great as that of any floating point type definable by a <code><a href="/docs/arm/AA-3/AA-3.5#S0045">floating_point_definition</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>6.a_3.5.6</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Static calculations using the operators of <i>root{"_"}real</i> are exact, as for all static calculations. See <a href="/docs/arm/AA-4/AA-4.9">4.9</a>. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.b_3.5.6</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>The Digits attribute of the type used to represent <i>root{"_"}real</i> at run time is at least as great as that of any other floating point type defined by a <code><a href="/docs/arm/AA-3/AA-3.5#S0045">floating_point_definition</a></code>, and its safe range includes that of any such floating point type with the same Digits attribute. On some machines, there might be real types with less accuracy but a wider range, and hence run-time calculations with <i>root{"_"}real</i> might not be able to accommodate all values that can be represented at run time in such floating point or fixed point types. <br />
</Admonition>
</AnnotatedOnly>


#### Implementation Permissions

<MarginText>7/5_3.5.6</MarginText>
<MarginInfo items={["AI95-00114-01", "AI12-0426-1", "AI12-0444-1"]} />
<p>[For the execution of a predefined operation of a real type, it is optional to raise Constraint{"_"}Error if the result is outside the base range of the type, so long as the correct result is produced, or the Machine{"_"}Overflows attribute of the type is False (see <a href="/docs/arm/AA-G/AA-G.2#Subclause_G.2.1">G.2.1</a>).]<br /></p>

<MarginText>8/5_3.5.6</MarginText>
<MarginInfo items={["AI12-0426-1", "AI12-0444-1"]} />
<p>An implementation may provide <i>nonstandard real types</i>, descendants of <i>root{"_"}real</i> that are declared outside of the specification of package Standard, which may have different characteristics than a type defined by a <code><a href="/docs/arm/AA-3/AA-3.5#S0044">real_type_definition</a></code>. For example, a nonstandard real type can have an asymmetric or unsigned base range, or its predefined operations can wrap around or &ldquo;saturate&rdquo; rather than overflow (modular or saturating arithmetic), or it can have a different accuracy model than is standard (see <a href="/docs/arm/AA-G/AA-G.2#Subclause_G.2.1">G.2.1</a>). Any type descended from a nonstandard real type is also nonstandard. An implementation may place arbitrary restrictions on the use of such types; it is implementation defined whether operators that are predefined for &ldquo;any real type&rdquo; are defined for a particular nonstandard real type. [In any case, such types are not permitted as <code><a href="/docs/arm/AA-12/AA-12.3#S0318">explicit_generic_actual_parameter</a></code>s for formal scalar types &mdash; see <a href="/docs/arm/AA-12/AA-12.5#Subclause_12.5.2">12.5.2</a>.] <br /></p>

<AnnotatedOnly>
<MarginText>8.a_3.5.6</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<b></b>Any nonstandard real types and the operators defined for them.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9_3.5.6</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   As stated, real literals are of the anonymous predefined real type <i>universal{"_"}real</i>. Other real types have no literals. However, the overload resolution rules (see <a href="/docs/arm/AA-8/AA-8.6">8.6</a>) allow expressions of the type <i>universal{"_"}real</i> whenever a real type is expected.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>

#### Wording Changes from Ada 83

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>9.a_3.5.6</MarginText>
<Admonition type="aarm" aarm="note">
The syntax rule for <code><a href="/docs/arm/AA-3/AA-3.5#S0044">real_type_definition</a></code> is modified to use the new syntactic categories <code><a href="/docs/arm/AA-3/AA-3.5#S0045">floating_point_definition</a></code> and <code><a href="/docs/arm/AA-3/AA-3.5#S0047">fixed_point_definition</a></code>, instead of <code>floating{"_"}point{"_"}constraint</code> and <code>fixed{"_"}point{"_"}constraint</code>, because the semantics of a type definition are significantly different than the semantics of a constraint.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.b_3.5.6</MarginText>
<Admonition type="aarm" aarm="note">

</Admonition>

<a id="Subclause_3.5.7"></a>

## 3.5.7  Floating Point Types

<MarginText>1_3.5.7</MarginText>
<p>For floating point types, the error bound is specified as a relative precision by giving the required minimum number of significant decimal digits. <br /></p>


#### Syntax

<MarginText>2_3.5.7</MarginText>

<CodeBlock>
<code>floating{"_"}point{"_"}definition</code><a id="S0045"></a><code> ::= </code><br />    <b>digits</b> <i>static{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> [<code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code>]<br />

</CodeBlock>
<MarginText>3_3.5.7</MarginText>

<CodeBlock>
<code>real{"_"}range{"_"}specification</code><a id="S0046"></a><code> ::= </code><br />    <b>range</b> <i>static{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code> .. <i>static{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code><br />

</CodeBlock>

#### Name Resolution Rules

<MarginText>4_3.5.7</MarginText>
<p>The <i>requested decimal precision</i>, which is the minimum number of significant decimal digits required for the floating point type, is specified by the value of the <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> given after the reserved word <b>digits</b>. This <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> is expected to be of any integer type.<br /></p>

<MarginText>5/5_3.5.7</MarginText>
<MarginInfo items={["AI12-0444-1"]} />
<p>Each <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code> of a <code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code> is expected to be of any real type[; the types can be different]. <br /></p>


#### Legality Rules

<MarginText>6_3.5.7</MarginText>
<p>The requested decimal precision shall be specified by a static <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> whose value is positive and no greater than System.Max{"_"}Base{"_"}Digits. Each <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code> of a <code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code> shall also be static. If the <code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code> is omitted, the requested decimal precision shall be no greater than System.Max{"_"}Digits. <br /></p>

<AnnotatedOnly>
<MarginText>6.a_3.5.7</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>We have added Max{"_"}Base{"_"}Digits to package System. It corresponds to the requested decimal precision of <i>root{"_"}real</i>. System.Max{"_"}Digits corresponds to the maximum value for Digits that may be specified in the absence of a <code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code>, for upward compatibility. These might not be the same if <i>root{"_"}real</i> has a base range that does not include ± 10.0{"*"}{"*"}(4{"*"}Max{"_"}Base{"_"}Digits). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>7_3.5.7</MarginText>
<p>A <code><a href="/docs/arm/AA-3/AA-3.5#S0045">floating_point_definition</a></code> is illegal if the implementation does not support a floating point type that satisfies the requested decimal precision and range. <br /></p>

<AnnotatedOnly>
<MarginText>7.a_3.5.7</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<b></b>What combinations of requested decimal precision and range are supported for floating point types.<br />
</Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>8_3.5.7</MarginText>
<p>The set of values for a floating point type is the (infinite) set of rational numbers. The <i>machine numbers</i> of a floating point type are the values of the type that can be represented exactly in every unconstrained variable of the type. The base range (see <a href="/docs/arm/AA-3/AA-3.5">3.5</a>) of a floating point type is symmetric around zero, except that it can include some extra negative values in some implementations.<br /></p>

<AnnotatedOnly>
<MarginText>8.a_3.5.7</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>For example, if a 2's complement representation is used for the mantissa rather than a sign-mantissa or 1's complement representation, then there is usually one extra negative machine number.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.b_3.5.7</MarginText>
<Admonition type="aarm" aarm="note">
<b>To be honest: </b>If the Signed{"_"}Zeros attribute is True, then minus zero could in a sense be considered a value of the type. However, for most purposes, minus zero behaves the same as plus zero.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>9_3.5.7</MarginText>
<p>The <i>base decimal precision</i> of a floating point type is the number of decimal digits of precision representable in objects of the type. The <i>safe range</i> of a floating point type is that part of its base range for which the accuracy corresponding to the base decimal precision is preserved by all predefined operations. <br /></p>

<AnnotatedOnly>
<MarginText>9.a_3.5.7</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>In most cases, the safe range and base range are the same. However, for some hardware, values near the boundaries of the base range might result in excessive inaccuracies or spurious overflows when used with certain predefined operations. For such hardware, the safe range would omit such values.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>10/5_3.5.7</MarginText>
<MarginInfo items={["AI12-0439-1"]} />
<p>A <code><a href="/docs/arm/AA-3/AA-3.5#S0045">floating_point_definition</a></code> defines a floating point type whose base decimal precision is no less than the requested decimal precision. If a <code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code> is given, the safe range of the floating point type (and hence, also its base range) includes at least the values of the simple expressions given in the <code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code>. If a <code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code> is not given, the safe (and base) range of the type includes at least the values of the range &ndash;10.0{"*"}{"*"}(4{"*"}D) .. +10.0{"*"}{"*"}(4{"*"}D) where D is the requested decimal precision. [The safe range can include other values as well. The attributes Safe{"_"}First and Safe{"_"}Last give the actual bounds of the safe range.]<br /></p>

<MarginText>11_3.5.7</MarginText>
<p>A <code><a href="/docs/arm/AA-3/AA-3.5#S0045">floating_point_definition</a></code> also defines a first subtype of the type. If a <code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code> is given, then the subtype is constrained to a range whose bounds are given by a conversion of the values of the <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code>s of the <code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code> to the type being defined. Otherwise, the subtype is unconstrained.<br /></p>

<AnnotatedOnly>
<MarginText>11.a.1/1_3.5.7</MarginText>
<Admonition type="aarm" aarm="note">
<b>To be honest: </b>The conversion mentioned above is not an <i>implicit subtype conversion</i> (which is something that happens at overload resolution, see <a href="/docs/arm/AA-4/AA-4.6">4.6</a>), although it happens implicitly. Therefore, the freezing rules are not invoked on the type (which is important so that representation items can be given for the type). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>12_3.5.7</MarginText>
<p>There is a predefined, unconstrained, floating point subtype named Float[, declared in the visible part of package Standard]. <br /></p>


#### Dynamic Semantics

<MarginText>13_3.5.7</MarginText>
<p>[The elaboration of a <code><a href="/docs/arm/AA-3/AA-3.5#S0045">floating_point_definition</a></code> creates the floating point type and its first subtype.] <br /></p>


#### Implementation Requirements

<MarginText>14_3.5.7</MarginText>
<p>In an implementation that supports floating point types with 6 or more digits of precision, the requested decimal precision for Float shall be at least 6.<br /></p>

<MarginText>15_3.5.7</MarginText>
<p>If Long{"_"}Float is predefined for an implementation, then its requested decimal precision shall be at least 11. <br /></p>


#### Implementation Permissions

<MarginText>16/5_3.5.7</MarginText>
<MarginInfo items={["AI12-0444-1"]} />
<p>An implementation is allowed to provide additional predefined floating point types[, declared in the visible part of Standard], whose (unconstrained) first subtypes have names of the form Short{"_"}Float, Long{"_"}Float, Short{"_"}Short{"_"}Float, Long{"_"}Long{"_"}Float, etc. Different predefined floating point types are allowed to have the same base decimal precision. However, the precision of Float should be no greater than that of Long{"_"}Float. Similarly, the precision of Short{"_"}Float (if provided) should be no greater than Float. Corresponding recommendations apply to any other predefined floating point types. An implementation may support base decimal precisions for which there is no corresponding named floating point type. <br /></p>

</Admonition>

<MarginText>17_3.5.7</MarginText>

<p>A fixed point type is either an ordinary fixed point type, or a decimal fixed point type. The error bound of a fixed point type is specified as an absolute value, called the <i>delta</i> of the fixed point type. <br /></p>

#### Name Resolution Rules

<MarginText>6_3.5.9</MarginText>
<p>For a type defined by a <code><a href="/docs/arm/AA-3/AA-3.5#S0047">fixed_point_definition</a></code>, the <i>delta</i> of the type is specified by the value of the <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> given after the reserved word <b>delta</b>; this <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> is expected to be of any real type. For a type defined by a <code><a href="/docs/arm/AA-3/AA-3.5#S0049">decimal_fixed_point_definition</a></code> (a <i>decimal</i> fixed point type), the number of significant decimal digits for its first subtype (the <i>digits</i> of the first subtype) is specified by the <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> given after the reserved word <b>digits</b>; this <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> is expected to be of any integer type.<br /></p>

<MarginText>6.1/4_3.5.9</MarginText>
<MarginInfo items={["AI12-0159-1"]} />
<p>The <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code> of a <code><a href="/docs/arm/AA-3/AA-3.5#S0050">digits_constraint</a></code> is expected to be of any integer type. <br /></p>


#### Legality Rules

<MarginText>7_3.5.9</MarginText>
<p>In a <code><a href="/docs/arm/AA-3/AA-3.5#S0047">fixed_point_definition</a></code> or <code><a href="/docs/arm/AA-3/AA-3.5#S0050">digits_constraint</a></code>, the <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code>s given after the reserved words <b>delta</b> and <b>digits</b> shall be static; their values shall be positive.<br /></p>

<MarginText>8/2_3.5.9</MarginText>
<MarginInfo items={["AI95-00100-01"]} />
<p>The set of values of a fixed point type comprise the integral multiples of a number called the <i>small</i> of the type. The <i>machine numbers</i> of a fixed point type are the values of the type that can be represented exactly in every unconstrained variable of the type. For a type defined by an <code><a href="/docs/arm/AA-3/AA-3.5#S0048">ordinary_fixed_point_definition</a></code> (an <i>ordinary</i> fixed point type), the <i>small</i> may be specified by an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute{"_"}definition{"_"}clause</a></code> (see <a href="/docs/arm/AA-13/AA-13.3">13.3</a>); if so specified, it shall be no greater than the <i>delta</i> of the type. If not specified, the <i>small</i> of an ordinary fixed point type is an implementation-defined power of two less than or equal to the <i>delta</i>. <br /></p>

<AnnotatedOnly>
<MarginText>8.a_3.5.9</MarginText>
<Admonition type="aarm" aarm="implementation-defined">

</Admonition>
</AnnotatedOnly>

<MarginText>9_3.5.9</MarginText>
<p>For a decimal fixed point type, the <i>small</i> equals the <i>delta</i>; the <i>delta</i> shall be a power of 10. If a <code><a href="/docs/arm/AA-3/AA-3.5#S0046">real_range_specification</a></code> is given, both bounds of the range shall be in the range &ndash;(10{"*"}{"*"}<i>digits</i>&ndash;1){"*"}<i>delta</i> .. +(10{"*"}{"*"}<i>digits</i>&ndash;1){"*"}<i>delta</i>.<br /></p>

<p>A <code><a href="/docs/arm/AA-3/AA-3.5#S0047">fixed_point_definition</a></code> is illegal if the implementation does not support a fixed point type with the given <i>small</i> and specified range or <i>digits</i>. <br /></p>

#### Dynamic Semantics

<MarginText>17_3.5.9</MarginText>

<MarginText>18/4_3.5.9</MarginText>
<MarginInfo items={["AI12-0152-1"]} />
<p>For a <code><a href="/docs/arm/AA-3/AA-3.5#S0050">digits_constraint</a></code> on a decimal fixed point subtype with a given <i>delta</i>, if it does not have a <code><a href="/docs/arm/AA-3/AA-3.5#S0036">range_constraint</a></code>, then it specifies an implicit range &ndash;(10{"*"}{"*"}<i>D</i>&ndash;1){"*"}<i>delta</i> .. +(10{"*"}{"*"}<i>D</i>&ndash;1){"*"}<i>delta</i>, where <i>D</i> is the value of the <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code>. Consider the following example: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18.c_3.5.9</MarginText>
<CodeBlock language="ada">
type D is delta 0.01 digits 7 range -0.00 .. 9999.99;{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>18.d/1_3.5.9</MarginText>
<Admonition type="aarm" aarm="note">
The compatibility rule implies that the <code><a href="/docs/arm/AA-3/AA-3.5#S0050">digits_constraint</a></code> "<b>digits</b> 6" specifies an implicit range of "&ndash;9999.99 .. 9999.99". Thus, "<b>digits</b> 6" is not compatible with the constraint of D, but "<b>digits</b> 6 range 0.00 .. 9999.99" is compatible.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18.e/2_3.5.9</MarginText>
<MarginInfo items={["AI95-00114-01"]} />
<Admonition type="aarm" aarm="note">
A value of a scalar type belongs to a constrained subtype of the type if it belongs to the range of the subtype. Attributes like Digits and Delta have no effect on this fundamental rule. So the obsolescent forms of <code><a href="/docs/arm/AA-3/AA-3.5#S0050">digits_constraint</a></code>s and <code><a href="/docs/arm/AA-J/AA-J.3#S0367">delta_constraint</a></code>s that are called &ldquo;accuracy constraints&rdquo; in RM83 don't really represent constraints on the values of the subtype, but rather primarily affect compatibility of the &ldquo;constraint&rdquo; with the subtype being &ldquo;constrained&rdquo;. In this sense, they might better be called &ldquo;subtype assertions&rdquo; rather than &ldquo;constraints&rdquo;.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18.f_3.5.9</MarginText>
<Admonition type="aarm" aarm="note">
Note that the <code><a href="/docs/arm/AA-3/AA-3.5#S0050">digits_constraint</a></code> on a decimal fixed point subtype is a combination of an assertion about the <i>digits</i> of the subtype being further constrained, and a constraint on the range of the subtype being defined, either explicit or implicit. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>19/4_3.5.9</MarginText>
<MarginInfo items={["AI12-0152-1"]} />
<p>The elaboration of a <code><a href="/docs/arm/AA-3/AA-3.5#S0050">digits_constraint</a></code> consists of the elaboration of the <code><a href="/docs/arm/AA-3/AA-3.5#S0036">range_constraint</a></code>, if any. If a <code><a href="/docs/arm/AA-3/AA-3.5#S0036">range_constraint</a></code> is given, a check is made that the bounds of the range are both in the range &ndash;(10{"*"}{"*"}<i>D</i>&ndash;1){"*"}<i>delta</i> .. +(10{"*"}{"*"}<i>D</i>&ndash;1){"*"}<i>delta</i>, where <i>D</i> is the value of the (static) <code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code> given after the reserved word <b>digits</b>. If this check fails, Constraint{"_"}Error is raised. <br /></p>


#### Implementation Requirements

<MarginText>20_3.5.9</MarginText>
<p>The implementation shall support at least 24 bits of precision (including the sign bit) for fixed point types. <br /></p>

<AnnotatedOnly>
<MarginText>20.a_3.5.9</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>This is sufficient to represent Standard.Duration with a <i>small</i> no more than 50 milliseconds. <br />
</Admonition>
</AnnotatedOnly>


#### Implementation Permissions

<MarginText>21_3.5.9</MarginText>
<p>Implementations are permitted to support only <i>small</i>s that are a power of two. In particular, all decimal fixed point type declarations can be disallowed. Note however that conformance with the Information Systems Annex requires support for decimal <i>small</i>s, and decimal fixed point type declarations with <i>digits</i> up to at least 18. <br /></p>

<AnnotatedOnly>
<MarginText>21.a_3.5.9</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>The accuracy requirements for multiplication, division, and conversion (see <a href="/docs/arm/AA-G/AA-G.2#Subclause_G.2.1">G.2.1</a>, &ldquo;<a href="/docs/arm/AA-G/AA-G.2#Subclause_G.2.1">Model of Floating Point Arithmetic</a>&rdquo;) are such that support for arbitrary <i>small</i>s should be practical without undue implementation effort. Therefore, implementations should support fixed point types with arbitrary values for <i>small</i> (within reason). One reasonable limitation would be to limit support to fixed point types that can be converted to the most precise floating point type without loss of precision (so that Fixed{"_"}IO is implementable in terms of Float{"_"}IO). <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>22/5_3.5.9</MarginText>
<MarginInfo items={["AI12-0442-1"]} />
<Admonition type="aarm" aarm="note">
NOTE   The specified bounds themselves can be outside the base range of an ordinary fixed point type so that the range specification can be given in a natural way, such as: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>23_3.5.9</MarginText>
<CodeBlock language="ada">
   type Fraction is delta 2.0{"*"}{"*"}(-15) range -1.0 .. 1.0;{"\n"}  {"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>24/5_3.5.9</MarginText>
<MarginInfo items={["AI12-0442-1"]} />
<Admonition type="aarm" aarm="note">
With 2's complement hardware, such a type would typically have a signed 16-bit representation, using 1 bit for the sign and 15 bits for fraction, resulting in a base range of &ndash;1.0 .. 1.0&ndash;2.0{"*"}{"*"}(&ndash;15). <br />
</Admonition>
</AnnotatedOnly>


#### Examples

<MarginText>25_3.5.9</MarginText>

<MarginText>28_3.5.9</MarginText>

</CodeBlock>

## 3.5.10  Operations of Fixed Point Types


#### Static Semantics

<MarginText>1_3.5.10</MarginText>
<p>The following attributes are defined for every fixed point subtype S: <br /></p>

<MarginText>2/5_3.5.10</MarginText>
<dt><br/>S'Small</dt>
<MarginInfo items={["AI95-00054-01", "AI12-0367-1"]} />
<dl>
<dd>{"{"}<i>8652/0005</i>{"}"} S'Small denotes the <i>small</i> of the type of S. The value of this attribute is of the type <i>universal{"_"}real</i>. Small may be specified for nonderived ordinary fixed point types via an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute{"_"}definition{"_"}clause</a></code> (see <a href="/docs/arm/AA-13/AA-13.3">13.3</a>); the expression of such a clause shall be static and positive.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>2.a/3_3.5.10</MarginText>
<Admonition type="aarm" aarm="note">
<b>Aspect Description for </b><b>Small: </b>Scale factor for a fixed point type.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>3_3.5.10</MarginText>
<dt><br/>S'Delta</dt>
<dl>
<dd>S'Delta denotes the <i>delta</i> of the fixed point subtype S. The value of this attribute is of the type <i>universal{"_"}real</i>. <br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>3.a_3.5.10</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>The <i>delta</i> is associated with the <i>sub</i>type as opposed to the type, because of the possibility of an (obsolescent) <code><a href="/docs/arm/AA-J/AA-J.3#S0367">delta_constraint</a></code>.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>4_3.5.10</MarginText>
<dt><br/>S'Fore</dt>
<dl>
<dd>S'Fore yields the minimum number of characters needed before the decimal point for the decimal representation of any value of the subtype S, assuming that the representation does not include an exponent, but includes a one-character prefix that is either a minus sign or a space. (This minimum number does not include superfluous zeros or underlines, and is at least 2.) The value of this attribute is of the type <i>universal{"_"}integer</i>.<br /></dd>
<MarginText>5_3.5.10</MarginText>
<dt><br/>S'Aft</dt>
<dd>S'Aft yields the number of decimal digits needed after the decimal point to accommodate the <i>delta</i> of the subtype S, unless the <i>delta</i> of the subtype S is greater than 0.1, in which case the attribute yields the value one. [(S'Aft is the smallest positive integer N for which (10{"*"}{"*"}N){"*"}S'Delta is greater than or equal to one.)] The value of this attribute is of the type <i>universal{"_"}integer</i>. <br /></dd>
</dl>
<MarginText>6_3.5.10</MarginText>
<p>The following additional attributes are defined for every decimal fixed point subtype S: <br /></p>

<MarginText>7_3.5.10</MarginText>
<dt><br/>S'Digits</dt>
<dl>
<dd>S'Digits denotes the <i>digits</i> of the decimal fixed point subtype S, which corresponds to the number of decimal digits that are representable in objects of the subtype. The value of this attribute is of the type <i>universal{"_"}integer</i>. Its value is determined as follows: <br /></dd>
</dl>
<MarginText>8_3.5.10</MarginText>
<ul>
<li>For a first subtype or a subtype defined by a <code><a href="/docs/arm/AA-3/AA-3.2#S0027">subtype_indication</a></code> with a <code><a href="/docs/arm/AA-3/AA-3.5#S0050">digits_constraint</a></code>, the digits is the value of the expression given after the reserved word <b>digits</b>;<br /></li>
<MarginText>9/5_3.5.10</MarginText>
<MarginInfo items={["AI12-0426-1"]} />
<li>For a subtype defined by a <code><a href="/docs/arm/AA-3/AA-3.2#S0027">subtype_indication</a></code> without a <code><a href="/docs/arm/AA-3/AA-3.5#S0050">digits_constraint</a></code>, the digits of the subtype is the same as that of the subtype denoted by the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code> in the <code><a href="/docs/arm/AA-3/AA-3.2#S0027">subtype_indication</a></code>; <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>9.a_3.5.10</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>Although a decimal subtype can be both range-constrained and digits-constrained, the digits constraint is intended to control the Size attribute of the subtype. For decimal types, Size can be important because input/output of decimal types is so common. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>10_3.5.10</MarginText>
<ul>
<li>The digits of a base subtype is the largest integer <i>D</i> such that the range &ndash;(10{"*"}{"*"}<i>D</i>&ndash;1){"*"}<i>delta</i> .. +(10{"*"}{"*"}<i>D</i>&ndash;1){"*"}<i>delta</i> is included in the base range of the type.<br /></li>
</ul>
<MarginText>11_3.5.10</MarginText>
<dt><br/>S'Scale</dt>
<dl>
<dd>S'Scale denotes the <i>scale</i> of the subtype S, defined as the value N such that S'Delta = 10.0{"*"}{"*"}(&ndash;N). [The scale indicates the position of the point relative to the rightmost significant digits of values of subtype S.] The value of this attribute is of the type <i>universal{"_"}integer</i>. <br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>11.a_3.5.10</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>S'Scale is negative if S'Delta is greater than one. By contrast, S'Aft is always positive. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>12_3.5.10</MarginText>
<dt><br/>S'Round</dt>
<dl>
<dd>S'Round denotes a function with the following specification: <br /></dd>
</dl>
<MarginText>13_3.5.10</MarginText>
<CodeBlock language="ada">
function S'Round(X : universal{"_"}real){"\n"}  return S'Base{"\n"}

</CodeBlock>
<MarginText>14_3.5.10</MarginText>
<dl>
<dd>The function returns the value obtained by rounding X (away from 0, if X is midway between two values of the type of S). <br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>15_3.5.10</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 1   All subtypes of a fixed point type will have the same value for the Delta attribute, in the absence of <code><a href="/docs/arm/AA-J/AA-J.3#S0367">delta_constraint</a></code>s (see <a href="/docs/arm/AA-J/AA-J.3">J.3</a>).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16_3.5.10</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 2   S'Scale is not always the same as S'Aft for a decimal subtype; for example, if S'Delta = 1.0 then S'Aft is 1 while S'Scale is 0.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>17_3.5.10</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 3   The predefined operations of a fixed point type include the assignment operation, qualification, the membership tests, and explicit conversion to and from other numeric types. They also include the relational operators and the following predefined arithmetic operators: the binary and unary adding operators &ndash; and +, multiplying operators, and the unary operator <b>abs</b>.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18_3.5.10</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 4   As for all types, objects of a fixed point type have Size and Address attributes (see <a href="/docs/arm/AA-13/AA-13.3">13.3</a>). Other attributes of fixed point types are defined in <a href="/docs/arm/AA-A/AA-A.5#Subclause_A.5.4">A.5.4</a>. <br />
</Admonition>
