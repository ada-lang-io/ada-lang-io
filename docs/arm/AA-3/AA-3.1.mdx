<MarginText>1/5</MarginText>
<MarginInfo items={["AI12-0373-1"]} />
<p>The language defines several kinds of named <em>entities</em> that are declared by declarations. The entity's <em>name</em> is defined by the declaration, usually by a <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code>, but sometimes by a <code><a href="../AA-3/AA-3.5#S0040">defining_character_literal</a></code> or <code><a href="../AA-6/AA-6.1#S0203">defining_operator_symbol</a></code>. There are also entities that are not directly declared; some of these are elements of other entities, or are allocated dynamically. Such entities can be denoted using <code><a href="../AA-4/AA-4.1#S0096">indexed_component</a></code>, <code><a href="../AA-4/AA-4.1#S0098">selected_component</a></code>, or dereference <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s (see <a href="../AA-4/AA-4.1">4.1</a>).<br /></p>

<AnnotatedOnly>
<MarginText>1.a/5</MarginText>
<MarginInfo items={["AI12-0373-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>Some entities are always anonymous. For instance, a type is never named (the name represents the first subtype). We don't mention those here as this paragraph is about <em>named</em> entities. <br /></Admonition>
</AnnotatedOnly>

#### Static Semantics

<MarginText>5</MarginText>
<p>A <em>declaration</em> is a language construct that associates a name with (a view of) an entity. A declaration may appear explicitly in the program text (an <em>explicit</em> declaration), or may be supposed to occur at a given place in the text as a consequence of the semantics of another construct (an <em>implicit</em> declaration). <br /></p>

<AnnotatedOnly>
<MarginText>5.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>An implicit declaration generally declares a predefined or inherited operation associated with the definition of a type. This term is used primarily when allowing explicit declarations to override implicit declarations, as part of a type declaration. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>declaration</strong> &mdash; language construct that associates a name with (a view of) an entity<br /><br /></Admonition>
</AnnotatedOnly>

<MarginText>6/5</MarginText>
<MarginInfo items={["AI95-00318-02", "AI05-0255-1", "AI05-0277-1", "AI12-0061-1", "AI12-0308-1"]} />
<p>Each of the following is defined to be a declaration: any <code><a href="../AA-3/AA-3.1#S0021">basic_declaration</a></code>; an <code><a href="../AA-3/AA-3.5#S0039">enumeration_literal_specification</a></code>; a <code><a href="../AA-3/AA-3.7#S0062">discriminant_specification</a></code>; a <code><a href="../AA-3/AA-3.8#S0070">component_declaration</a></code>; a <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code> of an <code><a href="../AA-4/AA-4.3#S0119">iterated_component_association</a></code>; a <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code>; a <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code> of a <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code>; an <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code>; a <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code> of an <code><a href="../AA-5/AA-5.5#S0186">iterator_parameter_specification</a></code>; a <code><a href="../AA-6/AA-6.1#S0207">parameter_specification</a></code>; a <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code>; an <code><a href="../AA-6/AA-6.5#S0224">extended_return_object_declaration</a></code>; an <code><a href="../AA-9/AA-9.5#S0257">entry_declaration</a></code>; an <code><a href="../AA-9/AA-9.5#S0263">entry_index_specification</a></code>; a <code><a href="../AA-11/AA-11.2#S0306">choice_parameter_specification</a></code>; a <code><a href="../AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This list (when <code><a href="../AA-3/AA-3.1#S0021">basic_declaration</a></code> is expanded out) contains all syntactic categories that end in "_declaration" or "_specification", except for program unit _specifications. Moreover, it contains <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code>. A <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> is a declaration, whether or not it completes a previous declaration. This is a bit strange, <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> is not part of the syntax of <code><a href="../AA-3/AA-3.1#S0021">basic_declaration</a></code> or <code><a href="../AA-10/AA-10.1#S0288">library_unit_declaration</a></code>. A renaming-as-body is considered a declaration. An <code><a href="../AA-9/AA-9.5#S0258">accept_statement</a></code> is not considered a declaration. Completions are sometimes declarations, and sometimes not. <br /></Admonition>
</AnnotatedOnly>

<MarginText>7</MarginText>
<p>All declarations contain a <em>definition</em> for a <em>view</em> of an entity. A view consists of an identification of the entity (the entity <em>of</em> the view), plus view-specific characteristics that affect the use of the entity through that view (such as mode of access to an object, formal parameter names and defaults for a subprogram, or visibility to components of a type). In most cases, a declaration also contains the definition for the entity itself (a <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code> is an example of a declaration that does not define a new entity, but instead defines a view of an existing entity (see <a href="../AA-8/AA-8.5">8.5</a>)).<br /></p>

<AnnotatedOnly>
<MarginText>7.a.1/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>view of an entity</strong> &mdash; representation of an entity that reveals some or all of the properties of the entity<br /><br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Most declarations define a view (of some entity) whose view-specific characteristics are unchanging for the life of the view. However, subtypes are somewhat unusual in that they inherit characteristics from whatever view of their type is currently visible. Hence, a subtype is not a <em>view</em> of a type; it is more of an indirect reference. By contrast, a private type provides a single, unchanging (partial) view of its full type. <br /></Admonition>
</AnnotatedOnly>

<MarginText>7.1/3</MarginText>
<MarginInfo items={["AI05-0080-1"]} />
<p>When it is clear from context, the term <em>object</em> is used in place of <em>view of an object</em>. Similarly, the terms <em>type</em> and <em>subtype</em> are used in place of <em>view of a type</em> and <em>view of a subtype</em>, respectively.<br /></p>

<AnnotatedOnly>
<MarginText>7.b/3</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Rules interpreted at compile time generally refer to views of entities, rather than the entities themselves. This is necessary to preserve privacy; characteristics that are not visible should not be used in compile-time rules. Thus, Static Semantics and Legality Rules generally implicitly have &ldquo;view of&rdquo;. Legality Rules that need to look into the private part are the exception to this interpretation.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.c/3</MarginText>
<Admonition type="aarm" aarm="note">
On the other hand, run-time rules can work either way, so &ldquo;view of&rdquo; should not be assumed in Dynamic Semantics rules.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.d/5</MarginText>
<MarginInfo items={["AI12-0191-1"]} />
<Admonition type="aarm" aarm="note">
For example, a reference to the components of an object in a rule that is interpreted at compile time would not apply to components that are not visible. On the other hand, a reference to the components of an object in a dynamic semantics rule would apply to all components of the object, visible or not, including (for tagged objects) components which are not components of the nominal type of the object (see <a href="../AA-3/AA-3.9#Subclause_3.9.1">3.9.1</a>). Other terms, such as &ldquo;subcomponent&rdquo; and &ldquo;part&rdquo;, are interpreted analogously. <br /></Admonition>
</AnnotatedOnly>

<MarginText>8</MarginText>
<p>For each declaration, the language rules define a certain region of text called the <em>scope</em> of the declaration (see <a href="../AA-8/AA-8.2">8.2</a>). Most declarations associate an <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> with a declared entity. Within its scope, and only there, there are places where it is possible to use the <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> to refer to the declaration, the view it defines, and the associated entity; these places are defined by the visibility rules (see <a href="../AA-8/AA-8.3">8.3</a>). At such places the <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> is said to be a <em>name</em> of the entity (the <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code> or <code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code>); the name is said to <em>denote</em> the declaration, the view, and the associated entity (see <a href="../AA-8/AA-8.6">8.6</a>). The declaration is said to <em>declare</em> the name, the view, and in most cases, the entity itself.<br /></p>

<MarginText>9</MarginText>
<p>As an alternative to an <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code>, an enumeration literal can be declared with a <code><a href="../AA-2/AA-2.5#S0015">character_literal</a></code> as its name (see <a href="../AA-3/AA-3.5#Subclause_3.5.1">3.5.1</a>), and a function can be declared with an <code><a href="../AA-6/AA-6.1#S0202">operator_symbol</a></code> as its name (see <a href="../AA-6/AA-6.1">6.1</a>).<br /></p>

<MarginText>10</MarginText>
<p>The syntax rules use the terms <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code>, <code><a href="../AA-3/AA-3.5#S0040">defining_character_literal</a></code>, and <code><a href="../AA-6/AA-6.1#S0203">defining_operator_symbol</a></code> for the defining occurrence of a name; these are collectively called <em>defining names</em>. The terms <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code> and <code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code> are used for usage occurrences of <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code>s, <code><a href="../AA-2/AA-2.5#S0015">character_literal</a></code>s, and <code><a href="../AA-6/AA-6.1#S0202">operator_symbol</a></code>s. These are collectively called <em>usage names</em>. <br /></p>

<AnnotatedOnly>
<MarginText>10.a</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>The terms <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code>, <code><a href="../AA-2/AA-2.5#S0015">character_literal</a></code>, and <code><a href="../AA-6/AA-6.1#S0202">operator_symbol</a></code> are used directly in contexts where the normal visibility rules do not apply (such as the <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> that appears after the <strong>end</strong> of a <code><a href="../AA-9/AA-9.1#S0248">task_body</a></code>). Analogous conventions apply to the use of <code><a href="../AA-6/AA-6.1#S0199">designator</a></code>, which is the collective term for <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> and <code><a href="../AA-6/AA-6.1#S0202">operator_symbol</a></code>. <br /></Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>11</MarginText>
<p>The process by which a construct achieves its run-time effect is called <em>execution</em>. This process is also called <em>elaboration</em> for declarations and <em>evaluation</em> for expressions. One of the terms execution, elaboration, or evaluation is defined by this Reference Manual for each construct that has a run-time effect. <br /></p>

<AnnotatedOnly>
<MarginText>11.a.1/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>execution</strong> &mdash; process by which a construct achieves its run-time effect<br /><br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.a</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>The term elaboration is also used for the execution of certain constructs that are not declarations, and the term evaluation is used for the execution of certain constructs that are not expressions. For example, <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code>s are elaborated, and <code><a href="../AA-3/AA-3.5#S0037">range</a></code>s are evaluated.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.b</MarginText>
<Admonition type="aarm" aarm="note">
For bodies, execution and elaboration are both explicitly defined. When we refer specifically to the execution of a body, we mean the explicit definition of execution for that kind of body, not its elaboration. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.c</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Technically, "the execution of a declaration" and "the elaboration of a declaration" are synonymous. We use the term "elaboration" of a construct when we know the construct is elaborable. When we are talking about more arbitrary constructs, we use the term "execution". For example, we use the term "erroneous execution", to refer to any erroneous execution, including erroneous elaboration or evaluation.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.d</MarginText>
<Admonition type="aarm" aarm="note">
When we explicitly define evaluation or elaboration for a construct, we are implicitly defining execution of that construct.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.e</MarginText>
<Admonition type="aarm" aarm="note">
We also use the term "execution" for things like <code><a href="../AA-5/AA-5.1#S0167">statement</a></code>s, which are executable, but neither elaborable nor evaluable. We considered using the term "execution" only for nonelaborable, nonevaluable constructs, and defining the term "action" to mean what we have defined "execution" to mean. We rejected this idea because we thought three terms that mean the same thing was enough &mdash; four would be overkill. Thus, the term "action" is used only informally in the standard (except where it is defined as part of a larger term, such as "protected action"). <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.e.1/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>elaboration</strong> &mdash; process by which a declaration achieves its run-time effect<br /><br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.e.2/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>evaluation</strong> &mdash; process by which an expression achieves its run-time effect<br /><br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.f</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>A construct is <em>elaborable</em> if elaboration is defined for it. A construct is <em>evaluable</em> if evaluation is defined for it. A construct is <em>executable</em> if execution is defined for it. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.g</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Don't confuse &ldquo;elaborable&rdquo; with &ldquo;preelaborable&rdquo; (defined in <a href="../AA-10/AA-10.2#Subclause_10.2.1">10.2.1</a>).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.h/2</MarginText>
<MarginInfo items={["AI95-00114-01"]} />
<Admonition type="aarm" aarm="note">
Evaluation of an evaluable construct produces a result that is either a value, a denotation, or a range. The following are evaluable: expression; <code><a href="../AA-4/AA-4.1#S0091">name</a></code> <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code>; <code><a href="../AA-3/AA-3.5#S0037">range</a></code>; <code><a href="../AA-9/AA-9.5#S0263">entry_index_specification</a></code>; and possibly <code><a href="../AA-3/AA-3.6#S0058">discrete_range</a></code>. The last one is curious &mdash; RM83 uses the term &ldquo;evaluation of a <code><a href="../AA-3/AA-3.6#S0058">discrete_range</a></code>&rdquo;, but never defines it. One might presume that the evaluation of a <code><a href="../AA-3/AA-3.6#S0058">discrete_range</a></code> consists of the evaluation of the <code><a href="../AA-3/AA-3.5#S0037">range</a></code> or the <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code>, depending on what it is. But <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code>s are not evaluated; they are elaborated.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.i</MarginText>
<Admonition type="aarm" aarm="note">
Intuitively, an <em>executable</em> construct is one that has a defined run-time effect (which may be null). Since execution includes elaboration and evaluation as special cases, all elaborable and all evaluable constructs are also executable. Hence, most constructs in Ada are executable. An important exception is that the constructs inside a generic unit are not executable directly, but rather are used as a template for (generally) executable constructs in instances of the generic. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   At compile time, the declaration of an entity <em>declares</em> the entity. At run time, the elaboration of the declaration <em>creates</em> the entity. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Syntactic categories for declarations are named either <em>entity_</em><code>declaration</code> (if they include a trailing semicolon) or <em>entity_</em><code>specification</code> (if not).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.b</MarginText>
<Admonition type="aarm" aarm="note">
The various kinds of named entities that can be declared are as follows: an object (including components and parameters), a named number, a type (the name always refers to its first subtype), a subtype, a subprogram (including enumeration literals and operators), a single entry, an entry family, a package, a protected or task unit (which corresponds to either a type or a single object), an exception, a generic unit, a label, and the name of a statement.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.c</MarginText>
<Admonition type="aarm" aarm="note">
Identifiers are also associated with names of pragmas, arguments to pragmas, and with attributes, but these are not user-definable. <br /></Admonition>
</AnnotatedOnly>

<MarginText>12.i</MarginText>
<Admonition type="aarm" aarm="note">
For declarations that come in &ldquo;two parts&rdquo; (program unit declaration plus body, private or incomplete type plus full type, deferred constant plus full constant), we consider both to be defining occurrences. Thus, for example, the syntax for <code><a href="../AA-7/AA-7.2#S0231">package_body</a></code> uses <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code> after the reserved word <strong>body</strong>, as opposed to <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code>.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.j</MarginText>
<Admonition type="aarm" aarm="note">
The defining occurrence of a statement name is in its implicit declaration, not where it appears in the program text. Considering the statement name itself to be the defining occurrence would complicate the visibility rules.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.k</MarginText>
<Admonition type="aarm" aarm="note">
The phrase &ldquo;visible by selection&rdquo; is not used in Ada 95. It is subsumed by simply &ldquo;visible&rdquo; and the Name Resolution Rules for <code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code>s.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.l/3</MarginText>
<MarginInfo items={["AI05-0299-1"]} />
<Admonition type="aarm" aarm="note">
(Note that in Ada 95, a declaration is visible at all places where one could have used a <code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code>, not just at places where a <code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code> was actually used. Thus, the places where a declaration is directly visible are a subset of the places where it is visible. See Clause <a href="../AA-8/">8</a> for details.)<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.m</MarginText>
<Admonition type="aarm" aarm="note">
We use the term &ldquo;declaration&rdquo; to cover <code>_specification</code>s that declare (views of) objects, such as <code><a href="../AA-6/AA-6.1#S0207">parameter_specification</a></code>s. In Ada 83, these are referred to as a &ldquo;form of declaration&rdquo;, but it is not entirely clear that they are considered simply &ldquo;declarations&rdquo;.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.n/3</MarginText>
<MarginInfo items={["AI05-0299-1"]} />
<Admonition type="aarm" aarm="note">
RM83 contains an incomplete definition of "elaborated" in this subclause: it defines "elaborated" for declarations, <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code>s, <code><a href="../AA-3/AA-3.11#S0087">declarative_item</a></code>s and <code><a href="../AA-10/AA-10.1#S0286">compilation_unit</a></code>s, but "elaboration" is defined elsewhere for various other constructs. To make matters worse, Ada 95 has a different set of elaborable constructs. Instead of correcting the list, it is more maintainable to refer to the term "elaborable," which is defined in a distributed manner.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.o</MarginText>
<Admonition type="aarm" aarm="note">
RM83 uses the term &ldquo;has no other effect&rdquo; to describe an elaboration that doesn't do anything except change the state from not-yet-elaborated to elaborated. This was a confusing wording, because the answer to &ldquo;other than what?&rdquo; was to be found many pages away. In Ada 95, we change this wording to &ldquo;has no effect&rdquo; (for things that truly do nothing at run time), and &ldquo;has no effect other than to establish that so-and-so can happen without failing the Elaboration_Check&rdquo; (for things where it matters).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.p</MarginText>
<Admonition type="aarm" aarm="note">
We make it clearer that the term "execution" covers elaboration and evaluation as special cases. This was implied in RM83. For example, "erroneous execution" can include any execution, and RM83-9.4(3) has, "The task designated by any other task object depends on the master whose execution creates the task object;" the elaboration of the master's <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code> is doing the task creation. <br /></Admonition>
</AnnotatedOnly>
