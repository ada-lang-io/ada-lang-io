#### Static Semantics

<MarginText>6.1/2</MarginText>
<MarginInfo items={["AI95-00420-01"]} />
<p>The list of <code><a href="../AA-3/AA-3.11#S0087">declarative_item</a></code>s of a <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code> is called the <em>declaration list</em> of the <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code>. <br /></p>


#### Dynamic Semantics

<MarginText>7</MarginText>
<p>The elaboration of a <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code> consists of the elaboration of the <code><a href="../AA-3/AA-3.11#S0087">declarative_item</a></code>s, if any, in the order in which they are given in the <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code>.<br /></p>

<MarginText>8</MarginText>
<p>An elaborable construct is in the <em>elaborated</em> state after the normal completion of its elaboration. Prior to that, it is <em>not yet elaborated</em>. <br /></p>

<AnnotatedOnly>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The elaborated state is only important for bodies; certain uses of a body raise an exception if the body is not yet elaborated.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.b</MarginText>
<Admonition type="aarm" aarm="note">
Note that "prior" implies before the start of elaboration, as well as during elaboration.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.c</MarginText>
<Admonition type="aarm" aarm="note">
The use of the term "normal completion" implies that if the elaboration propagates an exception or is aborted, the declaration is not elaborated. RM83 missed the aborted case. <br /></Admonition>
</AnnotatedOnly>

<MarginText>9</MarginText>
<p>For a construct that attempts to use a body, a check (Elaboration_Check) is performed, as follows: <br /></p>

<MarginText>10/1</MarginText>
<MarginInfo items={["AI95-00064-01"]} />
<ul>
<li>{"{"}<em>8652/0014</em>{"}"} For a call to a (non-protected) subprogram that has an explicit body, a check is made that the body is already elaborated. This check and the evaluations of any actual parameters of the call are done in an arbitrary order. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>10.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>AI83-00180 specifies that there is no elaboration check for a subprogram defined by a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Interface (or equivalently, <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Import). AI83-00430 specifies that there is no elaboration check for an enumeration literal. AI83-00406 specifies that the evaluation of parameters and the elaboration check occur in an arbitrary order. AI83-00406 applies to generic instantiation as well (see below).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.a.1/3</MarginText>
<MarginInfo items={["AI95-00064-01", "AI05-0177-1"]} />
<Admonition type="aarm" aarm="note">
{"{"}<em>8652/0014</em>{"}"} A subprogram can be completed by a renaming-as-body, a <code><a href="../AA-6/AA-6.7#S0227">null_procedure_declaration</a></code>, or an <code><a href="../AA-6/AA-6.8#S0228">expression_function_declaration</a></code>, and we need to make an elaboration check on such a body, so we use &ldquo;body&rdquo; rather than <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> above. <br /></Admonition>
</AnnotatedOnly>

<MarginText>11/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<ul>
<li>For a call to a protected operation of a protected type (that has a body &mdash; no check is performed if  the protected type is imported &mdash; see <a href="../AA-B/AA-B.1">B.1</a>), a check is made that the <code><a href="../AA-9/AA-9.4#S0254">protected_body</a></code> is already elaborated. This check and the evaluations of any actual parameters of the call are done in an arbitrary order. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>11.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>A protected type has only one elaboration &ldquo;bit&rdquo;, rather than one for each operation, because one call may result in evaluating the barriers of other entries, and because there are no elaborable declarations between the bodies of the operations. In fact, the elaboration of a <code><a href="../AA-9/AA-9.4#S0254">protected_body</a></code> does not elaborate the enclosed bodies, since they are not considered independently elaborable.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.b</MarginText>
<Admonition type="aarm" aarm="note">
Note that there is no elaboration check when calling a task entry. Task entry calls are permitted even before the associated <code><a href="../AA-9/AA-9.1#S0248">task_body</a></code> has been seen. Such calls are simply queued until the task is activated and reaches a corresponding <code><a href="../AA-9/AA-9.5#S0258">accept_statement</a></code>. We considered a similar rule for protected entries &mdash; simply queuing all calls until the <code><a href="../AA-9/AA-9.4#S0254">protected_body</a></code> was seen, but felt it was not worth the possible implementation overhead, particularly given that there might be multiple instances of the protected type. <br /></Admonition>
</AnnotatedOnly>

<MarginText>12</MarginText>
<ul>
<li>For the activation of a task, a check is made by the activator that the <code><a href="../AA-9/AA-9.1#S0248">task_body</a></code> is already elaborated. If two or more tasks are being activated together (see <a href="../AA-9/AA-9.2">9.2</a>), as the result of the elaboration of a <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code> or the initialization for the object created by an allocator, this check is done for all of them before activating any of them. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>12.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>As specified by AI83-00149, the check is done by the activator, rather than by the task itself. If it were done by the task itself, it would be turned into a Tasking_Error in the activator, and the other tasks would still be activated. <br /></Admonition>
</AnnotatedOnly>

<MarginText>13</MarginText>
<ul>
<li>For the instantiation of a generic unit that has a body, a check is made that this body is already elaborated. This check and the evaluation of any <code><a href="../AA-12/AA-12.3#S0318">explicit_generic_actual_parameter</a></code>s of the instantiation are done in an arbitrary order. <br /></li>
</ul>
<MarginText>14</MarginText>
<p>The exception Program_Error is raised if any of these checks fails. <br /></p>

<a id="Subclause_3.11.1"></a>

## 3.11.1  Completions of Declarations

<MarginText>1/3</MarginText>
<MarginInfo items={["AI95-00064-01", "AI05-0177-1"]} />
<p>{"{"}<em>8652/0014</em>{"}"} Declarations sometimes come in two parts. A declaration that requires a second part is said to <em>require completion</em>. The second part is called the <em>completion</em> of the declaration (and of the entity declared), and is either another declaration, a body, or a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>. A <em>body</em> is a <code><a href="../AA-3/AA-3.11#S0089">body</a></code>, an <code><a href="../AA-9/AA-9.5#S0260">entry_body</a></code>, a <code><a href="../AA-6/AA-6.7#S0227">null_procedure_declaration</a></code> or an <code><a href="../AA-6/AA-6.8#S0228">expression_function_declaration</a></code> that completes another declaration, or a renaming-as-body (see <a href="../AA-8/AA-8.5#Subclause_8.5.4">8.5.4</a>). <br /></p>

<AnnotatedOnly>
<MarginText>1.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Throughout the RM95, there are rules about completions that define the following: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>1.b</MarginText>
<ul>
<li>Which declarations require a corresponding completion.<br /></li>
<MarginText>1.c</MarginText>
<li>Which constructs can only serve as the completion of a declaration.<br /></li>
<MarginText>1.d</MarginText>
<li>Where the completion of a declaration is allowed to be.<br /></li>
<MarginText>1.e</MarginText>
<li>What kinds of completions are allowed to correspond to each kind of declaration that allows one. <br /></li>
</ul>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>1.f</MarginText>
<Admonition type="aarm" aarm="note">
Don't confuse this compile-time concept with the run-time concept of completion defined in <a href="../AA-7/AA-7.6#Subclause_7.6.1">7.6.1</a>.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>1.g</MarginText>
<Admonition type="aarm" aarm="note">
Note that the declaration of a private type (if limited) can be completed with the declaration of a task type, which is then completed with a body. Thus, a declaration can actually come in <em>three</em> parts.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>1.h/3</MarginText>
<MarginInfo items={["AI95-00217-06", "AI05-0162-1"]} />
<Admonition type="aarm" aarm="note">
An incomplete type (whether declared in the limited view of a package or not) may be completed by a private type declaration, so we can in fact have <em>four</em> parts.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>1.i/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<Admonition type="aarm" aarm="note">
In Ada 2012, there are no language-defined pragmas that act as completions. Pragma Import (which is obsolescent) has the effect of setting aspect Import to True; such an aspect makes giving a completion illegal. The wording that allows pragmas as completions was left as it is harmless and appears in many places in this Reference Manual. <br /></Admonition>
</AnnotatedOnly>


#### Name Resolution Rules

<MarginText>2</MarginText>
<p>A construct that can be a completion is interpreted as the completion of a prior declaration only if: <br /></p>

<MarginText>3</MarginText>
<ul>
<li>The declaration and the completion occur immediately within the same declarative region;<br /></li>
<MarginText>4</MarginText>
<li>The defining name or <code><a href="../AA-6/AA-6.1#S0201">defining_program_unit_name</a></code> in the completion is the same as in the declaration, or in the case of a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>, the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> applies to the declaration;<br /></li>
<MarginText>5</MarginText>
<li>If the declaration is overloadable, then the completion either has a type-conformant profile, or is a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>. <br /></li>
</ul>

#### Legality Rules

<MarginText>6/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<p>An implicit declaration shall not have a completion. For any explicit declaration that is specified to <em>require completion</em>, there shall be a corresponding explicit completion, unless the declared entity is imported (see <a href="../AA-B/AA-B.1">B.1</a>). <br /></p>

<AnnotatedOnly>
<MarginText>6.a.1/2</MarginText>
<MarginInfo items={["AI95-00217-06"]} />
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>The implicit declarations occurring in a limited view do have a completion (the explicit declaration occurring in the full view) but that's a special case, since the implicit declarations are actually built from the explicit ones. So they do not <em>require</em> a completion, they have one by <em>fiat</em>. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The implicit declarations of predefined operators are not allowed to have a completion. Enumeration literals, although they are subprograms, are not allowed to have a corresponding <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code>. That's because the completion rules are described in terms of constructs (<code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code>s) and not entities (subprograms). When a completion is required, it has to be explicit; the implicit null <code><a href="../AA-7/AA-7.2#S0231">package_body</a></code> that Clause <a href="../AA-7/">7</a> talks about cannot serve as the completion of a <code><a href="../AA-7/AA-7.1#S0229">package_declaration</a></code> if a completion is required. <br /></Admonition>
</AnnotatedOnly>

<MarginText>7</MarginText>
<p>At most one completion is allowed for a given declaration. Additional requirements on completions appear where each kind of completion is defined. <br /></p>

<AnnotatedOnly>
<MarginText>7.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>A subunit is not a completion; the stub is.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.b</MarginText>
<Admonition type="aarm" aarm="note">
If the completion of a declaration is also a declaration, then <em>that</em> declaration might have a completion, too. For example, a limited private type can be completed with a task type, which can then be completed with a task body. This is not a violation of the &ldquo;at most one completion&rdquo; rule. <br /></Admonition>
</AnnotatedOnly>

<MarginText>8</MarginText>
<p>A type is <em>completely defined</em> at a place that is after its full type definition (if it has one) and after all of its subcomponent types are completely defined. A type shall be completely defined before it is frozen (see <a href="../AA-13/AA-13.14">13.14</a> and <a href="../AA-7/AA-7.3">7.3</a>). <br /></p>

<AnnotatedOnly>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>Index types are always completely defined &mdash; no need to mention them. There is no way for a completely defined type to depend on the value of a (still) deferred constant. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 1   Completions are in principle allowed for any kind of explicit declaration. However, for some kinds of declaration, the only allowed completion is an implementation-defined pragma, and implementations are not required to have any such pragmas. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.a/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<Admonition type="aarm" aarm="note">
<em>This paragraph was deleted.</em><br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10/5</MarginText>
<MarginInfo items={["AI12-0449-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 2   There are rules that prevent premature uses of declarations that have a corresponding completion. The Elaboration_Checks of <a href="../AA-3/AA-3.11">3.11</a> prevent such uses at run time for subprograms, protected operations, tasks, and generic units. The freezing rules (see <a href="../AA-13/AA-13.14">13.14</a>)  prevent, at compile time, premature uses of other entities such as private types and deferred constants. <br /></Admonition>
</AnnotatedOnly>
