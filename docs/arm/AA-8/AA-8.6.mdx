<MarginText>1/3</MarginText>
<MarginInfo items={["AI05-0299-1"]} />
<p>[ Because declarations can be overloaded, it is possible for an occurrence of a usage name to have more than one possible interpretation; in most cases, ambiguity is disallowed. This subclause describes how the possible interpretations resolve to the actual interpretation.<br /></p>

<MarginText>2</MarginText>
<p>Certain rules of the language (the Name Resolution Rules) are considered &ldquo;overloading rules&rdquo;. If a possible interpretation violates an overloading rule, it is assumed not to be the intended interpretation; some other possible interpretation is assumed to be the actual interpretation. On the other hand, violations of nonoverloading rules do not affect which interpretation is chosen; instead, they cause the construct to be illegal. To be legal, there usually has to be exactly one acceptable interpretation of a construct that is a &ldquo;complete context&rdquo;, not counting any nested complete contexts.<br /></p>

<MarginText>3</MarginText>
<p>The syntax rules of the language and the visibility rules given in <a href="/docs/arm/AA-8/AA-8.3">8.3</a> determine the possible interpretations. Most type checking rules (rules that require a particular type, or a particular class of types, for example) are overloading rules. Various rules for the matching of formal and actual parameters are overloading rules.] <br /></p>


#### Language Design Principles

<AnnotatedOnly>
<MarginText>3.a</MarginText>
<Admonition type="aarm" aarm="note">
The type resolution rules are intended to minimize the need for implicit declarations and preference rules associated with implicit conversion and dispatching operations. <br />
</Admonition>
</AnnotatedOnly>


#### Name Resolution Rules

<MarginText>11</MarginText>
<ul>
<li>for each constituent of the complete context, to which syntactic categories it belongs, and by which syntax rules; and <br /></li>
</ul>

<MarginText>11.a</MarginText>

Basically, we're trying to capture all the information in the parse tree here, without using compiler-writer's jargon like &ldquo;parse tree&rdquo;. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>12</MarginText>
<ul>
<li>for each usage name, which declaration it denotes (and, therefore, which view and which entity it denotes); and <br /></li>

<Admonition type="aarm" aarm="ramification">

For example, within a <code><a href="/docs/arm/AA-9/AA-9.1#S0248">task_body</a></code> other than in an <code><a href="/docs/arm/AA-3/AA-3.10#S0084">access_definition</a></code>, a usage name that denotes the <code><a href="/docs/arm/AA-9/AA-9.1#S0244">task_type_declaration</a></code> denotes the object containing the currently executing task, and not the task type declared by the declaration. <br />
</Admonition>
</AnnotatedOnly>

<ul>
<li>for a complete context that is a <code><a href="/docs/arm/AA-3/AA-3.11#S0087">declarative_item</a></code>, whether or not it is a completion of a declaration, and (if so) which declaration it completes. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>13.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Unfortunately, we are not confident that the above list is complete. We'll have to live with that. <br />
</Admonition>

<AnnotatedOnly>
<MarginText>13.b</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>For &ldquo;possible&rdquo; interpretations, the above information is tentative. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>13.c</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>A possible interpretation (an <em>input</em> to overload resolution) contains information about what a usage name <em>might</em> denote, but what it actually <em>does</em> denote requires overload resolution to determine. Hence the term &ldquo;tentative&rdquo; is needed for possible interpretations; otherwise, the definition would be circular. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>14</MarginText>
<p>A <em>possible interpretation</em> is one that obeys the syntax rules and the visibility rules. An <em>acceptable interpretation</em> is a possible interpretation that obeys the <em>overloading rules</em>[, that is, those rules that specify an expected type or expected profile, or specify how a construct shall <em>resolve</em> or be <em>interpreted</em>.] <br /></p>

<AnnotatedOnly>
<MarginText>14.a</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>One rule that falls into this category, but does not use the above-mentioned magic words, is the rule about numbers of parameter associations in a call (see <a href="/docs/arm/AA-6/AA-6.4">6.4</a>). <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The Name Resolution Rules are the ones that appear under the Name Resolution Rules heading. Some Syntax Rules are written in English, instead of BNF. No rule is a Syntax Rule or Name Resolution Rule unless it appears under the appropriate heading. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>15</MarginText>
<p>The <em>interpretation</em> of a constituent of a complete context is determined from the overall interpretation of the complete context as a whole. [Thus, for example, &ldquo;interpreted as a <code><a href="/docs/arm/AA-6/AA-6.4#S0218">function_call</a></code>&rdquo;, means that the construct's interpretation says that it belongs to the syntactic category <code><a href="/docs/arm/AA-6/AA-6.4#S0218">function_call</a></code>.]<br /></p>

<MarginText>16</MarginText>
<p>[Each occurrence of] a usage name <em>denotes</em> the declaration determined by its interpretation. It also denotes the view declared by its denoted declaration, except in the following cases: <br /></p>

<AnnotatedOnly>
<MarginText>16.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>As explained below, a pragma argument is allowed to be ambiguous, so it can denote several declarations, and all of the views declared by those declarations. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>17/3</MarginText>
<MarginInfo items={["AI95-00382-01", "AI05-0287-1"]} />
<ul>
<li>If a usage name appears within the declarative region of a <code><a href="/docs/arm/AA-3/AA-3.2#S0023">type_declaration</a></code> and denotes that same <code><a href="/docs/arm/AA-3/AA-3.2#S0023">type_declaration</a></code>, then it denotes the <em>current instance</em> of the type (rather than the type itself); the current instance of a type is the object or value of the type that is associated with the execution that evaluates the usage name.  Similarly, if a usage name appears within the declarative region of a <code><a href="/docs/arm/AA-3/AA-3.2#S0026">subtype_declaration</a></code> and denotes that same <code><a href="/docs/arm/AA-3/AA-3.2#S0026">subtype_declaration</a></code>, then it denotes the current instance of the subtype. These rules do not apply if the usage name appears within the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code> of an <code><a href="/docs/arm/AA-3/AA-3.10#S0084">access_definition</a></code> for an access-to-object type, or within the subtype of a parameter or result of an access-to-subprogram type. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>17.a/2</MarginText>
<MarginInfo items={["AI95-00382-01"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>This is needed, for example, for references to the Access attribute from within the <code><a href="/docs/arm/AA-3/AA-3.2#S0023">type_declaration</a></code>. Also, within a <code><a href="/docs/arm/AA-9/AA-9.1#S0248">task_body</a></code> or <code><a href="/docs/arm/AA-9/AA-9.4#S0254">protected_body</a></code>, we need to be able to denote the current task or protected object. (For a <code><a href="/docs/arm/AA-9/AA-9.1#S0245">single_task_declaration</a></code> or <code><a href="/docs/arm/AA-9/AA-9.4#S0250">single_protected_declaration</a></code>, the rule about current instances is not needed.) We exclude anonymous access types so that they can be used to create self-referencing types in the natural manner (otherwise such types would be illegal). <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>17.b/2</MarginText>
<MarginInfo items={["AI95-00382-01"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>The phrase &ldquo;within the <code><a href="/docs/arm/AA-3/AA-3.2#S0028">subtype_mark</a></code>&rdquo; in the &ldquo;this rule does not apply&rdquo; part is intended to cover a case like <strong>access</strong> T'Class appearing within the declarative region of T: here T denotes the type, not the current instance. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>17.1/5</MarginText>
<MarginInfo items={["AI12-0068-1", "AI12-0324-1", "AI12-0427-1"]} />
<ul>
<li>Within an <code><a href="/docs/arm/AA-13/AA-13.1#S0346">aspect_specification</a></code> for a type or subtype, the current instance represents a value of the type; it is not an object. Unless otherwise specified, the nominal subtype of this value is given by the subtype itself (the first subtype in the case of a <code><a href="/docs/arm/AA-3/AA-3.2#S0023">type_declaration</a></code>), prior to applying any predicate specified directly on the type or subtype. If the type or subtype is by-reference, the associated object of the value is the object associated (see <a href="/docs/arm/AA-6/AA-6.2">6.2</a>) with the evaluation of the usage name.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>17.c/4</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>For the purposes of Legality Rules, the current instance acts as a value within an <code><a href="/docs/arm/AA-13/AA-13.1#S0346">aspect_specification</a></code>. It might really be an object (and has to be for a by-reference type), but that isn't discoverable by direct use of the name of the current instance. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>18</MarginText>
<ul>
<li>If a usage name appears within the declarative region of a <code><a href="/docs/arm/AA-12/AA-12.1#S0310">generic_declaration</a></code> (but not within its <code><a href="/docs/arm/AA-12/AA-12.1#S0313">generic_formal_part</a></code>) and it denotes that same <code><a href="/docs/arm/AA-12/AA-12.1#S0310">generic_declaration</a></code>, then it denotes the <em>current instance</em> of the generic unit (rather than the generic unit itself). </ul>

<MarginText>18.a</MarginText>

<strong>To be honest: </strong>The current instance of a generic unit is the instance created by whichever <code><a href="/docs/arm/AA-12/AA-12.3#S0315">generic_instantiation</a></code> is of interest at any given time. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Within a <code><a href="/docs/arm/AA-12/AA-12.1#S0313">generic_formal_part</a></code>, a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> that denotes the <code><a href="/docs/arm/AA-12/AA-12.1#S0310">generic_declaration</a></code> denotes the generic unit, which implies that it is not overloadable. <br />
</Admonition>

#### Legality Rules

<MarginText>27/2</MarginText>
<MarginInfo items={["AI95-00332-01"]} />
<p>When a construct is one that requires that its expected type be a <em>single</em> type in a given class, the type of the construct shall be determinable solely from the context in which the construct appears, excluding the construct itself, but using the requirement that it be in the given class. Furthermore, the context shall not be one that expects any type in some class that contains types of the given class; in particular, the construct shall not be the operand of a <code><a href="/docs/arm/AA-4/AA-4.6#S0162">type_conversion</a></code>.<br /></p>

<AnnotatedOnly>
<MarginText>27.a/5</MarginText>
<MarginInfo items={["AI95-00230-01", "AI12-0005-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>For example, the expected type for a character literal is required to be a single character type. But the expected type for the operand of a <code><a href="/docs/arm/AA-4/AA-4.6#S0162">type_conversion</a></code> is any type. Therefore, a character literal is not allowed as the operand of a <code><a href="/docs/arm/AA-4/AA-4.6#S0162">type_conversion</a></code>. This is true even if there is only one character type in scope (which is never the case). The reason for these rules is so that the compiler will not have to search &ldquo;everywhere&rdquo; to see if there is exactly one type in a class in scope. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.b/2</MarginText>
<MarginInfo items={["AI95-00332-01"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>The first sentence is carefully worded so that it only mentions &ldquo;expected type&rdquo; as part of identifying the interesting case, but doesn't require that the context actually provide such an expected type. This allows such constructs to be used inside of constructs that don't provide an expected type (like qualified expressions and renames). Otherwise, such constructs wouldn't allow <code><a href="/docs/arm/AA-4/AA-4.3#S0106">aggregate</a></code>s, 'Access, and so on. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>27.1/4</MarginText>
<MarginInfo items={["AI05-0102-1", "AI05-0149-1", "AI05-0299-1", "AI12-0039-1"]} />
<p>Other than for the <em>tested{"_"}</em><code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code> of a membership test, if the expected type for a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> or <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> is not the same as the actual type of the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> or <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code>, the actual type shall be convertible to the expected type (see <a href="/docs/arm/AA-4/AA-4.6">4.6</a>); further, if the expected type is a named access-to-object type with designated type <em>D1</em> and the actual type is an anonymous access-to-object type with designated type <em>D2</em>, then <em>D1</em> shall cover <em>D2</em>, and the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> or <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> shall denote a view with an accessibility level for which the statically deeper relationship applies[; in particular it shall not denote an access parameter nor a stand-alone access object].<br /></p>

<AnnotatedOnly>
<MarginText>27.c/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This rule prevents an implicit conversion that would be illegal if it was an explicit conversion. For instance, this prevents assigning an access-to-constant value into a stand-alone anonymous access-to-variable object. It also covers convertibility of the designated type and accessibility checks.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.d/3</MarginText>
<Admonition type="aarm" aarm="note">
The rule also minimizes cases of implicit conversions when the tag check or the accessibility check might fail. We word it this way because access discriminants should also be disallowed if their enclosing object is designated by an access parameter. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.e/3</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This rule does not apply to expressions that don't have expected types (such as the operand of a qualified expression or the expression of a renames). We don't need a rule like this in those cases, as the type needs to be the same; there is no implicit conversion. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>28</MarginText>
<p>A complete context shall have at least one acceptable interpretation; if there is exactly one, then that one is chosen. <br /></p>

<AnnotatedOnly>
<MarginText>28.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This, and the rule below about ambiguity, are the ones that suck in all the Syntax Rules and Name Resolution Rules as compile-time rules. Note that this and the ambiguity rule have to be Legality Rules. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>29</MarginText>
<p>There is a <em>preference</em> for the primitive operators (and <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code>s) of the root numeric types <em>root{"_"}integer</em> and <em>root{"_"}real</em>. In particular, if two acceptable interpretations of a constituent of a complete context differ only in that one is for a primitive operator (or <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code>) of the type <em>root{"_"}integer</em> or <em>root{"_"}real</em>, and the other is not, the interpretation using the primitive operator (or <code><a href="/docs/arm/AA-3/AA-3.5#S0037">range</a></code>) of the root numeric type is <em>preferred</em>.<br /></p>

<AnnotatedOnly>
<MarginText>29.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The reason for this preference is so that expressions involving literals and named numbers can be unambiguous. For example, without the preference rule, the following would be ambiguous: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>29.b/1</MarginText>
<CodeBlock language="ada">
N : constant := 123;{"\n"}if N {">"} 100 then -- Preference for root{"_"}integer "{">"}" operator.{"\n"}    ...{"\n"}end if;{"\n"}

</CodeBlock>
</AnnotatedOnly>
<MarginText>29.1/3</MarginText>
<MarginInfo items={["AI05-0149-1"]} />
<p>Similarly, there is a preference for the equality operators of the <em>universal{"_"}access</em> type (see <a href="/docs/arm/AA-4/AA-4.5#Subclause_4.5.2">4.5.2</a>). If two acceptable interpretations of a constituent of a complete context differ only in that one is for an equality operator of the <em>universal{"_"}access</em> type, and the other is not, the interpretation using the equality operator of the <em>universal{"_"}access</em> type is preferred.<br /></p>

<AnnotatedOnly>
<MarginText>29.c/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This preference is necessary because of implicit conversion from an anonymous access type to a named access type, which would allow the equality operator of any named access type to be used to compare anonymous access values (and that way lies madness). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>30</MarginText>
<p>For a complete context, if there is exactly one overall acceptable interpretation where each constituent's interpretation is the same as or preferred (in the above sense) over those in all other overall acceptable interpretations, then that one overall acceptable interpretation is chosen. Otherwise, the complete context is <em>ambiguous</em>.<br /></p>

<MarginText>31</MarginText>
<p>A complete context other than a <code><a href="/docs/arm/AA-2/AA-2.8#S0020">pragma_argument_association</a></code> shall not be ambiguous.<br /></p>

<MarginText>32</MarginText>
<p>A complete context that is a <code><a href="/docs/arm/AA-2/AA-2.8#S0020">pragma_argument_association</a></code> is allowed to be ambiguous (unless otherwise specified for the particular pragma), but only if every acceptable interpretation of the pragma argument is as a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> that statically denotes a callable entity. Such a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> denotes all of the declarations determined by its interpretations, and all of the views declared by these declarations. <br /></p>

<AnnotatedOnly>
<MarginText>32.a/3</MarginText>
<MarginInfo items={["AI95-00224-01", "AI05-0229-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>This applies to Inline, Suppress, Import, Export, and Convention <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code>s. For example, it is OK to say &ldquo;<strong>pragma</strong> Export(C, Entity{"_"}Name ={">"} P.Q);&rdquo;, even if there are two directly visible P's, and there are two Q's declared in the visible part of each P. In this case, P.Q denotes four different declarations. This rule also applies to certain pragmas defined in the Specialized Needs Annexes. It almost applies to Pure, Elaborate{"_"}Body, and Elaborate{"_"}All <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code>s, but those can't have overloading for other reasons.  Note that almost all of these pragmas are obsolescent (see <a href="/docs/arm/AA-J/AA-J.10">J.10</a> and <a href="/docs/arm/AA-J/AA-J.15">J.15</a>), and a major reason is that this rule has proven to be too broad in practice (it is common to want to specify something on a single subprogram of an overloaded set, that can't be done easily with this rule). <code><a href="/docs/arm/AA-13/AA-13.1#S0346">Aspect_specification</a></code>s, which are given on individual declarations, are preferred in Ada 2012.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>32.b</MarginText>
<Admonition type="aarm" aarm="note">
Note that if a pragma argument denotes a <em>call</em> to a callable entity, rather than the entity itself, this exception does not apply, and ambiguity is disallowed.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>32.c</MarginText>
<Admonition type="aarm" aarm="note">
Note that we need to carefully define which pragma-related rules are Name Resolution Rules, so that, for example, a <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code> Inline does not pick up subprograms declared in enclosing declarative regions, and therefore make itself illegal.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>32.d</MarginText>
<Admonition type="aarm" aarm="note">
We say &ldquo;statically denotes&rdquo; in the above rule in order to avoid having to worry about how many times the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is evaluated, in case it denotes more than one callable entity. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>33</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   If a usage name has only one acceptable interpretation, then it denotes the corresponding entity. However, this does not mean that the usage name is necessarily legal since other requirements exist which are not considered for overload resolution; for example, the fact that an expression is static, whether an object is constant, mode and subtype conformance rules, freezing rules, order of elaboration, and so on.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34</MarginText>
<Admonition type="aarm" aarm="note">
Similarly, subtypes are not considered for overload resolution (the violation of a constraint does not make a program illegal but raises an exception during program execution). <br />
</Admonition>
</AnnotatedOnly>

#### Wording Changes from Ada 83

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>34.c</MarginText>
<Admonition type="aarm" aarm="note">
In Ada 83, it is not clear what the &ldquo;syntax rules&rdquo; are. AI83-00157 states that a certain textual rule is a syntax rule, but it's still not clear how one tells in general which textual rules are syntax rules. We have solved the problem by stating exactly which rules are syntax rules &mdash; the ones that appear under the &ldquo;Syntax&rdquo; heading.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.d</MarginText>
<Admonition type="aarm" aarm="note">
RM83 has a long list of the &ldquo;forms&rdquo; of rules that are to be used in overload resolution (in addition to the syntax rules). It is not clear exactly which rules fall under each form. We have solved the problem by explicitly marking all rules that are used in overload resolution. Thus, the list of kinds of rules is unnecessary. It is replaced with some introductory (intentionally vague) text explaining the basic idea of what sorts of rules are overloading rules.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.e/3</MarginText>
<MarginInfo items={["AI05-0299-1"]} />
<Admonition type="aarm" aarm="note">
It is not clear from RM83 what information is embodied in a &ldquo;meaning&rdquo; or an &ldquo;interpretation&rdquo;. &ldquo;Meaning&rdquo; and &ldquo;interpretation&rdquo; were intended to be synonymous; we now use the latter only in defining the rules about overload resolution. &ldquo;Meaning&rdquo; is used only informally. This subclause attempts to clarify what is meant by &ldquo;interpretation&rdquo;.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.f</MarginText>
<Admonition type="aarm" aarm="note">
For example, RM83 does not make it clear that overload resolution is required in order to match <code>subprogram{"_"}bodies</code> with their corresponding declarations (and even to tell whether a given <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code> is the completion of a previous declaration). Clearly, the information needed to do this is part of the &ldquo;interpretation&rdquo; of a <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code>. The resolution of such things is defined in terms of the &ldquo;expected profile&rdquo; concept. Ada 95 has some new cases where expected profiles are needed &mdash; the resolution of P'Access, where P might denote a subprogram, is an example.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.g</MarginText>
<Admonition type="aarm" aarm="note">
RM83-8.7(2) might seem to imply that an interpretation embodies information about what is denoted by each usage name, but not information about which syntactic category each construct belongs to. However, it seems necessary to include such information, since the Ada grammar is highly ambiguous. For example, X(Y) might be a <code><a href="/docs/arm/AA-6/AA-6.4#S0218">function_call</a></code> or an <code><a href="/docs/arm/AA-4/AA-4.1#S0096">indexed_component</a></code>, and no context-free/syntactic information can tell the difference. It seems like we should view X(Y) as being, for example, &ldquo;interpreted as a <code><a href="/docs/arm/AA-6/AA-6.4#S0218">function_call</a></code>&rdquo; (if that's what overload resolution decides it is). Note that there are examples where the denotation of each usage name does not imply the syntactic category. However, even if that were not true, it seems that intuitively, the interpretation includes that information. Here's an example: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.h</MarginText>
<CodeBlock language="ada">
type T;{"\n"}type A is access T;{"\n"}type T is array(Integer range 1..10) of A;{"\n"}I : Integer := 3;{"\n"}function F(X : Integer := 7) return A;{"\n"}Y : A := F(I); -- Ambiguous? (We hope so.){"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>34.i/1</MarginText>
<Admonition type="aarm" aarm="note">
Consider the declaration of Y (a complete context). In the above example, overload resolution can easily determine the declaration, and therefore the entity, denoted by Y, A, F, and I. However, given all of that information, we still don't know whether F(I) is a <code><a href="/docs/arm/AA-6/AA-6.4#S0218">function_call</a></code> or an <code><a href="/docs/arm/AA-4/AA-4.1#S0096">indexed_component</a></code> whose <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> is a <code><a href="/docs/arm/AA-6/AA-6.4#S0218">function_call</a></code>. (In the latter case, it is equivalent to F(7).<strong>all</strong>(I).)<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.j</MarginText>
<Admonition type="aarm" aarm="note">
It seems clear that the declaration of Y ought to be considered ambiguous. We describe that by saying that there are two interpretations, one as a <code><a href="/docs/arm/AA-6/AA-6.4#S0218">function_call</a></code>, and one as an <code><a href="/docs/arm/AA-4/AA-4.1#S0096">indexed_component</a></code>. These interpretations are both acceptable to the overloading rules. Therefore, the complete context is ambiguous, and therefore illegal.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.k</MarginText>
<Admonition type="aarm" aarm="note">
It is the intent that the Ada 95 preference rule for root numeric operators is more locally enforceable than that of RM83-4.6(15). It should also eliminate interpretation shifts due to the addition or removal of a <code><a href="/docs/arm/AA-8/AA-8.4#S0235">use_clause</a></code> (the so called <em>Beaujolais</em> effect).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.l/2</MarginText>
<MarginInfo items={["AI95-00114-01"]} />
<Admonition type="aarm" aarm="note">
RM83-8.7 seems to be missing some complete contexts, such as <code><a href="/docs/arm/AA-2/AA-2.8#S0020">pragma_argument_association</a></code>s, <code><a href="/docs/arm/AA-3/AA-3.11#S0087">declarative_item</a></code>s that are not declarations or <code><a href="/docs/arm/AA-13/AA-13.1#S0343">aspect_clause</a></code>s, and <code><a href="/docs/arm/AA-10/AA-10.1#S0293">context_item</a></code>s. We have added these, and also replaced the &ldquo;must be determinable&rdquo; wording of RM83-5.4(3) with the notion that the expression of a <code><a href="/docs/arm/AA-5/AA-5.4#S0176">case_statement</a></code> is a complete context.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.m</MarginText>
<Admonition type="aarm" aarm="note">
Cases like the Val attribute are now handled using the normal type resolution rules, instead of having special cases that explicitly allow things like &ldquo;any integer type&rdquo;. <br />
</Admonition>

#### Inconsistencies With Ada 2012

<MarginText>34.y/4</MarginText>
<MarginInfo items={["AI12-0068-1"]} />
<Admonition type="aarm" aarm="note">
<strong>Corrigendum:</strong> Added a rule to specify that the current instance of a type or subtype is a value within an <code><a href="/docs/arm/AA-13/AA-13.1#S0346">aspect_specification</a></code>. More likely, a usage of a current instance as a prefix of an attribute will become illegal (such as Size or Alignment). Any such code is very tricky. Moreover, as this is a new feature of Ada 2012, there are not that many predicates and invariants, and the ones that exist are very unlikely to be this tricky. Thus we do not believe that there will be any practical effect to this change, other than to explicitly allow common implementation strategies. <br />
