<p class="Indented2">If a <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> has a <i>loop{"_"}</i><code><a href="/docs/arm/AA-5/AA-5.1#S0172">statement_identifier</a></code>, then the <code><a href="/docs/arm/AA-2/AA-2.3#S0002">identifier</a></code> shall be repeated after the <b>end loop</b>; otherwise, there shall not be an <code><a href="/docs/arm/AA-2/AA-2.3#S0002">identifier</a></code> after the <b>end loop</b>.<br /></p>

#### Name Resolution Rules

<MarginText>5.2/5</MarginText>
<MarginInfo items={["AI12-0251-1"]} />
<p>In a <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code> that is an <i>integer{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code>, the <i>integer{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code> is expected to be of any integer type. <br /></p>


#### Static Semantics

<MarginText>6/5</MarginText>
<MarginInfo items={["AI12-0061-1"]} />
<p>A <code><a href="/docs/arm/AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> declares a <i>loop parameter</i>, which is an object whose subtype (and nominal subtype) is that defined by the <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>. <br /></p>

<MarginText>6.1/5</MarginText>
<MarginInfo items={["AI12-0251-1"]} />
<p>In a <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code> that has a <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>, the <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code> declares a <i>chunk parameter</i> object whose subtype (and nominal subtype) is that defined by the <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>. <br /></p>


#### Dynamic Semantics

<MarginText>6.2/5</MarginText>
<MarginInfo items={["AI12-0250-1", "AI12-0327-1"]} />
<p>The <i>filter</i> of an <i>iterator construct</i> (a <code><a href="/docs/arm/AA-5/AA-5.5#S0181">loop_parameter_specification</a></code>, <code><a href="/docs/arm/AA-5/AA-5.5#S0183">iterator_specification</a></code>, or <code><a href="/docs/arm/AA-5/AA-5.5#S0185">procedural_iterator</a></code>) is defined to be <i>satisfied</i> when there is no <code><a href="/docs/arm/AA-5/AA-5.5#S0182">iterator_filter</a></code> for the iterator construct, or when the <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code> of the <code><a href="/docs/arm/AA-5/AA-5.5#S0182">iterator_filter</a></code> evaluates to True for a given iteration of the iterator construct.<br /></p>

<AnnotatedOnly>
<MarginText>6.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<b>Term entry: </b><b>iterator filter</b> &mdash; construct that is used to restrict the elements produced by an iteration to those for which a boolean condition evaluates to True<br />
</Admonition>
</AnnotatedOnly>

<MarginText>6.3/5</MarginText>
<MarginInfo items={["AI12-0250-1", "AI12-0327-1"]} />
<p>If a <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence_of_statements</a></code> of a <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> with an iterator construct is said to be <i>conditionally executed</i>, then the <code><a href="/docs/arm/AA-5/AA-5.1#S0167">statement</a></code>s are executed only when the filter of the iterator construct is satisfied.<br /></p>

<MarginText>6.4/5</MarginText>
<MarginInfo items={["AI12-0250-1", "AI12-0327-1"]} />
<p>The loop iterators <code><a href="/docs/arm/AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> and <code><a href="/docs/arm/AA-5/AA-5.5#S0183">iterator_specification</a></code> can also be used in contexts other than <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code>s (for example, see <a href="/docs/arm/AA-4/AA-4.3#Subclause_4.3.5">4.3.5</a> and <a href="/docs/arm/AA-4/AA-4.5#Subclause_4.5.8">4.5.8</a>). In such a context, the iterator <i>conditionally produces</i> values in the order specified for the associated construct below or in <a href="/docs/arm/AA-5/AA-5.5#Subclause_5.5.2">5.5.2</a>. The values produced are the values given to the loop parameter when the filter of the iterator construct is satisfied for that value. [No value is produced when the <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code> of an <code><a href="/docs/arm/AA-5/AA-5.5#S0182">iterator_filter</a></code> evaluates to False.]<br /></p>

<MarginText>7/5</MarginText>
<MarginInfo items={["AI12-0119-1"]} />
<p>For the execution of a <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code>, the <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence_of_statements</a></code> is executed zero or more times, until the <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> is complete. The <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> is complete when a transfer of control occurs that transfers control out of the loop, or, in the case of an <code><a href="/docs/arm/AA-5/AA-5.5#S0179">iteration_scheme</a></code>, as specified below.<br /></p>

<MarginText>8</MarginText>
<p>For the execution of a <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> with a <b>while</b> <code><a href="/docs/arm/AA-5/AA-5.5#S0179">iteration_scheme</a></code>, the condition is evaluated before each execution of the <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence{"_"}of{"_"}statements</a></code>; if the value of the <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code> is True, the <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence{"_"}of{"_"}statements</a></code> is executed; if False, the execution of the <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop{"_"}statement</a></code> is complete.<br /></p>

<MarginText>8.1/5</MarginText>
<MarginInfo items={["AI12-0251-1", "AI12-0294-1"]} />
<p>If the reserved word <b>parallel</b> is present in the <code><a href="/docs/arm/AA-5/AA-5.5#S0179">iteration_scheme</a></code> of a <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> (a <i>parallel loop</i>), the iterations are partitioned into one or more <i>chunks</i>, each with its own separate logical thread of control (see Clause <a href="/docs/arm/AA-9/">9</a>). If a <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code> is present in a parallel loop, it is elaborated first, and the result of the elaboration determines the maximum number of chunks used for the parallel loop. If the <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code> is an <i>integer{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.4#S0138">simple_expression</a></code>, the elaboration evaluates the expression, and the value of the expression determines the maximum number of chunks. If a <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code> is present, the elaboration elaborates the <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>, which defines the subtype of the chunk parameter, and the number of values in this subtype determines the maximum number of chunks. After elaborating the <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code>, a check is made that the determined maximum number of chunks is greater than zero. If this check fails, Program{"_"}Error is raised. <br /></p>

<MarginText>9/5</MarginText>
<MarginInfo items={["AI05-0139-2", "AI05-0262-1", "AI12-0071-1", "AI12-0119-1", "AI12-0250-1", "AI12-0251-1", "AI12-0294-1", "AI12-0355-2", "AI12-0416-1"]} />
<p>For the execution of a <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> that has an <code><a href="/docs/arm/AA-5/AA-5.5#S0179">iteration_scheme</a></code> including a <code><a href="/docs/arm/AA-5/AA-5.5#S0181">loop{"_"}parameter{"_"}specification</a></code>, after elaborating the <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code> and <code><a href="/docs/arm/AA-13/AA-13.1#S0346">aspect_specification</a></code>, if any, the <code><a href="/docs/arm/AA-5/AA-5.5#S0181">loop{"_"}parameter{"_"}specification</a></code> is elaborated. This elaborates the <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete{"_"}subtype{"_"}definition</a></code>, which defines the subtype of the loop parameter. If the <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete{"_"}subtype{"_"}definition</a></code> defines a subtype with a null range, the execution of the <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> is complete. Otherwise, the <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence{"_"}of{"_"}statements</a></code> is conditionally executed once for each value of the discrete subtype defined by the <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete{"_"}subtype{"_"}definition</a></code> that satisfies the predicates of the subtype (or until the loop is left as a consequence of a transfer of control). Prior to each such iteration, the corresponding value of the discrete subtype is assigned to the loop parameter associated with the given iteration. If the loop is a parallel loop, each chunk has its own logical thread of control with its own copy of the loop parameter; otherwise (a <i>sequential loop</i>), a single logical thread of control performs the loop, and there is a single copy of the loop parameter. Each logical thread of control handles a distinct subrange of the values of the subtype of the loop parameter such that all values are covered with no overlaps. Within each logical thread of control, the values are assigned to the loop parameter in increasing order unless the reserved word <b>reverse</b> is present, in which case the values are assigned in decreasing order. In the absence of a transfer of control, the associated parallel construct of a <code><a href="/docs/arm/AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> is complete when all of its logical threads of control are complete. <br /></p>

#### Examples

</CodeBlock>

<p><i>Example of a loop statement with a <b>while</b> iteration scheme:</i> <br /></p>

<MarginText>20</MarginText>
<CodeBlock language="ada">

#### Name Resolution Rules

<MarginText>3/3_5.5.2</MarginText>
<MarginInfo items={["AI05-0139-2", "AI05-0292-1"]} />
<p>For the first form of <code><a href="/docs/arm/AA-5/AA-5.5#S0183">iterator_specification</a></code>, called a <i>generalized iterator</i>, the expected type for the <i>iterator{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is any iterator type. For the second form of <code><a href="/docs/arm/AA-5/AA-5.5#S0183">iterator_specification</a></code>, the expected type for the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is any array or iterable container type. If the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> denotes an array object, the <code><a href="/docs/arm/AA-5/AA-5.5#S0183">iterator_specification</a></code> is called an <i>array component iterator</i>; otherwise it is called a <i>container element iterator</i>. <br /></p>

<AnnotatedOnly>
<MarginText>3.a/5_5.5.2</MarginText>
<Admonition type="aarm" aarm="note">
<b>Term entry: </b><b>iterator</b> &mdash; construct that is used to loop over the elements of an array or container<br />Note: Iterators can be user defined, and can perform arbitrary computations to access elements from a container.<br />
</Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>4/5_5.5.2</MarginText>
<MarginInfo items={["AI05-0139-2", "AI12-0266-1"]} />
<p>If the reserved word <b>reverse</b> appears, the <code><a href="/docs/arm/AA-5/AA-5.5#S0183">iterator_specification</a></code> is a <i>reverse iterator</i>. If the <code><a href="/docs/arm/AA-5/AA-5.5#S0183">iterator_specification</a></code> is for a parallel construct, the <code><a href="/docs/arm/AA-5/AA-5.5#S0183">iterator_specification</a></code> is a <i>parallel iterator</i>. Otherwise, it is a <i>forward iterator</i>. Forward and reverse iterators are collectively called <i>sequential</i> iterators. In a reverse generalized iterator, the <i>iterator{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> shall be of a reversible iterator type. In a parallel generalized iterator, the <i>iterator{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> shall be of a parallel iterator type. In a reverse container element iterator, the default iterator type for the type of the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> shall be a reversible iterator type. In a parallel container element iterator, the default iterator type for the type of the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> shall be of a parallel iterator type.<br /></p>

<MarginText>5/5_5.5.2</MarginText>
<MarginInfo items={["AI05-0139-2", "AI12-0151-1", "AI12-0156-1", "AI12-0183-1"]} />
<p>The subtype defined by the <code><a href="/docs/arm/AA-5/AA-5.5#S0184">loop{"_"}parameter{"_"}subtype{"_"}indication</a></code>, if any, of a generalized iterator shall statically match the iteration cursor subtype. The subtype defined by the <code><a href="/docs/arm/AA-5/AA-5.5#S0184">loop{"_"}parameter{"_"}subtype{"_"}indication</a></code>, if any, of an array component iterator shall statically match the component subtype of the type of the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code>. The subtype defined by the <code><a href="/docs/arm/AA-5/AA-5.5#S0184">loop{"_"}parameter{"_"}subtype{"_"}indication</a></code>, if any, of a container element iterator shall statically match the default element subtype for the type of the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code>.<br /></p>

<MarginText>6/3_5.5.2</MarginText>
<MarginInfo items={["AI05-0139-2"]} />
<p>In a container element iterator whose <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> has type <i>T</i>, if the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> denotes a constant or the Variable{"_"}Indexing aspect is not specified for <i>T</i>, then the Constant{"_"}Indexing aspect shall be specified for <i>T</i>.<br /></p>

<MarginText>6.1/4_5.5.2</MarginText>
<MarginInfo items={["AI12-0047-1"]} />
<p>The <i>iterator{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> or <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> of an <code><a href="/docs/arm/AA-5/AA-5.5#S0183">iterator_specification</a></code> shall not denote a subcomponent that depends on discriminants of an object whose nominal subtype is unconstrained, unless the object is known to be constrained.<br /></p>

<AnnotatedOnly>
<MarginText>6.a/4_5.5.2</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>This is the same rule that applies to renames; it serves the same purpose of preventing the object from disappearing while the iterator is still using it. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>6.2/4_5.5.2</MarginText>
<MarginInfo items={["AI12-0120-1"]} />
<p>A container element iterator is illegal if the call of the default iterator function that creates the loop iterator (see below) is illegal.<br /></p>

<AnnotatedOnly>
<MarginText>6.b/4_5.5.2</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>This can happen if the parameter to the default iterator function is <b>in out</b> and the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is a constant. The wording applies to any reason that the call would be illegal, as it's possible that one of the default parameters would be illegal, or that some accessibility check would fail. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>6.3/4_5.5.2</MarginText>
<MarginInfo items={["AI12-0120-1"]} />
<p>A generalized iterator is illegal if the iteration cursor subtype of the <i>iterator{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is a limited type at the point of the generalized iterator. A container element iterator is illegal if the default cursor subtype of the type of the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is a limited type at the point of the container element iterator.<br /></p>

<AnnotatedOnly>
<MarginText>6.c/4_5.5.2</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>If the cursor type is limited, the assignment to the loop parameter for a generalized iterator would be illegal. The same is true for a container element iterator. We have to say "at the point of the iterator" as the limitedness of a type can change due to visibility. <br />
</Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>7/5_5.5.2</MarginText>
<MarginInfo items={["AI05-0139-2", "AI05-0269-1", "AI05-0292-1", "AI12-0156-1"]} />
<p>An <code><a href="/docs/arm/AA-5/AA-5.5#S0183">iterator_specification</a></code> declares a <i>loop parameter</i>. In a generalized iterator, an array component iterator, or a container element iterator, if a <code><a href="/docs/arm/AA-5/AA-5.5#S0184">loop_parameter_subtype_indication</a></code> is present, it determines the nominal subtype of the loop parameter. In a generalized iterator, if a <code><a href="/docs/arm/AA-5/AA-5.5#S0184">loop_parameter_subtype_indication</a></code> is not present, the nominal subtype of the loop parameter is the iteration cursor subtype. In an array component iterator, if a <code><a href="/docs/arm/AA-5/AA-5.5#S0184">loop_parameter_subtype_indication</a></code> is not present, the nominal subtype of the loop parameter is the component subtype of the type of the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code>. In a container element iterator, if a <code><a href="/docs/arm/AA-5/AA-5.5#S0184">loop_parameter_subtype_indication</a></code> is not present, the nominal subtype of the loop parameter is the default element subtype for the type of the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code>.<br /></p>

<MarginText>8/3_5.5.2</MarginText>
<MarginInfo items={["AI05-0139-2", "AI05-0292-1"]} />
<p>In a generalized iterator, the loop parameter is a constant. In an array component iterator, the loop parameter is a constant if the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> denotes a constant; otherwise it denotes a variable. In a container element iterator, the loop parameter is a constant if the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> denotes a constant, or if the Variable{"_"}Indexing aspect is not specified for the type of the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code>; otherwise it is a variable.<br /></p>

<AnnotatedOnly>
<MarginText>8.a/5_5.5.2</MarginText>
<MarginInfo items={["AI12-0093-1", "AI12-0156-1"]} />
<Admonition type="aarm" aarm="ramification">
<b></b>The loop parameter of a generalized iterator has the same accessibility as the loop statement. This means that the loop parameter object is finalized when the loop statement is left. (It also may be finalized as part of assigning a new value to the loop parameter.) For array component iterators, the loop parameter directly denotes an element of the array and has the accessibility of the associated array. For container element iterators, the loop parameter denotes the result of the indexing function call (in the case of a constant indexing) or a generalized reference thereof (in the case of a variable indexing). Roughly speaking, the loop parameter has the accessibility level of a single iteration of the loop. More precisely, the function result (or the generalized reference thereof) is considered to be renamed in the declarative part of a notional block statement which immediately encloses the loop's <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence_of_statements</a></code>; the accessibility of the loop parameter is that of the block statement. <br />
</Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>9/3_5.5.2</MarginText>
<MarginInfo items={["AI05-0139-2"]} />
<p>For the execution of a <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> with an <code><a href="/docs/arm/AA-5/AA-5.5#S0183">iterator_specification</a></code>, the <code><a href="/docs/arm/AA-5/AA-5.5#S0183">iterator_specification</a></code> is first elaborated. This elaboration elaborates the <code><a href="/docs/arm/AA-3/AA-3.2#S0027">subtype_indication</a></code>, if any.<br /></p>

<MarginText>10/5_5.5.2</MarginText>
<MarginInfo items={["AI05-0139-2", "AI12-0250-1", "AI12-0266-1"]} />
<p>For a sequential generalized iterator, the loop parameter is created, the <i>iterator{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is evaluated, and the denoted iterator object becomes the <i>loop iterator</i>. In a forward generalized iterator, the operation First of the iterator type is called on the loop iterator, to produce the initial value for the loop parameter. If the result of calling Has{"_"}Element on the initial value is False, then the execution of the <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> is complete. Otherwise, the <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence_of_statements</a></code> is conditionally executed and then the Next operation of the iterator type is called with the loop iterator and the current value of the loop parameter to produce the next value to be assigned to the loop parameter. This repeats until the result of calling Has{"_"}Element on the loop parameter is False, or the loop is left as a consequence of a transfer of control. For a reverse generalized iterator, the operations Last and Previous are called rather than First and Next.<br /></p>

<AnnotatedOnly>
<MarginText>10.a/4_5.5.2</MarginText>
<MarginInfo items={["AI12-0093-1"]} />
<Admonition type="aarm" aarm="ramification">
<b></b>The loop parameter of a generalized iterator is a variable of which the user only has a constant view. It follows the normal rules for a variable of its nominal subtype. In particular, if the nominal subtype is indefinite, the variable is constrained by its initial value. Similarly, if the nominal subtype is class-wide, the variable (like all variables) has the tag of the initial value. Constraint{"_"}Error may be raised by a subsequent iteration if Next or Previous return an object with a different tag or constraint. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>10.1/5_5.5.2</MarginText>
<MarginInfo items={["AI12-0266-1"]} />
<p>For a parallel generalized iterator, the <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code>, if any, of the associated parallel construct, is first elaborated, to determine the maximum number of chunks (see <a href="/docs/arm/AA-5/AA-5.5">5.5</a>), and then the operation Split{"_"}Into{"_"}Chunks of the iterator type is called, with the determined maximum passed as the Max{"_"}Chunks parameter, specifying the upper bound for the number of loop parameter objects (and the number of logical threads of control) to be associated with the iterator. In the absence of a <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code>, the maximum number of chunks is determined in an implementation-defined manner.<br /></p>

<AnnotatedOnly>
<MarginText>10.b/5_5.5.2</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<b></b>The maximum number of chunks for a parallel generalized iterator without a <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code>.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.c/5_5.5.2</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>The Max{"_"}Chunks parameter of the Split{"_"}Into{"_"}Chunks procedure is an upper bound for the number of chunks to be associated with a loop. A container implementation may opt for a lower value for the number of chunks if a more optimal split can be determined. For instance, a tree-based container might create the split based on the number of branches at the top levels of the tree. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>10.2/5_5.5.2</MarginText>
<MarginInfo items={["AI12-0250-1", "AI12-0266-1", "AI12-0418-1"]} />
<p>Upon return from Split{"_"}Into{"_"}Chunks, the actual number of chunks for the loop is determined by calling the Chunk{"_"}Count operation of the iterator, at which point one logical thread of control is initiated for each chunk, with an associated chunk index in the range from one to the actual number of chunks.<br /></p>

<MarginText>10.3/5_5.5.2</MarginText>
<MarginInfo items={["AI12-0250-1", "AI12-0266-1", "AI12-0418-1"]} />
<p>Within each logical thread of control, a loop parameter is created. If a <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code> with a <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code> is present in the associated parallel construct, then a chunk parameter is created and initialized with a value from the discrete subtype defined by the <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>, so that the order of the chosen chunk parameter values correspond to the order of the chunk indices associated with the logical threads of control. The operation First of the iterator type that has a Chunk parameter is called on the loop iterator, with Chunk initialized from the corresponding chunk index, to produce the initial value for the loop parameter. If the result of calling Has{"_"}Element on this initial value is False, then the execution of the logical thread of control is complete. Otherwise, the <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence_of_statements</a></code> is conditionally executed, and then the Next operation of the iterator type that has a Chunk parameter is called with the loop iterator, the current value of the loop parameter, and the corresponding chunk index, to produce the next value to be assigned to the loop parameter. This repeats until the result of calling Has{"_"}Element on the loop parameter is False, or the associated parallel construct is left as a consequence of a transfer of control.<br /></p>

<MarginText>10.4/5_5.5.2</MarginText>
<MarginInfo items={["AI12-0250-1", "AI12-0266-1", "AI12-0418-1"]} />
<p>In the absence of a transfer of control, the associated parallel construct of a parallel generalized iterator is complete when all of its logical threads of control are complete.<br /></p>

<MarginText>11/5_5.5.2</MarginText>
<MarginInfo items={["AI05-0139-2", "AI05-0292-1", "AI12-0250-1", "AI12-0266-1"]} />
<p>For an array component iterator, the <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code> of the associated parallel construct, if any, is first elaborated to determine the maximum number of chunks (see <a href="/docs/arm/AA-5/AA-5.5">5.5</a>), and then the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is evaluated and the denoted array object becomes the <i>array for the loop</i>. If the array for the loop is a null array, then the execution of the <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> is complete. Otherwise, the <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence_of_statements</a></code> is conditionally executed with the loop parameter denoting each component of the array for the loop, using a <i>canonical</i> order of components, which is last dimension varying fastest (unless the array has convention Fortran, in which case it is first dimension varying fastest). For a forward array component iterator, the iteration starts with the component whose index values are each the first in their index range, and continues in the canonical order. For a reverse array component iterator, the iteration starts with the component whose index values are each the last in their index range, and continues in the reverse of the canonical order. For a parallel array component iterator, the iteration is broken up into contiguous chunks of the canonical order, such that all components are covered with no overlaps; each chunk has its own logical thread of control with its own loop parameter and iteration within each chunk is in the canonical order. The number of chunks is implementation defined, but is limited in the presence of a <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code> to the determined maximum. The loop iteration proceeds until the <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence_of_statements</a></code> has been conditionally executed for each component of the array for the loop, or until the loop is left as a consequence of a transfer of control.<br /></p>

<AnnotatedOnly>
<MarginText>11.a/5_5.5.2</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<b></b>The number of chunks for an array component iterator.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>11.1/5_5.5.2</MarginText>
<MarginInfo items={["AI12-0266-1"]} />
<p>If a <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code> with a <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code> is present in the associated parallel construct, then the logical thread of control associated with a given chunk has a chunk parameter initialized with a distinct value from the discrete subtype defined by the <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>. The values of the chunk parameters are assigned such that they increase in the canonical order of the starting array components for the chunks.<br /></p>

<MarginText>12/5_5.5.2</MarginText>
<MarginInfo items={["AI05-0139-2", "AI05-0292-1", "AI12-0111-1", "AI12-0266-1"]} />
<p>For a container element iterator, the <code><a href="/docs/arm/AA-5/AA-5.5#S0180">chunk_specification</a></code> of the associated parallel construct, if any, is first elaborated to determine the maximum number of chunks (see <a href="/docs/arm/AA-5/AA-5.5">5.5</a>), and then the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is evaluated. If the container type has Iterator{"_"}View specified, an object of the Iterator{"_"}View type is created with the discriminant referencing the iterable container object denoted by the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code>. This is the <i>iterable container object for the loop</i>. Otherwise, the iterable container object denoted by the <i>iterable{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> becomes the iterable container object for the loop. The default iterator function for the type of the iterable container object for the loop is called on the iterable container object and the result is the <i>loop iterator</i>. For a sequential container element iterator, an object of the default cursor subtype is created (the <i>loop cursor</i>). For a parallel container element iterator, each chunk of iterations will have its own loop cursor, again of the default cursor subtype.<br /></p>

<AnnotatedOnly>
<MarginText>12.a/5_5.5.2</MarginText>
<MarginInfo items={["AI12-0111-1"]} />
<Admonition type="aarm" aarm="reason">
<b></b>If Iterator{"_"}View is specified, we add an extra object and use that object for this iteration. This allows these iterators to automatically use the stable view (defined in each of the language-defined containers) to do the iteration. That eliminates the need to set and clear the tampering with elements indication each time Reference is called; that eliminates substantial overhead as finalization is typically used to implement the tampering reset. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>13/5_5.5.2</MarginText>
<MarginInfo items={["AI05-0139-2", "AI05-0292-1", "AI12-0266-1"]} />
<p>A container element iterator then proceeds as described above for a generalized iterator, except that each reference to a loop parameter is replaced by a reference to the corresponding loop cursor. For a container element iterator, the loop parameter for each iteration instead denotes an indexing (see <a href="/docs/arm/AA-4/AA-4.1#Subclause_4.1.6">4.1.6</a>) into the iterable container object for the loop, with the only parameter to the indexing being the value of the loop cursor for the given iteration. If the loop parameter is a constant (see above), then the indexing uses the default constant indexing function for the type of the iterable container object for the loop; otherwise it uses the default variable indexing function.<br /></p>

<MarginText>14/4_5.5.2</MarginText>
<MarginInfo items={["AI12-0120-1"]} />
<p>Any exception propagated by the execution of a generalized iterator or container element iterator is propagated by the immediately enclosing loop statement.<br /></p>

<AnnotatedOnly>
<MarginText>14.a/4_5.5.2</MarginText>

<MarginText>17.b/4_5.5.2</MarginText>
<MarginInfo items={["AI12-0047-1"]} />
<Admonition type="aarm" aarm="note">
<b>Corrigendum:</b> Added a rule to ensure that the object being iterated cannot be a component that could disappear before the loop completes. This could be incompatible by making a loop that was legal (and worked correctly, so long as the enclosing object is not modified during the loop) from the original Ada 2012 illegal in corrected Ada 2012. Such loops should be pretty rare, especially as these iterator forms are new to Ada 2012.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>17.c/4_5.5.2</MarginText>
<MarginInfo items={["AI12-0120-1"]} />
<Admonition type="aarm" aarm="note">
<b>Corrigendum:</b> Added rules to reject loops if the call to the default iterator function for a container element iterator is illegal, or if the cursor type of an iterator is limited. These are formally incompatible with original Ada 2012, but as it's unlikely that any Ada 2012 compiler ever allowed the illegal usages in an expansion of a loop (it's much more likely that they would have just caused an internal error in the compiler), this should have no effect in practice.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>17.d/4_5.5.2</MarginText>
<MarginInfo items={["AI12-0151-1"]} />
<Admonition type="aarm" aarm="note">
<b>Corrigendum:</b> Added a requirement that the given subtype statically match the subtype of the element or component for a component element iterator or array component iterator. Original Ada 2012 text allowed any type that covered the subtype of the element or component, but that led to questions of what the meaning was if they are different. In this case, the element is essentially a renaming of the container element, and it doesn't make sense for the constraints to be different. Ignoring explicitly defined constraints in renames is a mistake that we don't want to continue, thus we require static matching. This means that some programs might be illegal, but those programs were misleading at best, and potentially would raise unexpected exceptions because the element values might have been invalid or abnormal with respect to the declared constraint. <br />
</Admonition>
</AnnotatedOnly>

## 5.5.3  Procedural Iterators

<MarginText>1/5_5.5.3</MarginText>
<MarginInfo items={["AI12-0189-1"]} />
<p>A <code><a href="/docs/arm/AA-5/AA-5.5#S0185">procedural_iterator</a></code> invokes a user-defined procedure, passing in the body of the enclosing <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> as a parameter of an anonymous access-to-procedure type, to allow the loop body to be executed repeatedly as part of the invocation of the user-defined procedure. <br /></p>

#### Static Semantics

<MarginText>15/5_5.5.3</MarginText>
<MarginInfo items={["AI12-0189-1", "AI12-0250-1", "AI12-0308-1", "AI12-0326-2"]} />
<p>A <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> with an <code><a href="/docs/arm/AA-5/AA-5.5#S0179">iteration_scheme</a></code> that has a <code><a href="/docs/arm/AA-5/AA-5.5#S0185">procedural_iterator</a></code> is equivalent to a local declaration of a procedure P followed by a <code><a href="/docs/arm/AA-6/AA-6.4#S0217">procedure_call_statement</a></code> that is formed from the <code><a href="/docs/arm/AA-5/AA-5.5#S0187">iterator_procedure_call</a></code> by replacing the {"<"}{">"} of the <code><a href="/docs/arm/AA-5/AA-5.5#S0190">parameter_association_with_box</a></code> with P'Access. The <code><a href="/docs/arm/AA-6/AA-6.1#S0206">formal_part</a></code> of the locally declared procedure P is formed from the <code><a href="/docs/arm/AA-6/AA-6.1#S0206">formal_part</a></code> of the anonymous access-to-procedure type <i>A</i>, by replacing the <code><a href="/docs/arm/AA-2/AA-2.3#S0002">identifier</a></code> of each formal parameter of this <code><a href="/docs/arm/AA-6/AA-6.1#S0206">formal_part</a></code> with the <code><a href="/docs/arm/AA-2/AA-2.3#S0002">identifier</a></code> of the corresponding formal parameter or element of the list of <code><a href="/docs/arm/AA-3/AA-3.1#S0022">defining_identifier</a></code>s given in the <code><a href="/docs/arm/AA-5/AA-5.5#S0186">iterator_parameter_specification</a></code>. The body of <i>P</i> consists of the conditionally executed <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence{"_"}of{"_"}statements</a></code>. The procedure P is called the <i>loop body procedure</i>.<br /></p>

</CodeBlock>

<MarginText>17/5_5.5.3</MarginText>
<MarginInfo items={["AI12-0189-1", "AI12-0326-2", "AI12-0344-1"]} />
<p>The following aspects may be specified for a callable entity <i>S</i> that has exactly one formal parameter of an anonymous access-to-subprogram type:<br /></p>

<MarginText>18/5_5.5.3</MarginText>
<dt><br/>Allows{"_"}Exit</dt>
<dl>
<dd>The Allows{"_"}Exit aspect is of type Boolean. The specified value shall be static. The Allows{"_"}Exit aspect of an inherited primitive subprogram is True if Allows{"_"}Exit is True either for the corresponding subprogram of the progenitor type or for any other inherited subprogram that it overrides. If not specified or inherited as True, the Allows{"_"}Exit aspect of a callable entity is False. For an entry, only a confirming specification of False is permitted for the Allows{"_"}Exit aspect.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>18.a/5_5.5.3</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>An entry does not allow exit, because implementing a transfer of control out of a task or protected entry creates unnecessarily complex dynamic semantics. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>19/5_5.5.3</MarginText>
<dl>
<dd>Specifying the Allows{"_"}Exit aspect to be True for a subprogram indicates that the subprogram <i>allows exit</i>, meaning that it is prepared to be completed by arbitrary transfers of control from the loop body procedure[, including propagation of exceptions. A subprogram for which Allows{"_"}Exit is True should use finalization as appropriate rather than exception handling to recover resources and make any necessary final updates to data structures].<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>19.a/5_5.5.3</MarginText>
<Admonition type="aarm" aarm="note">
<b>Aspect Description for </b><b>Allows{"_"}Exit: </b>An indication of whether a subprogram will operate correctly for arbitrary transfers of control.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>19.b/5_5.5.3</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>A subprogram that does not need cleanup satisfies the requirements, and thus can specify Allows{"_"}Exit as True. If a subprogram <i>S</i> allows exit, it cannot expect to get control other than via finalization if the loop body procedure initiates a transfer of control as part of a <code><a href="/docs/arm/AA-5/AA-5.5#S0185">procedural_iterator</a></code>. In particular, exception handlers in <i>S</i>, even <b>when others</b> handlers, will not be executed when a transfer of control occurs. The mechanism that the implementation uses to implement such transfers of control needs to avoid triggering exception handlers. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>20/5_5.5.3</MarginText>
<dt><br/>Parallel{"_"}Iterator</dt>
<dl>
<dd>The Parallel{"_"}Iterator aspect is of type Boolean. The specified value shall be static. The Parallel{"_"}Iterator aspect of an inherited primitive subprogram is True if Parallel{"_"}Iterator is True either for the corresponding subprogram of the progenitor type or for any other inherited subprogram that it overrides. If not specified or inherited as True, the Parallel{"_"}Iterator aspect of a callable entity is False.<br /></dd>
<MarginText>21/5_5.5.3</MarginText>
<MarginInfo items={["AI12-0189-1", "AI12-0442-1"]} />
<dd>Specifying the Parallel{"_"}Iterator aspect to be True for a callable entity indicates that the entity is allowed to invoke the loop body procedure from multiple distinct logical threads of control. The Parallel{"_"}Iterator aspect for a subprogram shall be statically False if the subprogram allows exit.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>21.a/5_5.5.3</MarginText>
<Admonition type="aarm" aarm="note">
<b>Aspect Description for </b><b>Parallel{"_"}Iterator: </b>An indication of whether a subprogram may use multiple threads of control to invoke a loop body procedure.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>21.b/5_5.5.3</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>Permitting exit from a parallel procedural iterator introduces additional semantic and implementation complexity. <br />
</Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>22/5_5.5.3</MarginText>
<MarginInfo items={["AI12-0189-1", "AI12-0326-2"]} />
<p>If a callable entity overrides an inherited dispatching subprogram that allows exit, the overriding callable entity also shall allow exit. If a callable entity overrides an inherited dispatching subprogram that has a True Parallel{"_"}Iterator aspect, the overriding callable entity also shall have a True Parallel{"_"}Iterator aspect.<br /></p>

<AnnotatedOnly>
<MarginText>22.a/5_5.5.3</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Since an entry never allows exit, attempting to implement an allows exit subprogram with a task or protected entry is always illegal. However, the Parallel{"_"}Iterator aspect can be applied to an entry, so a subprogram with the Parallel{"_"}Iterator aspect True can be implemented by an entry. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>23/5_5.5.3</MarginText>
<MarginInfo items={["AI12-0326-2"]} />
<p>A <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> with a <code><a href="/docs/arm/AA-5/AA-5.5#S0185">procedural_iterator</a></code> as its <code><a href="/docs/arm/AA-5/AA-5.5#S0179">iteration_scheme</a></code> shall begin with the reserved word <b>parallel</b> if and only if the callable entity identified in the <code><a href="/docs/arm/AA-5/AA-5.5#S0187">iterator_procedure_call</a></code> has a Parallel{"_"}iterator aspect of True.<br /></p>

<MarginText>24/5_5.5.3</MarginText>

<MarginText>26/5_5.5.3</MarginText>
<MarginInfo items={["AI12-0294-1"]} />
<p>The <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence_of_statements</a></code> of a <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> with a <code><a href="/docs/arm/AA-5/AA-5.5#S0185">procedural_iterator</a></code> as its <code><a href="/docs/arm/AA-5/AA-5.5#S0179">iteration_scheme</a></code> shall not contain an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> whose <code><a href="/docs/arm/AA-9/AA-9.5#S0257">entry_declaration</a></code> occurs outside the <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code>.<br /></p>

#### Dynamic Semantics

<MarginText>27/5_5.5.3</MarginText>
<MarginInfo items={["AI12-0326-2", "AI12-0355-2", "AI12-0445-1"]} />
<p>[For the execution of a <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> with an <code><a href="/docs/arm/AA-5/AA-5.5#S0179">iteration_scheme</a></code> that has a <code><a href="/docs/arm/AA-5/AA-5.5#S0185">procedural_iterator</a></code>, the procedure denoted by the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> or <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> of the <code><a href="/docs/arm/AA-5/AA-5.5#S0187">iterator_procedure_call</a></code> (the <i>iterating procedure</i>) is invoked, passing an access value designating the loop body procedure as a parameter. The iterating procedure then calls the loop body procedure zero or more times and returns, whereupon the <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> is complete. If the <b>parallel</b> reserved word is present, the iterating procedure is allowed to invoke the loop body procedure from multiple distinct logical threads of control.] The <code><a href="/docs/arm/AA-13/AA-13.1#S0346">aspect_specification</a></code>, if any, is elaborated prior to the invocation of the iterating procedure.<br /></p>

#### Bounded (Run-Time) Errors

<MarginText>28/5_5.5.3</MarginText>
<MarginInfo items={["AI12-0326-2", "AI12-0445-1"]} />
<p>If the callable entity identified in the <code><a href="/docs/arm/AA-5/AA-5.5#S0187">iterator_procedure_call</a></code> allows exit, then it is a bounded error for a call of the loop body procedure to be performed from within an abort-deferred operation (see <a href="/docs/arm/AA-9/AA-9.8">9.8</a>), unless the entire <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> was within the same abort-deferred operation. If detected, Program{"_"}Error is raised at the point of the call; otherwise, a transfer of control from the <code><a href="/docs/arm/AA-5/AA-5.1#S0166">sequence_of_statements</a></code> of the <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> will not necessarily terminate the <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code>, and the loop body procedure can be called again.<br /></p>

<MarginText>29/5_5.5.3</MarginText>
<MarginInfo items={["AI12-0326-2", "AI12-0445-1"]} />
<p>If a <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code> with the <code><a href="/docs/arm/AA-5/AA-5.5#S0185">procedural_iterator</a></code> as its <code><a href="/docs/arm/AA-5/AA-5.5#S0179">iteration_scheme</a></code> (see <a href="/docs/arm/AA-5/AA-5.5">5.5</a>) does not begin with the reserved word <b>parallel</b>, it is a bounded error if the loop body procedure is invoked from a different logical thread of control than the one that initiates the <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code>. If detected, Program{"_"}Error is raised; otherwise, conflicts associated with concurrent executions of the loop body procedure can occur without being detected by the applicable conflict check policy (see <a href="/docs/arm/AA-9/AA-9.10#Subclause_9.10.1">9.10.1</a>). Furthermore, propagating an exception or making an attempt to exit in the presence of multiple threads of control will not necessarily terminate the <code><a href="/docs/arm/AA-5/AA-5.5#S0178">loop_statement</a></code>, deadlock can occur, or the loop body procedure can be called again.<br /></p>

<AnnotatedOnly>
<MarginText>29.a/5_5.5.3</MarginText>
<MarginInfo items={["AI12-0326-2"]} />
<Admonition type="aarm" aarm="discussion">
<b></b>Other Ada rules are still in effect for the allows exit subprogram <i>A</i>, of course. For instance, if a transfer of control causes finalization which raises an exception, Program{"_"}Error will be propagated by <i>A</i> (rather than the transfer of control). In such a case, the bounded error above would still apply. Another example is the case where an unrelated task is waiting on the normal completion of the loop body procedure call in <i>A</i>. Such a task might end up waiting forever if a transfer of control happens (this is a deadlock situation). This case does not require additional wording, as the same thing would happen if an exception is propagated from the loop body procedure or if <i>A</i> executed a transfer of control (such as a return statement). <br />
</Admonition>
</AnnotatedOnly>
