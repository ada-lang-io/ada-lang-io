<MarginInfo items={["AI12-0119-1"]} />
<p>[A <code><a href="../AA-5/AA-5.1#S0167">statement</a></code> is either simple or compound. A <code><a href="../AA-5/AA-5.1#S0168">simple_statement</a></code> encloses no other <code><a href="../AA-5/AA-5.1#S0167">statement</a></code>. A <code><a href="../AA-5/AA-5.1#S0169">compound_statement</a></code> can enclose <code><a href="../AA-5/AA-5.1#S0168">simple_statement</a></code>s and other <code><a href="../AA-5/AA-5.1#S0169">compound_statement</a></code>s.] A <em>parallel construct</em> is a construct that introduces additional logical threads of control (see Clause <a href="../AA-9/">9</a>) without creating a new task. Parallel loops (see <a href="../AA-5/AA-5.5">5.5</a>) and <code><a href="../AA-5/AA-5.6#S0192">parallel_block_statement</a></code>s (see <a href="../AA-5/AA-5.6#Subclause_5.6.1">5.6.1</a>) are parallel constructs. <br /></p>

<AnnotatedOnly>
<MarginText>1.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>parallel construct</strong> &mdash; executable construct that defines multiple activities of a single task that can proceed in parallel, via the execution of multiple logical threads of control<br /></Admonition>
</AnnotatedOnly>

#### Name Resolution Rules

<MarginText>10</MarginText>
<p>The <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code> of a <code><a href="../AA-5/AA-5.1#S0172">statement_identifier</a></code> shall resolve to denote its corresponding implicit declaration (see below). <br /></p>


#### Legality Rules

<MarginText>11</MarginText>
<p>Distinct <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code>s shall be used for all <code><a href="../AA-5/AA-5.1#S0172">statement_identifier</a></code>s that appear in the same body, including inner <code><a href="../AA-5/AA-5.6#S0191">block_statement</a></code>s but excluding inner program units. <br /></p>


#### Static Semantics

<MarginText>12</MarginText>
<p>For each <code><a href="../AA-5/AA-5.1#S0172">statement_identifier</a></code>, there is an implicit declaration (with the specified <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code>) at the end of the <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code> of the innermost <code><a href="../AA-5/AA-5.6#S0191">block_statement</a></code> or body that encloses the <code><a href="../AA-5/AA-5.1#S0172">statement_identifier</a></code>. The implicit declarations occur in the same order as the <code><a href="../AA-5/AA-5.1#S0172">statement_identifier</a></code>s occur in the source text. If a usage name denotes such an implicit declaration, the entity it denotes is the <code><a href="../AA-5/AA-5.1#S0171">label</a></code>, <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code>, or <code><a href="../AA-5/AA-5.6#S0191">block_statement</a></code> with the given <code><a href="../AA-5/AA-5.1#S0172">statement_identifier</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>12.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We talk in terms of individual <code><a href="../AA-5/AA-5.1#S0172">statement_identifier</a></code>s here rather than in terms of the corresponding statements, since a given <code><a href="../AA-5/AA-5.1#S0167">statement</a></code> may have multiple <code><a href="../AA-5/AA-5.1#S0172">statement_identifier</a></code>s.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.b</MarginText>
<Admonition type="aarm" aarm="note">
A <code><a href="../AA-5/AA-5.6#S0191">block_statement</a></code> that has no explicit <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code> has an implicit empty <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code>, so this rule can safely refer to the <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code> of a <code><a href="../AA-5/AA-5.6#S0191">block_statement</a></code>.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.c</MarginText>
<Admonition type="aarm" aarm="note">
The scope of a declaration starts at the place of the declaration itself (see <a href="../AA-8/AA-8.2">8.2</a>). In the case of a label, loop, or block name, it follows from this rule that the scope of the implicit declaration starts before the first explicit occurrence of the corresponding name, since this occurrence is either in a statement label, a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code>, a <code><a href="../AA-5/AA-5.6#S0191">block_statement</a></code>, or a <code><a href="../AA-5/AA-5.8#S0194">goto_statement</a></code>. An implicit declaration in a <code><a href="../AA-5/AA-5.6#S0191">block_statement</a></code> may hide a declaration given in an outer program unit or <code><a href="../AA-5/AA-5.6#S0191">block_statement</a></code> (according to the usual rules of hiding explained in <a href="../AA-8/AA-8.3">8.3</a>).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.d</MarginText>
<Admonition type="aarm" aarm="note">
The syntax rule for <code><a href="../AA-5/AA-5.1#S0171">label</a></code> uses <code><a href="../AA-5/AA-5.1#S0172">statement_identifier</a></code> which is a <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code> (not a <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code>), because labels are implicitly declared. The same applies to loop and block names. In other words, the <code><a href="../AA-5/AA-5.1#S0171">label</a></code> itself is not the defining occurrence; the implicit declaration is.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.e</MarginText>
<Admonition type="aarm" aarm="note">
We cannot consider the <code><a href="../AA-5/AA-5.1#S0171">label</a></code> to be a defining occurrence. An example that can tell the difference is this: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.f</MarginText>
<CodeBlock language="ada">
declare{"\n"}    -- Label Foo is implicitly declared here.{"\n"}begin{"\n"}    for Foo in ... loop{"\n"}        ...{"\n"}        {"<"}{"<"}Foo{">"}{">"} -- Illegal.{"\n"}        ...{"\n"}    end loop;{"\n"}end;{"\n"}  {"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>12.g/5</MarginText>
<MarginInfo items={["AI05-0299-1", "AI12-0449-1"]} />
<Admonition type="aarm" aarm="note">
The label in this example is hidden from itself by the loop parameter with the same name; the example is illegal. We considered creating a new syntactic category name, separate from <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code> and <code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code>, for use in the case of statement labels. However, that would confuse the rules in <a href="../AA-8/AA-8.3">8.3</a> , so we didn't do it. <br /></Admonition>
</AnnotatedOnly>

<MarginText>12.1/3</MarginText>
<MarginInfo items={["AI05-0179-1"]} />
<p>If one or more <code><a href="../AA-5/AA-5.1#S0171">label</a></code>s end a <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code>, an implicit <code><a href="../AA-5/AA-5.1#S0170">null_statement</a></code> follows the <code><a href="../AA-5/AA-5.1#S0171">label</a></code>s before any following constructs.<br /></p>

<AnnotatedOnly>
<MarginText>12.g.1/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The semantics of a <code><a href="../AA-5/AA-5.8#S0194">goto_statement</a></code> is defined in terms of the statement having (following) that label. Thus we ensure that every label has a following statement, which might be implicit. <br /></Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>13</MarginText>
<p>The execution of a <code><a href="../AA-5/AA-5.1#S0170">null_statement</a></code> has no effect.<br /></p>

<MarginText>14/2</MarginText>
<MarginInfo items={["AI95-00318-02"]} />
<p>A <em>transfer of control</em> is the run-time action of an <code><a href="../AA-5/AA-5.7#S0193">exit_statement</a></code>, return statement, <code><a href="../AA-5/AA-5.8#S0194">goto_statement</a></code>, or <code><a href="../AA-9/AA-9.5#S0265">requeue_statement</a></code>, selection of a <code><a href="../AA-9/AA-9.7#S0275">terminate_alternative</a></code>, raising of an exception, or an abort, which causes the next action performed to be one other than what would normally be expected from the other rules of the language. [As explained in <a href="../AA-7/AA-7.6#Subclause_7.6.1">7.6.1</a>, a transfer of control can cause the execution of constructs to be completed and then left, which may trigger finalization.]<br /></p>

<MarginText>15</MarginText>
<p>The execution of a <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> consists of the execution of the individual <code><a href="../AA-5/AA-5.1#S0167">statement</a></code>s in succession until the <code>sequence_</code> is completed. <br /></p>

<AnnotatedOnly>
<MarginText>15.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>It could be completed by reaching the end of it, or by a transfer of control. <br /></Admonition>
</AnnotatedOnly>

<MarginText>16/5</MarginText>
<MarginInfo items={["AI12-0119-1"]} />
<p>Within a parallel construct, if a transfer of control out of the construct is initiated by one of the logical threads of control, an attempt is made to <em>cancel</em> all other logical threads of control initiated by the parallel construct. Once all other logical threads of control of the construct either complete or are canceled, the transfer of control occurs. If two or more logical threads of control of the same construct initiate such a transfer of control concurrently, one of them is chosen arbitrarily and the others are canceled.<br /></p>

<MarginText>17/5</MarginText>
<MarginInfo items={["AI12-0119-1"]} />
<p>When a logical thread of control is canceled, the cancellation causes it to complete as though it had performed a transfer of control to the point where it would have finished its execution. Such a cancellation is deferred while the logical thread of control is executing within an abort-deferred operation (see <a href="../AA-9/AA-9.8">9.8</a>), and may be deferred further, but not past a point where the logical thread initiates a new nested parallel construct or reaches an exception handler that is outside such an abort-deferred operation. <br /></p>


#### Bounded (Run-Time) Errors

<MarginText>18/5</MarginText>
<MarginInfo items={["AI12-0119-1", "AI12-0442-1"]} />
<p>During the execution of a parallel construct, it is a bounded error to invoke an operation that is potentially blocking (see <a href="../AA-9/AA-9.5">9.5</a>). Program_Error is raised if the error is detected by the implementation; otherwise, the execution of the potentially blocking operation can either proceed normally, or it can result in the indefinite blocking of some or all of the logical threads of control making up the current task. <br /></p>

<AnnotatedOnly>
<MarginText>19/5</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   A <code><a href="../AA-5/AA-5.1#S0172">statement_identifier</a></code> that appears immediately within the declarative region of a named <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> or an <code><a href="../AA-9/AA-9.5#S0258">accept_statement</a></code> is nevertheless implicitly declared immediately within the declarative region of the innermost enclosing body or <code><a href="../AA-5/AA-5.6#S0191">block_statement</a></code>; in other words, the expanded name for a named statement is not affected by whether the statement occurs inside or outside a named loop or an <code><a href="../AA-9/AA-9.5#S0258">accept_statement</a></code> &mdash; only nesting within <code><a href="../AA-5/AA-5.6#S0191">block_statement</a></code>s is relevant to the form of its expanded name. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>19.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Each comment in the following example gives the expanded name associated with an entity declared in the task body: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>

task body Compute is{"\n"}   Sum : Integer := 0;                       -- Compute.Sum{"\n"}begin{"\n"} Outer:                                      -- Compute.Outer{"\n"}   for I in 1..10 loop     -- Compute.Outer.I{"\n"}    Blk:                                     -- Compute.Blk{"\n"}      declare{"\n"}         Sum : Integer := 0;                 -- Compute.Blk.Sum{"\n"}      begin{"\n"}         accept Ent(I : out Integer; J : in Integer) do{"\n"}                                             -- Compute.Ent.I, Compute.Ent.J{"\n"}            Compute.Ent.I := Compute.Outer.I;{"\n"}          Inner:                             -- Compute.Blk.Inner{"\n"}            for J in 1..10 loop{"\n"}                                             -- Compute.Blk.Inner.J{"\n"}               Sum := Sum + Compute.Blk.Inner.J * Compute.Ent.J;{"\n"}            end loop Inner;{"\n"}         end Ent;{"\n"}         Compute.Sum := Compute.Sum + Compute.Blk.Sum;{"\n"}      end Blk;{"\n"}   end loop Outer;{"\n"}   Record_Result(Sum);{"\n"}end Compute;{"\n"}
