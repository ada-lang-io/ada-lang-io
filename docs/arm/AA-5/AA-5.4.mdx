#### Name Resolution Rules

<MarginText>4/3</MarginText>
<MarginInfo items={["AI05-0188-1"]} /><p>{"{"}<em>{"AI05-0188-1"}</em>{"}"} <br />The <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is expected to be of any discrete type. The expected type for each <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> is the type of the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>. <br /></p>


#### Legality Rules

<MarginText>5/3</MarginText>
<MarginInfo items={["AI05-0153-3"]} /><p>{"{"}<em>{"AI05-0153-3"}</em>{"}"} <br />The <code><a href="../AA-4/AA-4.4#S0133">choice_expression</a></code>s, <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code>s, and <code><a href="../AA-3/AA-3.5#S0037">range</a></code>s given as <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>s of a <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> shall be static. [A <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> <strong>others</strong>, if present, shall appear alone and in the last <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code>.]<br /></p>

<MarginText>6/3</MarginText>
<MarginInfo items={["AI05-0188-1", "AI05-0240-1"]} /><p>{"{"}<em>{"AI05-0188-1"}</em>{"}"} {"{"}<em>{"AI05-0240-1"}</em>{"}"} <br />The possible values of the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> shall be covered (see <a href="../AA-3/AA-3.8#Subclause_3.8.1">3.8.1</a>) as follows: <br /></p>

<MarginText>6.a/3</MarginText>
<MarginInfo items={["AI05-0240-1"]} /><Admonition type="aarm" aarm="discussion" title="Discussion: ">
<strong></strong>{"{"}<em>{"AI05-0240-1"}</em>{"}"} <br />The meaning of "covered" here and in the following rules is that of the term "cover a value" that is defined in <a href="../AA-3/AA-3.8#Subclause_3.8.1">3.8.1</a>. <br /></Admonition>

<MarginText>7/4</MarginText>
<MarginInfo items={["AI05-0003-1", "AI05-0153-3", "AI05-0188-1", "AI05-0262-1", "AI12-0071-1"]} /><ul>
<li>{"{"}<em>{"AI05-0003-1"}</em>{"}"} {"{"}<em>{"AI05-0153-3"}</em>{"}"} {"{"}<em>{"AI05-0188-1"}</em>{"}"} {"{"}<em>{"AI05-0262-1"}</em>{"}"} {"{"}<em>{"AI12-0071-1"}</em>{"}"} <br />If the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> [(including a <code><a href="../AA-4/AA-4.6#S0162">type_conversion</a></code>, <code><a href="../AA-4/AA-4.7#S0163">qualified_expression</a></code>, or <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code>)] having a static and constrained nominal subtype, then each non-<strong>others</strong> <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> shall cover only values in that subtype that satisfy its predicates (see <a href="../AA-3/AA-3.2#Subclause_3.2.4">3.2.4</a>), and each value of that subtype that satisfies its predicates shall be covered by some <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> [(either explicitly or by <strong>others</strong>)]. <br /></li>
</ul>
<MarginText>7.a</MarginText>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>Although not official <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s of objects, a value conversion still has a defined nominal subtype, namely its target subtype. See <a href="../AA-4/AA-4.6">4.6</a>. <br /></Admonition>

<MarginText>8/3</MarginText>
<MarginInfo items={["AI05-0188-1"]} /><ul>
<li>{"{"}<em>{"AI05-0188-1"}</em>{"}"} <br />If the type of the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is <em>root_integer</em>, <em>universal_integer</em>, or a descendant of a formal scalar type, then the <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> shall have an <strong>others</strong> <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>. <br /></li>
</ul>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<strong></strong>This is because the base range is implementation defined for <em>root_integer</em> and <em>universal_integer</em>, and not known statically in the case of a formal scalar type. <br /></Admonition>

<MarginText>9/3</MarginText>
<MarginInfo items={["AI05-0188-1"]} /><ul>
<li>{"{"}<em>{"AI05-0188-1"}</em>{"}"} <br />Otherwise, each value of the base range of the type of the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> shall be covered [(either explicitly or by <strong>others</strong>)]. <br /></li>
</ul>
<MarginText>10</MarginText>
<p>Two distinct <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>s of a <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> shall not cover the same value. <br /></p>

<MarginText>10.a/3</MarginText>
<MarginInfo items={["AI05-0188-1"]} /><Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>{"{"}<em>{"AI05-0188-1"}</em>{"}"} <br />The goal of these coverage rules is that any possible value of the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> should be covered by exactly one <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> of the <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code>, and that this should be checked at compile time. The goal is achieved in most cases, but there are two minor loopholes: <br /></Admonition>

<MarginText>10.b</MarginText>
<ul>
<li>If the expression reads an object with an invalid representation (e.g. an uninitialized object), then the value can be outside the covered range. This can happen for static constrained subtypes, as well as nonstatic or unconstrained subtypes. It cannot, however, happen if the <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> has the <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> <strong>others</strong>, because <strong>others</strong> covers all values, even those outside the subtype.<br /></li>
<MarginText>10.c/3</MarginText>
<MarginInfo items={["AI95-00114-01", "AI05-0188-1"]} /><li>{"{"}<em>{"AI95-00114-01"}</em>{"}"} {"{"}<em>{"AI05-0188-1"}</em>{"}"} <br />If the compiler chooses to represent the value of an expression of an unconstrained subtype in a way that includes values outside the bounds of the subtype, then those values can be outside the covered range. For example, if X: Integer := Integer'Last;, and the case <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is X+1, then the implementation might choose to produce the correct value, which is outside the bounds of Integer. (It might raise Constraint_Error instead.) This case can only happen for nongeneric subtypes that are either unconstrained or nonstatic (or both). It can only happen if there is no <strong>others</strong> <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>. <br /></li>
</ul>
<MarginText>10.d</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
In the uninitialized variable case, the value might be anything; hence, any alternative can be chosen, or Constraint_Error can be raised. (We intend to prevent, however, jumping to random memory locations and the like.) In the out-of-range case, the behavior is more sensible: if there is an <strong>others</strong>, then the implementation may choose to raise Constraint_Error on the evaluation of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> (as usual), or it may choose to correctly evaluate the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> and therefore choose the <strong>others</strong> alternative. Otherwise (no <strong>others</strong>), Constraint_Error is raised either way &mdash; on the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> evaluation, or for the <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> itself.<br /></Admonition>

<MarginText>10.e</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
For an enumeration type with a discontiguous set of internal codes (see <a href="../AA-13/AA-13.4">13.4</a>), the only way to get values in between the proper values is via an object with an invalid representation; there is no &ldquo;out-of-range&rdquo; situation that can produce them. <br /></Admonition>


#### Dynamic Semantics

<MarginText>11/3</MarginText>
<MarginInfo items={["AI05-0188-1"]} /><p>{"{"}<em>{"AI05-0188-1"}</em>{"}"} <br />For the execution of a <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code>, the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is first evaluated.<br /></p>

<MarginText>12/3</MarginText>
<MarginInfo items={["AI05-0188-1"]} /><p>{"{"}<em>{"AI05-0188-1"}</em>{"}"} <br />If the value of the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is covered by the <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code> of some <code><a href="../AA-5/AA-5.4#S0177">case_statement_alternative</a></code>, then the <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> of the <code>_alternative</code> is executed.<br /></p>

<MarginText>13</MarginText>
<p>Otherwise (the value is not covered by any <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code>, perhaps due to being outside the base range), Constraint_Error is raised. <br /></p>

<MarginText>13.a/5</MarginText>
<MarginInfo items={["AI12-0005-1"]} /><Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>{"{"}<em>{"AI12-0005-1"}</em>{"}"} <br />In this case, the value fails to satisfy its (static) predicate (possible when the predicate is disabled), is outside the base range of its type, or is an invalid representation.<br /></Admonition>

<MarginText>14/5</MarginText>
<MarginInfo items={["AI12-0440-1"]} /><Admonition type="aarm" aarm="note" title="Note: ">
NOTE 1   {"{"}<em>{"AI12-0440-1"}</em>{"}"} <br />The execution of a <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> chooses one and only one alternative. Qualification of the expression of a <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> by a static subtype can often be used to limit the number of choices that can  be given explicitly. <br /></Admonition>

#### Incompatibilities With Ada 83

<MarginText>18.a.1/1</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
In Ada 95, <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code>s and <code><a href="../AA-4/AA-4.6#S0162">type_conversion</a></code>s are <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s, whereas in Ada 83, they were <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s. Therefore, if the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> is a <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code> or <code><a href="../AA-4/AA-4.6#S0162">type_conversion</a></code>, and the result subtype is static, it is illegal to specify a choice outside the bounds of the subtype. For this case in Ada 83 choices only are required to be in the base range of the type.<br /></Admonition>

<MarginText>18.a.2/1</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
In addition, the rule about which choices must be covered is unchanged in Ada 95. Therefore, for a <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> whose <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is a <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code> or <code><a href="../AA-4/AA-4.6#S0162">type_conversion</a></code>, Ada 83 required covering all choices in the base range, while Ada 95 only requires covering choices in the bounds of the subtype. If the <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> does not include an <strong>others</strong> <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>, then a legal Ada 83 <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> will be illegal in Ada 95 if the bounds of the subtype are different than the bounds of the base type. <br /></Admonition>


#### Extensions to Ada 83

<MarginText>18.a</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
In Ada 83, the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> in a <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> is not allowed to be of a generic formal type. This restriction is removed in Ada 95; an <strong>others</strong> <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> is required instead.<br /></Admonition>

<MarginText>18.b</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
In Ada 95, a function call is the name of an object; this was not true in Ada 83 (see <a href="../AA-4/AA-4.1">4.1</a>, &ldquo;<a href="../AA-4/AA-4.1">Names</a>&rdquo;). This change makes the following <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> legal: <br /></Admonition>

<MarginText>18.c</MarginText>
<CodeBlock language="ada">
subtype S is Integer range 1..2;{"\n"}function F return S;{"\n"}case F is{"\n"}   when 1 ={">"} ...;{"\n"}   when 2 ={">"} ...;{"\n"}   -- No others needed.{"\n"}end case;{"\n"}

</CodeBlock>
<MarginText>18.d/3</MarginText>
<MarginInfo items={["AI05-0005-1"]} /><Admonition type="aarm" aarm="note" title="Note: ">
{"{"}<em>{"AI05-0005-1"}</em>{"}"} <br />Note that the result subtype given in a function <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code> is ignored; for a <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> whose expression calls a such a function, the full coverage rules are checked using the result subtype of the original function. Note that predefined operators such as "+" have an unconstrained result subtype (see <a href="../AA-4/AA-4.5#Subclause_4.5.1">4.5.1</a>). Note that generic formal functions do not have static result subtypes. Note that the result subtype of an inherited subprogram need not correspond to any nameable subtype; there is still a perfectly good result subtype, though. <br /></Admonition>


#### Wording Changes from Ada 83

<MarginText>18.e</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
Ada 83 forgot to say what happens for &ldquo;legally&rdquo; out-of-bounds values.<br /></Admonition>

<MarginText>18.f</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
We take advantage of rules and terms (e.g. <em>cover a value</em>) defined for <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>s and <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code>s in <a href="../AA-3/AA-3.8#Subclause_3.8.1">3.8.1</a>, &ldquo;<a href="../AA-3/AA-3.8#Subclause_3.8.1">Variant Parts and Discrete Choices</a>&rdquo;.<br /></Admonition>

<MarginText>18.g</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
In the Name Resolution Rule for the case expression, we no longer need RM83-5.4(3)'s &ldquo;which must be determinable independently of the context in which the expression occurs, but using the fact that the expression must be of a discrete type&rdquo;, because the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is now a complete context. See <a href="../AA-8/AA-8.6">8.6</a>, &ldquo;<a href="../AA-8/AA-8.6">The Context of Overload Resolution</a>&rdquo;.<br /></Admonition>

<MarginText>18.h</MarginText>
<Admonition type="aarm" aarm="note" title="Note: ">
Since <code><a href="../AA-4/AA-4.6#S0162">type_conversion</a></code>s are now defined as <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s, their coverage rule is now covered under the general rule for <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s, rather than being separated out along with <code><a href="../AA-4/AA-4.7#S0163">qualified_expression</a></code>s. <br /></Admonition>
