<MarginText>1/5</MarginText>
<MarginInfo items={["AI12-0315-1"]} />
<p>An <em>image</em> of a value is a string representing the value in display form. The attributes Image, Wide_Image, and Wide_Wide_Image are available to produce the image of a value as a String, Wide_String, or Wide_Wide_String (respectively). User-defined images for a given type can be implemented by overriding the default implementation of the attribute Put_Image. <br /></p>


#### Static Semantics

<MarginText>2/5</MarginText>
<MarginInfo items={["AI12-0020-1", "AI12-0315-1"]} />
<p>For every subtype S of a type T other than <em>universal_real</em> or <em>universal_fixed</em>, the following type-related operational attribute is defined:<br /></p>

<MarginText>3/5</MarginText>
<dt><br/>S'Put_Image</dt>
<MarginInfo items={["AI12-0020-1", "AI12-0320-1", "AI12-0340-1"]} />
<dl>
<dd>S'Put_Image denotes a procedure with the following specification: <br /></dd>
</dl>
<MarginText>3.1/5</MarginText>
<CodeBlock language="ada">
procedure S'Put_Image{"\n"}   (Buffer : in out {"\n"}             Ada.Strings.Text_Buffers.Root_Buffer_Type'Class;{"\n"}    Arg   : in T);{"\n"}

</CodeBlock>
<MarginText>3.2/5</MarginText>
<MarginInfo items={["AI12-0315-1", "AI12-0340-1"]} />
<dl>
<dd>The default implementation of S'Put_Image writes (using Wide_Wide_Put) an <em>image</em> of the value of <em>Arg</em>.<br /></dd>
</dl>
<MarginText>4/5</MarginText>
<MarginInfo items={["AI12-0020-1", "AI12-0419-1"]} />
<p>The Put_Image attribute may be specified for any specific type T either via an <code><a href="../AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> or via an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> specifying the Put_Image aspect of the type.[ The Put_Image aspect is not inherited, but rather is implicitly composed for derived types, as defined below.]<br /></p>

<AnnotatedOnly>
<MarginText>4.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Aspect Description for </strong><strong>Put_Image: </strong>Procedure to define the image of a given type.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.b/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>In contrast, the Image, Wide_Image, and Wide_Wide_Image attributes and their associated aspects can not be specified. The behavior of any of these attributes is defined in terms of calls to the corresponding Put_Image procedure, so changes in their behavior may be accomplished via a Put_Image specification.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.c/5</MarginText>
<Admonition type="aarm" aarm="note">
In earlier versions of Ada, Image and related attributes were defined only for scalar types. The definition of these attributes is now very different, but there should be no change in the behavior of existing programs as a result of these changes. <br /></Admonition>
</AnnotatedOnly>

<MarginText>5/5</MarginText>
<MarginInfo items={["AI12-0427-1"]} />
<p>For an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> or <code><a href="../AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> specifying Put_Image, the subtype of the <em>Arg</em> parameter shall be the first subtype or the base subtype if scalar, and the first subtype if not scalar.<br /></p>

<MarginText>6/5</MarginText>
<MarginInfo items={["AI12-0020-1", "AI12-0419-1"]} />
<p>The behavior of the default implementation of S'Put_Image depends on the class of T.<br /></p>

<MarginText>7/5</MarginText>
<MarginInfo items={["AI12-0020-1", "AI12-0419-1", "AI12-0435-1"]} />
<p>For an untagged derived type, or a null extension, the default implementation of T'Put_Image invokes the Put_Image for its parent type on a conversion of the parameter of type T to the parent type.<br /></p>

<MarginText>8/5</MarginText>
<MarginInfo items={["AI12-0020-1", "AI12-0340-1", "AI12-0384-2", "AI12-0419-1"]} />
<p>For a nonderived elementary type, the implementation is equivalent to: <br /></p>

<MarginText>9/5</MarginText>
<CodeBlock language="ada">
procedure Scalar_Type'Put_Image{"\n"}  (Buffer : in out Ada.Strings.Text_Buffers.Root_Buffer_Type'Class;{"\n"}   Arg    : in Scalar_Type) is{"\n"}begin{"\n"}   Buffer.Wide_Wide_Put ({"<"}described below{">"});{"\n"}end Scalar_Type'Put_Image;{"\n"}

</CodeBlock>
<MarginText>10/5</MarginText>
<p>where the Wide_Wide_String value written out to the text buffer is defined as follows:<br /></p>

<MarginText>11/5</MarginText>
<MarginInfo items={["AI12-0020-1"]} />
<ul>
<li>For an integer type, the image written out is the corresponding decimal literal, without underlines, leading zeros, exponent, or trailing spaces, but with a single leading character that is either a minus sign or a space.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>11.a/5</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>If the machine supports negative zeros for signed integer types, it is not specified whether " 0" or "-0" should be returned for negative zero. We don't have enough experience with such machines to know what is appropriate, and what other languages do. In any case, the implementation should be consistent. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.b/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>We allow S'Put_Image when S is <em>universal_integer</em> or <em>root_integer</em>, because the details of the desired string do not depend on properties of an integer type. While S'Put_Image cannot be called directly for these types (as they cannot be named), it can be called as part of evaluating an Image attribute. Note that other rules of the language ensure that an implementation can evaluate any <em>universal_integer</em> attribute using type <em>root_integer</em>; therefore, Constraint_Error could be raised by the evaluation of an Image attribute if the value of the prefix is outside of the range of <em>root_integer</em>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>12/5</MarginText>
<MarginInfo items={["AI12-0020-1"]} />
<ul>
<li>For an enumeration type, the image written out is either the corresponding identifier in upper case or the corresponding character literal (including the two apostrophes); neither leading nor trailing spaces are included. For a <em>nongraphic character</em> (a value of a character type that has no enumeration literal associated with it), the value is a corresponding language-defined name in upper case (for example, the image of the nongraphic character identified as <em>nul</em> is "NUL" &mdash; the quotes are not part of the image).<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>12.a/5</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>For an enumeration type T that has "holes" (caused by an enumeration_representation_clause), T'Put_Image should raise Program_Error if the value is one of the holes (which is a bounded error anyway, since holes can be generated only via uninitialized variables and similar things). <br /></Admonition>
</AnnotatedOnly>

<MarginText>13/5</MarginText>
<MarginInfo items={["AI12-0020-1"]} />
<ul>
<li>For a floating point type, the image written out is a decimal real literal best approximating the value (rounded away from zero if halfway between) with a single leading character that is either a minus sign or a space, a single digit (that is nonzero unless the value is zero), a decimal point, S'Digits-1 (see <a href="../AA-3/AA-3.5#Subclause_3.5.8">3.5.8</a>) digits after the decimal point (but one if S'Digits is one), an upper case E, the sign of the exponent (either + or -), and two or more digits (with leading zeros if necessary) representing the exponent. If S'Signed_Zeros is True, then the leading character is a minus sign for a negatively signed zero.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>13.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>Leading zeros are present in the exponent only if necessary to make the exponent at least two digits. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>13.b/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This image is intended to conform to that produced by Text_IO.Float_IO.Put in its default format. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>13.c/5</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>The rounding direction is specified here to ensure portability of output results. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>13.d/5</MarginText>
<MarginInfo items={["AI12-0315-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>We do not allow S'Put_Image when S is <em>universal_real</em>, as the details of the desired string depend on the properties of the (specific) type of S. Specifically, <em>universal_real</em> does not have a defined value for S'Digits. <br /></Admonition>
</AnnotatedOnly>

<MarginText>14/5</MarginText>
<MarginInfo items={["AI12-0020-1"]} />
<ul>
<li>For a fixed point type, the image written out is a decimal real literal best approximating the value (rounded away from zero if halfway between) with a single leading character that is either a minus sign or a space, one or more digits before the decimal point (with no redundant leading zeros), a decimal point, and S'Aft (see <a href="../AA-3/AA-3.5#Subclause_3.5.10">3.5.10</a>) digits after the decimal point.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>14.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This image is intended to conform to that produced by Text_IO.Fixed_IO.Put. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.b/5</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>The rounding direction is specified here to ensure portability of output results.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.c/5</MarginText>
<Admonition type="aarm" aarm="note">
For a machine that supports negative zeros, it is not specified whether " 0.000" or "-0.000" is returned. See corresponding comment above about integer types with signed zeros. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.d/5</MarginText>
<MarginInfo items={["AI12-0315-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>We do not allow S'Put_Image when S is <em>universal_fixed</em>, as the details of the desired string depend on the properties of the (specific) type of S. Specifically, <em>universal_fixed</em> does not have a defined value for S'Aft. <br /></Admonition>
</AnnotatedOnly>

<MarginText>15/5</MarginText>
<MarginInfo items={["AI12-0020-1"]} />
<ul>
<li>For an access type (named or anonymous), the image written out depends on whether the value is <strong>null</strong>. If it is <strong>null</strong>, then the image is "NULL". Otherwise the image is a left parenthesis followed by "ACCESS", a space, and a sequence of graphic characters, other than space or right parenthesis, representing the location of the designated object, followed by a right parenthesis, as in "(ACCESS FF0012AC)".<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>15.a/5</MarginText>
<MarginInfo items={["AI12-0315-1"]} />
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>S'Put_Image is defined for <em>universal_access</em>, but it can never be called (as no legal <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of Image has that type, and that type cannot be named preventing direct calls). <br /></Admonition>
</AnnotatedOnly>

<MarginText>16/5</MarginText>
<MarginInfo items={["AI12-0020-1", "AI12-0419-1", "AI12-0435-1"]} />
<p>For a nonnull type extension, the default implementation of T'Put_Image depends on whether there exists a noninterface ancestor of T (other than T itself) for which the Put_Image aspect has been directly specified. If so, then T'Put_Image will generate an image based on extension aggregate syntax where the ancestor type of the extension aggregate is the nearest ancestor type whose Put_Image aspect has been specified. If no such ancestor exists, then the default implementation of T'Put_Image is the same as described below for a nonderived record type.<br /></p>

<AnnotatedOnly>
<MarginText>16.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This might generate an image such as "(This Text Was User-Generated with C1 ={">"}  123, C2 ={">"}  456)" where the "This Text was User-Generated" portion of the text was generated by the call to the user-specified Put_Image routine. <br /></Admonition>
</AnnotatedOnly>

<MarginText>17/5</MarginText>
<MarginInfo items={["AI12-0419-1"]} />
<p>For a specific, nonderived composite type:<br /></p>

<MarginText>18/5</MarginText>
<MarginInfo items={["AI12-0020-1"]} />
<ul>
<li>If the default implementation of Put_Image writes components, the order in which components are written is the same canonical order in which components of a composite type T are written out by the default implementation of T'Write. [This is also the order that is used in determining the meaning of a positional aggregate of type T.]<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>18.a/5</MarginText>
<MarginInfo items={["AI12-0020-1", "AI12-0384-2"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>In general, the default implementation of T'Put_Image for a composite type will involve some sequence of calls to Put and its Wide and Wide_Wide variants and calls to the Put_Image procedures of component types and, in the case of an array type, index types. The Put calls may pass in either literal values (e.g., "(", ")", "'(", " ={">"} ", or ", "), or other things (such as component names for record values, task_id images for tasks, or the Wide_Wide_Expanded_Name of the tag in the class-wide case). <br /></Admonition>
</AnnotatedOnly>

<MarginText>19/5</MarginText>
<MarginInfo items={["AI12-0020-1", "AI12-0419-1"]} />
<ul>
<li>For an array type T, the default implementation of T'Put_Image generates an image based on named (not positional) array aggregate syntax (with '[' and ']' as the delimiters) using calls to the Put_Image procedures of the index type(s) and the element type to generate images for values of those types.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>19.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>An array type might generate an image such as: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>19.b/5</MarginText>
<CodeBlock language="ada">
[ 1 ={">"} [ 1 ={">"} [ 123 ={">"} True,  124 ={">"} False]{"\n"}         2 ={">"} [ 123 ={">"} False,  124 ={">"} False]],{"\n"}  2 ={">"} [ 1 ={">"} [ 123 ={">"} True,  124 ={">"} True],{"\n"}         2 ={">"} [ 123 ={">"} True,  124 ={">"} False]]]{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>19.c/5</MarginText>
<Admonition type="aarm" aarm="note">
although perhaps with different white space and/or line breaking. <br /></Admonition>
</AnnotatedOnly>

<MarginText>20/5</MarginText>
<MarginInfo items={["AI12-0020-1"]} />
<ul>
<li>The case of a null array is handled specially, using ranges for index bounds and "{"<"}{">"}" as a syntactic component-value placeholder.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>20.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This might generate an image such as "[ 1 ..  3 ={">"} [ 1 ..  0 ={">"} [ 1 .. 5 ={">"} {"<"}{">"}]]]", where the use of "{"<"}{">"}" (among other things) indicates that the overall array is a null array and has no actual elements. <br /></Admonition>
</AnnotatedOnly>

<MarginText>21/5</MarginText>
<MarginInfo items={["AI12-0020-1", "AI12-0419-1"]} />
<ul>
<li>For a record type (or, as indicated above, a type extension with no noninterface ancestor specifying Put_Image), or a protected type, the default implementation of T'Put_Image generates an image based on named (not positional) record aggregate syntax (except that for a protected type, the initial left parenthesis is followed by "PROTECTED with "). Component names are displayed in upper case, following the rules for the image of an enumeration value. Component values are displayed via calls to the component type's Put_Image procedure.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>21.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This might generate an image such as: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>21.b/5</MarginText>
<CodeBlock language="ada">
"(FOO ={">"} [1 ={">"} 'c',  2 ={">"} 'a',  3 ={">"} 't'], BAR ={">"} TRUE)"{"\n"}

</CodeBlock>
</AnnotatedOnly>
<MarginText>22/5</MarginText>
<MarginInfo items={["AI12-0020-1"]} />
<ul>
<li>The image written out for a record having no components (including any interface type) is "(NULL RECORD)". The image written out for a componentless protected type is "(PROTECTED NULL RECORD)". In the case of a protected type T, a call to the default implementation of T'Put_Image begins only one protected (read-only) action.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>22.a/5</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>The expected, but not required, implementation model for generating the image of a protected record involves the compiler producing a "helper" protected function which T'Put_Image would call. The result type of this function might be a null record; it is only a function because it does not need a write-lock, not because it returns a meaningful result.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>22.b/5</MarginText>
<Admonition type="aarm" aarm="note">
The assertion in the following example should succeed: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>22.c/5</MarginText>
<CodeBlock language="ada">
type T1 (D1, D2 : Positive) is record ... end record; -- untagged{"\n"}type T2 (D : Positive) is new T1 (D1 ={">"} D, D2 ={">"} D);{"\n"}X : T2 (D ={">"} 123) := ... ;{"\n"}pragma Assert (X'Image /= T1(X)'Image);{"\n"}

</CodeBlock>
</AnnotatedOnly>
<MarginText>23/5</MarginText>
<MarginInfo items={["AI12-0020-1"]} />
<ul>
<li>For an undiscriminated task type, the default implementation of T'Put_Image generates an image of the form "(TASK {"<"}task_id_image{">"})" where {"<"}task_id_image{">"} is the result obtained by calling Task_Identification.Image with the id of the given task and then passing that String to Characters.Conversions.To_Wide_Wide_String.<br /></li>
<MarginText>24/5</MarginText>
<MarginInfo items={["AI12-0020-1"]} />
<li>For a discriminated task type, the default implementation of T'Put_Image also includes discriminant values, as in: <br /></li>
</ul>
<MarginText>25/5</MarginText>
<CodeBlock language="ada">
"(TASK {"<"}task_id_image{">"} with D1 ={">"}  123, D2 ={">"}  456)"{"\n"}

</CodeBlock>
<MarginText>26/5</MarginText>
<MarginInfo items={["AI12-0020-1", "AI12-0384-2"]} />
<p>For a class-wide type, the default implementation of T'Put_Image generates an image based on qualified expression syntax. Wide_Wide_Put is called with Wide_Wide_Expanded_Name of <em>Arg</em>'Tag. Then S'Put_Image is called, where S is the specific type identified by <em>Arg</em>'Tag.<br /></p>

<AnnotatedOnly>
<MarginText>26.a/5</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>This will typically require a dispatching call. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>26.b/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This might generate an image such as: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>26.c/5</MarginText>
<CodeBlock language="ada">
SOME_PACKAGE.SOME_TAGGED_TYPE'{"\n"}   (COMPONENT_1 ={">"}  123, COMPONENT_2 ={">"} 456){"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>26.d/5</MarginText>
<Admonition type="aarm" aarm="note">
The parentheses in this case are generated by the call to Some_Tagged_Type'Put_Image. <br /></Admonition>
</AnnotatedOnly>

<MarginText>27/5</MarginText>
<MarginInfo items={["AI12-0020-1", "AI12-0384-2"]} />
<p>[T'Put_Image is the same for both the partial view and full view of T, if T has a partial view.]<br /></p>

<AnnotatedOnly>
<MarginText>27.a/5</MarginText>
<MarginInfo items={["AI12-0384-2"]} />
<Admonition type="aarm" aarm="proof">
<strong></strong>A type-related operational aspect is the same for the full view and partial view of a type. See <a href="../AA-13/AA-13.1">13.1</a>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>28/5</MarginText>
<MarginInfo items={["AI12-0435-1"]} />
<p>In the <code><a href="../AA-6/AA-6.1#S0205">parameter_and_result_profile</a></code> for the default implementation of Put_Image, the subtype of the <em>Arg</em> parameter is the base subtype of <em>T</em> if <em>T</em> is a scalar type, and the first subtype otherwise. For an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> or <code><a href="../AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> specifying Put_Image, the subprogram name shall denote a nonabstract procedure whose second parameter is either of the first subtype of <em>T</em>, or as an option when <em>T</em> is scalar, the base subtype of <em>T</em>.<br /></p>

<AnnotatedOnly>
<MarginText>28.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Put_Image is never an abstract routine, even for an abstract type. Thus, Put_Image and Image can be called for any type. <br /></Admonition>
</AnnotatedOnly>

<MarginText>29/5</MarginText>
<MarginInfo items={["AI12-0020-1"]} />
<p>For every subtype S of a type T, the following attributes are defined:<br /></p>

<MarginText>30/5</MarginText>
<dt><br/>S'Wide_Wide_Image</dt>
<MarginInfo items={["AI12-0020-1", "AI12-0340-1", "AI12-0384-2"]} />
<dl>
<dd>S'Wide_Wide_Image denotes a function with the following specification: <br /></dd>
</dl>
<MarginText>30.1/5</MarginText>
<CodeBlock language="ada">
function S'Wide_Wide_Image(Arg : S'Base){"\n"}  return Wide_Wide_String{"\n"}

</CodeBlock>
<MarginText>30.2/5</MarginText>
<dl>
<dd>S'Wide_Wide_Image calls S'Put_Image passing <em>Arg</em> (which will typically store a sequence of character values in a text buffer) and then returns the result of retrieving the contents of that buffer with function Wide_Wide_Get. The lower bound of the result is one. [Any exception propagated by the call of S'Put_Image is propagated.]<br /></dd>
<MarginText>31/5</MarginText>
<dt><br/>S'Wide_Image</dt>
<MarginInfo items={["AI12-0020-1", "AI12-0340-1", "AI12-0384-2"]} />
<dd>S'Wide_Image denotes a function with the following specification: <br /></dd>
</dl>
<MarginText>32/5</MarginText>
<CodeBlock language="ada">
function S'Wide_Image(Arg : S'Base){"\n"}  return Wide_String{"\n"}

</CodeBlock>
<MarginText>33/5</MarginText>
<dl>
<dd>S'Wide_Image calls S'Put_Image passing <em>Arg</em> (which will typically store a sequence of character values in a text buffer) and then returns the result of retrieving the contents of that buffer with function Wide_Get. The lower bound of the result is one. [Any exception propagated by the call of S'Put_Image is propagated.]<br /></dd>
<MarginText>34/5</MarginText>
<dt><br/>S'Image</dt>
<MarginInfo items={["AI12-0020-1", "AI12-0340-1", "AI12-0384-2"]} />
<dd>S'Image denotes a function with the following specification: <br /></dd>
</dl>
<MarginText>35/5</MarginText>
<CodeBlock language="ada">
function S'Image(Arg : S'Base){"\n"}  return String{"\n"}

#### Implementation Permissions

<MarginText>41/5</MarginText>
<MarginInfo items={["AI12-0020-1"]} />
<p>An implementation may transform the image generated by the default implementation of S'Put_Image for a composite subtype S in the following ways:<br /></p>

<MarginText>42/5</MarginText>
<ul>
<li>If S is a composite subtype, the leading character of the image <em>M</em> of a component value or index value is a space, and the immediately preceding character (if any) is an open parenthesis, open bracket, or space, then the leading space of the image <em>M</em> may be omitted. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>42.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This means that it is permitted to generate "[1 ={">"} 123, 2 ={">"} 456]" instead of "[ 1 ={">"}  123,  2 ={">"}  456]". <br /></Admonition>
</AnnotatedOnly>

<MarginText>43/5</MarginText>
<ul>
<li>If S is an array subtype, the low bound of the array in each dimension equals the low bound of the corresponding index subtype, and the array value is not a null array value, then positional array aggregate syntax may be used.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>43.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This means that it is permitted to generate "[TRUE, TRUE, FALSE]" instead of "[ 1 ={">"} TRUE,  2 ={">"} TRUE,  3 ={">"} FALSE]" if the low bound of the index subtype is one. <br /></Admonition>
</AnnotatedOnly>

<MarginText>44/5</MarginText>
<ul>
<li>If S is an array subtype and the given value can be displayed using <code><a href="../AA-4/AA-4.3#S0116">named_array_aggregate</a></code> syntax where some <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code> identifies more than one index value by identifying a sequence of one or more ranges and values separated by vertical bars, then this image may be generated instead; this may involve the reordering of component values.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>44.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This means that it is permitted to generate "[ 1 ..  2 |  5 ={">"} TRUE,  3 ..  4 ={">"} FALSE]" instead of "[ 1 ={">"} TRUE,  2 ={">"} TRUE,  3 ={">"} FALSE,  4 ={">"} FALSE,  5 ={">"} TRUE]". <br /></Admonition>
</AnnotatedOnly>

<MarginText>45/5</MarginText>
<ul>
<li>Similarly, if S is a record subtype (or a discriminated type) and the given value can be displayed using named component association syntax where the length of some component_choice_list is greater than one, then this image may be generated instead; this may involve the reordering of component values.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>45.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This means that it is permitted to generate "(F1 | F2 ={">"} TRUE)" instead of "(F1 ={">"} TRUE, F2 ={">"} TRUE)". <br /></Admonition>
</AnnotatedOnly>

<MarginText>46/5</MarginText>
<MarginInfo items={["AI12-0020-1", "AI12-0340-1", "AI12-0384-2"]} />
<ul>
<li>Additional spaces (Wide_Wide_Characters with position 32), and calls to the New_Line operation of a text buffer, may be inserted to improve readability of the generated image, with the spaces inserted directly or via use of the Increase_Indent and Decrease_Indent procedures.<br /></li>
<MarginText>47/5</MarginText>
<MarginInfo items={["AI12-0384-2"]} />
<li>For a string type, implementations may produce an image corresponding to a string literal.<br /></li>
<MarginText>48/5</MarginText>
<MarginInfo items={["AI12-0384-2"]} />
<li>For an unchecked union type, implementations may raise Program_Error or produce some recognizable image (such as "(UNCHECKED UNION)") that does not require reading the discriminants. <br /></li>
</ul>
<MarginText>49/5</MarginText>
<MarginInfo items={["AI12-0304-1"]} />
<p>For each language-defined nonscalar type T, T'Put_Image may be specified.<br /></p>

<AnnotatedOnly>
<MarginText>49.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This permission applies, in particular, to nonscalar types declared in language-defined generic packages, and to any language-defined private type, even if an implementation chooses to complete it as a scalar type. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>49.b/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>For any language-defined scalar type T, T'Put_Image should not be specified; the Image attribute needs to return the language-defined image for such types. This is important for compatibility: the Image attribute has been available for scalar types for many Ada revisions, and programs can (and do!) depend on its behavior. <br /></Admonition>
</AnnotatedOnly>


#### Implementation Requirements

<MarginText>50/5</MarginText>
<MarginInfo items={["AI12-0304-1"]} />
<p>For each language-defined container type T (that is, each of the Vector, List, Map, Set, Tree, and Holder types defined in the various children of Ada.Containers), T'Put_Image shall be specified so that T'Image produces a result consistent with array aggregate syntax (using '[' and ']' as delimiters) as follows:<br /></p>

<MarginText>51/5</MarginText>
<ul>
<li>Vector images shall be consistent with the default image of an array type with the same index and component types.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>51.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>In particular, this means that the format is 	that of a named array aggregate. We have no recommendation on how to 	handle empty elements; if the implementation can identify them, it may 	wish to display them specially, but otherwise, they're just 	uninitialized elements. <br /></Admonition>
</AnnotatedOnly>

<MarginText>52/5</MarginText>
<ul>
<li>Map images shall be consistent with named array aggregate syntax, using key value images in place of discrete choice names. For example, [Key1 ={">"} Value1, Key2 ={">"} Value2].<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>52.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>There is no recommendation about the order in which key/element pairs occur within a map image. In the case of multiple key values whose corresponding element values have the same image, there is no recommendation about factoring (that is, generating Key1 | Key2 ={">"} Some_Value instead of Key1 ={">"} Some_Value, Key2 ={">"} Some_Value). <br /></Admonition>
</AnnotatedOnly>

<MarginText>53/5</MarginText>
<ul>
<li>Set, List, and Holder images shall be consistent with positional array aggregate syntax. List elements shall occur in order within an image of a list. The image of an empty holder shall be [].<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>53.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>There is no recommendation about the order in which set elements occur within the image of a set. <br /></Admonition>
</AnnotatedOnly>

<MarginText>54/5</MarginText>
<ul>
<li>Tree images (and images of subtrees of trees) shall be consistent with positional array aggregate syntax. For example, [[1, 2], [111, 222, 333]].<br /></li>
</ul>
<MarginText>55/5</MarginText>
<MarginInfo items={["AI12-0304-1"]} />
<p>For each language-defined nonscalar type T that has a primitive language-defined Image function whose profile is type conformant with that of T'Image (for example, Ada.Numerics.Float_Random.State has such an Image function), T'Put_Image shall be specified so that T'Image yields the same result as that Image function.<br /></p>


#### Implementation Advice

<MarginText>56/5</MarginText>
<p>For each language-defined private type T, T'Image should generate an image that would be meaningful based only on the relevant public interfaces, as opposed to requiring knowledge of the implementation of the private type.<br /></p>

<AnnotatedOnly>
<MarginText>56.a/5</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<strong></strong>For each language-defined private type T, T'Image should generate an image that would be meaningful based only on the relevant public interfaces.<br /></Admonition>
