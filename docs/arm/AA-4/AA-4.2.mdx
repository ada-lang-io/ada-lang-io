<MarginText>1</MarginText>
<p>[ A <i>literal</i> represents a value literally, that is, by means of notation suited to its kind.] A literal is either a <code><a href="/docs/arm/AA-2/AA-2.4#S0006">numeric_literal</a></code>, a <code><a href="/docs/arm/AA-2/AA-2.5#S0015">character_literal</a></code>, the literal <b>null</b>, or a <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>1.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>An enumeration literal that is an <code><a href="/docs/arm/AA-2/AA-2.3#S0002">identifier</a></code> rather than a <code><a href="/docs/arm/AA-2/AA-2.5#S0015">character_literal</a></code> is not considered a <i>literal</i> in the above sense, because it involves no special notation &ldquo;suited to its kind&rdquo;. It might more properly be called an <code>enumeration{"_"}identifier</code>, except for historical reasons. <br />
</Admonition>
</AnnotatedOnly>

<MarginInfo items={["AI12-0325-1", "AI12-0373-1"]} />
<p>The expected type for a <code><a href="/docs/arm/AA-4/AA-4.4#S0141">primary</a></code> that is a <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code> shall be a single string type or a type with a specified String{"_"}Literal aspect (see <a href="/docs/arm/AA-4/AA-4.2#Subclause_4.2.1">4.2.1</a>). In either case, the <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code> is interpreted to be of its expected type. If the expected type of an integer literal is a type with a specified Integer{"_"}Literal aspect (see <a href="/docs/arm/AA-4/AA-4.2#Subclause_4.2.1">4.2.1</a>), the literal is interpreted to be of its expected type; otherwise it is interpreted to be of type <i>universal{"_"}integer</i>. If the expected type of a real literal is a type with a specified Real{"_"}Literal aspect (see <a href="/docs/arm/AA-4/AA-4.2#Subclause_4.2.1">4.2.1</a>), it is interpreted to be of its expected type; otherwise, it is interpreted to be of type <i>universal{"_"}real</i>. <br /></p>


#### Legality Rules

<MarginText>5</MarginText>
<p>A <code><a href="/docs/arm/AA-2/AA-2.5#S0015">character_literal</a></code> that is a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> shall correspond to a <code><a href="/docs/arm/AA-3/AA-3.5#S0040">defining_character_literal</a></code> of the expected type, or of the result type of the expected profile.<br /></p>

<MarginText>6/5</MarginText>
<MarginInfo items={["AI12-0295-1", "AI12-0325-1"]} />
<p>If the expected type for a string{"_"}literal is a string type, then for each character of the <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code> there shall be a corresponding <code><a href="/docs/arm/AA-3/AA-3.5#S0040">defining_character_literal</a></code> of the component type of the expected string type.<br /></p>

#### Dynamic Semantics

<MarginText>9/5</MarginText>
<MarginInfo items={["AI12-0249-1"]} />
<p>If its expected type is a numeric type, the evaluation of a numeric literal yields the represented value. [In other cases, the effect of evaluating a numeric literal is determined by the Integer{"_"}Literal or Real{"_"}Literal aspect that applies (see <a href="/docs/arm/AA-4/AA-4.2#Subclause_4.2.1">4.2.1</a>).]<br /></p>

<MarginText>9.1/5</MarginText>
<MarginInfo items={["AI12-0249-1"]} />
<p>The evaluation of the literal <b>null</b> yields the null value of the expected type.<br /></p>

<MarginText>10/5</MarginText>
<MarginInfo items={["AI12-0295-1", "AI12-0325-1"]} />
<p>The evaluation of a <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code> that is a <code><a href="/docs/arm/AA-4/AA-4.4#S0141">primary</a></code> and has an expected type that is a string type, yields an array value containing the value of each character of the sequence of characters of the <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code>, as defined in <a href="/docs/arm/AA-2/AA-2.6">2.6</a>. The bounds of this array value are determined according to the rules for <code><a href="/docs/arm/AA-4/AA-4.3#S0114">positional_array_aggregate</a></code>s (see <a href="/docs/arm/AA-4/AA-4.3#Subclause_4.3.3">4.3.3</a>), except that for a null string literal, the upper bound is the predecessor of the lower bound. [In other cases, the effect of evaluating a <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code> is determined by the String{"_"}Literal aspect that applies (see <a href="/docs/arm/AA-4/AA-4.2#Subclause_4.2.1">4.2.1</a>).]<br /></p>

<MarginText>11/5</MarginText>
<MarginInfo items={["AI12-0295-1", "AI12-0325-1"]} />
<p>For the evaluation of a <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code> of a string type <i>T</i>, a check is made that the value of each character of the <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code> belongs to the component subtype of <i>T</i>. For the evaluation of a null string literal of a string type, a check is made that its lower bound is greater than the lower bound of the base range of the index type. The exception Constraint{"_"}Error is raised if either of these checks fails. <br /></p>

<AnnotatedOnly>
<MarginText>11.a/5</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
<Admonition type="aarm" aarm="ramification">
<b></b>If no predicates apply to the component subtype, the checks on the characters need not involve more than two checks altogether, since one need only check the characters of the string with the lowest and highest position numbers against the range of the component subtype. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   Enumeration literals that are <code><a href="/docs/arm/AA-2/AA-2.3#S0002">identifier</a></code>s rather than <code><a href="/docs/arm/AA-2/AA-2.5#S0015">character_literal</a></code>s follow the normal rules for <code><a href="/docs/arm/AA-2/AA-2.3#S0002">identifier</a></code>s when used in a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> (see <a href="/docs/arm/AA-4/AA-4.1">4.1</a> and <a href="/docs/arm/AA-4/AA-4.1#Subclause_4.1.3">4.1.3</a>). <code><a href="/docs/arm/AA-2/AA-2.5#S0015">Character_literal</a></code>s used as <code><a href="/docs/arm/AA-4/AA-4.1#S0099">selector_name</a></code>s follow the normal rules for expanded names (see <a href="/docs/arm/AA-4/AA-4.1#Subclause_4.1.3">4.1.3</a>). <br />
</Admonition>
</AnnotatedOnly>

## 4.2.1  User-Defined Literals

<MarginText>1/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0249-1"]} />
<p>Using one or more of the aspects defined below, a type may be specified to allow the use of one or more kinds of literals as values of the type. <br /></p>


#### Static Semantics

<MarginText>2/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0249-1", "AI12-0342-1"]} />
<p>The following type-related operational aspects (collectively known as <i>user-defined literal aspects</i>) may be specified for a type <i>T</i>:<br /></p>

<MarginText>3/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0249-1", "AI12-0342-1", "AI12-0373-1"]} />
<dt><br/>Integer{"_"}Literal</dt>
<dl>
<dd>This aspect is specified by a <i>function{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> that statically denotes a function with a result type of <i>T</i> and one <b>in</b> parameter that is of type String and is not explicitly aliased.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>3.a/5_4.2.1</MarginText>
<Admonition type="aarm" aarm="note">
<b>Aspect Description for </b><b>Integer{"_"}Literal: </b>Defines a function to implement user-defined integer literals.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>4/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0249-1", "AI12-0342-1", "AI12-0373-1", "AI12-0394-1"]} />
<dt><br/>Real{"_"}Literal</dt>
<dl>
<dd>This aspect is specified by a <i>function{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> that statically denotes a function with a result type of <i>T</i> and one <b>in</b> parameter that is of type String and is not explicitly aliased, and optionally a second function [(overloading the first) ]with a result type of <i>T</i> and two <b>in</b> parameters of type String that are not explicitly aliased.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>4.a/5_4.2.1</MarginText>
<Admonition type="aarm" aarm="note">
<b>Aspect Description for </b><b>Real{"_"}Literal: </b>Defines a function or functions to implement user-defined real literals.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>5/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0295-1", "AI12-0342-1", "AI12-0373-1"]} />
<dt><br/>String{"_"}Literal</dt>
<dl>
<dd>This aspect is specified by a <i>function{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> that statically denotes a function with a result type of <i>T</i> and one <b>in</b> parameter that is of type Wide{"_"}Wide{"_"}String and is not explicitly aliased.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>5.a/5_4.2.1</MarginText>
<Admonition type="aarm" aarm="note">
<b>Aspect Description for </b><b>String{"_"}Literal: </b>Defines a function to implement user-defined string literals.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0342-1"]} />
<Admonition type="aarm" aarm="ramification">
<b></b>The following example is legal because the resolution of an <code><a href="/docs/arm/AA-13/AA-13.1#S0348">aspect_definition</a></code> for an aspect that is defined to be a subprogram is based on the profile required for that aspect (see <a href="/docs/arm/AA-13/AA-13.1#Subclause_13.1.1">13.1.1</a>): <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.c/5_4.2.1</MarginText>
<CodeBlock language="ada">
package Pkg1 is{"\n"}   type T is record X, Y : Integer; end record{"\n"}     with Integer{"_"}Literal ={">"} Int{"_"}Lit;{"\n"}   function Int{"_"}Lit (X, Y : T) return Duration;    -- Wrong profile.{"\n"}   function Int{"_"}Lit (Lit{"_"}Image : String) return T; -- Right profile.{"\n"}end Pkg1;{"\n"}

</CodeBlock>
</AnnotatedOnly>
<MarginText>6/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0342-1", "AI12-0419-1"]} />
<p>User-defined literal aspects are nonoverridable (see <a href="/docs/arm/AA-13/AA-13.1#Subclause_13.1.1">13.1.1</a>). <br /></p>

<AnnotatedOnly>
<MarginText>6.a/5_4.2.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>This means that in this example<br />

<MarginText>7/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0342-1", "AI12-0427-1"]} />
<p>When a numeric literal is interpreted as a value of a non-numeric type <i>T</i> or a <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code> is interpreted as a value of a type <i>T</i> that is not a string type (see <a href="/docs/arm/AA-4/AA-4.2">4.2</a>), it is equivalent to a call to the subprogram denoted by the corresponding aspect of <i>T</i>: the Integer{"_"}Literal aspect for an integer literal, the Real{"_"}Literal aspect for a real literal, and the String{"_"}Literal aspect for a <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code>. The actual parameter of this notional call is a <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code> representing a sequence of characters that is the same as the sequence of characters in the original numeric literal, or the sequence represented by the original string literal.<br /></p>

<AnnotatedOnly>
<MarginText>7.a/5_4.2.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>This equivalence defines, for example, the nominal type, the nominal subtype, and the accessibility level of a user-defined literal. It also has the consequence that a user-defined literal shall not be of an abstract type (because that would be equivalent to a nondispatching call to an abstract function). This equivalence also defines the Dynamic Semantics of evaluating a user-defined literal.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.b/5_4.2.1</MarginText>
<Admonition type="aarm" aarm="note">
The (sub)type of the actual parameter to this call is determined by the profile of the appropriate aspect, and the bounds of the <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code> are defined by the usual rules for the bounds of a <code><a href="/docs/arm/AA-2/AA-2.6#S0016">string_literal</a></code>. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>8/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0342-1"]} />
<p>Such a literal is said to be a <i>user-defined literal</i>.<br /></p>

<MarginText>9/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0394-1"]} />
<p>When a named number that denotes a value of type <i>universal{"_"}integer</i> is interpreted as a value of a non-numeric type <i>T</i>, it is equivalent to a call to the function denoted by the Integer{"_"}Literal aspect of <i>T</i>. The actual parameter of this notional call is a String having a textual representation of a decimal integer literal optionally preceded by a minus sign, representing the same value as the named number.<br /></p>

<MarginText>10/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0394-1"]} />
<p>When a named number that denotes a value of type <i>universal{"_"}real</i> is interpreted as a value of a non-numeric type <i>T</i>, it is equivalent to a call to the two-parameter function denoted by the Real{"_"}Literal aspect of <i>T</i>, if any. The actual parameters of this notional call are each a String with the textual representation of a decimal integer literal, with the first optionally preceded by a minus sign, where the first String represents the same value as the numerator, and the second the same value as the denominator, of the named number when represented as a rational number in lowest terms, with a positive denominator.<br /></p>


#### Legality Rules

<MarginText>11/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0249-1", "AI12-0295-1", "AI12-0325-1", "AI12-0342-1"]} />
<p>The Integer{"_"}Literal or Real{"_"}Literal aspect shall not be specified for a type <i>T</i> if the full view of <i>T</i> is a numeric type. The String{"_"}Literal aspect shall not be specified for a type <i>T</i> if the full view of <i>T</i> is a string type.<br /></p>

<MarginText>12/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0342-1"]} />
<p>For a nonabstract type, the function directly specified for a user-defined literal aspect shall not be abstract.<br /></p>

<MarginText>13/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0342-1"]} />
<p>For a tagged type with a partial view, a user-defined literal aspect shall not be directly specified on the full type.<br /></p>

<MarginText>14/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0342-1", "AI12-0394-1"]} />
<p>If a nonabstract tagged type inherits any user-defined literal aspect, then each inherited aspect shall be directly specified as a nonabstract function for the type unless the inherited aspect denotes a nonabstract function, or functions, and the type is a null extension.<br /></p>

<MarginText>15/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0394-1"]} />
<p>If a named number that denotes a value of type <i>universal{"_"}integer</i> is interpreted as a value of a non-numeric type <i>T</i>, <i>T</i> shall have an Integer{"_"}Literal aspect. If a named number that denotes a value of type <i>universal{"_"}real</i> is interpreted as a value of a non-numeric type <i>T</i>, <i>T</i> shall have a Real{"_"}Literal aspect, and the aspect shall denote a function that has two <b>in</b> parameters, both of type String, with result of type <i>T</i>.<br /></p>

<MarginText>16/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0249-1", "AI12-0342-1"]} />
<p>In addition to the places where Legality Rules normally apply (see <a href="/docs/arm/AA-12/AA-12.3">12.3</a>), these rules also apply in the private part of an instance of a generic unit. <br /></p>


#### Bounded (Run-Time) Errors

<MarginText>17/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0249-1", "AI12-0325-1", "AI12-0342-1", "AI12-0394-1"]} />
<p>It is a bounded error if the evaluation of a literal or named number that has an expected type with a specified user-defined literal aspect propagates an exception. Either Program{"_"}Error or the exception propagated by the evaluation is raised at the point of use of the value of the literal or named number. If it is recognized prior to run time that evaluation of such a literal or named number will inevitably (if executed) result in such a bounded error, then this may be reported as an error prior to run time.<br /></p>

<AnnotatedOnly>
<MarginText>17.a/5_4.2.1</MarginText>
<MarginInfo items={["AI12-0249-1"]} />
<Admonition type="aarm" aarm="implementation-note">
<b></b>As always, an implementation may apply "as-if" optimizations (those that result in the same external effects in the absence of erroneous execution) to the function calls associated with user-defined literals. In particular, if the function associated with a user-defined literal aspect has a Global aspect that indicates no references to global variables, then a number of optimizations are available to the implementation: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>17.b/5_4.2.1</MarginText>
<ul>
<li>The implementation can evaluate a user-defined literal function at compile-time if it has access to the body of the function (for example, if it is inlined), and that body doesn't reference anything evaluated at runtime. If the compile-time evaluation results in an exception, this bounded error allows the compilation unit to be rejected.<br /></li>
<MarginText>17.c/5_4.2.1</MarginText>
<li>Implementations can use existing permissions (see <a href="/docs/arm/AA-6/AA-6.1#Subclause_6.1.2">6.1.2</a>) to avoid evaluating the function associated with a user-defined literal more than once for a particular literal value. This evaluation can be "hoisted" (done once per compilation unit during the elaboration of the unit) if the compiler can prove that the function doesn't depend on any constants or locals with a runtime value not yet elaborated.<br /></li>
<MarginText>17.d/5_4.2.1</MarginText>
<li>If the literal value is not needed by the execution of the program, the function call can be omitted even if it might have side-effects (again, see <a href="/docs/arm/AA-6/AA-6.1#Subclause_6.1.2">6.1.2</a>). <br /></li>
</ul>
</AnnotatedOnly>
