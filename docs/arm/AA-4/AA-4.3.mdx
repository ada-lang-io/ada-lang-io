<p>[ An <em>aggregate</em> combines component values into a composite value of an array type, record type, or record extension.] <br /></p>

<AnnotatedOnly>
<MarginText>1.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>aggregate</strong> &mdash; construct used to define a value of a composite type by specifying the values of the components of the type<br /></Admonition>
</AnnotatedOnly>

#### Name Resolution Rules

<MarginText>3/5</MarginText>
<MarginInfo items={["AI95-00287-01", "AI12-0127-1", "AI12-0212-1", "AI12-0307-1"]} />
<p>The expected type for an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> shall be a single array type, a single type with the Aggregate aspect specified, or a single descendant of a record type  or of a record extension. <br /></p>

<AnnotatedOnly>
<MarginText>3.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>See <a href="../AA-8/AA-8.6">8.6</a>, &ldquo;<a href="../AA-8/AA-8.6">The Context of Overload Resolution</a>&rdquo; for the meaning of &ldquo;shall be a single ... type&rdquo;. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.b/3</MarginText>
<MarginInfo items={["AI05-0005-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>There are additional rules for each kind of aggregate. These aggregate rules are additive; a legal expression needs to satisfy all of the applicable rules. That means the rule given here must be satisfied even when it is syntactically possible to tell which specific kind of aggregate is being used. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.c/5</MarginText>
<MarginInfo items={["AI12-0005-1", "AI12-0127-1", "AI12-0212-1", "AI12-0307-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>Generally, we don't want to look in <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>s to resolve them. For instance, Ada 95 allowed array types to match <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code>s, even though those have to be record types. Thus, we allow any record type with any visible (nondiscriminant) components to match an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, even though only <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code>s allow private types or private extensions. Similarly, we allow any container type to match an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, even though only <code><a href="../AA-4/AA-4.3#S0123">container_aggregate</a></code>s allow container types. <br /></Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>4/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<p>A <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code> or <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code>  shall not be of a class-wide type. <br /></p>

<AnnotatedOnly>
<MarginText>4.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>When the expected type in some context is class-wide, an aggregate has to be explicitly qualified by the specific type of value to be created, so that the expected type for the aggregate itself is specific. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.b</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>We used to disallow <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>s of a type with unknown discriminants. However, that was unnecessarily restrictive in the case of an extension aggregate, and irrelevant to a record aggregate (since a type that is legal for a record aggregate could not possibly have unknown discriminants) and to an array aggregate (the only specific types that can have unknown discriminants are private types, private extensions, and types derived from them). <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.c/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>We don't mention <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code>s, as those can get the specific type from the object represented by the <em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> (possibly at run time). We don't mention <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code>s, as those cannot even be of a tagged type, so being class-wide is impossible. <br /></Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>5</MarginText>
<p>For the evaluation of an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, an anonymous object is created and values for the components or ancestor part are obtained (as described in the subsequent subclause for each kind of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>) and assigned into the corresponding components or ancestor part of the anonymous object. Obtaining the values and the assignments occur in an arbitrary order. The value of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> is the value of this object. <br /></p>

<AnnotatedOnly>
<MarginText>5.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The ancestor part is the set of components inherited from the ancestor type. The syntactic category <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> is the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> or <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code> that specifies how the ancestor part of the anonymous object should be initialized. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The assignment operations do the necessary value adjustment, as described in <a href="../AA-7/AA-7.6">7.6</a>. Note that the value as a whole is not adjusted &mdash; just the subcomponents (and ancestor part, if any). <a href="../AA-7/AA-7.6">7.6</a> also describes when this anonymous object is finalized.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.c</MarginText>
<Admonition type="aarm" aarm="note">
If the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> is a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code> the Initialize procedure for the ancestor type is applied to the ancestor part after default-initializing it, unless the procedure is abstract, as described in <a href="../AA-7/AA-7.6">7.6</a>. The Adjust procedure for the ancestor type is not called in this case, since there is no assignment to the ancestor part as a whole. <br /></Admonition>
</AnnotatedOnly>

<MarginText>6</MarginText>
<p>If an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> is of a tagged type, a check is made that its value belongs to the first subtype of the type. Constraint_Error is raised if this check fails. <br /></p>

<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This check ensures that no values of a tagged type are ever outside the first subtype, as required for inherited dispatching operations to work properly (see <a href="../AA-3/AA-3.4">3.4</a>). This check will always succeed if the first subtype is unconstrained. This check is not extended to untagged types to preserve upward compatibility. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>

#### Incompatibilities With Ada 95

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>6.e/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />
<Admonition type="aarm" aarm="note">
In Ada 95, a limited type is not considered when resolving an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>. Since Ada 2005 now allows limited <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>s, we can have incompatibilities. For example: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.f/2</MarginText>
<CodeBlock language="ada">
type Lim is limited{"\n"}   record{"\n"}      Comp: Integer;{"\n"}   end record;{"\n"}
<MarginText>6.g/2</MarginText>
type Not_Lim is{"\n"}   record{"\n"}      Comp: Integer;{"\n"}   end record;{"\n"}
<MarginText>6.h/2</MarginText>
procedure P(X: Lim);{"\n"}procedure P(X: Not_Lim);{"\n"}
<MarginText>6.i/2</MarginText>
P((Comp ={">"} 123)); -- Illegal in Ada 2005, legal in Ada 95{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>6.j/2</MarginText>
<Admonition type="aarm" aarm="note">
The call to P is ambiguous in Ada 2005, while it would not be ambiguous in Ada 95 as the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> could not have a limited type. Qualifying the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> will eliminate any ambiguity. This construction would be rather confusing to a maintenance programmer, so it should be avoided, and thus we expect it to be rare. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>

#### Extensions to Ada 95

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>6.k/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />
<Admonition type="aarm" aarm="note">
<code><a href="../AA-4/AA-4.3#S0106">Aggregate</a></code>s can be of a limited type. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>

#### Incompatibilities With Ada 2012

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>6.l/5</MarginText>
<MarginInfo items={["AI12-0005-1", "AI12-0127-1", "AI12-0212-1"]} />
<Admonition type="aarm" aarm="note">
We now allow types with the Aggregate aspect specified ("container types"), as well as private types and extensions descended from a record type or extension, to match all forms of <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>. These types are only allowed for new types of <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> (<code><a href="../AA-4/AA-4.3#S0123">container_aggregate</a></code>s for the Aggregate aspect, and <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code>s for private types), but, consistent with other forms of <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, we do not look at the form of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> to determine resolution. This can be incompatible in usually unlikely cases, where overloading of a container or private type with a type that was previously allowed in <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>s makes an existing call ambiguous. Unfortunately, Ada.Containers.Vectors has a number of such overloadings for Insert, Append, Prepend, and "&", so the problem may appear for any element type of a Vector that allows aggregates. For instance, if My_Vector is an instance of Ada.Containers.Vectors with an element type of Not_Lim as defined above, and V is an object of My_Vector.Vector, then My_Vector.Append (V, (Comp ={">"} 123)); will be illegal in Ada 2022 and legal in Ada 2012. This can easily be fixed by qualifying the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> with the correct type. <br /></Admonition>
</AnnotatedOnly>

<a id="Subclause_4.3.1"></a>

## 4.3.1  Record Aggregates

<MarginText>1</MarginText>
<p>[In a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code>, a value is specified for each component of the record or record extension value, using either a named or a positional association.] <br /></p>


#### Syntax

<MarginText>2</MarginText>

<CodeBlock>
<code>record_aggregate</code><a id="S0107"></a><code> ::= </code>(<code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code>)<br />

</CodeBlock>
<MarginText>3</MarginText>

<CodeBlock>
<code>record_component_association_list</code><a id="S0108"></a><code> ::= </code><br />    <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> {"{"}, <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code>{"}"}<br />  | <strong>null record</strong><br />

</CodeBlock>
<MarginText>4/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />

<CodeBlock>
<code>record_component_association</code><a id="S0109"></a><code> ::= </code><br />    [<code><a href="../AA-4/AA-4.3#S0110">component_choice_list</a></code> ={">"}] <code><a href="../AA-4/AA-4.4#S0132">expression</a></code><br />   | <code><a href="../AA-4/AA-4.3#S0110">component_choice_list</a></code> ={">"} {"<"}{">"}<br />

</CodeBlock>
<MarginText>5/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />

<CodeBlock>
<code>component_choice_list</code><a id="S0110"></a><code> ::= </code><br />     <em>component_</em><code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code> {"{"}'|'  <em>component_</em><code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code>{"}"}<br />   | <strong>others</strong><br />

</CodeBlock>
<MarginText>6</MarginText>
<p>A <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> is a <em>named component association</em> if it has a <code><a href="../AA-4/AA-4.3#S0110">component_choice_list</a></code>; otherwise, it is a <em>positional component association</em>. Any positional component associations shall precede any named component associations. If there is a named association with a <code><a href="../AA-4/AA-4.3#S0110">component_choice_list</a></code> of <strong>others</strong>, it shall come last. <br /></p>

<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>These rules were implied by the BNF in an early version of the RM9X, but it made the grammar harder to read, and was inconsistent with how we handle discriminant constraints. Note that for array aggregates we still express some of the rules in the grammar, but array aggregates are significantly different because an array aggregate is either all positional (with a possible <strong>others</strong> at the end), or all named. <br /></Admonition>
</AnnotatedOnly>

<MarginText>7</MarginText>
<p>In the <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code> for a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code>, if there is only one association, it shall be a named association. <br /></p>

<AnnotatedOnly>
<MarginText>7.a/3</MarginText>
<MarginInfo items={["AI05-0264-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>Otherwise, the construct would be interpreted as a parenthesized expression. This is considered a syntax rule, since it is relevant to overload resolution. We choose not to express it with BNF so we can share the definition of <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code> in both <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code> and <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code>. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code> of an <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code> does not have such a restriction. <br /></Admonition>
</AnnotatedOnly>


#### Name Resolution Rules

<MarginText>8/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />
<p>The expected type for a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code> shall be a single record type or record extension. <br /></p>

<AnnotatedOnly>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This rule is used to resolve whether an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> is an <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> or a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code>. The presence of a <strong>with</strong> is used to resolve between a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code> and an <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>9/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<p>For the <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code> of a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code>, all components of the composite value defined by the aggregate are <em>needed</em>[; for the association list of an <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code>, only those components not determined by the ancestor expression or subtype are needed (see <a href="../AA-4/AA-4.3#Subclause_4.3.2">4.3.2</a>).] Each <em>component_</em><code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code>  in a <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> of a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code> or <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code> shall denote a needed component [(including possibly a discriminant)]. Each <em>component_</em><code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code> in a <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> of a <code><a href="../AA-4/AA-4.3#S0121">record_delta_aggregate</a></code> (see <a href="../AA-4/AA-4.3#Subclause_4.3.4">4.3.4</a>) shall denote a nondiscriminant component of the type of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>.<br /></p>

<AnnotatedOnly>
<MarginText>9.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>For the association list of a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code>, &ldquo;needed components&rdquo; includes every component of the composite value, but does not include those in unchosen <code><a href="../AA-3/AA-3.8#S0072">variant</a></code>s (see AI83-309). If there are <code><a href="../AA-3/AA-3.8#S0072">variant</a></code>s, then the value specified for the discriminant that governs them determines which <code><a href="../AA-3/AA-3.8#S0072">variant</a></code> is chosen, and hence which components are needed.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.b</MarginText>
<Admonition type="aarm" aarm="note">
If an extension defines a new <code><a href="../AA-3/AA-3.7#S0061">known_discriminant_part</a></code>, then all of its discriminants are needed in the component association list of an extension aggregate for that type, even if the discriminants have the same names and types as discriminants of the type of the ancestor expression. This is necessary to ensure that the positions in the <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code> are well defined, and that discriminants that govern <code><a href="../AA-3/AA-3.8#S0071">variant_part</a></code>s can be given by static expressions. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.c/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We don't define &ldquo;needed&rdquo; for <code><a href="../AA-4/AA-4.3#S0121">record_delta_aggregate</a></code>s so that there is no completeness requirement. But that means that we need to ensure that the rules using &ldquo;needed&rdquo; doesn't appear to apply to <code><a href="../AA-4/AA-4.3#S0121">record_delta_aggregate</a></code>s, and we also need Legality Rules to prevent giving the same component twice and giving components from two different variants. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.d/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>needed component</strong> &mdash; component of a record type or record extension that is required to have its value specified within a given aggregate<br /></Admonition>
</AnnotatedOnly>

<MarginText>10</MarginText>
<p>The expected type for the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> is the type of the <em>associated</em> component(s); the associated component(s) are as follows: <br /></p>

<MarginText>11</MarginText>
<ul>
<li>For a positional association, the component [(including possibly a discriminant)] in the corresponding relative position (in the declarative region of the type), counting only the needed components; <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>11.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This means that for an association list of an <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code>, only noninherited components are counted to determine the position.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.b/3</MarginText>
<MarginInfo items={["AI05-0005-1"]} />
<Admonition type="aarm" aarm="note">
For a derived type (including type extensions), the order of declaration is defined in <a href="../AA-3/AA-3.4">3.4</a>, &ldquo;<a href="../AA-3/AA-3.4">Derived Types and Classes</a>&rdquo;. In particular, all discriminants come first, regardless of whether they are defined for the parent type or are newly added to the derived type. <br /></Admonition>
</AnnotatedOnly>

<MarginText>12</MarginText>
<ul>

<MarginText>13</MarginText>

#### Legality Rules

<MarginText>14</MarginText>
<p>If the type of a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code> is a record extension, then it shall be a descendant of a record type, through one or more record extensions (and no private extensions).<br /></p>

<MarginText>15/5</MarginText>
<MarginInfo items={["AI05-0016-1", "AI12-0127-1"]} />
<p>A <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code> shall be  <strong>null record</strong> only if the list occurs in a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code> or <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code>, and  there are no components needed for that list .<br /></p>

<AnnotatedOnly>
<MarginText>15.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>For example, "(<strong>null record</strong>)" is a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code> for a null record type. Similarly, "(T'(A) <strong>with null record</strong>)" is an <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code> for a type defined as a null record extension of T.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>15.b/3</MarginText>
<MarginInfo items={["AI05-0016-1"]} />
<Admonition type="aarm" aarm="note">
If no components are needed and <strong>null record</strong> is not used, the <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> must necessarily be <strong>others</strong> ={">"} {"<"}{">"}, as that is the only <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> that does not require an associated component. <br /></Admonition>
</AnnotatedOnly>

<MarginText>16/5</MarginText>
<MarginInfo items={["AI95-00287-01", "AI05-0199-1", "AI12-0046-1", "AI12-0127-1"]} />
<p>For a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code> or <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code>, each  <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> other than an <strong>others</strong> choice with a {"<"}{">"} shall have at least one associated component, and each needed component shall be associated with exactly one <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code>. For a <code><a href="../AA-4/AA-4.3#S0121">record_delta_aggregate</a></code>, each <em>component_</em><code><a href="../AA-4/AA-4.1#S0099">selector_name</a></code> of each <code><a href="../AA-4/AA-4.3#S0110">component_choice_list</a></code> shall denote a distinct nondiscriminant component of the type of the aggregate. <br /></p>

<AnnotatedOnly>
<MarginText>16.a/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>These rules apply to an association with an <strong>others</strong> choice with an expression. An <strong>others</strong> choice with a {"<"}{">"} can match zero components or several components with different types. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16.b/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>Without these rules, there would be no way to know what was the expected type for the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the association. Note that some of the rules do not apply to {"<"}{">"} associations, as we do not need to resolve anything. We allow <strong>others</strong> ={">"} {"<"}{">"} to match no components as this is similar to array aggregates. That means that (<strong>others</strong> ={">"} {"<"}{">"}) always represents a default-initialized record or array value. <br /></Admonition>
</AnnotatedOnly>

<MarginText>16.c/4</MarginText>
<MarginInfo items={["AI12-0046-1"]} />
<Admonition type="aarm" aarm="note">
<em>This paragraph was deleted.</em><br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16.d</MarginText>

#### Dynamic Semantics

<MarginText>18</MarginText>
<p>The evaluation of a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code> consists of the evaluation of the <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code>.<br /></p>

<MarginText>19/5</MarginText>
<MarginInfo items={["AI12-0086-1", "AI12-0451-1"]} />
<p>For the evaluation of a <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code>, any per-object constraints (see <a href="../AA-3/AA-3.8">3.8</a>) for components specified in the association list are elaborated and any <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s are evaluated and converted to the subtype of the associated component. Any constraint elaborations and <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> evaluations (and conversions) occur in an arbitrary order, except that the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> for a discriminant is evaluated (and converted) prior to the elaboration of any per-object constraint that depends on it, which in turn occurs prior to the evaluation and conversion of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> for the component with the per-object constraint. If the value of a discriminant that governs a selected <code><a href="../AA-3/AA-3.8#S0071">variant_part</a></code> is given by a nonstatic <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, and the evaluation of that <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> yields a value that does not belong to the nominal subtype of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, then Constraint_Error is raised.<br /></p>

<AnnotatedOnly>
<MarginText>19.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The conversion in the first rule might raise Constraint_Error. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>19.b</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This check in the first rule presumably happened as part of the dependent compatibility check in Ada 83. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>19.c/5</MarginText>
<MarginInfo items={["AI12-0086-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>The test on the value of the discriminant can only fail if the value is outside the base range of its type, does not satisfy the (static) predicates of the subtype (possible when the predicate is disabled), or is an invalid representation. This is a rule similar to that used for case statements. As with case statements, this is not a &ldquo;check&rdquo;; it cannot be Suppressed. <br /></Admonition>
</AnnotatedOnly>

<MarginText>19.1/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />
<p>For a <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> with an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> defines the value for the associated component(s). For a <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> with {"<"}{">"}, if the <code><a href="../AA-3/AA-3.8#S0070">component_declaration</a></code> has a <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code>, that <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code> defines the value for the associated component(s); otherwise, the associated component(s) are initialized by default as for a stand-alone object of the component subtype (see <a href="../AA-3/AA-3.3#Subclause_3.3.1">3.3.1</a>).<br /></p>

<MarginText>20</MarginText>
<p>The <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> is evaluated (and converted) once for each associated component.<br /></p>

<AnnotatedOnly>
<MarginText>20.a/3</MarginText>
<MarginInfo items={["AI05-0005-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>We don't need similar language for {"<"}{">"}, as we're considering the value of {"<"}{">"} for each individual component. Each component has its own default expression or its own default initialization (they can be different for each component; the components even could have different types), and each one has to be evaluated. So there is no need to repeat that. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>21/5</MarginText>
<MarginInfo items={["AI12-0447-1"]} />
<Admonition type="aarm" aarm="note">
NOTE   By the rules given above, for  a <code><a href="../AA-4/AA-4.3#S0107">record_aggregate</a></code> with positional associations, expressions specifying discriminant values appear first and in the same order as their corresponding <code><a href="../AA-3/AA-3.7#S0062">discriminant_specification</a></code>s, since the <code><a href="../AA-3/AA-3.7#S0061">known_discriminant_part</a></code> occurs  first in the declaration of the type . <br /></Admonition>
</AnnotatedOnly>

## 4.3.2  Extension Aggregates

<MarginText>1</MarginText>
<p>[An <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code> specifies a value for a type that is a record extension by specifying a value or subtype for an ancestor of the type, followed by associations for any components not determined by the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code>.] <br /></p>


#### Language Design Principles

<AnnotatedOnly>
<MarginText>1.a</MarginText>
<Admonition type="aarm" aarm="note">
The model underlying this syntax is that a record extension can also be viewed as a regular record type with an ancestor "prefix". The <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code> corresponds to exactly what would be needed if there were no ancestor/prefix type. The <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> determines the value of the ancestor/prefix. <br /></Admonition>
</AnnotatedOnly>


#### Syntax

<MarginText>2</MarginText>

<CodeBlock>
<code>extension_aggregate</code><a id="S0111"></a><code> ::= </code><br />    (<code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> <strong>with</strong> <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code>)<br />

</CodeBlock>
<MarginText>3</MarginText>

<CodeBlock>
<code>ancestor_part</code><a id="S0112"></a><code> ::= </code><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> | <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code><br />

</CodeBlock>

#### Name Resolution Rules

<MarginText>4/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />
<p>The expected type for an <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code> shall be a single type that is a record extension. If the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> is an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, it is expected to be of any tagged type. <br /></p>

<AnnotatedOnly>
<MarginText>4.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We could have made the expected type <em>T'</em>Class where <em>T</em> is the ultimate ancestor of the type of the aggregate, or we could have made it even more specific than that. However, if the overload resolution rules get too complicated, the implementation gets more difficult and it becomes harder to produce good error messages. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.b/3</MarginText>
<MarginInfo items={["AI05-0005-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>This rule is additive with the rule given in <a href="../AA-4/AA-4.3">4.3</a>. That means the <a href="../AA-4/AA-4.3">4.3</a> rule must be satisfied even though it is always syntactically possible to tell that something is an extension aggregate rather than another kind of aggregate. Specifically, that means that an extension aggregate is ambiguous if the context is overloaded on array and/or untagged record types, even though those are never legal contexts for an extension aggregate. Thus, this rule acts more like a Legality Rule than a Name Resolution Rule. <br /></Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>5/3</MarginText>
<MarginInfo items={["AI95-00306-01", "AI05-0115-1"]} />
<p>If the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> is a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>, it shall denote a specific tagged subtype. If the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> is an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, it shall not be dynamically tagged. The type of the <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code> shall be a descendant of the type of the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> (the <em>ancestor</em> type), through one or more record extensions (and no private extensions). If the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> is a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>, the view of the ancestor type from which the type is descended (see <a href="../AA-7/AA-7.3#Subclause_7.3.1">7.3.1</a>) shall not have unknown discriminants. <br /></p>

<AnnotatedOnly>
<MarginText>5.a/2</MarginText>
<MarginInfo items={["AI95-00306-01"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>The expression cannot be dynamically tagged to prevent implicit "truncation" of a dynamically-tagged value to the specific ancestor type. This is similar to the rules in <a href="../AA-3/AA-3.9#Subclause_3.9.2">3.9.2</a>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>5.1/5</MarginText>
<MarginInfo items={["AI05-0067-1", "AI05-0244-1", "AI12-0236-1", "AI12-0317-1"]} />
<p>If the type of the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> is limited and at least one component is needed in the <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code>, then the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> shall not have an operative constituent expression (see <a href="../AA-4/AA-4.4">4.4</a>) that is a call to a function with an unconstrained result subtype.  <br /></p>

<MarginText>5.2/5</MarginText>
<MarginInfo items={["AI12-0317-1"]} />
<ul>
<li> <br /></li>
<MarginText>5.3/5</MarginText>
<MarginInfo items={["AI12-0317-1"]} />
<li> <br /></li>
<MarginText>5.4/5</MarginText>
<MarginInfo items={["AI12-0317-1"]} />
<li> <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>5.b/3</MarginText>
<MarginInfo items={["AI05-0067-1", "AI05-0244-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>This restriction simplifies implementation, because it ensures that either the caller or the callee knows the size to allocate for the aggregate. Without this restriction, information from both caller and callee would have to be combined to determine the appropriate size.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.c/3</MarginText>
<MarginInfo items={["AI05-0067-1"]} />
<Admonition type="aarm" aarm="note">
The (F(...) with null record) case is exempt from this rule, because such extension aggregates are created internally for inherited functions returning null-extension types &mdash; we can't very well make those illegal. Moreover, we don't need the rule for null extensions, as the result can simply use the space returned by the function call.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.d/5</MarginText>
<MarginInfo items={["AI12-0317-1"]} />
<Admonition type="aarm" aarm="note">
The mention of &ldquo;operative constituents&rdquo; means that constructs like parenthesized expressions, <code><a href="../AA-4/AA-4.7#S0163">qualified_expression</a></code>s, and <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>s are ignored when enforcing this rule. <br /></Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>6</MarginText>
<p>For the <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code> of an <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code>, the only components <em>needed</em> are those of the composite value defined by the aggregate that are not inherited from the type of the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code>, plus any inherited discriminants if the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> is a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code> that denotes an unconstrained subtype. <br /></p>


#### Dynamic Semantics

<MarginText>7</MarginText>
<p>For the evaluation of an <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code>, the <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code> is evaluated. If the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> is an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, it is also evaluated; if the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> is a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>, the components of the value of the aggregate not given by the <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code> are initialized by default as for an object of the ancestor type. Any implicit initializations or evaluations are performed in an arbitrary order, except that the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> for a discriminant is evaluated prior to any other evaluation or initialization that depends on it.<br /></p>

<MarginText>8/3</MarginText>
<MarginInfo items={["AI05-0282-1"]} />
<p>If the type of the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> has discriminants and the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> is not a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code> that denotes an unconstrained subtype, then a check is made that each discriminant determined by the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> has the value specified for a corresponding discriminant, if any, either in the <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code>, or in the <code><a href="../AA-3/AA-3.4#S0035">derived_type_definition</a></code> for some ancestor of the type of the <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code>. Constraint_Error is raised if this check fails. <br /></p>

<AnnotatedOnly>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Corresponding and specified discriminants are defined in <a href="../AA-3/AA-3.7">3.7</a>. The rules requiring static compatibility between new discriminants of a derived type and the parent discriminant(s) they constrain ensure that at most one check is required per discriminant of the ancestor expression.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.b/3</MarginText>
<MarginInfo items={["AI05-0282-1"]} />
<Admonition type="aarm" aarm="note">
The check needs to be made any time that the ancestor is constrained; the source of the discriminants or the constraints is irrelevant. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9/5</MarginText>
<MarginInfo items={["AI12-0447-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 1   By the rules given in <a href="../AA-4/AA-4.3#Subclause_4.3.1">4.3.1</a>, if  all components of the value of the <code><a href="../AA-4/AA-4.3#S0111">extension_aggregate</a></code> are determined by the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code>, then the <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code> will  be simply <strong>null record</strong>.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 2   If the <code><a href="../AA-4/AA-4.3#S0112">ancestor_part</a></code> is a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>, then its type can be abstract. If its type is controlled, then as the last step of evaluating the aggregate, the Initialize procedure of the ancestor type is called, unless the Initialize procedure is abstract (see <a href="../AA-7/AA-7.6">7.6</a>). <br /></Admonition>
</AnnotatedOnly>

## 4.3.3  Array Aggregates

<p>An <em>n-dimensional</em> <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> is one that is written as n levels of nested <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code>s (or at the bottom level, equivalent <code><a href="../AA-2/AA-2.6#S0016">string_literal</a></code>s). For the multidimensional case (n {">"}= 2) the <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code>s (or equivalent <code><a href="../AA-2/AA-2.6#S0016">string_literal</a></code>s) at the n&ndash;1 lower levels are called <em>subaggregate</em>s of the enclosing n-dimensional <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code>. The <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s of the bottom level subaggregates (or of the <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> itself if one-dimensional) are called the <em>array component expressions</em> of the enclosing n-dimensional <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Subaggregates do not have a type. They correspond to part of an array. For example, with a matrix, a subaggregate would correspond to a single row of the matrix. The definition of "n-dimensional" <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> applies to subaggregates as well as <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>s that have a type. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.b</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>An <em><strong>others</strong> choice</em> is the reserved word <strong>others</strong> as it appears in a <code><a href="../AA-4/AA-4.3#S0114">positional_array_aggregate</a></code> or as the <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> of the <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code> in an <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>6.1/5</MarginText>
<MarginInfo items={["AI12-0061-1"]} />
<p>The <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code> of an <code><a href="../AA-4/AA-4.3#S0119">iterated_component_association</a></code> declares an <em>index parameter</em>, an object of the corresponding index type.<br /></p>


#### Name Resolution Rules

<MarginText>7/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />
<p>The expected type for an <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> (that is not a subaggregate) shall be a single array type. The component type of this array type is the expected type for each array component expression of the <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>7.a/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>We already require a single array or record type or record extension for an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>. The above rule requiring a single array type (and similar ones for record and extension aggregates) resolves which kind of aggregate you have. <br /></Admonition>
</AnnotatedOnly>

<MarginText>8</MarginText>
<p>The expected type for each <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> in any <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code> of a <code><a href="../AA-4/AA-4.3#S0116">named_array_aggregate</a></code> is the type of the <em>corresponding index</em>; the corresponding index for an <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> that is not a subaggregate is the first index of its type; for an (n&ndash;m)-dimensional subaggregate within an <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> of an n-dimensional type, the corresponding index is the index in position m+1. <br /></p>


#### Legality Rules

<MarginText>9/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>An <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> of an n-dimensional array type shall be written as an n-dimensional <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code>, or as a <code><a href="../AA-4/AA-4.3#S0115">null_array_aggregate</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>9.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>In an m-dimensional <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> [(including a subaggregate)], where m {">"}= 2, each of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s has to be an (m&ndash;1)-dimensional subaggregate. <br /></Admonition>
</AnnotatedOnly>

<MarginText>10/5</MarginText>
<MarginInfo items={["AI12-0418-1"]} />
<p>An <strong>others</strong> choice is allowed for an <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> only if an <em>applicable index constraint</em> applies to the <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code>. <MarginText>11/4</MarginText>
<MarginInfo items={["AI95-00318-02", "AI12-0157-1"]} />
<ul>
<li>For an <code><a href="../AA-6/AA-6.4#S0221">explicit_actual_parameter</a></code>, an <code><a href="../AA-12/AA-12.3#S0318">explicit_generic_actual_parameter</a></code>, the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a return statement, the return expression of an expression function, the initialization expression in an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code>, or a <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code> [(for a parameter or a component)], when the nominal subtype of the corresponding formal parameter, generic formal parameter, function return object, expression function return object, object, or component is a constrained array subtype, the applicable index constraint is the constraint of the subtype;<br /></li>
<MarginText>12</MarginText>

<MarginText>12.a</MarginText>
<Admonition type="aarm" aarm="reason">

<MarginText>13</MarginText>
<ul>
<li>For the operand of a <code><a href="../AA-4/AA-4.7#S0163">qualified_expression</a></code> whose <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code> denotes a constrained array subtype, the applicable index constraint is the constraint of the subtype;<br /></li>
<MarginText>14</MarginText>
<li>For a component <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> in an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, if the component's nominal subtype is a constrained array subtype, the applicable index constraint is the constraint of the subtype; <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>14.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Here, the <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> with <strong>others</strong> is being used within a larger aggregate. <br /></Admonition>
</AnnotatedOnly>

<MarginText>14.1/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<ul>
<li>For the <em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code>, if the nominal subtype of the <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code> is a constrained array subtype, the applicable index constraint is the constraint of the subtype;<br /></li>
<MarginText>15/3</MarginText>
<MarginInfo items={["AI05-0147-1"]} />

<MarginText>15.1/5</MarginText>
<MarginInfo items={["AI05-0147-1", "AI12-0236-1"]} />
<ul>
<li>For a <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> (see <a href="../AA-4/AA-4.5#Subclause_4.5.7">4.5.7</a>), the applicable index constraint for each <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is that, if any, defined for the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>; <br /></li>
<MarginText>15.2/5</MarginText>
<MarginInfo items={["AI12-0236-1"]} />

<MarginText>17/5</MarginText>
<MarginInfo items={["AI05-0153-3", "AI12-0061-1", "AI12-0127-1"]} />
<p>The <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code> of an <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code> (including an <code><a href="../AA-4/AA-4.3#S0119">iterated_component_association</a></code>) is allowed to have a <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> that is a nonstatic <code><a href="../AA-4/AA-4.4#S0133">choice_expression</a></code> or that is a <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code> or <code><a href="../AA-3/AA-3.5#S0037">range</a></code> that defines a nonstatic or null range, only if it is the single <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> of its <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code>, and either there is only one <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code> in the enclosing <code><a href="../AA-4/AA-4.3#S0117">array_component_association_list</a></code> or the enclosing <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> is an <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code>[, not an <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code>] .<br /></p>

<AnnotatedOnly>
<MarginText>17.a</MarginText>

<AnnotatedOnly>
<MarginText>17.b/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />

<strong></strong>We allow multiple dynamic choices in <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code>s, but only one dynamic choice per association even in that case. <br /></Admonition>
</AnnotatedOnly>

<MarginText>17.1/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>Either all or none of the <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code>s of an <code><a href="../AA-4/AA-4.3#S0117">array_component_association_list</a></code> shall be <code><a href="../AA-4/AA-4.3#S0119">iterated_component_association</a></code>s with an <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code>.<br /></p>

<MarginText>18/3</MarginText>
<MarginInfo items={["AI05-0262-1"]} />
<p>In a <code><a href="../AA-4/AA-4.3#S0116">named_array_aggregate</a></code> where all <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>s are static, no two <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>s are allowed to cover the same value (see <a href="../AA-3/AA-3.8#Subclause_3.8.1">3.8.1</a>); if there is no <strong>others</strong> choice, the <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>s taken together shall exactly cover a contiguous sequence of values of the corresponding index type. <br /></p>

<AnnotatedOnly>
<MarginText>18.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This implies that each component must be specified exactly once. See AI83-309. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18.b/3</MarginText>
<MarginInfo items={["AI05-0262-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>This has to apply even if there is only one static <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>; a single choice has to represent a contiguous range (a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code> with a static predicate might represent a discontiguous set of values). If the (single) choice is a dynamic subtype, we don't need to make this check as no predicates are allowed (see <a href="../AA-3/AA-3.2#Subclause_3.2.4">3.2.4</a>) and thus the range has to be contiguous. <br /></Admonition>
</AnnotatedOnly>

#### Static Semantics

<MarginText>20</MarginText>
<p>A subaggregate that is a <code><a href="../AA-2/AA-2.6#S0016">string_literal</a></code> is equivalent to one that is a <code><a href="../AA-4/AA-4.3#S0114">positional_array_aggregate</a></code> of the same length, with each <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> being the <code><a href="../AA-2/AA-2.5#S0015">character_literal</a></code> for the corresponding character of the <code><a href="../AA-2/AA-2.6#S0016">string_literal</a></code>.<br /></p>

<AnnotatedOnly>
<MarginText>20.a/5</MarginText>
<MarginInfo items={["AI12-0306-1"]} />
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>This is true even in cases where there is no corresponding legal <code><a href="../AA-4/AA-4.3#S0114">positional_array_aggregate</a></code>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>20.1/5</MarginText>
<MarginInfo items={["AI12-0061-1"]} />
<p>The subtype (and nominal subtype) of an index parameter is the corresponding index subtype. <br /></p>


#### Dynamic Semantics

<MarginText>20.2/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>For an <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> that contains only <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code>s that are <code><a href="../AA-4/AA-4.3#S0119">iterated_component_association</a></code>s with <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code>s, evaluation proceeds in two steps: <br /></p>

<MarginText>20.3/5</MarginText>
<dt><br/>1.</dt>
<MarginInfo items={["AI12-0212-1", "AI12-0250-1"]} />
<dl>
<dd>Each <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code> is elaborated (in an arbitrary order) and an iteration is performed solely to determine a maximum count for the number of values produced by the iteration; all of these counts are combined to determine the overall length of the array, and ultimately the limits on the bounds of the array (defined below);<br /></dd>
<MarginText>20.4/5</MarginText>
<dt><br/>2.</dt>
<MarginInfo items={["AI12-0212-1", "AI12-0250-1", "AI12-0327-1"]} />
<dd>A second iteration is performed for each of the <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code>s, in the order given in the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, and for each value conditionally produced by the iteration (see <a href="../AA-5/AA-5.5">5.5</a> and <a href="../AA-5/AA-5.5#Subclause_5.5.2">5.5.2</a>), the associated <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is evaluated, its value is converted to the component subtype of the array type, and used to define the value of the next component of the array starting at the low bound and proceeding sequentially toward the high bound. A check is made that the second iteration results in an array length no greater than the maximum determined by the first iteration; Constraint_Error is raised if this check fails.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>20.b/5</MarginText>
<MarginInfo items={["AI12-0212-1", "AI12-0250-1"]} />
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>Constraint_Error should be raised no later than when the iterations exceed the maximum array length; memory that doesn't belong to the aggregate temporary should not be overwritten. <br /></Admonition>
</AnnotatedOnly>

<MarginText>21/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>The evaluation of any other  <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> of a given array type proceeds in two steps: <br /></p>

<MarginText>22</MarginText>
<dt><br/>1.</dt>
<dl>
<dd>Any <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>s of this aggregate and of its subaggregates are evaluated in an arbitrary order, and converted to the corresponding index type; <br /></dd>
<MarginText>23</MarginText>
<dt><br/>2.</dt>
<dd>The array component expressions of the aggregate are evaluated in an arbitrary order and their values are converted to the component subtype of the array type; an array component expression is evaluated once for each associated component. <br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>23.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Subaggregates are not separately evaluated. The conversion of the value of the component expressions to the component subtype might raise Constraint_Error.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>23.b/3</MarginText>
<MarginInfo items={["AI05-0005-1"]} />
<Admonition type="aarm" aarm="note">
We don't need to say that {"<"}{">"} is evaluated once for each component, as {"<"}{">"} means that each component is <em>initialized by default</em>. That means that the actions defined for default initialization are applied to each component individually. Initializing one component by default and copying that to the others would be an incorrect implementation in general (although it might be OK if the default initialization is known to be constant). <br /></Admonition>
</AnnotatedOnly>

<MarginText>23.1/4</MarginText>
<MarginInfo items={["AI95-00287-01", "AI12-0084-1"]} />
<p>Each <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> in an <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code> defines the value for the associated component(s). For an <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code> with {"<"}{">"}, the associated component(s) are initialized to the Default_Component_Value of the array type if this aspect has been specified for the array type; otherwise, they are initialized by default as for a stand-alone object of the component subtype (see <a href="../AA-3/AA-3.3#Subclause_3.3.1">3.3.1</a>).<br /></p>

<MarginText>23.2/5</MarginText>
<MarginInfo items={["AI12-0061-1", "AI12-0212-1"]} />
<p>During an evaluation of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of an <code><a href="../AA-4/AA-4.3#S0119">iterated_component_association</a></code> with a <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code>, the value of the corresponding index parameter is that of the corresponding index of the corresponding array component. During an evaluation of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of an <code><a href="../AA-4/AA-4.3#S0119">iterated_component_association</a></code> with an <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code>, the value of the loop parameter of the <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code> is the value produced by the iteration (as described in <a href="../AA-5/AA-5.5#Subclause_5.5.2">5.5.2</a>).<br /></p>

<AnnotatedOnly>
<MarginText>23.c/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Taken together with the preceding rule that &ldquo;The array component expressions of the aggregate are evaluated in an arbitrary order&rdquo;, this implies that an index parameter of an <code><a href="../AA-4/AA-4.3#S0119">iterated_component_association</a></code> can take on its values in an arbitrary order. This is different from other constructs, such as a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code>. In contrast, a loop parameter of an <code><a href="../AA-4/AA-4.3#S0119">iterated_component_association</a></code> takes its values in the order defined by the iteration, the same as a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>24</MarginText>
<p>The bounds of the index range of an <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> [(including a subaggregate)] are determined as follows: <br /></p>

<MarginText>25</MarginText>
<ul>
<li>For an <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> with an <strong>others</strong> choice, the bounds are those of the corresponding index range from the applicable index constraint;<br /></li>
<MarginText>26</MarginText>
<li>For a <code><a href="../AA-4/AA-4.3#S0114">positional_array_aggregate</a></code> [(or equivalent <code><a href="../AA-2/AA-2.6#S0016">string_literal</a></code>)] without an <strong>others</strong> choice, the lower bound is that of the corresponding index range in the applicable index constraint, if defined, or that of the corresponding index subtype, if not; in either case, the upper bound is determined from the lower bound and the number of <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s [(or the length of the <code><a href="../AA-2/AA-2.6#S0016">string_literal</a></code>)];<br /></li>
<MarginText>26.1/5</MarginText>
<MarginInfo items={["AI12-0212-1", "AI12-0306-1"]} />
<li>For a <code><a href="../AA-4/AA-4.3#S0115">null_array_aggregate</a></code>, bounds for each dimension are determined as for a <code><a href="../AA-4/AA-4.3#S0114">positional_array_aggregate</a></code> without an <strong>others</strong> choice that has no expressions for each dimension;<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>26.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We need a separate rule to describe what happens for a multidimensional <code><a href="../AA-4/AA-4.3#S0115">null_array_aggregate</a></code>; we could've combined the single-dimension case with the <code><a href="../AA-4/AA-4.3#S0114">positional_array_aggregate</a></code> rule. <br /></Admonition>
</AnnotatedOnly>

<MarginText>26.2/5</MarginText>
<MarginInfo items={["AI12-0212-1", "AI12-0250-1"]} />
<ul>
<li>For a <code><a href="../AA-4/AA-4.3#S0116">named_array_aggregate</a></code> containing only <code><a href="../AA-4/AA-4.3#S0119">iterated_component_association</a></code>s with an <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code>, the lower bound is determined as for a <code><a href="../AA-4/AA-4.3#S0114">positional_array_aggregate</a></code> without an <strong>others</strong> choice, and the upper bound is determined from the lower bound and the total number of values produced by the second set of iterations;<br /></li>
<MarginText>27/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<li>For any other  <code><a href="../AA-4/AA-4.3#S0116">named_array_aggregate</a></code> without an <strong>others</strong> choice, the bounds are determined by the smallest and largest index values covered by any <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code>. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>27.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We don't need to say that each index value has to be covered exactly once, since that is a ramification of the general rule on <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>s that each component's value has to be specified exactly once. <br /></Admonition>
</AnnotatedOnly>

<MarginText>28</MarginText>
<p>For an <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code>, a check is made that the index range defined by its bounds is compatible with the corresponding index subtype. <br /></p>

<AnnotatedOnly>
<MarginText>28.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>In RM83, this was phrased more explicitly, but once we define "compatibility" between a range and a subtype, it seems to make sense to take advantage of that definition. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>28.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The definition of compatibility handles the special case of a null range, which is always compatible with a subtype. See AI83-00313. <br /></Admonition>
</AnnotatedOnly>

<MarginText>29/3</MarginText>
<MarginInfo items={["AI05-0037-1"]} />
<p>For an <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> with an <strong>others</strong> choice, a check is made that no <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> or {"<"}{">"} is specified for an index value outside the bounds determined by the applicable index constraint. <br /></p>

<AnnotatedOnly>
<MarginText>29.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>RM83 omitted this case, apparently through an oversight. AI83-00309 defines this as a dynamic check, even though other Ada 83 rules ensured that this check could be performed statically. We now allow an <strong>others</strong> choice to be dynamic, even if it is not the only choice, so this check now needs to be dynamic, in some cases. Also, within a generic unit, this would be a nonstatic check in some cases. <br /></Admonition>
</AnnotatedOnly>

<MarginText>30</MarginText>
<p>For a multidimensional <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code>, a check is made that all subaggregates that correspond to the same index have the same bounds. <br /></p>

<AnnotatedOnly>
<MarginText>30.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>No array bounds &ldquo;sliding&rdquo; is performed on subaggregates. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>30.b</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>If sliding were performed, it would not be obvious which subaggregate would determine the bounds of the corresponding index. <br /></Admonition>
</AnnotatedOnly>

<MarginText>31</MarginText>
<p>The exception Constraint_Error is raised if any of the above checks fail. <br /></p>

<MarginText>32/5</MarginText>

## 4.3.4  Delta Aggregates

<MarginText>1/5</MarginText>
<MarginInfo items={["AI12-0127-1", "AI12-0324-1"]} />
<p>Evaluating a (record or array) delta aggregate yields a composite value that starts with a copy of another value of the same type and then assigns to some (but typically not all) components of the copy. <br /></p>


#### Syntax

<MarginText>2/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />

<CodeBlock>
<code>delta_aggregate</code><a id="S0120"></a><code> ::= </code><code><a href="../AA-4/AA-4.3#S0121">record_delta_aggregate</a></code> | <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code><br />

</CodeBlock>
<MarginText>3/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />

<CodeBlock>
<code>record_delta_aggregate</code><a id="S0121"></a><code> ::= </code><br />    (<em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> <strong>with</strong> <strong>delta</strong> <code><a href="../AA-4/AA-4.3#S0108">record_component_association_list</a></code>)<br />

</CodeBlock>
<MarginText>4/5</MarginText>
<MarginInfo items={["AI12-0127-1", "AI12-0212-1"]} />

<CodeBlock>
<code>array_delta_aggregate</code><a id="S0122"></a><code> ::= </code><br />    (<em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> <strong>with</strong> <strong>delta</strong> <code><a href="../AA-4/AA-4.3#S0117">array_component_association_list</a></code>)<br />  | '[' <em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> <strong>with</strong> <strong>delta</strong> <code><a href="../AA-4/AA-4.3#S0117">array_component_association_list</a></code> ']'<br />

</CodeBlock>

#### Name Resolution Rules

<MarginText>5/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<p>The expected type for a <code><a href="../AA-4/AA-4.3#S0121">record_delta_aggregate</a></code> shall be a single descendant of a record type or record extension.<br /></p>

<MarginText>6/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<p>The expected type for an <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code> shall be a single array type.<br /></p>

<MarginText>7/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<p>The expected type for the <em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of any <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code> is the type of the enclosing <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code>. <br /></p>

<MarginText>8/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />

<MarginText>9/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<p>For an <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code>, the expected type for each <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> in an <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code> is the index type of the type of the <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code>.<br /></p>

<MarginText>10/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<p>The expected type of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> in an <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code> is defined as for an <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code> occurring within an <code><a href="../AA-4/AA-4.3#S0113">array_aggregate</a></code> of the type of the <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code>.<br /></p>


#### Legality Rules

<MarginText>11/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<p>For an <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code>, the <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code> shall not use the box symbol {"<"}{">"}, and the <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> shall not be <strong>others</strong>.<br /></p>

<MarginText>12/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<p>For an <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code>, the dimensionality of the type of the <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code> shall be 1.<br /></p>

<MarginText>13/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<p>For an <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code>, the <em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> and each <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> in every <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code> shall be of a nonlimited type.<br /></p>

<AnnotatedOnly>
<MarginText>13.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The <em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a <code><a href="../AA-4/AA-4.3#S0121">record_delta_aggregate</a></code> may be of a limited type (for example a record with limited components), as it is not restricted. A rule in <a href="../AA-4/AA-4.3#Subclause_4.3.1">4.3.1</a> ensures that we do not assign to a limited component. We do not allow any part of an <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code> to be of a limited type, even the <em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, as this is a useless construct (you would not be able to update anything because the components necessarily are also limited except in pathological cases). <br /></Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>14/5</MarginText>
<MarginInfo items={["AI12-0127-1", "AI12-0324-1", "AI12-0381-1"]} />
<p>The evaluation of a <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code> begins with the evaluation of the <em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code>; then that value is used to create and initialize the anonymous object of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>. The bounds of the anonymous object of an <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code> and the discriminants (if any) of the anonymous object of a <code><a href="../AA-4/AA-4.3#S0121">record_delta_aggregate</a></code> are those of the <em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>. If a <code><a href="../AA-4/AA-4.3#S0121">record_delta_aggregate</a></code> is of a specific tagged type, its tag is that of the specific type; if it is of a class-wide type, its tag is that of the <em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>.<br /></p>

<AnnotatedOnly>
<MarginText>14.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This is the same anonymous object as described in <a href="../AA-7/AA-7.6">7.6</a>, &ldquo;<a href="../AA-7/AA-7.6">Assignment and Finalization</a>&rdquo;; in particular, it might be required to be built in place.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.b/5</MarginText>
<Admonition type="aarm" aarm="note">
For a specific tagged type, any extension components that the actual object underlying the <em>base_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> might have are not used to initialize the anonymous object, which is critical if the aggregate is required to be built-in-place. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.c/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>The anonymous object associated with the evaluation of a <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code> begins its life as a variable, not a constant (<a href="../AA-3/AA-3.3">3.3</a> notwithstanding). This must be the case because the object is initialized and then subsequently modified. After evaluation of the <code><a href="../AA-4/AA-4.3#S0120">delta_aggregate</a></code> is complete, the object is a constant object. This is similar to the way that an extended return statement can provide a variable view of an object that will eventually be a constant object after the function returns its result. <br /></Admonition>
</AnnotatedOnly>

<MarginText>15/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<p>For a <code><a href="../AA-4/AA-4.3#S0121">record_delta_aggregate</a></code>, for each component associated with each <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> (in an unspecified order): <br /></p>

<MarginText>16/5</MarginText>
<ul>
<li>if the associated component belongs to a <code><a href="../AA-3/AA-3.8#S0072">variant</a></code>, a check is made that the values of the discriminants are such that the anonymous object has this component. The exception Constraint_Error is raised if this check fails.<br /></li>
<MarginText>17/5</MarginText>
<li>the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the <code><a href="../AA-4/AA-4.3#S0109">record_component_association</a></code> is evaluated, converted to the nominal subtype of the associated component, and assigned to the component of the anonymous object.<br /></li>
</ul>
<MarginText>18/5</MarginText>
<MarginInfo items={["AI12-0127-1"]} />
<p>For an <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code>, for each <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> of each <code><a href="../AA-4/AA-4.3#S0118">array_component_association</a></code> (in the order given in the enclosing <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code> and <code><a href="../AA-4/AA-4.3#S0117">array_component_association_list</a></code>, respectively) the <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> is evaluated; for each represented index value (in ascending order, if the <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code> represents a range): <br /></p>

<MarginText>19/5</MarginText>
<ul>
<li>the index value is converted to the index type of the array type.<br /></li>
<MarginText>20/5</MarginText>
<li>a check is made that the index value belongs to the index range of the anonymous object of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>; Constraint_Error is raised if this check fails.<br /></li>
<MarginText>21/5</MarginText>
<li>the component <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is evaluated, converted to the array component subtype, and assigned to the component of the anonymous object identified by the index value. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>21.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>Unlike other <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>s, an <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code> is evaluated in the order that it is written. This is necessary to get deterministic behavior, as (unlike other <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>s, including <code><a href="../AA-4/AA-4.3#S0121">record_delta_aggregate</a></code>s) there is no requirement that the specified components be distinct. As such, the order requirement ensures that every <code><a href="../AA-4/AA-4.3#S0122">array_delta_aggregate</a></code> has a well-defined result, even if the same component is specified multiple times. <br /></Admonition>
</AnnotatedOnly>

<a id="Subclause_4.3.5"></a>

## 4.3.5  Container Aggregates

<MarginText>1/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>In a <code><a href="../AA-4/AA-4.3#S0123">container_aggregate</a></code>, values are specified for elements of a container; for a <code><a href="../AA-4/AA-4.3#S0125">positional_container_aggregate</a></code>, the elements are given sequentially; for a <code><a href="../AA-4/AA-4.3#S0126">named_container_aggregate</a></code>, the elements are specified by a sequence of key/value pairs, or using an iterator. The Aggregate aspect of the type of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> determines how the elements are combined to form the container.<br /></p>

<AnnotatedOnly>
<MarginText>1.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>container aggregate</strong> &mdash; construct used to define a value of a type that represents a collection of elements, by explicitly specifying the elements in the collection<br /></Admonition>
</AnnotatedOnly>

<MarginText>2/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>For a type other than an array type, the following type-related operational aspect may be specified:<br /></p>

<MarginText>3/5</MarginText>
<dt><br/>Aggregate</dt>
<dl>
<dd>This aspect is an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> of the form:<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>3.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Aspect Description for </strong><strong>Aggregate: </strong>Mechanism to define user-defined aggregates.<br /></Admonition>
</AnnotatedOnly>

<MarginText>4/5</MarginText>
<p>   (Empty ={">"} <code><a href="../AA-4/AA-4.1#S0091">name</a></code>[,<br />    Add_Named ={">"} <em>procedure_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>][,<br />    Add_Unnamed ={">"} <em>procedure_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>][,<br />    New_Indexed ={">"} <em>function_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>,<br />    Assign_Indexed ={">"} <em>procedure_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>])<br /></p>

<AnnotatedOnly>
<MarginText>4.a/5</MarginText>
<MarginInfo items={["AI22-0020-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>As this aspect is described with syntax, it has to be given exactly as specified here. That means it cannot be given as a positional aggregate, nor can the order of the elements be changed. For instance, New_Indexed always has to occur before Assign_Indexed, and Empty always has to be first. <br /></Admonition>
</AnnotatedOnly>

<MarginText>5/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<dl>
<dd>The type for which this aspect is specified is known as the <em>container type</em> of the Aggregate aspect. A <em>procedure_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> shall be specified for at least one of Add_Named, Add_Unnamed, or Assign_Indexed. If Add_Named is specified, neither Add_Unnamed nor Assign_Indexed shall be specified. Either both or neither of New_Indexed and Assign_Indexed shall be specified.<br /></dd>
</dl>

#### Name Resolution Rules

<MarginText>6/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>The <code><a href="../AA-4/AA-4.1#S0091">name</a></code> specified for Empty for an Aggregate aspect shall denote a constant of the container type, or denote exactly one function with a result type of the container type that has no parameters, or that has one <strong>in</strong> parameter of a signed integer type.<br /></p>

<AnnotatedOnly>
<MarginText>6.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>In the function case, the parameter, if present may be used to specify an initial size for the container, in anticipation of adding elements to it. For a positional aggregate, or a named aggregate that doesn't use an iterator, it will be initialized with the number of elements. For a named aggregate that uses an iterator, the implementation is permitted to estimate the number of elements that the iterator will produce, but it is not required to do so. <br /></Admonition>
</AnnotatedOnly>

<MarginText>7/5</MarginText>
<MarginInfo items={["AI12-0212-1", "AI12-0437-1"]} />
<p>The <em>procedure_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> specified for Add_Unnamed for an Aggregate aspect shall denote exactly one procedure that has two parameters, the first an <strong>in out</strong> parameter of the container type, and the second an <strong>in</strong> parameter of some nonlimited type, called the <em>element type</em> of the container type.<br /></p>

<MarginText>8/5</MarginText>
<MarginInfo items={["AI12-0212-1", "AI12-0437-1"]} />
<p>The <em>function_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> specified for New_Indexed for an Aggregate aspect shall denote exactly one function with a result type of the container type, and two parameters of the same discrete type, with that type being the <em>key type</em> of the container type.<br /></p>

<AnnotatedOnly>
<MarginText>8.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The New_Indexed function is used instead of Empty as the first step of creating an aggregate that is initialized using the Assign_Indexed procedure. <br /></Admonition>
</AnnotatedOnly>

<MarginText>9/5</MarginText>
<MarginInfo items={["AI12-0212-1", "AI12-0437-1"]} />
<p>The <em>procedure_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> specified for Add_Named or Assign_Indexed for an Aggregate aspect shall denote exactly one procedure that has three parameters, the first an <strong>in out</strong> parameter of the container type, the second an <strong>in</strong> parameter of a nonlimited type (the <em>key type</em> of the container type), and the third, an <strong>in</strong> parameter of a nonlimited type that is called the <em>element type</em> of the container type.<br /></p>


#### Legality Rules

<MarginText>10/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>If the container type of an Aggregate aspect is a private type, the full type of the container type shall not be an array type. If the container type is limited, the name specified for Empty shall denote a function rather than a constant object.<br /></p>

<MarginText>11/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>For an Aggregate aspect, the key type of Assign_Indexed shall be the same type as that of the parameters of New_Indexed. Additionally, if both Add_Unnamed and Assign_Indexed are specified, the final parameters shall be of the same type &mdash; the element type of the container type.<br /></p>


#### Static Semantics

<MarginText>12/5</MarginText>
<MarginInfo items={["AI12-0212-1", "AI12-0388-1"]} />
<p>The Aggregate aspect is nonoverridable (see <a href="../AA-13/AA-13.1#Subclause_13.1.1">13.1.1</a>). <br /></p>

#### Name Resolution Rules

<MarginText>22/5</MarginText>
<MarginInfo items={["AI12-0212-1", "AI12-0437-1"]} />
<p>The expected type for a <code><a href="../AA-4/AA-4.3#S0123">container_aggregate</a></code> shall be a single type for which the Aggregate aspect has been specified. The expected type for each <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a <code><a href="../AA-4/AA-4.3#S0123">container_aggregate</a></code> is the element type of the expected type.<br /></p>

<MarginText>23/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>The expected type for a <em>key_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, or a <code><a href="../AA-3/AA-3.6#S0058">discrete_range</a></code> of a <code><a href="../AA-4/AA-4.3#S0130">key_choice</a></code>, is the key type of the expected type of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>.<br /></p>


#### Legality Rules

<MarginText>24/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>The expected type for a <code><a href="../AA-4/AA-4.3#S0125">positional_container_aggregate</a></code> shall have an Aggregate aspect that includes a specification for an Add_Unnamed procedure or an Assign_Indexed procedure. The expected type for a <code><a href="../AA-4/AA-4.3#S0126">named_container_aggregate</a></code> that contains one or more <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code>s with a <em>key_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> shall have an Aggregate aspect that includes a specification for the Add_Named procedure. The expected type for a <code><a href="../AA-4/AA-4.3#S0126">named_container_aggregate</a></code> that contains one or more <code><a href="../AA-4/AA-4.3#S0129">key_choice_list</a></code>s shall have an Aggregate aspect that includes a specification for the Add_Named or Assign_Indexed procedure. [A <code><a href="../AA-4/AA-4.3#S0124">null_container_aggregate</a></code> can be of any type with an Aggregate aspect.]<br /></p>

<MarginText>25/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>A non-null container aggregate is called an <em>indexed aggregate</em> if the expected type <em>T</em> of the aggregate specifies an Assign_Indexed procedure in its Aggregate aspect, and either there is no Add_Unnamed procedure specified for the type, or the aggregate is a <code><a href="../AA-4/AA-4.3#S0126">named_container_aggregate</a></code> with a <code><a href="../AA-4/AA-4.3#S0128">container_element_association</a></code> that contains a <code><a href="../AA-4/AA-4.3#S0129">key_choice_list</a></code> or a <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code>. The key type of an indexed aggregate is also called the <em>index type</em> of the aggregate.<br /></p>

<MarginText>26/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>A <code><a href="../AA-4/AA-4.3#S0128">container_element_association</a></code> with a {"<"}{">"} rather than an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, or with a <code><a href="../AA-4/AA-4.3#S0130">key_choice</a></code> that is a <code><a href="../AA-3/AA-3.6#S0058">discrete_range</a></code>, is permitted only in an indexed aggregate.<br /></p>

<AnnotatedOnly>
<MarginText>26.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>Only an element of an indexed aggregate can be left uninitialized, because the compiler can simply omit producing an Assign_Indexed operation for the given index. For other kinds of aggregates, there are no operations that add an element without giving it a value. We could have defined such (optional) operations, but they would have added significant complication to an already complex feature without adding much additional expressive power. Note that, to be consistent with array aggregates, we do not support specifying {"<"}{">"} in an <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>27/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>For an <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code> without a <em>key_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, if the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> is an indexed aggregate or the expected type of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> specifies an Add_Named procedure in its Aggregate aspect, then the type of the loop parameter of the <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code> shall be the same as the key type of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>.<br /></p>

<AnnotatedOnly>
<MarginText>27.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>If there is a <em>key_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> in an <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code>, it determines the key of each added key/value pair, rather than the loop parameter. But if there is no <em>key_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, the loop parameter itself is used as the key. <br /></Admonition>
</AnnotatedOnly>

<MarginText>28/5</MarginText>
<MarginInfo items={["AI12-0212-1", "AI12-0250-1"]} />
<p>For a <code><a href="../AA-4/AA-4.3#S0126">named_container_aggregate</a></code> that is an indexed aggregate, all <code><a href="../AA-4/AA-4.3#S0128">container_element_association</a></code>s shall contain either a <code><a href="../AA-4/AA-4.3#S0129">key_choice_list</a></code>, or a <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> without a <em>key_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> or <code><a href="../AA-5/AA-5.5#S0182">iterator_filter</a></code>. Furthermore, for such an aggregate, either: <br /></p>

<MarginText>29/5</MarginText>
<ul>
<li>all <code><a href="../AA-4/AA-4.3#S0130">key_choice</a></code>s shall be static expressions or static ranges, and every <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> shall have a <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code> that defines a non-null static range, and the set of values of the index type covered by the <code><a href="../AA-4/AA-4.3#S0130">key_choice</a></code>s and the <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>s shall form a contiguous range of values with no duplications; or<br /></li>
<MarginText>30/5</MarginText>
<li>there shall be exactly one <code><a href="../AA-4/AA-4.3#S0128">container_element_association</a></code>, and if it has a <code><a href="../AA-4/AA-4.3#S0129">key_choice_list</a></code>, the list shall have exactly one <code><a href="../AA-4/AA-4.3#S0130">key_choice</a></code>. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>30.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The above is trying to mimic the rules for <code><a href="../AA-4/AA-4.3#S0116">named_array_aggregate</a></code>s, without <strong>others</strong>. <br /></Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>31/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>The evaluation of a <code><a href="../AA-4/AA-4.3#S0123">container_aggregate</a></code> starts by creating an anonymous object <em>A</em> of the expected type <em>T</em>, initialized as follows:<br /></p>

<MarginText>32/5</MarginText>
<ul>
<li>if the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> is an indexed aggregate, from the result of a call on the New_Indexed function; the actual parameters in this call represent the lower and upper bound of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, and are determined as follows: <br /></li>
<MarginText>33/5</MarginText>
<li>if the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> is a <code><a href="../AA-4/AA-4.3#S0125">positional_container_aggregate</a></code>, the lower bound is the low bound of the subtype of the key parameter of the Add_Indexed procedure, and the upper bound has a position number that is the sum of the position number of the lower bound and one less than the number of <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s in the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>;<br /></li>
<MarginText>34/5</MarginText>
<li>if the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> is a <code><a href="../AA-4/AA-4.3#S0126">named_container_aggregate</a></code>, the lower bound is the lowest value covered by a <code><a href="../AA-4/AA-4.3#S0129">key_choice_list</a></code> or is the low bound of a range defined by a <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code> of a <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code>; the upper bound is the highest value covered by a <code><a href="../AA-4/AA-4.3#S0129">key_choice_list</a></code> or is the high bound of a range defined by a <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code> of a <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code>. <br /></li>
<MarginText>35/5</MarginText>
<li>if the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> is not an indexed aggregate, by assignment from the Empty constant, or from a call on the Empty function specified in the Aggregate aspect. In the case of an Empty function with a formal parameter, the actual parameter has the following value:<br /></li>
<MarginText>36/5</MarginText>
<li>for a <code><a href="../AA-4/AA-4.3#S0124">null_container_aggregate</a></code>, the value zero;<br /></li>
<MarginText>37/5</MarginText>
<li>for a <code><a href="../AA-4/AA-4.3#S0125">positional_container_aggregate</a></code>, the number of <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s;<br /></li>
<MarginText>38/5</MarginText>
<li>for a <code><a href="../AA-4/AA-4.3#S0126">named_container_aggregate</a></code> without an <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code>, the number of <em>key_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s;<br /></li>
<MarginText>39/5</MarginText>
<li>for a <code><a href="../AA-4/AA-4.3#S0126">named_container_aggregate</a></code> where every <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code> contains a <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code>, the total number of elements specified by all of the <code><a href="../AA-4/AA-4.3#S0128">container_element_association</a></code>s;<br /></li>
<MarginText>40/5</MarginText>
<li>otherwise, to an implementation-defined value.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>40.a/5</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>This value ought to be an estimate for the number of elements in the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, if one is available. If not, it is suggested to use the default value for the parameter if one exists, and zero otherwise. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>40.b/5</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>The value of the parameter to Empty for some <code><a href="../AA-4/AA-4.3#S0123">container_aggregate</a></code>s.<br /></Admonition>
</AnnotatedOnly>

<MarginText>41/5</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<p>The evaluation then proceeds as follows:<br /></p>

<MarginText>42/5</MarginText>
<ul>
<li>for a <code><a href="../AA-4/AA-4.3#S0124">null_container_aggregate</a></code>, the anonymous object <em>A</em> is the result;<br /></li>
<MarginText>43/5</MarginText>
<li>for a <code><a href="../AA-4/AA-4.3#S0125">positional_container_aggregate</a></code> of a type with a specified Add_Unnamed procedure, each <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is evaluated in an arbitrary order, and the Add_Unnamed procedure is invoked in sequence with the anonymous object <em>A</em> as the first parameter and the result of evaluating each <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> as the second parameter, in the order of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s;<br /></li>
<MarginText>44/5</MarginText>
<li>for a <code><a href="../AA-4/AA-4.3#S0125">positional_container_aggregate</a></code> that is an indexed aggregate, each <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is evaluated in an arbitrary order, and the Assign_Indexed procedure is invoked in sequence with the anonymous object <em>A</em> as the first parameter, the key value as the second parameter, computed by starting with the low bound of the subtype of the key formal parameter of the Assign_Indexed procedure and taking the successor of this value for each successive <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, and the result of evaluating each <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> as the third parameter;<br /></li>
<MarginText>45/5</MarginText>
<li>for a <code><a href="../AA-4/AA-4.3#S0126">named_container_aggregate</a></code> for a type with an Add_Named procedure in its Aggregate aspect, the <code><a href="../AA-4/AA-4.3#S0128">container_element_association</a></code>s are evaluated in an arbitrary order: <br /></li>
<MarginText>46/5</MarginText>
<li>for a <code><a href="../AA-4/AA-4.3#S0128">container_element_association</a></code> with a <code><a href="../AA-4/AA-4.3#S0129">key_choice_list</a></code>, for each <code><a href="../AA-4/AA-4.3#S0130">key_choice</a></code> of the list in an arbitrary order, the <code><a href="../AA-4/AA-4.3#S0130">key_choice</a></code> is evaluated as is the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the <code><a href="../AA-4/AA-4.3#S0128">container_element_association</a></code> (in an arbitrary order), and the Add_Named procedure is invoked once for each value covered by the <code><a href="../AA-4/AA-4.3#S0130">key_choice</a></code>, with the anonymous object <em>A</em> as the first parameter, the value from the <code><a href="../AA-4/AA-4.3#S0130">key_choice</a></code> as the second parameter, and the result of evaluating the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> as the third parameter;<br /></li>
<MarginText>47/5</MarginText>
<MarginInfo items={["AI12-0212-1", "AI12-0327-1"]} />
<li>for a <code><a href="../AA-4/AA-4.3#S0128">container_element_association</a></code> with an <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code>, first the <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code> is elaborated, then an iteration is performed, and for each value conditionally produced by the iteration (see <a href="../AA-5/AA-5.5">5.5</a> and <a href="../AA-5/AA-5.5#Subclause_5.5.2">5.5.2</a>) the Add_Named procedure is invoked with the anonymous object <em>A</em> as the first parameter, the result of evaluating the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> as the third parameter, and:<br /></li>
<MarginText>48/5</MarginText>
<li>if there is a <em>key_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, the result of evaluating the <em>key_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> as the second parameter;<br /></li>
<MarginText>49/5</MarginText>
<li>otherwise, with the loop parameter as the second parameter;<br /></li>
<MarginText>50/5</MarginText>
<li>for a <code><a href="../AA-4/AA-4.3#S0126">named_container_aggregate</a></code> that is an indexed aggregate, the evaluation proceeds as above for the case of Add_Named, but with the Assign_Indexed procedure being invoked instead of Add_Named; in the case of a <code><a href="../AA-4/AA-4.3#S0128">container_element_association</a></code> with a {"<"}{">"} rather than an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, the corresponding call on Assign_Indexed is not performed, leaving the component as it was upon return from the New_Indexed function;<br /></li>
<MarginText>51/5</MarginText>
<li>for any other <code><a href="../AA-4/AA-4.3#S0126">named_container_aggregate</a></code>, the <code><a href="../AA-4/AA-4.3#S0128">container_element_association</a></code>s (which are necessarily <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code>s) are evaluated in the order given; each such evaluation comprises two steps: <br /></li>
</ul>
<MarginText>52/5</MarginText>
<dt><br/>1.</dt>
<dl>
<dd>the <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code> is elaborated;<br /></dd>
<MarginText>53/5</MarginText>
<dt><br/>2.</dt>
<MarginInfo items={["AI12-0212-1", "AI12-0327-1"]} />
<dd>an iteration is performed, and for each value conditionally produced by the iteration (see <a href="../AA-5/AA-5.5">5.5</a> and <a href="../AA-5/AA-5.5#Subclause_5.5.2">5.5.2</a>) the Add_Unnamed procedure is invoked, with the anonymous object <em>A</em> as the first parameter and the result of evaluating the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> as the second parameter.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>53.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>In this case, the value of the loop parameter is not directly relevant, though presumably it appears within the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>60/5</MarginText>
   Empty_Map : constant Map_Type;{"\n"}
<MarginText>61/5</MarginText>
   --  Vector_Type is an extensible array-like container type.{"\n"}   type Vector_Type is private{"\n"}      with Aggregate ={">"} (Empty          ={">"} Empty_Vector,{"\n"}                         Add_Unnamed    ={">"} Append_One,{"\n"}                         New_Indexed    ={">"} New_Vector,{"\n"}                         Assign_Indexed ={">"} Assign_Element);{"\n"}
<MarginText>62/5</MarginText>
   function Empty_Vector (Capacity : Integer := 0) return Vector_Type;{"\n"}
<MarginText>63/5</MarginText>
   procedure Append_One (V : in out Vector_Type; New_Item : in String);{"\n"}
<MarginText>64/5</MarginText>

   function New_Vector (First, Last : Positive) return Vector_Type{"\n"}      with Pre ={">"} First = Positive'First;{"\n"}      --  Vectors are always indexed starting at the{"\n"}      --  lower bound of their index subtype.{"\n"}

<p><em>Examples of container aggregates for Set_Type, Map_Type, and Vector_Type:</em><br /></p>

<MarginText>68/5</MarginText>
<CodeBlock language="ada">
--  Example aggregates using Set_Type.{"\n"}S : Set_Type;{"\n"}
<MarginText>69/5</MarginText>
--  Assign the empty set to S:{"\n"}S := [];{"\n"}
<MarginText>70/5</MarginText>
--  Is equivalent to:{"\n"}S := Empty_Set;{"\n"}
<MarginText>71/5</MarginText>
--  A positional set aggregate:{"\n"}S := [1, 2];{"\n"}
<MarginText>72/5</MarginText>
--  Is equivalent to:{"\n"}S := Empty_Set;{"\n"}Include (S, 1);{"\n"}Include (S, 2);{"\n"}
<MarginText>73/5</MarginText>
--  A set aggregate with an <a href="../AA-4/AA-4.3#S0131">iterated_element_association</a>:{"\n"}S := [for Item in 1 .. 5 ={">"} Item * 2];{"\n"}
<MarginText>74/5</MarginText>
--  Is equivalent to:{"\n"}S := Empty_Set;{"\n"}for Item in 1 .. 5 loop{"\n"}   Include (S, Item * 2);{"\n"}end loop;{"\n"}
<MarginText>75/5</MarginText>
