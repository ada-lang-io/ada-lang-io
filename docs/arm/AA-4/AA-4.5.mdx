The corresponding <code><a href="../AA-6/AA-6.1#S0202">operator_symbol</a></code>s, and only those, can be used as <code><a href="../AA-6/AA-6.1#S0199">designator</a></code>s in declarations of functions for user-defined operators. #### Static Semantics

<MarginText>8</MarginText>
<p>For a sequence of operators of the same precedence level, the operators are associated with their operands in textual order from left to right. Parentheses can be used to impose specific associations. <br /></p>

<AnnotatedOnly>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The left-associativity is not directly inherent in the grammar of <a href="../AA-4/AA-4.4">4.4</a>, though in <a href="../AA-1/AA-1.1#Subclause_1.1.4">1.1.4</a> the definition of the metasymbols <strong>{"{"}{"}"}</strong> implies left associativity. So this could be seen as redundant, depending on how literally one interprets the definition of the <strong>{"{"}{"}"}</strong> metasymbols.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.b</MarginText>
<Admonition type="aarm" aarm="note">
See the Implementation Permissions below regarding flexibility in reassociating operators of the same precedence. <br /></Admonition>
</AnnotatedOnly>

<MarginText>9</MarginText>
<p>For each form of type definition, certain of the above operators are <em>predefined</em>; that is, they are implicitly declared immediately after the type definition. For each such implicit operator declaration, the parameters are called Left and Right for <em>binary</em> operators; the single parameter is called Right for <em>unary</em> operators. [An expression of the form X op Y, where op is a binary operator, is equivalent to a <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code> of the form "op"(X, Y). An expression of the form op Y, where op is a unary operator, is equivalent to a <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code> of the form "op"(Y). The predefined operators and their effects are described in subclauses <a href="../AA-4/AA-4.5#Subclause_4.5.1">4.5.1</a> through <a href="../AA-4/AA-4.5#Subclause_4.5.6">4.5.6</a>.] <br /></p>


#### Dynamic Semantics

<MarginText>10</MarginText>
<p>[ The predefined operations on integer types either yield the mathematically correct result or raise the exception Constraint_Error. For implementations that support the Numerics Annex, the predefined operations on real types yield results whose accuracy is defined in <a href="../AA-G/">Annex G</a>, or raise the exception Constraint_Error. ]<br /></p>

<AnnotatedOnly>
<MarginText>10.a</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>Predefined operations on real types can &ldquo;silently&rdquo; give wrong results when the Machine_Overflows attribute is false, and the computation overflows. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.b/5</MarginText>
<Admonition type="aarm" aarm="proof">
<strong></strong>For integer types, this is normatively stated in the Dynamic Semantics of <a href="../AA-3/AA-3.5#Subclause_3.5.4">3.5.4</a>. For floating point types, this is normatively stated at the end of the Implementation Requirements of <a href="../AA-G/AA-G.2#Subclause_G.2.1">G.2.1</a>. For fixed point types, this is normatively stated at the end of the Implementation Requirements of <a href="../AA-G/AA-G.2#Subclause_G.2.3">G.2.3</a>. <br /></Admonition>
</AnnotatedOnly>


#### Implementation Requirements

<MarginText>11</MarginText>
<p>The implementation of a predefined operator that delivers a result of an integer or fixed point type may raise Constraint_Error only if the result is outside the base range of the result type.<br /></p>

<MarginText>12</MarginText>
<p>The implementation of a predefined operator that delivers a result of a floating point type may raise Constraint_Error only if the result is outside the safe range of the result type. <br /></p>

<AnnotatedOnly>
<MarginText>12.a</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>An exception is made for exponentiation by a negative exponent in <a href="../AA-4/AA-4.5#Subclause_4.5.6">4.5.6</a>.<br /></Admonition>
</AnnotatedOnly>


#### Implementation Permissions

<MarginText>13</MarginText>
<p>For a sequence of predefined operators of the same precedence level (and in the absence of parentheses imposing a specific association), an implementation may impose any association of the operators with operands so long as the result produced is an allowed result for the left-to-right association, but ignoring the potential for failure of language-defined checks in either the left-to-right or chosen order of association. <br /></p>

<AnnotatedOnly>
<MarginText>13.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Note that the permission to reassociate the operands in any way subject to producing a result allowed for the left-to-right association is not much help for most floating point operators, since reassociation may introduce significantly different round-off errors, delivering a result that is outside the model interval for the left-to-right association. Similar problems arise for division with integer or fixed point operands.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>13.b</MarginText>
<Admonition type="aarm" aarm="note">
Note that this permission does not apply to user-defined operators. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   The two operands of an expression of the form X op Y, where op is a binary operator, are evaluated in an arbitrary order, as for any <code><a href="../AA-6/AA-6.4#S0218">function_call</a></code> (see <a href="../AA-6/AA-6.4">6.4</a>).<br /></Admonition>
</AnnotatedOnly>

## 4.5.1  Logical Operators and Short-circuit Control Forms


#### Name Resolution Rules

<MarginText>1</MarginText>
<p>An <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> consisting of two <code><a href="../AA-4/AA-4.4#S0135">relation</a></code>s connected by <strong>and then</strong> or <strong>or else</strong> (a <em>short-circuit control form</em>) shall resolve to be of some boolean type; the expected type for both <code><a href="../AA-4/AA-4.4#S0135">relation</a></code>s is that same boolean type. <br /></p>

<AnnotatedOnly>
<MarginText>1.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This rule is written this way so that overload resolution treats the two operands symmetrically; the resolution of overloading present in either one can benefit from the resolution of the other. Furthermore, the type expected by context can help. <br /></Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>2</MarginText>
<p>The following logical operators are predefined for every boolean type <em>T</em>, for every modular type <em>T</em>, and for every one-dimensional array type <em>T</em> whose component type is a boolean type: <br /></p>

<MarginText>3</MarginText>
<CodeBlock language="ada">
function "and"(Left, Right : T) return T{"\n"}function "or" (Left, Right : T) return T{"\n"}function "xor"(Left, Right : T) return T{"\n"}

</CodeBlock>
<AnnotatedOnly>
<MarginText>3.a/2</MarginText>
<MarginInfo items={["AI95-00145-01"]} />
<Admonition type="aarm" aarm="note">
<em>This paragraph was deleted.</em><br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.b/2</MarginText>
<MarginInfo items={["AI95-00145-01"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>For these operators, we are talking about the type without any (interesting) subtype, and not some subtype with a constraint or exclusion. Since it's possible that there is no name for the &ldquo;uninteresting&rdquo; subtype, we denote the type with an italicized <em>T</em>. This applies to the italicized <em>T</em> in many other predefined operators and attributes as well.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.c/2</MarginText>
<MarginInfo items={["AI95-00145-01"]} />
<Admonition type="aarm" aarm="note">
In many cases, there is a subtype with the correct properties available. The italicized <em>T</em> means: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.d/2</MarginText>
<ul>
<li><em>T</em>'Base, for scalars;<br /></li>
<MarginText>3.e/2</MarginText>
<li>the first subtype of <em>T</em>, for tagged types;<br /></li>
<MarginText>3.f/2</MarginText>
<li>a subtype of the type <em>T</em> without any constraint or null exclusion, in other cases. <br /></li>
</ul>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>3.g/2</MarginText>
<Admonition type="aarm" aarm="note">
Note that &ldquo;without a constraint&rdquo; is not the same as unconstrained. For instance, a record type with no discriminant part is considered constrained; no subtype of it has a constraint, but the subtype is still constrained.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.h/2</MarginText>
<Admonition type="aarm" aarm="note">
Thus, the last case often is the same as the first subtype of <em>T</em>, but that isn't the case for constrained array types (where the correct subtype is unconstrained) and for access types with a <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code> (where the correct subtype does not exclude null).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.i/2</MarginText>
<Admonition type="aarm" aarm="note">
This italicized <em>T</em> is used for defining operators and attributes of the language. The meaning is intended to be as described here. <br /></Admonition>
</AnnotatedOnly>

<MarginText>4</MarginText>
<p>For boolean types, the predefined logical operators <strong>and</strong>, <strong>or</strong>, and <strong>xor</strong> perform the conventional operations of conjunction, inclusive disjunction, and exclusive disjunction, respectively.<br /></p>

<MarginText>5</MarginText>
<p>For modular types, the predefined logical operators are defined on a bit-by-bit basis, using the binary representation of the value of the operands to yield a binary representation for the result, where zero represents False and one represents True. If this result is outside the base range of the type, a final subtraction by the modulus is performed to bring the result into the base range of the type.<br /></p>

<MarginText>6</MarginText>
<p>The logical operators on arrays are performed on a component-by-component basis on matching components (as for equality &mdash; see <a href="../AA-4/AA-4.5#Subclause_4.5.2">4.5.2</a>), using the predefined logical operator for the component type. The bounds of the resulting array are those of the left operand.<br /></p>


#### Dynamic Semantics

<MarginText>7</MarginText>
<p>The short-circuit control forms <strong>and then</strong> and <strong>or else</strong> deliver the same result as the corresponding predefined <strong>and</strong> and <strong>or</strong> operators for boolean types, except that the left operand is always evaluated first, and the right operand is not evaluated if the value of the left operand determines the result.<br /></p>

<MarginText>8</MarginText>
<p>For the logical operators on arrays, a check is made that for each component of the left operand there is a matching component of the right operand, and vice versa. Also, a check is made that each component of the result belongs to the component subtype. The exception Constraint_Error is raised if either of the above checks fails. <br /></p>

<AnnotatedOnly>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The check against the component subtype is per AI83-00535. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   The conventional meaning of the logical operators is given by the following truth table: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10</MarginText>
<Admonition type="aarm" aarm="note">
	  A	  B	(A <strong>and</strong> B)	(A <strong>or</strong> B)	(A <strong>xor</strong> B)<br /><br />	True  	True  	True  	True  	False<br />	True  	False 	False 	True  	True<br />	False 	True  	False 	True  	True<br />	False 	False 	False 	False 	False<br /></Admonition>
</AnnotatedOnly>

## 4.5.2  Relational Operators and Membership Tests

<MarginText>1</MarginText>
<p>[ The <em>equality operators</em> = (equals) and /= (not equals) are predefined for nonlimited types. The other <code><a href="../AA-4/AA-4.5#S0143">relational_operator</a></code>s are the <em>ordering operators</em> {"<"} (less than), {"<"}= (less than or equal), {">"} (greater than), and {">"}= (greater than or equal). The ordering operators are predefined for scalar types, and for <em>discrete array types</em>, that is, one-dimensional array types whose components are of a discrete type. <br /></p>

<AnnotatedOnly>
<MarginText>1.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The equality operators are not defined for <em>every</em> nonlimited type &mdash; see below for the exact rule. <br /></Admonition>
</AnnotatedOnly>

<MarginText>2/3</MarginText>
<MarginInfo items={["AI05-0262-1", "AI05-0269-1"]} />
<p>A <em>membership test</em>, using <strong>in</strong> or <strong>not in</strong>, determines whether or not a value belongs to any given subtype or range, is equal to any given value, has a tag that identifies a type that is covered by a given type, or is convertible to and has an accessibility level appropriate for a given access type. Membership tests are allowed for all types.]<br /></p>


#### Name Resolution Rules

<MarginText>3/3</MarginText>
<MarginInfo items={["AI95-00251-01", "AI05-0158-1"]} />
<p>The <em>tested type</em> of a membership test is determined by the <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code>s of the <code><a href="../AA-4/AA-4.4#S0136">membership_choice_list</a></code>. Either all <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code>s of the <code><a href="../AA-4/AA-4.4#S0136">membership_choice_list</a></code> shall resolve to the same type, which is the tested type; or each <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code> shall be of an elementary type, and the tested type shall be covered by each of these elementary types.<br /></p>

<MarginText>3.1/5</MarginText>
<MarginInfo items={["AI05-0158-1", "AI12-0039-1", "AI12-0418-1"]} />
<p>If the tested type is tagged, then the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> shall resolve to be of a type that is convertible (see <a href="../AA-4/AA-4.6">4.6</a>) to the tested type; if untagged, the expected type of  the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> is the tested type. The expected type of a <em>choice_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> in a <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code>, and of a <code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> of a <code><a href="../AA-3/AA-3.5#S0037">range</a></code> in a <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code>, is the tested type of the membership operation.<br /></p>

<AnnotatedOnly>
<MarginText>3.a/2</MarginText>
<MarginInfo items={["AI95-00230-01"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>The part of the rule for untagged types is stated in a way that ensures that operands like a string literal are still legal as operands of a membership test.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.b/4</MarginText>
<MarginInfo items={["AI95-00251-01", "AI12-0039-1"]} />
<Admonition type="aarm" aarm="note">
The significance of &ldquo;is convertible to&rdquo; is that we allow the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> to be of any class-wide type that could be converted to the tested type, not just the one rooted at the tested type. This includes any class-wide type that covers the tested type, along with class-wide interfaces in some cases.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.c/3</MarginText>
<MarginInfo items={["AI05-0158-1"]} />
<Admonition type="aarm" aarm="note">
The special rule for determining the tested type for elementary types is to allow numeric literals in <code><a href="../AA-4/AA-4.4#S0136">membership_choice_list</a></code>s. Without the rule, A <strong>in</strong> B | 1 would be illegal as B and 1 would have different types (the literal having type <em>universal integer</em>). <br /></Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>4/4</MarginText>
<MarginInfo items={["AI12-0039-1"]} />
<p>For a membership test, if the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> is of a tagged class-wide type, then the tested type shall be (visibly) tagged. <br /></p>

<AnnotatedOnly>
<MarginText>4.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Untagged types covered by the tagged class-wide type are not permitted. Such types can exist if they are descendants of a private type whose full type is tagged. This rule is intended to avoid confusion since such derivatives don't have their &ldquo;own&rdquo; tag, and hence are indistinguishable from one another at run time once converted to a covering class-wide type. <br /></Admonition>
</AnnotatedOnly>

<MarginText>4.1/5</MarginText>
<MarginInfo items={["AI05-0158-1", "AI12-0039-1", "AI12-0328-1"]} />
<p>If a membership test includes one or more <em>choice_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code>s and the tested type of the membership test is limited, then the tested type of the membership test shall have a visible primitive equality operator; if the tested type of the membership test is nonlimited with a user-defined primitive equality operator that is defined at a point where the type is limited, the tested type shall be a record type or record extension. <br /></p>

<AnnotatedOnly>
<MarginText>4.b/3</MarginText>
<MarginInfo items={["AI05-0158-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>A visible equality operator is required in order to avoid breaking privacy; that is, we don't want to depend on a hidden equality operator.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.c/5</MarginText>
<MarginInfo items={["AI12-0328-1"]} />
<Admonition type="aarm" aarm="note">
We make the membership test on the nonlimited view of a type illegal if it would use a different equality operator than what would be used for a limited view of the same type (and such a limited view is known to exist). <br /></Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>5</MarginText>
<p>The result type of a membership test is the predefined type Boolean.<br /></p>

<MarginText>6</MarginText>
<p>The equality operators are predefined for every specific type <em>T</em> that is not limited, and not an anonymous access type, with the following specifications: <br /></p>

<MarginText>7</MarginText>

</CodeBlock>

<MarginText>7.2/2</MarginText>
<CodeBlock language="ada">

</CodeBlock>
<MarginText>8</MarginText>

<MarginText>9</MarginText>
<CodeBlock language="ada">

</CodeBlock>

#### Name Resolution Rules

<MarginText>9.1/2</MarginText>
<MarginInfo items={["AI95-00230-01", "AI95-00420-01"]} />
<p>At least one of the operands of an equality operator for <em>universal_access</em> shall be of a specific anonymous access type. Unless the predefined equality operator is identified using an expanded name with <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> denoting the package Standard, neither operand shall be of an access-to-object type whose designated type is <em>D</em> or <em>D</em>'Class, where <em>D</em> has a user-defined primitive equality operator such that: <br /></p>

<MarginText>9.2/2</MarginText>
<ul>
<li>its result type is Boolean;<br /></li>
<MarginText>9.3/3</MarginText>
<MarginInfo items={["AI05-0020-1"]} />
<li>it is declared immediately within the same declaration list as <em>D</em> or any partial or incomplete view of <em>D</em>; and<br /></li>
<MarginText>9.4/2</MarginText>
<li>at least one of its operands is an access parameter with designated type <em>D</em>. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>9.a/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The first sentence prevents compatibility problems by ensuring that these operators are not used for named access types. Also, universal access types do not count for the purposes of this rule. Otherwise, equality expressions like (X = <strong>null</strong>) would be ambiguous for normal access types.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.b/2</MarginText>
<Admonition type="aarm" aarm="note">
The rest of the rule makes it possible to call (including a dispatching call) user-defined "=" operators for anonymous access-to-object types (they'd be hidden otherwise), and to write user-defined "=" operations for anonymous access types (by making it possible to see the universal operator using the Standard prefix). <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.c/2</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>We don't need a similar rule for anonymous access-to-subprogram types because they can't be primitive for any type. Note that any nonprimitive user-defined equality operators still are hidden by the universal operators; they'll have to be called with a package prefix, but they are likely to be very uncommon. <br /></Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>9.5/2</MarginText>
<MarginInfo items={["AI95-00230-01"]} />
<p>At least one of the operands of the equality operators for <em>universal_access</em> shall be of type <em>universal_access</em>, or both shall be of access-to-object types, or both shall be of access-to-subprogram types. Further: <br /></p>

<MarginText>9.6/2</MarginText>
<ul>
<li>When both are of access-to-object types, the designated types shall be the same or one shall cover the other, and if the designated types are elementary or array types, then the designated subtypes shall statically match;<br /></li>
<MarginText>9.7/2</MarginText>
<li>When both are of access-to-subprogram types, the designated profiles shall be subtype conformant. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>9.d/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We don't want to allow completely arbitrary comparisons, as we don't want to insist that all access types are represented in ways that are convertible to one another. For instance, a compiler could use completely separate address spaces or incompatible representations. Instead, we allow compares if there exists an access parameter to which both operands could be converted. Since the user could write such an subprogram, and any reasonable meaning for "=" would allow using it in such a subprogram, this doesn't impose any further restrictions on Ada implementations. <br /></Admonition>
</AnnotatedOnly>

<MarginText>9.8/5</MarginText>
<MarginInfo items={["AI05-0123-1", "AI12-0101-1", "AI12-0352-1"]} />
<p>If the profile of an explicitly declared primitive equality operator of an untagged record type is type conformant with that of the corresponding predefined equality operator, the declaration shall occur before the type is frozen. In addition, no type shall have been derived from the untagged record type before the declaration of the primitive equality operator. In addition to the places where Legality Rules normally apply (see <a href="../AA-12/AA-12.3">12.3</a>), this rule applies also in the private part of an instance of a generic unit. <br /></p>


#### Dynamic Semantics

<MarginText>10</MarginText>
<p>For discrete types, the predefined relational operators are defined in terms of corresponding mathematical operations on the position numbers of the values of the operands.<br /></p>

<MarginText>11</MarginText>
<p>For real types, the predefined relational operators are defined in terms of the corresponding mathematical operations on the values of the operands, subject to the accuracy of the type. <br /></p>

<AnnotatedOnly>
<MarginText>11.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>For floating point types, the results of comparing <em>nearly</em> equal values depends on the accuracy of the implementation (see <a href="../AA-G/AA-G.2#Subclause_G.2.1">G.2.1</a>, &ldquo;<a href="../AA-G/AA-G.2#Subclause_G.2.1">Model of Floating Point Arithmetic</a>&rdquo; for implementations that support the Numerics Annex). <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.b</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>On a machine with signed zeros, if the generated code generates both plus zero and minus zero, plus and minus zero must be equal by the predefined equality operators. <br /></Admonition>
</AnnotatedOnly>

<MarginText>12</MarginText>
<p>Two access-to-object values are equal if they designate the same object, or if both are equal to the null value of the access type.<br /></p>

<MarginText>13</MarginText>
<p>Two access-to-subprogram values are equal if they are the result of the same evaluation of an Access <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code>, or if both are equal to the null value of the access type. Two access-to-subprogram values are unequal if they designate different subprograms. [It is unspecified whether two access values that designate the same subprogram but are the result of distinct evaluations of Access <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code>s are equal or unequal.] <br /></p>

<AnnotatedOnly>
<MarginText>13.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This allows each Access <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code> for a subprogram to designate a distinct &ldquo;wrapper&rdquo; subprogram if necessary to support an indirect call. <br /></Admonition>
</AnnotatedOnly>

<MarginText>14/3</MarginText>
<MarginInfo items={["AI05-0123-1"]} />
<p>For a type extension, predefined equality is defined in terms of the primitive [(possibly user-defined)] equals operator for the parent type and for any components that have a record type in the extension part, and predefined equality for any other components not inherited from the parent type. <br /></p>

<AnnotatedOnly>
<MarginText>14.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Two values of a type extension are not equal if there is a <code><a href="../AA-3/AA-3.8#S0071">variant_part</a></code> in the extension part and the two values have different <code><a href="../AA-3/AA-3.8#S0072">variant</a></code>s present. This is a ramification of the requirement that a discriminant governing such a <code><a href="../AA-3/AA-3.8#S0071">variant_part</a></code> has to be a &ldquo;new&rdquo; discriminant, and so has to be equal in the two values for the values to be equal. Note that <code><a href="../AA-3/AA-3.8#S0071">variant_part</a></code>s in the parent part need not match if the primitive equals operator for the parent type considers them equal.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.b/2</MarginText>
<MarginInfo items={["AI95-00349-01"]} />
<Admonition type="aarm" aarm="note">
The full type extension's operation is used for a private extension. This follows as only full types have parent types; the type specified in a private extension is an ancestor, but not necessarily the parent type. For instance, in: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.c/2</MarginText>
<CodeBlock language="ada">
with Pak1;{"\n"}package Pak2 is{"\n"}   type Typ3 is new Pak1.Typ1 with private;{"\n"}private{"\n"}   type Typ3 is new Pak1.Typ2 with null record;{"\n"}end Pak2;{"\n"}  {"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>14.d/2</MarginText>
<Admonition type="aarm" aarm="note">
the parent type is Pak1.Typ2, not Pak1.Typ1, and the equality operator of Pak1.Typ2 is used to create predefined equality for Typ3. <br /></Admonition>
</AnnotatedOnly>

<MarginText>14.1/5</MarginText>
<MarginInfo items={["AI05-0123-1", "AI12-0413-1"]} />
<p> <br /></p>

<AnnotatedOnly>
<MarginText>14.e/5</MarginText>
<MarginInfo items={["AI12-0413-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong> <br /></Admonition>
</AnnotatedOnly>

<MarginText>15/5</MarginText>
<MarginInfo items={["AI05-0123-1", "AI12-0328-1"]} />
<p>For a private type, if its full type is a record type or a record extension, predefined equality is defined in terms of the primitive equals operator of the full type; otherwise, predefined equality for the private type is that of its full type.<br /></p>

<MarginText>16</MarginText>
<p>For other composite types, the predefined equality operators [(and certain other predefined operations on composite types &mdash; see <a href="../AA-4/AA-4.5#Subclause_4.5.1">4.5.1</a> and <a href="../AA-4/AA-4.6">4.6</a>)] are defined in terms of the corresponding operation on <em>matching components</em>, defined as follows: <br /></p>

<MarginText>17</MarginText>
<ul>
<li>For two composite objects or values of the same non-array type, matching components are those that correspond to the same <code><a href="../AA-3/AA-3.8#S0070">component_declaration</a></code> or <code><a href="../AA-3/AA-3.7#S0062">discriminant_specification</a></code>;<br /></li>
<MarginText>18</MarginText>
<li>For two one-dimensional arrays of the same type, matching components are those (if any) whose index values match in the following sense: the lower bounds of the index ranges are defined to match, and the successors of matching indices are defined to match;<br /></li>
<MarginText>19</MarginText>
<li>For two multidimensional arrays of the same type, matching components are those whose index values match in successive index positions. <br /></li>
</ul>
<MarginText>20</MarginText>
<p>The analogous definitions apply if the types of the two objects or values are convertible, rather than being the same. <br /></p>

<AnnotatedOnly>
<MarginText>20.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Ada 83 seems to omit this part of the definition, though it is used in array type conversions. See <a href="../AA-4/AA-4.6">4.6</a>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>21</MarginText>
<p>Given the above definition of matching components, the result of the predefined equals operator for composite types (other than for those composite types covered earlier) is defined as follows: <br /></p>

<MarginText>22</MarginText>
<ul>
<li>If there are no components, the result is defined to be True;<br /></li>
<MarginText>23</MarginText>
<li>If there are unmatched components, the result is defined to be False;<br /></li>
<MarginText>24/3</MarginText>
<MarginInfo items={["AI05-0123-1"]} />
<li>Otherwise, the result is defined in terms of the primitive equals operator for any matching components that are records, and the predefined equals for any other matching components. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>24.a/3</MarginText>
<MarginInfo items={["AI05-0123-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>This asymmetry between components with and without a record type is necessary to preserve most upward compatibility and corresponds with the corresponding situation with generics, where the predefined operations &ldquo;reemerge&rdquo; in a generic for non-record types, but do not for record types. Also, only tagged types support user-defined assignment (see <a href="../AA-7/AA-7.6">7.6</a>), so only tagged types can fully handle levels of indirection in the implementation of the type. For untagged types, one reason for a user-defined equals operator might be to allow values with different bounds or discriminants to compare equal in certain cases. When such values are matching components, the bounds or discriminants will necessarily match anyway if the discriminants of the enclosing values match. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Two null arrays of the same type are always equal; two null records of the same type are always equal.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.c/3</MarginText>
<MarginInfo items={["AI05-0123-1"]} />
<Admonition type="aarm" aarm="note">
Note that if a composite object has a component of a floating point type, and the floating point type has both a plus and minus zero, which are considered equal by the predefined equality, then a block compare cannot be used for the predefined composite equality. Of course, with user-defined equals operators for components that are records, a block compare breaks down anyway, so this is not the only special case that requires component-by-component comparisons. On a one's complement machine, a similar situation might occur for integer types, since one's complement machines typically have both a plus and minus (integer) zero. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.d/2</MarginText>
<MarginInfo items={["AI95-00230-01"]} />
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>For a component with an anonymous access type, &ldquo;predefined equality&rdquo; is that defined for the <em>universal_access</em> type (anonymous access types have no equality operators of their own).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.e/3</MarginText>
<MarginInfo items={["AI05-0123-1"]} />
<Admonition type="aarm" aarm="note">
For a component with a record type <em>T</em>, &ldquo;the primitive equals operator&rdquo; is the one with two parameters of <em>T</em> which returns Boolean. We're not talking about some random other primitive function named "=". <br /></Admonition>
</AnnotatedOnly>

<MarginText>24.1/5</MarginText>
<MarginInfo items={["AI05-0123-1", "AI12-0413-1"]} />
<p>If the primitive equals operator for an untagged record type is abstract, then Program_Error is raised at the point of any  call to that abstract subprogram[, implicitly as part of an equality operation on an enclosing composite object, or in an instance of a generic with a formal private type where the actual type is a record type with an abstract "="]. <br /></p>

<AnnotatedOnly>
<MarginText>24.f/5</MarginText>
<MarginInfo items={["AI12-0413-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>An explicit call to an abstract subprogram is generally illegal. This rule is needed in order to define the effect of a call in an instance of a generic body, or an implicit call such as a call that is part of the predefined equality operation for an enclosing composite type that has a component of an untagged record type that has an abstract primitive equals operator. For tagged types, an abstract primitive equals operator is only allowed for an abstract type, and abstract types cannot be components, so this case does not occur. <br /></Admonition>
</AnnotatedOnly>

<MarginText>24.2/1</MarginText>
<MarginInfo items={["AI95-00123-01"]} />
<p>{"{"}<em>8652/0016</em>{"}"} For any composite type, the order in which "=" is called for components is unspecified. Furthermore, if the result can be determined before calling "=" on some components, it is unspecified whether "=" is called on those components.<br /></p>

<MarginText>25</MarginText>
<p>The predefined "/=" operator gives the complementary result to the predefined "=" operator. <br /></p>

<AnnotatedOnly>
<MarginText>25.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Furthermore, if the user defines an "=" operator that returns Boolean, then a "/=" operator is implicitly declared in terms of the user-defined "=" operator so as to give the complementary result. See <a href="../AA-6/AA-6.6">6.6</a>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>26/3</MarginText>
<MarginInfo items={["AI05-0264-1"]} />
<p>For a discrete array type, the predefined ordering operators correspond to <em>lexicographic order</em> using the predefined order relation of the component type: A null array is lexicographically less than any array having at least one component. In the case of nonnull arrays, the left operand is lexicographically less than the right operand if the first component of the left operand is less than that of the right; otherwise, the left operand is lexicographically less than the right operand only if their first components are equal and the tail of the left operand is lexicographically less than that of the right (the <em>tail</em> consists of the remaining components beyond the first and can be null).<br /></p>

<MarginText>26.1/3</MarginText>
<MarginInfo items={["AI05-0269-1"]} />
<p>An <em>individual membership test</em> is the membership test of a single <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code>.<br /></p>

<MarginText>27/4</MarginText>
<MarginInfo items={["AI05-0158-1", "AI12-0039-1"]} />
<p>For the evaluation of a membership test using <strong>in</strong> whose <code><a href="../AA-4/AA-4.4#S0136">membership_choice_list</a></code> has a single <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code>, the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> and the <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code> are evaluated in an arbitrary order; the result is the result of the individual membership test for the <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code>.<br /></p>

<MarginText>27.1/4</MarginText>
<MarginInfo items={["AI05-0158-1", "AI12-0039-1"]} />
<p>For the evaluation of a membership test using <strong>in</strong> whose <code><a href="../AA-4/AA-4.4#S0136">membership_choice_list</a></code> has more than one <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code>, the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> of the membership test is evaluated first and the result of the operation is equivalent to that of a sequence consisting of an individual membership test on each <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code> combined with the short-circuit control form <strong>or else</strong>.<br /></p>

<AnnotatedOnly>
<MarginText>27.a.1/3</MarginText>
<MarginInfo items={["AI05-0158-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>This equivalence includes the evaluation of the <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code>s; evaluation stops as soon as an individual choice evaluates to True. <br /></Admonition>
</AnnotatedOnly>

<MarginText>28/3</MarginText>
<MarginInfo items={["AI05-0158-1", "AI05-0269-1"]} />
<p>An individual membership test yields the result True if: <br /></p>

<MarginText>28.1/5</MarginText>
<MarginInfo items={["AI05-0158-1", "AI05-0264-1", "AI12-0039-1", "AI12-0328-1"]} />
<ul>
<li>The <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code> is a <em>choice_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code>, and the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> is equal to the value of the <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code>. If the tested type is a record type or a record extension, or is limited at the point where the membership test occurs , the test uses the primitive equality for the type; otherwise, the test uses predefined equality.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>28.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We use the predefined equality operator if the membership test occurs where the type is nonlimited if the type is not a record type or record extension. However, to avoid confusion, cases where a membership test could use different equality operators based on the view are illegal. <br /></Admonition>
</AnnotatedOnly>

<MarginText>28.2/4</MarginText>
<MarginInfo items={["AI05-0153-3", "AI05-0158-1", "AI12-0039-1"]} />
<ul>
<li>The <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code> is a <code><a href="../AA-3/AA-3.5#S0037">range</a></code> and the value of the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> belongs to the given <code><a href="../AA-3/AA-3.5#S0037">range</a></code>.<br /></li>
<MarginText>29/4</MarginText>
<MarginInfo items={["AI05-0153-3", "AI05-0158-1", "AI12-0039-1", "AI12-0071-1"]} />
<li>The <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code> is a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>, the tested type is scalar, the value of the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> belongs to the range of the named subtype, and the value satisfies the predicates of the named subtype. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>29.a/3</MarginText>
<MarginInfo items={["AI05-0153-3"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>The scalar membership test only does a range check and a predicate check. It does not perform any other check, such as whether a value falls in a &ldquo;hole&rdquo; of a &ldquo;holey&rdquo; enumeration type. The Pos attribute function can be used for that purpose.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>29.b</MarginText>
<Admonition type="aarm" aarm="note">
Even though Standard.Float is an unconstrained subtype, the test &ldquo;X in Float&rdquo; will still return False (presuming the evaluation of X does not raise Constraint_Error) when X is outside Float'Range. <br /></Admonition>
</AnnotatedOnly>

<MarginText>30/4</MarginText>
<MarginInfo items={["AI95-00231-01", "AI05-0153-3", "AI05-0158-1", "AI12-0039-1", "AI12-0071-1"]} />
<ul>
<li>The <code><a href="../AA-4/AA-4.4#S0137">membership_choice</a></code> is a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>, the tested type is not scalar, the value of the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> satisfies any constraints of the named subtype, the value satisfies the predicates of the named subtype, and: <br /></li>
</ul>
<MarginText>30.1/4</MarginText>
<MarginInfo items={["AI95-00231-01", "AI12-0039-1"]} />
<ul>
<li>if the type of the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> is class-wide, the value has a tag that identifies a type covered by the tested type; <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>30.a/4</MarginText>
<MarginInfo items={["AI12-0039-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>Note that the tag is not checked if the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> is of a specific type. <br /></Admonition>
</AnnotatedOnly>

<MarginText>30.2/4</MarginText>
<MarginInfo items={["AI95-00231-01", "AI05-0149-1", "AI12-0039-1"]} />
<ul>
<li>if the tested type is an access type and the named subtype excludes null, the value of the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> is not null;<br /></li>
<MarginText>30.3/4</MarginText>
<MarginInfo items={["AI05-0149-1", "AI12-0039-1"]} />
<li>if the tested type is a general access-to-object type, the type of the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> is convertible to the tested type and its accessibility level is no deeper than that of the tested type; further, if the designated type of the tested type is tagged and the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> is nonnull, the tag of the object designated by the value of the <em>tested_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> is covered by the designated type of the tested type. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>30.b/5</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>In some of these cases, the bulleted checks need to pass before any predicate check is executed. Otherwise, a predicate check could be performed on an object of the wrong type. Consider:<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>

## 4.5.3  Binary Adding Operators


#### Static Semantics

<MarginText>1</MarginText>
<p>The binary adding operators + (addition) and &ndash; (subtraction) are predefined for every specific numeric type <em>T</em> with their conventional meaning. They have the following specifications: <br /></p>

<MarginText>2</MarginText>
<CodeBlock language="ada">
function "+"(Left, Right : T) return T{"\n"}function "-"(Left, Right : T) return T{"\n"}

</CodeBlock>
<MarginText>3</MarginText>
<p>The concatenation operators & are predefined for every nonlimited, one-dimensional array type <em>T</em> with component type <em>C</em>. They have the following specifications: <br /></p>

<MarginText>4</MarginText>
<CodeBlock language="ada">
function "&"(Left : T; Right : T) return T{"\n"}function "&"(Left : T; Right : C) return T{"\n"}function "&"(Left : C; Right : T) return T{"\n"}function "&"(Left : C; Right : C) return T{"\n"}

</CodeBlock>

#### Dynamic Semantics

<MarginText>5</MarginText>
<p>For the evaluation of a concatenation with result type <em>T</em>, if both operands are of type <em>T</em>, the result of the concatenation is a one-dimensional array whose length is the sum of the lengths of its operands, and whose components comprise the components of the left operand followed by the components of the right operand. If the left operand is a null array, the result of the concatenation is the right operand. Otherwise, the lower bound of the result is determined as follows: <br /></p>

<MarginText>6</MarginText>
<ul>
<li>If the ultimate ancestor of the array type was defined by a <code><a href="../AA-3/AA-3.6#S0054">constrained_array_definition</a></code>, then the lower bound of the result is that of the index subtype; <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This rule avoids Constraint_Error when using concatenation on an array type whose first subtype is constrained. <br /></Admonition>
</AnnotatedOnly>

<MarginText>7</MarginText>
<ul>
<li>If the ultimate ancestor of the array type was defined by an <code><a href="../AA-3/AA-3.6#S0052">unconstrained_array_definition</a></code>, then the lower bound of the result is that of the left operand. <br /></li>
</ul>
<MarginText>8</MarginText>
<p>[The upper bound is determined by the lower bound and the length.] A check is made that the upper bound of the result of the concatenation belongs to the range of the index subtype, unless the result is a null array. Constraint_Error is raised if this check fails.<br /></p>

<MarginText>9</MarginText>
<p>If either operand is of the component type <em>C</em>, the result of the concatenation is given by the above rules, using in place of such an operand an array having this operand as its only component (converted to the component subtype) and having the lower bound of the index subtype of the array type as its lower bound. <br /></p>

<AnnotatedOnly>
<MarginText>9.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The conversion might raise Constraint_Error. The conversion provides &ldquo;sliding&rdquo; for the component in the case of an array-of-arrays, consistent with the normal Ada 95 rules that allow sliding during parameter passing. <br /></Admonition>
</AnnotatedOnly>

<MarginText>10</MarginText>
<p>The result of a concatenation is defined in terms of an assignment to an anonymous object, as for any function call (see <a href="../AA-6/AA-6.5">6.5</a>). <br /></p>

<AnnotatedOnly>
<MarginText>10.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This implies that value adjustment is performed as appropriate &mdash; see <a href="../AA-7/AA-7.6">7.6</a>. We don't bother saying this for other predefined operators, even though they are all function calls, because this is the only one where it matters. It is the only one that can return a value having controlled parts. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   As for all predefined operators on modular types, the binary adding operators + and &ndash; on modular types include a final reduction modulo the modulus if the result is outside the base range of the type. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.a</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>A full "modulus" operation need not be performed after addition or subtraction of modular types. For binary moduli, a simple mask is sufficient. For nonbinary moduli, a check after addition to see if the value is greater than the high bound of the base range can be followed by a conditional subtraction of the modulus. Conversely, a check after subtraction to see if a "borrow" was performed can be followed by a conditional addition of the modulus. <br /></Admonition>
</AnnotatedOnly>

## 4.5.5  Multiplying Operators


#### Static Semantics

<MarginText>1</MarginText>
<p>The multiplying operators * (multiplication), / (division), <strong>mod</strong> (modulus), and <strong>rem</strong> (remainder) are predefined for every specific integer type <em>T</em>: <br /></p>

<MarginText>2</MarginText>
<CodeBlock language="ada">
function "*"  (Left, Right : T) return T{"\n"}function "/"  (Left, Right : T) return T{"\n"}function "mod"(Left, Right : T) return T{"\n"}function "rem"(Left, Right : T) return T{"\n"}

</CodeBlock>
<MarginText>3</MarginText>
<p>Signed integer multiplication has its conventional meaning.<br /></p>

<MarginText>4</MarginText>
<p>Signed integer division and remainder are defined by the relation: <br /></p>

<MarginText>5</MarginText>
<CodeBlock language="ada">
A = (A/B)*B + (A rem B){"\n"}

</CodeBlock>
<MarginText>6</MarginText>
<p>where (A <strong>rem</strong> B) has the sign of A and an absolute value less than the absolute value of B. Signed integer division satisfies the identity: <br /></p>

<MarginText>7</MarginText>
<CodeBlock language="ada">
(-A)/B = -(A/B) = A/(-B){"\n"}

</CodeBlock>
<MarginText>8/3</MarginText>
<MarginInfo items={["AI05-0260-1"]} />
<p>The signed integer modulus operator is defined such that the result of A <strong>mod</strong> B is either zero, or has the sign of B and an absolute value less than the absolute value of B; in addition, for some signed integer value N, this result satisfies the relation: <br /></p>

<MarginText>9</MarginText>
<CodeBlock language="ada">
A = B*N + (A mod B){"\n"}

</CodeBlock>
<MarginText>10</MarginText>
<p>The multiplying operators on modular types are defined in terms of the corresponding signed integer operators[, followed by a reduction modulo the modulus if the result is outside the base range of the type] [(which is only possible for the "*" operator)]. <br /></p>

<AnnotatedOnly>
<MarginText>10.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The above identity satisfied by signed integer division is not satisfied by modular division because of the difference in effect of negation. <br /></Admonition>
</AnnotatedOnly>

<MarginText>11</MarginText>
<p>Multiplication and division operators are predefined for every specific floating point type <em>T</em>: <br /></p>

<MarginText>12</MarginText>
<CodeBlock language="ada">
function "*"(Left, Right : T) return T{"\n"}function "/"(Left, Right : T) return T{"\n"}

</CodeBlock>
<MarginText>13</MarginText>

#### Name Resolution Rules

<MarginText>19.1/2</MarginText>
<MarginInfo items={["AI95-00364-01", "AI95-00420-01"]} />
<p>The above two fixed-fixed multiplying operators shall not be used in a context where the expected type for the result is itself <em>universal_fixed</em> [&mdash; the context has to identify some other numeric type to which the result is to be converted, either explicitly or implicitly]. Unless the predefined universal operator is identified using an expanded name with <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> denoting the package Standard, an explicit conversion is required on the result when using the above fixed-fixed multiplication operator if either operand is of a type having a user-defined primitive multiplication operator such that: <br /></p>

<MarginText>19.2/3</MarginText>
<MarginInfo items={["AI05-0020-1", "AI05-0209-1"]} />
<ul>
<li>it is declared immediately within the same declaration list as the type or any partial or incomplete view thereof; and<br /></li>
<MarginText>19.3/2</MarginText>
<li>both of its formal parameters are of a fixed-point type. <br /></li>
</ul>
<MarginText>19.4/2</MarginText>
<MarginInfo items={["AI95-00364-01", "AI95-00420-01"]} />
<p>A corresponding requirement applies to the universal fixed-fixed division operator.<br /></p>

<AnnotatedOnly>
<MarginText>19.a/2</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The <em>small</em> of <em>universal_fixed</em> is infinitesimal; no loss of precision is permitted. However, fixed-fixed division is impractical to implement when an exact result is required, and multiplication will sometimes result in unanticipated overflows in such circumstances, so we require an explicit conversion to be inserted in expressions like A * B * C if A, B, and C are each of some fixed point type.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>19.b/2</MarginText>
<Admonition type="aarm" aarm="note">
On the other hand, X := A * B; is permitted by this rule, even if X, A, and B are all of different fixed point types, since the expected type for the result of the multiplication is the type of X, which is necessarily not <em>universal_fixed</em>.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>19.c/2</MarginText>
<MarginInfo items={["AI95-00364-01", "AI95-00420-01"]} />
<Admonition type="aarm" aarm="note">
We have made these into Name Resolution rules to ensure that user-defined primitive fixed-fixed operators are not made unusable due to the presence of these universal fixed-fixed operators. But we do allow these operators to be used if prefixed by package Standard, so that they can be used in the definitions of user-defined operators. <br /></Admonition>
</AnnotatedOnly>

## 4.5.6  Highest Precedence Operators


#### Static Semantics

<MarginText>1</MarginText>
<p>The highest precedence unary operator <strong>abs</strong> (absolute value) is predefined for every specific numeric type <em>T</em>, with the following specification: <br /></p>

<MarginText>2</MarginText>
<CodeBlock language="ada">
function "abs"(Right : T) return T{"\n"}

</CodeBlock>
<MarginText>3</MarginText>
<p>The highest precedence unary operator <strong>not</strong> (logical negation) is predefined for every boolean type <em>T</em>, every modular type <em>T</em>, and for every one-dimensional array type <em>T</em> whose components are of a boolean type, with the following specification: <br /></p>

<MarginText>4</MarginText>
<CodeBlock language="ada">
function "not"(Right : T) return T{"\n"}

</CodeBlock>
<MarginText>5</MarginText>
<p>The result of the operator <strong>not</strong> for a modular type is defined as the difference between the high bound of the base range of the type and the value of the operand. [For a binary modulus, this corresponds to a bit-wise complement of the binary representation of the value of the operand.]<br /></p>

<MarginText>6</MarginText>
<p>The operator <strong>not</strong> that applies to a one-dimensional array of boolean components yields a one-dimensional boolean array with the same bounds; each component of the result is obtained by logical negation of the corresponding component of the operand (that is, the component that has the same index value). A check is made that each component of the result belongs to the component subtype; the exception Constraint_Error is raised if this check fails. <br /></p>

<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The check against the component subtype is per AI83-00535. <br /></Admonition>
</AnnotatedOnly>

<MarginText>7</MarginText>
<p>The highest precedence <em>exponentiation</em> operator ** is predefined for every specific integer type <em>T</em> with the following specification: <br /></p>

<MarginText>8</MarginText>
<CodeBlock language="ada">
function "**"(Left : T; Right : Natural) return T{"\n"}

</CodeBlock>
<MarginText>9</MarginText>
<p>Exponentiation is also predefined for every specific floating point type as well as <em>root_real</em>, with the following specification (where <em>T</em> is <em>root_real</em> or the floating point type): <br /></p>

<MarginText>10</MarginText>
<CodeBlock language="ada">
function "**"(Left : T; Right : Integer'Base) return T{"\n"}

</CodeBlock>
<MarginText>11/3</MarginText>
<MarginInfo items={["AI05-0088-1"]} />
<p>The right operand of an exponentiation is the <em>exponent</em>. The value of X**N with the value of the exponent N positive is the same as the value of X*X*...X (with N&ndash;1 multiplications) except that the multiplications are associated in an arbitrary order. With N equal to zero, the result is one. With the value of N negative [(only defined for a floating point operand)], the result is the reciprocal of the result using the absolute value of N as the exponent. <br /></p>

<AnnotatedOnly>
<MarginText>11.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The language does not specify the order of association of the multiplications inherent in an exponentiation. For a floating point type, the accuracy of the result might depend on the particular association order chosen. <br /></Admonition>
</AnnotatedOnly>


#### Implementation Permissions

<MarginText>12</MarginText>
<p>The implementation of exponentiation for the case of a negative exponent is allowed to raise Constraint_Error if the intermediate result of the repeated multiplications is outside the safe range of the type, even though the final result (after taking the reciprocal) would not be. (The best machine approximation to the final result in this case would generally be 0.0.) <br /></p>

<AnnotatedOnly>
<MarginText>13</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   As implied by the specification given above for exponentiation of an integer type, a check is made that the exponent is not negative. Constraint_Error is raised if this check fails. <br /></Admonition>
</AnnotatedOnly>

## 4.5.7  Conditional Expressions

<MarginText>1/3</MarginText>
<MarginInfo items={["AI05-0147-1", "AI05-0188-1", "AI05-0262-1"]} />
<p>A <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> selects for evaluation at most one of the enclosed <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s, depending on a decision among the alternatives. One kind of <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is the <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code>, which selects for evaluation a <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> depending on the value of one or more corresponding conditions. The other kind of <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is the <code><a href="../AA-4/AA-4.5#S0151">case_expression</a></code>, which selects for evaluation one of a number of alternative <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s; the chosen alternative is determined by the value of a <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>. <br /></p>


#### Language Design Principles

<AnnotatedOnly>
<MarginText>1.a/3</MarginText>
<MarginInfo items={["AI05-0188-1"]} />
<Admonition type="aarm" aarm="note">
As previously noted, there are two kinds of <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>, <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code>s and <code><a href="../AA-4/AA-4.5#S0151">case_expression</a></code>s. Whenever possible, we have written the rules in terms of <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>s to avoid duplication.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>1.b/3</MarginText>
<MarginInfo items={["AI05-0147-1"]} />
<Admonition type="aarm" aarm="note">
The rules for <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>s have been designed as much as possible to work similarly to a parenthesized expression. The intent is that as much as possible, wherever a parenthesized expression would be allowed, a <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> would be allowed, and it should work the same way. <br /></Admonition>
</AnnotatedOnly>

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>7.d/3</MarginText>
<Admonition type="aarm" aarm="note">
The following procedure calls are syntactically legal; the first uses the above rule to eliminate the redundant parentheses found in the second:<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.e/3</MarginText>
<CodeBlock language="ada">
P(if X then Y else Z);{"\n"}P((if X then Y else Z)); -- redundant parentheses{"\n"}
<MarginText>7.f/3</MarginText>
P((if X then Y else Z), Some_Other_Param);{"\n"}P(Some_Other_Param, (if X then Y else Z));{"\n"}P(Formal ={">"} (if X then Y else Z));{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>7.g/3</MarginText>
<Admonition type="aarm" aarm="note">
whereas the following are illegal:<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.h/3</MarginText>
<CodeBlock language="ada">
P(if X then Y else Z, Some_Other_Param);{"\n"}P(Some_Other_Param, if X then Y else Z);{"\n"}P(Formal ={">"} if X then Y else Z);{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>7.i/3</MarginText>
<Admonition type="aarm" aarm="note">
because in these latter cases, the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is not immediately surrounded by parentheses (which means on both sides!).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.j/3</MarginText>
<Admonition type="aarm" aarm="note">
The English-language rule applies in all places that could surround an expression with parentheses, including pragma arguments, type conversion and qualified expression operands, and array index expressions.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.k/3</MarginText>
<MarginInfo items={["AI05-0147-1"]} />
<Admonition type="aarm" aarm="note">
This English-language rule could have been implemented instead by adding a nonterminal <code>expression_within_parentheses</code>, which would consist of <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s and <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>s. Then, that could be used in all of the syntax which could consist of parens directly around an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>. We did not do that because of the large amount of change required. A complete grammar is given in . <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.l/3</MarginText>
<MarginInfo items={["AI05-0147-1"]} />
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>Implementers are cautioned to consider error detection when implementing the syntax for <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>s. An <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code> and an <code><a href="../AA-5/AA-5.3#S0175">if_statement</a></code> are very similar syntactically, (as are a <code><a href="../AA-4/AA-4.5#S0151">case_expression</a></code> and a <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code>) and simple mistakes can appear to change one into the other, potentially causing errors to be moved far away from their actual location. The absence of <strong>end if</strong> to terminate an <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code> (and <strong>end case</strong> for a <code><a href="../AA-4/AA-4.5#S0151">case_expression</a></code>) also may make error handling harder. <br /></Admonition>
</AnnotatedOnly>


#### Name Resolution Rules

<MarginText>8/3</MarginText>
<MarginInfo items={["AI05-0147-1"]} />
<p>If a <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is expected to be of a type <em>T</em>, then each <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is expected to be of type <em>T</em>. Similarly, if a <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is expected to be of some class of types, then each <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is subject to the same expectation. If a <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> shall resolve to be of a type <em>T</em>, then each <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> shall resolve to be of type <em>T</em>.<br /></p>

<MarginText>9/3</MarginText>
<MarginInfo items={["AI05-0147-1"]} />
<p>The possible types of a <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> are further determined as follows:<br /></p>

<MarginText>10/3</MarginText>
<ul>
<li>If the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is the operand of a type conversion, the type of the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is the target type of the conversion; otherwise,<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>10.a/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This rule distributes an enclosing type conversion to the <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s. This means that <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.b/3</MarginText>
<CodeBlock language="ada">
T(if C then A else B){"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>10.c/3</MarginText>
<Admonition type="aarm" aarm="note">
has the same semantics as <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.d/3</MarginText>
<CodeBlock language="ada">
(if C then T(A) else T(B)){"\n"}

</CodeBlock>
</AnnotatedOnly>
<MarginText>11/3</MarginText>
<ul>
<li>If all of the <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s are of the same type, the type of the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is that type; otherwise,<br /></li>
<MarginText>12/3</MarginText>
<li>If a <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is of an elementary type, the type of the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> shall be covered by that type; otherwise,<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>12.a/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This rule supports the use of numeric literals and universal expressions within a <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>13/3</MarginText>
<ul>
<li>If the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is expected to be of type <em>T</em> or shall resolve to type <em>T</em>, then the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is of type <em>T</em>.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>13.a/3</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>If the type of the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> cannot be determined by one of these rules, then Name Resolution has failed for that expression, even if the <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s would resolve individually. <br /></Admonition>
</AnnotatedOnly>

<MarginText>14/3</MarginText>
<MarginInfo items={["AI05-0147-1"]} />
<p>A <code><a href="../AA-4/AA-4.5#S0150">condition</a></code> is expected to be of any boolean type. <br /></p>

<MarginText>15/3</MarginText>
<MarginInfo items={["AI05-0188-1"]} />
<p>The expected type for the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> and the <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>s are as for case statements (see <a href="../AA-5/AA-5.4">5.4</a>). <br /></p>


#### Legality Rules

<MarginText>16/3</MarginText>
<MarginInfo items={["AI05-0147-1", "AI05-0188-1"]} />
<p>All of the <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s shall be convertible (see <a href="../AA-4/AA-4.6">4.6</a>) to the type of the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>.<br /></p>

<MarginText>17/3</MarginText>
<MarginInfo items={["AI05-0147-1", "AI05-0188-1", "AI05-0269-1"]} />
<p>If the expected type of a <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is a specific tagged type, all of the <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s of the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> shall be dynamically tagged, or none shall be dynamically tagged. In this case, the <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code> is dynamically tagged if all of the <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s are dynamically tagged, is tag-indeterminate if all of the <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s are tag-indeterminate, and is statically tagged otherwise.<br /></p>

<MarginText>18/3</MarginText>
<MarginInfo items={["AI05-0147-1", "AI05-0262-1"]} />
<p>If there is no <strong>else</strong> <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, the <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code> shall be of a boolean type.<br /></p>

<MarginText>19/3</MarginText>
<MarginInfo items={["AI05-0188-1", "AI05-0269-1"]} />
<p>All Legality Rules that apply to the <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>s of a <code><a href="../AA-5/AA-5.4#S0176">case_statement</a></code> (see <a href="../AA-5/AA-5.4">5.4</a>) also apply to the <code><a href="../AA-3/AA-3.8#S0074">discrete_choice</a></code>s of a <code><a href="../AA-4/AA-4.5#S0151">case_expression</a></code> except within an instance of a generic unit. <br /></p>

<AnnotatedOnly>
<MarginText>19.a/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The exemption for a case expression that occurs in an instance allows the following example: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>19.b/3</MarginText>
<CodeBlock language="ada">
generic{"\n"}   with function Int_Func return Integer;{"\n"}package G is{"\n"}   X : Float := (case Int_Func is{"\n"}                  when Integer'First .. -1 ={">"} -1.0,{"\n"}                  when 0 ={">"} 0.0,{"\n"}                  when Positive ={">"} 1.0);{"\n"}end G;{"\n"}
<MarginText>19.c/3</MarginText>
function Nat_Func return Natural is (123);{"\n"}
<MarginText>19.d/3</MarginText>
package I is new G (Int_Func ={">"} Nat_Func); -- Legal{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>19.e/3</MarginText>
<Admonition type="aarm" aarm="note">
Note that the Legality Rules still apply in the generic unit itself; they are just not enforced in an instance of the unit. <br /></Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>20/3</MarginText>
<MarginInfo items={["AI05-0147-1", "AI05-0188-1"]} />
<p>For the evaluation of an <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code>, the <code><a href="../AA-4/AA-4.5#S0150">condition</a></code> specified after <strong>if</strong>, and any <code><a href="../AA-4/AA-4.5#S0150">condition</a></code>s specified after <strong>elsif</strong>, are evaluated in succession (treating a final <strong>else</strong> as <strong>elsif</strong> True <strong>then</strong>), until one evaluates to True or all <code><a href="../AA-4/AA-4.5#S0150">condition</a></code>s are evaluated and yield False. If a <code><a href="../AA-4/AA-4.5#S0150">condition</a></code> evaluates to True, the associated <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is evaluated, converted to the type of the <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code>, and the resulting value is the value of the <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code>. Otherwise (when there is no <strong>else</strong> clause), the value of the <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code> is True.<br /></p>

<AnnotatedOnly>
<MarginText>20.a/3</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong><strong>Else</strong> is required unless the <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code> has a boolean type, so the last sentence can only apply to <code><a href="../AA-4/AA-4.5#S0149">if_expression</a></code>s with a boolean type. <br /></Admonition>
</AnnotatedOnly>

<MarginText>21/3</MarginText>
<MarginInfo items={["AI05-0188-1"]} />
<p>For the evaluation of a <code><a href="../AA-4/AA-4.5#S0151">case_expression</a></code>, the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is first evaluated. If the value of the <em>selecting_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is covered by the <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code> of some <code><a href="../AA-4/AA-4.5#S0152">case_expression_alternative</a></code>, then the <em>dependent_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the <code><a href="../AA-4/AA-4.5#S0152">case_expression_alternative</a></code> is evaluated, converted to the type of the <code><a href="../AA-4/AA-4.5#S0151">case_expression</a></code>, and the resulting value is the value of the <code><a href="../AA-4/AA-4.5#S0151">case_expression</a></code>. Otherwise (the value is not covered by any <code><a href="../AA-3/AA-3.8#S0073">discrete_choice_list</a></code>, perhaps due to being outside the base range), Constraint_Error is raised. <br /></p>

## 4.5.8  Quantified Expressions

<MarginText>0.1/4</MarginText>
<MarginInfo items={["AI12-0158-1"]} />
<p>Quantified expressions provide a way to write universally and existentially quantified predicates over containers and arrays. <br /></p>


#### Syntax

<MarginText>1/3</MarginText>
<MarginInfo items={["AI05-0176-1"]} />

<CodeBlock>
<code>quantified_expression</code><a id="S0153"></a><code> ::= </code><br />    <strong>for</strong> <code><a href="../AA-4/AA-4.5#S0154">quantifier</a></code> <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> ={">"} <code><a href="../AA-4/AA-4.5#S0155">predicate</a></code><br />  | <strong>for</strong> <code><a href="../AA-4/AA-4.5#S0154">quantifier</a></code> <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code> ={">"} <code><a href="../AA-4/AA-4.5#S0155">predicate</a></code><br />

</CodeBlock>
<MarginText>2/3</MarginText>

<CodeBlock>
<code>quantifier</code><a id="S0154"></a><code> ::= </code><strong>all</strong> | <strong>some</strong><br />

</CodeBlock>
<MarginText>3/3</MarginText>

<CodeBlock>
<code>predicate</code><a id="S0155"></a><code> ::= </code><em>boolean_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code><br />

</CodeBlock>
<MarginText>4/3</MarginText>
<MarginInfo items={["AI05-0176-1"]} />
<p class="Indented2">Wherever the Syntax Rules allow an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, a <code><a href="../AA-4/AA-4.5#S0153">quantified_expression</a></code> may be used in place of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, so long as it is immediately surrounded by parentheses.<br /></p>

<AnnotatedOnly>
<MarginText>4.a/3</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The syntactic category <code><a href="../AA-4/AA-4.5#S0153">quantified_expression</a></code> appears only as a <code><a href="../AA-4/AA-4.4#S0141">primary</a></code> that is parenthesized. The above rule allows it to additionally be used in other contexts where it would be directly surrounded by parentheses. This is the same rule that is used for <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>s; see <a href="../AA-4/AA-4.5#Subclause_4.5.7">4.5.7</a> for a detailed discussion of the meaning and effects of this rule. <br /></Admonition>
</AnnotatedOnly>


#### Name Resolution Rules

<MarginText>5/3</MarginText>
<MarginInfo items={["AI05-0176-1"]} />
<p>The expected type of a <code><a href="../AA-4/AA-4.5#S0153">quantified_expression</a></code> is any Boolean type. The <code><a href="../AA-4/AA-4.5#S0155">predicate</a></code> in a <code><a href="../AA-4/AA-4.5#S0153">quantified_expression</a></code> is expected to be of the same type. <br /></p>


#### Dynamic Semantics

<MarginText>6/5</MarginText>
<MarginInfo items={["AI05-0176-1", "AI12-0158-1", "AI12-0327-1"]} />
<p>For the evaluation of a <code><a href="../AA-4/AA-4.5#S0153">quantified_expression</a></code>, the <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> or <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code> is first elaborated. The evaluation of a <code><a href="../AA-4/AA-4.5#S0153">quantified_expression</a></code> then performs an iteration, and evaluates the <code><a href="../AA-4/AA-4.5#S0155">predicate</a></code> for each value conditionally produced by the iteration  (see <a href="../AA-5/AA-5.5">5.5</a> and  <a href="../AA-5/AA-5.5#Subclause_5.5.2">5.5.2</a>).<br /></p>

<AnnotatedOnly>
<MarginText>6.a/5</MarginText>
<MarginInfo items={["AI12-0327-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>The order of evaluation of the predicates is that in which the values are produced, as specified in <a href="../AA-5/AA-5.5">5.5</a> or <a href="../AA-5/AA-5.5#Subclause_5.5.2">5.5.2</a>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>7/3</MarginText>
<MarginInfo items={["AI05-0176-1"]} />
<p>The value of the <code><a href="../AA-4/AA-4.5#S0153">quantified_expression</a></code> is determined as follows:<br /></p>

<MarginText>8/4</MarginText>
<MarginInfo items={["AI12-0158-1"]} />
<ul>
<li>If the <code><a href="../AA-4/AA-4.5#S0154">quantifier</a></code> is <strong>all</strong>, the expression is False if the evaluation of any <code><a href="../AA-4/AA-4.5#S0155">predicate</a></code> yields False; evaluation of the <code><a href="../AA-4/AA-4.5#S0153">quantified_expression</a></code> stops at that point. Otherwise (every predicate has been evaluated and yielded True), the expression is True. Any exception raised by evaluation of the <code><a href="../AA-4/AA-4.5#S0155">predicate</a></code> is propagated.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>8.a/3</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is True if the domain contains no values. <br /></Admonition>
</AnnotatedOnly>

<MarginText>9/4</MarginText>
<MarginInfo items={["AI12-0158-1"]} />
<ul>
<li>If the <code><a href="../AA-4/AA-4.5#S0154">quantifier</a></code> is <strong>some</strong>, the expression is True if the evaluation of any <code><a href="../AA-4/AA-4.5#S0155">predicate</a></code> yields True; evaluation of the <code><a href="../AA-4/AA-4.5#S0153">quantified_expression</a></code> stops at that point. Otherwise (every predicate has been evaluated and yielded False), the expression is False. Any exception raised by evaluation of the <code><a href="../AA-4/AA-4.5#S0155">predicate</a></code> is propagated.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>9.a/3</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is False if the domain contains no values. <br /></Admonition>
</AnnotatedOnly>


#### Examples

<MarginText>10/5</MarginText>
<MarginInfo items={["AI05-0176-1", "AI12-0429-1"]} />
<p><em>Example of a quantified expression as a postcondition for a sorting routine on an array A with an index subtype T:</em> <br /></p>

<MarginText>11/3</MarginText>
<CodeBlock language="ada">

## 4.5.9  Declare Expressions

<MarginText>1/5</MarginText>
<MarginInfo items={["AI12-0236-1"]} />
<p>Declare expressions provide a way to declare local constants and object renamings in an expression context. <br /></p>


#### Syntax

<MarginText>2/5</MarginText>
<MarginInfo items={["AI12-0236-1"]} />

<CodeBlock>
<code>declare_expression</code><a id="S0156"></a><code> ::= </code><br />    <strong>declare</strong> {"{"}<code><a href="../AA-4/AA-4.5#S0157">declare_item</a></code>{"}"}<br />    <strong>begin</strong> <em>body_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code><br />

</CodeBlock>
<MarginText>3/5</MarginText>
<MarginInfo items={["AI12-0236-1"]} />

<CodeBlock>
<code>declare_item</code><a id="S0157"></a><code> ::= </code><code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> | <code><a href="../AA-8/AA-8.5#S0239">object_renaming_declaration</a></code><br />

</CodeBlock>
<AnnotatedOnly>
<MarginText>3.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We allow (<strong>declare begin</strong> <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>) with no <code><a href="../AA-4/AA-4.5#S0157">declare_item</a></code>s, for uniformity with block statements, which also allow a pointless <strong>declare</strong>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>4/5</MarginText>
<MarginInfo items={["AI12-0236-1"]} />
<p class="Indented2">Wherever the Syntax Rules allow an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, a <code><a href="../AA-4/AA-4.5#S0156">declare_expression</a></code> may be used in place of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, so long as it is immediately surrounded by parentheses.<br /></p>

<AnnotatedOnly>
<MarginText>4.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The syntactic category <code><a href="../AA-4/AA-4.5#S0156">declare_expression</a></code> appears only as a <code><a href="../AA-4/AA-4.4#S0141">primary</a></code> that is parenthesized. The above rule allows it to additionally be used in other contexts where it would be directly surrounded by parentheses. This is the same rule that is used for <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>s; see <a href="../AA-4/AA-4.5#Subclause_4.5.7">4.5.7</a> for a detailed discussion of the meaning and effects of this rule. <br /></Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>5/5</MarginText>
<MarginInfo items={["AI12-0236-1"]} />
<p>A <code><a href="../AA-4/AA-4.5#S0157">declare_item</a></code> that is an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> shall declare a constant of a nonlimited type.<br /></p>

<AnnotatedOnly>
<MarginText>5.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We disallow limited objects to avoid the horror of task waiting in the middle of an expression. The solution used for controlled objects (wait until the entire expression is finished) isn't appropriate for tasks. This restriction also eliminates build-in-place complexities and the need to activate tasks found in a <code><a href="../AA-4/AA-4.5#S0157">declare_item</a></code>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>6/5</MarginText>
<MarginInfo items={["AI12-0236-1", "AI12-0317-1"]} />
<p>A <code><a href="../AA-4/AA-4.5#S0157">declare_item</a></code> that is an <code><a href="../AA-8/AA-8.5#S0239">object_renaming_declaration</a></code> (see <a href="../AA-8/AA-8.5#Subclause_8.5.1">8.5.1</a>) shall not rename an object of a limited type if any operative constituent of the <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> is a value conversion or is newly constructed (see <a href="../AA-4/AA-4.4">4.4</a>).<br /></p>

<AnnotatedOnly>
<MarginText>6.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We disallow renaming limited temporary objects (like the result of a function call) for the same reasons as for stand-alone objects. We do allow renaming existing limited objects because these don't cause any problems. Note that one cannot directly rename an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, parenthesized expression, <code><a href="../AA-4/AA-4.5#S0148">conditional_expression</a></code>, or <code><a href="../AA-4/AA-4.5#S0156">declare_expression</a></code> (these are not <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s), but any of these can appear inside of a <code><a href="../AA-4/AA-4.7#S0163">qualified_expression</a></code> or <code><a href="../AA-4/AA-4.6#S0162">type_conversion</a></code> (which are <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s and thus can be renamed). <br /></Admonition>
</AnnotatedOnly>

<MarginText>7/5</MarginText>
<MarginInfo items={["AI12-0236-1"]} />
<p>The following are not allowed within a <code><a href="../AA-4/AA-4.5#S0156">declare_expression</a></code>: a declaration containing the reserved word <strong>aliased</strong>; the <code><a href="../AA-4/AA-4.1#S0101">attribute_designator</a></code> Access or Unchecked_Access; or an anonymous access type.<br /></p>

<AnnotatedOnly>
<MarginText>7.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We do not want to define accessibility rules for <code><a href="../AA-4/AA-4.5#S0157">declare_item</a></code>s, as nested <code><a href="../AA-4/AA-4.5#S0156">declare_expression</a></code>s cause complexities or usage warts. We want to keep this feature simple to use, understand, and implement. Thus, we do not allow any of the features that would require accessibility rules. <br /></Admonition>
</AnnotatedOnly>


#### Name Resolution Rules

<MarginText>8/5</MarginText>
<MarginInfo items={["AI12-0236-1"]} />
<p>If a <code><a href="../AA-4/AA-4.5#S0156">declare_expression</a></code> is expected to be of a type <em>T</em>, then the <em>body_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is expected to be of type <em>T</em>. Similarly, if a <code><a href="../AA-4/AA-4.5#S0156">declare_expression</a></code> is expected to be of some class of types, then the <em>body_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is subject to the same expectation. If a <code><a href="../AA-4/AA-4.5#S0156">declare_expression</a></code> shall resolve to be of a type <em>T</em>, then the <em>body_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> shall resolve to be of type <em>T</em>.<br /></p>

<MarginText>9/5</MarginText>
<MarginInfo items={["AI12-0236-1"]} />
<p>The type of a <code><a href="../AA-4/AA-4.5#S0156">declare_expression</a></code> is the type of the <em>body_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>. <br /></p>


#### Dynamic Semantics

<MarginText>10/5</MarginText>
<MarginInfo items={["AI12-0236-1"]} />
<p>For the evaluation of a <code><a href="../AA-4/AA-4.5#S0156">declare_expression</a></code>, the <code><a href="../AA-4/AA-4.5#S0157">declare_item</a></code>s are elaborated in order, and then the <em>body_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is evaluated. The value of the <code><a href="../AA-4/AA-4.5#S0156">declare_expression</a></code> is that of the <em>body_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>. <br /></p>


#### Examples

<MarginText>11/5</MarginText>
<MarginInfo items={["AI12-0236-1", "AI12-0429-1"]} />
<p><em>Example of use of a declare expression as a replacement postcondition for Ada.Containers.Vectors."&" (see <a href="../AA-A/AA-A.18#Subclause_A.18.2">A.18.2</a>):</em><br /></p>

<MarginText>12/5</MarginText>
<MarginInfo items={["AI12-0236-1"]} />
<CodeBlock language="ada">
with Post ={">"}{"\n"}   (declare{"\n"}      Result renames Vectors."&"'Result;{"\n"}      Length : constant Count_Type := Left.Length + Right.Length;{"\n"}    begin{"\n"}      Result.Length = Length and then{"\n"}      not Tampering_With_Elements_Prohibited (Result) and then{"\n"}      not Tampering_With_Cursors_Prohibited (Result) and then{"\n"}      Result.Capacity {">"}= Length){"\n"}

</CodeBlock>

## 4.5.10  Reduction Expressions

<MarginText>1/5</MarginText>
<MarginInfo items={["AI12-0242-1", "AI12-0262-1"]} />
<p>Reduction expressions provide a way to map or transform a collection of values into a new set of values, and then summarize the values produced by applying an operation to reduce the set to a single value result. A reduction expression is represented as an <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code> of the reduction attributes Reduce or Parallel_Reduce.<br /></p>

<AnnotatedOnly>
<MarginText>1.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>reduction expression</strong> &mdash; expression that defines how to map or transform a collection of values into a new set of values, and then summarize the values by applying an operation to reduce the set to a single value<br /></Admonition>
</AnnotatedOnly>


#### Syntax

<MarginText>2/5</MarginText>
<MarginInfo items={["AI12-0242-1", "AI12-0262-1"]} />

<CodeBlock>
<code>reduction_attribute_reference</code><a id="S0158"></a><code> ::= </code><br />    <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code>'<code><a href="../AA-4/AA-4.5#S0160">reduction_attribute_designator</a></code><br />  | <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code>'<code><a href="../AA-4/AA-4.5#S0160">reduction_attribute_designator</a></code><br />

</CodeBlock>
<MarginText>3/5</MarginText>
<MarginInfo items={["AI12-0262-1", "AI12-0355-2"]} />

<CodeBlock>
<code>value_sequence</code><a id="S0159"></a><code> ::= </code><br />    '[' [<strong>parallel</strong>[(<code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code>)] [<code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>]]<br />        <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code> ']'<br />

</CodeBlock>
<MarginText>4/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />

<CodeBlock>
<code>reduction_attribute_designator</code><a id="S0160"></a><code> ::= </code><br />    <em>reduction_</em><code><a href="../AA-2/AA-2.3#S0002">identifier</a></code>(<code><a href="../AA-4/AA-4.5#S0161">reduction_specification</a></code>)<br />

</CodeBlock>
<MarginText>5/5</MarginText>
<MarginInfo items={["AI12-0262-1", "AI12-0348-1"]} />

<CodeBlock>
<code>reduction_specification</code><a id="S0161"></a><code> ::= </code><em>reducer_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>, <em>initial_value_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code><br />

</CodeBlock>
<MarginText>6/5</MarginText>
<MarginInfo items={["AI12-0250-1", "AI12-0262-1"]} />
<p class="Indented2">The <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code> of a <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code> shall not have a <em>key_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, nor shall it have a <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> that has the reserved word <strong>reverse</strong>.<br /></p>

<AnnotatedOnly>
<MarginText>6.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The intent is that the syntax matches as closely as possible array or container aggregate notation. Syntax that matches a <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> with the <strong>reverse</strong> reserved word would not be permitted in an array aggregate, so we disallow that here. <br /></Admonition>
</AnnotatedOnly>

<MarginText>7/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<p class="Indented2">The <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code>, if any, of a <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code> shall be an <em>integer_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code>.<br /></p>


#### Name Resolution Rules

<MarginText>8/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<p>The expected type for a <code><a href="../AA-4/AA-4.5#S0158">reduction_attribute_reference</a></code> shall be a single nonlimited type.<br /></p>

<MarginText>9/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<p>In the remainder of this subclause, we will refer to nonlimited subtypes <em>Value_Type</em> and <em>Accum_Type</em> of a <code><a href="../AA-4/AA-4.5#S0158">reduction_attribute_reference</a></code>. These subtypes and interpretations of the <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s and <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>s of a <code><a href="../AA-4/AA-4.5#S0158">reduction_attribute_reference</a></code> are determined by the following rules:<br /></p>

<AnnotatedOnly>
<MarginText>9.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong><em>Accum_Type</em> represents the result of the reduction (the <em>accumulator</em> type), and <em>Value_Type</em> represents the type of the input values to the reduction. <br /></Admonition>
</AnnotatedOnly>

<MarginText>10/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<ul>
<li><em>Accum_Type</em> is a subtype of the expected type of the <code><a href="../AA-4/AA-4.5#S0158">reduction_attribute_reference</a></code>.<br /></li>
<MarginText>11/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<li>A <em>reducer subprogram</em> is subtype conformant with one of the following specifications:<br /></li>
</ul>
<MarginText>12/5</MarginText>
<CodeBlock language="ada">
   function Reducer(Accumulator : Accum_Type;{"\n"}                    Value : Value_Type) return Accum_Type;{"\n"}
<MarginText>13/5</MarginText>
   procedure Reducer(Accumulator : in out Accum_Type;{"\n"}                     Value : in Value_Type);{"\n"}

</CodeBlock>
<MarginText>14/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<ul>
<li>The <em>reducer_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> of a <code><a href="../AA-4/AA-4.5#S0161">reduction_specification</a></code> denotes a reducer subprogram.<br /></li>
<MarginText>15/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<li>The expected type of an <em>initial_value_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a <code><a href="../AA-4/AA-4.5#S0161">reduction_specification</a></code> is that of subtype <em>Accum_Type</em>.<br /></li>
<MarginText>16/5</MarginText>
<MarginInfo items={["AI12-0250-1", "AI12-0262-1"]} />
<li>The expected type of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code> of a <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code> is that of subtype <em>Value_Type</em>. <br /></li>
</ul>

#### Legality Rules

<MarginText>17/5</MarginText>
<MarginInfo items={["AI12-0262-1", "AI12-0348-1"]} />
<p>If the <code><a href="../AA-4/AA-4.5#S0158">reduction_attribute_reference</a></code> has a <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code> with the reserved word <strong>parallel</strong>, the subtypes <em>Accum_Type</em> and <em>Value_Type</em> shall statically match.<br /></p>

<MarginText>18/5</MarginText>
<MarginInfo items={["AI12-0262-1", "AI12-0348-1"]} />
<p>If the <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> of a <code><a href="../AA-4/AA-4.5#S0160">reduction_attribute_designator</a></code> is Parallel_Reduce, the subtypes <em>Accum_Type</em> and <em>Value_Type</em> shall statically match.<br /></p>

<AnnotatedOnly>
<MarginText>18.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>For a <code><a href="../AA-4/AA-4.5#S0158">reduction_attribute_reference</a></code> with a <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code> that does not have the reserved word <strong>parallel</strong> or has a <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> and the <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> of the <code><a href="../AA-4/AA-4.5#S0160">reduction_attribute_designator</a></code> is Reduce, the subtypes <em>Accum_Type</em> and <em>Value_Type</em> can be different because only one logical thread of control is presumed so there is no need to combine multiple results. <br /></Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>19/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<p>A <code><a href="../AA-4/AA-4.5#S0158">reduction_attribute_reference</a></code> denotes a value, with its nominal subtype being the subtype of the first parameter of the subprogram denoted by the <em>reducer_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>. <br /></p>


#### Dynamic Semantics

<MarginText>20/5</MarginText>
<MarginInfo items={["AI12-0250-1", "AI12-0262-1", "AI12-0327-1", "AI12-0355-2"]} />
<p>For the evaluation of a <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code>, the <code><a href="../AA-4/AA-4.3#S0131">iterated_element_association</a></code>, the <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code>, and the <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>, if any, are elaborated in an arbitrary order. Next an iteration is performed, and for each value conditionally produced by the iteration (see <a href="../AA-5/AA-5.5">5.5</a> and <a href="../AA-5/AA-5.5#Subclause_5.5.2">5.5.2</a>), the associated <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is evaluated with the loop parameter having this value, which produces a result that is converted to Value_Type and is used to define the next value in the sequence.<br /></p>

<MarginText>21/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<p>If the <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code> does not have the reserved word <strong>parallel</strong>, it is produced as a single sequence of values by a single logical thread of control. If the reserved word <strong>parallel</strong> is present in the <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code>, the enclosing <code><a href="../AA-4/AA-4.5#S0158">reduction_attribute_reference</a></code> is a parallel construct, and the sequence of values is generated by a parallel iteration (as defined in <a href="../AA-5/AA-5.5">5.5</a>, <a href="../AA-5/AA-5.5#Subclause_5.5.1">5.5.1</a>, and <a href="../AA-5/AA-5.5#Subclause_5.5.2">5.5.2</a>), as a set of non-empty, non-overlapping contiguous chunks (<em>subsequences</em>) with one logical thread of control (see Clause <a href="../AA-9/">9</a>) associated with each subsequence. If there is a <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code>, it determines the maximum number of chunks, as defined in <a href="../AA-5/AA-5.5">5.5</a>; otherwise the maximum number of chunks is implementation defined.<br /></p>

<AnnotatedOnly>
<MarginText>21.a/5</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>The maximum number of chunks for a parallel reduction expression without a <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code>.<br /></Admonition>
</AnnotatedOnly>

<MarginText>22/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<p>For a <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code> V, the following attribute is defined:<br /></p>

<MarginText>23/5</MarginText>
<dt><br/>V'Reduce(Reducer, Initial_Value)</dt>
<MarginInfo items={["AI12-0262-1", "AI12-0348-1"]} />
<dl>
<dd>This attribute represents a <em>reduction expression</em>, and is in the form of a <code><a href="../AA-4/AA-4.5#S0158">reduction_attribute_reference</a></code>. <br /></dd>
<MarginText>24/5</MarginText>
<MarginInfo items={["AI12-0262-1", "AI12-0348-1"]} />
<dd>The evaluation of a use of this attribute begins by evaluating the parts of the <code><a href="../AA-4/AA-4.5#S0160">reduction_attribute_designator</a></code> (the <em>reducer_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> Reducer and the <em>initial_value_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> Initial_Value), in an arbitrary order. It then initializes the <em>accumulator</em> of the reduction expression to the value of the <em>initial_value_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> (the <em>initial value</em>).  The <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code> V is then evaluated.<br /></dd>
<MarginText>25/5</MarginText>
<MarginInfo items={["AI12-0262-1", "AI12-0348-1"]} />
<dd>If the <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code> does not have the reserved word <strong>parallel</strong>, each value of the <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code> is passed, in order, as the second (Value) parameter to a call on Reducer, with the first (Accumulator) parameter being the prior value of the accumulator, saving the result as the new value of the accumulator. The reduction expression yields the final value of the accumulator.<br /></dd>
<MarginText>26/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<dd>If the reserved word <strong>parallel</strong> is present in a <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code>, then the (parallel) reduction expression is a parallel construct and the sequence has been partitioned into one or more subsequences (see above) each with its own separate logical thread of control.<br /></dd>
<MarginText>27/5</MarginText>
<MarginInfo items={["AI12-0262-1", "AI12-0348-1"]} />
<dd>Each logical thread of control creates a local accumulator for processing its subsequence. The accumulator for a subsequence is initialized to the first value of the subsequence, and calls on Reducer start with the second value of the subsequence (if any). The result for the subsequence is the final value of its local accumulator.<br /></dd>
<MarginText>28/5</MarginText>
<MarginInfo items={["AI12-0262-1", "AI12-0348-1"]} />
<dd>After all logical threads of control of a parallel reduction expression have completed, Reducer is called for each subsequence, in the original sequence order, passing the local accumulator for that subsequence as the second (Value) parameter, and the overall accumulator [(initialized above to the initial value)] as the first (Accumulator) parameter, with the result saved back in the overall accumulator. The parallel reduction expression yields the final value of the overall accumulator.<br /></dd>
<MarginText>29/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<dd>If the evaluation of the <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code> yields an empty sequence of values, the reduction expression yields the initial value.<br /></dd>
<MarginText>30/5</MarginText>
<MarginInfo items={["AI12-0262-1", "AI12-0348-1"]} />
<dd>If an exception is propagated by one of the calls on Reducer, that exception is propagated from the reduction expression. If different exceptions are propagated in different logical threads of control, one is chosen arbitrarily to be propagated from the reduction expression as a whole.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>30.a/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>For a <code><a href="../AA-4/AA-4.5#S0158">reduction_attribute_reference</a></code> that has a <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code> without the reserved word <strong>parallel</strong> or a <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> where the <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> of the <code><a href="../AA-4/AA-4.5#S0160">reduction_attribute_designator</a></code> is Reduce (see below), generally the compiler can still choose to execute the reduction in parallel, presuming doing so would not change the results. However sequential execution is necessary if the subtypes of the parameters of Reducer do not statically match, since there is no subprogram identified in the construct that could be used for combining the results in parallel. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>30.b/5</MarginText>
<MarginInfo items={["AI12-0262-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>We say the calls to Reducer that combine the results of parallel execution are sequentially ordered in increasing order because certain reductions, such as vector concatenation, can be non-commutative (but still associative) operations. In order to return a deterministic result for parallel execution that is consistent with sequential execution, we need to specify an order for the iteration, and for the combination of results from the logical threads of control. It is also necessary that combining calls to Reducer are issued sequentially with respect to each other, which may require extra synchronization if the calls to Reducer are being executed by different logical threads of control. <br /></Admonition>
</AnnotatedOnly>

<MarginText>31/5</MarginText>
<MarginInfo items={["AI12-0242-1"]} />

<dd>X'Reduce is a reduction expression that yields a result equivalent to replacing the <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of the attribute with the <code><a href="../AA-4/AA-4.5#S0159">value_sequence</a></code>: <br /></dd>
</dl>

<CodeBlock language="ada">
[for Item of X ={">"} Item]{"\n"}

</CodeBlock>
<MarginText>33/5</MarginText>

<MarginInfo items={["AI12-0242-1", "AI12-0348-1"]} />

#### Bounded (Run-Time) Errors

<MarginText>34/5</MarginText>
<MarginInfo items={["AI12-0262-1", "AI12-0348-1"]} />
<p>For a parallel reduction expression, it is a bounded error if the reducer subprogram is not associative. That is, for any arbitrary values of subtype <em>Value_Type</em> <em>A</em>, <em>B</em>, <em>C</em> and a reducer function <em>R</em>, the result of <em>R</em> (<em>A</em>, <em>R</em> (<em>B</em>, <em>C</em>)) should produce a result equal to <em>R</em> (<em>R</em> (<em>A</em>, <em>B</em>), <em>C</em>)); it is a bounded error if <em>R</em> does not. The possible consequences are Program_Error, or a result that does not match the equivalent sequential reduction expression due to the order of calls on the reducer subprogram being unspecified in the overall reduction. Analogous rules apply in the case of a reduction procedure.<br /></p>

<AnnotatedOnly>
<MarginText>34.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>In a sequential reduction expression, the reducer subprogram is called in a left-to-right order, whereas in a parallel reduction expression, the reducer subprogram is called in an order that depends on the number of logical threads of control that execute the reduction and on the elements/components given to each chunk. If the reducer is associative, this order does not matter, but in other cases, very different results are possible. While one can specify the <em>maximum</em> number of chunks, the actual number of chunks is unspecified. Similarly, the split of elements has only weak requirements. Thus, to get a consistent and portable result, an associative reducer is required for a parallel reduction. We define this as a Bounded (Run-Time) Errors to provide a stern warning about the required nature of the reducer subprogram and to let compilers detect the problem when possible. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.b/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>In this rule, &ldquo;equal&rdquo; means semantically equal. We don't care if the bit patterns differ but that the results mean the same thing. In particular, if the primitive equal is user-defined, that equality would be the one used to determine if this rule is violated. <br /></Admonition>
</AnnotatedOnly>
