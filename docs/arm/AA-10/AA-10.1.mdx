<MarginText>1</MarginText>
<p>[ A <i>program unit</i> is either a package, a task unit, a protected unit, a protected entry, a generic unit, or an explicitly declared subprogram other than an enumeration literal. Certain kinds of program units can be separately compiled. Alternatively, they can appear physically nested within other program units. <br /></p>

<AnnotatedOnly>
<MarginText>1.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<b>Term entry: </b><b>program unit</b> &mdash; language construct that is a package, a task unit, a protected unit, a protected entry, a generic unit, or an explicitly declared subprogram other than an enumeration literal<br />Note: Certain kinds of program units can be separately compiled. Alternatively, they can appear physically nested within other program units.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>2</MarginText>
<p>The text of a program can be submitted to the compiler in one or more <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code>s. Each <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code> is a succession of <code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code>s. A <code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code> contains either the declaration, the body, or a renaming of a program unit.] The representation for a <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code> is implementation-defined. <br /></p>

<AnnotatedOnly>
<MarginText>2.a</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<b></b>The representation for a <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code>.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>2.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Some implementations might choose to make a <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code> be a source (text) file. Others might allow multiple source files to be automatically concatenated to form a single <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code>. Others still may represent the source in a nontextual form such as a parse tree. Note that the RM95 does not even define the concept of a source file.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>2.c</MarginText>
<Admonition type="aarm" aarm="note">
Note that a protected subprogram is a subprogram, and therefore a program unit. An instance of a generic unit is a program unit.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>2.d</MarginText>
<Admonition type="aarm" aarm="note">
A protected entry is a program unit, but protected entries cannot be separately compiled. <br />
</Admonition>
</AnnotatedOnly>

</Admonition>
</AnnotatedOnly>

<MarginText>3</MarginText>
<p>A library unit is a separately compiled program unit, and is a package, subprogram, or generic unit. Library units may have other (logically nested) library units as children, and may have other program units physically nested within them. A root library unit, together with its children and grandchildren and so on, form a <i>subsystem</i>. <br /></p>

<AnnotatedOnly>
<MarginText>3.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<b>Term entry: </b><b>library unit</b> &mdash; separately compiled program unit, which is a package, a subprogram, or a generic unit<br />Note: Library units can have other (logically nested) library units as children, and can have other program units physically nested within them. A root library unit, together with its children and grandchildren and so on, form a subsystem.<br />
</Admonition>
</AnnotatedOnly>


#### Implementation Permissions

<MarginText>4</MarginText>
<p>An implementation may impose implementation-defined restrictions on <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code>s that contain multiple <code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code>s. <br /></p>

<AnnotatedOnly>
<MarginText>4.a</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<b></b>Any restrictions on <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code>s that contain multiple <code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code>s.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.b</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>For example, an implementation might disallow a <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code> that contains two versions of the same compilation unit, or that contains the declarations for library packages P1 and P2, where P1 precedes P2 in the <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code> but P1 has a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> that mentions P2. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>

#### Wording Changes from Ada 83

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>4.c</MarginText>
<Admonition type="aarm" aarm="note">
The interactions between language issues and environmental issues are left open in Ada 95. The environment concept is new. In Ada 83, the concept of the program library, for example, appeared to be quite concrete, although the rules had no force, since implementations could get around them simply by defining various mappings from the concept of an Ada program library to whatever data structures were actually stored in support of separate compilation. Indeed, implementations were encouraged to do so.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.d</MarginText>
<Admonition type="aarm" aarm="note">
In RM83, it was unclear which was the official definition of &ldquo;program unit&rdquo;. Definitions appeared in RM83-5, 6, 7, and 9, but not 12. Placing it here seems logical, since a program unit is sort of a potential compilation unit. <br />
</Admonition>
</AnnotatedOnly>

## 10.1.1  Compilation Units - Library Units

<MarginText>1_10.1.1</MarginText>
<p>[A <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> is a compilation unit that is the declaration, body, or renaming of a library unit. Each library unit (except Standard) has a <i>parent unit</i>, which is a library package or generic library package.] A library unit is a <i>child</i> of its parent unit. The <i>root</i> library units are the children of the predefined library package Standard. <br /></p>

<MarginText>9_10.1.1</MarginText>
<p>A <i>library unit</i> is a program unit that is declared by a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>. When a program unit is a library unit, the prefix &ldquo;library&rdquo; is used to refer to it (or &ldquo;generic library&rdquo; if generic), as well as to its declaration and body, as in &ldquo;library procedure&rdquo;, &ldquo;library <code><a href="/docs/arm/AA-7/AA-7.2#S0231">package_body</a></code>&rdquo;, or &ldquo;generic library package&rdquo;. The term <i>compilation unit</i> is used to refer to a <code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code>. When the meaning is clear from context, the term is also used to refer to the <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> of a <code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code> or to the <code><a href="/docs/arm/AA-3/AA-3.11#S0090">proper_body</a></code> of a <code><a href="/docs/arm/AA-10/AA-10.1#S0302">subunit</a></code> [(that is, the <code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code> without the <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code> and the <b>separate</b> (<code><a href="/docs/arm/AA-10/AA-10.1#S0291">parent_unit_name</a></code>))]. <br /></p>

<MarginText>9.a_10.1.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>In this example: <br />
</Admonition>
</AnnotatedOnly>

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>9.c_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
the term &ldquo;compilation unit&rdquo; can refer to this text: &ldquo;<b>with</b> Ada.Text{"_"}IO; <b>package</b> P <b>is</b> ... <b>end</b> P;&rdquo; or to this text: &ldquo;<b>package</b> P <b>is</b> ... <b>end</b> P;&rdquo;. We use this shorthand because it corresponds to common usage.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.d_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
We like to use the word &ldquo;unit&rdquo; for declaration-plus-body things, and &ldquo;item&rdquo; for declaration or body separately (as in <code><a href="/docs/arm/AA-3/AA-3.11#S0087">declarative_item</a></code>). The terms &ldquo;<code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code>&rdquo;, &ldquo;compilation unit&rdquo;, and &ldquo;<code><a href="/docs/arm/AA-10/AA-10.1#S0302">subunit</a></code>&rdquo; are exceptions to this rule. We considered changing &ldquo;<code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code>&rdquo;, &ldquo;compilation unit&rdquo; to &ldquo;<code>compilation{"_"}item</code>&rdquo;, &ldquo;compilation item&rdquo;, respectively, but we decided not to. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>10_10.1.1</MarginText>
<p>The <i>parent declaration</i> of a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> (and of the library unit) is the declaration denoted by the <code><a href="/docs/arm/AA-10/AA-10.1#S0291">parent{"_"}unit{"_"}name</a></code>, if any, of the <code><a href="/docs/arm/AA-6/AA-6.1#S0201">defining{"_"}program{"_"}unit{"_"}name</a></code> of the <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>. If there is no <code><a href="/docs/arm/AA-10/AA-10.1#S0291">parent{"_"}unit{"_"}name</a></code>, the parent declaration is the declaration of Standard, the <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> is a <i>root</i> <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>, and the library unit (renaming) is a <i>root</i> library unit (renaming). The declaration and body of Standard itself have no parent declaration. The <i>parent unit</i> of a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> or library unit is the library unit declared by its parent declaration. <br /></p>

<AnnotatedOnly>
<MarginText>10.a_10.1.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>The declaration and body of Standard are presumed to exist from the beginning of time, as it were. There is no way to actually write them, since there is no syntactic way to indicate lack of a parent. An attempt to compile a package Standard would result in Standard.Standard. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.b_10.1.1</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>Library units (other than Standard) have &ldquo;parent declarations&rdquo; and &ldquo;parent units&rdquo;. Subunits have &ldquo;parent bodies&rdquo;. We didn't bother to define the other possibilities: parent body of a library unit, parent declaration of a subunit, parent unit of a subunit. These are not needed, and might get in the way of a correct definition of &ldquo;child&rdquo;. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>11_10.1.1</MarginText>
<p>[The children of a library unit occur immediately within the declarative region of the declaration of the library unit.] The <i>ancestors</i> of a library unit are itself, its parent, its parent's parent, and so on. [(Standard is an ancestor of every library unit.)] The <i>descendant</i> relation is the inverse of the ancestor relation. <br /></p>

<AnnotatedOnly>
<MarginText>11.a_10.1.1</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>These definitions are worded carefully to avoid defining subunits as children. Only library units can be children.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.b_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
We use the unadorned term &ldquo;ancestors&rdquo; here to concisely define both &ldquo;ancestor unit&rdquo; and &ldquo;ancestor declaration&rdquo;. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>12_10.1.1</MarginText>
<p>A <code><a href="/docs/arm/AA-10/AA-10.1#S0288">library_unit_declaration</a></code> or a <code><a href="/docs/arm/AA-10/AA-10.1#S0289">library{"_"}unit{"_"}renaming{"_"}declaration</a></code> is <i>private</i> if the declaration is immediately preceded by the reserved word <b>private</b>; it is otherwise <i>public</i>. A library unit is private or public according to its declaration. The <i>public descendants</i> of a library unit are the library unit itself, and the public descendants of its public children. Its other descendants are <i>private descendants</i>. <br /></p>

<AnnotatedOnly>
<MarginText>12.a_10.1.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>The first concept defined here is that a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> is either public or private (not in relation to anything else &mdash; it's just a property of the library unit). The second concept is that a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> is a public descendant or private descendant <i>of a given ancestor</i>. A given <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> can be a public descendant of one of its ancestors, but a private descendant of some other ancestor.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.b_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
A subprogram declared by a <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code> (as opposed to a <code><a href="/docs/arm/AA-6/AA-6.1#S0195">subprogram_declaration</a></code>) is always public, since the syntax rules disallow the reserved word <b>private</b> on a body.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.c_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
Note that a private library unit is a <i>public</i> descendant of itself, but a <i>private</i> descendant of its parent. This is because it is visible outside itself &mdash; its privateness means that it is not visible outside its parent.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.d_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
Private children of Standard are legal, and follow the normal rules. It is intended that implementations might have some method for taking an existing environment, and treating it as a package to be &ldquo;imported&rdquo; into another environment, treating children of Standard in the imported environment as children of the imported package. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.e_10.1.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Suppose we have a public library unit A, a private library unit A.B, and a public library unit A.B.C. A.B.C is a public descendant of itself and of A.B, but a private descendant of A; since A.B is private to A, we don't allow A.B.C to escape outside A either. This is similar to the situation that would occur with physical nesting, like this: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.f_10.1.1</MarginText>
<CodeBlock language="ada">
package A is{"\n"}private{"\n"}    package B is{"\n"}        package C is{"\n"}        end C;{"\n"}    private{"\n"}    end B;{"\n"}end A;{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>12.g_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
Here, A.B.C is visible outside itself and outside A.B, but not outside A. (Note that this example is intended to illustrate the visibility of program units from the outside; the visibility within child units is not quite identical to that of physically nested units, since child units are nested after their parent's declaration.) <br />
</Admonition>
</AnnotatedOnly>

<p>For each library <code><a href="/docs/arm/AA-7/AA-7.1#S0229">package_declaration</a></code> in the environment, there is an implicit declaration of a <i>limited view</i> of that library package. The limited view of a package contains:<br /></p>

<MarginText>12.g.1/3_10.1.1</MarginText>
<MarginInfo items={["AI05-0108-1"]} />
<Admonition type="aarm" aarm="reason">
<b></b>The incomplete view of a type does not have a discriminant{"_"}part even if the <code><a href="/docs/arm/AA-3/AA-3.2#S0023">type_declaration</a></code> does have one. This is necessary because semantic analysis (and the associated dependence on <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s) would be necessary to determine the types of the discriminants.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.g.2/3_10.1.1</MarginText>
<MarginInfo items={["AI05-0129-1"]} />
<Admonition type="aarm" aarm="note">
No incomplete views of incomplete types are included in the limited view. The rules of <a href="/docs/arm/AA-3/AA-3.10#Subclause_3.10.1">3.10.1</a> ensure that the completion of any visible incomplete type is declared in the same visible part, so such an incomplete view would simply be redundant. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.g.3/2_10.1.1</MarginText>
<MarginInfo items={["AI95-00217-06"]} />
<Admonition type="aarm" aarm="discussion">
<b></b>The implementation model of a limited view is that it can be determined solely from the syntax of the source of the unit, without any semantic analysis. That allows it to be created without the semantic dependences of a full unit, which is necessary for it to break mutual dependences of units. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.g.4/2_10.1.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>The limited view does not include package instances and their contents. Semantic analysis of a unit (and dependence on its <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s) would be needed to determine the contents of an instance. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>12.4/2_10.1.1</MarginText>
<p>The limited view of a library <code><a href="/docs/arm/AA-7/AA-7.1#S0229">package_declaration</a></code> is private if that library <code><a href="/docs/arm/AA-7/AA-7.1#S0229">package_declaration</a></code> is immediately preceded by the reserved word <b>private</b>.<br /></p>

<MarginText>12.5/2_10.1.1</MarginText>
<p>[There is no syntax for declaring limited views of packages, because they are always implicit.] The implicit declaration of a limited view of a library package [is not the declaration of a library unit (the library <code><a href="/docs/arm/AA-7/AA-7.1#S0229">package_declaration</a></code> is); nonetheless, it] is a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>. The implicit declaration of the limited view of a library package forms an (implicit) compilation unit whose <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code> is empty.<br /></p>

<MarginText>12.6/2_10.1.1</MarginText>
<p>A library <code><a href="/docs/arm/AA-7/AA-7.1#S0229">package_declaration</a></code> is the completion of the declaration of its limited view.<br /></p>

<AnnotatedOnly>
<MarginText>12.h/2_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
<b>To be honest: </b>This is notwithstanding the rule in <a href="/docs/arm/AA-3/AA-3.11#Subclause_3.11.1">3.11.1</a> that says that implicit declarations don't have completions. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.i/2_10.1.1</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>This rule explains where to find the completions of the incomplete views defined by the limited view. <br />
</Admonition>

#### Legality Rules

<MarginText>13_10.1.1</MarginText>
<p>The parent unit of a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> shall be a [library] package or generic [library] package.<br /></p>

<MarginText>14_10.1.1</MarginText>
<p>If a <code><a href="/docs/arm/AA-6/AA-6.1#S0201">defining_program_unit_name</a></code> of a given declaration or body has a <code><a href="/docs/arm/AA-10/AA-10.1#S0291">parent_unit_name</a></code>, then the given declaration or body shall be a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>. The body of a program unit shall be a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> if and only if the declaration of the program unit is a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>. In a <code><a href="/docs/arm/AA-10/AA-10.1#S0289">library{"_"}unit{"_"}renaming{"_"}declaration</a></code>, the [(old)] <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> shall denote a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>14.a_10.1.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>We could have allowed nested program units to be children of other program units; their semantics would make sense. We disallow them to keep things simpler and because they wouldn't be particularly useful. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>15/2_10.1.1</MarginText>
<MarginInfo items={["AI95-00217-06"]} />
<p>A <code><a href="/docs/arm/AA-10/AA-10.1#S0291">parent_unit_name</a></code> [(which can be used within a <code><a href="/docs/arm/AA-6/AA-6.1#S0201">defining_program_unit_name</a></code> of a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> and in the <b>separate</b> clause of a <code><a href="/docs/arm/AA-10/AA-10.1#S0302">subunit</a></code>)], and each of its <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code>es, shall not denote a <code><a href="/docs/arm/AA-8/AA-8.5#S0238">renaming_declaration</a></code>. [On the other hand, a name that denotes a <code><a href="/docs/arm/AA-10/AA-10.1#S0289">library{"_"}unit{"_"}renaming{"_"}declaration</a></code> is allowed in a <code><a href="/docs/arm/AA-10/AA-10.1#S0296">nonlimited_with_clause</a></code> and other places where the name of a library unit is allowed.]<br /></p>

<MarginText>16_10.1.1</MarginText>
<p>If a library package is an instance of a generic package, then every child of the library package shall either be itself an instance or be a renaming of a library unit. <br /></p>

<AnnotatedOnly>
<MarginText>16.a_10.1.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>A child of an instance of a given generic unit will often be an instance of a (generic) child of the given generic unit. This is not required, however. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16.b_10.1.1</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>Instances are forbidden from having noninstance children for two reasons: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16.c_10.1.1</MarginText>
<dt><br/>1.</dt>
<dl>
<dd>We want all source code that can depend on information from the private part of a library unit to be inside the "subsystem" rooted at the library unit. If an instance of a generic unit were allowed to have a noninstance as a child, the source code of that child might depend on information from the private part of the generic unit, even though it is outside the subsystem rooted at the generic unit.<br /></dd>
<MarginText>16.d_10.1.1</MarginText>
<dt><br/>2.</dt>
<dd>Disallowing noninstance children simplifies the description of the semantics of children of generic packages. <br /></dd>
</dl>
</AnnotatedOnly>
<MarginText>17/3_10.1.1</MarginText>
<MarginInfo items={["AI05-0004-1"]} />
<p>A child of a generic library package shall either be itself a generic unit or be a renaming of some other child of the same generic unit.<br /></p>

<MarginText>18_10.1.1</MarginText>
<p>A child of a parent generic package shall be instantiated or renamed only within the declarative region of the parent generic.<br /></p>

<MarginText>19/2_10.1.1</MarginText>
<MarginInfo items={["AI95-00331-01"]} />
<p>For each child <i>C</i> of some parent generic package <i>P</i>, there is a corresponding declaration <i>C</i> nested immediately within each instance of <i>P</i>. For the purposes of this rule, if a child <i>C</i> itself has a child <i>D</i>, each corresponding declaration for <i>C</i> has a corresponding child <i>D</i>. [The corresponding declaration for a child within an instance is visible only within the scope of a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> that mentions the (original) child generic unit.]<br /></p>

<AnnotatedOnly>
<MarginText>19.a_10.1.1</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>Within the child, like anything nested in a generic unit, one can make up-level references to the current instance of its parent, and thereby gain access to the formal parameters of the parent, to the types declared in the parent, etc. This &ldquo;nesting&rdquo; model applies even within the <code><a href="/docs/arm/AA-12/AA-12.1#S0313">generic_formal_part</a></code> of the child, as it does for a generic child of a nongeneric unit. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>19.b_10.1.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Suppose P is a generic library package, and P.C is a generic child of P. P.C can be instantiated inside the declarative region of P. Outside P, P.C can be mentioned only in a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>. Conceptually, an instance I of P is a package that has a nested generic unit called I.C. Mentioning P.C in a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> allows I.C to be instantiated. I need not be a library unit, and the instantiation of I.C need not be a library unit. If I is a library unit, and an instance of I.C is a child of I, then this instance has to be called something other than C.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>20_10.1.1</MarginText>
<p>A library subprogram shall not override a primitive subprogram. <br /></p>

<AnnotatedOnly>
<MarginText>20.a_10.1.1</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>This prevents certain obscure anomalies. For example, if a library subprogram were to override a subprogram declared in its parent package, then in a compilation unit that depends <i>in</i>directly on the library subprogram, the library subprogram could hide the overridden operation from all visibility, but the library subprogram itself would not be visible.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>20.b_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
Note that even without this rule, such subprograms would be illegal for tagged types, because of the freezing rules. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>21_10.1.1</MarginText>
<p>The defining name of a function that is a compilation unit shall not be an <code><a href="/docs/arm/AA-6/AA-6.1#S0202">operator_symbol</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>21.a_10.1.1</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>Since overloading is not permitted among compilation units, it seems unlikely that it would be useful to define one as an operator. Note that a subunit could be renamed within its parent to be an operator. <br />
</Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>22_10.1.1</MarginText>
<p>A <code><a href="/docs/arm/AA-8/AA-8.5#S0242">subprogram_renaming_declaration</a></code> that is a <code><a href="/docs/arm/AA-10/AA-10.1#S0289">library{"_"}unit{"_"}renaming{"_"}declaration</a></code> is a renaming-as-declaration, not a renaming-as-body.<br /></p>

<MarginText>23_10.1.1</MarginText>
<p>[There are two kinds of dependences among compilation units:] <br /></p>

<MarginText>24/5_10.1.1</MarginText>
<MarginInfo items={["AI12-0445-1"]} />
<ul>
<li>[The <i>semantic dependences</i> (see below) are the ones necessary to check the compile-time rules across compilation unit boundaries; a compilation unit depends semantically on the other compilation units necessary to determine its legality. The visibility rules are based on the semantic dependences.<br /></li>
<MarginText>25_10.1.1</MarginText>
<li>The <i>elaboration dependences</i> (see <a href="/docs/arm/AA-10/AA-10.2">10.2</a>) determine the order of elaboration of <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>s.] <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>25.a_10.1.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>Don't confuse these kinds of dependences with the run-time dependences among tasks and masters defined in <a href="/docs/arm/AA-9/AA-9.3">9.3</a>, &ldquo;<a href="/docs/arm/AA-9/AA-9.3">Task Dependence - Termination of Tasks</a>&rdquo;. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>26/2_10.1.1</MarginText>
<MarginInfo items={["AI95-00217-06"]} />
<p>A <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> depends semantically upon its parent declaration. A subunit depends semantically upon its parent body. A <code><a href="/docs/arm/AA-10/AA-10.1#S0290">library_unit_body</a></code> depends semantically upon the corresponding <code><a href="/docs/arm/AA-10/AA-10.1#S0288">library_unit_declaration</a></code>, if any. The declaration of the limited view of a library package depends semantically upon the declaration of the limited view of its parent. The declaration of a library package depends semantically upon the declaration of its limited view. A compilation unit depends semantically upon each <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> mentioned in a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> of the compilation unit. In addition, if a given compilation unit contains an <code><a href="/docs/arm/AA-4/AA-4.1#S0100">attribute_reference</a></code> of a type defined in another compilation unit, then the given compilation unit depends semantically upon the other compilation unit. The semantic dependence relationship is transitive.<br /></p>

<AnnotatedOnly>
<MarginText>26.a_10.1.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>The &ldquo;if any&rdquo; in the third sentence is necessary because library subprograms are not required to have a <code><a href="/docs/arm/AA-6/AA-6.1#S0195">subprogram_declaration</a></code>. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>26.b_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
<b>To be honest: </b>If a given compilation unit contains a <code><a href="/docs/arm/AA-11/AA-11.2#S0306">choice_parameter_specification</a></code>, then the given compilation unit depends semantically upon the declaration of Ada.Exceptions.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>26.c_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
If a given compilation unit contains a <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code> with an argument of a type defined in another compilation unit, then the given compilation unit depends semantically upon the other compilation unit. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>26.d_10.1.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>For example, a compilation unit containing X'Address depends semantically upon the declaration of package System.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>26.e_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
For the Address attribute, this fixes a hole in Ada 83. Note that in almost all cases, the dependence will need to exist due to <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s, even without this rule. Hence, the rule has very little effect on programmers.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>26.f_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
Note that the semantic dependence does not have the same effect as a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>; in order to denote a declaration in one of those packages, a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> will generally be needed.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>26.g_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
Note that no special rule is needed for an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code>, since an expression after <b>use</b> will require semantic dependence upon the compilation unit containing the <code><a href="/docs/arm/AA-3/AA-3.2#S0023">type_declaration</a></code> of interest.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>26.h/2_10.1.1</MarginText>
<MarginInfo items={["AI95-00217-06"]} />
<Admonition type="aarm" aarm="note">
Unlike a full view of a package, a limited view does not depend semantically on units mentioned in <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s of the <code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code> that defines the package. Formally, this is achieved by saying that the limited view has an empty <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code>. This is necessary so that they can be useful for their intended purpose: allowing mutual dependences between packages. The lack of semantic dependence limits the contents of a limited view to the items that can be determined solely from the syntax of the source of the package, without any semantic analysis. That allows it to be created without the semantic dependences of a full package. <br />
</Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>26.1/2_10.1.1</MarginText>
<MarginInfo items={["AI95-00217-06"]} />
<p>The elaboration of the declaration of the limited view of a package has no effect. <br /></p>

<AnnotatedOnly>
<MarginText>27/5_10.1.1</MarginText>
<MarginInfo items={["AI12-0442-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 1   A simple program can consist of a single compilation unit. A <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code> can have no compilation units; for example, its text can consist of <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code>s. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.a_10.1.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Such <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code>s cannot have any arguments that are <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code>s, by a previous rule of this subclause. A <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code> can even be entirely empty, which is probably not useful.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.b_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
Some interesting properties of the three kinds of dependence: The elaboration dependences also include the semantic dependences, except that subunits are taken together with their parents. The semantic dependences partly determine the order in which the compilation units appear in the environment at compile time. At run time, the order is partly determined by the elaboration dependences.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.c_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
The model whereby a child is inside its parent's declarative region, after the parent's declaration, as explained in <a href="/docs/arm/AA-8/AA-8.1">8.1</a>, has the following ramifications: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.d_10.1.1</MarginText>
<ul>
<li>The restrictions on &ldquo;early&rdquo; use of a private type (RM83-7.4.1(4)) or a deferred constant (RM83-7.4.3(2)) do not apply to uses in child units, because they follow the full declaration.<br /></li>
<MarginText>27.e_10.1.1</MarginText>
<li>A library subprogram is never primitive, even if its profile includes a type declared immediately within the parent's <code><a href="/docs/arm/AA-7/AA-7.1#S0230">package_specification</a></code>, because the child is not declared immediately within the same <code><a href="/docs/arm/AA-7/AA-7.1#S0230">package_specification</a></code> as the type (so it doesn't declare a new primitive subprogram), and because the child is forbidden from overriding an old primitive subprogram. It is immediately within the same declarative region, but not the same <code><a href="/docs/arm/AA-7/AA-7.1#S0230">package_specification</a></code>. Thus, for a tagged type, it is not possible to call a child subprogram in a dispatching manner. (This is also forbidden by the freezing rules.) Similarly, it is not possible for the user to declare primitive subprograms of the types declared in the declaration of Standard, such as Integer (even if the rules were changed to allow a library unit whose name is an operator symbol).<br /></li>
<MarginText>27.f_10.1.1</MarginText>
<li>When the parent unit is &ldquo;used&rdquo; the simple names of the with'd child units are directly visible (see <a href="/docs/arm/AA-8/AA-8.4">8.4</a>, &ldquo;<a href="/docs/arm/AA-8/AA-8.4">Use Clauses</a>&rdquo;).<br /></li>
<MarginText>27.g_10.1.1</MarginText>
<li>When a parent body with's its own child, the defining name of the child is directly visible, and the parent body is not allowed to include a declaration of a homograph of the child unit immediately within the <code><a href="/docs/arm/AA-3/AA-3.11#S0086">declarative_part</a></code> of the body (RM83-8.3(17)). <br /></li>
</ul>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>27.h_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
Note that &ldquo;declaration of a library unit&rdquo; is different from &ldquo;<code><a href="/docs/arm/AA-10/AA-10.1#S0288">library_unit_declaration</a></code>&rdquo; &mdash; the former includes <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code>. Also, we sometimes really mean &ldquo;declaration of a view of a library unit&rdquo;, which includes <code><a href="/docs/arm/AA-10/AA-10.1#S0289">library{"_"}unit{"_"}renaming{"_"}declaration</a></code>s.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.i_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
The visibility rules generally imply that the renamed view of a <code><a href="/docs/arm/AA-10/AA-10.1#S0289">library{"_"}unit{"_"}renaming{"_"}declaration</a></code> has to be mentioned in a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with{"_"}clause</a></code> of the <code><a href="/docs/arm/AA-10/AA-10.1#S0289">library{"_"}unit{"_"}renaming{"_"}declaration</a></code>. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.j_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
<b>To be honest: </b>The real rule is that the renamed library unit has to be visible in the <code><a href="/docs/arm/AA-10/AA-10.1#S0289">library{"_"}unit{"_"}renaming{"_"}declaration</a></code>. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.k_10.1.1</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>In most cases, &ldquo;has to be visible&rdquo; means there has to be a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>. However, it is possible in obscure cases to avoid the need for a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>; in particular, a compilation unit such as &ldquo;<b>package</b> P.Q <b>renames</b> P;&rdquo; is legal with no <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s (though not particularly interesting). ASCII is physically nested in Standard, and so is not a library unit, and cannot be renamed as a library unit. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>28_10.1.1</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 2   The <code><a href="/docs/arm/AA-6/AA-6.1#S0199">designator</a></code> of a library function cannot be an <code><a href="/docs/arm/AA-6/AA-6.1#S0202">operator_symbol</a></code>, but a nonlibrary <code><a href="/docs/arm/AA-8/AA-8.5#S0238">renaming_declaration</a></code> is allowed to rename a library function as an operator. </Admonition>

## 10.1.2  Context Clauses - With Clauses

<MarginText>1_10.1.2</MarginText>
<p>[A <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code> is used to specify the <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>s whose names are needed within a compilation unit.] <br /></p>


#### Language Design Principles

<AnnotatedOnly>
<MarginText>1.a_10.1.2</MarginText>
<Admonition type="aarm" aarm="note">
The reader should be able to understand a <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code> without looking ahead. Similarly, when compiling a <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code>, the compiler should not have to look ahead at subsequent <code><a href="/docs/arm/AA-10/AA-10.1#S0293">context_item</a></code>s, nor at the compilation unit to which the <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code> is attached. (We have not completely achieved this.)<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>1.b/2_10.1.2</MarginText>
<MarginInfo items={["AI95-00217-06"]} />
<Admonition type="aarm" aarm="note">
A <i>ripple effect</i> occurs when the legality of a compilation unit could be affected by adding or removing an otherwise unneeded <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> on some compilation unit on which the unit depends, directly or indirectly. We try to avoid ripple effects because they make understanding and maintenance more difficult. However, ripple effects can occur because of direct visibility (as in child units); this seems impossible to eliminate. The ripple effect for <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s is somewhat similar to the Beaujolais effect (see <a href="/docs/arm/AA-8/AA-8.4">8.4</a>) for <code><a href="/docs/arm/AA-8/AA-8.4#S0235">use_clause</a></code>s, which we also try to avoid. <br />
</Admonition>
</AnnotatedOnly>

#### Name Resolution Rules

<MarginText>5_10.1.2</MarginText>
<p>The <i>scope</i> of a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> that appears on a <code><a href="/docs/arm/AA-10/AA-10.1#S0288">library{"_"}unit{"_"}declaration</a></code> or <code><a href="/docs/arm/AA-10/AA-10.1#S0289">library{"_"}unit{"_"}renaming{"_"}declaration</a></code> consists of the entire declarative region of the declaration[, which includes all children and subunits]. The scope of a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> that appears on a body consists of the body[, which includes all subunits]. <br /></p>

<AnnotatedOnly>
<MarginText>5.a/2_10.1.2</MarginText>
<MarginInfo items={["AI95-00262-01"]} />
<Admonition type="aarm" aarm="discussion">
<b></b>Suppose a nonprivate  <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> of a public library unit mentions one of its private siblings. (This is only allowed on the body of the public library unit.) We considered making the scope of that <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> not include the visible part of the public library unit. (This would only matter for a <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code>, since those are the only kinds of body that have a visible part, and only if the <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code> completes a <code><a href="/docs/arm/AA-6/AA-6.1#S0195">subprogram_declaration</a></code>, since otherwise the <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> would be illegal.) We did not put in such a rule for two reasons: (1) It would complicate the wording of the rules, because we would have to split each <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> into pieces, in order to correctly handle &ldquo;<b>with</b> P, Q;&rdquo; where P is public and Q is private. (2) The conformance rules prevent any problems. It doesn't matter if a type name in the spec of the body denotes the completion of a <code><a href="/docs/arm/AA-7/AA-7.3#S0232">private_type_declaration</a></code>.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b_10.1.2</MarginText>
<Admonition type="aarm" aarm="note">
A <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> also affects visibility within subsequent <code><a href="/docs/arm/AA-8/AA-8.4#S0235">use_clause</a></code>s and <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code>s of the same <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code>, even though those are not in the scope of the <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>6/2_10.1.2</MarginText>
<MarginInfo items={["AI95-00217-06"]} />
<p>A <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> (and the corresponding library unit) is <i>named</i> in a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> if it is denoted by a <i>library{"_"}unit{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> in the <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>. A <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> (and the corresponding library unit) is <i>mentioned</i> in a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> if it is named in the <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> or if it is denoted by a <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> in the <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>.<br /></p>

<AnnotatedOnly>
<MarginText>6.a_10.1.2</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b><code><a href="/docs/arm/AA-10/AA-10.1#S0294">With_clause</a></code>s control the visibility of declarations or renamings of library units. Mentioning a root library unit in a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> makes its declaration directly visible. Mentioning a nonroot library unit makes its declaration visible. See Clause <a href="/docs/arm/AA-8/">8</a> for details.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.b/2_10.1.2</MarginText>
<MarginInfo items={["AI95-00114-01"]} />
<Admonition type="aarm" aarm="note">
Note that this rule implies that &ldquo;<b>with</b> A.B.C;&rdquo; is almost equivalent to &ldquo;<b>with</b> A, A.B, A.B.C;&rdquo;. The reason for making a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> apply to all the ancestor units is to avoid &ldquo;visibility holes&rdquo; &mdash; situations in which an inner program unit is visible while an outer one is not. Visibility holes would cause semantic complexity and implementation difficulty. (This is not exactly equivalent because the latter <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> names A and A.B, while the previous one does not. Whether a unit is &ldquo;named&rdquo; does not have any effect on visibility, however, so it is equivalent for visibility purposes.)<br />
</Admonition>
</AnnotatedOnly>

<MarginText>7_10.1.2</MarginText>
<p>[Outside its own declarative region, the declaration or renaming of a library unit can be visible only within the scope of a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> that mentions it. The visibility of the declaration or renaming of a library unit otherwise follows from its placement in the environment.] <br /></p>


#### Legality Rules

<MarginText>8/2_10.1.2</MarginText>
<MarginInfo items={["AI95-00262-01"]} />
<p>If a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> of a given <code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code> mentions a private child of some library unit, then the given <code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code> shall be one of: <br /></p>

<MarginText>9/2_10.1.2</MarginText>
<MarginInfo items={["AI95-00262-01"]} />
<ul>
<li>the declaration, body, or subunit of a private descendant of that library unit;<br /></li>
<MarginText>10/2_10.1.2</MarginText>
<MarginInfo items={["AI95-00220-01", "AI95-00262-01"]} />
<li>the body or subunit of a public descendant of that library unit, but not a subprogram body acting as a subprogram declaration (see <a href="/docs/arm/AA-10/AA-10.1#Subclause_10.1.4">10.1.4</a>); or<br /></li>
<MarginText>11/2_10.1.2</MarginText>
<MarginInfo items={["AI95-00262-01"]} />
<li>the declaration of a public descendant of that library unit, in which case the <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> shall include the reserved word <b>private</b>. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>11.a/2_10.1.2</MarginText>
<MarginInfo items={["AI95-00262-01"]} />
<Admonition type="aarm" aarm="reason">
<b></b>The purpose of this rule is to prevent a private child from being visible from outside the subsystem rooted at its parent. A private child can be semantically depended-on without violating this principle if it is used in a private <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.b_10.1.2</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>This rule violates the one-pass <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code>s Language Design Principle. We rationalize this by saying that at least that Language Design Principle works for legal compilation units.<br />
</Admonition>
</AnnotatedOnly>

<Admonition type="aarm" aarm="note">
(1) is OK because it's a private child of A.B &mdash; it would be illegal if we made A.B.X a public child of A.B. (2) is OK because it's the body of a child of A.B. </Admonition>
</AnnotatedOnly>

<MarginText>12/5_10.1.2</MarginText>

<p>A <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> denoting a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> (or the corresponding declaration for a child of a generic within an instance &mdash; see <a href="/docs/arm/AA-10/AA-10.1#Subclause_10.1.1">10.1.1</a>), if it is visible only due to being mentioned in one or more <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s of a unit <i>U</i> that include the reserved word <b>private</b>, shall appear only within: <br /></p>

<MarginText>13/2_10.1.2</MarginText>
<ul>
<li>a private part;<br /></li>
<MarginText>14/5_10.1.2</MarginText>
<MarginInfo items={["AI12-0261-1"]} />
<li>a body of a public descendant of <i>U</i>, but not within the <code><a href="/docs/arm/AA-6/AA-6.1#S0196">subprogram_specification</a></code> of a body of a subprogram that is a public descendant of <i>U</i>;<br /></li>
<MarginText>15/5_10.1.2</MarginText>
<MarginInfo items={["AI12-0261-1"]} />

<MarginText>16/2_10.1.2</MarginText>
<li>a pragma within a context clause. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>16.a/2_10.1.2</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>These rules apply only if all of the <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s that mention the name include the reserved word <b>private</b>. <AnnotatedOnly>
<MarginText>16.b/3_10.1.2</MarginText>
<MarginInfo items={["AI05-0077-1"]} />
<Admonition type="aarm" aarm="reason">
<b></b>These rules make the <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> visible anywhere that is not visible outside the subsystem rooted at the <code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code> having the private <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>, including private parts of packages nested in the visible part, private parts of child packages, the visible part of private children, and context clause pragmas like Elaborate{"_"}All.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16.c/2_10.1.2</MarginText>
<Admonition type="aarm" aarm="note">
We considered having the scope of a private <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> not include the visible part. However, that rule would mean that moving a declaration between the visible part and the private part could change its meaning from one legal interpretation to a different legal interpretation. For example: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16.d/2_10.1.2</MarginText>
<CodeBlock language="ada">
package A is{"\n"}    function B return Integer;{"\n"}end A;{"\n"}
<MarginText>16.e/2_10.1.2</MarginText>
function B return Integer;{"\n"}
<MarginText>16.f/2_10.1.2</MarginText>

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>16.g/2_10.1.2</MarginText>
<Admonition type="aarm" aarm="note">
If we say that library subprogram B is not in scope in the visible part of C, then the B at (1) resolves to A.B, while (2) resolves to library unit B. Simply moving a declaration could silently change its meaning. With the legality rule defined above, the B at (1) is illegal. If the user really meant A.B, they still can say that. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>18.a/2_10.1.2</MarginText>
<MarginInfo items={["AI95-00412-01"]} />

<b></b>We don't allow a <code><a href="/docs/arm/AA-10/AA-10.1#S0295">limited_with_clause</a></code> on a <code><a href="/docs/arm/AA-10/AA-10.1#S0289">library{"_"}unit{"_"}renaming{"_"}declaration</a></code> because it would be useless and therefore probably is a mistake. </Admonition>

</Admonition>

<Admonition type="aarm" aarm="note">

Thus, within a compilation unit that mentions its declaration, the name of a library package can be given in <code><a href="/docs/arm/AA-8/AA-8.4#S0235">use_clause</a></code>s and can be used to form expanded names, a library subprogram can be called, and instances of a generic library unit can be declared. If a child of a parent generic package is mentioned in a <code><a href="/docs/arm/AA-10/AA-10.1#S0296">nonlimited_with_clause</a></code>, then the corresponding declaration nested within each visible instance is visible within the compilation unit. Similarly, a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> mentioned in a <code><a href="/docs/arm/AA-10/AA-10.1#S0295">limited_with_clause</a></code> of a compilation unit is visible within the compilation unit and thus can be used to form expanded names.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>23.a_10.1.2</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>The rules given for <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s are such that the same effect is obtained whether the name of a library unit is mentioned once or more than once by the applicable <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s, or even within a given <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>23.b_10.1.2</MarginText>
<Admonition type="aarm" aarm="note">
If a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> mentions a <code><a href="/docs/arm/AA-10/AA-10.1#S0289">library{"_"}unit{"_"}renaming{"_"}declaration</a></code>, it only &ldquo;mentions&rdquo; the <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code>es appearing explicitly in the <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> (and the renamed view itself); the <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> is not defined to mention the ancestors of the renamed entity. Thus, if X renames Y.Z, then &ldquo;with X;&rdquo; does not make the declarations of Y or Z visible. Note that this does not cause the dreaded visibility holes mentioned above. <br />
</Admonition>
</AnnotatedOnly>

#### Wording Changes from Ada 83

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>32.c_10.1.2</MarginText>
<Admonition type="aarm" aarm="note">
The syntax rule for <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code> is modified to more closely reflect the semantics. The Ada 83 syntax rule implies that the <code><a href="/docs/arm/AA-8/AA-8.4#S0235">use_clause</a></code>s that appear immediately after a particular <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> are somehow attached to that <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>, which is not true. The new syntax allows a <code><a href="/docs/arm/AA-8/AA-8.4#S0235">use_clause</a></code> to appear first, but that is prevented by a textual rule that already exists in Ada 83.<br />
</Admonition>

## 10.1.3  Subunits of Compilation Units

<MarginText>1_10.1.3</MarginText>
<p>[Subunits are like child units, with these (important) differences: subunits support the separate compilation of bodies only (not declarations); the parent contains a <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code> to indicate the existence and place of each of its subunits; declarations appearing in the parent's body can be visible within the subunits.] <br /></p>


#### Syntax

<MarginText>2_10.1.3</MarginText>

<CodeBlock>
<code>body{"_"}stub</code><a id="S0297"></a><code> ::= </code><br />    <code><a href="/docs/arm/AA-10/AA-10.1#S0298">subprogram_body_stub</a></code> | <code><a href="/docs/arm/AA-10/AA-10.1#S0299">package_body_stub</a></code><br />  | <code><a href="/docs/arm/AA-10/AA-10.1#S0300">task_body_stub</a></code> | <code><a href="/docs/arm/AA-10/AA-10.1#S0301">protected_body_stub</a></code><br />

</CodeBlock>

<MarginInfo items={["AI95-00218-03", "AI05-0267-1"]} />

<CodeBlock>
<code>subprogram{"_"}body{"_"}stub</code><a id="S0298"></a><code> ::= </code><br />    [<code><a href="/docs/arm/AA-8/AA-8.3#S0234">overriding_indicator</a></code>]<br />    <code><a href="/docs/arm/AA-6/AA-6.1#S0196">subprogram_specification</a></code> <b>is</b> <b>separate</b><br />       [<code><a href="/docs/arm/AA-13/AA-13.1#S0346">aspect_specification</a></code>];<br />

</CodeBlock>

#### Legality Rules

<MarginText>8/2_10.1.3</MarginText>
<MarginInfo items={["AI95-00243-01"]} />
<p>The <i>parent body</i> of a subunit is the body of the program unit denoted by its <code><a href="/docs/arm/AA-10/AA-10.1#S0291">parent_unit_name</a></code>. The term <i>subunit</i> is used to refer to a <code><a href="/docs/arm/AA-10/AA-10.1#S0302">subunit</a></code> and also to the <code><a href="/docs/arm/AA-3/AA-3.11#S0090">proper_body</a></code> of a <code><a href="/docs/arm/AA-10/AA-10.1#S0302">subunit</a></code>. The <i>subunits of a program unit</i> include any subunit that names that program unit as its parent, as well as any subunit that names such a subunit as its parent (recursively). <br /></p>

<AnnotatedOnly>
<MarginText>8.a.1/2_10.1.3</MarginText>
<MarginInfo items={["AI95-00243-01"]} />
<Admonition type="aarm" aarm="reason">
<b></b>We want any rule that applies to a subunit to apply to a subunit of a subunit as well. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.a/5_10.1.3</MarginText>
<Admonition type="aarm" aarm="note">
<b>Term entry: </b><b>subunit</b> &mdash; body of a program unit that can be compiled separately from its enclosing program unit<br />
</Admonition>
</AnnotatedOnly>

<MarginText>9_10.1.3</MarginText>
<p>The parent body of a subunit shall be present in the current environment, and shall contain a corresponding <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code> with the same <code><a href="/docs/arm/AA-3/AA-3.1#S0022">defining_identifier</a></code> as the subunit. <br /></p>

<AnnotatedOnly>
<MarginText>9.a_10.1.3</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>This can't be a Name Resolution Rule, because a <code><a href="/docs/arm/AA-10/AA-10.1#S0302">subunit</a></code> is not a complete context. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>10/3_10.1.3</MarginText>
<MarginInfo items={["AI05-0004-1"]} />
<p>A <code><a href="/docs/arm/AA-10/AA-10.1#S0299">package_body_stub</a></code> shall be the completion of a <code><a href="/docs/arm/AA-7/AA-7.1#S0229">package{"_"}declaration</a></code> or <code><a href="/docs/arm/AA-12/AA-12.1#S0312">generic{"_"}package{"_"}declaration</a></code>; a <code><a href="/docs/arm/AA-10/AA-10.1#S0300">task{"_"}body{"_"}stub</a></code> shall be the completion of a task declaration; a <code><a href="/docs/arm/AA-10/AA-10.1#S0301">protected{"_"}body{"_"}stub</a></code> shall be the completion of a protected declaration.<br /></p>

<MarginText>11/5_10.1.3</MarginText>
<MarginInfo items={["AI12-0444-1"]} />
<p>In contrast, a <code><a href="/docs/arm/AA-10/AA-10.1#S0298">subprogram_body_stub</a></code> can be defined without it being the completion of a previous declaration, [in which case the <code>{"_"}stub</code> declares the subprogram]. If the <code>{"_"}stub</code> is a completion, it shall be the completion of a <code><a href="/docs/arm/AA-6/AA-6.1#S0195">subprogram_declaration</a></code> or <code><a href="/docs/arm/AA-12/AA-12.1#S0311">generic_subprogram_declaration</a></code>. The profile of a <code><a href="/docs/arm/AA-10/AA-10.1#S0298">subprogram_body_stub</a></code> that completes a declaration shall conform fully to that of the declaration. <br /></p>

<AnnotatedOnly>
<MarginText>11.a_10.1.3</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>The part about <code><a href="/docs/arm/AA-10/AA-10.1#S0298">subprogram_body_stub</a></code>s echoes the corresponding rule for <code>subprogram{"_"}bodies</code> in <a href="/docs/arm/AA-6/AA-6.3">6.3</a>, &ldquo;<a href="/docs/arm/AA-6/AA-6.3">Subprogram Bodies</a>&rdquo;. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>12_10.1.3</MarginText>
<p>A subunit that corresponds to a <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code> shall be of the same kind (<code>package{"_"}</code>, <code>subprogram{"_"}</code>, <code>task{"_"}</code>, or <code>protected{"_"}</code>) as the <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code>. The profile of a <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code> subunit shall be fully conformant to that of the corresponding <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code>. <br /></p>

<MarginText>13_10.1.3</MarginText>
<p>A <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code> shall appear immediately within the <code><a href="/docs/arm/AA-3/AA-3.11#S0086">declarative_part</a></code> of a compilation unit body. This rule does not apply within an instance of a generic unit. <br /></p>

<AnnotatedOnly>
<MarginText>13.a_10.1.3</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>This is a methodological restriction; that is, it is not necessary for the semantics of the language to make sense. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>14_10.1.3</MarginText>
<p>The <code><a href="/docs/arm/AA-3/AA-3.1#S0022">defining_identifier</a></code>s of all <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code>s that appear immediately within a particular <code><a href="/docs/arm/AA-3/AA-3.11#S0086">declarative_part</a></code> shall be distinct. <br /></p>


#### Post-Compilation Rules

<MarginText>15_10.1.3</MarginText>
<p>For each <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code>, there shall be a subunit containing the corresponding <code><a href="/docs/arm/AA-3/AA-3.11#S0090">proper_body</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>16_10.1.3</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   The rules in <a href="/docs/arm/AA-10/AA-10.1#Subclause_10.1.4">10.1.4</a>, &ldquo;<a href="/docs/arm/AA-10/AA-10.1#Subclause_10.1.4">The Compilation Process</a>&rdquo; say that a <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code> is equivalent to the corresponding <code><a href="/docs/arm/AA-3/AA-3.11#S0090">proper_body</a></code>. This implies: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>17_10.1.3</MarginText>
<ul>
<li>Visibility within a subunit is the visibility that would be obtained at the place of the corresponding <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code> (within the parent body) if the <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code> of the subunit were appended to that of the parent body. <br /></li>
</ul>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>17.a_10.1.3</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Recursively. Note that this transformation might make the parent illegal; hence it is not a true equivalence, but applies only to visibility within the subunit. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18_10.1.3</MarginText>
<ul>
<li>The effect of the elaboration of a <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code> is to elaborate the subunit. <br /></li>
</ul>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>18.a_10.1.3</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>The elaboration of a subunit is part of its parent body's elaboration, whereas the elaboration of a child unit is not part of its parent declaration's elaboration. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18.b_10.1.3</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>A <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> that is mentioned in a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> of a subunit can be hidden (from direct visibility) by a declaration (with the same <code><a href="/docs/arm/AA-2/AA-2.3#S0002">identifier</a></code>) given in the subunit. Moreover, such a <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> can even be hidden by a declaration given within the parent body since a library unit is declared in its parent's declarative region; this however does not affect the interpretation of the <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s themselves, since only <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>s are visible or directly visible in <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18.c_10.1.3</MarginText>
<Admonition type="aarm" aarm="note">
The body of a protected operation cannot be a subunit. This follows from the syntax rules. The body of a protected unit can be a subunit. <br />
</Admonition>
</AnnotatedOnly>

## 10.1.4  The Compilation Process

<MarginText>1_10.1.4</MarginText>
<p>Each compilation unit submitted to the compiler is compiled in the context of an <i>environment</i> <code><a href="/docs/arm/AA-3/AA-3.11#S0086">declarative_part</a></code> (or simply, an <i>environment</i>), which is a conceptual <code><a href="/docs/arm/AA-3/AA-3.11#S0086">declarative_part</a></code> that forms the outermost declarative region of the context of any <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code>. At run time, an environment forms the <code><a href="/docs/arm/AA-3/AA-3.11#S0086">declarative_part</a></code> of the body of the environment task of a partition (see <a href="/docs/arm/AA-10/AA-10.2">10.2</a>, &ldquo;<a href="/docs/arm/AA-10/AA-10.2">Program Execution</a>&rdquo;). <br /></p>

<AnnotatedOnly>
<MarginText>1.a_10.1.4</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>At compile time, there is no particular construct that the declarative region is considered to be nested within &mdash; the environment is the universe. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>1.b_10.1.4</MarginText>
<Admonition type="aarm" aarm="note">
<b>To be honest: </b>The environment is really just a portion of a <code><a href="/docs/arm/AA-3/AA-3.11#S0086">declarative_part</a></code>, since there might, for example, be bodies that do not yet exist. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>2/5_10.1.4</MarginText>
<MarginInfo items={["AI12-0445-1"]} />
<p>The <code><a href="/docs/arm/AA-3/AA-3.11#S0087">declarative_item</a></code>s of the environment are <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>s appearing in an order such that there are no forward semantic dependences. Each included subunit occurs in place of the corresponding stub. The visibility rules apply as if the environment were the outermost declarative region, except that <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with{"_"}clause</a></code>s are necessary to make declarations of library units visible (see <a href="/docs/arm/AA-10/AA-10.1#Subclause_10.1.2">10.1.2</a>).<br /></p>

<MarginText>3/2_10.1.4</MarginText>
<MarginInfo items={["AI95-00217-06"]} />
<p>The mechanisms for creating an environment and for adding and replacing compilation units within an environment are implementation defined. The mechanisms for adding a compilation unit mentioned in a <code><a href="/docs/arm/AA-10/AA-10.1#S0295">limited_with_clause</a></code> to an environment are implementation defined. <br /></p>

<AnnotatedOnly>
<MarginText>3.a_10.1.4</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<b></b>The mechanisms for creating an environment and for adding and replacing compilation units.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.a.1/2_10.1.4</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<b></b>The mechanisms for adding a compilation unit mentioned in a <code><a href="/docs/arm/AA-10/AA-10.1#S0295">limited_with_clause</a></code> to an environment.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.b_10.1.4</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>The traditional model, used by most Ada 83 implementations, is that one places a compilation unit in the environment by compiling it. Other models are possible. For example, an implementation might define the environment to be a directory; that is, the compilation units in the environment are all the compilation units in the source files contained in the directory. In this model, the mechanism for replacing a compilation unit with a new one is simply to edit the source file containing that compilation unit. <br />
</Admonition>
</AnnotatedOnly>


#### Name Resolution Rules

<MarginText>4/3_10.1.4</MarginText>
<MarginInfo items={["AI95-00192-01", "AI05-0264-1"]} />
<p>{"{"}<i>8652/0032</i>{"}"} If a <code><a href="/docs/arm/AA-10/AA-10.1#S0290">library_unit_body</a></code> that is a <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code> is submitted to the compiler, it is interpreted only as a completion if a <code><a href="/docs/arm/AA-10/AA-10.1#S0288">library_unit_declaration</a></code> with the same <code><a href="/docs/arm/AA-6/AA-6.1#S0201">defining_program_unit_name</a></code> already exists in the environment for a subprogram other than an instance of a generic subprogram or for a generic subprogram (even if the profile of the body is not type conformant with that of the declaration); otherwise, the <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code> is interpreted as both the declaration and body of a library subprogram. <br /></p>

<AnnotatedOnly>
<MarginText>4.a_10.1.4</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>The principle here is that a <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code> should be interpreted as only a completion if and only if it &ldquo;might&rdquo; be legal as the completion of some preexisting declaration, where &ldquo;might&rdquo; is defined in a way that does not require overload resolution to determine.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.b_10.1.4</MarginText>
<Admonition type="aarm" aarm="note">
Hence, if the preexisting declaration is a <code><a href="/docs/arm/AA-6/AA-6.1#S0195">subprogram_declaration</a></code> or <code><a href="/docs/arm/AA-12/AA-12.1#S0311">generic_subprogram_declaration</a></code>, we treat the new <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code> as its completion, because it &ldquo;might&rdquo; be legal. If it turns out that the profiles don't fully conform, it's an error. In all other cases (the preexisting declaration is a package or a generic package, or an instance of a generic subprogram, or a renaming, or a &ldquo;spec-less&rdquo; subprogram, or in the case where there is no preexisting thing), the <code><a href="/docs/arm/AA-6/AA-6.3#S0216">subprogram_body</a></code> declares a new subprogram.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.c_10.1.4</MarginText>
<Admonition type="aarm" aarm="note">
See also AI83-00266/09. <br />
</Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>5_10.1.4</MarginText>
<p>When a compilation unit is compiled, all compilation units upon which it depends semantically shall already exist in the environment; the set of these compilation units shall be <i>consistent</i> in the sense that the new compilation unit shall not semantically depend (directly or indirectly) on two different versions of the same compilation unit, nor on an earlier version of itself. <br /></p>

<AnnotatedOnly>
<MarginText>5.a_10.1.4</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>For example, if package declarations A and B both say &ldquo;<b>with</b> X;&rdquo;, and the user compiles a compilation unit that says &ldquo;<b>with</b> A, B;&rdquo;, then the A and B have to be talking about the same version of X. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b_10.1.4</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>What it means to be a &ldquo;different version&rdquo; is not specified by the language. In some implementations, it means that the compilation unit has been recompiled. In others, it means that the source of the compilation unit has been edited in some significant way.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.c_10.1.4</MarginText>
<Admonition type="aarm" aarm="note">
Note that an implementation cannot require the existence of compilation units upon which the given one does not semantically depend. For example, an implementation is required to be able to compile a compilation unit that says "<b>with</b> A;" when A's body does not exist. It has to be able to detect errors without looking at A's body.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.d/3_10.1.4</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<Admonition type="aarm" aarm="note">
Similarly, the implementation has to be able to compile a call to a subprogram for which aspect Inline has been specified without seeing the body of that subprogram &mdash; inlining would not be achieved in this case, but the call is still legal.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.e/3_10.1.4</MarginText>
<MarginInfo items={["AI95-00217-06", "AI05-0005-1"]} />
<Admonition type="aarm" aarm="note">
The consistency rule applies to limited views as well as the full view of a compilation unit. That means that an implementation needs a way to enforce consistency of limited views, not just of full views. <br />
</Admonition>
</AnnotatedOnly>


#### Implementation Permissions

<MarginText>6/2_10.1.4</MarginText>
<MarginInfo items={["AI95-00217-06"]} />
<p>The implementation may require that a compilation unit be legal before it can be mentioned in a <code><a href="/docs/arm/AA-10/AA-10.1#S0295">limited_with_clause</a></code> or it can be inserted into the environment.<br /></p>

<MarginText>7/3_10.1.4</MarginText>
<MarginInfo items={["AI95-00214-01", "AI05-0229-1"]} />
<p>When a compilation unit that declares or renames a library unit is added to the environment, the implementation may remove from the environment any preexisting <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> or <code><a href="/docs/arm/AA-10/AA-10.1#S0302">subunit</a></code> with the same full expanded name. When a compilation unit that is a subunit or the body of a library unit is added to the environment, the implementation may remove from the environment any preexisting version of the same compilation unit. When a compilation unit that contains a <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code> is added to the environment, the implementation may remove any preexisting <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> or <code><a href="/docs/arm/AA-10/AA-10.1#S0302">subunit</a></code> with the same full expanded name as the <code><a href="/docs/arm/AA-10/AA-10.1#S0297">body_stub</a></code>. When a given compilation unit is removed from the environment, the implementation may also remove any compilation unit that depends semantically upon the given one. If the given compilation unit contains the body of a subprogram for which aspect Inline is True, the implementation may also remove any compilation unit containing a call to that subprogram. <br /></p>

<AnnotatedOnly>
<MarginText>7.a/3_10.1.4</MarginText>
<MarginInfo items={["AI05-0005-1"]} />
<Admonition type="aarm" aarm="ramification">
<b></b>The permissions given in this paragraph correspond to the traditional model, where compilation units enter the environment by being compiled into it, and the compiler checks their legality at that time. An implementation model in which the environment consists of all source files in a given directory might not want to take advantage of these permissions. Compilation units would not be checked for legality as soon as they enter the environment; legality checking would happen later, when compilation units are compiled. In this model, compilation units might never be automatically removed from the environment; they would be removed when the user explicitly deletes a source file.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.b_10.1.4</MarginText>
<Admonition type="aarm" aarm="note">
Note that the rule is recursive: if the above permission is used to remove a compilation unit containing an inlined subprogram call, then compilation units that depend semantically upon the removed one may also be removed, and so on.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.c_10.1.4</MarginText>
<Admonition type="aarm" aarm="note">
Note that here we are talking about dependences among existing compilation units in the environment; it doesn't matter what <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code>s are attached to the new compilation unit that triggered all this.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.d/3_10.1.4</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<Admonition type="aarm" aarm="note">
An implementation may have other modes in which compilation units in addition to the ones mentioned above are removed. For example, an implementation might inline subprogram calls without an explicit aspect Inline. If so, it either has to have a mode in which that optimization is turned off, or it has to automatically regenerate code for the inlined calls without requiring the user to resubmit them to the compiler. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.d.1/2_10.1.4</MarginText>
<MarginInfo items={["AI95-00077-01", "AI95-00114-01"]} />
<Admonition type="aarm" aarm="discussion">
<b></b>{"{"}<i>8652/0108</i>{"}"} In the standard mode, implementations may only remove units from the environment for one of the reasons listed here, or in response to an explicit user command to modify the environment. It is not intended that the act of compiling a unit is one of the &ldquo;mechanisms&rdquo; for removing units other than those specified by this document.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.e/2_10.1.4</MarginText>
<MarginInfo items={["AI95-00214-01"]} />
<Admonition type="aarm" aarm="note">
These rules are intended to ensure that an implementation never need keep more than one compilation unit with any full expanded name. In particular, it is not necessary to be able to have a subunit and a child unit with the same name in the environment at one time.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8_10.1.4</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 1   The rules of the language are enforced across <code><a href="/docs/arm/AA-10/AA-10.1#S0285">compilation</a></code> and compilation unit boundaries, just as they are enforced within a single compilation unit. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.a/3_10.1.4</MarginText>
<MarginInfo items={["AI05-0299-1"]} />
<Admonition type="aarm" aarm="ramification">
<b></b>Note that Clause <a href="/docs/arm/AA-1/">1</a> requires an implementation to detect illegal compilation units at compile time. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9/5_10.1.4</MarginText>
<MarginInfo items={["AI12-0442-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 2   An implementation can support a concept of a <i>library</i>, which contains <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>s. If multiple libraries are supported, the implementation can document how a single environment is constructed when a compilation unit is submitted to the compiler. Naming conflicts between different libraries can, for example, be resolved by treating each library as the root of a hierarchy of child library units. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.a_10.1.4</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>Alternatively, naming conflicts could be resolved via some sort of hiding rule. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9.b_10.1.4</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>For example, the implementation might support a command to import library Y into library X. If a root library unit called LU (that is, Standard.LU) exists in Y, then from the point of view of library X, it could be called Y.LU. X might contain library units that say, &ldquo;<b>with</b> Y.LU;&rdquo;. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10_10.1.4</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 3   A compilation unit containing an instantiation of a separately compiled generic unit does not semantically depend on the body of the generic unit. Therefore, replacing the generic body in the environment does not result in the removal of the compilation unit containing the instantiation. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.a_10.1.4</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>Therefore, implementations have to be prepared to automatically instantiate generic bodies at link-time, as needed. This might imply a complete automatic recompilation, but it is the intent of the language that generic bodies can be (re)instantiated without forcing all of the compilation units that semantically depend on the compilation unit containing the instantiation to be recompiled. <br />
</Admonition>
</AnnotatedOnly>

</AnnotatedOnly>

## 10.1.6  Environment-Level Visibility Rules

<MarginText>1_10.1.6</MarginText>
<p>[The normal visibility rules do not apply within a <code><a href="/docs/arm/AA-10/AA-10.1#S0291">parent_unit_name</a></code> or a <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code>, nor within a <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code> that appears at the place of a compilation unit. The special visibility rules for those contexts are given here.] <br /></p>


#### Static Semantics

<MarginText>2/2_10.1.6</MarginText>
<MarginInfo items={["AI95-00217-06", "AI95-00312-01"]} />
<p>Within the <code><a href="/docs/arm/AA-10/AA-10.1#S0291">parent_unit_name</a></code> at the beginning of an explicit <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>, and within a <code><a href="/docs/arm/AA-10/AA-10.1#S0296">nonlimited_with_clause</a></code>, the only declarations that are visible are those that are explicit <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>s of the environment, and the only declarations that are directly visible are those that are explicit root <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code>s of the environment. Within a <code><a href="/docs/arm/AA-10/AA-10.1#S0295">limited_with_clause</a></code>, the only declarations that are visible are those that are the implicit declaration of the limited view of a library package of the environment, and the only declarations that are directly visible are those that are the implicit declaration of the limited view of a root library package.<br /></p>

<AnnotatedOnly>
<MarginText>2.a_10.1.6</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>In &ldquo;<b>package</b> P.Q.R <b>is</b> ... <b>end</b> P.Q.R;&rdquo;, this rule requires P to be a root library unit, and Q to be a library unit (because those are the things that are directly visible and visible). Note that visibility does not apply between the &ldquo;<b>end</b>&rdquo; and the &ldquo;;&rdquo;.<br />
</Admonition>
</AnnotatedOnly>

<Admonition type="aarm" aarm="note">
Only compilation units defining limited views can be mentioned in a <code><a href="/docs/arm/AA-10/AA-10.1#S0295">limited_with_clause</a></code>, while only compilation units defining full views (that is, the explicit declarations) can be mentioned in a <code><a href="/docs/arm/AA-10/AA-10.1#S0296">nonlimited_with_clause</a></code>. <MarginText>3_10.1.6</MarginText>
<p>Within a <code><a href="/docs/arm/AA-8/AA-8.4#S0235">use_clause</a></code> or <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code> that is within a <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code>, each <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> mentioned in a previous <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> of the same <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code> is visible, and each root <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> so mentioned is directly visible. In addition, within such a <code><a href="/docs/arm/AA-8/AA-8.4#S0235">use_clause</a></code>, if a given declaration is visible or directly visible, each declaration that occurs immediately within the given declaration's visible part is also visible. No other declarations are visible or directly visible. <br /></p>

<AnnotatedOnly>
<MarginText>3.a_10.1.6</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>Note the word &ldquo;same&rdquo;. For example, if a <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> on a declaration mentions X, this does not make X visible in <code><a href="/docs/arm/AA-8/AA-8.4#S0235">use_clause</a></code>s and <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code>s that are on the body. The reason for this rule is the one-pass <code><a href="/docs/arm/AA-10/AA-10.1#S0292">context_clause</a></code>s Language Design Principle.<br />
</Admonition>
</AnnotatedOnly>

The ancestor root <code><a href="/docs/arm/AA-10/AA-10.1#S0287">library_item</a></code> is directly visible.<br /></p>
