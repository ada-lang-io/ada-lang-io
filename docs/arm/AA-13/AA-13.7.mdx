<MarginText>19</MarginText>
<p>Name is an enumeration subtype. Values of type Name are the names of alternative machine configurations handled by the implementation. System{"_"}Name represents the current machine configuration.<br /></p>

<MarginText>20</MarginText>
<p>The named numbers Fine{"_"}Delta and Tick are of the type <em>universal{"_"}real</em>; the others are of the type <em>universal{"_"}integer</em>.<br /></p>

<MarginText>21</MarginText>
<p>The meanings of the named numbers are: <br /></p>

<MarginText>22</MarginText>
<dt><br/>[ Min{"_"}Int </dt>
<dl>
<dd>The smallest (most negative) value allowed for the expressions of a <code><a href="../AA-3/AA-3.5#S0042">signed{"_"}integer{"_"}type{"_"}definition</a></code>.<br /></dd>
<MarginText>23</MarginText>
<dt><br/>Max{"_"}Int </dt>
<dd>The largest (most positive) value allowed for the expressions of a <code><a href="../AA-3/AA-3.5#S0042">signed{"_"}integer{"_"}type{"_"}definition</a></code>.<br /></dd>
<MarginText>24</MarginText>
<dt><br/>Max{"_"}Binary{"_"}Modulus </dt>
<dd>A power of two such that it, and all lesser positive powers of two, are allowed as the modulus of a <code><a href="../AA-3/AA-3.5#S0043">modular_type_definition</a></code>.<br /></dd>
<MarginText>25</MarginText>
<dt><br/>Max{"_"}Nonbinary{"_"}Modulus </dt>
<dd>A value such that it, and all lesser positive integers, are allowed as the modulus of a <code><a href="../AA-3/AA-3.5#S0043">modular_type_definition</a></code>. <br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>25.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>There is no requirement that Max{"_"}Nonbinary{"_"}Modulus be less than or equal to Max{"_"}Binary{"_"}Modulus, although that's what makes most sense. On a typical 32-bit machine, for example, Max{"_"}Binary{"_"}Modulus will be 2{"*"}{"*"}32 and Max{"_"}Nonbinary{"_"}Modulus will be 2{"*"}{"*"}31, because supporting nonbinary moduli in above 2{"*"}{"*"}31 causes implementation difficulties. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>26</MarginText>
<dt><br/>Max{"_"}Base{"_"}Digits </dt>
<dl>
<dd>The largest value allowed for the requested decimal precision in a <code><a href="../AA-3/AA-3.5#S0045">floating{"_"}point{"_"}definition</a></code>.<br /></dd>
<MarginText>27</MarginText>
<dt><br/>Max{"_"}Digits </dt>
<dd>The largest value allowed for the requested decimal precision in a <code><a href="../AA-3/AA-3.5#S0045">floating{"_"}point{"_"}definition</a></code> that has no <code><a href="../AA-3/AA-3.5#S0046">real{"_"}range{"_"}specification</a></code>. Max{"_"}Digits is less than or equal to Max{"_"}Base{"_"}Digits.<br /></dd>
<MarginText>28</MarginText>
<dt><br/>Max{"_"}Mantissa </dt>
<dd>The largest possible number of binary digits in the mantissa of machine numbers of a user-defined ordinary fixed point type. (The mantissa is defined in <a href="../AA-G/">Annex G</a>.)<br /></dd>
<MarginText>29</MarginText>
<dt><br/>Fine{"_"}Delta </dt>
<dd>The smallest delta allowed in an <code><a href="../AA-3/AA-3.5#S0048">ordinary_fixed_point_definition</a></code> that has the <code><a href="../AA-3/AA-3.5#S0046">real{"_"}range{"_"}specification</a></code> <strong>range</strong> &ndash;1.0 .. 1.0. ]<br /></dd>
<MarginText>30</MarginText>
<dt><br/>Tick </dt>
<dd>A period in seconds approximating the real time interval during which the value of Calendar.Clock remains constant. <br /></dd>

<MarginText>33.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>It is unspecified whether this refers to the size of the address space, the amount of physical memory on the machine, or perhaps some other interpretation of &ldquo;memory size&rdquo;. In any case, the value has to be given by a static expression, even though the amount of memory on many modern machines is a dynamic quantity in several ways. Thus, Memory{"_"}Size is not very useful. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>34/2</MarginText>
<MarginInfo items={["AI95-00161-01"]} />
<p>Address is a definite, nonlimited type with preelaborable initialization (see <a href="../AA-10/AA-10.2#Subclause_10.2.1">10.2.1</a>). Address represents machine addresses capable of addressing individual storage elements. Null{"_"}Address is an address that is distinct from the address of any object or program unit. <br /></p>

<AnnotatedOnly>
<MarginText>34.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The implementation has to ensure that there is at least one address that nothing will be allocated to; Null{"_"}Address will be one such address. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Address is the type of the result of the attribute Address. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.c</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>Address is required to be nonlimited and definite because it is important to be able to assign addresses, and to declare uninitialized address variables. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.d/5</MarginText>
<MarginInfo items={["AI95-00161-01", "AI12-0417-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>If System.Address is defined as a private type (as suggested below), it might be necessary to add the Preelaborable{"_"}Initialization aspect to the declaration of Address in order that it have preelaborable initialization as required. <br />
</Admonition>
</AnnotatedOnly>

#### Implementation Permissions

<MarginText>36/2</MarginText>
<MarginInfo items={["AI95-00362-01"]} />
<p>An implementation may add additional implementation-defined declarations to package System and its children. [However, it is usually better for the implementation to provide additional functionality via implementation-defined children of System.] <br /></p>

<AnnotatedOnly>
<MarginText>36.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The declarations in package System and its children can be implicit. For example, since Address is not limited, the predefined "=" and "/=" operations are probably sufficient. However, the implementation is not <em>required</em> to use the predefined "=". <br />
</Admonition>
</AnnotatedOnly>


#### Implementation Advice

<MarginText>37</MarginText>
<p>Address should be a private type. <br /></p>

<AnnotatedOnly>
<MarginText>37.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This promotes uniformity by avoiding having implementation-defined predefined operations for the type. We don't require it, because implementations may want to stick with what they have. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>37.a.1/2</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<strong></strong>Type System.Address should be a private type.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>37.b</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>It is not necessary for Address to be able to point at individual bits within a storage element. Nor is it necessary for it to be able to point at machine registers. It is intended as a memory address that matches the hardware's notion of an address.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>37.c</MarginText>
<Admonition type="aarm" aarm="note">
The representation of the <strong>null</strong> value of a general access type should be the same as that of Null{"_"}Address; instantiations of Unchecked{"_"}Conversion should work accordingly. If the implementation supports interfaces to other languages, the representation of the <strong>null</strong> value of a general access type should be the same as in those other languages, if appropriate.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>37.d</MarginText>
<Admonition type="aarm" aarm="note">
Note that the children of the Interfaces package will generally provide foreign-language-specific null values where appropriate. See UI-0065 regarding Null{"_"}Address. <br />
</Admonition>
</AnnotatedOnly>

#### Implementation Advice

<MarginText>16_13.7.1</MarginText>
<p>Operations in System and its children should reflect the target environment semantics as closely as is reasonable. For example, on most machines, it makes sense for address arithmetic to &ldquo;wrap around&rdquo;. Operations that do not make sense should raise Program{"_"}Error. <br /></p>

<AnnotatedOnly>
<MarginText>16.a.1/2_13.7.1</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<strong></strong>Operations in System and its children should reflect the target environment; operations that do not make sense should raise Program{"_"}Error.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16.a_13.7.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>For example, on a segmented architecture, X {"<"} Y might raise Program{"_"}Error if X and Y do not point at the same segment (assuming segments are unordered). Similarly, on a segmented architecture, the conversions between Integer{"_"}Address and Address might not make sense for some values, and so might raise Program{"_"}Error. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16.b_13.7.1</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>We considered making Storage{"_"}Element a private type. However, it is better to declare it as a modular type in the visible part, since code that uses it is already low level, and might as well have access to the underlying representation. We also considered allowing Storage{"_"}Element to be any integer type, signed integer or modular, but it is better to have uniformity across implementations in this regard, and viewing storage elements as unsigned seemed to make the most sense. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16.c_13.7.1</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>To{"_"}Address is intended for use in Address clauses. Implementations should overload To{"_"}Address if appropriate. For example, on a segmented architecture, it might make sense to have a record type representing a segment/offset pair, and have a To{"_"}Address conversion that converts from that record type to type Address. <br />
</Admonition>
</AnnotatedOnly>

## 13.7.2  The Package System.Address_To_Access_Conversions


#### Static Semantics

<MarginText>1_13.7.2</MarginText>
<p>The following language-defined generic library package exists: <br /></p>

<MarginText>2/5_13.7.2</MarginText>

<MarginText>5/2_13.7.2</MarginText>
<MarginInfo items={["AI95-00230-01"]} />
<p>The To{"_"}Pointer and To{"_"}Address subprograms convert back and forth between values of types Object{"_"}Pointer and Address. To{"_"}Pointer(X'Address) is equal to X'Unchecked{"_"}Access for any X that allows Unchecked{"_"}Access. To{"_"}Pointer(Null{"_"}Address) returns <strong>null</strong>. For other addresses, the behavior is unspecified. To{"_"}Address(<strong>null</strong>) returns Null{"_"}Address. To{"_"}Address(Y), where Y /= <strong>null</strong>, returns Y.<strong>all</strong>'Address. <br /></p>

<AnnotatedOnly>
<MarginText>5.a/3_13.7.2</MarginText>
<MarginInfo items={["AI95-00114-01", "AI05-0005-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>The programmer should ensure that the address passed to To{"_"}Pointer is either Null{"_"}Address, or the address of an object of type Object. (If Object is not a by-reference type, the object ought to be aliased; recall that the Address attribute is not required to provide a useful result for other objects.) Otherwise, the behavior of the program is unspecified; it might raise an exception or crash, for example. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b_13.7.2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>Unspecified is almost the same thing as erroneous; they both allow arbitrarily bad behavior. We don't say erroneous here, because the implementation might allow the address passed to To{"_"}Pointer to point at some memory that just happens to &ldquo;look like&rdquo; an object of type Object. That's not necessarily an error; it's just not portable. However, if the actual type passed to Object is (for example) an array type, the programmer would need to be aware of any dope that the implementation expects to exist, when passing an address that did not come from the Address attribute of an object of type Object.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.c_13.7.2</MarginText>
<Admonition type="aarm" aarm="note">
One might wonder why To{"_"}Pointer and To{"_"}Address are any better than unchecked conversions. The answer is that Address does not necessarily have the same representation as an access type. For example, an access value might point at the bounds of an array when an address would point at the first element. Or an access value might be an offset in words from someplace, whereas an address might be an offset in bytes from the beginning of memory. <br />
</Admonition>
</AnnotatedOnly>


#### Implementation Permissions

<MarginText>6_13.7.2</MarginText>
<p>An implementation may place restrictions on instantiations of Address{"_"}To{"_"}Access{"_"}Conversions. <br /></p>

<AnnotatedOnly>
<MarginText>6.a_13.7.2</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>For example, if the hardware requires aligned loads and stores, then dereferencing an access value that is not properly aligned might raise an exception.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.b_13.7.2</MarginText>
<Admonition type="aarm" aarm="note">
For another example, if the implementation has chosen to use negative component offsets (from an access value), it might not be possible to preserve the semantics, since negative offsets from the Address are not allowed. (The Address attribute always points at &ldquo;the first of the storage elements...&rdquo;.) Note that while the implementation knows how to convert an access value into an address, it might not be able to do the reverse. To avoid generic contract model violations, the restriction might have to be detected at run time in some cases. <br />
</Admonition>
</AnnotatedOnly>
