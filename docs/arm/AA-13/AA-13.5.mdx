<MarginText>1</MarginText>
<p>The <em>(record) layout</em> aspect of representation consists of the <em>storage places</em> for some or all components, that is, storage place attributes of the components. The layout can be specified with a <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>. <br /></p>

<a id="Subclause_13.5.1"></a>

## 13.5.1  Record Representation Clauses

<MarginText>1</MarginText>
<p>[A <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> specifies the storage representation of records and record extensions, that is, the order, position, and size of components (including discriminants, if any). ]<br /></p>


#### Language Design Principles

<AnnotatedOnly>
<MarginText>1.a/2</MarginText>
<MarginInfo items={["AI95-00114-01"]} />
<Admonition type="aarm" aarm="note">
It should be feasible for an implementation to use negative offsets in the representation of composite types. However, no implementation should be forced to support negative offsets. Therefore, in the interest of uniformity, negative offsets should be disallowed in <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>s. <br /></Admonition>
</AnnotatedOnly>


#### Syntax

<MarginText>2/5</MarginText>
<MarginInfo items={["AI12-0213-1"]} />

<CodeBlock>
<code>record_representation_clause</code><a id="S0352"></a><code> ::= </code><br />    <strong>for</strong> <em>first_subtype_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> <strong>use</strong><br />       <strong>record</strong> [<code><a href="../AA-J/AA-J.8#S0369">mod_clause</a></code>]<br />          {"{"}<code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>{"}"}<br />       <strong>end</strong> <strong>record</strong> [<code><a href="../AA-13/AA-13.1#S0345">local_name</a></code>];<br />

</CodeBlock>

<CodeBlock>
<code>first_bit</code><a id="S0355"></a><code> ::= </code><em>static_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code><br />

</CodeBlock>
<MarginText>6</MarginText>

<CodeBlock>
<code>last_bit</code><a id="S0356"></a><code> ::= </code><em>static_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code><br />

</CodeBlock>
<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong><code><a href="../AA-13/AA-13.5#S0355">First_bit</a></code> and <code><a href="../AA-13/AA-13.5#S0356">last_bit</a></code> need to be <code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> instead of <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> for the same reason as in <code><a href="../AA-3/AA-3.5#S0037">range</a></code> (see <a href="../AA-3/AA-3.5">3.5</a>, &ldquo;<a href="../AA-3/AA-3.5">Scalar Types</a>&rdquo;). <br /></Admonition>
</AnnotatedOnly>

<MarginText>6.1/5</MarginText>
<MarginInfo items={["AI12-0213-1"]} />
<p class="Indented2">If a <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> appears at the end of the <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>, it shall repeat the <em>first_subtype_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code>. <br /></p>


#### Name Resolution Rules

<MarginText>7</MarginText>
<p>Each <code><a href="../AA-13/AA-13.5#S0354">position</a></code>, <code><a href="../AA-13/AA-13.5#S0355">first_bit</a></code>, and <code><a href="../AA-13/AA-13.5#S0356">last_bit</a></code> is expected to be of any integer type. <br /></p>

<AnnotatedOnly>
<MarginText>7.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>These need not have the same integer type. <br /></Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>8/2</MarginText>
<MarginInfo items={["AI95-00436-01"]} />
<p>The <em>first_subtype_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> of a <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> shall denote a specific record or record extension subtype. <br /></p>

<AnnotatedOnly>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>As for all type-related representation items, the <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> is required to denote a first subtype. <br /></Admonition>
</AnnotatedOnly>

<MarginText>9</MarginText>
<p>If the <em>component_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> is a <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code>, the <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> shall denote a component of the type. For a record extension, the component shall not be inherited, and shall not be a discriminant that corresponds to a discriminant of the parent type. If the <em>component_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> has an <code><a href="../AA-4/AA-4.1#S0101">attribute_designator</a></code>, the <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code> of the <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> shall denote either the declaration of the type or a component of the type, and the <code><a href="../AA-4/AA-4.1#S0101">attribute_designator</a></code> shall denote an implementation-defined implicit component of the type.<br /></p>

<MarginText>10</MarginText>
<p>The <code><a href="../AA-13/AA-13.5#S0354">position</a></code>, <code><a href="../AA-13/AA-13.5#S0355">first_bit</a></code>, and <code><a href="../AA-13/AA-13.5#S0356">last_bit</a></code> shall be static expressions. The value of <code><a href="../AA-13/AA-13.5#S0354">position</a></code> and <code><a href="../AA-13/AA-13.5#S0355">first_bit</a></code> shall be nonnegative. The value of <code><a href="../AA-13/AA-13.5#S0356">last_bit</a></code> shall be no less than <code><a href="../AA-13/AA-13.5#S0355">first_bit</a></code> &ndash; 1. <br /></p>

<AnnotatedOnly>
<MarginText>10.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>A <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> such as &ldquo;X <strong>at</strong> 4 <strong>range</strong> 0..&ndash;1;&rdquo; is allowed if X can fit in zero bits. <br /></Admonition>
</AnnotatedOnly>

<MarginText>10.1/2</MarginText>
<MarginInfo items={["AI95-00133-01"]} />
<p>If the nondefault bit ordering applies to the type, then either: <br /></p>

<MarginText>10.2/2</MarginText>
<ul>
<li>the value of <code><a href="../AA-13/AA-13.5#S0356">last_bit</a></code> shall be less than the size of the largest machine scalar; or<br /></li>
<MarginText>10.3/2</MarginText>
<li>the value of <code><a href="../AA-13/AA-13.5#S0355">first_bit</a></code> shall be zero and the value of <code><a href="../AA-13/AA-13.5#S0356">last_bit</a></code> + 1 shall be a multiple of System.Storage_Unit. <br /></li>
</ul>
<MarginText>11</MarginText>
<p>At most one <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> is allowed for each component of the type, including for each discriminant (<code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>s may be given for some, all, or none of the components). Storage places within a <code><a href="../AA-3/AA-3.8#S0068">component_list</a></code> shall not overlap, unless they are for components in distinct <code><a href="../AA-3/AA-3.8#S0072">variant</a></code>s of the same <code><a href="../AA-3/AA-3.8#S0071">variant_part</a></code>.<br /></p>

<MarginText>12</MarginText>
<p>A name that denotes a component of a type is not allowed within a <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> for the type, except as the <em>component_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> of a <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>12.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>It might seem strange to make the <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> part of the declarative region, and then disallow mentions of the components within almost all of the <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>. The alternative would be to treat the <em>component_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> like a formal parameter name in a subprogram call (in terms of visibility). However, this rule would imply slightly different semantics, because (given the actual rule) the components can hide other declarations. This was the rule in Ada 83, and we see no reason to change it. The following, for example, was and is illegal: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.b</MarginText>
<CodeBlock language="ada">
type T is{"\n"}    record{"\n"}        X : Integer;{"\n"}    end record;{"\n"}X : constant := 31; -- Same defining name as the component.{"\n"}for T use{"\n"}    record{"\n"}        X at 0 range 0..X; -- Illegal!{"\n"}    end record;{"\n"}    {"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>12.c</MarginText>
<Admonition type="aarm" aarm="note">
The component X hides the named number X throughout the <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>. <br /></Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>13/2</MarginText>
<MarginInfo items={["AI95-00133-01"]} />
<p>A <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> (without the <code><a href="../AA-J/AA-J.8#S0369">mod_clause</a></code>) specifies the layout.<br /></p>

<AnnotatedOnly>
<MarginText>13.a/3</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Aspect Description for </strong><strong>Layout (record): </strong>Layout of record components. Specified by a <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>, not by an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>13.b/3</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Aspect Description for </strong><strong>Record layout: </strong>See Layout.<br /></Admonition>
</AnnotatedOnly>

<MarginText>13.1/2</MarginText>
<MarginInfo items={["AI95-00133-01"]} />
<p>If the default bit ordering applies to the type, the <code><a href="../AA-13/AA-13.5#S0354">position</a></code>, <code><a href="../AA-13/AA-13.5#S0355">first_bit</a></code>, and <code><a href="../AA-13/AA-13.5#S0356">last_bit</a></code> of each <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> directly specify the position and size of the corresponding component.<br /></p>

<MarginText>13.2/3</MarginText>
<MarginInfo items={["AI95-00133-01", "AI05-0264-1"]} />
<p>If the nondefault bit ordering applies to the type, then the layout is determined as follows: <br /></p>

<MarginText>13.3/2</MarginText>
<ul>
<li>the <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>s for which the value of <code><a href="../AA-13/AA-13.5#S0356">last_bit</a></code> is greater than or equal to the size of the largest machine scalar directly specify the position and size of the corresponding component;<br /></li>
<MarginText>13.4/2</MarginText>
<li>for other <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>s, all of the components having the same value of <code><a href="../AA-13/AA-13.5#S0354">position</a></code> are considered to be part of a single machine scalar, located at that <code><a href="../AA-13/AA-13.5#S0354">position</a></code>; this machine scalar has a size which is the smallest machine scalar size larger than the largest <code><a href="../AA-13/AA-13.5#S0356">last_bit</a></code> for all <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>s at that <code><a href="../AA-13/AA-13.5#S0354">position</a></code>; the <code><a href="../AA-13/AA-13.5#S0355">first_bit</a></code> and <code><a href="../AA-13/AA-13.5#S0356">last_bit</a></code> of each <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> are then interpreted as bit offsets in this machine scalar. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>13.c/2</MarginText>
<MarginInfo items={["AI95-00133-01"]} />
<Admonition type="aarm" aarm="note">
<em>This paragraph was deleted.</em><br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>13.d</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>A <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> also determines the value of the Size attribute of the component, since this attribute is related to First_Bit and Last_Bit. <br /></Admonition>
</AnnotatedOnly>

<MarginText>14</MarginText>
<p>[A <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> for a record extension does not override the layout of the parent part;] if the layout was specified for the parent type, it is inherited by the record extension. <br /></p>


#### Implementation Permissions

<MarginText>15</MarginText>
<p>An implementation may generate implementation-defined components (for example, one containing the offset of another component). An implementation may generate names that denote such implementation-defined components; such names shall be implementation-defined <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code>s. An implementation may allow such implementation-defined names to be used in <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>s. An implementation can restrict such <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>s in any manner it sees fit. <br /></p>

<AnnotatedOnly>
<MarginText>15.a</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>Implementation-defined components.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>15.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Of course, since the semantics of implementation-defined attributes is implementation defined, the implementation need not support these names in all situations. They might be purely for the purpose of <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>s, for example. The visibility rules for such names are up to the implementation.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>15.c</MarginText>
<Admonition type="aarm" aarm="note">
We do not allow such component names to be normal identifiers &mdash; that would constitute blanket permission to do all kinds of evil things. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>15.d</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Such implementation-defined components are known in the vernacular as &ldquo;dope&rdquo;. Their main purpose is for storing offsets of components that depend on discriminants. <br /></Admonition>
</AnnotatedOnly>

<MarginText>16</MarginText>
<p>If a <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> is given for an untagged derived type, the storage place attributes for all of the components of the derived type may differ from those of the corresponding components of the parent type, even for components whose storage place is not specified explicitly in the <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>16.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This is clearly necessary, since the whole record may need to be laid out differently. <br /></Admonition>
</AnnotatedOnly>


#### Implementation Advice

<MarginText>17</MarginText>
<p>The recommended level of support for <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>s is: <br /></p>

<MarginText>17.1/2</MarginText>
<MarginInfo items={["AI95-00133-01"]} />
<ul>
<li>An implementation should support machine scalars that correspond to all of the integer, floating point, and address formats supported by the machine.<br /></li>
<MarginText>18</MarginText>
<li>An implementation should support storage places that can be extracted with a load, mask, shift sequence of machine code, and set with a load, shift, mask, store sequence, given the available machine instructions and run-time model.<br /></li>
<MarginText>19</MarginText>
<li>A storage place should be supported if its size is equal to the Size of the component subtype, and it starts and ends on a boundary that obeys the Alignment of the component subtype.<br /></li>
<MarginText>20/2</MarginText>
<MarginInfo items={["AI95-00133-01"]} />
<li>For a component with a subtype whose Size is less than the word size, any storage place that does not cross an aligned word boundary should be supported.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>20.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The above recommendations are sufficient to define interfaces to most interesting hardware. This causes less implementation burden than the definition in ACID, which requires arbitrary bit alignments of arbitrarily large components. Since the ACID definition is neither enforced by the ACVC, nor supported by all implementations, it seems OK for us to weaken it. <br /></Admonition>
</AnnotatedOnly>

<MarginText>21</MarginText>
<ul>
<li>An implementation may reserve a storage place for the tag field of a tagged type, and disallow other components from overlapping that place. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>21.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Similar permission for other dope is not granted. <br /></Admonition>
</AnnotatedOnly>

<MarginText>22/5</MarginText>
<MarginInfo items={["AI12-0444-1"]} />
<ul>
<li>An implementation is not required to  support a <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> for a component of an extension part if the storage place is not after the storage places of all components of the parent type, whether or not those storage places had been specified. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>22.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>These restrictions are probably necessary if block equality operations are to be feasible for class-wide types. For block comparison to work, the implementation typically has to fill in any gaps with zero (or one) bits. If a &ldquo;gap&rdquo; in the parent type is filled in with a component in a type extension, then this won't work when a class-wide object is passed by reference, as is required. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>22.b/2</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<strong></strong>The recommended level of support for <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>s should be followed.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>23</MarginText>
<Admonition type="aarm" aarm="note">
NOTE   If no <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> is given for a component, then the choice of the storage place for the component is left to the implementation. If <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>s are given for all components, the <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> completely specifies the representation of the type and will be obeyed exactly by the implementation. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>23.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The visibility rules prevent the name of a component of the type from appearing in a <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> at any place <em>except</em> for the <em>component_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> of a <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>. However, since the <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> is part of the declarative region of the type declaration, the component names hide outer homographs throughout.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>23.b/1</MarginText>
<MarginInfo items={["AI95-00137-01"]} />
<Admonition type="aarm" aarm="note">
{"{"}<em>8652/0009</em>{"}"} A <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> cannot be given for a protected type, even though protected types, like record types, have components. The primary reason for this rule is that there is likely to be too much dope in a protected type &mdash; entry queues, bit maps for barrier values, etc. In order to control the representation of the user-defined components, simply declare a record type, give it a <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>, and give the protected type one component whose type is the record type. Alternatively, if the protected object is protecting something like a device register, it makes more sense to keep the thing being protected outside the protected object (possibly with a pointer to it in the protected object), in order to keep implementation-defined components out of the way. <br /></Admonition>
</AnnotatedOnly>


#### Examples

<MarginText>24</MarginText>
<p><em>Example of specifying the layout of a record type:</em> <br /></p>

<MarginText>25</MarginText>
<CodeBlock language="ada">
Word : constant := 4;  --  storage element is byte, 4 bytes per word{"\n"}
<MarginText>26</MarginText>
type State         is (A,M,W,P);{"\n"}type Mode          is (Fix, Dec, Exp, Signif);{"\n"}
<MarginText>27/5</MarginText>
<MarginInfo items={["AI12-0178-1"]} />
type Byte_Mask     is array (0..7)  of Boolean with Component_Size ={">"} 1;{"\n"}type State_Mask    is array (State) of Boolean with Component_Size ={">"} 1;{"\n"}type Mode_Mask     is array (Mode)  of Boolean with Component_Size ={">"} 1;{"\n"}
<MarginText>28</MarginText>

## 13.5.2  Storage Place Attributes


#### Static Semantics

<MarginText>1</MarginText>
<p>For a component C of a composite, non-array object R, the <em>storage place attributes</em> are defined: <br /></p>

<AnnotatedOnly>
<MarginText>1.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The storage place attributes are not (individually) specifiable, but the user may control their values by giving a <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>2/2</MarginText>
<dt><br/>R.C'Position</dt>
<MarginInfo items={["AI95-00133-01"]} />
<dl>
<dd>If the nondefault bit ordering applies to the composite type, and if a <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> specifies the placement of C, denotes the value given for the <code><a href="../AA-13/AA-13.5#S0354">position</a></code> of the <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>; otherwise, denotes the same value as R.C'Address &ndash; R'Address. The value of this attribute is of the type <em>universal_integer</em>. <br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>2.a/2</MarginText>
<MarginInfo items={["AI95-00133-01"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>Thus, for the default bit order, R.C'Position is the offset of C in storage elements from the beginning of the object, where the first storage element of an object is numbered zero. R'Address + R.C'Position = R.C'Address. For record extensions, the offset is not measured from the beginning of the extension part, but from the beginning of the whole object, as usual.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>2.b</MarginText>
<Admonition type="aarm" aarm="note">
In &ldquo;R.C'Address &ndash; R'Address&rdquo;, the "&ndash;" operator is the one in System.Storage_Elements that takes two Addresses and returns a Storage_Offset. <br /></Admonition>
</AnnotatedOnly>

<MarginText>3/2</MarginText>
<dt><br/>R.C'First_Bit</dt>
<MarginInfo items={["AI95-00133-01"]} />
<dl>
<dd>If the nondefault bit ordering applies to the composite type, and if a <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> specifies the placement of C, denotes the value given for the <code><a href="../AA-13/AA-13.5#S0355">first_bit</a></code> of the <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>; otherwise, denotes the offset, from the start of the first of the storage elements occupied by C, of the first bit occupied by C. This offset is measured in bits. The first bit of a storage element is numbered zero. The value of this attribute is of the type <em>universal_integer</em>.<br /></dd>
<MarginText>4/2</MarginText>
<dt><br/>R.C'Last_Bit</dt>
<MarginInfo items={["AI95-00133-01"]} />
<dd>If the nondefault bit ordering applies to the composite type, and if a <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> specifies the placement of C, denotes the value given for the <code><a href="../AA-13/AA-13.5#S0356">last_bit</a></code> of the <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>; otherwise, denotes the offset, from the start of the first of the storage elements occupied by C, of the last bit occupied by C. This offset is measured in bits. The value of this attribute is of the type <em>universal_integer</em>. <br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>4.a/2</MarginText>
<MarginInfo items={["AI95-00114-01"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>The ordering of bits in a storage element is defined in <a href="../AA-13/AA-13.5#Subclause_13.5.3">13.5.3</a>, &ldquo;<a href="../AA-13/AA-13.5#Subclause_13.5.3">Bit Ordering</a>&rdquo;.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.b</MarginText>
<Admonition type="aarm" aarm="note">
R.C'Size = R.C'Last_Bit &ndash; R.C'First_Bit + 1. (Unless the implementation chooses an indirection representation.)<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.c</MarginText>
<Admonition type="aarm" aarm="note">
If a <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> applies to a component, then that component will be at the same relative storage place in all objects of the type. Otherwise, there is no such requirement. <br /></Admonition>
</AnnotatedOnly>


#### Implementation Advice

<MarginText>5</MarginText>
<p>If a component is represented using some form of pointer (such as an offset) to the actual data of the component, and this data is contiguous with the rest of the object, then the storage place attributes should reflect the place of the actual data, not the pointer. If a component is allocated discontiguously from the rest of the object, then a warning should be generated upon reference to one of its storage place attributes. <br /></p>

<AnnotatedOnly>
<MarginText>5.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>For discontiguous components, these attributes make no sense. For example, an implementation might allocate dynamic-sized components on the heap. For another example, an implementation might allocate the discriminants separately from the other components, so that multiple objects of the same subtype can share discriminants. Such representations cannot happen if there is a <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> for that component. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b/2</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<strong></strong>If a component is represented using a pointer to the actual data of the component which is contiguous with the rest of the object, then the storage place attributes should reflect the place of the actual data. If a component is allocated discontiguously from the rest of the object, then a warning should be generated upon reference to one of its storage place attributes.<br /></Admonition>
</AnnotatedOnly>

#### Incompatibilities With Ada 95

</AnnotatedOnly>

<MarginInfo items={["AI95-00133-01"]} />

## 13.5.3  Bit Ordering

<MarginText>1</MarginText>
<p>[The Bit_Order attribute specifies the interpretation of the storage place attributes.] <br /></p>

<AnnotatedOnly>
<MarginText>1.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The intention is to provide uniformity in the interpretation of storage places across implementations on a particular machine by allowing the user to specify the Bit_Order. It is not intended to fully support data interoperability across different machines, although it can be used for that purpose in some situations.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>1.b/2</MarginText>
<MarginInfo items={["AI95-00114-01"]} />
<Admonition type="aarm" aarm="note">
We can't require all implementations on a given machine to use the same bit ordering by default; if the user cares, a Bit_Order <code><a href="../AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> can be used to force all implementations to use the same bit ordering. <br /></Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>2</MarginText>
<p>A bit ordering is a method of interpreting the meaning of the storage place attributes. High_Order_First [(known in the vernacular as &ldquo;big endian&rdquo;)] means that the first bit of a storage element (bit 0) is the most significant bit (interpreting the sequence of bits that represent a component as an unsigned integer value). Low_Order_First [(known in the vernacular as &ldquo;little endian&rdquo;)] means the opposite: the first bit is the least significant.<br /></p>

<MarginText>3/5</MarginText>
<MarginInfo items={["AI12-0373-1"]} />
<p>For every specific record subtype S, the following representation attribute is defined: <br /></p>

<MarginText>4</MarginText>
<dt><br/>S'Bit_Order</dt>
<dl>
<dd>Denotes the bit ordering for the type of S. The value of this attribute is of type System.Bit_Order. Bit_Order may be specified for specific record types via an <code><a href="../AA-13/AA-13.3#S0349">attribute_definition_clause</a></code>; the expression of such a clause shall be static.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>4.a/3</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Aspect Description for </strong><strong>Bit_Order: </strong>Order of bit numbering in a <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>.<br /></Admonition>
</AnnotatedOnly>

<MarginText>5</MarginText>
<p>If Word_Size = Storage_Unit, the default bit ordering is implementation defined. If Word_Size {">"} Storage_Unit, the default bit ordering is the same as the ordering of storage elements in a word, when interpreted as an integer. <br /></p>

<AnnotatedOnly>
<MarginText>5.a</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>If Word_Size = Storage_Unit, the default bit ordering.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Consider machines whose Word_Size = 32, and whose Storage_Unit = 8. Assume the default bit ordering applies. On a machine with big-endian addresses, the most significant storage element of an integer is at the address of the integer. Therefore, bit zero of a storage element is the most significant bit. On a machine with little-endian addresses, the least significant storage element of an integer is at the address of the integer. Therefore, bit zero of a storage element is the least significant bit. <br /></Admonition>
</AnnotatedOnly>

<MarginText>6</MarginText>
<p>The storage place attributes of a component of a type are interpreted according to the bit ordering of the type. <br /></p>

<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This implies that the interpretation of the <code><a href="../AA-13/AA-13.5#S0354">position</a></code>, <code><a href="../AA-13/AA-13.5#S0355">first_bit</a></code>, and <code><a href="../AA-13/AA-13.5#S0356">last_bit</a></code> of a <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code> of a <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> obey the bit ordering given in a representation item. <br /></Admonition>
</AnnotatedOnly>


#### Implementation Advice

<MarginText>7</MarginText>
<p>The recommended level of support for the nondefault bit ordering is: <br /></p>

<MarginText>8/2</MarginText>
<MarginInfo items={["AI95-00133-01"]} />
<ul>
<li>The implementation should support the nondefault bit ordering in addition to the default bit ordering. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>8.a/2</MarginText>
<MarginInfo items={["AI95-00133-01"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>The implementation should support both bit orderings. Implementations are required to support storage positions that cross storage element boundaries when Word_Size {">"} Storage_Unit but the definition of the storage place attributes for the nondefault bit order ensures that such storage positions will not be split into two or three pieces. Thus, there is no significant implementation burden to supporting the nondefault bit order, given that the set of machine scalars is implementation-defined. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.b/2</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<strong></strong>The recommended level of support for the nondefault bit ordering should be followed.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>9/2</MarginText>
<MarginInfo items={["AI95-00133-01"]} />
<Admonition type="aarm" aarm="note">
NOTE   Bit_Order clauses make it possible to write <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code>s that can be ported between machines having different bit ordering. They do not guarantee transparent exchange of data between such machines. <br /></Admonition>
</AnnotatedOnly>
