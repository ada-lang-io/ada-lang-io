<MarginText>1/1</MarginText>
<MarginInfo items={["AI95-00137-01"]} />
<p>{"{"}<em>8652/0009</em>{"}"} [ The values of certain implementation-dependent characteristics can be obtained by interrogating appropriate operational or representation attributes. Some of these attributes are specifiable via an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code>.] <br /></p>


#### Language Design Principles

<AnnotatedOnly>
<MarginText>1.a</MarginText>
<Admonition type="aarm" aarm="note">
In general, the meaning of a given attribute should not depend on whether the attribute was specified via an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code>, or chosen by default by the implementation. <br />
</Admonition>
</AnnotatedOnly>

#### Name Resolution Rules

<MarginText>3</MarginText>
<p>For an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> that specifies an attribute that denotes a value, the form with an <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> shall be used. Otherwise, the form with a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> shall be used.<br /></p>

<MarginText>4</MarginText>
<p>For an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> that specifies an attribute that denotes a value or an object, the expected type for the expression or <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is that of the attribute. For an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> that specifies an attribute that denotes a subprogram, the expected profile for the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is the profile required for the attribute. For an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> that specifies an attribute that denotes some other kind of entity, the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> shall resolve to denote an entity of the appropriate kind. <br /></p>

<AnnotatedOnly>
<MarginText>4.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>For example, the Size attribute is of type <em>universal{"_"}integer</em>. Therefore, the expected type for Y in &ldquo;<strong>for</strong> X'Size <strong>use</strong> Y;&rdquo; is <em>universal{"_"}integer</em>, which means that Y can be of any integer type. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.b</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>For attributes that denote subprograms, the required profile is indicated separately for the individual attributes. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.c</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>For an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> with a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code>, the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> need not statically denote the entity it denotes. For example, the following kinds of things are allowed: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.d</MarginText>
<CodeBlock language="ada">
for Some{"_"}Access{"_"}Type'Storage{"_"}Pool use Storage{"_"}Pool{"_"}Array(I);{"\n"}for Some{"_"}Type'Read use Subprogram{"_"}Pointer.all;{"\n"}

#### Legality Rules

<MarginText>5/3</MarginText>
<MarginInfo items={["AI95-00137-01", "AI05-0183-1"]} />
<p>{"{"}<em>8652/0009</em>{"}"} An <code><a href="/docs/arm/AA-4/AA-4.1#S0101">attribute_designator</a></code> is allowed in an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> only if this Reference Manual explicitly allows it, or for an implementation-defined attribute if the implementation allows it. Each specifiable attribute constitutes an operational aspect or aspect of representation; the name of the aspect is that of the attribute. <br /></p>

<AnnotatedOnly>
<MarginText>5.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>For each specifiable attribute, we generally say something like, &ldquo;The ... attribute may be specified for ... via an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code>.&rdquo;<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b</MarginText>
<Admonition type="aarm" aarm="note">
The above wording allows for T'Class'Alignment, T'Class'Size, T'Class'Input, and T'Class'Output to be specifiable.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.c</MarginText>
<Admonition type="aarm" aarm="note">
A specifiable attribute is not necessarily specifiable for all entities for which it is defined. For example, one is allowed to ask T'Component{"_"}Size for an array subtype T, but &ldquo;<strong>for</strong> T'Component{"_"}Size <strong>use</strong> ...&rdquo; is only allowed if T is a first subtype, because Component{"_"}Size is a type-related aspect. <br />
</Admonition>
</AnnotatedOnly>

#### Static Semantics

<MarginText>7/2</MarginText>
<MarginInfo items={["AI95-00270-01"]} />
<p>A <em>Size clause</em> is an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> whose <code><a href="/docs/arm/AA-4/AA-4.1#S0101">attribute_designator</a></code> is Size. Similar definitions apply to the other specifiable attributes. <br /></p>

<AnnotatedOnly>
<MarginText>7.a</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>An <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> is type-related or subtype-specific if the <code><a href="/docs/arm/AA-4/AA-4.1#S0101">attribute_designator</a></code> denotes a type-related or subtype-specific attribute, respectively. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>8</MarginText>
<p>A <em>storage element</em> is an addressable element of storage in the machine. A <em>word</em> is the largest amount of storage that can be conveniently and efficiently manipulated by the hardware, given the implementation's run-time model. A word consists of an integral number of storage elements. <br /></p>

<AnnotatedOnly>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>A storage element is not intended to be a single bit, unless the machine can efficiently address individual bits. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>For example, on a machine with 8-bit storage elements, if there exist 32-bit integer registers, with a full set of arithmetic and logical instructions to manipulate those registers, a word ought to be 4 storage elements &mdash; that is, 32 bits. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.c</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The &ldquo;given the implementation's run-time model&rdquo; part is intended to imply that, for example, on an 80386 running MS-DOS, the word might be 16 bits, even though the hardware can support 32 bits.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.d</MarginText>
<Admonition type="aarm" aarm="note">
A word is what ACID refers to as a &ldquo;natural hardware boundary&rdquo;.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.e</MarginText>
<Admonition type="aarm" aarm="note">
Storage elements may, but need not be, independently addressable (see <a href="/docs/arm/AA-9/AA-9.10">9.10</a>, &ldquo;<a href="/docs/arm/AA-9/AA-9.10">Shared Variables</a>&rdquo;). Words are expected to be independently addressable. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>8.1/3</MarginText>
<MarginInfo items={["AI95-00133-01", "AI05-0092-1"]} />
<p>A <em>machine scalar</em> is an amount of storage that can be conveniently and efficiently loaded, stored, or operated upon by the hardware. Machine scalars consist of an integral number of storage elements. The set of machine scalars is implementation defined, but includes at least the storage element and the word. Machine scalars are used to interpret <code><a href="/docs/arm/AA-13/AA-13.5#S0353">component_clause</a></code>s when the nondefault bit ordering applies. <br /></p>

<AnnotatedOnly>
<MarginText>8.e.1/2</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>The set of machine scalars.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>8.f/3</MarginText>
<MarginInfo items={["AI05-0092-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>A single storage element is a machine scalar in all Ada implementations. Similarly, a word is a machine scalar in all implementations (although it might be the same as a storage element). An implementation may define other machine scalars that make sense on the target (a half-word, for instance). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>9/5</MarginText>
<MarginInfo items={["AI95-00137-01", "AI05-0191-1", "AI12-0059-1"]} />
<p>{"{"}<em>8652/0009</em>{"}"} The following representation attributes are defined: Address, Alignment, Size, Object{"_"}Size, Storage{"_"}Size, Component{"_"}Size, Has{"_"}Same{"_"}Storage, and Overlaps{"_"}Storage.<br /></p>

<MarginText>10/1</MarginText>
<p>For a <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> X that denotes an object, program unit, or label: <br /></p>

<MarginText>11</MarginText>
<dt><br/>X'Address</dt>
<dl>
<dd>Denotes the address of the first of the storage elements allocated to X. For a program unit or label, this value refers to the machine code associated with the corresponding body or <code><a href="/docs/arm/AA-5/AA-5.1#S0167">statement</a></code>. The value of this attribute is of type System.Address.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>11.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Here, the &ldquo;first of the storage elements&rdquo; is intended to mean the one with the lowest address; the endianness of the machine doesn't matter. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>11.1/3</MarginText>
<MarginInfo items={["AI05-0095-1"]} />
<dl>
<dd>The prefix of X'Address shall not statically denote a subprogram that has convention Intrinsic. X'Address raises Program{"_"}Error if X denotes a subprogram that has convention Intrinsic.<br /></dd>
<MarginText>12</MarginText>
<dd>Address may be specified for stand-alone objects and for program units via an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code>. <br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>12.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Address is not allowed for enumeration literals, predefined operators, derived task types, or derived protected types, since they are not program units.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.b/3</MarginText>
<Admonition type="aarm" aarm="note">
Address is not allowed for intrinsic subprograms, either. That can be checked statically unless the prefix is a generic formal subprogram and the attribute reference is in the body of a generic unit. We define that case to raise Program{"_"}Error, in order that the compiler does not have to build a wrapper for intrinsic subprograms.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.c</MarginText>
<Admonition type="aarm" aarm="note">
The validity of a given address depends on the run-time model; thus, in order to use Address clauses correctly, one needs intimate knowledge of the run-time model.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.d/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<Admonition type="aarm" aarm="note">
If the Address of an object is specified, any explicit or implicit initialization takes place as usual, unless the Import aspect is also specified for the object (in which case any necessary initialization is presumably done in the foreign language).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.e</MarginText>
<Admonition type="aarm" aarm="note">
Any compilation unit containing an <code><a href="/docs/arm/AA-4/AA-4.1#S0100">attribute_reference</a></code> of a given type depends semantically on the declaration of the package in which the type is declared, even if not mentioned in an applicable <code><a href="/docs/arm/AA-10/AA-10.1#S0294">with_clause</a></code> &mdash; see <a href="/docs/arm/AA-10/AA-10.1#Subclause_10.1.1">10.1.1</a>. In this case, it means that if a compilation unit contains X'Address, then it depends on the declaration of System. Otherwise, the fact that the value of Address is of a type in System wouldn't make sense; it would violate the &ldquo;legality determinable via semantic dependences&rdquo; Language Design Principle.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.f</MarginText>
<Admonition type="aarm" aarm="note">
AI83-00305 &mdash; If X is a task type, then within the body of X, X denotes the current task object; thus, X'Address denotes the object's address.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.g</MarginText>
<Admonition type="aarm" aarm="note">
Interrupt entries and their addresses are described in <a href="/docs/arm/AA-J/AA-J.7#Subclause_J.7.1">J.7.1</a>, &ldquo;<a href="/docs/arm/AA-J/AA-J.7#Subclause_J.7.1">Interrupt Entries</a>&rdquo;.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.h</MarginText>
<Admonition type="aarm" aarm="note">
If X is not allocated on a storage element boundary, X'Address points at the first of the storage elements that contains any part of X. This is important for the definition of the Position attribute to be sensible. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>12.i/3</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Aspect Description for </strong><strong>Address: </strong>Machine address of an entity.<br />
</Admonition>
</AnnotatedOnly>


#### Erroneous Execution

<MarginText>13/3</MarginText>
<MarginInfo items={["AI05-0009-1"]} />
<p>If an Address is specified, it is the programmer's responsibility to ensure that the address is valid and appropriate for the entity and its use; otherwise, program execution is erroneous.<br /></p>

<AnnotatedOnly>
<MarginText>13.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>&ldquo;Appropriate for the entity and its use&rdquo; covers cases such as misaligned addresses, read-only code addresses for variable data objects (and nonexecutable data addresses for code units), and addresses which would force objects that are supposed to be independently addressable to not be. Such addresses may be &ldquo;valid&rdquo; as they designate locations that are accessible to the program, but the program execution is still erroneous (meaning that implementations do not have to worry about these cases). <br />
</Admonition>
</AnnotatedOnly>


#### Implementation Advice

<MarginText>14</MarginText>
<p>For an array X, X'Address should point at the first component of the array, and not at the array bounds. <br /></p>

<AnnotatedOnly>
<MarginText>14.a.1/2</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<strong></strong>For an array X, X'Address should point at the first component of the array rather than the array bounds.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>On the other hand, we have no advice to offer about discriminants and tag fields; whether or not the address points at them is not specified by the language. If discriminants are stored separately, then the Position of a discriminant might be negative, or might raise an exception. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>15</MarginText>
<p>The recommended level of support for the Address attribute is: <br /></p>

<MarginText>16</MarginText>
<ul>
<li>X'Address should produce a useful result if X is an object that is aliased or of a by-reference type, or is an entity whose Address has been specified. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>16.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>Aliased objects are the ones for which the Unchecked{"_"}Access attribute is allowed; hence, these have to be allocated on an addressable boundary anyway. Similar considerations apply to objects of a by-reference type.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16.b</MarginText>
<Admonition type="aarm" aarm="note">
An implementation need not go to any trouble to make Address work in other cases. For example, if an object X is not aliased and not of a by-reference type, and the implementation chooses to store it in a register, X'Address might return System.Null{"_"}Address (assuming registers are not addressable). For a subprogram whose calling convention is Intrinsic, or for a package, the implementation need not generate an out-of-line piece of code for it. <br />
</Admonition>
</AnnotatedOnly>

#### Language Design Principles

<AnnotatedOnly>
<MarginText>21.d</MarginText>
<Admonition type="aarm" aarm="note">
By default, the Alignment of a subtype should reflect the &ldquo;natural&rdquo; alignment for objects of the subtype on the machine. The Alignment, whether specified or default, should be known at compile time, even though Addresses are generally not known at compile time. (The generated code should never need to check at run time the number of zero bits at the end of an address to determine an alignment).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>21.e</MarginText>
<Admonition type="aarm" aarm="note">
There are two symmetric purposes of Alignment clauses, depending on whether or not the implementation has control over object allocation. If the implementation allocates an object, the implementation should ensure that the Address and Alignment are consistent with each other. If something outside the implementation allocates an object, the implementation should be allowed to assume that the Address and Alignment are consistent, but should not assume stricter alignments than that. <br />
</Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginInfo items={["AI95-00291-02"]} />

<MarginText>23/2</MarginText>
<dt><br/>X'Alignment</dt>

#### Implementation Advice

<MarginText>28.1/3</MarginText>
<MarginInfo items={["AI05-0116-1"]} />
<p>For any tagged specific subtype <em>S</em>, <em>S</em>'Class'Alignment should equal <em>S</em>'Alignment.<br /></p>

<AnnotatedOnly>
<MarginText>28.a/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>A tagged object should never be less aligned than the alignment of the type of its view, so for a class-wide type T'Class, the alignment should be no greater than that of any type covered by T'Class. If the implementation only supports alignments that are required by the recommended level of support (and this is most likely), then the alignment of any covered type has to be the same or greater than that of T &mdash; which leaves the only reasonable value of T'Class'Alignment being T'Alignment. Thus we recommend this, but don't require it so that in the unlikely case that the implementation does support smaller alignments for covered types, it can select a smaller value for T'Class'Alignment. <br />
</Admonition>

<AnnotatedOnly>
<MarginText>38/3</MarginText>
<MarginInfo items={["AI05-0229-1", "AI05-0269-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 4   A <code><a href="/docs/arm/AA-13/AA-13.5#S0353">component_clause</a></code>, Component{"_"}Size clause, or specifying the Pack aspect as True can override a specified Alignment. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>38.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>Most objects are allocated by the implementation; for these, the implementation obeys the Alignment. The implementation is of course allowed to make an object <em>more</em> aligned than its Alignment requires &mdash; an object whose Alignment is 4 might just happen to land at an address that's a multiple of 4096. For formal parameters, the implementation might want to force an Alignment stricter than the parameter's subtype. For example, on some systems, it is customary to always align parameters to 4 storage elements.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>38.b</MarginText>
<Admonition type="aarm" aarm="note">
Hence, one might initially assume that the implementation could evilly make all Alignments 1 by default, even though integers, say, are normally aligned on a 4-storage-element boundary. However, the implementation cannot get away with that &mdash; if the Alignment is 1, the generated code cannot assume an Alignment of 4, at least not for objects allocated outside the control of the implementation.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>38.c</MarginText>
<Admonition type="aarm" aarm="note">
Of course implementations can assume anything they can prove, but typically an implementation will be unable to prove much about the alignment of, say, an imported object. Furthermore, the information about where an address &ldquo;came from&rdquo; can be lost to the compiler due to separate compilation.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>38.d/3</MarginText>
<MarginInfo items={["AI95-00114-01", "AI05-0229-1"]} />
<Admonition type="aarm" aarm="note">
The Alignment of an object that is a component of a packed composite object will usually be 0, to indicate that the component is not necessarily aligned on a storage element boundary. For a subtype, an Alignment of 0 means that objects of the subtype are not normally aligned on a storage element boundary at all. For example, an implementation might choose to make Component{"_"}Size be 1 for an array of Booleans, even when the Pack aspect has not been specified for the array. In this case, Boolean'Alignment would be 0. (In the presence of tasking, this would in general be feasible only on a machine that had atomic test-bit and set-bit instructions.)<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>38.e</MarginText>
<Admonition type="aarm" aarm="note">
If the machine has no particular natural alignments, then all subtype Alignments will probably be 1 by default.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>38.f</MarginText>
<Admonition type="aarm" aarm="note">
Specifying an Alignment of 0 in an <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> does not require the implementation to do anything (except return 0 when the Alignment is queried). However, it might be taken as advice on some implementations.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>38.g</MarginText>
<Admonition type="aarm" aarm="note">
It is an error for an Address clause to disobey the object's Alignment. The error cannot be detected at compile time, in general, because the Address is not necessarily known at compile time (and is almost certainly not static). We do not require a runtime check, since efficiency seems paramount here, and Address clauses are treading on thin ice anyway. Hence, this misuse of Address clauses is just like any other misuse of Address clauses &mdash; it's erroneous.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>38.h</MarginText>
<Admonition type="aarm" aarm="note">
A type extension can have a stricter Alignment than its parent. This can happen, for example, if the Alignment of the parent is 4, but the extension contains a component with Alignment 8. The Alignment of a class-wide type or object will have to be the maximum possible Alignment of any extension.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>38.i</MarginText>
<Admonition type="aarm" aarm="note">
The recommended level of support for the Alignment attribute is intended to reflect a minimum useful set of capabilities. An implementation can assume that all Alignments are multiples of each other &mdash; 1, 2, 4, and 8 might be the only supported Alignments for subtypes. An Alignment of 3 or 6 is unlikely to be useful. For objects that can be allocated statically, we recommend that the implementation support larger alignments, such as 4096. We do not recommend such large alignments for subtypes, because the maximum subtype alignment will also have to be used as the alignment of stack frames, heap objects, and class-wide objects. Similarly, we do not recommend such large alignments for stack-allocated objects.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>38.j</MarginText>
<Admonition type="aarm" aarm="note">
If the maximum default Alignment is 8 (say, Long{"_"}Float'Alignment = 8), then the implementation can refuse to accept stricter alignments for subtypes. This simplifies the generated code, since the compiler can align the stack and class-wide types to this maximum without a substantial waste of space (or time).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>38.k</MarginText>
<Admonition type="aarm" aarm="note">
Note that the recommended level of support takes into account interactions between Size and Alignment. For example, on a 32-bit machine with 8-bit storage elements, where load and store instructions have to be aligned according to the size of the thing being loaded or stored, the implementation might accept an Alignment of 1 if the Size is 8, but might reject an Alignment of 1 if the Size is 32. On a machine where unaligned loads and stores are merely inefficient (as opposed to causing hardware traps), we would expect an Alignment of 1 to be supported for any Size. <br />
</Admonition>
</AnnotatedOnly>

</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>52.b</MarginText>
<Admonition type="aarm" aarm="note">
Note that &ldquo;<strong>for</strong> S'Size <strong>use</strong> 32;&rdquo; does not cause things like formal parameters to have Size = 32 &mdash; the implementation is allowed to make all parameters be at least 64 bits, for example.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>52.c</MarginText>
<Admonition type="aarm" aarm="note">
Note that &ldquo;<strong>for</strong> S2'Size <strong>use</strong> 5;&rdquo; requires record components whose subtype is S2 to be exactly 5 bits if the record type is packed. The same is not true of array components; their Size may be rounded up to the nearest factor of the word size. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>52.d/2</MarginText>
<MarginInfo items={["AI95-00291-02"]} />
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>On most machines, arrays don't contain gaps between elementary components; if the Component{"_"}Size is greater than the Size of the component subtype, the extra bits are generally considered part of each component, rather than gaps between components. On the other hand, a record might contain gaps between elementary components, depending on what sorts of loads, stores, and masking operations are generally done by the generated code.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>52.e/2</MarginText>
<MarginInfo items={["AI95-00291-02"]} />
<Admonition type="aarm" aarm="note">
For an array, any extra bits stored for each elementary component will generally be part of the component &mdash; the whole point of storing extra bits is to make loads and stores more efficient by avoiding the need to mask out extra bits. The PDP-10 is one counter-example; since the hardware supports byte strings with a gap at the end of each word, one would want to pack in that manner. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>52.f/5</MarginText>
<MarginInfo items={["AI12-0059-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>We talk about the explicit specification of Object{"_"}Size so that we have specified what happens if both Object{"_"}Size and Size are specified incompatibly; we give priority to Object{"_"}Size. Note that the value of Size no longer has any direct effect on the Size of objects; what happens instead is that the value of Size can have an effect on the value of Object{"_"}Size in the absence of a specification for Object{"_"}Size, and it is always the value of Object{"_"}Size that determines the size of an object. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>53</MarginText>
<p>A Size clause on a composite subtype should not affect the internal layout of components. <br /></p>

<AnnotatedOnly>
<MarginText>53.a.1/2</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<strong></strong>A Size clause on a composite subtype should not affect the internal layout of components.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>53.a/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>That's what Pack aspects, <code><a href="/docs/arm/AA-13/AA-13.5#S0352">record_representation_clause</a></code>s, and Component{"_"}Size clauses are for. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>54</MarginText>
<p>The recommended level of support for the Size attribute of subtypes is: <br /></p>

<MarginText>55/5</MarginText>
<MarginInfo items={["AI12-0445-1"]} />
<ul>
<li>The Size (if not specified) of a static discrete or fixed point subtype should be the number of bits necessary to represent each value belonging to the subtype using an unbiased representation, leaving space for a sign bit only if the subtype contains negative values. If such a subtype is a first subtype, then an implementation should support a specified Size for it that reflects this representation.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>55.a</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>This applies to static enumeration subtypes, using the internal codes used to represent the values.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.b</MarginText>
<Admonition type="aarm" aarm="note">
For a two's-complement machine, this implies that for a static signed integer subtype S, if all values of S are in the range 0 .. 2<em>n</em>&ndash;1, or all values of S are in the range &ndash;2<em>n&ndash;1</em> .. 2<em>n&ndash;1</em>&ndash;1, for some <em>n</em> less than or equal to the word size, then S'Size should be {"<"}= the smallest such <em>n</em>. For a one's-complement machine, it is the same except that in the second range, the lower bound &ldquo;&ndash;2<em>n&ndash;1</em>&rdquo; is replaced by &ldquo;&ndash;2<em>n&ndash;1</em>+1&rdquo;.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.c</MarginText>
<Admonition type="aarm" aarm="note">
If an integer subtype (whether signed or unsigned) contains no negative values, the Size should not include space for a sign bit.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.d</MarginText>
<Admonition type="aarm" aarm="note">
Typically, the implementation will choose to make the Size of a subtype be exactly the smallest such <em>n</em>. However, it might, for example, choose a biased representation, in which case it could choose a smaller value.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.e/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<Admonition type="aarm" aarm="note">
On most machines, it is in general not a good idea to pack (parts of) multiple stand-alone objects into the same storage element, because (1) it usually doesn't save much space, and (2) it requires locking to prevent tasks from interfering with each other, since separate stand-alone objects are independently addressable. Therefore, if S'Size = 2 on a machine with 8-bit storage elements, the size of a stand-alone object of subtype S will probably not be 2. It might, for example, be 8, 16 or 32, depending on the availability and efficiency of various machine instructions. The same applies to components of composite types, unless Pack, Component{"_"}Size, or record layout is specified.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.f</MarginText>
<Admonition type="aarm" aarm="note">
For an unconstrained discriminated object, if the implementation allocates the maximum possible size, then the Size attribute should return that maximum possible size. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.g</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The Size of an object X is not usually the same as that of its subtype S. If X is a stand-alone object or a parameter, for example, most implementations will round X'Size up to a storage element boundary, or more, so X'Size might be greater than S'Size. On the other hand, X'Size cannot be less than S'Size, even if the implementation can prove, for example, that the range of values actually taken on by X during execution is smaller than the range of S.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.h</MarginText>
<Admonition type="aarm" aarm="note">
For example, if S is a first integer subtype whose range is 0..3, S'Size will be probably be 2 bits, and components of packed composite types of this subtype will be 2 bits (assuming Storage{"_"}Unit is a multiple of 2), but stand-alone objects and parameters will probably not have a size of 2 bits; they might be rounded up to 32 bits, for example. On the other hand, Unchecked{"_"}Conversion will use the 2-bit size, even when converting a stand-alone object, as one would expect.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.i</MarginText>
<Admonition type="aarm" aarm="note">
Another reason for making the Size of an object bigger than its subtype's Size is to support the run-time detection of uninitialized variables. The implementation might add an extra value to a discrete subtype that represents the uninitialized state, and check for this value on use. In some cases, the extra value will require an extra bit in the representation of the object. Such detection is not required by the language. If it is provided, the implementation has to be able to turn it off. For example, if the programmer gives a <code><a href="/docs/arm/AA-13/AA-13.5#S0352">record_representation_clause</a></code> or Component{"_"}Size clause that makes a component too small to allow the extra bit, then the implementation will not be able to perform the checking (not using this method, anyway).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.j</MarginText>
<Admonition type="aarm" aarm="note">
The fact that the size of an object is not necessarily the same as its subtype can be confusing: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.k</MarginText>
<CodeBlock language="ada">
type Device{"_"}Register is range 0..2{"*"}{"*"}8 - 1;{"\n"}for Device{"_"}Register'Size use 8; -- Confusing!{"\n"}My{"_"}Device : Device{"_"}Register;{"\n"}for My{"_"}Device'Address use To{"_"}Address(16{"#"}FF00{"#"});{"\n"}  {"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>55.l/5</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
<Admonition type="aarm" aarm="note">
The programmer might think that My{"_"}Device'Size is 8, and that My{"_"}Device'Address points at an 8-bit location. However, this is not guaranteed to be true. In Ada 83 (and in Ada 95), My{"_"}Device'Size might well be 32, and My{"_"}Device'Address might well point at the high-order 8 bits of the 32-bit object, which are always all zero bits. If My{"_"}Device'Address is passed to an assembly language subprogram, based on the programmer's assumption, the program will not work properly. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.m</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>It is not reasonable to require that an implementation allocate exactly 8 bits to all objects of subtype Device{"_"}Register. For example, in many run-time models, stand-alone objects and parameters are always aligned to a word boundary. Such run-time models are generally based on hardware considerations that are beyond the control of the implementer. (It is reasonable to require that an implementation allocate exactly 8 bits to all components of subtype Device{"_"}Register, if packed.) <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.n</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The correct way to write the above code is like this: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.o</MarginText>
<CodeBlock language="ada">
type Device{"_"}Register is range 0..2{"*"}{"*"}8 - 1;{"\n"}My{"_"}Device : Device{"_"}Register;{"\n"}for My{"_"}Device'Size use 8;{"\n"}for My{"_"}Device'Address use To{"_"}Address(16{"#"}FF00{"#"});{"\n"}  {"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>55.p</MarginText>
<Admonition type="aarm" aarm="note">
If the implementation cannot accept 8-bit stand-alone objects, then this will be illegal. However, on a machine where an 8-bit device register exists, the implementation will probably be able to accept 8-bit stand-alone objects. Therefore, My{"_"}Device'Size will be 8, and My{"_"}Device'Address will point at those 8 bits, as desired.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>55.q</MarginText>
<Admonition type="aarm" aarm="note">
If an object of subtype Device{"_"}Register is passed to a foreign language subprogram, it will be passed according to that subprogram's conventions. Most foreign language implementations have similar run-time model restrictions. For example, when passing to a C function, where the argument is of the C type char{"*"} (that is, pointer to char), the C compiler will generally expect a full word value, either on the stack, or in a register. It will <em>not</em> expect a single byte. Thus, Size clauses for subtypes really have nothing to do with passing parameters to foreign language subprograms. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>56</MarginText>
<ul>
<li>For a subtype implemented with levels of indirection, the Size should include the size of the pointers, but not the size of what they point at. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>56.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>For example, if a task object is represented as a pointer to some information (including a task stack), then the size of the object should be the size of the pointer. The Storage{"_"}Size, on the other hand, should include the size of the stack. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>56.1/2</MarginText>
<MarginInfo items={["AI95-00051-02"]} />
<ul>
<li>An implementation should support a Size clause for a discrete type, fixed point type, record type, or array type, subject to the following: <br /></li>
</ul>
<MarginText>56.2/5</MarginText>
<MarginInfo items={["AI95-00051-02", "AI12-0444-1"]} />
<ul>
<li>An implementation is not required to support a Size clause for a signed integer type specifying a Size greater than that of the largest signed integer type supported by the implementation in the absence of a size clause (that is, when the size is chosen by default). A corresponding limitation may be imposed for modular integer types, fixed point types, enumeration types, record types, and array types.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>56.b/2</MarginText>
<MarginInfo items={["AI95-00051-02"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>Note that the &ldquo;corresponding limitation&rdquo; for a record or array type implies that an implementation may impose some reasonable maximum size for records and arrays (e.g. 2{"*"}{"*"}32 bits), which is an upper bound (&ldquo;capacity&rdquo; limit) on the size, whether chosen by default or by being specified by the user. The largest size supported for records need not be the same as the largest size supported for arrays. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>56.b.1/3</MarginText>
<MarginInfo items={["AI05-0155-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>Only Size clauses with a size greater than or equal to the Size that would be chosen by default may be safely presumed to be supported on nonstatic elementary subtypes. Implementations may choose to support smaller sizes, but only if the Size allows any value of the subtype to be represented, for any possible value of the bounds. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>56.3/5</MarginText>
<MarginInfo items={["AI95-00291-02", "AI12-0444-1"]} />
<ul>
<li>A nonconfirming size clause for the first subtype of a derived untagged by-reference type is not required to be supported. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>56.c/2</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<strong></strong>The recommended level of support for the Size attribute should be followed.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>56.d/2</MarginText>
<MarginInfo items={["AI95-00291-02"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>There is no recommendation to support any nonconfirming Size clauses for types not mentioned above. Remember that <a href="/docs/arm/AA-13/AA-13.1">13.1</a> requires support for confirming Size clauses for all types. <br />
</Admonition>

#### Static Semantics

<MarginText>58.1/5</MarginText>
<MarginInfo items={["AI12-0059-1"]} />
<p>For every subtype S: <br /></p>

<MarginText>58.2/5</MarginText>
<dt><br/>S'Object{"_"}Size</dt>
<MarginInfo items={["AI12-0059-1"]} />
<dl>
<dd>If S is definite, denotes the size (in bits) of a stand-alone aliased object, or a component of subtype S in the absence of an <code><a href="/docs/arm/AA-13/AA-13.1#S0346">aspect_specification</a></code> or representation item that specifies the size of the object or component. </dl>

<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>The meaning of Object{"_"}Size for indefinite subtypes.<br />
</Admonition>
</AnnotatedOnly>

#### Static Semantics

<MarginText>59/1</MarginText>
<p>For a <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> T that denotes a task object [(after any implicit dereference)]: <br /></p>

<MarginText>60/3</MarginText>
<dt><br/>T'Storage{"_"}Size</dt>
<MarginInfo items={["AI05-0229-1"]} />
<dl>
<dd>Denotes the number of storage elements reserved for the task. The value of this attribute is of the type <em>universal{"_"}integer</em>. The Storage{"_"}Size includes the size of the task's stack, if any. The language does not specify whether or not it includes other storage associated with the task (such as the &ldquo;task control block&rdquo; used by some implementations.) #### Implementation Advice

<MarginText>71</MarginText>
<p>The recommended level of support for the Component{"_"}Size attribute is: <br /></p>

<MarginText>72/5</MarginText>
<MarginInfo items={["AI12-0444-1"]} />
<ul>
<li>An implementation is not required to support specified Component{"_"}Sizes that are less than the Size of the component subtype.<br /></li>
<MarginText>73/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<li>An implementation should support specified Component{"_"}Sizes that are factors and multiples of the word size. For such Component{"_"}Sizes, the array should contain no gaps between components. For other Component{"_"}Sizes (if supported), the array should contain no gaps between components when Pack is also specified; the implementation should forbid this combination in cases where it cannot support a no-gaps representation. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>73.a/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>For example, if Storage{"_"}Unit = 8, and Word{"_"}Size = 32, then the user is allowed to specify a Component{"_"}Size of 1, 2, 4, 8, 16, and 32, with no gaps. In addition, <em>n</em>{"*"}32 is allowed for positive integers <em>n</em>, again with no gaps. If the implementation accepts Component{"_"}Size = 3, then it might allocate 10 components per word, with a 2-bit gap at the end of each word (unless Pack is also specified), or it might not have any internal gaps at all. (There can be gaps at either end of the array.) <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>73.b/2</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<strong></strong>The recommended level of support for the Component{"_"}Size attribute should be followed.<br />
</Admonition>
</AnnotatedOnly>

#### Dynamic Semantics

<MarginText>75.1/3</MarginText>
<MarginInfo items={["AI05-0113-1"]} />
<p>If a user-specified external tag S'External{"_"}Tag is the same as T'External{"_"}Tag for some other tagged type declared by a different declaration in the partition, Program{"_"}Error is raised by the elaboration of the <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code>.<br /></p>

<AnnotatedOnly>
<MarginText>75.c/3</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This rule does not depend on the visibility of the other tagged type, but it does depend on the existence of the other tagged type. The other tagged type could have the default external tag or a user-specified external tag.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>75.d/3</MarginText>
<Admonition type="aarm" aarm="note">
This rule allows the same declaration to be elaborated multiple times. In that case, different types could have the same external tag. If that happens, Internal{"_"}Tag would return some unspecified tag, and Descendant{"_"}Tag probably would return the intended tag (using the given ancestor to determine which type is intended). However, in some cases (such as multiple instantiations of a derived tagged type declared in a generic body), Tag{"_"}Error might be raised by Descendant{"_"}Tag if multiple types are identified.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>75.e/3</MarginText>
<Admonition type="aarm" aarm="note">
Note that while there is a race condition inherent in this definition (which <code><a href="/docs/arm/AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> raises Program{"_"}Error depends on the order of elaboration), it doesn't matter as a program with two such clauses is simply wrong. Two types that both come from the same declaration are allowed, as noted previously. <br />
</Admonition>
</AnnotatedOnly>


#### Implementation Requirements

<MarginText>76</MarginText>
<p>In an implementation, the default external tag for each specific tagged type declared in a partition shall be distinct, so long as the type is declared outside an instance of a generic body. If the compilation unit in which a given tagged type is declared, and all compilation units on which it semantically depends, are the same in two different partitions, then the external tag for the type shall be the same in the two partitions. What it means for a compilation unit to be the same in two different partitions is implementation defined. At a minimum, if the compilation unit is not recompiled between building the two different partitions that include it, the compilation unit is considered the same in the two partitions. <br /></p>

<AnnotatedOnly>
<MarginText>76.a</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>What determines whether a compilation unit is the same in two different partitions.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>76.b</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>These requirements are important because external tags are used for input/output of class-wide types. These requirements ensure that what is written by one program can be read back by some other program so long as they share the same declaration for the type (and everything it depends on).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>76.c</MarginText>
<Admonition type="aarm" aarm="note">
The user may specify the external tag if (s)he wishes its value to be stable even across changes to the compilation unit in which the type is declared (or changes in some unit on which it depends).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>76.d/2</MarginText>
<MarginInfo items={["AI95-00114-01"]} />
<Admonition type="aarm" aarm="note">
We use a String rather than a Stream{"_"}Element{"_"}Array to represent an external tag for portability. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>76.e</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Note that the characters of an external tag need not all be graphic characters. In other words, the external tag can be a sequence of arbitrary 8-bit bytes. <br />
</Admonition>
</AnnotatedOnly>

<strong></strong> User-specified external tags that conflict with other external tags raise Program{"_"}Error (or are optionally illegal). This was legal and did not raise an exception in the past, although the effects were not defined.
