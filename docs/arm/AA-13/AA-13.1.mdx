<MarginText>0.1/3</MarginText>
<MarginInfo items={["AI95-00137-01", "AI05-0295-1"]} />
<p>{"{"}<em>8652/0009</em>{"}"} [Two kinds of aspects of entities can be specified: representation aspects and operational aspects. Representation aspects affect how the types and other entities of the language are to be mapped onto the underlying machine. Operational aspects determine other properties of entities.]<br /></p>

<AnnotatedOnly>
<MarginText>0.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>operational aspect</strong> &mdash; aspect that indicates a logical property of an entity, such as the precondition of a subprogram, or the procedure used to write a given type of object to a stream<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>0.b/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>representation aspect</strong> &mdash; aspect that indicates how an entity is mapped onto the underlying hardware, for example the size or alignment of an object<br />
</Admonition>
</AnnotatedOnly>

<MarginText>0.2/3</MarginText>
<MarginInfo items={["AI05-0183-1", "AI05-0295-1"]} />
<p>[Either kind of aspect of an entity may be specified by means of an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> (see <a href="../AA-13/AA-13.1#Subclause_13.1.1">13.1.1</a>), which is an optional element of most kinds of declarations and applies to the entity or entities being declared. Aspects may also be specified by certain other constructs occurring subsequent to the declaration of the affected entity: a representation aspect value may be specified by means of a representation item and an operational aspect value may be specified by means of an operational item.]<br /></p>

<MarginText>1/1</MarginText>
<MarginInfo items={["AI95-00137-01"]} />
<p>{"{"}<em>8652/0009</em>{"}"} There are six kinds of <em>representation items</em>: <code><a href="../AA-13/AA-13.3#S0349">attribute{"_"}definition{"_"}clause</a></code>s for representation attributes, <code><a href="../AA-13/AA-13.4#S0350">enumeration{"_"}representation{"_"}clause</a></code>s, <code><a href="../AA-13/AA-13.5#S0352">record{"_"}representation{"_"}clause</a></code>s, <code><a href="../AA-J/AA-J.7#S0368">at_clause</a></code>s, <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>s, and <em>representation pragmas</em>. [ They can be provided to give more efficient representation or to interface with features that are outside the domain of the language (for example, peripheral hardware). ]<br /></p>

<MarginText>1.1/1</MarginText>
<MarginInfo items={["AI95-00137-01"]} />
<p>{"{"}<em>8652/0009</em>{"}"} An <em>operational item</em> is an <code><a href="../AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> for an operational attribute.<br /></p>

<MarginText>1.2/1</MarginText>
<MarginInfo items={["AI95-00137-01"]} />
<p>{"{"}<em>8652/0009</em>{"}"} [An operational item or a representation item applies to an entity identified by a <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code>, which denotes an entity declared local to the current declarative region, or a library unit declared immediately preceding a representation pragma in a <code><a href="../AA-10/AA-10.1#S0285">compilation</a></code>.] <br /></p>


#### Language Design Principles

<AnnotatedOnly>
<MarginText>1.a/3</MarginText>
<MarginInfo items={["AI95-00137-01", "AI05-0295-1"]} />
<Admonition type="aarm" aarm="note">
{"{"}<em>8652/0009</em>{"}"} Representation aspects are intended to refer to properties that need to be known before the compiler can generate code to create or access an entity. For instance, the size of an object needs to be known before the object can be created. Conversely, operational aspects are those that only need to be known before they can be used. For instance, how an object is read from a stream only needs to be known when a stream read is executed. Thus, representation aspects have stricter rules as to when they can be specified.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>1.a.1/3</MarginText>
<MarginInfo items={["AI95-00291-02", "AI05-0295-1"]} />
<Admonition type="aarm" aarm="note">
Confirming the value of an aspect should never change the semantics of the aspect. Thus Size = 8 (for example) means the same thing whether it was specified with a representation item or whether the compiler chose this value by default. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>1.b/5</MarginText>
<Admonition type="aarm" aarm="note">
<strong>Term entry: </strong><strong>aspect</strong> &mdash; specifiable property of an entity<br /><code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> on the declaration of the entity. Some aspects can be queried via attributes.<br />
</Admonition>
</AnnotatedOnly>

#### Name Resolution Rules

<MarginText>5/1</MarginText>
<MarginInfo items={["AI95-00137-01"]} />
<p>{"{"}<em>8652/0009</em>{"}"} In an operational item or representation item, if the <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> is a <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code>, then it shall resolve to denote a declaration (or, in the case of a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>, one or more declarations) that occurs immediately within the same declarative region as the item. If the <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> has an <code><a href="../AA-4/AA-4.1#S0101">attribute_designator</a></code>, then it shall resolve to denote an implementation-defined component (see <a href="../AA-13/AA-13.5#Subclause_13.5.1">13.5.1</a>) or a class-wide type implicitly declared immediately within the same declarative region as the item. A <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> that is a <em>library{"_"}unit{"_"}</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> (only permitted in a representation pragma) shall resolve to denote the <code><a href="../AA-10/AA-10.1#S0287">library_item</a></code> that immediately precedes (except for other pragmas) the representation pragma. <br /></p>

<AnnotatedOnly>
<MarginText>5.a/1</MarginText>
<MarginInfo items={["AI95-00137-01"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>{"{"}<em>8652/0009</em>{"}"} This is a Name Resolution Rule, because we don't want an operational or representation item for X to be ambiguous just because there's another X declared in an outer declarative region. It doesn't make much difference, since most operational or representation items are for types or subtypes, and type and subtype names can't be overloaded. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.b/1</MarginText>
<MarginInfo items={["AI95-00137-01"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>{"{"}<em>8652/0009</em>{"}"} The visibility rules imply that the declaration has to occur before the operational or representation item.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>5.c/1</MarginText>
<MarginInfo items={["AI95-00137-01"]} />
<Admonition type="aarm" aarm="note">
{"{"}<em>8652/0009</em>{"}"} For objects, this implies that operational or representation items can be applied only to stand-alone objects. <br />
</Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>6/1</MarginText>
<MarginInfo items={["AI95-00137-01"]} />
<p>{"{"}<em>8652/0009</em>{"}"} The <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> of an <code><a href="../AA-13/AA-13.1#S0343">aspect_clause</a></code> or representation pragma shall statically denote an entity (or, in the case of a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>, one or more entities) declared immediately preceding it in a <code><a href="../AA-10/AA-10.1#S0285">compilation</a></code>, or within the same <code><a href="../AA-3/AA-3.11#S0086">declarative{"_"}part</a></code>, <code><a href="../AA-7/AA-7.1#S0230">package{"_"}specification</a></code>, <code><a href="../AA-9/AA-9.1#S0246">task{"_"}definition</a></code>, <code><a href="../AA-9/AA-9.4#S0251">protected{"_"}definition</a></code>, or <code><a href="../AA-3/AA-3.8#S0067">record{"_"}definition</a></code> as the representation or operational item. If a <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> denotes a [local] callable entity, it may do so through a [local] <code><a href="../AA-8/AA-8.5#S0242">subprogram{"_"}renaming{"_"}declaration</a></code> [(as a way to resolve ambiguity in the presence of overloading)]; otherwise, the <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> shall not denote a <code><a href="../AA-8/AA-8.5#S0238">renaming{"_"}declaration</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The &ldquo;statically denote&rdquo; part implies that it is impossible to specify the representation of an object that is not a stand-alone object, except in the case of a representation item like pragma Atomic that is allowed inside a <code><a href="../AA-3/AA-3.8#S0068">component_list</a></code> (in which case the representation item specifies the representation of components of all objects of the type). It also prevents the problem of renamings of things like &ldquo;P.<strong>all</strong>&rdquo; (where P is an access-to-subprogram value) or &ldquo;E(I)&rdquo; (where E is an entry family).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.b</MarginText>
<Admonition type="aarm" aarm="note">
The part about where the denoted entity has to have been declared appears twice &mdash; once as a Name Resolution Rule, and once as a Legality Rule. Suppose P renames Q, and we have a representation item in a <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code> whose <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> is P. The fact that the representation item has to appear in the same <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code> as P is a Name Resolution Rule, whereas the fact that the representation item has to appear in the same <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code> as Q is a Legality Rule. This is subtle, but it seems like the least confusing set of rules. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.c</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>A separate Legality Rule applies for <code><a href="../AA-13/AA-13.5#S0353">component_clause</a></code>s. See <a href="../AA-13/AA-13.5#Subclause_13.5.1">13.5.1</a>, &ldquo;<a href="../AA-13/AA-13.5#Subclause_13.5.1">Record Representation Clauses</a>&rdquo;. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>7/5</MarginText>
<MarginInfo items={["AI95-00291-02", "AI12-0445-1"]} />
<p>The <em>representation</em> of an object consists of a certain number of bits (the <em>size</em> of the object). For an object of an elementary type, these are the bits that are normally read or updated by the machine code when loading, storing, or operating-on the value of the object. For an object of a composite type, these are the bits reserved for this object, and include bits occupied by subcomponents of the object. If the size of an object is greater than that of its subtype, the additional bits are padding bits. For an elementary object, these padding bits are normally read and updated along with the others. For a composite object, it is unspecified whether padding bits are read or updated in any given composite operation.<br /></p>

<AnnotatedOnly>
<MarginText>7.a/2</MarginText>
<MarginInfo items={["AI95-00291-02"]} />
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>Discontiguous representations are allowed, but the ones we're interested in here are generally contiguous sequences of bits. For a discontiguous representation, the size doesn't necessarily describe the &ldquo;footprint&rdquo; of the object in memory (that is, the amount of space taken in the address space for the object). <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.a.1/2</MarginText>
<MarginInfo items={["AI95-00291-02"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>In the case of composite objects, we want the implementation to have the flexibility to either do operations component-by-component, or with a block operation covering all of the bits. We carefully avoid giving a preference in the wording. There is no requirement for the choice to be documented, either, as the implementation can make that choice based on many factors, and could make a different choice for different operations on the same object.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.a.2/2</MarginText>
<MarginInfo items={["AI95-00291-02"]} />
<Admonition type="aarm" aarm="note">
In the case of a properly aligned, contiguous object whose size is a multiple of the storage unit size, no other bits should be read or updated as part of operating on the object. We don't say this normatively because it would be difficult to normatively define &ldquo;properly aligned&rdquo; or &ldquo;contiguous&rdquo;. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Two objects with the same value do not necessarily have the same representation. For example, an implementation might represent False as zero and True as any odd value. Similarly, two objects (of the same type) with the same sequence of bits do not necessarily have the same value. For example, an implementation might use a biased representation in some cases but not others: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.c/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<CodeBlock language="ada">
subtype S is Integer range 1..256;{"\n"}type A is array(Natural range 1..4) of S{"\n"}   with Pack;{"\n"}X : S := 3;{"\n"}Y : A := (1, 2, 3, 4);{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>7.d</MarginText>
<Admonition type="aarm" aarm="note">
The implementation might use a biased-by-1 representation for the array elements, but not for X. X and Y(3) have the same value, but different representation: the representation of X is a sequence of (say) 32 bits: 0...011, whereas the representation of Y(3) is a sequence of 8 bits: 00000010 (assuming a two's complement representation).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.e</MarginText>
<Admonition type="aarm" aarm="note">
Such tricks are not required, but are allowed. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.f</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The value of any padding bits is not specified by the language, though for a numeric type, it will be much harder to properly implement the predefined operations if the padding bits are not either all zero, or a sign extension. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.g/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>For example, suppose S'Size = 2, and an object X is of subtype S. If the machine code typically uses a 32-bit load instruction to load the value of X, then X'Size should be 32, even though 30 bits of the value are just zeros or sign-extension bits. On the other hand, if the machine code typically masks out those 30 bits, then X'Size should be 2. Usually, such masking only happens for components of a composite type for which Pack, Component{"_"}Size, or record layout is specified.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.h</MarginText>
<Admonition type="aarm" aarm="note">
Note, however, that the formal parameter of an instance of Unchecked{"_"}Conversion is a special case. Its Size is required to be the same as that of its subtype.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.i</MarginText>
<Admonition type="aarm" aarm="note">
Note that we don't generally talk about the representation of a value. A value is considered to be an amorphous blob without any particular representation. An object is considered to be more concrete. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>8/5</MarginText>
<MarginInfo items={["AI05-0112-1", "AI05-0295-1", "AI12-0396-1", "AI12-0407-1"]} />
<p>A representation item <em>directly specifies</em> a <em>representation aspect</em> of the entity denoted by the <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code>, except in the case of a type-related representation item, whose <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> shall denote a first subtype, and which directly specifies an aspect of the subtype's type. A representation item that names a subtype is either <em>subtype-specific</em> (Size, Object{"_"}Size, and Alignment clauses) or <em>type-related</em> (all others).<br /></p>

Each representation pragma specifies a separate aspect. <br />
</Admonition>

See <a href="../AA-J/AA-J.7">J.7</a>, &ldquo;<a href="../AA-J/AA-J.7">At Clauses</a>&rdquo;, and <a href="../AA-J/AA-J.8">J.8</a>, &ldquo;<a href="../AA-J/AA-J.8">Mod Clauses</a>&rdquo;.<br />
</Admonition>

<AnnotatedOnly>
<MarginText>8.e.1/3</MarginText>
<MarginInfo items={["AI05-0112-1"]} />
<Admonition type="aarm" aarm="note">
We give a default naming for representation aspects of representation pragmas so we don't have to do that for every pragma. Operational and representation attributes are given a default naming in <a href="../AA-13/AA-13.3">13.3</a>. We don't want any anonymous aspects; that would make other rules more difficult to write and understand. <br />
</Admonition>

<MarginText>9.3/5</MarginText>
<MarginInfo items={["AI05-0106-1", "AI05-0295-1", "AI12-0116-1", "AI12-0181-1"]} />
<p>Unless otherwise specified, it is illegal to specify an operational or representation aspect of a generic formal parameter. <br /></p>

<AnnotatedOnly>
<MarginText>9.d/5</MarginText>
<MarginInfo items={["AI12-0064-2"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>Specifying an aspect on a generic formal parameter implies an added contract for a generic unit. That contract needs to be defined via generic parameter matching rules, and, as aspects vary widely, that has to be done for each such aspect. Since most aspects do not need this complexity, we avoid the complexity by saying that such contract-forming aspect specifications are banned unless the rules defining them explicitly exist. Note that the method of specification does not matter: <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>s, representation items, and operational items are all covered by this rule. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>10/5</MarginText>
<MarginInfo items={["AI05-0295-1", "AI12-0109-1", "AI12-0376-1", "AI12-0427-1"]} />
<p>A <em>by-reference primitive</em> is a user-defined primitive subprogram for a type <em>T</em> that has an access result designating type <em>T</em>, or that has a formal parameter that is an access parameter designating type <em>T</em> or is aliased and of type <em>T</em>. It is illegal to specify a nonconfirming type-related representation aspect for an untagged type <em>T</em> if it is derived from a by-reference type or inherits one or more by-reference primitives, or if one or more types have been derived from <em>T</em> prior to the specification of the aspect and type <em>T</em> is a by-reference type or defines one or more by-reference primitives that are inherited by these descendants.<br /></p>

<AnnotatedOnly>
<MarginText>10.a/3</MarginText>
<MarginInfo items={["AI95-00137-01", "AI05-0295-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>{"{"}<em>8652/0009</em>{"}"} On the other hand, subtype-specific representation aspects may be specified for the first subtype of such a type, as can operational aspects. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.b/5</MarginText>
<MarginInfo items={["AI05-0229-1", "AI05-0295-1", "AI12-0109-1", "AI12-0376-1", "AI12-0427-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>The reason for forbidding specification of type-related representation aspects on untagged by-reference types is because a change of representation is impossible when passing by reference (to an inherited subprogram). (A by-reference object cannot be copied to change its representation.) The reason for forbidding specification of type-related representation aspects on untagged types with by-reference primitives is that access parameters, access results, and aliased parameters cannot be converted as part of invoking an inherited subprogram if the representation of the designated types might be different. This rule is not needed for tagged types, because other rules prevent a type-related representation aspect from changing the representation of the parent part; we want to allow specifying a type-related representation aspect on a type extension to specify aspects of the extension part. For example, specifying aspect Pack will cause packing of the extension part, but not of the parent part. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>10.c/4</MarginText>
<MarginInfo items={["AI12-0109-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>&ldquo;By-reference type&rdquo; usually cannot be used in Legality Rules, as it is privacy breaking. Our use here is privacy breaking, but we're stuck with it for compatibility reasons. Since representation aspects cannot be specified on partial views, privacy violations only can happen when a type includes a component of a private type. In that case, whether these rules are triggered depends on the full type of the private type &mdash; which is clearly privacy breaking. <br />
</Admonition>
</AnnotatedOnly>

#### Static Semantics

<MarginText>14/5</MarginText>
<MarginInfo items={["AI12-0059-1"]} />
<p>If two subtypes statically match, then their subtype-specific aspects (for example, Size and Alignment) are the same. <br /></p>

<AnnotatedOnly>
<MarginText>14.a/5</MarginText>
<MarginInfo items={["AI05-0295-1", "AI12-0059-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>This is necessary because we allow (for example) conversion between access types whose designated subtypes statically match. Note that most aspects (including the subtype-specific aspects Size and Alignment) may not be specified for a nonfirst subtype. The only language-defined exceptions to this rule are the Object{"_"}Size, Static{"_"}Predicate, and Dynamic{"_"}Predicate aspects.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.b</MarginText>
<Admonition type="aarm" aarm="note">
Consider, for example: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.c/5</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
<CodeBlock language="ada">
package P1 is{"\n"}   subtype S1 is Integer range 0..2{"*"}{"*"}16-1{"\n"}      with Size ={">"} 16; -- Illegal!{"\n"}      -- S1'Size would be 16 by default.{"\n"}   type A1 is access all S1;{"\n"}   X1: A1;{"\n"}end P1;{"\n"}
<MarginText>14.d/5</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
package P2 is{"\n"}   subtype S2 is Integer range 0..2{"*"}{"*"}16-1{"\n"}      with Size ={">"} 32; -- Illegal!{"\n"}   type A2 is access all S2;{"\n"}   X2: A2;{"\n"}end P2;{"\n"}
<MarginText>14.e/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
procedure Q is{"\n"}   use P1, P2;{"\n"}   type Array1 is array(Integer range {"<"}{">"}) of aliased S1{"\n"}      with Pack;{"\n"}   Obj1: Array1(1..100);{"\n"}   type Array2 is array(Integer range {"<"}{">"}) of aliased S2{"\n"}      with Pack;{"\n"}   Obj2: Array2(1..100);{"\n"}begin{"\n"}   X1 := Obj2(17)'Unchecked{"_"}Access;{"\n"}   X2 := Obj1(17)'Unchecked{"_"}Access;{"\n"}end Q;{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>14.f</MarginText>
<Admonition type="aarm" aarm="note">
Loads and stores through X1 would read and write 16 bits, but X1 points to a 32-bit location. Depending on the endianness of the machine, loads might load the wrong 16 bits. Stores would fail to zero the other half in any case.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.g</MarginText>
<Admonition type="aarm" aarm="note">
Loads and stores through X2 would read and write 32 bits, but X2 points to a 16-bit location. Thus, adjacent memory locations would be trashed.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.h</MarginText>
<Admonition type="aarm" aarm="note">
Hence, the above is illegal. Furthermore, the compiler is forbidden from choosing different Sizes by default, for the same reason.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.i/5</MarginText>
<MarginInfo items={["AI12-0059-1"]} />
<Admonition type="aarm" aarm="note">
The same issues apply to Alignment. Similar issues apply to Object{"_"}Size, Static{"_"}Predicate, and Dynamic{"_"}Predicate, but the chosen solution is different: explicit rules in <a href="../AA-4/AA-4.9#Subclause_4.9.1">4.9.1</a> that ensure that the aspects are the same if specified.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.j/5</MarginText>
<MarginInfo items={["AI12-0059-1"]} />
<Admonition type="aarm" aarm="note">
The above static matching rule combined with the definition of static matching in <a href="../AA-4/AA-4.9#Subclause_4.9.1">4.9.1</a> (after updating to add Object{"_"}Size) does not cause incompatibilities in existing Ada 2012 code that does not mention Object{"_"}Size. But it does constrain the implementation-defined value of Object{"_"}Size when it is not specified for a subtype; the above rule applies even in that case. (The effects noted in the example above would occur otherwise.)<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>14.k/5</MarginText>
<MarginInfo items={["AI12-0059-1"]} />
<Admonition type="aarm" aarm="note">
We need this rule even though static matching explicitly excludes confirming values of Object{"_"}Size. That's because a general access type can designate any aliased object whose subtype statically matches the designated subtype. Since the Object{"_"}Size of a subtype determines the number of bits allocated for an aliased object of the subtype, if we allowed different Object{"_"}Sizes for statically matching subtypes, we'd be allowing the access type to designate objects with differing numbers of bits. That isn't going to work. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>15/3</MarginText>
<MarginInfo items={["AI95-00108-01", "AI05-0009-1", "AI05-0295-1"]} />
<p>{"{"}<em>8652/0040</em>{"}"} A derived type inherits each type-related representation aspect of its parent type that was directly specified before the declaration of the derived type, or (in the case where the parent is derived) that was inherited by the parent type from the grandparent type. A derived subtype inherits each subtype-specific representation aspect of its parent subtype that was directly specified before the declaration of the derived type, or (in the case where the parent is derived) that was inherited by the parent subtype from the grandparent subtype, but only if the parent subtype statically matches the first subtype of the parent type. An inherited representation aspect is overridden by a subsequent <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> or representation item that specifies a different value for the same aspect of the type or subtype. <br /></p>

<AnnotatedOnly>
<MarginText>15.a</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>A <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> for a record extension does not override the layout of the parent part; if the layout was specified for the parent type, it is inherited by the record extension. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>15.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>If a representation item for the parent appears after the <code><a href="../AA-3/AA-3.4#S0035">derived{"_"}type{"_"}definition</a></code>, then inheritance does not happen for that representation item.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>15.b.1/3</MarginText>
<MarginInfo items={["AI05-0009-1", "AI05-0295-1"]} />
<Admonition type="aarm" aarm="note">
If an inherited aspect is confirmed by an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> or a later representation item for a derived type, the confirming specification does not override the inherited one. Thus the derived type has both a specified confirming value and an inherited nonconfirming representation value &mdash; this means that rules that apply only to nonconfirming representation values still apply to this type.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>15.b.2/4</MarginText>
<MarginInfo items={["AI12-0109-1"]} />
<Admonition type="aarm" aarm="note">
If an aspect was specified by an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> and the parent type has not yet been frozen, then the inherited aspect might not yet have been resolved and evaluated. The implementation will need to have a mechanism to handle such an aspect. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>15.1/5</MarginText>
<MarginInfo items={["AI95-00108-01", "AI95-00444-01", "AI05-0183-1", "AI05-0295-1", "AI12-0396-1"]} />
<p>{"{"}<em>8652/0040</em>{"}"} In contrast, whether type-related operational aspects are inherited by a derived type depends on each specific aspect; unless specified, an operational aspect is not inherited. When type-related operational aspects are inherited by a derived type, aspects that were directly specified by <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>s or operational items that are visible at any point within the immediate scope of the derived type declaration, or (in the case where the parent is derived) that were inherited by the parent type from the grandparent type, are inherited. An inherited operational aspect is overridden by an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> or operational item that specifies the same aspect of the type. <br /></p>

<AnnotatedOnly>
<MarginText>15.b.3/1</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>As with representation items, if an operational item for the parent appears after the <code><a href="../AA-3/AA-3.4#S0035">derived{"_"}type{"_"}definition</a></code>, then inheritance does not happen for that operational item.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>15.b.4/5</MarginText>
<MarginInfo items={["AI12-0396-1"]} />
<Admonition type="aarm" aarm="note">
Unlike representation aspects, operational aspects can be inherited at any point in the immediate scope, and can be overridden by aspect{"_"}specifications or operational items that occur at a point before or after the inheritance takes place. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>15.2/5</MarginText>
<MarginInfo items={["AI12-0396-1"]} />
<p>When a type-related operational aspect is inherited, the rules for inheritance depend on the nature of the aspect (see <a href="../AA-13/AA-13.1#Subclause_13.1.1">13.1.1</a>). Unless otherwise specified for a given aspect, these rules are as follows:<br /></p>

<MarginText>15.3/5</MarginText>
<ul>
<li>For an operational aspect that is a value, the inherited aspect has the same value;<br /></li>
<MarginText>15.4/5</MarginText>
<li>For an operational aspect that is a <code><a href="../AA-4/AA-4.1#S0091">name</a></code>:<br /></li>
<MarginText>15.5/5</MarginText>
<li>if the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> denotes one or more primitive subprograms of the type, the inherited aspect is a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes the corresponding primitive subprogram(s) of the derived type;<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>15.b.6/5</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>A primitive subprogram that is declared after the end of the enclosing declaration list of a derived type (in particular, in the private part of a package if the derived type is declared in the package's visible part) is not the &ldquo;corresponding primitive subprogram&rdquo; in the sense of this rule. In a case like this, the corresponding primitive subprogram is actually the inherited subprogram. This doesn't make a difference for tagged types, because of the &ldquo;dispatching&rdquo; semantics used, which means a call on the inherited subprogram actually &ldquo;reaches&rdquo; the body of the overriding. But for untagged types, overriding an inherited subprogram of an untagged derived type in the private part is &ldquo;too late&rdquo; to have an effect on external uses of the primitive, since non-visible overridings are not generally reachable, and a call that only &ldquo;sees&rdquo; the inherited subprogram goes to the body of the parent's operation, effectively ignoring the private overriding. This same rule applies to aspects that denote an inherited subprogram of an untagged derived type &mdash; the private overriding is effectively ignored. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>15.6/5</MarginText>
<ul>
<li>otherwise, the inherited aspect is a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes the same entity or entities as the original aspect; <br /></li>
<MarginText>15.7/5</MarginText>
<MarginInfo items={["AI12-0396-1", "AI12-0423-1"]} />
<li>For an operational aspect that is an identifier specific to the aspect, the inherited aspect is the same identifier;<br /></li>
<MarginText>15.8/5</MarginText>
<MarginInfo items={["AI12-0396-1", "AI12-0423-1"]} />
<li>For an operational aspect that is an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> or an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, the inherited aspect is a corresponding <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> or <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> where each <code><a href="../AA-4/AA-4.1#S0091">name</a></code>, value, and identifier follows these same rules for inheritance.<br /></li>
</ul>
<MarginText>16</MarginText>
<p>Each aspect of representation of an entity is as follows: <br /></p>

<MarginText>17</MarginText>
<ul>
<li>If the aspect is <em>specified</em> for the entity, meaning that it is either directly specified or inherited, then that aspect of the entity is as specified, except in the case of Storage{"_"}Size, which specifies a minimum. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>17.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This rule implies that queries of the aspect return the specified value. For example, if the user writes &ldquo;<strong>for</strong> X'Size <strong>use</strong> 32;&rdquo;, then a query of X'Size will return 32. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>18</MarginText>
<ul>
<li>If an aspect of representation of an entity is not specified, it is chosen by default in an unspecified manner. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>18.a/3</MarginText>
<MarginInfo items={["AI95-00137-01", "AI05-0295-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>{"{"}<em>8652/0009</em>{"}"} Note that specifying a representation aspect can affect the semantics of the entity.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18.b</MarginText>
<Admonition type="aarm" aarm="note">
The rules forbid things like &ldquo;<strong>for</strong> S'Base'Alignment <strong>use</strong> ...&rdquo; and &ldquo;<strong>for</strong> S'Base <strong>use</strong> record ...&rdquo;. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18.c</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The intent is that implementations will represent the components of a composite value in the same way for all subtypes of a given composite type. Hence, Component{"_"}Size and record layout are type-related aspects. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18.d/3</MarginText>
<MarginInfo items={["AI05-0083-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>As noted previously, in the case of an object, the entity mentioned in this text is a specific view of an object. That means that only references to the same view of an object that has a specified value for a representation aspect <em>R</em> necessarily have that value for the aspect <em>R</em>. The value of the aspect <em>R</em> for a different view of that object is unspecified. In particular, this means that the representation values for by-reference parameters is unspecified; they do not have to be the same as those of the underlying object. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>18.1/5</MarginText>
<MarginInfo items={["AI95-00108-01", "AI12-0396-1"]} />
<p>{"{"}<em>8652/0040</em>{"}"} If an operational aspect is <em>specified</em> for an entity (meaning that it is either directly specified or, if type-related or subtype-specific, inherited), then that aspect of the entity is as specified. Otherwise, the aspect of the entity has the default value for that aspect.[ For aspects that are neither type-related nor subtype-specific, the terms &ldquo;specified&rdquo; and &ldquo;directly specified&rdquo; are equivalent.]<br /></p>

<AnnotatedOnly>
<MarginText>18.e/5</MarginText>
<MarginInfo items={["AI12-0396-1"]} />
<Admonition type="aarm" aarm="proof">
<strong></strong>Aspect inheritance is only defined for types and subtypes. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>18.2/5</MarginText>
<MarginInfo items={["AI95-00291-02", "AI05-0295-1", "AI12-0396-1"]} />
<p>An <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> or representation item that specifies a representation aspect that would have been chosen in the absence of the <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> or representation item is said to be <em>confirming</em>. The aspect value specified in this case is said to be a <em>confirming</em> representation aspect value. Other values of the aspect are said to be <em>nonconfirming</em>, as are the <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>s and representation items that specified them. Similarly, an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> or operational item that specifies an operational aspect to be the same as the definition it would have by default is said to be <em>confirming</em>; otherwise it is <em>nonconfirming</em>. <br /></p>

<AnnotatedOnly>
<MarginText>18.f/5</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>As noted at the beginning of this subclause, we have a Language Design Principle that confirming aspects do not change the semantics. However, it is not a Language Design Principle principle that one can always specify a confirming aspect. Some aspects can never be confirmed. For instance, the default implementation of a stream-oriented attribute cannot be confirmed, as any subprogram that can be specified is not the same as the default subprogram, and thus is nonconfirming. Whether the effect is the same is not relevant for this purpose; it would be very difficult to formally define what it means to have the same effect. (Note that one can name any implementation for a stream-oriented attribute by using the associated attribute but specifying that by name would violate the circular aspect definition rule.) This is true for any implicitly composed aspect. Similarly, aspect Default{"_"}Value cannot be confirmed as the default is that there is no default value; any value that can be specified is not that. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>21.1/3</MarginText>
<MarginInfo items={["AI95-00291-02", "AI05-0295-1"]} />

<AnnotatedOnly>
<MarginText>22.a</MarginText>
<Admonition type="aarm" aarm="reason">

<AnnotatedOnly>
<MarginText>22.b/5</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
<CodeBlock language="ada">
X : Integer := F(...){"\n"}   with Address ={">"} Y;{"\n"}Y : Address := G(...);{"\n"}

</CodeBlock>

<AnnotatedOnly>

<MarginInfo items={["AI12-0059-1", "AI12-0444-1"]} />
<ul>
<li>An implementation is not required to support a specification for the Object{"_"}Size or Size for a given composite subtype, nor the size or storage place for an object (including a component) of a given composite subtype, unless the constraints on the subtype and its composite subcomponents (if any) are all static constraints.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>23.a/5</MarginText>
<MarginInfo items={["AI12-0059-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>We don't want to require that Object{"_"}Size or Size be supported on types that might have a representation not completely specified at compile time, or are represented discontiguously, or are represented differently for different constraints. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>24/5</MarginText>
<MarginInfo items={["AI95-00291-02", "AI05-0295-1", "AI12-0444-1"]} />
<ul>
<li>An implementation is not required to support specifying a nonconfirming representation aspect value if it can cause an aliased object or an object of a by-reference type to be allocated at a nonaddressable location or, when the alignment attribute of the subtype of such an object is nonzero, at an address that is not an integral multiple of that alignment. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>24.a/1</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The intent is that access types, type System.Address, and the pointer used for a by-reference parameter should be implementable as a single machine address &mdash; bit-field pointers should not be required. (There is no requirement that this implementation be used &mdash; we just want to make sure it's feasible.) <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.b/2</MarginText>
<MarginInfo items={["AI95-00291-02"]} />
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>We want subprograms to be able to assume the properties of the types of their parameters inside of subprograms. While many objects can be copied to allow this (and thus do not need limitations), aliased or by-reference objects cannot be copied (their memory location is part of their identity). Thus, the above rule does not apply to types that merely allow by-reference parameter passing; for such types, a copy typically needs to be made at the call site when a bit-aligned component is passed as a parameter. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>25/5</MarginText>
<MarginInfo items={["AI95-00291-02", "AI05-0295-1", "AI12-0444-1"]} />
<ul>
<li>An implementation is not required to support specifying a nonconfirming representation aspect value if it can cause an aliased object of an elementary type to have a size other than that which would have been chosen by default. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>25.a/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>Since all bits of elementary objects participate in operations, aliased objects must not have a different size than that assumed by users of the access type. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>26/5</MarginText>
<MarginInfo items={["AI95-00291-02", "AI05-0295-1", "AI12-0444-1"]} />
<ul>
<li>An implementation is not required to support specifying a nonconfirming representation aspect value if it can cause an aliased object of a composite type, or an object whose type is by-reference, to have a size smaller than that which would have been chosen by default. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>26.a/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>Unlike elementary objects, there is no requirement that all bits of a composite object participate in operations. Thus, as long as the object is the same or larger in size than that expected by the access type, all is well. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>26.b/2</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>This rule presumes that the implementation allocates an object of a size specified to be larger than the default size in such a way that access of the default size suffices to correctly read and write the value of the object. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>27/5</MarginText>
<MarginInfo items={["AI95-00291-02", "AI05-0295-1", "AI12-0444-1"]} />
<ul>
<li>An implementation is not required to support specifying a nonconfirming subtype-specific representation aspect value for an indefinite or abstract subtype. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>27.a/3</MarginText>
<MarginInfo items={["AI05-0295-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>Representation aspects are often not well-defined for such types. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.b/3</MarginText>
<MarginInfo items={["AI95-00291-02", "AI05-0229-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>A type with the Pack aspect specified will typically not be packed so tightly as to disobey the above rules. A Component{"_"}Size clause or <code><a href="../AA-13/AA-13.5#S0352">record_representation_clause</a></code> will typically be illegal if it disobeys the above rules. Atomic components have similar restrictions (see <a href="../AA-C/AA-C.6">C.6</a>, &ldquo;<a href="../AA-C/AA-C.6">Shared Variable Control</a>&rdquo;). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>28/5</MarginText>
<MarginInfo items={["AI95-00291-02", "AI05-0295-1", "AI12-0444-1"]} />
<p>For purposes of these rules, the determination of whether specifying a representation aspect value for a type <em>can cause</em> an object to have some property is based solely on the properties of the type itself, not on any available information about how the type is used. In particular, it presumes that minimally aligned objects of this type can be declared at some point. <br /></p>

</Admonition>

#### Name Resolution Rules

<MarginText>5/5_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1", "AI12-0212-1"]} />
<p>An <code><a href="../AA-13/AA-13.1#S0347">aspect_mark</a></code> identifies an aspect of the entity defined by the associated declaration (the <em>associated entity</em>); the aspect denotes an object, a value, an expression, an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, a subprogram, or some other kind of entity. If the <code><a href="../AA-13/AA-13.1#S0347">aspect_mark</a></code> identifies: <br /></p>

<MarginText>6/3_13.1.1</MarginText>
<ul>
<li>an aspect that denotes an object, the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> shall be a <code><a href="../AA-4/AA-4.1#S0091">name</a></code>. The expected type for the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> is the type of the identified aspect of the associated entity; <br /></li>
<MarginText>7/3_13.1.1</MarginText>
<li>an aspect that is a value or an expression, the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> shall be an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>. The expected type for the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is the type of the identified aspect of the associated entity;<br /></li>
<MarginText>7.1/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<li>an aspect that is an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, the aspect definition shall be an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> that is an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, with the form of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> determined by the identified aspect;<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>7.a/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0212-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>The only requirement here is that the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is syntactically an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>; there is no requirement that it resolve to some particular type or even that it should resolve like an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>. Each aspect that uses an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> is responsible for specifying how the choice(s), component(s), and other contents of the <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code> are resolved and interpreted. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>8/3_13.1.1</MarginText>
<ul>
<li>an aspect that denotes a subprogram, the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> shall be a <code><a href="../AA-4/AA-4.1#S0091">name</a></code>; the expected profile for the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> is the profile required for the aspect of the associated entity;<br /></li>
<MarginText>9/3_13.1.1</MarginText>
<li>an aspect that denotes some other kind of entity, the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> shall be a <code><a href="../AA-4/AA-4.1#S0091">name</a></code>, and the name shall resolve to denote an entity of the appropriate kind;<br /></li>
<MarginText>10/3_13.1.1</MarginText>
<li>an aspect that is given by an identifier specific to the aspect, the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> shall be an <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code>, and the <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> shall be one of the identifiers specific to the identified aspect. <br /></li>
</ul>
<MarginText>11/5_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1", "AI12-0373-1", "AI12-0427-1"]} />
<p>The usage names in an <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> associated with a declaration [ are not resolved at the point of the associated declaration, but rather] are resolved at the end of the immediately enclosing declaration list, or in the case of the declaration of a library unit, at the end of the visible part of the entity.<br /></p>

<AnnotatedOnly>
<MarginText>11.a_13.1.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>If an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> is not associated with a declaration (instead being part of a <code><a href="../AA-5/AA-5.1#S0167">statement</a></code> or <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>), then it is resolved when the associated construct is resolved. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>12/5_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1", "AI12-0180-1", "AI12-0220-1"]} />
<p>If the associated declaration is for a subprogram, entry,  or access-to-subprogram type, the names of the formal parameters are directly visible within the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code>, as are certain attributes, as specified elsewhere in this document for the identified aspect. If the associated declaration is a <code><a href="../AA-3/AA-3.2#S0023">type_declaration</a></code>, within the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> the names of any visible components, protected subprograms, and entries are directly visible, and the name of the first subtype denotes the current instance of the type (see <a href="../AA-8/AA-8.6">8.6</a>). If the associated declaration is a <code><a href="../AA-3/AA-3.2#S0026">subtype_declaration</a></code>, within the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> the name of the new subtype denotes the current instance of the subtype. <br /></p>


#### Legality Rules

<MarginText>13/3_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1"]} />
<p>If the first freezing point of the associated entity comes before the end of the immediately enclosing declaration list, then each usage name in the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> shall resolve to the same entity at the first freezing point as it does at the end of the immediately enclosing declaration list.<br /></p>

<MarginText>13.1/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0181-1", "AI12-0396-1"]} />
<p>An <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> or <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that causes freezing of an entity shall not occur within an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> that specifies a representation or operational aspect of that entity.<br /></p>

<AnnotatedOnly>
<MarginText>13.a/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0181-1", "AI12-0396-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>The rule prevents an aspect of an entity from depending (directly or indirectly) on properties of the entity itself. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>13.b/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0396-1"]} />
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>This check needs to be deferred at least to the freezing of the entity (as the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> is not resolved until then), and might be accomplished during the freezing of the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> (since it is closely related). Keep in mind that multiple other entities could be involved. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>14/3_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1"]} />
<p>At most one occurrence of each <code><a href="../AA-13/AA-13.1#S0347">aspect_mark</a></code> is allowed within a single <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>. The aspect identified by the <code><a href="../AA-13/AA-13.1#S0347">aspect_mark</a></code> shall be an aspect that can be specified for the associated entity (or view of the entity defined by the associated declaration).<br /></p>

<AnnotatedOnly>
<MarginText>14.a/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0005-1", "AI12-0116-1", "AI12-0396-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>This rule prevents multiple specifications in the same <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>. Rules in <a href="../AA-13/AA-13.1">13.1</a> prevent multiple specifications in different <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>s (on different views of the same type, for instance), or between operational or representation items and an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>15/3_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1"]} />
<p>The <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> associated with a given <code><a href="../AA-13/AA-13.1#S0347">aspect_mark</a></code> may be omitted only when the <code><a href="../AA-13/AA-13.1#S0347">aspect_mark</a></code> identifies an aspect of a boolean type, in which case it is equivalent to the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> being specified as True.<br /></p>

<MarginText>16/3_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1"]} />
<p>If the <code><a href="../AA-13/AA-13.1#S0347">aspect_mark</a></code> includes 'Class, then the associated entity shall be a tagged type or a primitive subprogram of a tagged type.<br /></p>

<MarginText>17/5_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1", "AI05-0267-1", "AI12-0064-2", "AI12-0194-1", "AI12-0380-1", "AI12-0407-1"]} />
<p>Unless otherwise specified for a specific aspect, a language-defined aspect cannot be specified on a <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code> or a <code><a href="../AA-12/AA-12.1#S0314">generic_formal_parameter_declaration</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>17.a/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0064-2"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>Implementation-defined aspects can be allowed on these, of course; the implementation will need to define the semantics. In addition, the language does not define default aspect matching rules for generic formals; only the handful of aspects allowed on formals have such rules. Therefore, the implementation will need to define actual type matching rules for any aspects allowed on formal types. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>18/5_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1", "AI05-0267-1", "AI12-0105-1", "AI12-0380-1"]} />
<p>Unless specified otherwise, a language-defined aspect shall not be specified in an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> given on a completion of a program unit.<br /></p>

<AnnotatedOnly>
<MarginText>18.a/3_13.1.1</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>Most language-defined aspects (for example, preconditions) are intended to be available to callers, and specifying them on a body that has a separate declaration hides them from callers. Specific language-defined aspects may allow this, but they have to do so explicitly (by defining an alternative Legality Rule), and provide any needed rules about visibility. Note that this rule does not apply to implementation-defined aspects, so implementers need to carefully define whether such aspects can be applied to bodies and stubs, and what happens if they are specified on both the declaration and body of a unit. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>18.1/4_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1", "AI12-0138-1"]} />
<p>If an aspect of a derived type is inherited from an ancestor type and has the boolean value True, the inherited value shall not be overridden to have the value False for the derived type, unless otherwise specified in this document.<br /></p>

<AnnotatedOnly>
<MarginText>18.b/4_13.1.1</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>Most boolean-valued language-defined aspects are associated with a representation pragma. The existing rules for such pragmas assume that the aspect cannot be removed. For instance, if a type <em>T</em> is declared to be Atomic, then all descendants of <em>T</em> are also Atomic. This rule ensures that remains the case when using the aspect notation instead of pragmas. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18.b.1/5_13.1.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>This definition leaves holes for Boolean aspects that can be specified on non-first subtypes. Such aspects (for instance, Nonblocking) must have their own rules (that is, "otherwise specify" rules) that define the effects of inheriting from subtypes (both first subtypes and nonfirst subtypes). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>18.2/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0396-1"]} />
<p>If a given aspect is type-related and inherited, then within an <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> for the aspect, if a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> resolves to denote multiple visible subprograms, all or none of the denoted subprograms shall be primitives of the associated type.<br /></p>

<AnnotatedOnly>
<MarginText>18.b.2/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0396-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>This is necessary because the inheritance rules for <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s denoting primitive subprograms are different from those for <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s denoting other entities &mdash; see <a href="../AA-13/AA-13.1">13.1</a>. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>18.3/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0138-1", "AI12-0206-1", "AI12-0419-1"]} />
<p>Certain type-related aspects are defined to be <em>nonoverridable</em>; all such aspects are inherited by derived types according to the rules given in <a href="../AA-13/AA-13.1">13.1</a>. Any legality rule associated with a nonoverridable aspect is re-checked for the derived type, if the derived type is not abstract. Certain type-related and subtype-specific aspects are defined to be <em>additive</em>; such aspects are not inherited, but they can <em>apply</em> to the types derived from, or the subtypes based on, the original type or subtype, as defined for each such aspect. Finally, certain type-related aspects are <em>implicitly composed</em>; such aspects are not inherited, but rather a default implementation for a derived type is provided, as defined for each such aspect, based on that of its parent type, presuming the aspect for the parent type is available where the derived type is declared, plus those of any new components added as part of a type extension.<br /></p>

<MarginText>18.4/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0138-1", "AI12-0206-1", "AI12-0211-1", "AI12-0396-1"]} />
<p>If a nonoverridable aspect is directly specified for a type <em>T</em>, then any explicit specification of that aspect for any descendant of <em>T</em> (other than <em>T</em> itself) shall be <em>confirming</em>. In the case of an aspect that is a <code><a href="../AA-4/AA-4.1#S0091">name</a></code>, this means that the specified <code><a href="../AA-4/AA-4.1#S0091">name</a></code> shall <em>match</em> the inherited aspect in the sense that it shall denote the same declarations as would the inherited <code><a href="../AA-4/AA-4.1#S0091">name</a></code>. Similarly, for an aspect that is an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> or an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, confirming means the defining <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is fully conformant (see <a href="../AA-6/AA-6.3#Subclause_6.3.1">6.3.1</a>) with the defining <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> for the inherited aspect, with the added rule that an identifier that is specific to the aspect is the same as the corresponding identifier in the inherited aspect.<br /></p>

<MarginText>18.5/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0138-1", "AI12-0396-1"]} />
<p>If a full type has a partial view, and a given nonoverridable aspect is allowed for both the full view and the partial view, then the given aspect for the partial view and the full view shall be the same: the aspect shall be directly specified only on the partial view; if the full type inherits the aspect, then a matching definition shall be specified (directly or by inheritance) for the partial view.<br /></p>

<AnnotatedOnly>
<MarginText>18.c/4_13.1.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>In order to enforce these rules without breaking privacy, we cannot allow a private type that could have a particular overridable aspect to have a hidden definition of that aspect. There is no problem if the private type does not allow the aspect (as the aspect could not be specified on descendants in that case). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>18.6/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0211-1"]} />
<p>If a type inherits a nonoverridable aspect from multiple ancestors, the value of the aspect inherited from any given ancestor shall be confirming of the values inherited from all other ancestors.<br /></p>

<AnnotatedOnly>
<MarginText>18.d/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0005-1", "AI12-0211-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>If more than one progenitor of a type <em>T</em> specifies a nonoverridable aspect, they all have to specify the same or matching values for that aspect. Otherwise, we'd have two different values for the aspect that depend on which progenitor we inherit from. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>18.7/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0138-1", "AI12-0211-1"]} />
<p>In addition to the places where Legality Rules normally apply (see <a href="../AA-12/AA-12.3">12.3</a>), these rules about nonoverridable aspects also apply in the private part of an instance of a generic unit.<br /></p>

<AnnotatedOnly>
<MarginText>18.e/5_13.1.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>We don't need an assume-the-worst rule for most nonoverridable aspects as they only work on tagged types and deriving from formal tagged types is not allowed in generic bodies. In the case of Implicit{"_"}Dereference, a derivation in a generic body does not cause problems (the ancestor necessarily cannot have the aspect, else specifying the aspect would be illegal), as there could be no place with visibility on both aspects. In the case of Max{"_"}Entry{"_"}Queue{"_"}Length, it is only allowed on task and protected types, and on entries, and there are not formal versions of any of those things. In the case of No{"_"}Controlled{"_"}Parts, we defined an assume-the-worst rule with the aspect.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>18.f/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0437-1"]} />
<Admonition type="aarm" aarm="note">
A list of all nonoverridable aspects can be found in the index, under &ldquo;nonoverridable aspect&rdquo;. <br />
</Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>19/3_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1"]} />
<p>Depending on which aspect is identified by the <code><a href="../AA-13/AA-13.1#S0347">aspect_mark</a></code>, an <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> specifies: <br /></p>

<MarginText>20/3_13.1.1</MarginText>
<ul>
<li>a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes a subprogram, object, or other kind of entity;<br /></li>
<MarginText>21/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0427-1"]} />
<li>an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> (other than an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>), which is either evaluated to produce a single value, or which (as in a precondition) is to be evaluated at particular points during later execution;<br /></li>
<MarginText>22/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0427-1"]} />
<li>an <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> specific to the aspect; or<br /></li>
<MarginText>22.1/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0427-1"]} />
<li>an <code><a href="../AA-4/AA-4.3#S0106">aggregate</a></code>, which is positional or named, and is composed of elements of any of these four kinds of constructs. <br /></li>
</ul>
<MarginText>23/3_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1"]} />
<p>The identified aspect of the associated entity, or in some cases, the view of the entity defined by the declaration, is as specified by the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> (or by the default of True when boolean). Whether an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> <em>applies</em> to an entity or only to the particular view of the entity defined by the declaration is determined by the <code><a href="../AA-13/AA-13.1#S0347">aspect_mark</a></code> and the kind of entity. The following aspects are view specific:<br /></p>

<MarginText>24/3_13.1.1</MarginText>
<ul>
<li>An aspect specified on an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code>;<br /></li>
<MarginText>25/3_13.1.1</MarginText>
<li>An aspect specified on a <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code>;<br /></li>
<MarginText>26/3_13.1.1</MarginText>
<li>An aspect specified on a <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code>.<br /></li>
</ul>
<MarginText>27/3_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1"]} />
<p>All other <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>s are associated with the entity, and <em>apply</em> to all views of the entity, unless otherwise specified in this document.<br /></p>

<MarginText>28/4_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1", "AI12-0106-1"]} />
<p>If the <code><a href="../AA-13/AA-13.1#S0347">aspect_mark</a></code> includes 'Class (a <em>class-wide aspect</em>), then, unless specified otherwise for a particular class-wide aspect: <br /></p>

<MarginText>29/3_13.1.1</MarginText>
<ul>
<li>if the associated entity is a tagged type, the specification <em>applies</em> to all descendants of the type;<br /></li>
<MarginText>30/3_13.1.1</MarginText>
<li>if the associated entity is a primitive subprogram of a tagged type <em>T</em>, the specification <em>applies</em> to the corresponding primitive subprogram of all descendants of <em>T</em>. <br /></li>
</ul>
<MarginText>31/3_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1", "AI05-0229-1"]} />
<p>All specifiable operational and representation attributes may be specified with an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> instead of an <code><a href="../AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> (see <a href="../AA-13/AA-13.3">13.3</a>).<br /></p>

<AnnotatedOnly>
<MarginText>31.a/3_13.1.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The name of the aspect is the same as that of the attribute (see <a href="../AA-13/AA-13.3">13.3</a>), so the <code><a href="../AA-13/AA-13.1#S0347">aspect_mark</a></code> is the <code><a href="../AA-4/AA-4.1#S0101">attribute_designator</a></code> of the attribute.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>31.b/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
<Admonition type="aarm" aarm="note">
Unless specified otherwise, all of the requirements for specifying a particular aspect with an <code><a href="../AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> also apply to an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> for the aspect. These are enforced at the freezing point of the entity. For example, when specifying the Size aspect of a subtype, the expression has to be a static expression with an integer type and a nonnegative value, all of the recommended level of support requirements apply if <a href="../AA-C/">Annex C</a> is supported (see <a href="../AA-C/AA-C.2">C.2</a>), and so on. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>32/5_13.1.1</MarginText>
<MarginInfo items={["AI05-0229-1", "AI12-0154-1", "AI12-0417-1"]} />
<p>Some aspects are defined to be <em>library unit aspects</em>. Library unit aspects are of type Boolean. The expression specifying a library unit aspect shall be static. Library unit aspects are defined for all program units, but shall be specified only for library units. Notwithstanding what this document says elsewhere, the expression of a library unit aspect is resolved and evaluated at the point where it occurs in the <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>[, rather than the first freezing point of the associated unit].<br /></p>

<AnnotatedOnly>
<MarginText>32.a/3_13.1.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The name of the aspect is the same as that of the pragma (see <a href="../AA-13/AA-13.1">13.1</a>), so the <code><a href="../AA-13/AA-13.1#S0347">aspect_mark</a></code> is the name of the pragma. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>33/3_13.1.1</MarginText>
<MarginInfo items={["AI05-0229-1"]} />

<MarginText>35/3_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1"]} />
<p>If a Legality Rule or Static Semantics rule only applies when a particular aspect has been specified, the aspect is considered to have been specified only when the <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> or <code><a href="../AA-13/AA-13.3#S0349">attribute_definition_clause</a></code> is visible (see <a href="../AA-8/AA-8.3">8.3</a>) at the point of the application of the rule.<br /></p>

#### Dynamic Semantics

<MarginText>37/5_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1", "AI12-0355-2"]} />
<p>At the freezing point of the associated entity, the <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> is elaborated. When appearing in a construct other than a declaration, an <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> is elaborated as part of the execution of the construct. The elaboration of the <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> consists of the elaboration of each <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> in an arbitrary order. The elaboration of an <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> includes the evaluation of any <code><a href="../AA-4/AA-4.1#S0091">name</a></code> or <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> that is part of the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> unless the part is itself an expression. If the corresponding aspect (or part thereof) represents an expression (as in a precondition), the elaboration of that part has no effect; the expression is evaluated later at points within the execution as specified elsewhere in this document for the particular aspect. <br /></p>


#### Implementation Permissions

<MarginText>38/3_13.1.1</MarginText>
<MarginInfo items={["AI05-0183-1"]} />
<p>Implementations may support implementation-defined aspects. The <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> for an implementation-defined aspect may use an implementation-defined syntax for the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code>, and may follow implementation-defined legality and semantics rules. <br /></p>

<AnnotatedOnly>
<MarginText>38.a/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0389-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>The intent is to allow implementations to support aspects that are defined, for example, by a <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code> rather than an <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> or a <code><a href="../AA-4/AA-4.1#S0091">name</a></code>. We chose not to try to enumerate all possible <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> syntaxes, but to give implementations maximum freedom. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>38.a.1/3_13.1.1</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>Implementation-defined aspects, including the syntax for specifying such aspects and the legality rules for such aspects.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>39/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0389-1"]} />
<p>An implementation may ignore the specification of an unrecognized aspect; if an implementation chooses to ignore such an aspect specification (as opposed to rejecting it), then it has no effect on the semantics of the program except for possibly (and this is not required) the rejection of syntax errors within the <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code>.<br /></p>

<AnnotatedOnly>
<MarginText>39.a/5_13.1.1</MarginText>
<MarginInfo items={["AI12-0389-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>Identifying the textual end of an <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> for an unrecognized aspect may be challenging, particularly if the syntax for the unrecognized aspect's <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> is implementation-defined. It is not specified how an implementation might accomplish this. Note that an implementation is never <em>required</em> to be able to do this; if an <code><a href="../AA-13/AA-13.1#S0348">aspect_definition</a></code> for an unrecognized aspect is problematic in any way, then it can always be rejected (as opposed to being ignored). <br />
</Admonition>
