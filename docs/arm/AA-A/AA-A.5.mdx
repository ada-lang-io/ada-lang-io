One child, the generic package Generic_Elementary_Functions, is defined in <a href="../AA-A/AA-A.5#Subclause_A.5.1">A.5.1</a>, together with nongeneric equivalents; two others, the package Float_Random and the generic package Discrete_Random, are defined in <a href="../AA-A/AA-A.5#Subclause_A.5.2">A.5.2</a>. ## A.5.1  Elementary Functions

<MarginText>1</MarginText>
<p>Implementation-defined approximations to the mathematical functions known as the &ldquo;elementary functions&rdquo; are provided by the subprograms in Numerics.Generic_Elementary_Functions. Nongeneric equivalents of this generic package for each of the predefined floating point types are also provided as children of Numerics. <br /></p>

<AnnotatedOnly>
<MarginText>1.a</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>The accuracy actually achieved by the elementary functions.<br /></Admonition>
</AnnotatedOnly>

<MarginText>9/1</MarginText>
<MarginInfo items={["AI95-00126-01"]} />
<p>{"{"}<em>8652/0020</em>{"}"} The library package Numerics.Elementary_Functions is declared pure and defines the same subprograms as Numerics.Generic_Elementary_Functions, except that the predefined type Float is systematically substituted for Float_Type'Base throughout. Nongeneric equivalents of Numerics.Generic_Elementary_Functions for each of the other predefined floating point types are defined similarly, with the names Numerics.Short_Elementary_Functions, Numerics.Long_Elementary_Functions, etc. <br /></p>

<AnnotatedOnly>
<MarginText>9.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The nongeneric equivalents are provided to allow the programmer to construct simple mathematical applications without being required to understand and use generics. <br /></Admonition>
</AnnotatedOnly>

<MarginText>10</MarginText>
<p>The functions have their usual mathematical meanings. When the Base parameter is specified, the Log function computes the logarithm to the given base; otherwise, it computes the natural logarithm. When the Cycle parameter is specified, the parameter X of the forward trigonometric functions (Sin, Cos, Tan, and Cot) and the results of the inverse trigonometric functions (Arcsin, Arccos, Arctan, and Arccot) are measured in units such that a full cycle of revolution has the given value; otherwise, they are measured in radians.<br /></p>

<MarginText>11</MarginText>
<p>The computed results of the mathematically multivalued functions are rendered single-valued by the following conventions, which are meant to imply the principal branch: <br /></p>

<MarginText>12</MarginText>
<ul>
<li>The results of the Sqrt and Arccosh functions and that of the exponentiation operator are nonnegative.<br /></li>
<MarginText>13</MarginText>
<li>The result of the Arcsin function is in the quadrant containing the point (1.0, <em>x</em>), where <em>x</em> is the value of the parameter X. This quadrant is I or IV; thus, the range of the Arcsin function is approximately &ndash;&pi;/2.0 to &pi;/2.0 (&ndash;Cycle/4.0 to Cycle/4.0, if the parameter Cycle is specified).<br /></li>
<MarginText>14</MarginText>
<li>The result of the Arccos function is in the quadrant containing the point (<em>x</em>, 1.0), where <em>x</em> is the value of the parameter X. This quadrant is I or II; thus, the Arccos function ranges from 0.0 to approximately &pi; (Cycle/2.0, if the parameter Cycle is specified).<br /></li>
<MarginText>15</MarginText>
<li>The results of the Arctan and Arccot functions are in the quadrant containing the point (<em>x</em>, <em>y</em>), where <em>x</em> and <em>y</em> are the values of the parameters X and Y, respectively. This may be any quadrant (I through IV) when the parameter X (resp., Y) of Arctan (resp., Arccot) is specified, but it is restricted to quadrants I and IV (resp., I and II) when that parameter is omitted. Thus, the range when that parameter is specified is approximately &ndash;&pi; to &pi; (&ndash;Cycle/2.0 to Cycle/2.0, if the parameter Cycle is specified); when omitted, the range of Arctan (resp., Arccot) is that of Arcsin (resp., Arccos), as given above. When the point (<em>x</em>, <em>y</em>) lies on the negative x-axis, the result approximates <br /></li>
<MarginText>16</MarginText>
<li>&pi; (resp., &ndash;&pi;) when the sign of the parameter Y is positive (resp., negative), if Float_Type'Signed_Zeros is True;<br /></li>
<MarginText>17</MarginText>
<li>&pi;, if Float_Type'Signed_Zeros is False. <br /></li>
</ul>
<MarginText>18</MarginText>
<p>(In the case of the inverse trigonometric functions, in which a result lying on or near one of the axes may not be exactly representable, the approximation inherent in computing the result may place it in an adjacent quadrant, close to but on the wrong side of the axis.) <br /></p>


#### Dynamic Semantics

<MarginText>19</MarginText>
<p>The exception Numerics.Argument_Error is raised, signaling a parameter value outside the domain of the corresponding mathematical function, in the following cases: <br /></p>

<MarginText>20</MarginText>
<ul>
<li>by any forward or inverse trigonometric function with specified cycle, when the value of the parameter Cycle is zero or negative;<br /></li>
<MarginText>21</MarginText>
<li>by the Log function with specified base, when the value of the parameter Base is zero, one, or negative;<br /></li>
<MarginText>22</MarginText>
<li>by the Sqrt and Log functions, when the value of the parameter X is negative;<br /></li>
<MarginText>23</MarginText>
<li>by the exponentiation operator, when the value of the left operand is negative or when both operands have the value zero;<br /></li>
<MarginText>24</MarginText>
<li>by the Arcsin, Arccos, and Arctanh functions, when the absolute value of the parameter X exceeds one;<br /></li>
<MarginText>25</MarginText>
<li>by the Arctan and Arccot functions, when the parameters X and Y both have the value zero;<br /></li>
<MarginText>26</MarginText>
<li>by the Arccosh function, when the value of the parameter X is less than one; and<br /></li>
<MarginText>27</MarginText>
<li>by the Arccoth function, when the absolute value of the parameter X is less than one. <br /></li>
</ul>
<MarginText>28</MarginText>
<p>The exception Constraint_Error is raised, signaling a pole of the mathematical function (analogous to dividing by zero), in the following cases, provided that Float_Type'Machine_Overflows is True: <br /></p>

<MarginText>29</MarginText>
<ul>
<li>by the Log, Cot, and Coth functions, when the value of the parameter X is zero;<br /></li>
<MarginText>30</MarginText>
<li>by the exponentiation operator, when the value of the left operand is zero and the value of the exponent is negative;<br /></li>
<MarginText>31</MarginText>
<li>by the Tan function with specified cycle, when the value of the parameter X is an odd multiple of the quarter cycle;<br /></li>
<MarginText>32</MarginText>
<li>by the Cot function with specified cycle, when the value of the parameter X is zero or a multiple of the half cycle; and<br /></li>
<MarginText>33</MarginText>
<li>by the Arctanh and Arccoth functions, when the absolute value of the parameter X is one. <br /></li>
</ul>
<MarginText>34</MarginText>
<p>[Constraint_Error can also be raised when a finite result overflows (see <a href="../AA-G/AA-G.2#Subclause_G.2.4">G.2.4</a>); this may occur for parameter values sufficiently <em>near</em> poles, and, in the case of some of the functions, for parameter values with sufficiently large magnitudes.] When Float_Type'Machine_Overflows is False, the result at poles is unspecified. <br /></p>

<AnnotatedOnly>
<MarginText>34.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>The purpose of raising Constraint_Error (rather than Numerics.Argument_Error) at the poles of a function, when Float_Type'Machine_Overflows is True, is to provide continuous behavior as the actual parameters of the function approach the pole and finally reach it. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>34.b</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>It is anticipated that an Ada binding to IEC 559:1989 will be developed in the future. As part of such a binding, the Machine_Overflows attribute of a conformant floating point type will be specified to yield False, which will permit both the predefined arithmetic operations and implementations of the elementary functions to deliver signed infinities (and set the overflow flag defined by the binding) instead of raising Constraint_Error in overflow situations, when traps are disabled. Similarly, it is appropriate for the elementary functions to deliver signed infinities (and set the zero-divide flag defined by the binding) instead of raising Constraint_Error at poles, when traps are disabled. Finally, such a binding should also specify the behavior of the elementary functions, when sensible, given parameters with infinite values. <br /></Admonition>
</AnnotatedOnly>

<MarginText>35/5</MarginText>
<MarginInfo items={["AI12-0453-1"]} />
<p>When one parameter of a function with multiple parameters represents a pole and another is outside the function's domain, the latter takes precedence and  Numerics.Argument_Error is raised . <br /></p>


#### Implementation Requirements

<MarginText>36</MarginText>
<p>In the implementation of Numerics.Generic_Elementary_Functions, the range of intermediate values allowed during the calculation of a final result shall not be affected by any range constraint of the subtype Float_Type. <br /></p>

<AnnotatedOnly>
<MarginText>36.a</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>Implementations of Numerics.Generic_Elementary_Functions written in Ada should therefore avoid declaring local variables of subtype Float_Type; the subtype Float_Type'Base should be used instead. <br /></Admonition>
</AnnotatedOnly>

<MarginText>37</MarginText>
<p>In the following cases, evaluation of an elementary function shall yield the <em>prescribed result</em>, provided that the preceding rules do not call for an exception to be raised: <br /></p>

<MarginText>38</MarginText>
<ul>
<li>When the parameter X has the value zero, the Sqrt, Sin, Arcsin, Tan, Sinh, Arcsinh, Tanh, and Arctanh functions yield a result of zero, and the Exp, Cos, and Cosh functions yield a result of one.<br /></li>
<MarginText>39</MarginText>
<li>When the parameter X has the value one, the Sqrt function yields a result of one, and the Log, Arccos, and Arccosh functions yield a result of zero.<br /></li>
<MarginText>40</MarginText>
<li>When the parameter Y has the value zero and the parameter X has a positive value, the Arctan and Arccot functions yield a result of zero.<br /></li>
<MarginText>41</MarginText>
<li>The results of the Sin, Cos, Tan, and Cot functions with specified cycle are exact when the mathematical result is zero; those of the first two are also exact when the mathematical result is ± 1.0.<br /></li>
<MarginText>42</MarginText>
<li>Exponentiation by a zero exponent yields the value one. Exponentiation by a unit exponent yields the value of the left operand. Exponentiation of the value one yields the value one. Exponentiation of the value zero yields the value zero. <br /></li>
</ul>
<MarginText>43</MarginText>
<p>Other accuracy requirements for the elementary functions, which apply only in implementations conforming to the Numerics Annex, and then only in the &ldquo;strict&rdquo; mode defined there (see <a href="../AA-G/AA-G.2">G.2</a>), are given in <a href="../AA-G/AA-G.2#Subclause_G.2.4">G.2.4</a>.<br /></p>

<MarginText>44</MarginText>
<p>When Float_Type'Signed_Zeros is True, the sign of a zero result shall be as follows: <br /></p>

<MarginText>45</MarginText>
<ul>
<li>A prescribed zero result delivered <em>at the origin</em> by one of the odd functions (Sin, Arcsin, Sinh, Arcsinh, Tan, Arctan or Arccot as a function of Y when X is fixed and positive, Tanh, and Arctanh) has the sign of the parameter X (Y, in the case of Arctan or Arccot).<br /></li>
<MarginText>46</MarginText>
<li>A prescribed zero result delivered by one of the odd functions <em>away from the origin</em>, or by some other elementary function, has an implementation-defined sign. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>46.a</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>The sign of a zero result from some of the operators or functions in Numerics.Generic_Elementary_Functions, when Float_Type'Signed_Zeros is True.<br /></Admonition>
</AnnotatedOnly>

<MarginText>47/5</MarginText>
<MarginInfo items={["AI12-0453-1"]} />
<ul>
<li>[A zero result that is not a prescribed result (that is , one that results from rounding or underflow) has the correct mathematical sign.] <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>47.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This is a consequence of the rules specified in IEC 559:1989 as they apply to underflow situations with traps disabled. <br /></Admonition>
</AnnotatedOnly>

## A.5.2  Random Number Generation

<MarginText>1</MarginText>
<p>[Facilities for the generation of pseudo-random floating point numbers are provided in the package Numerics.Float_Random; the generic package Numerics.Discrete_Random provides similar facilities for the generation of pseudo-random integers and pseudo-random values of enumeration types. For brevity, pseudo-random values of any of these types are called <em>random numbers</em>.<br /></p>

<MarginText>2</MarginText>
<p>Some of the facilities provided are basic to all applications of random numbers. These include a limited private type each of whose objects serves as the generator of a (possibly distinct) sequence of random numbers; a function to obtain the &ldquo;next&rdquo; random number from a given sequence of random numbers (that is, from its generator); and subprograms to initialize or reinitialize a given generator to a time-dependent state or a state denoted by a single integer.<br /></p>

<MarginText>3</MarginText>
<p>Other facilities are provided specifically for advanced applications. These include subprograms to save and restore the state of a given generator; a private type whose objects can be used to hold the saved state of a generator; and subprograms to obtain a string representation of a given generator state, or, given such a string representation, the corresponding state.] <br /></p>

<AnnotatedOnly>
<MarginText>3.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>These facilities support a variety of requirements ranging from repeatable sequences (for debugging) to unique sequences in each execution of a program. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>41.a/5</MarginText>
<MarginInfo items={["AI12-0005-1", "AI12-0144-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>In this rule, &ldquo;consecutive&rdquo; means at least that there are no intervening explicit calls involving the same generator. This restricts the rule to only applying to cases where just the Random function changes the generator. Operations that use the resulting random values (for instance, to store them somewhere) are not considered in determining if calls are consecutive. <br /></Admonition>

<Admonition type="aarm" aarm="reason">
<strong></strong>One might think that a simple transformation of the result of the floating point Random function such as Integer(Float(M) * Random(G)) <strong>mod</strong> M would give a uniform distribution. But this is only true if the period of the underlying generator is a multiple of M. (This usually requires that M be a power of two.) In other cases, the <strong>mod</strong> operation maps slightly more random values to some result values than others. It is easy to see this: consider a 4-bit random integer (with a range of 0 .. 15). If one <strong>mod</strong>s this by 6 to get a value in 0 .. 5 (to which one would add 1 to get the value of a die roll), 3 values would be mapped to each value 0 .. 3, but only 2 values would be mapped to 4 and 5. Even when the input is uniformly distributed, the output clearly is not. A similar effect occurs regardless of the number of bits in the random integer. Since it takes care to get this right, users should use the provided functions (which presumably do this correctly &ndash;  contains a correct algorithm) and resist the urge to &ldquo;roll-their-own&rdquo;. <br /></Admonition>

<dt><br/>S'Machine_Radix</dt>
<dl>
<dd>Yields the radix of the hardware representation of the type <em>T</em>. The value of this attribute is of the type <em>universal_integer</em>. <br /></dd>
</dl>
<MarginText>3</MarginText>
<p>The values of other representation-oriented attributes of a floating point subtype, and of the &ldquo;primitive function&rdquo; attributes of a floating point subtype described later, are defined in terms of a particular representation of nonzero values called the <em>canonical form</em>. The canonical form (for the type <em>T</em>) is the form<br />    ± <em>mantissa</em> · <em>T</em>'Machine_Radix<em>exponent</em><br />where <br /></p>

<MarginText>4</MarginText>
<ul>
<li><em>mantissa</em> is a fraction in the number base <em>T</em>'Machine_Radix, the first digit of which is nonzero, and<br /></li>
<MarginText>5</MarginText>
<li><em>exponent</em> is an integer. <br /></li>
</ul>
<MarginText>6</MarginText>
<dt><br/>S'Machine_Mantissa</dt>
<dl>
<dd>Yields the largest value of <em>p</em> such that every value expressible in the canonical form (for the type <em>T</em>), having a <em>p</em>-digit <em>mantissa</em> and an <em>exponent</em> between <em>T</em>'Machine_Emin and <em>T</em>'Machine_Emax, is a machine number (see <a href="../AA-3/AA-3.5#Subclause_3.5.7">3.5.7</a>) of the type <em>T</em>. This attribute yields a value of the type <em>universal_integer</em>. <br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Values of a type held in an extended register are, in general, not machine numbers of the type, since they cannot be expressed in the canonical form with a sufficiently short <em>mantissa</em>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>7</MarginText>
<dt><br/>S'Machine_Emin</dt>
<dl>
<dd>Yields the smallest (most negative) value of <em>exponent</em> such that every value expressible in the canonical form (for the type <em>T</em>), having a <em>mantissa</em> of <em>T</em>'Machine_Mantissa digits, is a machine number (see <a href="../AA-3/AA-3.5#Subclause_3.5.7">3.5.7</a>) of the type <em>T</em>. This attribute yields a value of the type <em>universal_integer</em>.<br /></dd>
<MarginText>8</MarginText>
<dt><br/>S'Machine_Emax</dt>
<dd>Yields the largest (most positive) value of <em>exponent</em> such that every value expressible in the canonical form (for the type <em>T</em>), having a <em>mantissa</em> of <em>T</em>'Machine_Mantissa digits, is a machine number (see <a href="../AA-3/AA-3.5#Subclause_3.5.7">3.5.7</a>) of the type <em>T</em>. This attribute yields a value of the type <em>universal_integer</em>. <br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Note that the above definitions do not determine unique values for the representation-oriented attributes of floating point types. The implementation may choose any set of values that collectively satisfies the definitions.<br /></Admonition>
</AnnotatedOnly>

<MarginText>9</MarginText>
<dt><br/>S'Denorm</dt>
<dl>
<dd>Yields the value True if every value expressible in the form<br />    ± <em>mantissa</em> · <em>T</em>'Machine_Radix<em>T</em>'Machine_Emin<br />where <em>mantissa</em> is a nonzero <em>T</em>'Machine_Mantissa-digit fraction in the number base <em>T</em>'Machine_Radix, the first digit of which is zero, is a machine number (see <a href="../AA-3/AA-3.5#Subclause_3.5.7">3.5.7</a>) of the type <em>T</em>; yields the value False otherwise. The value of this attribute is of the predefined type Boolean. <br /></dd>
</dl>
<MarginText>10</MarginText>
<p>The values described by the formula in the definition of S'Denorm are called <em>denormalized numbers</em>. A nonzero machine number that is not a denormalized number is a <em>normalized number</em>. A normalized number <em>x</em> of a given type <em>T</em> is said to be <em>represented in canonical form</em> when it is expressed in the canonical form (for the type <em>T</em>) with a <em>mantissa</em> having <em>T</em>'Machine_Mantissa digits; the resulting form is the <em>canonical-form representation</em> of <em>x</em>. <br /></p>

<AnnotatedOnly>
<MarginText>10.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The intent is that S'Denorm be True when such denormalized numbers exist and are generated in the circumstances defined by IEC 559:1989, though the latter requirement is not formalized here. <br /></Admonition>
</AnnotatedOnly>

<MarginText>11</MarginText>
<dt><br/>S'Machine_Rounds</dt>
<dl>
<dd>Yields the value True if rounding is performed on inexact results of every predefined operation that yields a result of the type <em>T</em>; yields the value False otherwise. The value of this attribute is of the predefined type Boolean. <br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>11.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>It is difficult to be more precise about what it means to round the result of a predefined operation. If the implementation does not use extended registers, so that every arithmetic result is necessarily a machine number, then rounding seems to imply two things: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.b</MarginText>
<ul>
<li>S'Model_Mantissa = S'Machine_Mantissa, so that operand preperturbation never occurs;<br /></li>
<MarginText>11.c</MarginText>
<li>when the exact mathematical result is not a machine number, the result of a predefined operation must be the nearer of the two adjacent machine numbers. <br /></li>
</ul>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>11.d</MarginText>
<Admonition type="aarm" aarm="note">
Technically, this attribute should yield False when extended registers are used, since a few computed results will cross over the half-way point as a result of double rounding, if and when a value held in an extended register has to be reduced in precision to that of the machine numbers. It does not seem desirable to preclude the use of extended registers when S'Machine_Rounds could otherwise be True. <br /></Admonition>
</AnnotatedOnly>

<MarginText>12</MarginText>
<dt><br/>S'Machine_Overflows</dt>
<dl>
<dd>Yields the value True if overflow and divide-by-zero are detected and reported by raising Constraint_Error for every predefined operation that yields a result of the type <em>T</em>; yields the value False otherwise. The value of this attribute is of the predefined type Boolean.<br /></dd>
<MarginText>13</MarginText>
<dt><br/>S'Signed_Zeros</dt>
<dd>Yields the value True if the hardware representation for the type <em>T</em> has the capability of representing both positively and negatively signed zeros, these being generated and used by the predefined operations of the type <em>T</em> as specified in IEC 559:1989; yields the value False otherwise. The value of this attribute is of the predefined type Boolean. <br /></dd>
</dl>
<MarginText>14</MarginText>
<p>For every value <em>x</em> of a floating point type <em>T</em>, the <em>normalized exponent</em> of <em>x</em> is defined as follows: <br /></p>

<MarginText>15</MarginText>
<ul>
<li>the normalized exponent of zero is (by convention) zero;<br /></li>
<MarginText>16</MarginText>
<li>for nonzero <em>x</em>, the normalized exponent of <em>x</em> is the unique integer <em>k</em> such that <em>T</em>'Machine_Radix<em>k</em>&ndash;1 &le; |<em>x</em>| {"<"} <em>T</em>'Machine_Radix<em>k</em>. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>16.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The normalized exponent of a normalized number <em>x</em> is the value of <em>exponent</em> in the canonical-form representation of <em>x</em>.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16.b</MarginText>
<Admonition type="aarm" aarm="note">
The normalized exponent of a denormalized number is less than the value of <em>T</em>'Machine_Emin. <br /></Admonition>
</AnnotatedOnly>

<dt><br/>S'Machine</dt>
<dl>
<dd>S'Machine denotes a function with the following specification: <br /></dd>
</dl>
<MarginText>61</MarginText>
<CodeBlock language="ada">
function S'Machine (X : T){"\n"}  return T{"\n"}

</CodeBlock>
<MarginText>62</MarginText>
<dl>
<dd>If <em>X</em> is a machine number of the type <em>T</em>, the function yields <em>X</em>; otherwise, it yields the value obtained by rounding or truncating <em>X</em> to either one of the adjacent machine numbers of the type <em>T</em>. Constraint_Error is raised if rounding or truncating <em>X</em> to the precision of the machine numbers results in a value outside the base range of S. A zero result has the sign of <em>X</em> when S'Signed_Zeros is True. <br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>62.a/3</MarginText>
<MarginInfo items={["AI05-0005-1"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>All of the primitive function attributes except Rounding and Machine correspond to subprograms in the Generic_Primitive_Functions generic package that was proposed as a separate ISO standard (ISO/IEC DIS 11729) for Ada 83. The Scaling, Unbiased_Rounding, and Truncation attributes correspond to the Scale, Round, and Truncate functions, respectively, in Generic_Primitive_Functions. The Rounding attribute rounds away from zero; this functionality was not provided in Generic_Primitive_Functions. The name Round was not available for either of the primitive function attributes that perform rounding, since an attribute of that name is used for a different purpose for decimal fixed point types. Likewise, the name Scale was not available, since an attribute of that name is also used for a different purpose for decimal fixed point types. The functionality of the Machine attribute was also not provided in Generic_Primitive_Functions. The functionality of the Decompose procedure of Generic_Primitive_Functions is only provided in the form of the separate attributes Exponent and Fraction. The functionality of the Successor and Predecessor functions of Generic_Primitive_Functions is provided by the extension of the existing Succ and Pred attributes. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>62.b</MarginText>
