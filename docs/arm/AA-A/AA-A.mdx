#### Implementation Requirements

<MarginText>3/5</MarginText>
<MarginInfo items={["AI95-00434-01", "AI12-0052-1", "AI12-0114-1", "AI12-0200-1", "AI12-0439-1"]} />
<p>The implementation shall ensure that  concurrent calls on any two (possibly the same) language-defined subprograms  perform as specified, so long as all pairs of objects (one from each call) that are either denoted by parameters that can  be passed by reference, or are designated by parameters of an access type, are nonoverlapping. <br /></p>

<AnnotatedOnly>
<MarginText>3.a.1/4</MarginText>
<MarginInfo items={["AI12-0052-1", "AI12-0114-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>So long as the parameters are disjoint, concurrent calls on the same language-defined subprogram, and concurrent calls on two different language-defined subprograms are required to work. But concurrent calls operating on overlapping objects (be they of the same or different language-defined subprograms) are <em>not</em> required to work (being an erroneous use of shared variables) unless both subprograms are required to pass the associated parameter by-copy.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.a</MarginText>
<Admonition type="aarm" aarm="note">
For example, simultaneous calls to Text_IO.Put will work properly, so long as they are going to two different files. On the other hand, simultaneous output to the same file constitutes erroneous use of shared variables. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.b</MarginText>
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>Here, &ldquo;language defined subprogram&rdquo; means a language defined library subprogram, a subprogram declared in the visible part of a language defined library package, an instance of a language defined generic library subprogram, or a subprogram declared in the visible part of an instance of a language defined generic library package. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.b.1/4</MarginText>
<MarginInfo items={["AI12-0052-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>This rule applies to all language-defined subprograms, including those defined in packages that manage some global state (like environment variables or the current directory). Unless specified above, such subprograms need to work when the explicit parameters are not overlapping; in particular, the existence of the global state is not considered.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>3.c/4</MarginText>
<MarginInfo items={["AI12-0052-1"]} />
<Admonition type="aarm" aarm="note">
The rule implies that any data local to the private part or body of the package (including global state as described above) has to be somehow protected against simultaneous access. <br /></Admonition>
</AnnotatedOnly>

<MarginText>3.1/4</MarginText>
<MarginInfo items={["AI12-0052-1", "AI12-0159-1"]} />
<p>For the purpose of determining whether concurrent calls on text input-output subprograms are required to perform as specified above, when calling a subprogram within Text_IO or its children that implicitly operates on one of the default input-output files, the subprogram is considered to have a parameter of Current_Input or Current_Output (as appropriate).<br /></p>

<MarginText>3.2/3</MarginText>
<MarginInfo items={["AI05-0048-1"]} />
<p>If a descendant of a language-defined tagged type is declared, the implementation shall ensure that each inherited language-defined subprogram behaves as described in this Reference Manual. In particular, overriding a language-defined subprogram shall not alter the effect of any inherited language-defined subprogram. <br /></p>

<AnnotatedOnly>
<MarginText>3.d/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This means that internally the implementation must not do redispatching unless it is required by the Standard. So when we say that some subprogram Bar is equivalent to Foo, overriding Foo for a derived type doesn't change the semantics of Bar, and in particular it means that Bar may no longer be equivalent to Foo. The word &ldquo;equivalent&rdquo; is always a bit of a lie anyway. <br /></Admonition>
</AnnotatedOnly>


#### Implementation Permissions

<MarginText>4</MarginText>
<p>The implementation may restrict the replacement of language-defined compilation units. The implementation may restrict children of language-defined library units (other than Standard). <br /></p>

<AnnotatedOnly>
<MarginText>4.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>For example, the implementation may say, &ldquo;you cannot compile a library unit called System&rdquo; or &ldquo;you cannot compile a child of package System&rdquo; or &ldquo;if you compile a library unit called System, it has to be a package, and it has to contain at least the following declarations: ...&rdquo;. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.b/5</MarginText>
<MarginInfo items={["AI12-0112-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>Implementations are of course allowed to make changes to the specifications of language-defined units, so long as those changes are semantically neutral (that is, no program could change legality or effect because of the changes). In particular, an implementation can <strong>with</strong> additional units (especially implementation-defined units) so long as those units do not change the elaboration of the language-defined unit.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.c/5</MarginText>
<Admonition type="aarm" aarm="note">
Similarly, an implementation can add postconditions to language-defined subprograms, so long as those postconditions always evaluate to True. This is useful if the implementation can use those postconditions for optimization. <br /></Admonition>
</AnnotatedOnly>

<MarginInfo items={["AI12-0005-1"]} />
<Admonition type="aarm" aarm="note">
When a new entity <em>E</em> is added to a package <em>P</em> that is <strong>use</strong>d in client code, <em>use clause conflicts</em> are possible. Specifically, if <em>P</em> is referenced in a <code><a href="../AA-8/AA-8.4#S0235">use_clause</a></code>, and an entity <em>F</em> with the same defining identifier as <em>E</em> is defined in some other package that is also referenced in a <em>use_clause</em>, the user-defined entity <em>F</em> may no longer be use-visible, resulting in errors. This is an incompatibility when the new entity is added to a language-defined package. Note that use clause conflicts are rare and easily fixed by using an expanded name. <br /></Admonition>
