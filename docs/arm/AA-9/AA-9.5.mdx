<MarginText>1</MarginText>
<p>The primary means for intertask communication is provided by calls on entries and protected subprograms. Calls on protected subprograms allow coordinated access to shared data objects. Entry calls allow for blocking the caller until a given condition is satisfied (namely, that the corresponding entry is open &mdash; see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.3">9.5.3</a>), and then communicating data or control information directly with another task or indirectly via a shared protected object.<br /></p>


#### Static Semantics

<MarginText>2/3</MarginText>
<MarginInfo items={["AI05-0225-1", "AI05-0291-1"]} />
<p>When a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> or <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> denotes an entry, protected subprogram, or a prefixed view of a primitive subprogram of a limited interface whose first parameter is a controlling parameter, the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> or <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> determines a <i>target object</i>, as follows: <br /></p>

<AnnotatedOnly>
<MarginText>2.a/3</MarginText>
<MarginInfo items={["AI05-0291-1"]} />
<Admonition type="aarm" aarm="note">
<b>To be honest: </b>This wording uses "denotes" to mean "denotes a view of an entity" (when the term is used in Legality Rules), and "denotes an entity" (when the term is used in Dynamic Semantics rules). It does not mean "view of a declaration", as that would not include renames (a renames is not an entry or protected subprogram). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>3/3</MarginText>
<MarginInfo items={["AI05-0291-1"]} />
<ul>
<li>If it is a <code><a href="/docs/arm/AA-4/AA-4.1#S0092">direct_name</a></code> or expanded name that denotes the declaration (or body) of the operation, then the target object is implicitly specified to be the current instance of the task or protected unit immediately enclosing the operation; a call using such a name is defined to be an <i>internal call</i>;<br /></li>
<MarginText>4/3</MarginText>
<MarginInfo items={["AI05-0291-1"]} />
<li>If it is a <code><a href="/docs/arm/AA-4/AA-4.1#S0098">selected_component</a></code> that is not an expanded name, then the target object is explicitly specified to be the object denoted by the <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> of the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code>; a call using such a name is defined to be an <i>external call</i>; <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>4.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>For example: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>4.b</MarginText>
<CodeBlock language="ada">
protected type Pt is{"\n"}  procedure Op1;{"\n"}  procedure Op2;{"\n"}end Pt;{"\n"}
<MarginText>4.c</MarginText>
PO : Pt;{"\n"}Other{"_"}Object : Some{"_"}Other{"_"}Protected{"_"}Type;{"\n"}
<MarginText>4.d</MarginText>
protected body Pt is{"\n"}  procedure Op1 is begin ... end Op1;{"\n"}
<MarginText>4.e/5</MarginText>
<MarginInfo items={["AI12-0005-1"]} />
  procedure Op2 is{"\n"}  begin{"\n"}    Op1; -- An internal call.{"\n"}    Pt.Op1; -- Another internal call.{"\n"}    PO.Op1; -- An external call. If the current instance is PO, then{"\n"}            -- this is a bounded error (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.1">9.5.1</a>).{"\n"}    Other{"_"}Object.Some{"_"}Op; -- An external call.{"\n"}  end Op2;{"\n"}end Pt;{"\n"}

</CodeBlock>
</AnnotatedOnly>
<MarginText>5/3</MarginText>
<MarginInfo items={["AI05-0291-1"]} />
<ul>
<li>If the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> or <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> is a dereference (implicit or explicit) of an access-to-protected-subprogram value, then the target object is determined by the <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> of the Access <code><a href="/docs/arm/AA-4/AA-4.1#S0100">attribute_reference</a></code> that produced the access value originally; a call using such a name is defined to be an <i>external call</i>;<br /></li>
<MarginText>6</MarginText>
<li>If the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> or <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> denotes a <code><a href="/docs/arm/AA-8/AA-8.5#S0242">subprogram_renaming_declaration</a></code>, then the target object is as determined by the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> of the renamed entity.<br /></li>
</ul>
<MarginText>6.1/3</MarginText>
<MarginInfo items={["AI05-0291-1"]} />
<p>A call on an entry or a protected subprogram either uses a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> or <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> that determines a target object implicitly, as above, or is a call on (a non-prefixed view of) a primitive subprogram of a limited interface whose first parameter is a controlling parameter, in which case the target object is identified explicitly by the first parameter. This latter case is an <i>external call</i>.<br /></p>

<MarginText>7</MarginText>
<p>A corresponding definition of target object applies to a <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code> (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.4">9.5.4</a>), with a corresponding distinction between an <i>internal requeue</i> and an <i>external requeue</i>.<br /></p>


#### Legality Rules

<MarginText>7.1/3</MarginText>
<MarginInfo items={["AI95-00345-01", "AI05-0225-1", "AI05-0291-1"]} />
<p>If a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> or <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> determines a target object, and the name denotes a protected entry or procedure, then the target object shall be a variable, unless the <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> is for an <code><a href="/docs/arm/AA-4/AA-4.1#S0100">attribute_reference</a></code> to the Count attribute (see <a href="/docs/arm/AA-9/AA-9.9">9.9</a>). <br /></p>

<AnnotatedOnly>
<MarginText>7.a/3</MarginText>
<MarginInfo items={["AI05-0225-1"]} />
<Admonition type="aarm" aarm="reason">
<b></b>The point is to prevent any calls to such a <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> whose target object is a constant view of a protected object, directly, or via an access value, renames, or generic formal subprogram. It is, however, legal to say P'Count in a protected function body, even though the protected object is a constant view there. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.b/3</MarginText>
<MarginInfo items={["AI05-0291-1"]} />
<Admonition type="aarm" aarm="ramification">
<b></b>This rule does not apply to calls that are not to a prefixed view. Specifically a "normal" call to a primitive operation of a limited interface is not covered by this rule. In that case, the normal parameter passing mode checks will prevent passing a constant protected object to an operation implemented by a protected entry or procedure as the mode is required to be <b>in out</b> or <b>out</b>. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>7.2/5</MarginText>
<MarginInfo items={["AI12-0166-1"]} />
<p>An internal call on a protected function shall not occur within a precondition expression (see <a href="/docs/arm/AA-6/AA-6.1#Subclause_6.1.1">6.1.1</a>) of a protected operation nor within a <code><a href="/docs/arm/AA-3/AA-3.7#S0063">default_expression</a></code> of a <code><a href="/docs/arm/AA-6/AA-6.1#S0207">parameter_specification</a></code> of a protected operation.<br /></p>

<AnnotatedOnly>
<MarginText>7.c/5</MarginText>
<MarginInfo items={["AI125-0166-1"]} />
<Admonition type="aarm" aarm="reason">
<b></b>These calls will be made before the start of the protected action, and thus would not be subject to the expected mutual exclusion. As such, they would be an automatic race condition (the state of the called object could change before the start of the protected action for the call on the protected entry or subprogram). <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.d/5</MarginText>
<MarginInfo items={["AI125-0166-1"]} />
<Admonition type="aarm" aarm="note">
<b>To be honest: </b><a href="/docs/arm/AA-6/AA-6.1#Subclause_6.1.1">6.1.1</a> actually defines "specific precondition expression" and "class-wide precondition expression". This rule is intended to apply to both. <br />
</Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>8</MarginText>
<p>Within the body of a protected operation, the current instance (see <a href="/docs/arm/AA-8/AA-8.6">8.6</a>) of the immediately enclosing protected unit is determined by the target object specified (implicitly or explicitly) in the call (or requeue) on the protected operation. <br /></p>

<AnnotatedOnly>
<MarginText>8.a</MarginText>
<Admonition type="aarm" aarm="note">
<b>To be honest: </b>The current instance is defined in the same way within the body of a subprogram declared immediately within a <code><a href="/docs/arm/AA-9/AA-9.4#S0254">protected_body</a></code>. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>9</MarginText>
<p>Any call on a protected procedure or entry of a target protected object is defined to be an update to the object, as is a requeue on such an entry. <br /></p>

<AnnotatedOnly>
<MarginText>9.a</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>Read/write access to the components of a protected object is granted while inside the body of a protected procedure or entry. Also, any protected entry call can change the value of the Count attribute, which represents an update. Any protected procedure call can result in servicing the entries, which again might change the value of a Count attribute. <br />
</Admonition>
</AnnotatedOnly>

#### Legality Rules

<MarginText>14/3</MarginText>
<MarginInfo items={["AI05-0030-2", "AI05-0215-1"]} />
<p>The <code><a href="/docs/arm/AA-9/AA-9.5#S0256">synchronization_kind</a></code> By{"_"}Protected{"_"}Procedure shall not be applied to a primitive procedure of a task interface.<br /></p>

<MarginText>15/3</MarginText>
<MarginInfo items={["AI05-0030-2", "AI05-0215-1"]} />
<p>A procedure for which the specified <code><a href="/docs/arm/AA-9/AA-9.5#S0256">synchronization_kind</a></code> is By{"_"}Entry shall be implemented by an entry. A procedure for which the specified <code><a href="/docs/arm/AA-9/AA-9.5#S0256">synchronization_kind</a></code> is By{"_"}Protected{"_"}Procedure shall be implemented by a protected procedure. A procedure for which the specified <code><a href="/docs/arm/AA-9/AA-9.5#S0256">synchronization_kind</a></code> is Optional may be implemented by an entry or by a procedure (including a protected procedure).<br /></p>

<MarginText>16/3</MarginText>
<MarginInfo items={["AI05-0030-2", "AI05-0215-1"]} />
<p>If a primitive procedure overrides an inherited operation for which the Synchronization aspect has been specified to be By{"_"}Entry or By{"_"}Protected{"_"}Procedure, then any specification of the aspect Synchronization applied to the overriding operation shall have the same <code><a href="/docs/arm/AA-9/AA-9.5#S0256">synchronization_kind</a></code>.<br /></p>

<MarginText>17/3</MarginText>
<MarginInfo items={["AI05-0030-2"]} />
<p>In addition to the places where Legality Rules normally apply (see <a href="/docs/arm/AA-12/AA-12.3">12.3</a>), these rules also apply in the private part of an instance of a generic unit. <br /></p>


#### Static Semantics

<MarginText>18/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2"]} />
<p>For a program unit, task entry, formal package, formal subprogram, formal object of an anonymous access-to-subprogram type, enumeration literal, and for a subtype (including a formal subtype), the following language-defined operational aspect is defined:<br /></p>

<MarginText>19/5</MarginText>
<dt><br/>Nonblocking</dt>
<dl>
<dd>This aspect specifies the blocking restriction for the entity; it shall be specified by a static Boolean expression. [The <code><a href="/docs/arm/AA-13/AA-13.1#S0348">aspect_definition</a></code> can be omitted from the specification of this aspect; in that case, the aspect for the entity is True.]<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>19.a/5</MarginText>
<Admonition type="aarm" aarm="note">
<b>Aspect Description for </b><b>Nonblocking: </b>Specifies that an associated subprogram does not block.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>19.b/5</MarginText>
<Admonition type="aarm" aarm="proof">
<b></b><a href="/docs/arm/AA-13/AA-13.1#Subclause_13.1.1">13.1.1</a> allows omitting the aspect <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> for any aspect with type Boolean; we take advantage of that here. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>20/5</MarginText>
<MarginInfo items={["AI12-0064-2"]} />
<dl>
<dd>The Nonblocking aspect may be specified for all entities for which it is defined, except for protected operations and task entries. In particular, Nonblocking may be specified for generic formal parameters.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>20.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>The Nonblocking aspect cannot be specified for predefined operators or enumeration literals but we don't need to mention that above. One would have to declare a subprogram in order to specify the aspect in those cases, but that defines a user-defined subprogram which is itself not a predefined operator or an enumeration literal. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>21/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2", "AI12-0439-1"]} />
<dl>
<dd>When aspect Nonblocking is False for an entity, the entity can contain a potentially blocking operation; such an entity <i>allows blocking</i>. If the aspect is True for an entity, the entity is said to be <i>nonblocking</i>.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>21.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Specifying Nonblocking as False imposes no restrictions. Specifying Nonblocking as True imposes additional compile-time checks to prevent blocking, but does not prevent deadlock. A pragma Detect{"_"}Blocking can be used to ensure that Program{"_"}Error is raised in a deadlock situation. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>22/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2"]} />
<dl>
<dd>For a generic instantiation and entities declared within such an instance, the aspect is determined by the Nonblocking aspect for the corresponding entity of the generic unit, <b>and</b>ed with the Nonblocking aspects of the actual generic parameters <i>used</i> by the entity. If the aspect is directly specified for an instance, the specified expression shall have the same value as the Nonblocking aspect of the instance (after <b>and</b>ing with the aspects of the used actual parameters). In the absence of a Use{"_"}Formal aspect, all actual generic parameters are presumed to be <i>used</i> by an entity (see <a href="/docs/arm/AA-H/AA-H.7#Subclause_H.7.1">H.7.1</a>).<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>22.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>We want to allow confirming aspects for instances, but nothing else. The Legality Rules of the generic body were checked assuming the Nonblocking aspect of the generic unit combined with the Nonblocking aspects of the formals where they are used, and if that is overridden on the instance, the instance body might make calls that allow blocking in subprograms that are declared nonblocking. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>23/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2"]} />
<dl>
<dd>For a (protected or task) entry, the Nonblocking aspect is False.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>23.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>An entry can be renamed as a procedure, so the value of the aspect has to be well-defined (as the attribute can be applied to a procedure). We do not want a nonblocking subprogram to be able to call an entry, no matter how it occurs, so the value ought to be False. Moreover, we do not want a subprogram that renames an entry to be able to override a nonblocking subprogram. We could have used individual rules for these cases, but there were already many of them, and this solution avoids the need for extra rules for entries. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>24/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2"]} />
<dl>
<dd>For an enumeration literal, the Nonblocking aspect is True.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>24.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>Enumeration literals can be renamed as functions, and passed to generic formal functions, so we need to define the value of the aspect to ensure the other rules are meaningful. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>25/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2"]} />
<dl>
<dd>For a predefined operator of an elementary type, the Nonblocking aspect is True. For a predefined operator of a composite type, the Nonblocking aspect of the operator is the same as the Nonblocking aspect for the type.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>25.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>Predefined operators of elementary types can never include any potentially blocking operations, so we want them to declare that. Record equality can be composed of operations including user-defined "=" operators, which might allow blocking. Array equality might use some record equality. So we have to have the possibility of allowing blocking for them. We don't just copy the Nonblocking aspect of the type in every case, as someone could declare an elementary type to allow blocking. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>25.b/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>It's not possible to specify the nonblocking expression of a predefined operator; if an operator is declared in order to do that, it is no longer predefined. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>26/5</MarginText>
<MarginInfo items={["AI12-0064-2"]} />
<dl>
<dd>For a dereference of an access-to-subprogram type, the Nonblocking aspect of the designated subprogram is that of the access-to-subprogram type.<br /></dd>
<MarginText>27/5</MarginText>
<MarginInfo items={["AI12-0374-2"]} />
<dd>For the base subtype of a scalar (sub)type, the Nonblocking aspect is True.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>27.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>The first subtype of a scalar type can allow blocking (which can be useful so a predicate can allow blocking), but the base subtype is always Nonblocking. We need this so the Nonblocking value is well-defined for any subtype that is built from the base subtype (T'Base). T'Base of any scalar type, including a generic formal type, is always nonblocking. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>28/5</MarginText>
<MarginInfo items={["AI12-0064-2"]} />
<dl>
<dd>For an inherited primitive dispatching subprogram that is null or abstract, the subprogram is nonblocking if and only if a corresponding subprogram of at least one ancestor is nonblocking. For any other inherited subprogram, it is nonblocking if and only if the corresponding subprogram of the parent is nonblocking.<br /></dd>
<MarginText>29/5</MarginText>
<MarginInfo items={["AI12-0064-2"]} />
<dd>Unless directly specified, overridings of dispatching operations inherit this aspect.<br /></dd>
<MarginText>30/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2"]} />
<dd>Unless directly specified, for a formal subtype, formal package, or formal subprogram, the Nonblocking aspect is that of the actual subtype, package, or subprogram.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>30.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>This means that Nonblocking legality checking for the actual parameters of the instance is only necessary when the aspect is explicitly specified for the formal type. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>31/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2"]} />
<dl>
<dd>Unless directly specified, for a non-first subtype <i>S</i>, the Nonblocking aspect is that of the subtype identified in the subtype{"_"}indication defining <i>S</i>; unless directly specified for the first subtype of a derived type, the Nonblocking aspect is that of the ancestor subtype.<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>31.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>The expressions that can be specified for a such a subtype are limited by a Legality Rule, see below. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>32/5</MarginText>
<MarginInfo items={["AI12-0064-2"]} />
<dl>
<dd>Unless directly specified, for any other program unit, first subtype, or formal object, the Nonblocking aspect of the entity is determined by the Nonblocking aspect for the innermost program unit enclosing the entity.<br /></dd>
<MarginText>33/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2"]} />
<dd>If not specified for a library unit, the Nonblocking aspect is True if the library unit is declared pure, or False otherwise.<br /></dd>
</dl>
<MarginText>34/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0247-1"]} />
<p>The following are defined to be <i>potentially blocking</i> operations: <br /></p>

<AnnotatedOnly>
<MarginText>34.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>The primary purpose of these rules is to define what operations are not allowed in a protected operation (blocking is not allowed). Some of these operations are not directly blocking. However, they are still treated as potentially blocking, because allowing them in a protected action might impose an undesirable implementation burden. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>35/5</MarginText>
<ul>
<li>a <code><a href="/docs/arm/AA-9/AA-9.7#S0269">select_statement</a></code>;<br /></li>
<MarginText>36/5</MarginText>
<li>an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code>;<br /></li>
<MarginText>37/5</MarginText>
<li>an <code><a href="/docs/arm/AA-9/AA-9.5#S0264">entry_call_statement</a></code>, or a call on a procedure that renames or is implemented by an entry;<br /></li>
<MarginText>38/5</MarginText>
<li>a <code><a href="/docs/arm/AA-9/AA-9.6#S0266">delay_statement</a></code>;<br /></li>
<MarginText>39/5</MarginText>
<li>an <code><a href="/docs/arm/AA-9/AA-9.8#S0284">abort_statement</a></code>;<br /></li>
<MarginText>40/5</MarginText>
<li>task creation or activation;<br /></li>
<MarginText>41/5</MarginText>
<li>during a protected action, an external call on a protected subprogram (or an external requeue) with the same target object as that of the protected action.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>41.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>This is really a deadlocking call, rather than a blocking call, but we include it in this list for simplicity. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>42/5</MarginText>
<MarginInfo items={["AI12-0064-2"]} />
<p>If a language-defined subprogram allows blocking, then a call on the subprogram is a potentially blocking operation.<br /></p>

<AnnotatedOnly>
<MarginText>42.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Calls on other subprograms that allow blocking are not themselves potentially blocking; the execution of the body could execute a potentially blocking operation.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>42.b/5</MarginText>
<Admonition type="aarm" aarm="note">
A user-defined instance of a language-defined generic creates user-defined subprograms for the purpose of this rule. A dispatching call to a language-defined abstract subprogram always calls a user-defined concrete subprogram, so that too is not potentially blocking for the purposes of this rule. <br />
</Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>43/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0267-1", "AI12-0374-2"]} />
<p>A portion of program text is called a <i>nonblocking region</i> if it is anywhere within a parallel construct, or if the innermost enclosing program unit is nonblocking. A nonblocking region shall not contain any of the following:<br /></p>

<MarginText>44/5</MarginText>
<ul>
<li>a <code><a href="/docs/arm/AA-9/AA-9.7#S0269">select_statement</a></code>;<br /></li>
<MarginText>45/5</MarginText>
<li>an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code>;<br /></li>
<MarginText>46/5</MarginText>
<li>a <code><a href="/docs/arm/AA-9/AA-9.6#S0266">delay_statement</a></code>;<br /></li>
<MarginText>47/5</MarginText>
<li>an <code><a href="/docs/arm/AA-9/AA-9.8#S0284">abort_statement</a></code>;<br /></li>
<MarginText>48/5</MarginText>
<li>task creation or activation. <br /></li>
</ul>
<MarginText>49/5</MarginText>
<MarginInfo items={["AI12-0374-2"]} />
<p>Furthermore, a parallel construct shall neither contain a call on a callable entity for which the Nonblocking aspect is False, nor shall it contain a call on a callable entity declared within a generic unit that uses a generic formal parameter with Nonblocking aspect False (see Use{"_"}Formal aspect in <a href="/docs/arm/AA-H/AA-H.7#Subclause_H.7.1">H.7.1</a>).<br /></p>

<MarginText>50/5</MarginText>
<MarginInfo items={["AI12-0374-2"]} />
<p>Finally, a nonblocking region that is outside of a parallel construct shall not contain a call on a callable entity for which the Nonblocking aspect is False, unless the region is within a generic unit and the callable entity is associated with a generic formal parameter of the generic unit, or the call is within the <code><a href="/docs/arm/AA-13/AA-13.1#S0348">aspect_definition</a></code> of an assertion aspect for an entity that allows blocking.<br /></p>

<AnnotatedOnly>
<MarginText>50.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>A generic unit or entity declared within one is presumed to use its "used" generic formal parameters at least once each time it is invoked, and this passes through to the parallel construct check. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>50.b/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Implicit calls for finalization, storage pools, and the like are covered by the above prohibition. The rules above say &ldquo;a call&rdquo;, not &ldquo;an explicit call&rdquo;. Such calls are considered statically bound when that is possible, that is, when the controlling object has a known specific type (even if the actual implementation uses dispatching). <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>50.c/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>We don't need to worry specially about entry calls (even if the entry has been renamed as a procedure), as they will be detected by the prohibition against calls to entities with the Nonblocking aspect False.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>50.d/5</MarginText>
<Admonition type="aarm" aarm="note">
Similarly, we don't need to specially worry about subprograms of limited interfaces that are implemented by entries, as any such subprogram necessarily has the value statically False for the Nonblocking aspect, and thus is already covered by the prohibition against calling such subprograms. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>51/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2"]} />
<p>For the purposes of the above rules, an <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code> is considered nonblocking if the immediately enclosing protected unit is nonblocking.<br /></p>

<AnnotatedOnly>
<MarginText>51.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>An entry declaration always allows blocking (by rule); but we want to be able to compile-time check for most violations of the prohibition against potentially blocking operations in a protected action (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.1">9.5.1</a>). We do that by using the nonblocking status of the protected unit as the controlling factor, and enforce that by not allowing the specification of the Nonblocking aspect for any protected operation. We can't do this checking unconditionally, as that would be incompatible: existing Ada protected units might call subprograms that allow blocking. Thus a protected unit that allows blocking (which is the default) must allow calling any subprogram from an entry body. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>52/5</MarginText>
<MarginInfo items={["AI12-0374-2"]} />
<p>For a subtype for which aspect Nonblocking is True, any predicate expression that applies to the subtype shall only contain constructs that are allowed immediately within a nonblocking program unit.<br /></p>

<MarginText>53/5</MarginText>
<MarginInfo items={["AI12-0064-2"]} />
<p>A subprogram shall be nonblocking if it overrides a nonblocking dispatching operation. An entry shall not implement a nonblocking procedure. If an inherited dispatching subprogram allows blocking, then the corresponding subprogram of each ancestor shall allow blocking.<br /></p>

<AnnotatedOnly>
<MarginText>53.a/5</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>Rules elsewhere in the standard (<a href="/docs/arm/AA-4/AA-4.6">4.6</a> and <a href="/docs/arm/AA-3/AA-3.10#Subclause_3.10.2">3.10.2</a>) ensure that access-to-subprogram conversion and the Access attribute enforce nonblocking. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>53.b/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>A nonblocking subprogram can override one that allows blocking, but the reverse is illegal. Thus one can declare a Finalize subprogram to be nonblocking, even though it overrides a routine that allows blocking. (This works because a nonblocking subprogram allows a strict subset of the operations allowed in allows blocking subprograms, so calling such a subprogram as if it allows blocking &mdash; as is necessary in a dispatching call &mdash; is harmless.) <br />
</Admonition>
</AnnotatedOnly>

<MarginText>54/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2", "AI12-0396-1", "AI12-0399-1"]} />
<p>It is illegal to directly specify aspect Nonblocking for the first subtype of the full view of a type that has a partial view. If the Nonblocking aspect of the full view is inherited, it shall have the same value as that of the partial view, or have the value True.<br /></p>

<AnnotatedOnly>
<MarginText>54.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>We need completions to agree on nonblocking with the original view. One reason this is necessary to prevent the predefined equality operator from being nonblocking in the partial view and allowing blocking in the full view. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>55/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2"]} />
<p>Aspect Nonblocking shall be directly specified for the first subtype of a derived type only if it has the same value as the Nonblocking aspect of the ancestor subtype or if it is specified True. Aspect Nonblocking shall be directly specified for a nonfirst subtype <i>S</i> only if it has the same value as the Nonblocking aspect of the subtype identified in the <code><a href="/docs/arm/AA-3/AA-3.2#S0027">subtype_indication</a></code> defining <i>S</i> or if it is specified True.<br /></p>

<AnnotatedOnly>
<MarginText>55.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>Boolean-valued aspects have a similar rule to the first rule here (see <a href="/docs/arm/AA-13/AA-13.1#Subclause_13.1.1">13.1.1</a>), we want this one to work similarly. We need non-first subtypes to allow blocking only if the original first subtype allows blocking, as that allows the programmer to know that any operation on any subtype of a type are nonblocking if the first subtype is nonblocking. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>56/5</MarginText>
<MarginInfo items={["AI12-0319-1"]} />
<p>For an access-to-object type that is nonblocking, the Allocate, Deallocate, and Storage{"_"}Size operations on its storage pool shall be nonblocking.<br /></p>

<AnnotatedOnly>
<MarginText>56.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Standard storage pools always have nonblocking operations by definition (see <a href="/docs/arm/AA-13/AA-13.11">13.11</a>), so this rule only can fail for user-defined storage pools. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>57/5</MarginText>
<MarginInfo items={["AI12-0319-1"]} />
<p>For a composite type that is nonblocking:<br /></p>

<MarginText>58/5</MarginText>
<ul>
<li>All component subtypes shall be nonblocking;<br /></li>
<MarginText>59/5</MarginText>
<li>For a record type or extension, every call in the <code><a href="/docs/arm/AA-3/AA-3.7#S0063">default_expression</a></code> of a component (including discriminants) shall call an operation that is nonblocking;<br /></li>
<MarginText>60/5</MarginText>
<li>For a controlled type, the Initialize, Finalize, and Adjust (if any) subprograms shall be nonblocking.<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>60.a/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>These rules ensure that if a type is nonblocking, the default initialization, finalization, and assignment of the type are also nonblocking. This ensures that if a generic formal type is nonblocking, all of the basic operations of the actual type are nonblocking.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>60.b/5</MarginText>
<Admonition type="aarm" aarm="note">
Default initialization, finalization, and assignment of elementary types are always nonblocking, so we don't need any rules for those. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>61/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2"]} />
<p>The predefined equality operator for a composite type, unless it is for a record type or record extension and the operator is overridden by a primitive equality operator, is illegal if it is nonblocking and:<br /></p>

<MarginText>62/5</MarginText>
<ul>
<li>for a record type or record extension, the parent primitive "=" allows blocking; or<br /></li>
<MarginText>63/5</MarginText>
<li>some component is of a type <i>T</i>, and:<br /></li>
<MarginText>64/5</MarginText>
<li><i>T</i> is a record type or record extension that has a primitive "=" that allows blocking; or<br /></li>
<MarginText>65/5</MarginText>
<li><i>T</i> is neither a record type nor a record extension, and <i>T</i> has a predefined "=" that allows blocking. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>65.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>This applies to both record and array "=".<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>65.b/5</MarginText>
<Admonition type="aarm" aarm="note">
This check occurs when the equality operator is declared, so this rule effectively makes the type illegal if the rule is violated. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>65.c/5</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>We don't need to check this when the operator is overridden for a record type, as the body of the new definition of equality will enforce the rules, and there is no case where the predefined operator will re-emerge. We do have to check this for array types even if the operator is overridden, as the predefined operator will re-emerge in generics and record equality. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>66/5</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0374-2"]} />
<p>In a generic instantiation:<br /></p>

<MarginText>67/5</MarginText>
<ul>
<li>the actual subprogram corresponding to a nonblocking formal subprogram shall be nonblocking [(an actual that is an entry is not permitted in this case)];<br /></li>
<MarginText>68/5</MarginText>
<li>the actual subtype corresponding to a nonblocking formal subtype shall be nonblocking;<br /></li>
</ul>
<AnnotatedOnly>
<MarginText>68.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>We require matching for formal scalar or access-to-object types, even though their predefined operators are always nonblocking (and they re-emerge in the generic unit) - because a "blocking" predicate might apply to the actual subtype, which will also be enforced on operations of the formal type. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>69/5</MarginText>
<ul>
<li>the actual object corresponding to a formal object of a nonblocking access-to-subprogram type shall be of a nonblocking access-to-subprogram type;<br /></li>
<MarginText>70/5</MarginText>
<li>the actual instance corresponding to a nonblocking formal package shall be nonblocking.<br /></li>
</ul>
<MarginText>71/5</MarginText>
<MarginInfo items={["AI12-0064-2"]} />
<p>In addition to the places where Legality Rules normally apply (see <a href="/docs/arm/AA-12/AA-12.3">12.3</a>), the above rules also apply in the private part of an instance of a generic unit.<br /></p>

<AnnotatedOnly>
<MarginText>71.a/5</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>For a generic formal parameter to be nonblocking (thus, for these rules to apply), it has to explicitly specify aspect Nonblocking to be True. However, if not specified True, these rules do apply in the instance of the specification of the generic unit (the normal re-checking is needed). For instance, the body of an expression function might make a prohibited call. <br />
</Admonition>
</AnnotatedOnly>

NOTE   The <code><a href="/docs/arm/AA-9/AA-9.5#S0256">synchronization_kind</a></code> By{"_"}Protected{"_"}Procedure implies that the operation will not block. <br />
</Admonition>

<a id="Subclause_9.5.1"></a>

## 9.5.1  Protected Subprograms and Protected Actions

<MarginText>1_9.5.1</MarginText>
<p>A <i>protected subprogram</i> is a subprogram declared immediately within a <code><a href="/docs/arm/AA-9/AA-9.4#S0251">protected_definition</a></code>. Protected procedures provide exclusive read-write access to the data of a protected object; protected functions provide concurrent read-only access to the data. <br /></p>

<AnnotatedOnly>
<MarginText>1.a_9.5.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>A subprogram declared immediately within a <code><a href="/docs/arm/AA-9/AA-9.4#S0254">protected_body</a></code> is not a protected subprogram; it is an intrinsic subprogram. See <a href="/docs/arm/AA-6/AA-6.3#Subclause_6.3.1">6.3.1</a>, &ldquo;<a href="/docs/arm/AA-6/AA-6.3#Subclause_6.3.1">Conformance Rules</a>&rdquo;. <br />
</Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>2_9.5.1</MarginText>
<p>[Within the body of a protected function (or a function declared immediately within a <code><a href="/docs/arm/AA-9/AA-9.4#S0254">protected_body</a></code>), the current instance of the enclosing protected unit is defined to be a constant (that is, its subcomponents may be read but not updated). Within the body of a protected procedure (or a procedure declared immediately within a <code><a href="/docs/arm/AA-9/AA-9.4#S0254">protected_body</a></code>), and within an <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code>, the current instance is defined to be a variable (updating is permitted).] <br /></p>

<AnnotatedOnly>
<MarginText>2.a.1/3_9.5.1</MarginText>
<MarginInfo items={["AI05-0120-1"]} />
<Admonition type="aarm" aarm="proof">
<b></b>All constant views are defined in <a href="/docs/arm/AA-3/AA-3.3">3.3</a>, &ldquo;<a href="/docs/arm/AA-3/AA-3.3">Objects and Named Numbers</a>&rdquo;, anything not named there is a variable view. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>2.a_9.5.1</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>The current instance is like an implicit parameter, of mode <b>in</b> for a protected function, and of mode <b>in out</b> for a protected procedure (or protected entry). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>2.1/4_9.5.1</MarginText>
<MarginInfo items={["AI12-0129-1"]} />
<p>For a type declared by a <code><a href="/docs/arm/AA-9/AA-9.4#S0249">protected_type_declaration</a></code> or for the anonymous type of an object declared by a <code><a href="/docs/arm/AA-9/AA-9.4#S0250">single_protected_declaration</a></code>, the following language-defined type-related representation aspect may be specified:<br /></p>

<MarginText>2.2/4_9.5.1</MarginText>
<dt><br/>Exclusive{"_"}Functions</dt>
<dl>
<dd>The type of aspect Exclusive{"_"}Functions is Boolean. If not specified (including by inheritance), the aspect is False.<br /></dd>
<MarginText>2.3/4_9.5.1</MarginText>
<dt><br/></dt>
<dd>A value of True for this aspect indicates that protected functions behave in the same way as protected procedures with respect to mutual exclusion and queue servicing (see below).<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>2.a.1/4_9.5.1</MarginText>
<Admonition type="aarm" aarm="note">
<b>Aspect Description for </b><b>Exclusive{"_"}Functions: </b>Specifies mutual exclusion behavior of protected functions in a protected type.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>2.4/4_9.5.1</MarginText>
<MarginInfo items={["AI12-0129-1"]} />
<p>A protected procedure or entry is an <i>exclusive</i> protected operation. A protected function of a protected type <i>P</i> is an exclusive protected operation if the Exclusive{"_"}Functions aspect of <i>P</i> is True.<br /></p>


#### Dynamic Semantics

<MarginText>3_9.5.1</MarginText>
<p>For the execution of a call on a protected subprogram, the evaluation of the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> or <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> and of the parameter associations, and any assigning back of <b>in out</b> or <b>out</b> parameters, proceeds as for a normal subprogram call (see <a href="/docs/arm/AA-6/AA-6.4">6.4</a>). If the call is an internal call (see <a href="/docs/arm/AA-9/AA-9.5">9.5</a>), the body of the subprogram is executed as for a normal subprogram call. If the call is an external call, then the body of the subprogram is executed as part of a new <i>protected action</i> on the target protected object; the protected action completes after the body of the subprogram is executed. [A protected action can also be started by an entry call (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.3">9.5.3</a>).]<br /></p>

<MarginText>4/4_9.5.1</MarginText>
<MarginInfo items={["AI12-0129-1"]} />
<p>A new protected action is not started on a protected object while another protected action on the same protected object is underway, unless both actions are the result of a call on a nonexclusive protected function. This rule is expressible in terms of the execution resource associated with the protected object: <br /></p>

<MarginText>5/4_9.5.1</MarginText>
<MarginInfo items={["AI12-0129-1"]} />
<ul>
<li><i>Starting</i> a protected action on a protected object corresponds to <i>acquiring</i> the execution resource associated with the protected object, either for exclusive read-write access if the protected action is for a call on an exclusive protected operation, or for concurrent read-only access otherwise;<br /></li>
<MarginText>6_9.5.1</MarginText>
<li><i>Completing</i> the protected action corresponds to <i>releasing</i> the associated execution resource. <br /></li>
</ul>
<MarginText>7/4_9.5.1</MarginText>
<MarginInfo items={["AI12-0129-1"]} />
<p>[After performing an exclusive protected operation on a protected object, but prior to completing the associated protected action, the entry queues (if any) of the protected object are serviced (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.3">9.5.3</a>).]<br /></p>

<MarginText>7.1/5_9.5.1</MarginText>
<MarginInfo items={["AI12-0119-1"]} />
<p>If a parallel construct occurs within a protected action, no new logical threads of control are created. Instead, each element of the parallel construct that would have become a separate logical thread of control executes on the logical thread of control that is performing the protected action. If there are multiple such elements initiated at the same point, they execute in an arbitrary order.<br /></p>

<AnnotatedOnly>
<MarginText>7.a/5_9.5.1</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>It would be feasible to allow multiple logical threads of control within a protected action, but it would significantly complicate the definition of &ldquo;sequential&rdquo; and &ldquo;concurrent&rdquo; actions, since we generally presume that everything occuring within protected actions of a given protected object is sequential. We could simply disallow any use of parallel constructs, but that seems unnecessary, particularly as a parallel construct might be buried within a subprogram that is declared Nonblocking. <br />
</Admonition>
</AnnotatedOnly>


#### Bounded (Run-Time) Errors

<MarginText>8/5_9.5.1</MarginText>
<MarginInfo items={["AI12-0064-2"]} />
<p>During a protected action, it is a bounded error to invoke an operation that is potentially blocking (see <a href="/docs/arm/AA-9/AA-9.5">9.5</a>). <br /></p>

<p><i>Paragraphs 9 through 16 were moved to <a href="/docs/arm/AA-9/AA-9.5">9.5</a>.</i> <br /></p>

<MarginText>17/5_9.5.1</MarginText>
<MarginInfo items={["AI12-0439-1"]} />
<p>If the bounded error is detected, Program{"_"}Error is raised. If not detected, the bounded error can result in deadlock or a (nested) protected action on the same target object.<br /></p>

<AnnotatedOnly>
<MarginText>17.a/2_9.5.1</MarginText>
<MarginInfo items={["AI95-00305-01"]} />
<Admonition type="aarm" aarm="discussion">
<b></b>By &ldquo;nested protected action&rdquo;, we mean that an additional protected action can be started by another task on the same protected object. This means that mutual exclusion may be broken in this bounded error case. A way to ensure that this does not happen is to use pragma Detect{"_"}Blocking (see <a href="/docs/arm/AA-H/AA-H.5">H.5</a>). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>18/5_9.5.1</MarginText>
<MarginInfo items={["AI12-0064-2", "AI12-0247-1"]} />
<p>During a protected action, a call on a subprogram whose body contains a potentially blocking operation is a bounded error. If the bounded error is detected, Program{"_"}Error is raised; otherwise, the call proceeds normally. <br /></p>

<AnnotatedOnly>
<MarginText>18.b/5_9.5.1</MarginText>
<MarginInfo items={["AI12-0247-1"]} />
<Admonition type="aarm" aarm="reason">
<b></b>This allows an implementation to check and raise Program{"_"}Error as soon as a subprogram is called, rather than waiting to find out whether it actually reaches the potentially blocking operation. If the call proceeds normally, reaching the potentially blocking operation is a separate bounded error, covered by the previous rules. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>19/5_9.5.1</MarginText>
<MarginInfo items={["AI12-0276-1", "AI12-0440-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 1   If two tasks both try to start a protected action on a protected object, and at most one is calling a protected nonexclusive function, then only one of the tasks can proceed. Although the other task cannot proceed, it is not considered blocked, and it can be consuming processing resources while it awaits its turn. Unless there is an admission policy (see <a href="/docs/arm/AA-D/AA-D.4#Subclause_D.4.1">D.4.1</a>) in effect, there is no language-defined ordering or queuing presumed for tasks competing to start a protected action &mdash; on a multiprocessor such tasks can use busy-waiting; for further monoprocessor and multiprocessor considerations, see <a href="/docs/arm/AA-D/AA-D.3">D.3</a>, &ldquo;<a href="/docs/arm/AA-D/AA-D.3">Priority Ceiling Locking</a>&rdquo;. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>19.a_9.5.1</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>The intended implementation on a multi-processor is in terms of &ldquo;spin locks&rdquo; &mdash; the waiting task will spin. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>20/5_9.5.1</MarginText>
<MarginInfo items={["AI12-0440-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 2   The body of a protected unit can contain declarations and bodies for local subprograms. These are not visible outside the protected unit.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>21_9.5.1</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 3   The body of a protected function can contain internal calls on other protected functions, but not protected procedures, because the current instance is a constant. On the other hand, the body of a protected procedure can contain internal calls on both protected functions and procedures.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>22_9.5.1</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 4   From within a protected action, an internal call on a protected subprogram, or an external call on a protected subprogram with a different target object is not considered a potentially blocking operation. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>22.a_9.5.1</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>This is because a task is not considered blocked while attempting to acquire the execution resource associated with a protected object. The acquisition of such a resource is rather considered part of the normal competition for execution resources between the various tasks that are ready. External calls that turn out to be on the same target object are considered potentially blocking, since they can deadlock the task indefinitely. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>23/5_9.5.1</MarginText>
<MarginInfo items={["AI95-00305-01", "AI12-0064-2", "AI12-0440-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 5   The aspect Nonblocking can be specified True on the definition of a protected unit in order to reject most attempts to use potentially blocking operations within the protected unit (see <a href="/docs/arm/AA-9/AA-9.5">9.5</a>). The <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code> Detect{"_"}Blocking can be used to ensure that any remaining executions of potentially blocking operations during a protected action raise Program{"_"}Error. See <a href="/docs/arm/AA-H/AA-H.5">H.5</a>. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>23.a/5_9.5.1</MarginText>
<MarginInfo items={["AI12-0064-2"]} />
<Admonition type="aarm" aarm="discussion">
<b></b>The deadlock case cannot be detected at compile-time, so pragma Detect{"_"}Blocking is needed to give it consistent behavior. <br />
</Admonition>
</AnnotatedOnly>

#### Name Resolution Rules

<MarginText>11_9.5.2</MarginText>
<p>In an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code>, the expected profile for the <i>entry{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0092">direct_name</a></code> is that of the <code><a href="/docs/arm/AA-9/AA-9.5#S0257">entry{"_"}declaration</a></code>; the expected type for an <code><a href="/docs/arm/AA-9/AA-9.5#S0259">entry_index</a></code> is that of the subtype defined by the <code><a href="/docs/arm/AA-3/AA-3.6#S0055">discrete{"_"}subtype{"_"}definition</a></code> of the corresponding <code><a href="/docs/arm/AA-9/AA-9.5#S0257">entry{"_"}declaration</a></code>.<br /></p>

<MarginText>12_9.5.2</MarginText>
<p>Within the <code><a href="/docs/arm/AA-11/AA-11.2#S0304">handled_sequence_of_statements</a></code> of an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code>, if a <code><a href="/docs/arm/AA-4/AA-4.1#S0098">selected{"_"}component</a></code> has a <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> that denotes the corresponding <code><a href="/docs/arm/AA-9/AA-9.5#S0257">entry{"_"}declaration</a></code>, then the entity denoted by the <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> is the <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept{"_"}statement</a></code>, and the <code><a href="/docs/arm/AA-4/AA-4.1#S0098">selected{"_"}component</a></code> is interpreted as an expanded name (see <a href="/docs/arm/AA-4/AA-4.1#Subclause_4.1.3">4.1.3</a>)[; the <code><a href="/docs/arm/AA-4/AA-4.1#S0099">selector_name</a></code> of the <code><a href="/docs/arm/AA-4/AA-4.1#S0098">selected{"_"}component</a></code> has to be the <code><a href="/docs/arm/AA-2/AA-2.3#S0002">identifier</a></code> for some formal parameter of the <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept{"_"}statement</a></code>]. <br /></p>

<AnnotatedOnly>
<MarginText>12.a_9.5.2</MarginText>
<Admonition type="aarm" aarm="proof">
<b></b>The only declarations that occur immediately within the declarative region of an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> are those for its formal parameters. <br />
</Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>13_9.5.2</MarginText>
<p>An <code><a href="/docs/arm/AA-9/AA-9.5#S0257">entry_declaration</a></code> in a task declaration shall not contain a specification for an access parameter (see <a href="/docs/arm/AA-3/AA-3.10">3.10</a>). <br /></p>

<AnnotatedOnly>
<MarginText>13.a_9.5.2</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>Access parameters for task entries would require a complex implementation. For example: <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>13.b_9.5.2</MarginText>
<CodeBlock language="ada">
task T is{"\n"}   entry E(Z : access Integer); -- Illegal!{"\n"}end T;{"\n"}
<MarginText>13.c_9.5.2</MarginText>
task body T is{"\n"}begin{"\n"}   declare{"\n"}      type A is access all Integer;{"\n"}      X : A;{"\n"}      Int : aliased Integer;{"\n"}      task Inner;{"\n"}      task body Inner is{"\n"}      begin{"\n"}         T.E(Int'Access);{"\n"}      end Inner;{"\n"}   begin{"\n"}      accept E(Z : access Integer) do{"\n"}         X := A(Z); -- Accessibility{"_"}Check{"\n"}      end E;{"\n"}   end;{"\n"}end T;{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>13.d_9.5.2</MarginText>
<Admonition type="aarm" aarm="note">
Implementing the Accessibility{"_"}Check inside the <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> for E is difficult, since one does not know whether the entry caller is calling from inside the immediately enclosing declare block or from outside it. This means that the lexical nesting level associated with the designated object is not sufficient to determine whether the Accessibility{"_"}Check should pass or fail.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>13.e_9.5.2</MarginText>
<Admonition type="aarm" aarm="note">

<MarginText>15.a_9.5.2</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b><code><a href="/docs/arm/AA-9/AA-9.5#S0258">Accept_statement</a></code>s are required to be immediately within the enclosing <code><a href="/docs/arm/AA-9/AA-9.1#S0248">task_body</a></code> (as opposed to being in a nested subprogram) to ensure that a nested task does not attempt to accept the entry of its enclosing task. We considered relaxing this restriction, either by making the check a run-time check, or by allowing a nested task to accept an entry of its enclosing task. However, neither change seemed to provide sufficient benefit to justify the additional implementation burden.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>15.b_9.5.2</MarginText>
<Admonition type="aarm" aarm="note">

This could be relaxed by allowing the inner one to hide the outer one from all visibility, but again the small added benefit didn't seem to justify making the change for Ada 95.<br />
</Admonition>
</AnnotatedOnly>

</Admonition>

<MarginText>24/5_9.5.2</MarginText>

<p>For the execution of an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code>, the <code><a href="/docs/arm/AA-9/AA-9.5#S0259">entry_index</a></code>, if any, is first evaluated and converted to the entry index subtype; this index value identifies which entry of the family is to be accepted. Execution of the rendezvous consists of the execution of the <code><a href="/docs/arm/AA-11/AA-11.2#S0304">handled_sequence_of_statements</a></code>, performance of any postcondition or type invariant checks associated with the entry, and any initialization or finalization associated with these checks, as described in <a href="/docs/arm/AA-6/AA-6.1#Subclause_6.1.1">6.1.1</a> and <a href="/docs/arm/AA-7/AA-7.3#Subclause_7.3.2">7.3.2</a>. <b></b>An <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> need not have a <code><a href="/docs/arm/AA-11/AA-11.2#S0304">handled_sequence_of_statements</a></code> even if the corresponding entry has parameters. Equally, it can have a <code><a href="/docs/arm/AA-11/AA-11.2#S0304">handled_sequence_of_statements</a></code> even if the corresponding entry has no parameters. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>29.b_9.5.2</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>A single entry overloads a subprogram, an enumeration literal, or another single entry if they have the same <code><a href="/docs/arm/AA-3/AA-3.1#S0022">defining_identifier</a></code>. Overloading is not allowed for entry family names. A single entry or an entry of an entry family can be renamed as a procedure as explained in <a href="/docs/arm/AA-8/AA-8.5#Subclause_8.5.4">8.5.4</a>. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>30/5_9.5.2</MarginText>
<MarginInfo items={["AI12-0440-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 4   The <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code> in the <code><a href="/docs/arm/AA-9/AA-9.5#S0262">entry_barrier</a></code> can reference anything visible except the formal parameters of the entry. This includes the entry index (if any), the components (including discriminants) of the protected object, the Count attribute of an entry of that protected object, and data global to the protected unit.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>31_9.5.2</MarginText>
<Admonition type="aarm" aarm="note">
The restriction against referencing the formal parameters within an <code><a href="/docs/arm/AA-9/AA-9.5#S0262">entry_barrier</a></code> ensures that all calls of the same entry see the same barrier value. If it is necessary to look at the parameters of an entry call before deciding whether to handle it, the <code><a href="/docs/arm/AA-9/AA-9.5#S0262">entry_barrier</a></code> can be &ldquo;<b>when</b> True&rdquo; and the caller can be requeued (on some private entry) when its parameters indicate that it cannot be handled immediately. <br />
</Admonition>
</AnnotatedOnly>

## 9.5.3  Entry Calls

<MarginText>1_9.5.3</MarginText>
<p>[An <code><a href="/docs/arm/AA-9/AA-9.5#S0264">entry_call_statement</a></code> (an <i>entry call</i>) can appear in various contexts.] A <i>simple</i> entry call is a stand-alone statement that represents an unconditional call on an entry of a target task or a protected object. [Entry calls can also appear as part of <code><a href="/docs/arm/AA-9/AA-9.7#S0269">select_statement</a></code>s (see <a href="/docs/arm/AA-9/AA-9.7">9.7</a>).] <br /></p>


#### Syntax

<MarginText>2_9.5.3</MarginText>

<CodeBlock>
<code>entry{"_"}call{"_"}statement</code><a id="S0264"></a><code> ::= </code><i>entry{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> [<code><a href="/docs/arm/AA-6/AA-6.4#S0219">actual_parameter_part</a></code>];<br />

</CodeBlock>

#### Name Resolution Rules

<MarginText>3_9.5.3</MarginText>
<p>The <i>entry{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> given in an <code><a href="/docs/arm/AA-9/AA-9.5#S0264">entry_call_statement</a></code> shall resolve to denote an entry. The rules for parameter associations are the same as for subprogram calls (see <a href="/docs/arm/AA-6/AA-6.4">6.4</a> and <a href="/docs/arm/AA-6/AA-6.4#Subclause_6.4.1">6.4.1</a>). <br /></p>


#### Static Semantics

<MarginText>4_9.5.3</MarginText>
<p>[The <i>entry{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> of an <code><a href="/docs/arm/AA-9/AA-9.5#S0264">entry_call_statement</a></code> specifies (explicitly or implicitly) the target object of the call, the entry or entry family, and the entry index, if any (see <a href="/docs/arm/AA-9/AA-9.5">9.5</a>).] <br /></p>


#### Dynamic Semantics

<MarginText>5_9.5.3</MarginText>
<p>Under certain circumstances (detailed below), an entry of a task or protected object is checked to see whether it is <i>open</i> or <i>closed</i>: <br /></p>

<MarginText>6/3_9.5.3</MarginText>
<MarginInfo items={["AI05-0264-1"]} />
<ul>
<li>An entry of a task is open if the task is blocked on an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> that corresponds to the entry (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.2">9.5.2</a>), or on a <code><a href="/docs/arm/AA-9/AA-9.7#S0270">selective_accept</a></code> (see <a href="/docs/arm/AA-9/AA-9.7#Subclause_9.7.1">9.7.1</a>) with an open <code><a href="/docs/arm/AA-9/AA-9.7#S0273">accept_alternative</a></code> that corresponds to the entry; otherwise, it is closed.<br /></li>
<MarginText>7/3_9.5.3</MarginText>
<MarginInfo items={["AI05-0264-1"]} />
<li>An entry of a protected object is open if the <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code> of the <code><a href="/docs/arm/AA-9/AA-9.5#S0262">entry_barrier</a></code> of the corresponding <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code> evaluates to True; otherwise, it is closed. If the evaluation of the <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code> propagates an exception, the exception Program{"_"}Error is propagated to all current callers of all entries of the protected object. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>7.a_9.5.3</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>An exception during barrier evaluation is considered essentially a fatal error. All current entry callers are notified with a Program{"_"}Error. In a fault-tolerant system, a protected object might provide a Reset protected procedure, or equivalent, to support attempts to restore such a "broken" protected object to a reasonable state. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>7.b_9.5.3</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>Note that the definition of when a task entry is open is based on the state of the (accepting) task, whereas the "openness" of a protected entry is defined only when it is explicitly checked, since the barrier expression needs to be evaluated. Implementation permissions are given (below) to allow implementations to evaluate the barrier expression more or less often than it is checked, but the basic semantic model presumes it is evaluated at the times when it is checked. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>8_9.5.3</MarginText>
<p>For the execution of an <code><a href="/docs/arm/AA-9/AA-9.5#S0264">entry_call_statement</a></code>, evaluation of the <code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> and of the parameter associations is as for a subprogram call (see <a href="/docs/arm/AA-6/AA-6.4">6.4</a>). The entry call is then <i>issued</i>: For a call on an entry of a protected object, a new protected action is started on the object (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.1">9.5.1</a>). The named entry is checked to see if it is open; if open, the entry call is said to be <i>selected immediately</i>, and the execution of the call proceeds as follows: <br /></p>

<MarginText>9_9.5.3</MarginText>
<ul>
<li>For a call on an open entry of a task, the accepting task becomes ready and continues the execution of the corresponding <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.2">9.5.2</a>).<br /></li>
<MarginText>10_9.5.3</MarginText>
<li>For a call on an open entry of a protected object, the corresponding <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code> is executed (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.2">9.5.2</a>) as part of the protected action. <br /></li>
</ul>
<MarginText>11_9.5.3</MarginText>
<p>If the <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> or <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code> completes other than by a requeue (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.4">9.5.4</a>), return is made to the caller (after servicing the entry queues &mdash; see below); any necessary assigning back of formal to actual parameters occurs, as for a subprogram call (see <a href="/docs/arm/AA-6/AA-6.4#Subclause_6.4.1">6.4.1</a>); such assignments take place outside of any protected action. <br /></p>

<AnnotatedOnly>
<MarginText>11.a_9.5.3</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>The return to the caller will generally not occur until the protected action completes, unless some other thread of control is given the job of completing the protected action and releasing the associated execution resource. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>12_9.5.3</MarginText>
<p>If the named entry is closed, the entry call is added to an <i>entry queue</i> (as part of the protected action, for a call on a protected entry), and the call remains queued until it is selected or cancelled; there is a separate (logical) entry queue for each entry of a given task or protected object [(including each entry of an entry family)].<br /></p>

<MarginText>13_9.5.3</MarginText>
<p>When a queued call is <i>selected</i>, it is removed from its entry queue. Selecting a queued call from a particular entry queue is called <i>servicing</i> the entry queue. An entry with queued calls can be serviced under the following circumstances: <br /></p>

<MarginText>14_9.5.3</MarginText>
<ul>
<li>When the associated task reaches a corresponding <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code>, or a <code><a href="/docs/arm/AA-9/AA-9.7#S0270">selective_accept</a></code> with a corresponding open <code><a href="/docs/arm/AA-9/AA-9.7#S0273">accept_alternative</a></code>;<br /></li>
<MarginText>15/4_9.5.3</MarginText>
<MarginInfo items={["AI12-0129-1"]} />
<li>If after performing, as part of a protected action on the associated protected object, an exclusive protected operation on the object, the entry is checked and found to be open. <br /></li>
</ul>
<MarginText>16_9.5.3</MarginText>
<p>If there is at least one call on a queue corresponding to an open entry, then one such call is selected according to the <i>entry queuing policy</i> in effect (see below), and the corresponding <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> or <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code> is executed as above for an entry call that is selected immediately.<br /></p>

<MarginText>17_9.5.3</MarginText>
<p>The entry queuing policy controls selection among queued calls both for task and protected entry queues. The default entry queuing policy is to select calls on a given entry queue in order of arrival. If calls from two or more queues are simultaneously eligible for selection, the default entry queuing policy does not specify which queue is serviced first. Other entry queuing policies can be specified by <code><a href="/docs/arm/AA-2/AA-2.8#S0019">pragma</a></code>s (see <a href="/docs/arm/AA-D/AA-D.4">D.4</a>).<br /></p>

<MarginText>18_9.5.3</MarginText>
<p>For a protected object, the above servicing of entry queues continues until there are no open entries with queued calls, at which point the protected action completes. <br /></p>

<AnnotatedOnly>
<MarginText>18.a_9.5.3</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>While servicing the entry queues of a protected object, no new calls can be added to any entry queue of the object, except due to an internal requeue (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.4">9.5.4</a>). This is because the first step of a call on a protected entry is to start a new protected action, which implies acquiring (for exclusive read-write access) the execution resource associated with the protected object, which cannot be done while another protected action is already in progress. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>19_9.5.3</MarginText>
<p>For an entry call that is added to a queue, and that is not the <code><a href="/docs/arm/AA-9/AA-9.7#S0282">triggering_statement</a></code> of an <code><a href="/docs/arm/AA-9/AA-9.7#S0280">asynchronous{"_"}select</a></code> (see <a href="/docs/arm/AA-9/AA-9.7#Subclause_9.7.4">9.7.4</a>), the calling task is blocked until the call is cancelled, or the call is selected and a corresponding <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> or <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code> completes without requeuing. In addition, the calling task is blocked during a rendezvous.<br /></p>

<AnnotatedOnly>
<MarginText>19.a_9.5.3</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>For a call on a protected entry, the caller is not blocked if the call is selected immediately, unless a requeue causes the call to be queued.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>20_9.5.3</MarginText>
<p>An attempt can be made to cancel an entry call upon an abort (see <a href="/docs/arm/AA-9/AA-9.8">9.8</a>) and as part of certain forms of <code><a href="/docs/arm/AA-9/AA-9.7#S0269">select_statement</a></code> (see <a href="/docs/arm/AA-9/AA-9.7#Subclause_9.7.2">9.7.2</a>, <a href="/docs/arm/AA-9/AA-9.7#Subclause_9.7.3">9.7.3</a>, and <a href="/docs/arm/AA-9/AA-9.7#Subclause_9.7.4">9.7.4</a>). The cancellation does not take place until a point (if any) when the call is on some entry queue, and not protected from cancellation as part of a requeue (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.4">9.5.4</a>); at such a point, the call is removed from the entry queue and the call completes due to the cancellation. The cancellation of a call on an entry of a protected object is a protected action[, and as such cannot take place while any other protected action is occurring on the protected object. Like any protected action, it includes servicing of the entry queues (in case some entry barrier depends on a Count attribute).] <br /></p>

<AnnotatedOnly>
<MarginText>20.a/2_9.5.3</MarginText>
<MarginInfo items={["AI95-00114-01"]} />
<Admonition type="aarm" aarm="implementation-note">
<b></b>In the case of an attempted cancellation due to abort, this removal might have to be performed by the calling task itself if the ceiling priority of the protected object is lower than the priority of the task initiating the abort. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>21_9.5.3</MarginText>
<p>A call on an entry of a task that has already completed its execution raises the exception Tasking{"_"}Error at the point of the call; similarly, this exception is raised at the point of the call if the called task completes its execution or becomes abnormal before accepting the call or completing the rendezvous (see <a href="/docs/arm/AA-9/AA-9.8">9.8</a>). This applies equally to a simple entry call and to an entry call as part of a <code><a href="/docs/arm/AA-9/AA-9.7#S0269">select_statement</a></code>.<br /></p>


#### Implementation Permissions

<MarginText>22/5_9.5.3</MarginText>
<MarginInfo items={["AI12-0444-1"]} />
<p>An implementation may perform the sequence of steps of a protected action using any thread of control; it can be a thread other than that of the task that started the protected action. If an <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code> completes without requeuing, then the corresponding calling task may be made ready without waiting for the entire protected action to complete. <br /></p>

<AnnotatedOnly>
<MarginText>22.a_9.5.3</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>These permissions are intended to allow flexibility for implementations on multiprocessors. On a monoprocessor, which thread of control executes the protected action is essentially invisible, since the thread is not abortable in any case, and the "current{"_"}task" function is not guaranteed to work during a protected action (see <a href="/docs/arm/AA-C/AA-C.7#Subclause_C.7.1">C.7.1</a>). <br />
</Admonition>
</AnnotatedOnly>

<MarginText>23/5_9.5.3</MarginText>
<MarginInfo items={["AI12-0129-1", "AI12-0444-1"]} />
<p>When the entry of a protected object is checked to see whether it is open, the implementation can bypass reevaluating the <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code> of the corresponding <code><a href="/docs/arm/AA-9/AA-9.5#S0262">entry_barrier</a></code> if no variable or attribute referenced by the <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code> (directly or indirectly) has been altered by the execution (or cancellation) of a call to an exclusive protected operation of the object since the <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code> was last evaluated. <br /></p>

<AnnotatedOnly>
<MarginText>23.a/4_9.5.3</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Changes to variables referenced by an entry barrier that result from actions outside of a call to an exclusive protected operation of the protected object need not be "noticed". For example, if a global variable is referenced by an entry barrier, it should not be altered (except as part of a protected action on the object) any time after the barrier is first evaluated. In other words, globals can be used to "parameterize" a protected object, but they cannot reliably be used to control it after the first use of the protected object. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>23.b_9.5.3</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>Note that even if a global variable is volatile, the implementation need only reevaluate a barrier if the global is updated during a protected action on the protected object. This ensures that an entry-open bit-vector implementation approach is possible, where the bit-vector is computed at the end of a protected action, rather than upon each entry call. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>24_9.5.3</MarginText>
<p>An implementation may evaluate the <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code>s of all <code><a href="/docs/arm/AA-9/AA-9.5#S0262">entry_barrier</a></code>s of a given protected object any time any entry of the object is checked to see if it is open. <br /></p>

<AnnotatedOnly>
<MarginText>24.a_9.5.3</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>In other words, any side effects of evaluating an entry barrier should be innocuous, since an entry barrier might be evaluated more or less often than is implied by the "official" dynamic semantics. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.b_9.5.3</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>It is anticipated that when the number of entries is known to be small, all barriers will be evaluated any time one of them needs to be, to produce an "entry-open bit-vector". The appropriate bit will be tested when the entry is called, and only if the bit is false will a check be made to see whether the bit-vector might need to be recomputed. This should allow an implementation to maximize the performance of a call on an open entry, which seems like the most important case.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.c_9.5.3</MarginText>
<Admonition type="aarm" aarm="note">
In addition to the entry-open bit-vector, an "is-valid" bit is needed per object, which indicates whether the current bit-vector setting is valid. A "depends-on-Count-attribute" bit is needed per type. The "is-valid" bit is set to false (as are all the bits of the bit-vector) when the protected object is first created, as well as any time an exception is propagated from computing the bit-vector. Is-valid would also be set false any time the Count is changed and "depends-on-Count-attribute" is true for the type, or a protected procedure or entry returns indicating it might have updated a variable referenced in some barrier.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.d_9.5.3</MarginText>
<Admonition type="aarm" aarm="note">
A single procedure can be compiled to evaluate all of the barriers, set the entry-open bit-vector accordingly, and set the is-valid bit to true. It could have a "when others" handler to set them all false, and call a routine to propagate Program{"_"}Error to all queued callers.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.e_9.5.3</MarginText>
<Admonition type="aarm" aarm="note">
For protected types where the number of entries is not known to be small, it makes more sense to evaluate a barrier only when the corresponding entry is checked to see if it is open. It isn't worth saving the state of the entry between checks, because of the space that would be required. Furthermore, the entry queues probably want to take up space only when there is actually a caller on them, so rather than an array of all entry queues, a linked list of nonempty entry queues make the most sense in this case, with the first caller on each entry queue acting as the queue header. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>25/5_9.5.3</MarginText>
<MarginInfo items={["AI12-0444-1"]} />
<p>When an attempt is made to cancel an entry call, the implementation can use a thread of control other than that of the task (or interrupt) that initiated the cancellation; in particular, it may use the thread of control of the caller itself to attempt the cancellation, even if this can allow the entry call to be selected in the interim. <br /></p>

<AnnotatedOnly>
<MarginText>25.a_9.5.3</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>Because cancellation of a protected entry call is a protected action (which helps make the Count attribute of a protected entry meaningful), it might not be practical to attempt the cancellation from the thread of control that initiated the cancellation. For example, if the cancellation is due to the expiration of a delay, it is unlikely that the handler of the timer interrupt could perform the necessary protected action itself (due to being on the interrupt level). Similarly, if the cancellation is due to an abort, it is possible that the task initiating the abort has a priority higher than the ceiling priority of the protected object (for implementations that support ceiling priorities). Similar considerations could apply in a multiprocessor situation. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>26_9.5.3</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 1   If an exception is raised during the execution of an <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code>, it is propagated to the corresponding caller (see <a href="/docs/arm/AA-11/AA-11.4">11.4</a>).<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27_9.5.3</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 2   For a call on a protected entry, the entry is checked to see if it is open prior to queuing the call, and again thereafter if its Count attribute (see <a href="/docs/arm/AA-9/AA-9.9">9.9</a>) is referenced in some entry barrier. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.a_9.5.3</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Given this, extra care is required if a reference to the Count attribute of an entry appears in the entry's own barrier. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>27.b_9.5.3</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>An entry is checked to see if it is open prior to queuing to maximize the performance of a call on an open entry. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>28_9.5.3</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 3   In addition to simple entry calls, the language permits timed, conditional, and asynchronous entry calls (see <a href="/docs/arm/AA-9/AA-9.7#Subclause_9.7.2">9.7.2</a>, <a href="/docs/arm/AA-9/AA-9.7#Subclause_9.7.3">9.7.3</a>, and see <a href="/docs/arm/AA-9/AA-9.7#Subclause_9.7.4">9.7.4</a>). <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>28.a_9.5.3</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>A task can call its own entries, but the task will deadlock if the call is a simple entry call. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>29/5_9.5.3</MarginText>
<MarginInfo items={["AI12-0442-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 4   The <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code> of an <code><a href="/docs/arm/AA-9/AA-9.5#S0262">entry_barrier</a></code> is allowed to be evaluated by an implementation more often than strictly necessary, even if the evaluation can have side effects. On the other hand, an implementation can avoid reevaluating the <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code> if nothing it references was updated by an intervening protected action on the protected object, even if the <code><a href="/docs/arm/AA-4/AA-4.5#S0150">condition</a></code> references some global variable that is updated by an action performed from outside of a protected action. <br />
</Admonition>
</AnnotatedOnly>

## 9.5.4  Requeue Statements

<MarginText>1_9.5.4</MarginText>
<p>[A <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code> can be used to complete an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> or <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code>, while redirecting the corresponding entry call to a new (or the same) entry queue. Such a <i>requeue</i> can be performed with or without allowing an intermediate cancellation of the call, due to an abort or the expiration of a delay. ]<br /></p>


#### Syntax

<MarginText>2/3_9.5.4</MarginText>
<MarginInfo items={["AI05-0030-2"]} />

<CodeBlock>
<code>requeue{"_"}statement</code><a id="S0265"></a><code> ::= </code><b>requeue</b> <i>procedure{"_"}or{"_"}entry{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> [<b>with</b> <b>abort</b>];<br />

</CodeBlock>

#### Name Resolution Rules

<MarginText>3/3_9.5.4</MarginText>
<MarginInfo items={["AI05-0030-2", "AI05-0215-1"]} />
<p>The <i>procedure{"_"}or{"_"}entry{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> of a <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code> shall resolve to denote a procedure or an entry (the <i>requeue </i><i>target</i>). The profile of the entry, or the profile or prefixed profile of the procedure, shall either have no parameters, or be type conformant (see <a href="/docs/arm/AA-6/AA-6.3#Subclause_6.3.1">6.3.1</a>) with the profile of the innermost enclosing <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry{"_"}body</a></code> or <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept{"_"}statement</a></code>. <br /></p>


#### Legality Rules

<MarginText>4_9.5.4</MarginText>
<p>A <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code> shall be within a callable construct that is either an <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code> or an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code>, and this construct shall be the innermost enclosing body or callable construct.<br /></p>

<MarginText>5/3_9.5.4</MarginText>
<MarginInfo items={["AI05-0030-2", "AI05-0215-1"]} />
<p>If the requeue target has parameters, then its (prefixed) profile shall be subtype conformant with the profile of the innermost enclosing callable construct. <br /></p>

<MarginText>5.1/4_9.5.4</MarginText>
<MarginInfo items={["AI12-0090-1"]} />
<p>Given a requeue{"_"}statement where the innermost enclosing callable construct is for an entry <i>E1</i>, for every [specific or class-wide ]postcondition expression <i>P1</i> that applies to <i>E1</i>, there shall exist a postcondition expression <i>P2</i> that applies to the requeue target <i>E2</i> such that <br /></p>

<MarginText>5.2/4_9.5.4</MarginText>
<ul>
<li><i>P1</i> is fully conformant with the expression produced by replacing each reference in <i>P2</i> to a formal parameter of <i>E2</i> with a reference to the corresponding formal paramter of <i>E1</i>; and<br /></li>
<MarginText>5.3/4_9.5.4</MarginText>
<li>if <i>P1</i> is enabled, then <i>P2</i> is also enabled. <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>5.a/4_9.5.4</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>Roughly speaking, the postcondition of the requeue target is required to imply that of the enclosing callable construct. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>5.4/5_9.5.4</MarginText>
<MarginInfo items={["AI12-0090-1", "AI12-0143-1"]} />
<p>The requeue target shall not have an applicable specific or class-wide postcondition that includes an Old or Index <code><a href="/docs/arm/AA-4/AA-4.1#S0100">attribute_reference</a></code>.<br /></p>

<MarginText>5.5/4_9.5.4</MarginText>
<MarginInfo items={["AI12-0090-1"]} />
<p>If the requeue target is declared immediately within the <code><a href="/docs/arm/AA-9/AA-9.1#S0246">task_definition</a></code> of a named task type or the <code><a href="/docs/arm/AA-9/AA-9.4#S0251">protected_definition</a></code> of a named protected type, and if the requeue statement occurs within the body of that type, and if the requeue is an external requeue, then the requeue target shall not have a specific or class-wide postcondition which includes a name denoting either the current instance of that type or any entity declared within the declaration of that type.<br /></p>

<AnnotatedOnly>
<MarginText>5.b/4_9.5.4</MarginText>
<Admonition type="aarm" aarm="discussion">
<b></b>The above pair of rules always apply; they don't depend on whether or not any of the postconditions are enabled. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>5.6/4_9.5.4</MarginText>
<MarginInfo items={["AI05-0030-2", "AI05-0215-1", "AI12-0090-1"]} />
<p>If the target is a procedure, the name shall denote a renaming of an entry, or shall denote a view or a prefixed view of a primitive subprogram of a synchronized interface, where the first parameter of the unprefixed view of the primitive subprogram shall be a controlling parameter, and the Synchronization aspect shall be specified with <code><a href="/docs/arm/AA-9/AA-9.5#S0256">synchronization_kind</a></code> By{"_"}Entry for the primitive subprogram.<br /></p>

<MarginText>6/3_9.5.4</MarginText>
<MarginInfo items={["AI05-0030-2"]} />
<p>In a <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code> of an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> of some task unit, either the target object shall be a part of a formal parameter of the <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code>, or the accessibility level of the target object shall not be equal to or statically deeper than any enclosing <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> of the task unit. In a <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue{"_"}statement</a></code> of an <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry{"_"}body</a></code> of some protected unit, either the target object shall be a part of a formal parameter of the <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry{"_"}body</a></code>, or the accessibility level of the target object shall not be statically deeper than that of the <code><a href="/docs/arm/AA-9/AA-9.5#S0257">entry_declaration</a></code> for the <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code>.<br /></p>

<AnnotatedOnly>
<MarginText>6.a_9.5.4</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>In the <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code> case, the intent is that the target object can be global, or can be a component of the protected unit, but cannot be a local variable of the <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code>. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.b_9.5.4</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>These restrictions ensure that the target object of the requeue outlives the completion and finalization of the enclosing callable construct. They also prevent requeuing from a nested <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> on a parameter of an outer <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code>, which could create some strange "long-distance" connections between an entry caller and its server.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.c_9.5.4</MarginText>
<Admonition type="aarm" aarm="note">
Note that in the strange case where a <code><a href="/docs/arm/AA-9/AA-9.1#S0248">task_body</a></code> is nested inside an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code>, it is permissible to requeue from an <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> of the inner <code><a href="/docs/arm/AA-9/AA-9.1#S0248">task_body</a></code> on parameters of the outer <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code>. This is not a problem because all calls on the inner task have to complete before returning from the outer <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code>, meaning no "dangling calls" will be created. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>6.d_9.5.4</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<b></b>By disallowing certain requeues, we ensure that the normal <code><a href="/docs/arm/AA-9/AA-9.7#S0275">terminate_alternative</a></code> rules remain sensible, and that explicit clearing of the entry queues of a protected object during finalization is rarely necessary. In particular, such clearing of the entry queues is necessary only (ignoring premature Unchecked{"_"}Deallocation) for protected objects declared in a <code><a href="/docs/arm/AA-9/AA-9.1#S0248">task_body</a></code> (or created by an allocator for an access type declared in such a body) containing one or more <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code>s. Protected objects declared in subprograms, or at the library level, will never need to have their entry queues explicitly cleared during finalization. <br />
</Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>7/5_9.5.4</MarginText>
<MarginInfo items={["AI05-0030-2", "AI12-0090-1", "AI12-0335-1"]} />
<p>The execution of a <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code> begins with the following sequence of steps:<br /></p>

<MarginText>7.1/5_9.5.4</MarginText>
<dt><br/>1.</dt>
<dl>
<dd>The <i>procedure{"_"}or{"_"}entry{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> is evaluated. This includes evaluation of the <code><a href="/docs/arm/AA-4/AA-4.1#S0093">prefix</a></code> (if any) identifying the target task or protected object and of the <code><a href="/docs/arm/AA-4/AA-4.4#S0132">expression</a></code> (if any) identifying the entry within an entry family.<br /></dd>
<MarginText>7.2/5_9.5.4</MarginText>
<dt><br/>2.</dt>
<MarginInfo items={["AI12-0335-1"]} />
<dd>If the target object is not a part of a formal parameter of the innermost enclosing callable construct, a check is made that the accessibility level of the target object is not equal to or deeper than the level of the innermost enclosing callable construct. If this check fails, Program{"_"}Error is raised.<br /></dd>
<MarginText>7.3/5_9.5.4</MarginText>
<dt><br/>3.</dt>
<MarginInfo items={["AI12-0090-1"]} />
<dd>Precondition checks are performed as for a call to the requeue target.<br /></dd>
<MarginText>7.4/5_9.5.4</MarginText>
<dt><br/>4.</dt>
<dd>The <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code> or <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> enclosing the <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code> is then completed[, finalized, and left (see <a href="/docs/arm/AA-7/AA-7.6#Subclause_7.6.1">7.6.1</a>)].<br /></dd>
</dl>
<MarginText>8_9.5.4</MarginText>
<p>For the execution of a requeue on an entry of a target task, after leaving the enclosing callable construct, the named entry is checked to see if it is open and the requeued call is either selected immediately or queued, as for a normal entry call (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.3">9.5.3</a>).<br /></p>

<MarginText>9_9.5.4</MarginText>
<p>For the execution of a requeue on an entry of a target protected object, after leaving the enclosing callable construct: <br /></p>

<MarginText>10_9.5.4</MarginText>
<ul>
<li>if the requeue is an internal requeue (that is, the requeue is back on an entry of the same protected object &mdash; see <a href="/docs/arm/AA-9/AA-9.5">9.5</a>), the call is added to the queue of the named entry and the ongoing protected action continues (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.1">9.5.1</a>); <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>10.a_9.5.4</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>Note that for an internal requeue, the call is queued without checking whether the target entry is open. This is because the entry queues will be serviced before the current protected action completes anyway, and considering the requeued call immediately might allow it to "jump" ahead of existing callers on the same queue. <br />
</Admonition>
</AnnotatedOnly>

<MarginText>11_9.5.4</MarginText>
<ul>
<li>if the requeue is an external requeue (that is, the target protected object is not implicitly the same as the current object &mdash; see <a href="/docs/arm/AA-9/AA-9.5">9.5</a>), a protected action is started on the target object and proceeds as for a normal entry call (see <a href="/docs/arm/AA-9/AA-9.5#Subclause_9.5.3">9.5.3</a>). <br /></li>
</ul>
<MarginText>12/4_9.5.4</MarginText>
<MarginInfo items={["AI05-0030-2", "AI05-0090-1"]} />
<p>If the requeue target named in the <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code> has formal parameters, then during the execution of the <code><a href="/docs/arm/AA-9/AA-9.5#S0258">accept_statement</a></code> or <code><a href="/docs/arm/AA-9/AA-9.5#S0260">entry_body</a></code> corresponding to the new entry and during the checking of any preconditions of the new entry, the formal parameters denote the same objects as did the corresponding formal parameters of the callable construct completed by the requeue. [In any case, no parameters are specified in a <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code>; any parameter passing is implicit.]<br /></p>

<MarginText>13_9.5.4</MarginText>
<p>If the <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code> includes the reserved words <b>with abort</b> (it is a <i>requeue-with-abort</i>), then: <br /></p>

<MarginText>14_9.5.4</MarginText>
<ul>
<li>if the original entry call has been aborted (see <a href="/docs/arm/AA-9/AA-9.8">9.8</a>), then the requeue acts as an abort completion point for the call, and the call is cancelled and no requeue is performed;<br /></li>
<MarginText>15_9.5.4</MarginText>
<li>if the original entry call was timed (or conditional), then the original expiration time is the expiration time for the requeued call. <br /></li>
</ul>
<MarginText>16_9.5.4</MarginText>
<p>If the reserved words <b>with abort</b> do not appear, then the call remains protected against cancellation while queued as the result of the <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>16.a_9.5.4</MarginText>
<Admonition type="aarm" aarm="ramification">
<b></b>This protection against cancellation lasts only until the call completes or a subsequent requeue-with-abort is performed on the call. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>16.b_9.5.4</MarginText>
<Admonition type="aarm" aarm="reason">
<b></b>We chose to protect a requeue, by default, against abort or cancellation. This seemed safer, since it is likely that extra steps need to be taken to allow for possible cancellation once the servicing of an entry call has begun. This also means that in the absence of <b>with abort</b> the usual Ada 83 behavior is preserved, namely that once an entry call is accepted, it cannot be cancelled until it completes. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>17/5_9.5.4</MarginText>
<MarginInfo items={["AI12-0300-1"]} />
<Admonition type="aarm" aarm="note">
NOTE   A requeue is permitted from a single entry to an entry of an entry family, or vice versa. The entry index, if any, plays no part in the subtype conformance check between the profiles of the two entries; an entry index is part of the <i>entry{"_"}</i><code><a href="/docs/arm/AA-4/AA-4.1#S0091">name</a></code> for an entry of a family. <br />
</Admonition>
</AnnotatedOnly>

#### Inconsistencies With Ada 2012

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>20.c/4_9.5.4</MarginText>
<MarginInfo items={["AI12-0090-1"]} />
<Admonition type="aarm" aarm="note">
<b>Corrigendum:</b> We now define that any preconditions of the requeue target are evaluated as part of a <code><a href="/docs/arm/AA-9/AA-9.5#S0265">requeue_statement</a></code>. Original Ada 2012 did not specify this, so a program that requeues when the preconditions fail will raise an exception when none would happen in original Ada 2012. We don't expect this to be a problem, as in that case, the entry body would be called with some of its preconditions evaluating as False; the body is likely to assume that they are true and probably will have failed in some other way anyway.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>20.d/5_9.5.4</MarginText>
<MarginInfo items={["AI12-0335-1"]} />
<Admonition type="aarm" aarm="correction">
<b></b> We now include an accessibility check on requeues. This means Program{"_"}Error could be raised for a requeue that worked in Ada 2012. This can only fail for an object for which the statically deeper relationship does not apply, for instance a stand-alone object of an anonymous access type. Most programs that are affected are erroneous anyway (as they will eventually use a nonexistent object), so we do not believe this will matter in practice. <br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>

#### Incompatibilities With Ada 2012

</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>20.e/4_9.5.4</MarginText>
<MarginInfo items={["AI12-0090-1"]} />
<Admonition type="aarm" aarm="note">
<b>Corrigendum:</b> If a requeue target has a different postcondition than the original entry, the requeue is now illegal. In such a case, the original postcondition would never have been evaluated, and assumptions that the caller relied upon might not be true. A requeue should be invisible to the caller with respect to any postconditions; thus we only allow it when the original entry has no postconditions or the requeue target has (at least) the same postconditions. <br />
</Admonition>
</AnnotatedOnly>
