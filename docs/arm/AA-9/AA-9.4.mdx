<MarginText>1</MarginText>
<p>A <em>protected object</em> provides coordinated access to shared data, through calls on its visible <em>protected operations</em>, which can be <em>protected subprograms</em> or <em>protected entries</em>. A <em>protected unit</em> is declared by a <em>protected declaration</em>, which has a corresponding <code><a href="../AA-9/AA-9.4#S0254">protected_body</a></code>. A protected declaration may be a <code><a href="../AA-9/AA-9.4#S0249">protected_type_declaration</a></code>, in which case it declares a named protected type; alternatively, it may be a <code><a href="../AA-9/AA-9.4#S0250">single_protected_declaration</a></code>, in which case it defines an anonymous protected type, as well as declaring a named protected object of that type. <br /></p>

#### Static Semantics

<MarginText>11/2</MarginText>
<MarginInfo items={["AI95-00345-01", "AI95-00401-01"]} />
<p>A <code><a href="../AA-9/AA-9.4#S0251">protected_definition</a></code> defines a protected type and its first subtype. The list of <code><a href="../AA-9/AA-9.4#S0252">protected_operation_declaration</a></code>s of a <code><a href="../AA-9/AA-9.4#S0251">protected_definition</a></code>, together with the <code><a href="../AA-3/AA-3.7#S0061">known_discriminant_part</a></code>, if any, is called the visible part of the protected unit. [ The optional list of <code><a href="../AA-9/AA-9.4#S0253">protected_element_declaration</a></code>s after the reserved word <strong>private</strong> is called the private part of the protected unit.] <br /></p>

<AnnotatedOnly>
<MarginText>11.a</MarginText>
<Admonition type="aarm" aarm="proof">
<strong></strong>Private part is defined in Clause <a href="../AA-8/">8</a>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>11.1/3</MarginText>
<MarginInfo items={["AI95-00345-01", "AI95-00397-01", "AI95-00399-01", "AI95-00419-01", "AI05-0042-1"]} />
<p>For a protected declaration with an <code><a href="../AA-3/AA-3.9#S0078">interface_list</a></code>, the protected type inherits user-defined primitive subprograms from each progenitor type (see <a href="../AA-3/AA-3.9#Subclause_3.9.4">3.9.4</a>), in the same way that a derived type inherits user-defined primitive subprograms from its progenitor types (see <a href="../AA-3/AA-3.4">3.4</a>). If the first parameter of a primitive inherited subprogram is of the protected type or an access parameter designating the protected type, and there is a <code><a href="../AA-9/AA-9.4#S0252">protected_operation_declaration</a></code> for a protected subprogram or single entry with the same identifier within the protected declaration, whose profile is type conformant with the prefixed view profile of the inherited subprogram, the inherited subprogram is said to be <em>implemented</em> by the conforming protected subprogram or entry using an implicitly declared nonabstract subprogram which has the same profile as the inherited subprogram and which overrides it. <br /></p>

<AnnotatedOnly>
<MarginText>11.b/2</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The inherited subprograms can only come from an interface given as part of the protected declaration. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>11.b.1/3</MarginText>
<MarginInfo items={["AI05-0042-1"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>The part about the implicitly declared subprogram is needed so that a subprogram implemented by an entry or subprogram is considered to be overridden for the purpose of the other rules of the language. Without it, it would for instance be illegal for an abstract subprogram to be implemented by an entry, because the abstract subprogram would not be overridden. The Legality Rules below ensure that there is no conflict between the implicit overriding subprogram and a user-defined overriding subprogram. <br /></Admonition>
</AnnotatedOnly>


#### Legality Rules

<MarginText>11.2/2</MarginText>
<MarginInfo items={["AI95-00345-01"]} />
<p>A protected declaration requires a completion[, which shall be a <code><a href="../AA-9/AA-9.4#S0254">protected_body</a></code>,] and every <code><a href="../AA-9/AA-9.4#S0254">protected_body</a></code> shall be the completion of some protected declaration. <br /></p>

<AnnotatedOnly>
<MarginText>11.c/3</MarginText>
<MarginInfo items={["AI05-0229-1"]} />
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>If the implementation supports it, the protected body can be imported (using aspect Import, see <a href="../AA-B/AA-B.1">B.1</a>), in which case no explicit <code><a href="../AA-9/AA-9.4#S0254">protected_body</a></code> is allowed. <br /></Admonition>
</AnnotatedOnly>

<MarginText>11.3/2</MarginText>
<MarginInfo items={["AI95-00345-01", "AI95-00399-01"]} />
<p>[Each <em>interface_</em><code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code> of an <code><a href="../AA-3/AA-3.9#S0078">interface_list</a></code> appearing within a protected declaration shall denote a limited interface type that is not a task interface.] <br /></p>

<AnnotatedOnly>
<MarginText>11.d/2</MarginText>
<Admonition type="aarm" aarm="proof">
<strong></strong><a href="../AA-3/AA-3.9#Subclause_3.9.4">3.9.4</a> requires that an <code><a href="../AA-3/AA-3.9#S0078">interface_list</a></code> only name interface types, and limits the descendants of the various kinds of interface types. Only a limited, protected, or synchronized interface can have a protected type descendant. Nonlimited or task interfaces are not allowed, as they offer operations that a protected type does not have. <br /></Admonition>
</AnnotatedOnly>

<MarginText>11.4/3</MarginText>
<MarginInfo items={["AI95-00397-01", "AI05-0042-1"]} />
<p>The prefixed view profile of an explicitly declared primitive subprogram of a tagged protected type shall not be type conformant with any protected operation of the protected type, if the subprogram has the same defining name as the protected operation and the first parameter of the subprogram is of the protected type or is an access parameter designating the protected type. <br /></p>

<AnnotatedOnly>
<MarginText>11.e/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This prevents the existence of two operations with the same name and profile which could be called with a prefixed view. If the operation was inherited, this would be illegal by the following rules; this rule puts inherited and noninherited routines on the same footing. Note that this only applies to tagged protected types (that is, those with an interface in their declaration); we do that as there is no problem with prefixed view calls of primitive operations for &ldquo;normal&rdquo; protected types, and having this rule apply to all protected types would be incompatible with Ada 95. <br /></Admonition>
</AnnotatedOnly>

<MarginText>11.5/2</MarginText>
<MarginInfo items={["AI95-00345-01", "AI95-00399-01"]} />
<p>For each primitive subprogram inherited by the type declared by a protected declaration, at most one of the following shall apply:<br /></p>

<MarginText>11.6/2</MarginText>
<MarginInfo items={["AI95-00345-01"]} />
<ul>
<li>the inherited subprogram is overridden with a primitive subprogram of the protected type, in which case the overriding subprogram shall be subtype conformant with the inherited subprogram and not abstract; or<br /></li>
<MarginText>11.7/2</MarginText>
<MarginInfo items={["AI95-00345-01", "AI95-00397-01"]} />
<li>the inherited subprogram is implemented by a protected subprogram or single entry of the protected type, in which case its prefixed view profile shall be subtype conformant with that of the protected subprogram or entry. <br /></li>
</ul>
<MarginText>11.8/2</MarginText>
<p>If neither applies, the inherited subprogram shall be a null procedure. In addition to the places where Legality Rules normally apply (see <a href="../AA-12/AA-12.3">12.3</a>), these rules also apply in the private part of an instance of a generic unit. <br /></p>

<AnnotatedOnly>
<MarginText>11.f/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>Each inherited subprogram can only have a single implementation (either from overriding a subprogram, implementing a subprogram, or implementing an entry), and must have an implementation unless the subprogram is a null procedure. <br /></Admonition>
</AnnotatedOnly>

<MarginText>11.9/3</MarginText>
<MarginInfo items={["AI95-00345-01", "AI05-0291-1"]} />
<p>If an inherited subprogram is implemented by a protected procedure or an entry, then the first parameter of the inherited subprogram shall be of mode <strong>out</strong> or <strong>in out</strong>, or an access-to-variable parameter. If an inherited subprogram is implemented by a protected function, then the first parameter of the inherited subprogram shall be of mode <strong>in</strong>, but not an access-to-variable parameter. <br /></p>

<AnnotatedOnly>
<MarginText>11.g/3</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>For a protected procedure or entry, the protected object can be read or written (see <a href="../AA-9/AA-9.5#Subclause_9.5.1">9.5.1</a>). A subprogram that is implemented by a protected procedure or entry must have a profile which reflects that in order to avoid confusion. Similarly, a protected function has a parameter that is a constant, and the inherited routine should reflect that. <br /></Admonition>
</AnnotatedOnly>

<MarginText>11.10/2</MarginText>
<MarginInfo items={["AI95-00397-01"]} />
<p>If a protected subprogram declaration has an <code><a href="../AA-8/AA-8.3#S0234">overriding_indicator</a></code>, then at the point of the declaration:<br /></p>

<MarginText>11.11/2</MarginText>
<ul>
<li>if the <code><a href="../AA-8/AA-8.3#S0234">overriding_indicator</a></code> is <strong>overriding</strong>, then the subprogram shall implement an inherited subprogram;<br /></li>
<MarginText>11.12/2</MarginText>
<li>if the <code><a href="../AA-8/AA-8.3#S0234">overriding_indicator</a></code> is <strong>not overriding</strong>, then the subprogram shall not implement any inherited subprogram.<br /></li>
</ul>
<MarginText>11.13/2</MarginText>
<p>In addition to the places where Legality Rules normally apply (see <a href="../AA-12/AA-12.3">12.3</a>), these rules also apply in the private part of an instance of a generic unit.<br /></p>

<AnnotatedOnly>
<MarginText>11.h/2</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>These rules are subtly different than those for subprograms (see <a href="../AA-8/AA-8.3#Subclause_8.3.1">8.3.1</a>) because there cannot be &ldquo;late&rdquo; inheritance of primitives from interfaces. Hidden (that is, private) interfaces are prohibited explicitly (see <a href="../AA-7/AA-7.3">7.3</a>), as are hidden primitive operations (as private operations of public abstract types are prohibited &mdash; see <a href="../AA-3/AA-3.9#Subclause_3.9.3">3.9.3</a>). <br /></Admonition>
</AnnotatedOnly>


#### Dynamic Semantics

<MarginText>12</MarginText>
<p>[The elaboration of a protected declaration elaborates the <code><a href="../AA-9/AA-9.4#S0251">protected_definition</a></code>. The elaboration of a <code><a href="../AA-9/AA-9.4#S0250">single_protected_declaration</a></code> also creates an object of an (anonymous) protected type.] <br /></p>

<AnnotatedOnly>
<MarginText>12.a</MarginText>
<Admonition type="aarm" aarm="proof">
<strong></strong>This is redundant with the general rules for the elaboration of a <code><a href="../AA-3/AA-3.2#S0024">full_type_declaration</a></code> and an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code>. <br /></Admonition>
</AnnotatedOnly>

<MarginText>13</MarginText>
<p>[The elaboration of a <code><a href="../AA-9/AA-9.4#S0251">protected_definition</a></code> creates the protected type and its first subtype;] it also includes the elaboration of the <code><a href="../AA-3/AA-3.8#S0070">component_declaration</a></code>s and <code><a href="../AA-9/AA-9.4#S0252">protected_operation_declaration</a></code>s in the given order.<br /></p>

<MarginText>14</MarginText>
<p>[As part of the initialization of a protected object, any per-object constraints (see <a href="../AA-3/AA-3.8">3.8</a>) are elaborated.] <br /></p>

<AnnotatedOnly>
<MarginText>14.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>We do not mention pragmas since each pragma has its own elaboration rules. <br /></Admonition>
</AnnotatedOnly>

<MarginText>15</MarginText>
<p>The elaboration of a <code><a href="../AA-9/AA-9.4#S0254">protected_body</a></code> has no other effect than to establish that protected operations of the type can from then on be called without failing the Elaboration_Check.<br /></p>

<MarginText>16</MarginText>
<p>The content of an object of a given protected type includes: <br /></p>

<MarginText>17</MarginText>
<ul>
<li>The values of the components of the protected object, including (implicitly) an entry queue for each entry declared for the protected object; <br /></li>
</ul>
<AnnotatedOnly>
<MarginText>17.a</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>"For each entry" implies one queue for each single entry, plus one for each entry of each entry family. <br /></Admonition>
</AnnotatedOnly>

<MarginText>18</MarginText>
<ul>
<li>A representation of the state of the execution resource <em>associated</em> with the protected object (one such resource is associated with each protected object). <br /></li>
</ul>
<MarginText>19</MarginText>
<p>[The execution resource associated with a protected object has to be acquired to read or update any components of the protected object; it can be acquired (as part of a protected action &mdash; see <a href="../AA-9/AA-9.5#Subclause_9.5.1">9.5.1</a>) either for concurrent read-only access, or for exclusive read-write access.]<br /></p>

<MarginText>20</MarginText>
<p>As the first step of the <em>finalization</em> of a protected object, each call remaining on any entry queue of the object is removed from its queue and Program_Error is raised at the place of the corresponding <code><a href="../AA-9/AA-9.5#S0264">entry_call_statement</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>20.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This is analogous to the raising of Tasking_Error in callers of a task that completes before accepting the calls. This situation can only occur due to a requeue (ignoring premature unchecked_deallocation), since any task that has accessibility to a protected object is awaited before finalizing the protected object. For example: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>20.b</MarginText>
<CodeBlock language="ada">
procedure Main is{"\n"}    task T is{"\n"}        entry E;{"\n"}    end T;{"\n"}
<MarginText>20.c</MarginText>
    task body T is{"\n"}        protected PO is{"\n"}            entry Ee;{"\n"}        end PO;{"\n"}
<MarginText>20.d</MarginText>
        protected body PO is{"\n"}            entry Ee when False is{"\n"}            begin{"\n"}                null;{"\n"}            end Ee;{"\n"}        end PO;{"\n"}    begin{"\n"}        accept E do{"\n"}            requeue PO.Ee;{"\n"}        end E;{"\n"}    end T;{"\n"}begin{"\n"}    T.E;{"\n"}end Main;{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>20.e/3</MarginText>
<MarginInfo items={["AI05-0005-1"]} />
<Admonition type="aarm" aarm="note">
The environment task is queued on PO.Ee when PO is finalized.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>20.f</MarginText>
<Admonition type="aarm" aarm="note">
In a real example, a server task might park callers on a local protected object for some useful purpose, so we didn't want to disallow this case. <br /></Admonition>
</AnnotatedOnly>


#### Bounded (Run-Time) Errors

<MarginText>20.1/2</MarginText>
<MarginInfo items={["AI95-00280-01"]} />
<p>It is a bounded error to call an entry or subprogram of a protected object after that object is finalized. If the error is detected, Program_Error is raised. Otherwise, the call proceeds normally, which may leave a task queued forever. <br /></p>

<AnnotatedOnly>
<MarginText>20.g/2</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This is very similar to the finalization rule. It is a bounded error so that an implementation can avoid the overhead of the check if it can ensure that the call still will operate properly. Such an implementation cannot need to return resources (such as locks) to an executive that it needs to execute calls.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>20.h/2</MarginText>
<Admonition type="aarm" aarm="note">
This case can happen (and has happened in production code) when a protected object is accessed from the Finalize routine of a type. For example: <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>20.i/2</MarginText>
<CodeBlock language="ada">
with Ada.Finalization.Controlled;{"\n"}package Window_Manager is{"\n"}    ...{"\n"}    type Root_Window is new Ada.Finalization.Controlled with private;{"\n"}    type Any_Window is access all Root_Window;{"\n"}    ...{"\n"}private{"\n"}    ...{"\n"}    procedure Finalize (Object : in out Root_Window);{"\n"}    ...{"\n"}end Window_Manager;{"\n"}
<MarginText>20.j/2</MarginText>
package body Window_Manager is{"\n"}   protected type Lock is{"\n"}       entry Get_Lock;{"\n"}       procedure Free_Lock;{"\n"}   ...{"\n"}   end Lock;{"\n"}
<MarginText>20.k/2</MarginText>
   Window_Lock : Lock;{"\n"}
<MarginText>20.l/2</MarginText>
   procedure Finalize (Object : in out Root_Window) is{"\n"}   begin{"\n"}       Window_Lock.Get_Lock;{"\n"}       ...{"\n"}       Window_Lock.Free_Lock;{"\n"}   end Finalize;{"\n"}   ...{"\n"}   A_Window : Any_Window := new Root_Window;{"\n"}end Window_Manager;{"\n"}

</CodeBlock>
</AnnotatedOnly>
<AnnotatedOnly>
<MarginText>20.m/2</MarginText>
<Admonition type="aarm" aarm="note">
The environment task will call Window_Lock for the object allocated for A_Window when the collection for Any_Window is finalized, which will happen after the finalization of Window_Lock (because finalization of the package body will occur before that of the package specification). <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>21/2</MarginText>
<MarginInfo items={["AI95-00382-01"]} />
<Admonition type="aarm" aarm="note">
NOTE 1   Within the declaration or body of a protected unit other than in an <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>, the name of the protected unit denotes the current instance of the unit (see <a href="../AA-8/AA-8.6">8.6</a>), rather than the first subtype of the corresponding protected type (and thus the name cannot be used as a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>). <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>21.a/2</MarginText>
<MarginInfo items={["AI95-00382-01"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>It can be used as a <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code> in an anonymous access type. In addition, it is possible to refer to some other subtype of the protected type within its body, presuming such a subtype has been declared between the <code><a href="../AA-9/AA-9.4#S0249">protected_type_declaration</a></code> and the <code><a href="../AA-9/AA-9.4#S0254">protected_body</a></code>. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>22</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 2   A <code><a href="../AA-4/AA-4.1#S0098">selected_component</a></code> can be used to denote a discriminant of a protected object (see <a href="../AA-4/AA-4.1#Subclause_4.1.3">4.1.3</a>). Within a protected unit, the name of a discriminant of the protected type denotes the corresponding discriminant of the current instance of the unit.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>23/2</MarginText>
<MarginInfo items={["AI95-00287-01"]} />
<Admonition type="aarm" aarm="note">
NOTE 3   A protected type is a limited type (see <a href="../AA-7/AA-7.5">7.5</a>), and hence precludes use of <code><a href="../AA-5/AA-5.2#S0173">assignment_statement</a></code>s and predefined equality operators.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 4   The bodies of the protected operations given in the <code><a href="../AA-9/AA-9.4#S0254">protected_body</a></code> define the actions that take place upon calls to the protected operations.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>25</MarginText>
<Admonition type="aarm" aarm="note">
NOTE 5   The declarations in the private part are only visible within the private part and the body of the protected unit. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>25.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong><code><a href="../AA-3/AA-3.8#S0070">Component_declaration</a></code>s are disallowed in a <code><a href="../AA-9/AA-9.4#S0254">protected_body</a></code> because, for efficiency, we wish to allow the compiler to determine the size of protected objects (when not dynamic); the compiler cannot necessarily see the body. Furthermore, the semantics of initialization of such objects would be problematic &mdash; we do not wish to give protected objects complex initialization semantics similar to task activation.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>25.b</MarginText>
<Admonition type="aarm" aarm="note">
The same applies to <code><a href="../AA-9/AA-9.5#S0257">entry_declaration</a></code>s, since an entry involves an implicit component &mdash; the entry queue. <br /></Admonition>
</AnnotatedOnly>


#### Examples

<MarginText>26</MarginText>
<p><em>Example of declaration of protected type and corresponding body:</em> <br /></p>

<MarginText>27</MarginText>
<CodeBlock language="ada">
protected type Resource is{"\n"}   entry Seize;{"\n"}   procedure Release;{"\n"}private{"\n"}   Busy : Boolean := False;{"\n"}end Resource;{"\n"}
<MarginText>28</MarginText>
protected body Resource is{"\n"}   entry Seize when not Busy is{"\n"}   begin{"\n"}      Busy := True;{"\n"}   end Seize;{"\n"}
<MarginText>29</MarginText>
   procedure Release is{"\n"}   begin{"\n"}      Busy := False;{"\n"}   end Release;{"\n"}end Resource;{"\n"}

</CodeBlock>
<MarginText>30</MarginText>
<p><em>Example of a single protected declaration and corresponding body:</em> <br /></p>

<MarginText>31/5</MarginText>
<MarginInfo items={["AI12-0430-1"]} />
<CodeBlock language="ada">
protected Shared_Array is{"\n"}   --  Index, Item, and Item_Array are global types{"\n"}   function  Component    (N : in Index) return  Item;{"\n"}   procedure Set_Component(N : in Index; E : in  Item);{"\n"}private{"\n"}   Table : Item_Array(Index) := (others  ={">"} Null_Item);{"\n"}end Shared_Array;{"\n"}
<MarginText>32</MarginText>
protected body Shared_Array is{"\n"}   function Component(N : in Index) return Item is{"\n"}   begin{"\n"}      return Table(N);{"\n"}   end Component;{"\n"}
<MarginText>33</MarginText>
   procedure Set_Component(N : in Index; E : in Item) is{"\n"}   begin{"\n"}      Table(N) := E;{"\n"}   end Set_Component;{"\n"}end Shared_Array;{"\n"}

</CodeBlock>

<p><em>Examples of protected objects:</em> <br /></p>

<MarginText>35</MarginText>

Control  : Resource;{"\n"}Flags    : array(1 .. 100) of Resource;{"\n"}

</CodeBlock>
