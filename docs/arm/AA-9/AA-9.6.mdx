<MarginText>1</MarginText>
<p>[ A <code><a href="../AA-9/AA-9.6#S0266">delay_statement</a></code> is used to block further execution until a specified <em>expiration time</em> is reached. The expiration time can be specified either as a particular point in time (in a <code><a href="../AA-9/AA-9.6#S0267">delay_until_statement</a></code>), or in seconds from the current time (in a <code><a href="../AA-9/AA-9.6#S0268">delay_relative_statement</a></code>). The language-defined package Calendar provides definitions for a type Time and associated operations, including a function Clock that returns the current time. ]<br /></p>


#### Syntax

<MarginText>2</MarginText>

<CodeBlock>
<code>delay_statement</code><a id="S0266"></a><code> ::= </code><code><a href="../AA-9/AA-9.6#S0267">delay_until_statement</a></code> | <code><a href="../AA-9/AA-9.6#S0268">delay_relative_statement</a></code><br />

</CodeBlock>

<CodeBlock>

</CodeBlock>
<MarginText>4</MarginText>

#### Name Resolution Rules

<MarginText>5</MarginText>
<p>The expected type for the <em>delay_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> in a <code><a href="../AA-9/AA-9.6#S0268">delay_relative_statement</a></code> is the predefined type Duration. The <em>delay_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> in a <code><a href="../AA-9/AA-9.6#S0267">delay_until_statement</a></code> is expected to be of any nonlimited type.<br /></p>


#### Legality Rules

<MarginText>6/3</MarginText>
<MarginInfo items={["AI05-0092-1"]} />
<p>There can be multiple time bases, each with a corresponding clock, and a corresponding <em>time type</em>. The type of the <em>delay_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> in a <code><a href="../AA-9/AA-9.6#S0267">delay_until_statement</a></code> shall be a time type &mdash; either the type Time defined in the language-defined package Calendar (see below), the type Time in the package Real_Time (see <a href="../AA-D/AA-D.8">D.8</a>), or some other implementation-defined time type. <br /></p>

<AnnotatedOnly>
<MarginText>6.a</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>Any implementation-defined time types.<br /></Admonition>
</AnnotatedOnly>


#### Static Semantics

<MarginText>7</MarginText>
<p>[There is a predefined fixed point type named Duration, declared in the visible part of package Standard;] a value of type Duration is used to represent the length of an interval of time, expressed in seconds. [The type Duration is not specific to a particular time base, but can be used with any time base.]<br /></p>

<MarginText>8/3</MarginText>
<MarginInfo items={["AI05-0092-1"]} />
<p>A value of the type Time in package Calendar, or of some other time type, represents a time as reported by a corresponding clock.<br /></p>

#### Dynamic Semantics

<MarginText>20</MarginText>
<p>For the execution of a <code><a href="../AA-9/AA-9.6#S0266">delay_statement</a></code>, the <em>delay_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is first evaluated. For a <code><a href="../AA-9/AA-9.6#S0267">delay_until_statement</a></code>, the expiration time for the delay is the value of the <em>delay_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>, in the time base associated with the type of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>. For a <code><a href="../AA-9/AA-9.6#S0268">delay_relative_statement</a></code>, the expiration time is defined as the current time, in the time base associated with relative delays, plus the value of the <em>delay_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> converted to the type Duration, and then rounded up to the next clock tick. The time base associated with relative delays is as defined in <a href="../AA-D/AA-D.9">D.9</a>, &ldquo;<a href="../AA-D/AA-D.9">Delay Accuracy</a>&rdquo; or is implementation defined. <br /></p>

<AnnotatedOnly>
<MarginText>20.a</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>The time base associated with relative delays.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>20.b</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>Rounding up to the next clock tick means that the reading of the delay-relative clock when the delay expires should be no less than the current reading of the delay-relative clock plus the specified duration. <br /></Admonition>
</AnnotatedOnly>

<MarginText>21</MarginText>
<p>The task executing a <code><a href="../AA-9/AA-9.6#S0266">delay_statement</a></code> is blocked until the expiration time is reached, at which point it becomes ready again. If the expiration time has already passed, the task is not blocked. <br /></p>

<AnnotatedOnly>
<MarginText>21.a</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>For a <code><a href="../AA-9/AA-9.6#S0268">delay_relative_statement</a></code>, this case corresponds to when the value of the <em>delay_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is zero or negative.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>21.b</MarginText>
<Admonition type="aarm" aarm="note">
Even though the task is not blocked, it might be put back on the end of its ready queue. See <a href="../AA-D/AA-D.2">D.2</a>, &ldquo;<a href="../AA-D/AA-D.2">Priority Scheduling</a>&rdquo;. <br /></Admonition>
</AnnotatedOnly>

<MarginText>22/3</MarginText>
<MarginInfo items={["AI05-0092-1"]} />
<p>If an attempt is made to <em>cancel</em> the <code><a href="../AA-9/AA-9.6#S0266">delay_statement</a></code> [(as part of an <code><a href="../AA-9/AA-9.7#S0280">asynchronous_select</a></code> or abort &mdash; see <a href="../AA-9/AA-9.7#Subclause_9.7.4">9.7.4</a> and <a href="../AA-9/AA-9.8">9.8</a>)], the statement is cancelled if the expiration time has not yet passed, thereby completing the <code><a href="../AA-9/AA-9.6#S0266">delay_statement</a></code>. <br /></p>

<AnnotatedOnly>
<MarginText>22.a</MarginText>
<Admonition type="aarm" aarm="reason">
<strong></strong>This is worded this way so that in an <code><a href="../AA-9/AA-9.7#S0280">asynchronous_select</a></code> where the <code><a href="../AA-9/AA-9.7#S0282">triggering_statement</a></code> is a <code><a href="../AA-9/AA-9.6#S0266">delay_statement</a></code>, an attempt to cancel the delay when the <code><a href="../AA-9/AA-9.7#S0283">abortable_part</a></code> completes is ignored if the expiration time has already passed, in which case the optional statements of the <code><a href="../AA-9/AA-9.7#S0281">triggering_alternative</a></code> are executed. <br /></Admonition>
</AnnotatedOnly>

<MarginText>23</MarginText>
<p>The time base associated with the type Time of package Calendar is implementation defined. The function Clock of package Calendar returns a value representing the current time for this time base. [The implementation-defined value of the named number System.Tick (see <a href="../AA-13/AA-13.7">13.7</a>) is an approximation of the length of the real-time interval during which the value of Calendar.Clock remains constant.] <br /></p>

<AnnotatedOnly>
<MarginText>23.a</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>The time base of the type Calendar.Time.<br /></Admonition>
</AnnotatedOnly>

<MarginText>24/2</MarginText>
<MarginInfo items={["AI95-00351-01"]} />
<p>The functions Year, Month, Day, and Seconds return the corresponding values for a given value of the type Time, as appropriate to an implementation-defined time zone; the procedure Split returns all four corresponding values. Conversely, the function Time_Of combines a year number, a month number, a day number, and a duration, into a value of type Time. The operators "+" and "&ndash;" for addition and subtraction of times and durations, and the relational operators for times, have the conventional meaning. <br /></p>

<AnnotatedOnly>
<MarginText>24.a/2</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>The time zone used for package Calendar operations.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24.b/3</MarginText>
<MarginInfo items={["AI05-0119-1"]} />
<Admonition type="aarm" aarm="ramification">
<strong></strong>The behavior of these values and subprograms if the time zone changes is also implementation-defined. In particular, the changes associated with summer time adjustments (like Daylight Savings Time in the United States) should be treated as a change in the implementation-defined time zone. The language does not specify whether the time zone information is stored in values of type Time; therefore the results of binary operators are unspecified when the operands are the two values with different effective time zones. In particular, the results of "-" may differ from the "real" result by the difference in the time zone adjustment. Similarly, the result of UTC_Time_Offset (see 9.6.1) may or may not reflect a time zone adjustment. <br /></Admonition>
</AnnotatedOnly>

<MarginText>25</MarginText>
<p>If Time_Of is called with a seconds value of 86_400.0, the value returned is equal to the value of Time_Of for the next day with a seconds value of 0.0. The value returned by the function Seconds or through the Seconds parameter of the procedure Split is always less than 86_400.0.<br /></p>

<MarginText>26/1</MarginText>
<MarginInfo items={["AI95-00113-01"]} />
<p>{"{"}<em>8652/0030</em>{"}"} The exception Time_Error is raised by the function Time_Of if the actual parameters do not form a proper date. This exception is also raised by the operators "+" and "&ndash;" if the result is not representable in the type Time or Duration, as appropriate. This exception is also raised by the functions Year, Month, Day, and Seconds and the procedure Split if the year number of the given date is outside of the range of the subtype Year_Number. <br /></p>

<AnnotatedOnly>
<MarginText>26.a/1</MarginText>
<MarginInfo items={["AI95-00160-01"]} />
<Admonition type="aarm" aarm="note">
<strong>To be honest: </strong>{"{"}<em>8652/0106</em>{"}"} By "proper date" above we mean that the given year has a month with the given day. For example, February 29th is a proper date only for a leap year. We do not mean to include the Seconds in this notion; in particular, we do not mean to require implementations to check for the &ldquo;missing hour&rdquo; that occurs when Daylight Savings Time starts in the spring. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>26.b/2</MarginText>
<MarginInfo items={["AI95-00113-01", "AI95-00351-01"]} />
<Admonition type="aarm" aarm="reason">
<strong></strong>{"{"}<em>8652/0030</em>{"}"} We allow Year and Split to raise Time_Error because the arithmetic operators are allowed (but not required) to produce times that are outside the range of years from 1901 to 2399. This is similar to the way integer operators may return values outside the base range of their type so long as the value is mathematically correct. We allow the functions Month, Day and Seconds to raise Time_Error so that they can be implemented in terms of Split. <br /></Admonition>
</AnnotatedOnly>


#### Implementation Requirements

<MarginText>27</MarginText>
<p>The implementation of the type Duration shall allow representation of time intervals (both positive and negative) up to at least 86400 seconds (one day); Duration'Small shall not be greater than twenty milliseconds. The implementation of the type Time shall allow representation of all dates with year numbers in the range of Year_Number[; it may allow representation of other dates as well (both earlier and later).] <br /></p>


#### Implementation Permissions

<MarginText>28/3</MarginText>
<MarginInfo items={["AI05-0092-1"]} />
<p>An implementation may define additional time types.<br /></p>

<MarginText>29</MarginText>
<p>An implementation may raise Time_Error if the value of a <em>delay_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> in a <code><a href="../AA-9/AA-9.6#S0267">delay_until_statement</a></code> of a <code><a href="../AA-9/AA-9.7#S0269">select_statement</a></code> represents a time more than 90 days past the current time. The actual limit, if any, is implementation-defined. <br /></p>

<AnnotatedOnly>
<MarginText>29.a</MarginText>
<Admonition type="aarm" aarm="implementation-defined">
<strong></strong>Any limit on <code><a href="../AA-9/AA-9.6#S0267">delay_until_statement</a></code>s of <code><a href="../AA-9/AA-9.7#S0269">select_statement</a></code>s.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>29.b</MarginText>
<Admonition type="aarm" aarm="implementation-note">
<strong></strong>This allows an implementation to implement <code><a href="../AA-9/AA-9.7#S0269">select_statement</a></code> timeouts using a representation that does not support the full range of a time type. In particular 90 days of seconds can be represented in 23 bits, allowing a signed 24-bit representation for the seconds part of a timeout. There is no similar restriction allowed for stand-alone <code><a href="../AA-9/AA-9.6#S0267">delay_until_statement</a></code>s, as these can be implemented internally using a loop if necessary to accommodate a long delay. <br /></Admonition>
</AnnotatedOnly>

<MarginText>40.c/2</MarginText>
<MarginInfo items={["AI95-00351-01"]} />
<Admonition type="aarm" aarm="note">
The type Time may represent dates with year numbers outside of Year_Number. Therefore, the operations "+" and "&ndash;" need only raise Time_Error if the result is not representable in Time (or Duration); also, Split or Year will now raise Time_Error if the year number is outside of Year_Number. This change is intended to simplify the implementation of "+" and "&ndash;" (allowing them to depend on overflow for detecting when to raise Time_Error) and to allow local time zone information to be considered at the time of Split rather than Clock (depending on the implementation approach). For example, in a POSIX environment, it is natural for the type Time to be based on GMT, and the results of procedure Split (and the functions Year, Month, Day, and Seconds) to depend on local time zone information. In other environments, it is more natural for the type Time to be based on the local time zone, with the results of Year, Month, Day, and Seconds being pure functions of their input.<br /></Admonition>
</AnnotatedOnly>

#### Inconsistencies With Ada 95

<AnnotatedOnly>
<MarginText>40.e/2</MarginText>
<MarginInfo items={["AI95-00351-01"]} />
<Admonition type="aarm" aarm="note">
The upper bound of Year_Number has been changed to avoid a year 2100 problem. A program which expects years past 2099 to raise Constraint_Error will fail in Ada 2005. We don't expect there to be many programs which are depending on an exception to be raised. A program that uses Year_Number'Last as a magic number may also fail if values of Time are stored outside of the program. procedure Split (Date       : in Time;{"\n"}                 Year       : out Year_Number;{"\n"}                 Month      : out Month_Number;{"\n"}                 Day        : out Day_Number;{"\n"}                 Seconds    : out Day_Duration;{"\n"}                 Leap_Second: out Boolean;{"\n"}                 Time_Zone  : in Time_Zones.Time_Offset := 0);{"\n"}

</CodeBlock>
<MarginText>80/2</MarginText>
<MarginInfo items={["AI95-00351-01", "AI95-00427-01"]} />
<p>If Date does not represent a time within a leap second, splits Date into its constituent parts (Year, Month, Day, Seconds), relative to the specified time zone offset, and sets Leap_Second to False. If Date represents a time within a leap second, set the constituent parts to values corresponding to a time one second earlier than that given by Date, relative to the specified time zone offset, and sets Leap_Seconds to True. The value returned in the Seconds parameter is always less than 86_400.0.<br /></p>

<MarginText>81/2</MarginText>
<CodeBlock language="ada">
function Image (Date : Time;{"\n"}                Include_Time_Fraction : Boolean := False;{"\n"}                Time_Zone  : Time_Zones.Time_Offset := 0) return String;{"\n"}

</CodeBlock>
<MarginText>82/2</MarginText>
<MarginInfo items={["AI95-00351-01"]} />
<p>Returns a string form of the Date relative to the given Time_Zone. The format is "Year-Month-Day Hour:Minute:Second", where the Year is a 4-digit value, and all others are 2-digit values, of the functions defined in Calendar and Calendar.Formatting, including a leading zero, if needed. The separators between the values are a minus, another minus, a colon, and a single space between the Day and Hour. If Include_Time_Fraction is True, the integer part of Sub_Seconds*100 is suffixed to the string as a point followed by a 2-digit value. <br /></p>

<AnnotatedOnly>
<MarginText>82.a/2</MarginText>
<Admonition type="aarm" aarm="discussion">
<strong></strong>The Image provides a string in ISO 8601 format, the international standard time format. Alternative representations allowed in ISO 8601 are not supported here.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>82.b/2</MarginText>
<Admonition type="aarm" aarm="note">
ISO 8601 allows 24:00:00 for midnight; and a seconds value of 60 for leap seconds. These are not allowed here (the routines mentioned above cannot produce those results). <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>82.c/2</MarginText>
<Admonition type="aarm" aarm="ramification">
<strong></strong>The fractional part is truncated, not rounded. It would be quite hard to define the result with proper rounding, as it can change all of the values of the image. Values can be rounded up by adding an appropriate constant (0.5 if Include_Time_Fraction is False, 0.005 otherwise) to the time before taking the image. <br /></Admonition>
</AnnotatedOnly>

#### Implementation Advice

<MarginText>89/2</MarginText>
<MarginInfo items={["AI95-00351-01"]} />
<p>An implementation should support leap seconds if the target system supports them. If leap seconds are not supported, Difference should return zero for Leap_Seconds, Split should return False for Leap_Second, and Time_Of should raise Time_Error if Leap_Second is True. <br /></p>

<AnnotatedOnly>
<MarginText>89.a/2</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<strong></strong>Leap seconds should be supported if the target system supports them. Otherwise, operations in Calendar.Formatting should return results consistent with no leap seconds.<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>89.b/2</MarginText>
<Admonition type="aarm" aarm="implementation-advice">
<strong></strong>An implementation can always support leap seconds when the target system does not; indeed, this isn't particularly hard (all that is required is a table of when leap seconds were inserted). As such, leap second support isn't &ldquo;impossible or impractical&rdquo; in the sense of <a href="../AA-1/AA-1.1#Subclause_1.1.3">1.1.3</a>. However, for some purposes, it may be important to follow the target system's lack of leap second support (if the target is a GPS satellite, which does not use leap seconds, leap second support would be a handicap to work around). Thus, this  should be read as giving permission to not support leap seconds on target systems that don't support leap seconds. Implementers should use the needs of their customers to determine whether or not support leap seconds on such targets. <br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>90/5</MarginText>
<MarginInfo items={["AI95-00351-01", "AI12-0336-1", "AI12-0442-1"]} />
<Admonition type="aarm" aarm="note">
NOTE 1   The implementation-defined time zone of package Calendar can  be the local time zone. Local_Time_Offset  always returns the difference relative to the implementation-defined time zone of package Calendar. If Local_Time_Offset  does not raise Unknown_Zone_Error, UTC time can be safely calculated (within the accuracy of the underlying time-base).<br /></Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>90.a/2</MarginText>
<MarginInfo items={["AI95-00351-01"]} />
<Admonition type="aarm" aarm="discussion">
<strong></strong>The time in the time zone known as Greenwich Mean Time (GMT) is generally very close to UTC time; for most purposes they can be treated the same. GMT is the time based on the rotation of the Earth; UTC is the time based on atomic clocks, with leap seconds periodically inserted to realign with GMT (because most human activities depend on the rotation of the Earth). At any point in time, there will be a sub-second difference between GMT and UTC. <br /></Admonition>
</AnnotatedOnly>

NOTE 2   Calling Split on the results of subtracting Duration(Local_Time_Offset *60) from Clock provides the components (hours, minutes, and so on) of the UTC time. In the United States, for example, Local_Time_Offset  will generally be negative. <br /></Admonition>
