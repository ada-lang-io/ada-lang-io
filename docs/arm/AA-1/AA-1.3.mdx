To see the difference, consider &ldquo;aliased view&rdquo;. The descriptive definition is &ldquo;a view of an object that can be designated by an access value&rdquo;. The formal definition is a lengthy paragraph in <a href="/docs/arm/AA-3/AA-3.10">3.10</a> that defines 10 separate constructs as aliased views. In order to reason about a language rule that requires an aliased view in a particular context, one needs to look at the list of constructs in order to determine if the object in hand is actually an aliased view. The descriptive definition is no help in this case. <br />
</Admonition>

<Admonition type="aarm" aarm="discussion">
<b></b>Here are some AARM-only definitions: The Ada Rapporteur Group (ARG) interprets the Ada Reference Manual. An Ada Issue (AI) is a numbered ruling from the ARG. Ada Issues created for Ada 83 are denoted as "AI83", while Ada Issues created for Ada 95 are denoted as "AI95" in this document. Similarly, Ada Issues created for Ada 2005 are denoted as "AI05", Ada Issues created for Ada 2012 are denoted as "AI12", and Ada Issues created for Ada 2022 are denoted as "AI22". The Ada Commentary Integration Document (ACID) is an edition of the Ada 83 RM in which clearly marked insertions and deletions indicate the effect of integrating the approved AIs. The Uniformity Rapporteur Group (URG) issued recommendations intended to increase uniformity across Ada implementations. The functions of the URG have been assumed by the ARG. A Uniformity Issue (UI) was a numbered recommendation from the URG. A Defect Report and Response is an official query to WG9 about an error in the standard. Defect Reports are processed by the ARG, and are referenced here by their ISO numbers: 8652/nnnn. Most changes to the Ada 95 standard include reference(s) to the Defect Report(s) that prompted the change. The <i>Ada Conformity Assessment Test Suite (ACATS)</i> is a set of tests intended to check the conformity of Ada implementations to this standard. This set of tests was previously known as the Ada Compiler Validation Capability (ACVC). <br />
</Admonition>

## 1.3.1  Types, Objects, and their Properties

<MarginText>1/5_1.3.1</MarginText>
<dt><br/><b>abstract type</b></dt>
<dl>
<dd>tagged type intended for use as an ancestor of other types, but which is not allowed to have objects of its own<br /></dd>
<MarginText>2/5_1.3.1</MarginText>
<dt><br/><b>access type</b></dt>
<dd>type that has values that designate aliased objects<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>3/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Access types correspond to &ldquo;pointer types&rdquo; or &ldquo;reference types&rdquo; in some other languages.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>4/5_1.3.1</MarginText>
<dt><br/><b>accessibility level</b></dt>
<dl>
<dd>representation of the lifetime of an entity in terms of the level of dynamic nesting within which the entity is known to exist<br /></dd>
<MarginText>5/5_1.3.1</MarginText>
<dt><br/><b>aliased view</b></dt>
<dd>view of an object that can be designated by an access value<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>6/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Objects allocated by allocators are aliased. Objects can also be explicitly declared as aliased with the reserved word aliased. The Access attribute can be used to create an access value designating an aliased object.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>7/5_1.3.1</MarginText>
<dt><br/><b>ancestor of a type</b></dt>
<dl>
<dd>type itself or, in the case of a type derived from other types, its parent type or one of its progenitor types or one of their ancestors<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>8/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Ancestor and descendant are inverse relationships.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>9/5_1.3.1</MarginText>
<dt><br/><b>array type</b></dt>
<dl>
<dd>composite type whose components are all of the same type<br /></dd>
<MarginText>10/5_1.3.1</MarginText>
<dt><br/><b>aspect</b></dt>
<dd>specifiable property of an entity<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>11/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
<code><a href="/docs/arm/AA-13/AA-13.1#S0346">aspect_specification</a></code> on the declaration of the entity. Some aspects can be queried via attributes.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>12/5_1.3.1</MarginText>
<dt><br/><b>attribute</b></dt>
<dl>
<dd>characteristic or property of an entity that can be queried, and in some cases specified<br /></dd>
<MarginText>13/5_1.3.1</MarginText>
<dt><br/><b>category of types</b></dt>
<dd>set of types with one or more common properties, such as primitive operations<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>14/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: A category of types that is closed under derivation is also known as a class.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>15/5_1.3.1</MarginText>
<dt><br/><b>character type</b></dt>
<dl>
<dd>enumeration type whose values include characters<br /></dd>
<MarginText>16/5_1.3.1</MarginText>
<dt><br/><b>class of types</b></dt>
<dd>set of types that is closed under derivation, which means that if a given type is in the class, then all types derived from that type are also in the class<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>17/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: The set of types of a class share common properties, such as their primitive operations.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>18/5_1.3.1</MarginText>
<dt><br/><b>composite type</b></dt>
<dl>
<dd>type with components, such as an array or record<br /></dd>
<MarginText>19/5_1.3.1</MarginText>
<dt><br/><b>controlled type</b></dt>
<dd>type that supports user-defined assignment and finalization<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>20/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Objects are always finalized before being destroyed.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>21/5_1.3.1</MarginText>
<dt><br/><b>default initial condition</b></dt>
<dl>
<dd>property that holds for every default-initialized object of a given type<br /></dd>
<MarginText>22/5_1.3.1</MarginText>
<dt><br/><b>derived type</b></dt>
<dd>type defined in terms of a parent type and zero or more progenitor types given in a derived type definition<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>23/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note 1: A derived type inherits properties such as components and primitive operations from its parent and progenitors.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>24/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note 2: A type together with the types derived from it (directly or indirectly) form a derivation class.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>25/5_1.3.1</MarginText>
<dt><br/><b>descendant of a type</b></dt>
<dl>
<dd>type itself or a type derived (directly or indirectly) from it<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>26/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Descendant and ancestor are inverse relationships.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>27/5_1.3.1</MarginText>
<dt><br/><b>discrete type</b></dt>
<dl>
<dd>type that is either an integer type or an enumeration type<br /></dd>
<MarginText>28/5_1.3.1</MarginText>
<dt><br/><b>discriminant</b></dt>
<dd>parameter for a composite type, which can control, for example, the bounds of a component that is an array<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>29/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: A discriminant for a task type can be used to pass data to a task of the type upon its creation.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>30/5_1.3.1</MarginText>
<dt><br/><b>elementary type</b></dt>
<dl>
<dd>type that does not have components<br /></dd>
<MarginText>31/5_1.3.1</MarginText>
<dt><br/><b>enumeration type</b></dt>
<dd>type defined by an enumeration of its values, which can be denoted by identifiers or character literals<br /></dd>
<MarginText>32/5_1.3.1</MarginText>
<dt><br/><b>full type</b></dt>
<dd>type that defines a full view<br /></dd>
<MarginText>33/5_1.3.1</MarginText>
<dt><br/><b>full view</b></dt>
<dd>view of a type that reveals all of its properties<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>34/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: There can be other views of the type that reveal fewer properties.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>35/5_1.3.1</MarginText>
<dt><br/><b>incomplete type</b></dt>
<dl>
<dd>type that defines an incomplete view<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>36/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Incomplete types can be used for defining recursive data structures.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>37/5_1.3.1</MarginText>
<dt><br/><b>incomplete view</b></dt>
<dl>
<dd>view of a type that reveals minimal properties<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>38/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: The remaining properties are defined by the full view given elsewhere.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>39/5_1.3.1</MarginText>
<dt><br/><b>indexable container type</b></dt>
<dl>
<dd>type that has user-defined behavior for indexing, via the Constant{"_"}Indexing or Variable{"_"}Indexing aspects<br /></dd>
<MarginText>40/5_1.3.1</MarginText>
<dt><br/><b>integer type</b></dt>
<dd>type that represents signed or modular integers<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>41/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: A signed integer type has a base range that includes both positive and negative numbers, and has operations that can raise an exception when the result is outside the base range. A modular type has a base range whose lower bound is zero, and has operations with &ldquo;wraparound&rdquo; semantics. Modular types subsume what are called &ldquo;unsigned types&rdquo; in some other languages.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>42/5_1.3.1</MarginText>
<dt><br/><b>interface type</b></dt>
<dl>
<dd>abstract tagged type that has no components or concrete operations except possibly null procedures<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>43/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Interface types are used for composing other interfaces and tagged types and thereby provide multiple inheritance. Only an interface type can be used as a progenitor of another type.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>44/5_1.3.1</MarginText>
<dt><br/><b>invariant</b></dt>
<dl>
<dd>assertion that is expected to be True for all objects of a given private type when viewed from outside the defining package<br /></dd>
<MarginText>45/5_1.3.1</MarginText>
<dt><br/><b>iterable container type</b></dt>
<dd>type that has user-defined behavior for iteration, via the Default{"_"}Iterator and Iterator{"_"}Element aspects<br /></dd>
<MarginText>46/5_1.3.1</MarginText>
<dt><br/><b>limited type</b></dt>
<dd>type for which copying (such as in an <code><a href="/docs/arm/AA-5/AA-5.2#S0173">assignment_statement</a></code>) is not allowed<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>47/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: All types are either limited types or nonlimited types.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>48/5_1.3.1</MarginText>
<dt><br/><b>needed component</b></dt>
<dl>
<dd>component of a record type or record extension that is required to have its value specified within a given aggregate<br /></dd>
<MarginText>49/5_1.3.1</MarginText>
<dt><br/><b>nominal subtype</b></dt>
<dd>subtype specified when a view of an object is defined<br /></dd>
<MarginText>50/5_1.3.1</MarginText>
<dt><br/><b>nonlimited type</b></dt>
<dd>type for which copying is allowed<br /></dd>
<MarginText>51/5_1.3.1</MarginText>
<dt><br/><b>object</b></dt>
<dd>entity that contains a value, and is either a constant or a variable<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>52/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
<code><a href="/docs/arm/AA-3/AA-3.3#S0032">object_declaration</a></code> or by an <code><a href="/docs/arm/AA-4/AA-4.8#S0164">allocator</a></code>. A formal parameter is (a view of) an object. A subcomponent of an object is an object.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>53/5_1.3.1</MarginText>
<dt><br/><b>operational aspect</b></dt>
<dl>
<dd>aspect that indicates a logical property of an entity, such as the precondition of a subprogram, or the procedure used to write a given type of object to a stream<br /></dd>
<MarginText>54/5_1.3.1</MarginText>
<dt><br/><b>parent of a derived type</b></dt>
<dd>first ancestor type given in the definition of the derived type<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>55/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: The parent can be almost any kind of type, including an interface type.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>56/5_1.3.1</MarginText>
<dt><br/><b>partial view</b></dt>
<dl>
<dd>view of a type that reveals only some of its properties<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>57/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: The remaining properties are defined by the full view given elsewhere.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>58/5_1.3.1</MarginText>
<dt><br/><b>primitive operations of a type</b></dt>
<dl>
<dd>operations (such as subprograms) declared together with the type declarations<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>59/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Primitive operations are inherited by other types in the same derivation class of types.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>60/5_1.3.1</MarginText>
<dt><br/><b>private extension</b></dt>
<dl>
<dd>type that extends another type, with the additional properties hidden from its clients<br /></dd>
<MarginText>61/5_1.3.1</MarginText>
<dt><br/><b>private type</b></dt>
<dd>type that defines a partial view<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>62/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Private types can be used for defining abstractions that hide unnecessary details from their clients.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>63/5_1.3.1</MarginText>
<dt><br/><b>progenitor</b></dt>
<dl>
<dd>type given in the interface list, if any, of an interface, task, protected, or derived type definition<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>64/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: A progenitor is always an interface type.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>65/5_1.3.1</MarginText>
<dt><br/><b>protected type</b></dt>
<dl>
<dd>composite type whose components are accessible only through one of its protected operations, which synchronize concurrent access by multiple tasks<br /></dd>
<MarginText>66/5_1.3.1</MarginText>
<dt><br/><b>real type</b></dt>
<dd>type that has values that are approximations of the real numbers<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>67/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Floating point and fixed point types are real types.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>68/5_1.3.1</MarginText>
<dt><br/><b>record extension</b></dt>
<dl>
<dd>type that extends another type optionally with additional components<br /></dd>
<MarginText>69/5_1.3.1</MarginText>
<dt><br/><b>record type</b></dt>
<dd>composite type consisting of zero or more named components, possibly of different types<br /></dd>
<MarginText>70/5_1.3.1</MarginText>
<dt><br/><b>reference type</b></dt>
<dd>type that has user-defined behavior for &ldquo;.all&rdquo;, defined by the Implicit{"_"}Dereference aspect<br /></dd>
<MarginText>71/5_1.3.1</MarginText>
<dt><br/><b>representation aspect</b></dt>
<dd>aspect that indicates how an entity is mapped onto the underlying hardware, for example the size or alignment of an object<br /></dd>
<MarginText>72/5_1.3.1</MarginText>
<dt><br/><b>scalar type</b></dt>
<dd>either a discrete type or a real type<br /></dd>
<MarginText>73/5_1.3.1</MarginText>
<dt><br/><b>stable property</b></dt>
<dd>characteristic associated with objects of a given type that is preserved by many of the primitive operations of the type<br /></dd>
<MarginText>74/5_1.3.1</MarginText>
<dt><br/><b>storage pool object</b></dt>
<dd>object associated with one or more access types from which the storage for objects created by allocators of the access type(s) is obtained<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>75/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Some storage pools can be partitioned into subpools in order to support finer-grained storage management.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>76/5_1.3.1</MarginText>
<dt><br/><b>stream</b></dt>
<dl>
<dd>sequence of elements that can be used, along with the stream-oriented attributes, to support marshalling and unmarshalling of values of most types<br /></dd>
<MarginText>77/5_1.3.1</MarginText>
<dt><br/><b>subtype</b></dt>
<dd>type together with optional constraints, null exclusions, and predicates, which constrain the values of the type to the subset that satisfies the implied conditions<br /></dd>
<MarginText>78/5_1.3.1</MarginText>
<dt><br/><b>synchronized</b></dt>
<dd>can be safely operated on by multiple tasks concurrently<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>79/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Synchronized is used to qualify entities, as in a synchronized interface.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>80/5_1.3.1</MarginText>
<dt><br/><b>tagged type</b></dt>
<dl>
<dd>type whose objects each have a run-time type tag, which indicates the specific type for which the object was originally created<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>81/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Tagged types can be extended with additional components.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>82/5_1.3.1</MarginText>
<dt><br/><b>task type</b></dt>
<dl>
<dd>composite type used to represent active entities which execute concurrently and that can communicate via queued task entries<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>83/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: The top-level task of a partition is called the environment task.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>84/5_1.3.1</MarginText>
<dt><br/><b>type</b></dt>
<dl>
<dd>defining characteristic of each object and expression of the language, with an associated set of values, and a set of primitive operations that implement the fundamental aspects of its semantics<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>85/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: Types are grouped into categories. Most language-defined categories of types are also classes of types.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>86/5_1.3.1</MarginText>
<dt><br/><b>view of an entity</b></dt>
<dl>
<dd>representation of an entity that reveals some or all of the properties of the entity<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>87/5_1.3.1</MarginText>
<Admonition type="aarm" aarm="note">
Note: A single entity can have multiple views.<br />
</Admonition>
</AnnotatedOnly>

## 1.3.2  Subprograms and their Properties

<MarginText>1/5_1.3.2</MarginText>
<dt><br/><b>function</b></dt>
<dl>
<dd>form of subprogram that returns a result and can be called as part of an expression<br /></dd>
<MarginText>2/5_1.3.2</MarginText>
<dt><br/><b>overriding operation</b></dt>
<dd>operation that replaces an inherited primitive operation<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>3/5_1.3.2</MarginText>
<Admonition type="aarm" aarm="note">
Note: Operations can be marked explicitly as overriding or not overriding.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>4/5_1.3.2</MarginText>
<dt><br/><b>postcondition</b></dt>
<dl>
<dd>assertion that is expected to be True when a given subprogram returns normally<br /></dd>
<MarginText>5/5_1.3.2</MarginText>
<dt><br/><b>precondition</b></dt>
<dd>assertion that is expected to be True when a given subprogram is called<br /></dd>
<MarginText>6/5_1.3.2</MarginText>
<dt><br/><b>procedure</b></dt>
<dd>form of subprogram that does not return a result and can only be invoked by a statement<br /></dd>
<MarginText>7/5_1.3.2</MarginText>
<dt><br/><b>subprogram</b></dt>
<dd>unit of a program that can be brought into execution in various contexts, with the invocation being a subprogram call that can parameterize the effect of the subprogram through the passing of operands<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>8/5_1.3.2</MarginText>
<Admonition type="aarm" aarm="note">
Note: There are two forms of subprograms: functions, which return values, and procedures, which do not.<br />
</Admonition>
</AnnotatedOnly>

<a id="Subclause_1.3.3"></a>

## 1.3.3  Other Syntactic Constructs

<MarginText>1/5_1.3.3</MarginText>
<dt><br/><b>aggregate</b></dt>
<dl>
<dd>construct used to define a value of a composite type by specifying the values of the components of the type<br /></dd>
<MarginText>2/5_1.3.3</MarginText>
<dt><br/><b>compilation unit</b></dt>
<dd>program unit that is separately compiled<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>3/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
<code><a href="/docs/arm/AA-10/AA-10.1#S0286">compilation_unit</a></code> contains either the declaration, the body, or a renaming of a program unit.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>4/5_1.3.3</MarginText>
<dt><br/><b>construct</b></dt>
<dl>
<dd>piece of text (explicit or implicit) that is an instance of a syntactic category defined under Syntax<br /></dd>
<MarginText>5/5_1.3.3</MarginText>
<dt><br/><b>container</b></dt>
<dd>structured object that represents a collection of elements all of the same (potentially class-wide) type, such as a vector or a tree<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>6/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
Note: Several predefined container types are provided by the children of package Ada.Containers (see <a href="/docs/arm/AA-A/AA-A.18#Subclause_A.18.1">A.18.1</a>).<br />
</Admonition>
</AnnotatedOnly>

<MarginText>7/5_1.3.3</MarginText>
<dt><br/><b>container aggregate</b></dt>
<dl>
<dd>construct used to define a value of a type that represents a collection of elements, by explicitly specifying the elements in the collection<br /></dd>
<MarginText>8/5_1.3.3</MarginText>
<dt><br/><b>core language</b></dt>
<dd>clause or annex in which are defined language constructs or capabilities that are provided by all conforming implementations<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>9/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
Note: A construct is said to be part of the core language if it is defined in a core language clause or annex.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>10/5_1.3.3</MarginText>
<dt><br/><b>declaration</b></dt>
<dl>
<dd>language construct that associates a name with (a view of) an entity<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>11/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
Note: A declaration can appear explicitly in the program text (an explicit declaration), or can be supposed to occur at a given place in the text as a consequence of the semantics of another construct (an implicit declaration).<br />
</Admonition>
</AnnotatedOnly>

<MarginText>12/5_1.3.3</MarginText>
<dt><br/><b>generic instance</b></dt>
<dl>
<dd>nongeneric unit created by the instantiation of a generic unit<br /></dd>
<MarginText>13/5_1.3.3</MarginText>
<dt><br/><b>generic unit</b></dt>
<dd>template for a (nongeneric) program unit<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>14/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
Note 1: The template can be parameterized by objects, types, subprograms, and packages.<br />
</Admonition>
</AnnotatedOnly>

<AnnotatedOnly>
<MarginText>15/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
Note 2: Generic units can be used to perform the role that macros sometimes play in other languages.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>16/5_1.3.3</MarginText>
<dt><br/><b>iterator</b></dt>
<dl>
<dd>construct that is used to loop over the elements of an array or container<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>17/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
Note: Iterators can be user defined, and can perform arbitrary computations to access elements from a container.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>18/5_1.3.3</MarginText>
<dt><br/><b>iterator filter</b></dt>
<dl>
<dd>construct that is used to restrict the elements produced by an iteration to those for which a boolean condition evaluates to True<br /></dd>
<MarginText>19/5_1.3.3</MarginText>
<dt><br/><b>library unit</b></dt>
<dd>separately compiled program unit, which is a package, a subprogram, or a generic unit<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>20/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
Note: Library units can have other (logically nested) library units as children, and can have other program units physically nested within them. A root library unit, together with its children and grandchildren and so on, form a subsystem.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>21/5_1.3.3</MarginText>
<dt><br/><b>master construct</b></dt>
<dl>
<dd>one of certain executable constructs for which there can be objects or tasks whose lifetime ends when the construct completes<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>22/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
Note: Execution of a master construct is a master, with which objects and tasks are associated for the purposes of waiting and finalization.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>23/5_1.3.3</MarginText>
<dt><br/><b>needed compilation unit</b></dt>
<dl>
<dd>compilation unit that is necessary to produce an executable partition, because some entity declared or defined within the unit is used elsewhere in the partition<br /></dd>
<MarginText>24/5_1.3.3</MarginText>
<dt><br/><b>package</b></dt>
<dd>program unit that defines the interface to a group of logically related entities, along with their implementation<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>25/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
Note: Typically, a package contains the declaration of a type (often a private type or private extension) along with the declarations of primitive subprograms of the type, which can be called from outside the package, while their inner workings remain hidden from outside users.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>26/5_1.3.3</MarginText>
<dt><br/><b>parallel construct</b></dt>
<dl>
<dd>executable construct that defines multiple activities of a single task that can proceed in parallel, via the execution of multiple logical threads of control<br /></dd>
<MarginText>27/5_1.3.3</MarginText>
<dt><br/><b>partition</b></dt>
<dd>part of a program, which consists of a set of interdependent library units<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>28/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
Note: Each partition can run in a separate address space, possibly on a separate computer. A program can contain just one partition, or it can be distributed across multiple partitions, which can execute concurrently.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>29/5_1.3.3</MarginText>
<dt><br/><b>pragma</b></dt>
<dl>
<dd>compiler directive to provide control over and above that provided by the other syntactic constructs of the language<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>30/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
Note: There are language-defined pragmas that give instructions for optimization, listing control, etc. An implementation can support additional (implementation-defined) pragmas.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>31/5_1.3.3</MarginText>
<dt><br/><b>program</b></dt>
<dl>
<dd>set of partitions, each of which can execute in a separate address space, possibly on a separate computer<br /></dd>
<MarginText>32/5_1.3.3</MarginText>
<dt><br/><b>program unit</b></dt>
<dd>language construct that is a package, a task unit, a protected unit, a protected entry, a generic unit, or an explicitly declared subprogram other than an enumeration literal<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>33/5_1.3.3</MarginText>
<Admonition type="aarm" aarm="note">
Note: Certain kinds of program units can be separately compiled. Alternatively, they can appear physically nested within other program units.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>34/5_1.3.3</MarginText>
<dt><br/><b>reduction expression</b></dt>
<dl>
<dd>expression that defines how to map or transform a collection of values into a new set of values, and then summarize the values by applying an operation to reduce the set to a single value<br /></dd>
<MarginText>35/5_1.3.3</MarginText>
<dt><br/><b>renaming</b></dt>
<dd>declaration that does not define a new entity, but instead defines a new view of an existing entity<br /></dd>
<MarginText>36/5_1.3.3</MarginText>
<dt><br/><b>specialized needs annex</b></dt>
<dd>annex in which are defined language constructs or capabilities that are not necessarily provided by all conforming implementations<br /></dd>
<MarginText>37/5_1.3.3</MarginText>
<dt><br/><b>subunit</b></dt>
<dd>body of a program unit that can be compiled separately from its enclosing program unit<br /></dd>
<a id="Subclause_1.3.4"></a>
</dl>

## 1.3.4  Runtime Actions

<MarginText>1/5_1.3.4</MarginText>
<dt><br/><b>assertion</b></dt>
<dl>
<dd>boolean expression that is expected to be True at run time at certain specified places<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>2/5_1.3.4</MarginText>
<Admonition type="aarm" aarm="note">
Note: Certain pragmas and aspects define various kinds of assertions.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>3/5_1.3.4</MarginText>
<dt><br/><b>elaboration</b></dt>
<dl>
<dd>process by which a declaration achieves its run-time effect<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>4/5_1.3.4</MarginText>
<Admonition type="aarm" aarm="note">
Note: Elaboration is one of the forms of execution.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>5/5_1.3.4</MarginText>
<dt><br/><b>evaluation</b></dt>
<dl>
<dd>process by which an expression achieves its run-time effect<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>6/5_1.3.4</MarginText>
<Admonition type="aarm" aarm="note">
Note: Evaluation is one of the forms of execution.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>7/5_1.3.4</MarginText>
<dt><br/><b>execution</b></dt>
<dl>
<dd>process by which a construct achieves its run-time effect<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>8/5_1.3.4</MarginText>
<Admonition type="aarm" aarm="note">
Note: Execution of a declaration is also called elaboration. Execution of an expression is also called evaluation.<br />
</Admonition>
</AnnotatedOnly>

<MarginText>9/5_1.3.4</MarginText>
<dt><br/><b>logical thread of control</b></dt>
<dl>
<dd>activity within the execution of a program that can proceed in parallel with other activities of the same task, or of separate tasks<br /></dd>
<MarginText>10/5_1.3.4</MarginText>
<dt><br/><b>master</b></dt>
<dd>execution of a master construct<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>11/5_1.3.4</MarginText>
<Admonition type="aarm" aarm="note">
Note: Each object and task is associated with a master. When a master is left, associated tasks are awaited and associated objects are finalized.<br />
</Admonition>
</AnnotatedOnly>

## 1.3.5  Exceptional Situations

<MarginText>1/5_1.3.5</MarginText>
<dt><br/><b>check</b></dt>
<dl>
<dd>test made during execution to determine whether a language rule has been violated<br /></dd>
<MarginText>2/5_1.3.5</MarginText>
<dt><br/><b>exception</b></dt>
<dd>kind of exceptional situation<br /></dd>
<MarginText>3/5_1.3.5</MarginText>
<dt><br/><b>exception occurrence</b></dt>
<dd>run-time occurrence of an exceptional situation<br /></dd>
<MarginText>4/5_1.3.5</MarginText>
<dt><br/><b>handle an exception</b></dt>
<dd>perform some actions in response to the arising of an exception<br /></dd>
<MarginText>5/5_1.3.5</MarginText>
<dt><br/><b>raise an exception</b></dt>
<dd>abandon normal program execution so as to draw attention to the fact that the corresponding situation has arisen<br /></dd>
<MarginText>6/5_1.3.5</MarginText>
<dt><br/><b>suppress a check</b></dt>
<dd>assert that the check cannot fail, and request that the compiler optimize by disabling the check<br /></dd>
</dl>
<AnnotatedOnly>
<MarginText>7/5_1.3.5</MarginText>
<Admonition type="aarm" aarm="note">
Note: The compiler is not required to honor this request. Suppressing checks that can fail can cause a program to behave in arbitrary ways.<br />
</Admonition>
</AnnotatedOnly>
